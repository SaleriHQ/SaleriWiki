## 摘要

本章讨论开发人员测试的方法和最佳实践。核心观点：测试优先开发是最有价值实践之一；测试本身不足以保证质量；结构化基础测试确保覆盖所有路径。

## 测试在质量中的角色

**测试的限制**：
- 开发人员测试往往是"干净测试"
- 实际覆盖率只有50-60%而非声称的95%
- 跳过复杂覆盖率类型

**推荐时间分配**：
- 小型项目：20-25%
- 中型项目：15-20%
- 大型项目：8-15%

## 测试优先开发

**先测试的优势**：
1. 重新排序而非增加工作
2. 更早发现和修复缺陷
3. 迫使提前思考需求和设计
4. 尽早暴露需求问题

## 结构化基础测试

**计算所需测试用例**：
1. 从1开始
2. 每个 if/while/for/and/or 加1
3. 每个 case 选项加1
4. 无 default 的 case 再加1

**示例**：
```java
if (x < 10) { ... }  // 需要2个用例：x<10, x>=10
```

## 数据流测试

**数据状态**：
- 定义：初始化但未使用
- 使用：用于计算
- 销毁：取消定义

**异常模式**：
- 定义-定义：浪费
- 定义-销毁：缺少使用
- 销毁-使用：逻辑错误

## 边界分析

**三种边界情况**：
- 刚好小于边界
- 边界值
- 刚好大于边界

## 错误类型分布

| 错误类型 | 百分比 |
|---------|-------|
| 结构错误 | 25% |
| 数据错误 | 22% |
| 功能误解 | 16% |
| 构建错误 | 10% |

**关键发现**：
- 80%错误集中在20%代码中
- 50%错误集中在5%类中
- 85%错误可在单例程内修复
