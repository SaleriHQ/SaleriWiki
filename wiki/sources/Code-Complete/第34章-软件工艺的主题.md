## 摘要

本章是《Code Complete》抽象主题的总结，涵盖软件工艺的核心主题：复杂性管理、过程选择、为人编程、编程进入语言、约定、问题领域编程、警告信号、迭代和避免编程宗教。这些主题解释了 hacking 和软件工艺之间的区别。

## 关键概念

### 征服复杂性

**核心原则**：管理复杂性是软件开发的核心技术 imperatives。

**具体技术**：
- 将系统划分为子系统
- 仔细定义类接口，隐藏内部工作
- 避免全局数据（增加需要 juggle 的代码百分比）
- 避免深层继承层次
- 避免深度嵌套循环
- 避免 goto
- 系统化错误处理
- 保持例程简短
- 使用清晰、自解释的变量名
- 最小化参数数量
- 使用约定免除任意差异的记忆负担

**抽象是管理复杂性的关键**：
- 机器语言 → 高级语言（最大进步）
- 例程（第二大进步）
- 类和包
- 功能命名变量
- 命名常量

### 选择过程

**核心原则**：开发软件的过程比想象的更重要。

**关键洞察**：
- 小项目：个人程序员才能是最大因素
- 大项目：组织特征比个人技能产生更大差异
- 质量必须从第一步开始构建
- 过早优化是过程错误
- 遵循伪代码编程过程保证有注释

**过程影响质量**：
- 需求错误比构建错误成本高得多
- 架构后跟审查、设计后跟审查、代码后跟审查
- 测试只能显示错误存在，不能证明无错

### 为人编写程序

**核心原则**：代码可读性影响可理解性、可审查性、错误率、调试、可修改性、开发时间。

**关键洞察**：
- 可读代码不比混乱代码花更长时间编写
- 私有程序→公共程序转换需要可读性
- 平均程序被10代维护程序员重写
- 维护程序员花50-60%时间理解代码
- 专业程序员编写可读代码，句号

### 编程进入语言

**核心原则**：思考想做什么，评估如何用可用工具实现，而非受语言限制。

**示例**：
- 语言不支持断言？编写自己的 assert()
- 语言不支持枚举？用命名约定定义枚举
- 使用约定弥补语言弱点

### 借助约定

**约定的好处**：
1. **节省决定任意问题的时间**
2. **简洁传达重要信息**（命名约定、大小写、缩进）
3. **防范已知危险**（禁止危险实践）
4. **增加可预测性**（读者知道期待什么）
5. **弥补语言弱点**

### 问题领域编程

**抽象层次结构**：

| 层次 | 说明 |
|------|------|
| 0 | 操作系统操作和机器指令 |
| 1 | 编程语言结构和工具 |
| 2 | 低级实现结构（栈、队列、链表、树） |
| 3 | 低级问题领域术语（业务对象层） |
| 4 | 高级问题领域术语 |

**实践技术**：
- 使用类实现问题领域结构
- 隐藏低级数据类型实现
- 使用命名常量文档化字面量
- 分配中间变量文档化计算
- 使用布尔函数澄清复杂测试

### 警惕警告信号

**警告信号类型**：
- "棘手代码" = 坏代码的代号
- 一个类比平均有更多错误
- 大量调试 = 没有聪明工作
- 设计度量警告：
  - 类超过7个成员
  - 例程超过10个决策点
  - 超过3层逻辑嵌套
  - 不寻常数量的变量
  - 高耦合、低内聚

**响应策略**：
- 将警告信号视为"怀疑的刺激"
- 使程序中的问题难以忽略（释放后将指针设为 null）
- 不要忽略编译器警告

### 迭代

**迭代适用场景**：
- 需求规格说明（与用户迭代多个版本）
- 系统设计（分几次交付的灵活性）
- 原型设计（快速廉价开发替代方案）
- 日程估算（多种技术比较）
- 设计（heuristic 过程，需要迭代改进）
- 代码调优（调优、测量、再调优）
- 审查（检查点验证质量）

### 避免编程宗教

**核心原则**：折中主义优于教条主义。

**关键观点**：
- 宗教形式：对单一设计方法的教条坚持、对特定格式的坚定信念、对全局数据的狂热回避
- 成功 = 实验 + 愿意根据结果改变信念
- 不要抓住最新时尚，使用混合方法
- 工具箱 metaphor：工具选择不是固定的

**实验的层次**：
- 架构设计：三种不同设计方法
- 详细设计：三种不同低级设计
- 编程语言：实验程序测试语言特性
- 代码调优：benchmark 验证
- 开发过程：收集质量/生产力数据

## 关键要点

1. **复杂性是敌人** - 编程的主要目标是管理复杂性
2. **过程决定质量** - 开发过程比想象的更重要
3. **人为先，计算机次之** - 代码可读性决定可维护性
4. **编程进入语言** - 用工具实现目标，不受工具限制
5. **约定节省脑力** - 管理任意决定，专注重要问题
6. **问题领域术语** - 在最高抽象层次工作
7. **警惕警告信号** - 编程是纯脑力活动，智力警告尤其重要
8. **迭代改进** - 开发活动迭代越多，成果越好
9. **折中主义** - 填充工具箱，选择正确工具
