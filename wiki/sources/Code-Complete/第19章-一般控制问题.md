## 摘要

本章讨论布尔表达式、深度嵌套和复杂度管理。核心观点：布尔表达式应简单易读；深度嵌套应避免；复杂度是衡量代码质量的关键指标。

## 布尔表达式

### 使用 true 和 false

**原则**：
- 使用 true/false 而非 1/0
- 隐式比较布尔变量
- 显式比较数值

```cpp
// 好
if (statusOK) { ... }
if (!error) { ... }
while (balance != 0) { ... }

// 坏
if (statusOK == true) { ... }
while (balance) { ... }  // 对数值不清晰
```

### 简化复杂表达式

**方法**：
1. 分解为中间变量
2. 移入布尔函数
3. 使用决策表

```vb
' 复杂测试
If (document.AtEndOfStream And Not inputError) And _
   (MIN_LINES <= lineCount And lineCount <= MAX_LINES) And _
   Not ErrorProcessing() Then ...

' 简化后
If DocumentIsValid(document, lineCount, inputError) Then ...
```

### 德摩根定律

| 初始 | 等价 |
|------|------|
| not A and not B | not (A or B) |
| not (A and B) | not A or not B |

### 括号使用

- 为逻辑表达式完全加括号
- 不依赖求值顺序
- 括号很便宜

### 短路求值

```java
// 安全：先检查分母
if (denominator != 0 && (item / denominator) > MIN_VALUE) { ... }

// 不安全：顺序错误
if ((item / denominator) > MIN_VALUE && denominator != 0) { ... }
```

## 复合语句和空语句

**复合语句**：成对写大括号
```cpp
for (i = 0; i < maxLines; i++) { }  // 先写框架
```

**空语句**：使空语句显而易见
```cpp
while (recordArray.Read(index++) != EmptyRecord())
   ;  // 单独一行的空语句

// 或使用 DoNothing()
#define DoNothing()
while (...) { DoNothing(); }
```

## 驯服深度嵌套

### 嵌套问题

研究表明很少有人能理解超过三层嵌套。深度嵌套违背"管理复杂度"原则。

### 减少嵌套技术

| 技术 | 说明 |
|------|------|
| 重新测试部分条件 | 展开嵌套 |
| 转换为 if-then-else | 将灌木变为平铺 |
| 转换为 case | 整数测试 |
| 分解为子例程 | 功能提取 |
| 面向对象分派 | 多态替代 |

**示例 - 嵌套 if 转为 if-then-else**：
```java
// 坏
if (10 < quantity) {
    if (100 < quantity) {
        if (1000 < quantity) {
            discount = 0.10;
        } else { discount = 0.05; }
    } else { discount = 0.025; }
} else { discount = 0.0; }

// 好
if (1000 < quantity) discount = 0.10;
else if (100 < quantity) discount = 0.05;
else if (10 < quantity) discount = 0.025;
else discount = 0.0;
```

## 结构化编程

**核心**：程序只使用一进一出控制构造

**三种构造**：
1. **顺序**：语句按序执行
2. **选择**：if-then-else, case
3. **迭代**：while, for

任何控制流都可用这三种构造创建。

## 复杂度管理

### McCabe 圈复杂度

**计算方法**：
1. 从1开始
2. 每个 if/while/repeat/for/and/or 加1
3. 每个 case 选项加1

**复杂度评级**：
| 决策点 | 建议 |
|--------|------|
| 0-5 | 可能没问题 |
| 6-10 | 开始考虑简化 |
| 10+ | 分解为例程 |

### 复杂度影响

- 复杂度与错误率正相关
- 高复杂度例程需要更多测试
- 分解不能减少总复杂度，但减少同时处理的复杂度
