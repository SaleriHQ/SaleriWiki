## 摘要

本章讨论程序员和技术负责人需要做出的关键构建决策：编程语言选择、编程约定、技术浪潮位置、主要构建实践。研究表明使用高级语言可提高5-15倍生产力，使用熟悉语言可提高30%生产力。本章还区分"在语言中编程"与"编程进入语言"——后者是更成熟的做法。

## 编程语言的选择

**语言对生产力的影响**：
- 熟悉语言3年以上 → 比新手高30%生产力
- 高级语言比低级语言 → 生产力/可靠性提高5-15倍

**高级语言语句与等效C代码比率**：

| 语言 | 相对于C的级别 |
|------|-------------|
| C | 1 |
| C++/Java | 2.5 |
| Fortran 95 | 2 |
| Perl/Python/Smalltalk | 6 |
| Visual Basic | 4.5 |

**Sapir-Whorf假说**：语言表达能力影响思考能力。不熟悉的语言会限制思维。

## 编程约定

**约定的重要性**：
- 架构概念完整性需要低层实现一致
- 没有统一纪律，程序变成风格变体的混乱
- 约定节省大脑处理任意差异的精力

**在构建开始前必须阐明**：
- 变量名、类名、例程名约定
- 格式化约定
- 注释约定

## 技术浪潮位置

| 浪潮阶段 | 特点 |
|---------|------|
| 后期（成熟） | 丰富的工具、文档、调试器、优化器、编译器无bug |
| 早期（原始） | 语言有bug、文档差、工具原始、缺乏参考资料 |

**关键洞察**：无论在哪个阶段，"编程进入语言"而非被语言限制。

## 在语言中编程 vs 编程进入语言

| 在语言中编程 | 编程进入语言 |
|-------------|-------------|
| 想法被语言限制 | 先决定要表达什么，再确定如何用工具表达 |
| 语言工具原始→想法原始 | 用约定弥补语言弱点 |
| 受语言限制 | 控制语言 |

**示例**：Visual Basic早期没有内置MVC支持 → 用.frm文件约定（只允许数据库操作）弥补语言不足。

## 主要构建实践选择

### 编码实践
- [ ] 定义设计在编码前完成多少
- [ ] 定义编码约定（名称、注释、布局）
- [ ] 定义特定编码实践（错误处理、安全、接口约定）
- [ ] 识别技术浪潮位置并调整方法
- [ ] 确定如何"编程进入语言"

### 团队实践
- [ ] 定义集成程序
- [ ] 确定是否结对编程

### 质量保证
- [ ] 编写测试用例的时机（先写还是后写）
- [ ] 是否进行单元测试
- [ ] 是否单步调试代码
- [ ] 是否进行集成测试
- [ ] 是否进行代码审查

### 工具选择
- [ ] 版本控制工具
- [ ] 语言和编译器版本
- [ ] 框架选择（J2EE, .NET等）
- [ ] 编辑器、重构工具、调试器、测试框架

## 关键要点

1. 每种语言都有优缺点，要意识到你正在使用的语言的特定优缺点
2. 在编码前建立约定，之后更改几乎不可能
3. 存在的构建实践比任何单个项目能使用的多，有意识选择
4. 编程进入语言，而非被语言控制
5. 技术浪潮位置决定什么方法有效
