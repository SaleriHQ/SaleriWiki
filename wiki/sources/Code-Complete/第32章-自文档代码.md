## 摘要

本章探讨代码文档的核心原则，强调好的编程风格是最佳文档，注释应补充而非重复代码。核心观点是：糟糕的代码不应该被注释，而应该被重写；注释应该解释代码"为什么"而非"怎么做"；注释的风格应该易于维护。

## 关键概念

### 自文档代码检查清单

**类层面**：
- 接口呈现一致抽象
- 命名描述核心目的
- 接口明显说明使用方式
- 足够抽象，无需了解实现

**例程层面**：
- 名称准确描述功能
- 执行单一定义良好的任务
- 复杂部分已提取为独立例程
- 接口明显清晰

**数据命名**：
- 类型名称描述性
- 变量命名良好
- 循环计数器有信息名称
- 使用枚举类型而非临时标志
- 使用命名常量而非幻数
- 命名约定区分不同类型

**控制结构**：
- 规范路径清晰
- 相关语句分组
- 正常情况遵循 if 而非 else
- 控制结构简单
- 循环功能单一
- 最小化嵌套

### 注释类型与价值

| 类型 | 说明 | 价值 |
|------|------|------|
| 代码重复 | 用不同词重述代码 | 无价值 |
| 代码解释 | 解释复杂代码 | 低价值（应重写代码） |
| 标记 | 工作未完成的提示 | 仅开发期间 |
| 代码总结 | 归纳几行代码为一两句话 | 有价值 |
| 意图描述 | 解释代码目的（问题层次） | 高价值 |
| 代码无法表达的信息 | 版权、版本、设计笔记等 | 必要 |

**有效注释**：总结注释、意图注释、代码无法表达的信息

### 高效注释原则

1. **风格不易破坏** - 避免难以维护的花哨样式（如前导点、下划线加号）
2. **伪代码编程过程** - 先用伪代码设计，注释自然完成
3. **集成到开发中** - 不要留到项目结束
4. **注释数量** - IBM研究：每10语句约1注释是清晰度峰值

### 注释技术

**推荐做法**：
- 在意图级别编写注释（"为什么"而非"怎么做"）
- 在代码之前放置注释
- 用空行分隔注释
- 注释控制结构的结束（长循环、嵌套）
- 文档化数据：单位、允许范围、编码含义
- 文档化全局数据的目的和原因

**避免做法**：
- 行尾注释（难以维护、格式问题）
- 自我放纵的注释
- 重复代码的注释
- 棘手代码的注释（应重写）

### 注释数据声明

- 单位（长度、时间、坐标）
- 允许的数值范围
- 编码含义（使用枚举更好）
- 输入数据限制
- 位级标志含义
- 全局数据的用途

### 注释控制结构

- 在每个 if/case/loop 之前放置注释
- 注释长循环的结束
- 循环结束注释是代码复杂的警告信号

## 关键要点

1. **代码是最好的文档** - 改进代码使其不需要注释
2. **注释应解释代码无法表达的内容** - 意图级别、总结级别
3. **糟糕的注释比没有注释更糟** - 错误的注释误导
4. **风格应易于维护** - 避免繁琐的格式化
5. **伪代码编程过程** - 注释自动完成
6. **棘手的代码应重写** - 而非添加注释
