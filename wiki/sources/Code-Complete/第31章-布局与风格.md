## 摘要

本章探讨程序源代码的视觉布局与风格。核心观点是：良好的布局应该显示程序的逻辑结构，而非仅仅追求美观。布局影响代码的可读性、可维护性和理解难度。研究表明，专家程序员对布局有强烈期望，违反这些期望会显著降低其表现。

## 关键概念

### 格式化基本定理

**良好布局的核心目标**：准确显示程序的逻辑结构。

评估标准：
1. **准确性** - 准确表示代码的逻辑结构
2. **一致性** - 一致地表示代码的逻辑结构
3. **可读性** - 提高代码可读性
4. **可维护性** - 在代码修改下表现良好

### 布局技术

| 技术 | 说明 |
|------|------|
| 空格 | 分组相关语句，分隔不相关语句 |
| 空行 | 8-16% 的空行比例最佳（>16% 调试时间增加） |
| 缩进 | 2-4 空格最佳（6空格缩进看起来漂亮但更不可读） |
| 括号 | 超过两项时使用括号澄清 |

### 布局风格对比

| 风格 | 说明 | 适用语言 |
|------|------|----------|
| 纯块 | 控制构造有自己的终止符 | Visual Basic |
| 模拟纯块 | `{` 在控制语句末尾 | Java（标准）、C++ |
| begin-end 块边界 | `{` 在行首 | C++ |
| Endline 布局 | 代码在行中缩进 | 不推荐（难以维护） |

**推荐**：Java 用纯块风格，C++ 用模拟纯块或 begin-end 块边界均可。

### 控制结构布局

**关键规则**：
- 避免双重缩进 begin-end 对
- 单语句块一致格式化
- case 语句使用标准缩进而非 endline
- 复杂表达式每部分单独一行
- 避免 goto（完全避免格式化问题）

### 单语句布局

**最佳实践**：
- 每行一个语句
- 语句长度：80字符限制已过时，偶尔超过可接受
- 使用空格增强可读性（逻辑表达式、数组引用、例程参数）
- 续行按标准缩进量缩进
- 每行一个数据声明

### 注释布局

- 注释与代码缩进相同
- 用空行分隔注释
- 不要让注释掩盖代码结构

### 类/例程布局

**类接口顺序**：
1. 头注释（描述类）
2. 构造函数/析构函数
3. 公共例程
4. 受保护例程
5. 私有例程和数据

**文件组织**：
- 一个类一个文件
- 用空行分隔例程（至少2行）
- 每个类清晰标识

## 关键要点

1. **逻辑结构优先** - 布局的首要目标是显示代码逻辑，而非美观
2. **一致性最重要** - 遵循约定的细节不如一致应用重要
3. **可维护性是关键** - 难以维护的风格不会被维护
4. **空格帮助理解** - 研究显示2-4空格缩进最有效
5. **专家有强烈期望** - 违反布局期望会显著降低专家表现
