## 摘要

变量命名是程序可读性的关键。本章讨论命名原则、好的命名特征、不同类型变量的命名约定。核心观点：好的命名是唯一最重要的可读性因素；名称应描述"什么"而非"如何"；变量名长度应由作用域决定。

## 好的命名特征

**有效命名原则**：
- 完整且描述性（避免缩写）
- 发音可读
- 易于键盘输入
- 唯一性（避免歧义）

**名称长度**：
| 作用域 | 推荐长度 |
|--------|----------|
| 小作用域 | 5-8字符 |
| 大作用域 | 15-20字符 |
| 全局/长生命 | 更长、更描述性 |

**例外**：常见短名可用于传统场景（i用于循环计数）

## 特定类型变量命名

### 循环变量

| 类型 | 示例 |
|------|------|
| 简单循环 | i, j, k |
| 嵌套循环 | recordIndex, customerIndex |
| 复杂循环 | bytesRemaining, linesRemaining |

### 状态变量

- 使用有意义的名称（status、state、condition）
- 避免 flag 作为名称
- 用 isStatusOk 代替 statusFlag

### 临时变量

- 避免用 temp 命名
- 使用描述实际用途的名称

### 布尔变量

- 使用 is、has、can、should 前缀
- 使用肯定形式命名
- 避免否定命名（notFound、notReady）

### 枚举

- 组名前缀：Color_Red、Color_Blue
- 类型前缀：AccountType.Checking

## 命名约定

**驼峰 vs 下划线**：
```
// 驼峰
customerRecord, customerIndex

// 下划线
customer_record, customer_index
```

**选择原则**：
- 与项目风格一致
- 在整个项目中保持一致
- 选择一种并坚持使用

## 应避免的命名

| 避免 | 原因 |
|------|------|
| 误导性名称 | 引起误解 |
| 无意义区分 | record、file、data |
| 数字结尾 | value1、value2 |
| 特殊字符 | 含空格的名称 |
| 语言关键字 | 与语言保留字冲突 |

## 关键要点

1. 命名是提高可读性最重要的手段
2. 名称应描述"什么"而非"如何"
3. 变量名长度应由作用域决定
4. 循环变量可短，内部复杂循环应用描述性名
5. 一致使用命名约定
6. 避免误导性名称
