## 摘要

例程（包括函数和过程）是软件构建的基本单位。本章讨论创建高质量例程的理由、良好的命名、内聚性、长度、参数使用和特殊考虑。核心观点：创建例程最重要的理由是提高程序智力可管理性；功能内聚是最好的内聚类型；例程长度应由功能自然决定而非人为限制。

## 创建例程的有效理由

| 理由 | 说明 |
|------|------|
| 降低复杂性 | 隐藏信息，不必考虑内部工作 |
| 引入中间抽象 | 命名良好的例程是自文档化的 |
| 避免重复代码 | 修改只在一处 |
| 支持子类化 | 覆盖简短良好分解的例程更容易 |
| 隐藏顺序 | 隐藏事件处理顺序 |
| 隐藏指针操作 | 隔离复杂指针操作 |
| 提高可移植性 | 隔离不可移植的能力 |
| 简化布尔测试 | 函数化复杂条件判断 |
| 提高性能 | 优化一处使所有调用受益 |

## 良好的例程命名

**命名原则**：
- 描述例程的所有操作
- 避免无意义动词（HandleCalculation、ProcessInput）
- 不只用数字区分（Part1、Part2）
- 使名称根据需要尽可能长
- 函数根据返回值命名（cos()、GetName()）
- 过程使用强动词+对象（PrintDocument()）
- 精确使用反义词（add/remove, show/hide）

## 内聚性

| 类型 | 说明 | 评价 |
|------|------|------|
| 功能内聚 | 做一件事且做好 | 最好 |
| 顺序内聚 | 按特定顺序执行，共享数据 | 可接受 |
| 通信内聚 | 操作使用相同数据 | 可接受 |
| 时间内聚 | 同时执行的操作组合 | 可接受 |
| 过程内聚 | 按指定顺序执行 | 通常不可接受 |
| 逻辑内聚 | 通过标志选择操作 | 通常不可接受 |
| 偶然内聚 | 操作无明显关系 | 不可接受 |

## 例程长度

**研究结论**：
- 例程长度与错误率关系不明确
- 50-150行是常见建议
- IBM限制50行，TRW限制两页
- 超过200行需谨慎
- 超过500行错误率上升

**建议**：让内聚性、嵌套深度、变量数量等决定长度，而非人为限制。

## 参数使用

**参数顺序**：输入-修改-输出

**参数原则**：
- 使用所有参数
- 状态/错误变量放在最后
- 不将参数用作工作变量
- 记录接口假设
- 限制参数数量（约7个）
- 确保实参与形参匹配

## 函数 vs 过程

- 函数：主要目的是返回值
- 过程：执行操作不返回值
- 不要用函数做过程的工作（返回状态值）

## 宏和内联

**宏注意事项**：
- 完全括号化表达式
- 用花括号包围多语句宏
- 谨慎使用，优先选择例程

**内联注意事项**：
- 违反封装（代码在头文件）
- 增加代码大小
- 谨慎使用

## 关键要点

1. 创建例程最重要的是提高智力可管理性
2. 简单操作也值得放入例程
3. 功能内聚是最好的内聚类型
4. 名称是质量的指示
5. 函数应只在主要目的是返回值时使用
6. 细心的程序员谨慎使用宏
