## 摘要

防御式编程使错误更易发现、更易修复、对生产代码损害更小。本章讨论保护程序免受无效输入、断言、错误处理技术、异常和调试辅助。核心观点："垃圾进，垃圾出"不够；好的程序使用"垃圾进，什么都不出"或"垃圾进，错误消息出"；断言用于绝不应该发生的情况，错误处理用于非正常但预期的情况。

## 保护程序免受无效输入

**三种处理垃圾的方法**：
1. 检查来自外部源的所有数据值
2. 检查所有例程输入参数值
3. 决定如何处理坏输入

## 断言

**断言定义**：运行时检查自身，发现代码中意外错误

**断言使用**：
- 检查前置条件和后置条件
- 记录假设
- 发现不匹配的接口

**断言vs错误处理**：
| 断言 | 错误处理 |
|------|---------|
| 绝不应该发生的情况 | 非正常但预期的情况 |
| 需要更改源代码 | 生产代码处理 |

**指导原则**：
- 不在断言中放置可执行代码
- 使用断言记录前置条件和后置条件
- 高健壮代码可同时使用断言和错误处理

## 错误处理技术

| 技术 | 适用场景 |
|------|---------|
| 返回中立值 | 继续运行更重要 |
| 替换下一条有效数据 | 数据流处理 |
| 返回与上次相同的答案 | 短时间内值不变 |
| 替换最接近的合法值 | 校准仪器读数 |
| 记录警告到文件 | 需要审计跟踪 |
| 返回错误代码 | 信任调用层次 |
| 调用错误处理例程 | 集中处理 |
| 显示错误消息 | 本地处理 |
| 关闭程序 | 安全关键应用 |

**健壮性vs正确性**：
- 安全关键→正确性（不返回错误结果）
- 消费者应用→健壮性（任何结果比关闭好）

## 异常

**异常使用原则**：
- 只为真正例外的情况抛出
- 不用于正常处理
- 不使用异常推卸责任
- 避免在构造函数/析构函数中抛出
- 在一致的抽象层次抛出
- 包含导致异常的所有信息
- 避免空的catch块
- 考虑异常的替代方案

## 栅栏隔离

**概念**：定义"安全"区域边界，清理跨越边界的数据

**应用**：
- 公共方法检查数据有效性
- 私有方法假设数据已清理
- 栅栏外用错误处理，栅栏内用断言

## 调试辅助

**进攻式编程**：
- 断言中止程序（不让程序员绕过）
- 完全填充分配的内存
- 每个case语句的default失败严重
- 删除对象前用垃圾填充

**计划移除**：
- 使用版本控制工具
- 使用预处理器开关
- 使用调试存根

## 决定生产代码保留多少

**保留**：
- 检查重要错误
- 允许优雅崩溃
- 记录错误给技术支持

**移除**：
- 检查无关紧要错误
- 导致硬崩溃的代码
- 不友好的错误消息

## 关键要点

1. "垃圾进，垃圾出"不够——好的程序更复杂
2. 断言用于绝不应该发生的情况
3. 错误处理是架构/高层设计决策
4. 异常增加复杂性，谨慎使用
5. 开发版本约束不适用于生产版本
6. 防御式编程要平衡——过多也会造成问题
