## 摘要

本章涵盖了软件构建阶段的管理实践，涵盖六个核心主题：鼓励良好编码、配置管理、估算构建进度、度量、将程序员当人看、以及管理你的经理。这些主题共同构成了构建阶段有效管理的基础框架。

## 关键概念

### 鼓励良好编码

**核心原则**：从管理职位强制执行严格技术标准通常不是好主意——程序员需要认同标准。

**有效技术**：
- 为项目各部分指派双人合作（结对编程、师徒、伙伴审查）
- 审查每一行代码（至少三人阅读）
- 要求代码签核（资深技术人员签字）
- 路由良好代码示例以供审查
- 强调代码是公共资产而非私有财产
- 奖励良好代码

**关键洞察**："我必须能够阅读和理解项目编写的任何代码"——这是一种简单有效的编码质量标准。

### 配置管理

**定义**：识别项目工件并系统处理变更的实践，确保系统随时间保持完整性。

**核心要素**：

| 要素 | 说明 |
|------|------|
| 需求和设计变更控制 | 系统化变更控制程序、成批处理变更请求 |
| 源代码变更 | 版本控制软件（检查进出、历史回溯） |
| 工具版本 | 将编译器、链接器等纳入版本控制 |
| 机器配置 | 标准化开发工作站磁盘映像 |
| 备份计划 | 定期备份、异地存储、测试恢复 |

**变更控制委员会 (CCB)**：将变更请求中的精华与糟粕分开，评审、批准或否决变更请求。

### 估算构建进度

**估算方法**：
- 使用估算软件
- 算法方法（如 Cocomo II）
- 外部专家估算
- 各部分相加估算
- 自我任务估算
- 参考以往项目经验
- 保留历史数据迭代校准

**估算流程**：
1. 确定目标（为何估算、估算什么、精确度要求）
2. 留出时间计划估算
3. 明确软件需求
4. 在低细节级别估算
5. 使用多种技术比较结果
6. 定期重新估算

**影响进度的关键因素**：

| 因素 | 有帮助影响 | 有害影响 |
|------|------------|----------|
| 程序员能力 | −24% | +34% |
| 需求分析师能力 | −29% | +42% |
| 人员连续性 | −19% | +29% |
| 产品复杂性 | −27% | +74% |
| 前沿应用 | −11% | +46% |

### 度量

**度量理由**：
- 度量优于不度量
- 量化是科学评估的前提
- 警惕度量副作用（人们关注被度量的工作）

**有用度量类别**：

| 类别 | 指标示例 |
|------|----------|
| 规模 | 总代码行数、注释行数、类/例程数 |
| 缺陷跟踪 | 严重程度、位置、来源、修复时间 |
| 生产力 | 总工作时间、每行成本、每缺陷成本 |
| 质量 | 缺陷总数、每KLOC缺陷数 |
| 可维护性 | 圈复杂度、参数数、代码行数 |

### 将程序员当人看

**程序员时间分配**：
- 代码编写：35%
- 商务/个人：29%
- 会议/培训：13%
- 技术手册阅读：6%
- 其他：17%

**个人差异**：
- 最佳与最差程序员：编码时间比 20:1，调试时间比 25:1
- 生产力可相差数量级
- 程序员经验数量与质量/生产力无直接关系

**工作环境影响**：

| 环境因素 | 前25% | 后25% |
|----------|-------|-------|
| 专用楼层空间 | 78平方英尺 | 46平方英尺 |
| 可接受安静空间 | 57% | 29% |
| 可接受私密空间 | 62% | 19% |
| 频繁中断 | 38% | 76% |

**结论**：前25%程序员生产力是后25%的2.6倍。改善工作环境可实现约100%的生产力提升。

### 管理你的经理

**策略**：
1. 种下想法，等待经理"发现"是你的想法
2. 持续教育经理正确方式
3. 关注经理兴趣，封装实现细节
4. 拒绝错误指令，坚持正确方式
5. 另找工作（最后手段）

## 关键要点

1. **编码标准应该被认同而非强制**——让受尊敬的架构师定义标准
2. **配置管理使程序员工作更轻松**——包括变更控制、版本控制、备份
3. **估算是重大挑战**——使用多种方法、迭代收紧估算
4. **度量优于不度量**——选择有意义的度量，避免过度复杂
5. **程序员是人有差异**——物理环境和个人能力都影响生产力
6. **管理你的经理**——教育是长期解决方案
