## 摘要

本章讨论协作构建实践：结对编程、正式检查和其他评审技术。核心观点：协作实践比测试更有效发现缺陷，且成本更低；检查和结对编程是两种主要方法。

## 协作构建技术对比

| 属性 | 结对编程 | 正式检查 | 非正式评审 |
|------|---------|---------|-----------|
| 缺陷检测率 | 40-60% | 45-70% | 20-40% |
| 明确角色 | 是 | 是 | 否 |
| 后续修复 | 是 | 是 | 否 |
| 适用于设计/需求 | 可能 | 是 | 是 |

## 结对编程

**关键成功因素**：
- 使用编码标准支持
- 双方积极参与，非被动观看
- 选择性使用（复杂代码）
- 定期轮换配对
- 匹配步伐和性格
- 确保能看到显示器

**好处**：
- 压力下表现更好
- 代码质量提升到最佳程序员水平
- 缩短进度
- 传播知识和企业文化

## 正式检查

**检查流程**：
1. 规划：分发材料，确定参与者
2. 概述：作者描述技术环境（可选）
3. 准备：评审员独立审查，发现问题
4. 检查会议：逐行解释代码，记录错误
5. 检查报告：记录缺陷和分配
6. 修复：作者修复缺陷
7. 跟进：主持人确保完成

**角色**：
- 主持人：保持进度，不参与评审
- 作者：解释不清晰部分
- 评审员：发现缺陷
- 记录员：记录错误
- 管理层：不参加

**效率数据**：
- IBM：每小时检查防止100小时相关工作
- Raytheon：缺陷修复从40%降至20%
- 惠普：每年节省2150万美元

## 其他协作技术

### 走查
- 定义宽松，可灵活适应
- 通常比检查效率低
- 适用于大型评审组

### 代码阅读
- 评审员独立阅读代码
- 90%缺陷在准备阶段发现
- 会议专注于讨论问题

## 关键要点

1. 协作实践比测试更有效发现缺陷
2. 检查和评审发现测试未发现的错误类型
3. 正式检查使用检查表、准备、明确角色
4. 结对编程成本与检查相当
5. 走查和代码阅读是检查的替代方案
