## 摘要

本章讨论软件设计在构建阶段的作用。设计是邪恶问题——只有通过解决它才能清楚定义。设计是松散的、启发式的过程，涉及权衡和优先级。核心观点：软件的主要技术要务是管理复杂性。设计应该最小化复杂性、保持松散耦合、提供高扇入低扇出。

## 设计挑战

**设计是"邪恶问题"**：
- 只有通过解决它才能清楚定义问题
- 必须先"解决"问题才能清楚定义它

**设计的特点**：
- 松散的过程（即使产生整洁结果）
- 关于权衡和优先级
- 涉及限制
- 非确定性的（同一问题有多种可接受设计）
- 启发式的（经验法则而非保证过程）
- 涌现的（通过迭代改进演化）

## 关键设计概念

### 软件的主要技术要务：管理复杂性

**Brooks的观点**：
- 本质困难：复杂现实世界的接口、精确识别依赖关系
- 偶然困难：已被解决（语言语法、集成工具等）

**管理复杂性的双管齐下**：
1. 最小化任何人必须处理的基本复杂性数量
2. 防止偶然复杂性不必要扩散

### 设计的理想特征

| 特征 | 说明 |
|------|------|
| 最小复杂性 | 避免"聪明"设计，做出"简单"设计 |
| 易于维护 | 为维护程序员设计 |
| 松散耦合 | 最小化部分间连接 |
| 可扩展性 | 可在不损害结构的情况下增强 |
| 可重用性 | 可在其他系统中复用 |
| 高扇入 | 大量类使用给定类 |
| 低到中扇出 | 给定类使用少量其他类 |
| 可移植性 | 可移动到其他环境 |
| 精简 | 无多余部分 |
| 分层 | 分解层次保持分层 |
| 标准技术 | 使用常见标准化方法 |

### 设计层次

| 层次 | 说明 |
|------|------|
| 第1层 | 软件系统整体 |
| 第2层 | 子系统/包划分 |
| 第3层 | 分为类 |
| 第4层 | 分为例程 |
| 第5层 | 内部例程设计 |

## 设计启发式方法

### 寻找现实世界对象
1. 识别对象及其属性
2. 确定可对每个对象做什么
3. 确定对象间允许的交互
4. 确定哪些部分对其他对象可见
5. 定义每个对象的公共接口

### 形成一致的抽象
- 抽象允许在高级细节级别查看对象
- 好的程序员在例程、类、包级别创建抽象

### 封装实现细节
- 封装补充抽象："不可以在其他细节级别查看对象"
- 封装通过禁止查看复杂性帮助管理复杂性

### 信息隐藏
**秘密的两类**：
1. 隐藏复杂性（大脑不需要处理）
2. 隐藏变更源（变更效果本地化）

**信息隐藏示例**：使用NewId()例程隐藏ID生成实现细节

## 关键要点

1. 设计是非确定性的——同一问题有多种可接受解决方案
2. 软件的主要技术要务是管理复杂性
3. 好的设计通过抽象和封装隐藏复杂性
4. 分层设计：在任何单一级别查看系统获得一致视图
5. 最小化子系统间通信限制，避免熵增加
