# 零日漏洞 (Zero-Day Vulnerability)

## 定义

**零日漏洞**（Zero-Day Vulnerability）是指：
- **尚未被软件开发者和安全社区知晓**的漏洞
- 攻击者可能已经在使用，但**没有可用的修复补丁**
- "零日"指漏洞被发现并被利用时，开发者有**零天时间**来修复

> "Zero-day" 中的 "zero" 指的是软件开发者有 0 天的时间来修复——漏洞被公开时完全没有防御手段。

## 为什么零日如此危险

| 维度 | 传统漏洞 | 零日漏洞 |
|------|---------|---------|
| **知晓状态** | 已知，有补丁 | 未知，无补丁 |
| **利用窗口** | 数天至数周（厂商可响应） | 数分钟（攻击已发生） |
| **发现难度** | 人工或自动化工具可发现 | 需要极高专业水平 |
| **历史案例** | 数十万个 CVE 记录在案 | 多数被国家级攻击者保密使用 |

## 历史案例（Project Glasswing 发现）

Claude Mythos Preview 在数周内发现了数千个零日漏洞，其中包括：

### OpenBSD 27 年漏洞
- **影响**：OpenBSD 是全球最安全的操作系统之一，常用于防火墙
- **漏洞类型**：远程拒绝服务
- **危害**：攻击者只需连接到运行该系统的机器即可使其崩溃
- **存活时间**：27 年未被任何人发现

### FFmpeg 16 年漏洞
- **影响**：FFmpeg 用于无数软件进行视频编解码
- **漏洞类型**：代码处理缺陷
- **关键**：自动化安全测试工具触发了 **500 万次**都没发现该问题
- **存活时间**：16 年

### Linux 内核漏洞链
- **影响**：全球大多数服务器运行 Linux
- **漏洞类型**：权限提升（从普通用户到完全控制）
- **利用方式**：AI 自主发现并**链式组合**多个漏洞完成利用
- **影响**：可从普通用户权限提升为 root

## 零日漏洞生命周期

```
发现 → 储备（stashing）→ 利用 → 披露 → 修复
 ↑                           ↑
←←←←←←← 漏洞被利用窗口 ←←←←←←←←←
         （此时厂商零天时间）
```

- **储备**：攻击者（或 AI）发现漏洞后不披露，保留利用能力
- **披露**：公开或通知厂商
- **修复**：厂商发布补丁

## AI 如何改变零日发现

传统方法：
- 人工代码审计（成本高、速度慢）
- 模糊测试（自动化但覆盖率有限）
- 人工渗透测试（依赖专家经验）

AI 方法：
- **自主分析代码**：Claude 级别的模型可以完整理解大型代码库
- **跨系统关联**：发现漏洞之间的链式利用关系
- **无限并行**：AI 可以同时扫描数百个代码库
- **突破性发现**：AI 能发现人工审查数十年都未发现的漏洞

```python
# AI 发现漏洞的能力增长（示意）
2023年: 能发现中等难度漏洞
2024年: 能发现高级漏洞
2025年(Mythos): 能发现所有主要系统中的数千个漏洞
              包括存活数十年的深层漏洞
```

## 零日披露与负责任披露

| 模式 | 说明 |
|------|------|
| **负责任披露** | 发现后通知厂商，公开补丁后公布细节 |
| **协调披露** | 多方协调，在修复就绪后同时公布 |
| **AI 时代的挑战** | AI 可以在几分钟内完成发现和利用，人工披露流程来不及响应 |

Project Glasswing 采取的策略：
- 向所有相关厂商报告发现的漏洞
- 提供漏洞细节的加密哈希（验证存在但不公开）
- 补丁就绪后公布完整细节

## 防御策略

1. **AI 驱动的主动扫描**：在攻击者之前发现自身系统中的零日
2. **纵深防御**：即使一个漏洞被利用，也有其他层阻止攻击
3. **最小权限原则**：限制单个漏洞造成的损害范围
4. **快速补丁流程**：缩短从披露到修复的响应时间
5. **开源安全投入**：支持开源维护者获得 AI 安全工具（Project Glasswing $400 万捐赠）
