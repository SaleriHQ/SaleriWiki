## 设计工具

当前的设计工具主要由创建设计图的图形工具组成。设计工具有时嵌入在功能更广泛的计算机辅助软件工程（CASE）工具中；一些供应商将独立设计工具广告为 CASE 工具。图形设计工具通常允许你用通用图形表示法表达设计：UML、架构框图、层次结构图、实体关系图或类图。一些图形设计工具只支持一种表示法。其他支持多种。

## 源代码工具

用于处理源代码的工具比用于处理设计的工具更丰富、更成熟。

### 编辑

这组工具与编辑源代码相关。

#### 集成开发环境（IDE）

一些程序员估计他们花费多达40%的时间编辑源代码（Parikh 1986, Ratliff 1987）。如果是这样，花几美元购买尽可能好的 IDE 是一笔好投资。

除了基本的文字处理功能外，好的 IDE 还提供这些功能：

- 在编辑器内编译和错误检测
- 与源代码控制、构建、测试和调试工具集成
- 程序的压缩或大纲视图（仅类名或逻辑结构，无内容，也称为"折叠"）
- 跳转到类、例程和变量的定义
- 跳转到类、例程或变量被使用的所有位置
- 特定语言的格式化
- 正在编辑的语言的交互式帮助
- 大括号（开始-结束）匹配
- 常见语言结构的模板（例如，程序员输入 for 后编辑器完成 for 循环的结构）
- 智能缩进（包括在逻辑变化时轻松更改语句块的缩进）
- 自动化代码转换或重构
- 可用熟悉编程语言编程的宏
- 搜索字符串列表，以便常用字符串不需要重新输入
- 搜索和替换中的正则表达式
- 跨一组文件搜索和替换
- 同时编辑多个文件
- 并排差异比较
- 多级撤销

考虑到仍在使用的一些原始编辑器，你可能会惊讶地发现几个编辑器包含所有这些功能。

### 多文件字符串搜索和替换

如果你的编辑器不支持跨多个文件搜索和替换，你仍然可以找到补充工具来完成这项工作。这些工具对于搜索类名或例程名的所有出现很有用。当你在代码中发现错误时，你可以使用此类工具检查其他文件中类似的错误。

你可以搜索确切字符串、相似字符串（忽略大小写差异）或正则表达式。正则表达式特别强大，因为它们允许你搜索复杂字符串模式。如果你想要找到所有包含幻数的数组引用（数字"0"到"9"），你可以搜索"["，后跟零个或多个空格，后跟一个或多个数字，后跟零个或多个空格，后跟"]"。一个广泛可用的搜索工具叫做"grep"。幻数的 grep 查询如下所示：

```bash
grep "\[ *[0-9]+ *\]" *.cpp
```

你可以使搜索条件更复杂以微调搜索。

能够在多个文件中更改字符串通常很有帮助。例如，如果你想给例程、常量或全局变量一个更好的名称，你可能必须在几个文件中更改该名称。允许跨多个文件进行字符串更改的工具使这变得容易，这很好，因为你应该尽可能少地阻碍创建优秀的类名、例程名和常量名。处理多文件字符串更改的常用工具包括 Perl、AWK 和 sed。

### 差异工具

程序员经常需要比较两个文件。如果你做了几次尝试来纠正错误，需要删除不成功的尝试，文件比较器将对原始文件和修改后的文件进行比较，并列出你更改的行。如果你与其他人在一个程序上工作，想要查看他们自上次处理代码以来所做的更改，比较器工具（如 Diff）将对当前版本与你处理的代码的最后版本进行比较并显示差异。如果你发现一个你不记得在旧版本程序中遇到的新缺陷，而不是去看神经科医生关于健忘症，你可以使用比较器来比较当前和旧版本的源代码，准确确定发生了什么变化，并找到问题 source。这个功能 often 内置于修订控制工具中。

### 合并工具

一种风格的修订控制锁定源文件，以便一次只有一个人可以修改文件。另一种风格允许多人同时处理文件，并在检查时处理合并更改。在这种工作模式下，合并更改的工具至关重要。这些工具通常自动执行简单合并，并在与其他合并冲突或更复杂的合并时查询用户。

### 源代码美化器

源代码美化器美化你的源代码，使其看起来一致。它们突出类名和例程名，标准化你的缩进风格，一致格式化注释，并执行其他类似功能。一些美化器可以将每个例程放在单独的网页或打印页上，或执行更醒目的格式化。许多美化器让你自定义代码被美化的方式。

至少有两类源代码美化器。一类以源代码作为输入，在不更改原始源代码的情况下产生看起来好得多的输出。另一类工具更改源代码本身——标准化缩进、参数列表格式等。在处理大量遗留代码时，此功能很有用。该工具可以完成将遗留代码 conform 到你的编码风格约定所需的大量繁琐格式工作。

### 接口文档工具

一些工具从源代码文件中提取详细的程序员接口文档。源代码文件内的代码使用 @tag 字段等线索来识别应提取的文本。然后接口文档工具提取该标记文本并以良好格式化 presentation。Javadoc 是此类工具的一个突出示例。

### 模板

模板帮助你利用简化你经常做且想一致地做的键盘输入任务的简单想法。假设你希望在例程开始处有一个标准注释序言。你可以使用正确语法构建一个骨架序言，其中包含你希望在标准序言中包含的所有项目的位置。这个骨架将是一个"模板"，你将其存储在文件或键盘宏中。当你创建一个新例程时，你可以轻松地将模板插入到源文件中。你可以将模板技术用于设置更大的实体，如类和文件，或更小的实体，如循环。

如果你正在一个 group 项目上工作，模板是鼓励一致编码和文档风格的一种简单方法。在项目开始时将模板提供给整个团队，团队会使用它们，因为它们使工作更容易——你将一致性作为附带好处获得。

### 交叉引用工具

交叉引用工具列出变量和例程以及它们被使用的所有位置——通常在网页上。

### 类层次结构生成器

类层次结构生成器产生关于继承树的信息。这有时在调试中有用，但更常用于分析程序结构或将程序模块化为包或子系统。此功能 also 在一些 IDE 中可用。

## 分析代码质量

此类别中的工具检查静态源代码以评估其质量。

### 挑剔的语法和语义检查器

语法和语义检查器通过比编译器通常更彻底地检查代码来补充你的编译器。你的编译器可能只检查基本的语法错误。挑剔的语法检查器可能使用语言的细微差别来检查更 subtle 的错误——从编译器角度来看没有错误但你可能不打算写的东西。例如，在 C++ 中，语句

```cpp
while ( i = 0 ) ...
```

是一个完全合法的语句，但它通常应该是

```cpp
while ( i == 0 ) ...
```

第一行在语法上正确，但切换 = 和 == 是一个常见错误，该行可能是错误的。Lint 是一个挑剔的语法和语义检查器，你可以在许多 C/C++ 环境中找到。Lint 警告你有关未初始化的变量、完全未使用的变量、被赋值但从未使用的变量、在例程中被传出但未赋值的例程参数、可疑指针操作、可疑逻辑比较（如刚刚显示的示例中的比较）、不可达代码以及许多其他常见问题。其他语言提供类似工具。

### 指标报告器

一些工具分析你的代码并报告其质量。例如，你可以获取报告每个例程复杂度的工具，以便你可以将最复杂的例程作为额外审查、测试或重新设计的目标。一些工具计算整个程序或单个例程中的代码行数、数据声明、注释和空行。它们跟踪缺陷并将缺陷与制作它们的程序员、更正缺陷的更改和进行更正的程序员关联。他们计算软件的修改次数，并注意被修改最多的例程。发现复杂度分析工具对维护生产力有约20%的积极影响（Jones 2000）。

## 重构源代码

一些工具帮助将源代码从一种格式转换为另一种格式。

### 重构器

重构程序支持常见的代码重构，可以独立使用或集成到 IDE 中。重构浏览器允许你轻松地跨整个代码库更改类名。它们允许你 simply 通过突出显示你想要转换为新例程的代码、输入新例程的名称以及在参数列表中 ordering 参数来提取例程。重构器使代码更改更快且更不易出错。它们可用于 Java 和 Smalltalk，并且正在为其他语言提供。更多关于重构工具的信息，请参见《重构》（Fowler 1999）第14章。

### 重组器

重组器将一盘带有 goto 的 spaghetti 代码转换为更 nutritious 的不含 goto 的更好结构代码的主菜。Capers Jones 报告，在维护环境中，代码重组工具对维护生产力有25-30%的积极影响（Jones 2000）。重组器在转换代码时必须做很多假设，如果原始代码中的逻辑很糟糕，转换后的版本仍然会很糟糕。然而，如果你手动进行转换，你可以将重组器用于一般情况，并手动调整困难的情况。或者，你可以运行代码通过重组器，并用它作为手动转换的灵感。

### 代码翻译器

一些工具将代码从一种语言翻译成另一种语言。当你有一个要将之转移到另一个环境的大量代码库时，翻译器很有用。使用语言翻译器的危险是，如果你的起始代码不好，翻译器 simply 将坏代码翻译成一种不熟悉的语言。

## 数据字典

数据字典是描述项目中所有重要数据的数据库。在许多情况下，数据字典主要关注数据库模式。在大型项目上，数据字典 also 对于跟踪数百或数千个类定义很有用。在大型团队项目上，它对于避免命名冲突很有用。冲突可能是直接的、句法的冲突，其中同一名称被使用两次，或者可能是更微妙的冲突（或差距），其中不同名称被用来表示同一事物或相同名称被用来表示微妙不同的事物。对于每个数据项（数据库表或类），数据字典包含该项的名称和描述。字典还可能包含关于该项使用方式的注释。

## 可执行代码工具

用于处理可执行代码的工具与用于处理源代码的工具一样丰富。

### 代码创建

本节描述的工具帮助代码创建。

#### 编译器和链接器

编译器将源代码转换为可执行代码。大多数程序被编写为可编译的，尽管有些仍然被解释。

标准链接器将编译器从源文件生成的一个或多个对象文件与使可执行程序所需的 standard 代码链接起来。链接器通常可以链接来自多种语言的文件，允许你为程序的每个部分选择最合适的语言，而无需自己处理集成细节。

覆盖链接器通过开发在少于它们消耗的总空间量的内存中执行的程序来帮助你"在五磅袋子里放十磅"。覆盖链接器创建一个可执行文件，在任何时候只将自身的一部分加载到内存中，将其 rest 留在磁盘上直到需要。

#### 构建工具

构建工具的目的是最小化使用程序源文件当前版本构建程序所需的时间。对于项目中的每个目标文件，你指定目标文件所依赖的源文件以及如何创建它。构建工具 also 消除与源处于不一致状态相关的错误；构建工具确保它们都被带到一致状态。常见构建工具包括与 UNIX 关联的 make 实用程序和用于 Java 程序的 ant 工具。

假设你有一个名为 userface.obj 的目标文件。在 make 文件中，你指出为了制作 userface.obj，你必须编译文件 userface.cpp。你还指出 userface.cpp 依赖于 userface.h、stdlib.h 和 project.h。"依赖于"的概念 simply 意味着如果 userface.h、stdlib.h 或 project.h 更改，userface.cpp 需要重新编译。

当你构建程序时，make 工具检查你描述的所有依赖项，并确定需要重新编译的文件。如果你的250个源文件中有五个依赖于 userface.h 中的数据定义，而它更改了，make 自动重新编译依赖它的五个文件。它不会重新编译不依赖 userface.h 的245个文件。使用 make 或 ant 比重新编译所有250个文件或手动重新编译每个文件、忘记一个并获得奇怪的 out-of-synch 错误的替代方案更好。总的来说，像 make 或 ant 这样的构建工具显著改善了平均编译-链接-运行 cycle 的时间和可靠性。

一些组发现了 make 等依赖检查工具的有趣替代方案。例如，Microsoft Word 组发现，只要源文件本身被优化（头文件内容等），简单地重建所有源文件比使用 make 进行 extensive 依赖检查更快。使用这种方法，Word 项目上普通开发者的机器可以在约13分钟内重建整个 Word 可执行文件——数百万行代码。

在短时间内编写高质量代码的一个好方法是不 all 编写它，而是找到一个开源版本或购买它。你至少可以在这些领域找到高质量库：

- 容器类
- 信用卡交易服务（电子商务服务）
- 跨平台开发工具。你可能编写在 Microsoft Windows、Apple Macintosh 和 X Window System 中执行的代码，只需为每个环境重新编译
- 数据压缩工具
- 数据类型和算法
- 数据库操作和数据文件操作工具
- 制图、图形和图表工具
- 图像工具
- 许可证管理器
- 数学运算
- 网络和互联网通信工具
- 报告生成器和报告查询构建器
- 安全和加密工具
- 电子表格和网格工具
- 文本和拼写工具
- 语音、电话和传真工具

#### 代码生成向导

如果你找不到你想要的代码，让人帮你写怎么样？你不必穿上你的黄色格子外套，溜进汽车推销员的花言巧语来哄骗别人为你写代码。你可以找到为你编写代码的工具，此类工具通常集成在 IDE 中。

代码生成工具 tend to 专注于数据库应用程序，但这包括很多应用程序。常用代码生成器为数据库、用户界面和编译器编写代码。它们生成的代码 rarely 像人类程序员生成的代码那样好，但许多应用程序不需要手工精心制作的代码。对于某些用户来说，拥有10个工作应用程序比拥有一个工作异常出色的应用程序更有价值。

代码生成器也 useful for 为生产代码制作原型。使用代码生成器，你可能在几个小时内就能黑出一个演示用户界面关键方面的原型，或者你可能能够尝试各种设计方法。手动编码同样多的功能可能需要几周时间。如果只是在实验，为什么不用最便宜的方式做？

代码生成器的 common 缺点是它们 tend to 生成几乎不可读的代码。如果你曾经必须维护这样的代码，你可能会后悔最初没有手工编写它。

#### 设置和安装

众多供应商提供支持创建设置程序的工具。这些工具通常支持创建磁盘、CD 或 DVD 或通过 Web 安装。它们检查目标安装机器上是否已存在 common 库文件，执行版本检查等。

#### 预处理器

预处理器和预处理器宏函数对调试有用，因为它们使在开发代码和生产代码之间切换变得容易。在开发期间，如果你想在每个例程开始时检查内存碎片，你可以使用宏在每个例程开始时。你可能不想将检查留在生产代码中，所以对于生产代码，你可以重新定义宏，使其根本不生成任何代码。基于类似原因，预处理器宏对于编写针对在多种环境中编译的代码很有用——例如，在 Windows 和 Linux 中。

### 代码调优

这些工具可以帮助你微调代码。

#### 执行分析器

执行分析器在代码运行时观察你的代码，并告诉你每个语句被执行了多少次，或者程序在每个语句或执行路径上花费了多少时间。在代码运行时对其进行性能分析就像让医生将听诊器压在你的胸口并让你咳嗽。它让你深入了解你的程序如何工作、热点在哪里以及你应该在哪里集中代码调优工作。

#### 汇编列表和反汇编器

某天你可能想要查看由高级语言生成的汇编代码。一些高级语言编译器生成汇编列表。其他的不生成，你必须使用反汇编器从编译器生成的机器代码重新创建汇编。查看编译器生成的实际汇编代码向你展示了编译器将高级语言代码翻译成机器代码的效率。它可以告诉你为什么看起来 fast 的高级代码运行缓慢。在第26章中，一些基准测试结果是反直觉的。在对该代码进行基准测试时，我经常参考汇编列表以更好地理解在高级语言中 make 没有 sense 的结果。

如果你对汇编语言不舒服，想要一个介绍，没有比将你写的每个高级语言语句与编译器生成的汇编指令进行比较更好的了。第一次接触汇编通常是天真的丧失。当你看到编译器创建了多少代码——比它需要的多多少——你再也不会以同样的方式看待你的编译器了。

反之，在某些环境中，编译器必须生成极其复杂的代码。研究编译器输出可以培养对在低级语言中编程所需工作量的欣赏。

## 工具导向环境

一些环境已被证明比其他环境更适合工具导向编程。

UNIX 环境因其收集的名称有趣的小工具而闻名，这些工具可以很好地协同工作：grep、diff、sort、make、crypt、tar、lint、ctags、sed、awk、vi 等。与 UNIX 紧密耦合的 C 和 C++ 语言体现了相同的哲学；标准 C++ 库由可以轻松组合成更大函数的小函数组成，因为它们可以很好地协同工作。

一些程序员在 UNIX 中工作效率如此之高，以至于他们随身携带。它们使用 UNIX 工作类似的工具来在 Windows 和其他环境中支持它们的 UNIX 习惯。UNIX paradigm 成功的一个 tribute 是可以在其他机器上放置 UNIX costume 的工具的可用性。例如，cygwin 提供了在 Windows 下工作的 UNIX 等效工具（http://www.cygwin.com）。

## 构建你自己的编程工具

假设你被给予五个小时来做这项工作，你有一个选择：

舒适地在五小时内完成工作，或者花费四小时45分钟疯狂构建一个工具来做这项工作，然后在15分钟内让工具完成工作。

大多数好程序员在百万分之一的情况下会选择第一个选项，在所有其他情况下选择第二个选项。构建工具是编程的一部分。近乎所有大型组织（拥有超过1000名程序员的组织）都有内部工具和支持组。许多组织拥有比市场上更优越的专有需求和设计工具（Jones 2000）。

你可以编写本章描述的许多工具。这样做可能不具有成本效益，但没有任何巨大的技术障碍来做到这一点。

### 项目特定工具

大多数中大型项目需要特定于项目的特殊工具。例如，你可能需要工具来生成特殊类型的测试数据、验证数据文件的质量或模拟尚未可用的硬件。以下是一些项目特定工具支持的示例：

一个航空航天团队负责开发飞行中软件来控制红外传感器并分析其数据。为了验证软件的性能，飞行数据记录器记录了飞行中软件的操作。工程师编写了自定义数据分析工具来分析飞行系统的性能。每次飞行后，他们使用自定义工具检查主要系统。

Microsoft 计划在其 Windows 图形环境的一个版本中包含新的字体技术。由于字体数据文件和处理字体显示的软件都是新的，错误可能来自数据或软件。Microsoft 开发者编写了几个自定义工具来检查数据文件中的错误，这提高了他们区分字体数据错误和软件错误的能力。

一家保险公司开发了一个雄心勃勃的系统来计算其费率增长。因为系统很复杂且准确性至关重要，数百个计算出的费率需要仔细检查，即使手动计算单个费率需要几分钟。该公司编写了一个独立的软件工具来一次计算一个费率。使用该工具，该公司可以在几秒钟内计算单个费率，并以比手动检查主程序费率所需时间的一小部分来检查主程序的费率。

规划项目的一部分应该是考虑可能需要的工具，并分配时间来构建它们。

### 脚本

脚本是自动化重复琐碎任务的工具。在某些系统中，脚本称为批处理文件或宏。脚本可以是简单的或复杂的，一些最有用的脚本是最容易编写的。例如，我保留日记，为了保护我的隐私，我对它进行加密，除了我正在写它的时候。为了确保我总是正确地加密和解密它，我有一个脚本解密我的日记，执行文字处理器，然后加密日记。脚本如下：

```bash
crypto c:\word\journal.* %1 /d /Es /s
word c:\word\journal.doc
crypto c:\word\journal.* %1 /Es /s
```

%1 是我的密码字段，由于显而易见的原因，密码不包含在脚本中。脚本节省了我输入（和打错）所有参数的工作，并确保我总是执行所有操作并以正确顺序执行它们。

如果你发现自己一天多次输入超过约五个字符的内容，那就是脚本或批处理文件的好候选。示例包括编译/链接序列、备份命令以及带有大量参数的任何命令。

## 工具幻想乐园

几十年来，工具供应商和行业权威人士承诺消除编程所需的工具就在眼前。第一个获得此称号的工具，也许是最具讽刺意味的，是 Fortran。Fortran 或"公式翻译语言"的 conception 是为了让科学家和工程师 simply 输入公式，从而 supposedly 消除对程序员的需求。

交叉引用

工具可用性部分取决于技术环境的成熟度。更多关于此的信息，请参见技术浪潮中的你的位置。

Fortran 确实成功地使科学家和工程师能够编写程序，但从我们今天的 vantage 点来看，Fortran 似乎是一种比较 low-level 的编程语言。它 hardly 消除了对程序员的需求，而行业在使用 Fortran 方面的 experience 表明了整个软件行业的 progress。

软件行业不断开发新工具来减少或消除编程最繁琐的方面：源语句布局的细节；编辑、编译、链接和运行程序所需的步骤；查找不匹配大括号所需的工作；创建标准消息框所需的步骤数量；等等。当这些新工具中的每一个开始展示生产力的增量收益时，权威人士将这些收益 extrapolate 到无穷大，假设收益最终将"消除对编程的需求"。但现实中发生的是，每个新的编程创新都带有一些瑕疵。随着时间推移，瑕疵被移除，该创新的全部潜力被实现。然而，一旦基本工具概念被实现，通过剥离作为创建新工具的副作用而创建的偶然困难来实现 further 收益。消除这些偶然困难本身不会增加生产力；它 simply 消除了典型"两步 forward，一步 back"等式中的"一步 back"。

在过去的几十年里，程序员已经看到了许多本应消除编程的工具。首先是第三代语言。然后是第四代语言。然后是自动编程。然后是 CASE 工具。然后是可视化编程。这些进步中的每一个都产生了有价值的、编程的增量改进——总的来说，它们使编程对于在这些进步之前学习编程的任何人来说都变得无法辨认。但这些创新中没有一项成功消除了编程。

这种动态的原因是，从本质上讲，编程是 fundamentally 困难的——即使有良好的工具支持。无论有什么工具可用，程序员都必须与混乱的现实世界搏斗；我们必须 rigourously 思考序列、依赖和异常；我们必须 deal with 无法下定决心的最终用户。我们将 always 必须与定义不明确的与其他软件和硬件的接口搏斗，我们必须考虑法规、业务规则以及其他源于计算机编程世界之外的复杂性来源。

交叉引用

编程困难的原因在关键设计概念中的"偶然和本质困难"中描述。

我们将 always 需要能够弥合要解决的现实世界问题与 supposed to 正在解决该问题的计算机之间差距的人。无论我们是在汇编器中操作机器寄存器还是在 Microsoft Visual Basic 中操作对话框，这些人将被称为程序员。只要我们有计算机，我们就需要告诉计算机该做什么的人，而该活动将被称为编程。

当你听到工具供应商声称"这个新工具将消除计算机编程"时，跑！或者至少对你自己对供应商天真乐观的笑容感到满意。
