## 编程语言的选择

> 通过免除大脑所有不必要的工作，好的符号系统使其能够集中精力处理更高级的问题，实际上提高了种族的思维能力。在引入阿拉伯数字之前，乘法是困难的，整数除法甚至需要运用最高级的数学能力。在现代世界中，很可能没有什么比以下事实更让希腊数学家惊讶：西欧大部分人口可以对最大的数字执行除法运算。这个事实对他来说似乎完全不可能……我们现代对小数进行轻松计算的简便能力几乎是逐渐发现完美符号系统的近乎奇迹般的结果。
> —— Alfred North Whitehead

系统将用哪种编程语言实现应该对你非常感兴趣，因为你将从构建开始到结束都沉浸在其中。

研究表明，编程语言的选择以几种方式影响生产力和代码质量。

**使用熟悉的语言的程序员比使用不熟悉的语言的程序员生产力更高。** 来自 Cocomo II 估算模型的数据显示，使用一种语言三年或更长时间的程序员比具有同等经验但对该语言新手的程序员生产力高约 30%（Boehm et al. 2000）。IBM 早期的一项研究发现，对编程语言有丰富经验的程序员的生产力是最少经验程序员的 3 倍多（Walston and Felix 1977）。（Cocomo II 更谨慎地隔离了个别因素的影响，这解释了两个研究的不同结果。）

**使用高级语言的程序员比使用低级语言的程序员实现更好的生产力和质量。** 像 C++、Java、Smalltalk 和 Visual Basic 这样的语言被认为比汇编和 C 等低级语言在生产力、可靠性、简单性和可理解性方面提高了 5 到 15 倍（Brooks 1987, Jones 1998, Boehm 2000）。当 C 语句做它应该做的事情时，你不需要每次都举行颁奖典礼，从而节省时间。此外，高级语言比低级语言更具表现力。每一行代码表达更多。**表 4-1** 显示了几种高级语言源代码语句与等效 C 代码的典型比率。比率越高意味着列表中语言每一行代码比 C 的每一行代码完成更多工作。

一些语言在表达编程概念方面比其他语言更好。你可以在自然语言（如英语）和编程语言（如 Java 和 C++）之间进行类比。在自然语言的情况下，语言学家 Sapir 和 Whorf 假设了语言表达能力与思考某些思想能力之间的关系。Sapir-Whorf 假说说你思考一个思想的能力取决于知道能够表达那个思想的词汇。如果你不知道这些词，你就不能表达那个思想，你甚至可能无法构思它（Whorf 1956）。

程序员可能同样受到他们语言的影响。编程语言中可用于表达你的编程思想的词汇确实决定了你如何表达你的思想，甚至可能决定你能表达什么思想。

编程语言对程序员思维影响的表现很常见。一个典型的故事是这样的："我们正在用 C++ 编写一个新系统，但我们的大多数程序员在 C++ 方面没有太多经验。他们来自 Fortran 背景。他们写的代码在 C++ 中编译，但他们实际上是在写伪装的 Fortran。他们拉伸 C++ 来模拟 Fortran 的坏特性（如 goto 和全局数据），而忽略了 C++ 丰富的面向对象能力集。"这种现象在整个行业中已经报道了很多年（Hanson 1984, Yourdon 1986a）。

### 语言描述

一些语言的发展历史很有趣，它们的通用能力也很丰富。以下是当今使用最广泛的语言的描述。

#### Ada

Ada 是一种基于 Pascal 的通用高级编程语言。它是在美国国防部的主持下开发的，特别适合实时和嵌入式系统。Ada 强调数据抽象和信息隐藏，并强制你区分每个类和包的公共和私有部分。选择"Ada"作为语言名称是为了纪念 Ada Lovelace，她是一位数学家，被认为是世界上第一位程序员。今天，Ada 主要用于军事、空间和航空电子系统。

#### 汇编语言

汇编语言或"汇编器"是一种低级语言，其中每个语句对应一条机器指令。因为语句使用特定的机器指令，汇编语言特定于特定处理器——例如，特定的 Intel 或 Motorola CPU。汇编器被视为第二代语言。大多数程序员避免使用它，除非他们在执行速度或代码大小方面达到极限。

#### C

C 是一种最初与 UNIX 操作系统关联的通用、中级语言。C 有一些高级语言特性，如结构化数据、结构化控制流、机器独立性和丰富的运算符集。它也被称为"可移植汇编语言"，因为它大量使用指针和地址，具有一些低级构造如位操作，并且是弱类型的。

C 是 1970 年代在贝尔实验室开发的。它最初是为 DEC PDP-11 设计和使用的——其操作系统、C 编译器和 UNIX 应用程序都是用 C 编写的。1988 年发布了 ANSI 标准来编纂 C，并在 1999 年进行了修订。C 是 1980 年代和 1990 年代微型计算机和工作站编程的事实标准。

#### C++

C++ 是一种建立在 C 之上的面向对象语言，于 1980 年代在贝尔实验室开发。除了与 C 兼容外，C++ 还提供了类、多态、异常处理、模板，以及比 C 更强大的类型检查。它还提供了一个广泛而强大的标准库。

#### C#

C# 是一种通用、面向对象的语言和编程环境，由 Microsoft 开发，语法类似于 C、C++ 和 Java，它提供了广泛的工具来帮助在 Microsoft 平台上进行开发。

#### Cobol

Cobol 是一种类似英语的编程语言，最初于 1959-1961 年为国防部开发。Cobol 主要用于业务应用程序，至今仍是最广泛使用的语言之一，人气仅次于 Visual Basic（Feiman and Driver 2002）。Cobol 多年来已更新，包括数学函数和面向对象能力。"Cobol"这个缩写代表 COmmon Business-Oriented Language（通用业务导向语言）。

#### Fortran

Fortran 是第一种高级计算机语言，引入了变量和高级循环的概念。"Fortran"代表 FORmula TRANslation（公式翻译）。Fortran 最初于 1950 年代开发，已经经历了多次重大修订，包括 1977 年的 Fortran 77，它添加了块结构化的 if-then-else 语句和字符串操作。Fortran 90 添加了用户定义的数据类型、指针、类和丰富的数组操作。Fortran 主要用于科学和工程应用程序。

#### Java

Java 是一种面向对象的语言，语法类似于 C 和 C++，由 Sun Microsystems, Inc. 开发。Java 被设计为在任何平台上运行，方法是将 Java 源代码转换为字节码，然后在称为虚拟机的环境中在每个平台上运行。Java 被广泛用于编程 Web 应用程序。

#### JavaScript

JavaScript 是一种解释型脚本语言，与 Java 松散相关。它主要用于客户端编程，如向网页添加简单功能和在线应用程序。

#### Perl

Perl 是一种基于 C 和几种 UNIX 实用程序处理字符串的语言。Perl 常用于系统管理任务，如创建构建脚本，以及报告生成和处理。它也用于创建 Web 应用程序如 Slashdot。"Perl"这个缩写代表 Practical Extraction and Report Language（实用提取和报告语言）。

#### PHP

PHP 是一种开源脚本语言，语法类似于 Perl、Bourne Shell、JavaScript 和 C。PHP 在所有主要操作系统上运行以执行服务器端交互功能。它可以嵌入网页以访问和呈现数据库信息。"PHP"这个缩写最初代表 Personal Home Page，但现在代表 PHP: Hypertext Processor。

#### Python

Python 是一种解释型、交互式、面向对象的语言，运行在众多环境中。它最常用于编写脚本和小型 Web 应用程序，也包含一些用于创建更大程序的支持。

#### SQL

SQL 是用于查询、更新和管理关系数据库的事实标准语言。"SQL"代表 Structured Query Language（结构化查询语言）。与本节列出的其他语言不同，SQL 是一种"声明性语言"，这意味着它不定义操作序列，而是定义某些操作的结果。

#### Visual Basic

原始版本的 Basic 是一种高级语言，于 1960 年代在达特茅斯学院开发。BASIC 这个缩写代表 Beginner's All-purpose Symbolic Instruction Code（初学者通用符号指令代码）。Visual Basic 是 Microsoft 开发的 Basic 的高级、面向对象、可视化编程版本，最初设计用于创建 Microsoft Windows 应用程序。它后来被扩展以支持桌面应用程序（如 Microsoft Office）的定制、Web 程序的创建和其他应用程序。专家报告说，到 2000 年代初，更多的专业开发人员在 Visual Basic 中工作而不是任何其他语言（Feiman and Driver 2002）。

## 编程约定

在高质量软件中，你可以看到架构概念完整性与其在低层实现之间的关系。实现必须与指导它的架构一致，并在内部一致。这就是为变量名、类名、例程名、格式化约定和注释约定制定构建指南的要点。

> **交叉参考**
>
> 有关约定力量的更多详细信息，请参阅"命名约定的力量"到"标准化前缀"。

在复杂程序中，架构指南给程序结构性平衡，构建指南提供低层和谐，将每个类表达为全面设计的忠实部分。任何大型程序都需要一个控制结构来统一其编程语言细节。大型结构的美感之一在于其详细部分实现其架构含义的方式。没有统一的纪律，你的创作将是一个风格草率变体的混乱。这样的变体消耗你的大脑——而只是为了理解本质上武断的编码风格差异。成功编程的关键之一是避免武断的变体，这样你的大脑可以自由地专注于真正需要的变体。更多内容请参阅"关键设计概念"中的"软件的主要技术要务：管理复杂性"。

如果一幅画有一个很棒的设计，但一部分是古典的，一部分是印象派的，一部分是立体派的，那会怎样？无论你多么紧密地遵循其宏伟设计，它都不会有概念完整性。它看起来像一幅拼贴画。程序也需要低层完整性。

在构建开始之前，阐明你将使用的编程约定。编码约定细节处于如此精确的水平，以至于在软件编写之后几乎不可能将其改装进去。本书通篇提供了此类约定的细节。

## 你在技术浪潮中的位置

在我的职业生涯中，我看到 PC 的明星崛起，而大型机的明星向地平线倾斜。我看到 GUI 程序取代了基于字符的程序。我还看到 Web 兴起而 Windows 衰落。我只能假设当你读到这本书时，一些新技术将处于上升期，而我所知道的 Web 编程（2004 年）将逐渐衰落。这些技术周期或浪潮意味着根据你在浪潮中的位置，编程实践会有所不同。

在成熟的技术环境——浪潮的末端，如 2000 年代中期的 Web 编程——我们受益于丰富的软件开发基础设施。后期浪潮环境提供了众多编程语言选择、对用这些语言编写的代码的全面错误检查、强大的调试工具，以及自动、可靠的性能优化。编译器几乎无 bug。工具有完善的文档，包括供应商文献、第三方书籍和文章以及广泛的 Web 资源。工具是集成的，所以你可以在单一环境中做 UI、数据库、报告和业务逻辑。如果你确实遇到问题，你可以很容易地在 FAQ 中找到描述的工具怪癖。还有许多顾问和培训课程可用。

在早期浪潮环境——例如 1990 年代中期的 Web 编程——情况相反。可用的编程语言选择很少，而且这些语言往往有 bug 且文档质量差。程序员花费大量时间仅仅试图弄清楚语言如何工作，而不是编写新代码。程序员还花费无数小时处理语言产品、底层操作系统和其他工具中的 bug。早期浪潮环境中的编程工具往往是原始的。调试器可能根本不存在，编译器优化器仍然是一些程序员眼中的闪光。供应商经常修订他们的编译器版本，看起来每个新版本都会破坏你代码的重要部分。工具不是集成的，所以你倾向于使用不同的工具处理 UI、数据库、报告和业务逻辑。工具往往不太兼容，你可以花费大量精力仅仅是为了在编译器库的猛烈攻击下保持现有功能正常工作。如果你遇到麻烦，参考文献以某种形式存在于 Web 上，但它并不总是可靠的，而且，如果可用文献有任何指导，每次你遇到问题，似乎你是第一个这样做的人。

这些评论可能看起来像是避免早期浪潮编程的建议，但这不是它们的意图。一些最具创新性的应用程序来自早期浪潮程序，如 Turbo Pascal、Lotus 123、Microsoft Word 和 Mosaic 浏览器。关键是，你如何度过你的编程日子将取决于你在技术浪潮中的位置。如果你处于浪潮的后期，你可以计划大部分时间稳定地编写新功能。如果你处于浪潮的早期，你可以假设你会花费大部分时间试图弄清楚你的编程语言未记录的特性，调试实际上是库代码中缺陷的错误，修改代码以便它可以使用供应商库的某个新版本工作，等等。

当你发现自己在原始环境中工作时，要意识到本书中描述的编程实践可以帮助你，甚至比在成熟环境中更有帮助。正如 David Gries 指出的，你的编程工具不需要决定你如何思考编程（1981）。Gries 区分了"在语言中编程"和"编程进入语言"。用语言"编程"的程序员将他们的想法限制在语言直接支持的构造中。如果语言工具是原始的，程序员的想法也将是原始的。

编程"进入"语言的程序员首先决定他们想要表达什么想法，然后他们确定如何使用特定语言提供的工具表达这些想法。

### 编程进入语言的示例

在 Visual Basic 的早期，我很沮丧，因为我想在我正在开发的产品中将业务逻辑、UI 和数据库分开，但语言中没有内置方法可以做到这一点。我知道如果我不小心，随着时间的推移，我的一些 Visual Basic"表单"最终会包含业务逻辑，一些表单会包含数据库代码，一些两者都不包含——我将永远无法记住哪段代码位于哪个位置。我刚刚完成了一个 C++ 项目，那个项目在分离这些问题方面做得很差，我不想在不同的语言中体验那些头痛的 déjà vu。

因此，我采用了一个设计约定，即 .frm 文件（表单文件）只允许从数据库检索数据并将数据存储回数据库。它不允许直接将数据传达给程序的其他部分。每个表单支持一个 IsFormCompleted() 例程，调用例程使用它来确定已激活的表单是否已保存其数据。IsFormCompleted() 是表单允许拥有的唯一公共例程。表单也不允许包含任何业务逻辑。所有其他代码必须包含在关联的 .bas 文件中，包括表单中条目的有效性检查。

Visual Basic 不鼓励这种做法。它鼓励程序员尽可能将代码放入 .frm 文件，并且不让 .frm 文件容易地回调到关联的 .bas 文件。

这个约定相当简单，但随着我深入项目，我发现它帮助我避免了许多如果没有约定我会写混乱代码的情况。我会加载表单但保持它们隐藏，这样我就可以调用其中的数据有效性检查例程，或者我会将代码从表单复制到其他位置，然后在多个位置维护并行代码。IsFormCompleted() 约定也保持事情简单。因为每个表单都以完全相同的方式工作，我永远不必事后猜测 IsFormCompleted() 的语义——它每次使用时的意思都是相同的。

Visual Basic 不直接支持这个约定，但我使用一个简单的编程约定——编程进入语言——弥补了当时语言缺乏结构的不足，并帮助保持项目在智力上可控。

理解"在语言中编程"和"编程进入一种语言"之间的区别对于理解本书至关重要。大多数重要的编程原则不依赖于特定语言，而是依赖于你使用它们的方式。如果你的语言缺少你想要使用的构造或容易出现其他类型的问题，请尝试弥补它们。发明你自己的编码约定、标准、类库和其他增强功能。

## 主要构建实践的选择

为构建做准备的部分是决定你将强调的众多可用良好实践中的哪些。一些项目使用结对编程和测试先行开发，而其他项目使用单独开发和正式检查。任何一种技术组合都可以工作良好，取决于项目的具体情况。

以下检查表总结了你应该在构建期间有意识地决定包括或排除的具体实践。这些实践的详细信息包含在本书通篇中。

cc2e.com/0496

#### 编码

- 你是否定义了将在前面完成多少设计，以及在键盘旁——编写代码时——完成多少设计？
- 你是否为名称、注释和布局定义了编码约定？
- 你是否为架构隐含的特定编码实践进行了定义，例如将如何处理错误条件、如何解决安全问题、类接口将使用什么约定、重用代码将应用什么标准、在编码时考虑多少性能问题等等？
- 你是否识别了你自己在技术浪潮中的位置并调整了你的方法来匹配？
- 如果必要，你是否识别了你将如何编程进入语言而不是被限制在语言中编程？

#### 团队合作

- 你是否定义了集成程序——即，你是否定义了程序员在将代码检入主源码之前必须经过的特定步骤？
- 程序员将结对编程，还是单独编程，或者两者的某种组合？

#### 质量保证

- 程序员是否在编写代码本身之前为其代码编写测试用例？
- 程序员是否为其代码编写单元测试，无论他们是先写还是后写？
- 程序员是否在检入之前通过调试器单步执行他们的代码？
- 程序员是否在检入之前对其代码进行集成测试？
- 程序员是否相互审查或检查彼此的代码？

## 关键点

- 每种编程语言都有优点和缺点。要意识到你正在使用的语言的特定优点和缺点。
- 在开始编程之前建立编程约定。之后将代码更改为匹配它们几乎是不可能的。
- 存在的构建实践比你可以在任何单个项目上使用的要多。有意识地选择最适合你的项目的实践。
- 问问你自己，你正在使用的编程实践是对你正在使用的编程语言的响应，还是被它所控制。
- 记住编程进入语言，而不是在语言中编程。
- 你在技术浪潮中的位置决定了什么方法将有效——甚至是可能的。识别你在技术浪潮中的位置，并相应地调整你的计划和期望。
