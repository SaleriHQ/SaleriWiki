## 选择好名称时需要考虑的事项

你不能像给狗起名字那样给变量起名——因为它听起来可爱或好听。与狗及其名字不同（狗和名字是两个不同的实体），变量及其变量名本质上是同一个东西。因此，一个变量的好坏很大程度上取决于它的名字。要谨慎地选择变量名。

下面是一个使用糟糕变量名的代码示例：

**变量名不佳的 Java 示例**

示例 11-1.

```java
x = x - xx;
xxx = fido + SalesTax( fido );
x = x + LateFee( x1, x ) + xxx;
x = x + Interest( x1, x );
```

这段代码中发生了什么？x1、xx 和 xxx 是什么意思？fido 是什么意思？假设有人告诉你，这段代码是根据未结余额和新购买商品计算客户的总账单。你会使用哪个变量来仅打印新购买商品的客户账单？

下面是同一段代码的另一个版本，使这些问题更容易回答：

示例 11-2. 变量名良好的 Java 示例

```java
balance = balance - lastPayment;
monthlyTotal = newPurchases + SalesTax( newPurchases );
balance = balance + LateFee( customerID, balance ) + monthlyTotal;
balance = balance + Interest( customerID, balance );
```

---

鉴于这两段代码之间的对比，一个好的变量名应该是可读的、易记的，并且是恰当的。你可以使用几个通用的经验法则来实现这些目标。

## 命名中最重要的考虑事项

命名变量时最重要的考虑是，名称应完全准确地描述变量所代表的实体。提出好名字的一个有效技巧是用语言陈述变量代表什么。通常这个陈述本身就是最好的变量名。它很容易阅读，因为它不包含神秘的缩写，而且含义明确。因为它是对实体的完整描述，所以不会与其他东西混淆。而且由于名称与概念相似，所以很容易记住。

对于代表美国奥运代表团队人数的变量，你会创建名为 `numberOfPeopleOnTheUsOlympicTeam` 的名称。代表体育场座位数的变量应该是 `numberOfSeatsInTheStadium`。代表一个国家队在任何现代奥运会中获得的最高分数的变量应该是 `maximumNumberOfPointsInModernOlympics`。包含当前利率的变量最好命名为 `rate` 或 `interestRate`，而不是 `r` 或 `x`。你明白这个意思了。

注意这些名称的两个特点。首先，它们很容易解密。事实上，它们根本不需要解密，因为你可以直接阅读它们。但其次，其中一些名称很长——长到不切实际。我稍后会讨论变量名长度的问题。

表 11-1 显示了几个变量名示例，包括好的和坏的：

**表 11-1. 好变量名和坏变量名示例**

| 变量的用途 | 好的名称，好的描述符 | 坏的名称，差的描述符 |
|-----------|---------------------|---------------------|
| 迄今为止写入的支票累计总额 | runningTotal, checkTotal | written, ct, checks, CHKTTL, x, x1, x2 |
| 子弹列车的速度 | velocity, trainVelocity, velocityInMph | velt, v, tv, x, x1, x2, train |
| 当前日期 | currentDate, todaysDate | cd, current, c, x, x1, x2, date |
| 每页行数 | linesPerPage | lpp, lines, l, x, x1, x2 |

`currentDate` 和 `todaysDate` 是好名称，因为它们完全准确地描述了"当前日期"的概念。事实上，它们使用的是显而易见的词。程序员有时会忽略使用普通词汇，而这往往是最简单的解决方案。因为它们太短且根本没有描述性，`cd` 和 `c` 是糟糕的名称。`current` 是糟糕的，因为它没有告诉你什么是"当前"的。`date` 几乎是个好名字，但最终分析起来它是个糟糕的名字，因为涉及的日期不仅仅是任何日期，而是当前日期；仅 `date` 本身没有给出这样的指示。`x`、`x1` 和 `x2` 是糟糕的名称，因为它们总是糟糕的名称——`x` 传统上代表未知量；如果你不想让你的变量成为未知量，就想个好点的名字。

名称应该尽可能具体。像 `x`、`temp` 和 `i` 这样足够通用、可以用在多个地方的名称不如它们应有的那样信息丰富，而且通常是坏名字。

## 问题导向

一个好的助记名称通常针对问题说话，而不是解决方案。一个好的名字倾向于更多地表达"是什么"而不是"怎么做"。一般来说，如果一个名字指的是计算的某个方面而不是问题本身，那就是"怎么做"而不是"是什么"。避免使用这种名字，而应使用指向问题本身的名字。

员工数据记录可以称为 `inputRec` 或 `employeeData`。`inputRec` 是一个计算机术语，指的是计算思想——输入和记录。`employeeData` 指的是问题域而不是计算世界。同样，对于表示打印机状态的位字段，`bitFlag` 是一个更具计算机味的名称，而不是 `printerReady`。在会计应用程序中，`calcVal` 比 `sum` 更具计算机味。

## 最佳名称长度

名称的最佳长度似乎介于 `x` 和 `maximumNumberOfPointsInModernOlympics` 之间。 太短的名称不能传达足够的意义。`x1` 和 `x2` 这类名称的问题在于，即使你能发现 `x` 是什么，你也不会知道 `x1` 和 `x2` 之间的关系。 太长的名称难以输入，而且会掩盖程序的视觉结构。

Gorla、Benander 和 Benander 发现，当变量名平均长度为 10 到 16 个字符时，调试程序所需的努力最小（1990）。名称平均长度为 8 到 20 个字符的程序几乎一样容易调试。这个指导原则并不意味着你应该努力使所有变量名都变成 9 到 15 或 10 到 16 个字符。它确实意味着，如果你查看你的代码，看到很多更短的名称，你应该检查以确保这些名称尽可能清晰。

你可能会通过采用"金发姑娘原则"（Goldilocks-and-the-Three-Bears approach）来命名变量而胜出，如表 11-2 所示。

**表 11-2. 太长、太短或刚刚好的变量名**

| 类别 | 示例 |
|------|------|
| 太长 | numberOfPeopleOnTheUsOlympicTeam |
| | numberOfSeatsInTheStadium |
| | maximumNumberOfPointsInModernOlympics |
| 太短 | n, np, ntm |
| | n, ns, nsisd |
| | m, mp, max, points |
| 刚刚好 | numTeamMembers, teamMemberCount |
| | numSeatsInStadium, seatCount |

## 作用域对变量名的影响

交叉参考

作用域的更详细讨论请参见"作用域"。

短变量名总是坏的吗？不，并不总是。当你给变量起一个像 `i` 这样的短名字时，长度本身就说明了一些关于变量的信息——即变量是一个临时值，作用域有限。

阅读这种变量的程序员应该能够假设它的值不会在几行代码之外使用。当你给变量命名为 `i` 时，你是在说："这个变量是一个普通的循环计数器或数组索引，在这几行代码之外没有任何意义。"

J. Hansen 的研究发现，较长的名称对于很少使用的变量或全局变量更好，较短的名称对于局部变量或循环变量更好（Shneiderman 1980）。然而，短名称容易出现许多问题，一些谨慎的程序员出于防御性编程策略的考虑，完全避免使用它们。

在全局命名空间中使用的名称上加修饰符。如果你有在全局命名空间中的变量（命名常量、类名等），考虑你是否需要采用一种约定来划分全局命名空间并避免命名冲突。在 C++ 和 C# 中，你可以使用 `namespace` 关键字来划分全局命名空间。

示例 11-3. 在 C++ 中使用 namespace 关键字划分全局命名空间的示例

```cpp
namespace UserInterfaceSubsystem {
   ...
   // lots of declarations
   ...
}

namespace DatabaseSubsystem {
   ...
   // lots of declarations
   ...
}
```

如果你在 `UserInterfaceSubsystem` 和 `DatabaseSubsystem` 中都声明了一个 `Employee` 类，你可以通过写 `UserInterfaceSubsystem::Employee` 或 `DatabaseSubsystem::Employee` 来确定你想引用哪个。在 Java 中，你可以使用包来完成同样的事情。

在不支持命名空间或包的语言中，你仍然可以使用命名约定来划分全局命名空间。一个约定是要求全局可见的类加上子系统助记符前缀。用户界面员工类可能变成 `uiEmployee`，数据库员工类可能变成 `dbEmployee`。这最大限度地降低了全局命名空间冲突的风险。

## 变量名中的计算值修饰符

许多程序有包含计算值的变量：总计、平均值、最大值等。如果你用 Total、Sum、Average、Max、Min、Record、String 或 Pointer 这样的修饰符修改名称，把修饰符放在名称的末尾。

这个做法有几个优点。首先，变量名最重要的部分——赋予变量大部分含义的部分——在前面，所以它最突出，最先被阅读。其次，通过建立这个约定，你可以避免如果在同一程序中同时使用 `totalRevenue` 和 `revenueTotal` 可能产生的混淆。这些名字在语义上是等价的，这个约定会阻止它们被当作不同的东西使用。第三，像 `revenueTotal`、`expenseTotal`、`revenueAverage` 和 `expenseAverage` 这样一组名称具有令人愉悦的对称性。像 `totalRevenue`、`expenseTotal`、`revenueAverage` 和 `averageExpense` 这样一组名称则不吸引人。最后，一致性提高了可读性并便于维护。

计算值放在名称末尾这一规则的一个例外是 `Num` 修饰符的习惯位置。`Num` 放在变量名的开头，指的是一个总数：`numCustomers` 是客户的总数。`Num` 放在变量名的末尾，指的是一个索引：`customerNum` 是当前客户的编号。`numCustomers` 末尾的 `s` 是含义差异的另一个提示。但是，由于使用 `Num` 如此频繁地造成混淆，最好通过使用 `Count` 或 `Total` 来指代客户总数，使用 `Index` 来指代特定客户，从而回避整个问题。因此，`customerCount` 是客户的总数，`customerIndex` 指的是特定客户。

## 命名循环索引

交叉参考

关于循环的详细信息，请参见第16章。

关于在循环中命名变量的指导原则已经形成，因为循环是计算机编程中如此常见的特性。`i`、`j` 和 `k` 是习惯性的名称：

示例 11-4. 简单循环变量名的 Java 示例

```java
for ( i = firstItem; i < lastItem; i++ ) {
   data[ i ] = 0;
}
```

如果一个变量要在循环之外使用，它应该被赋予比 `i`、`j` 或 `k` 更有意义的名称。例如，如果你正在从文件中读取记录，并且需要记住已经读取了多少条记录，像 `recordCount` 这样的名称是合适的：

示例 11-5. 描述性良好的循环变量名的 Java 示例

```java
recordCount = 0;
while ( moreScores() ) {
   score[ recordCount ] = GetNextScore();
   recordCount++;
}

// lines using recordCount
...
```

如果循环超过几行，很容易忘记 `i` 应该代表什么，所以你最好给循环索引一个更有意义的名称。因为代码经常被更改、扩展和复制到其他程序中，许多有经验的程序员完全避免使用 `i` 这样的名字。

循环变长的一个常见原因是嵌套。如果你有几个嵌套循环，为循环变量分配较长的名称以提高可读性。

示例 11-6. 嵌套循环中良好循环名的 Java 示例

```java
for ( teamIndex = 0; teamIndex < teamCount; teamIndex++ ) {
   for ( eventIndex = 0; eventIndex < eventCount[ teamIndex ]; eventIndex++ ) {
      score[ teamIndex ][ eventIndex ] = 0;

   }
}
```

为循环索引变量精心选择的名称避免了常见的问题：索引串扰——想说 `i` 的时候说成了 `j`，想说 `j` 的时候说成了 `i`。它们还使数组访问更清晰：`score[ teamIndex ][ eventIndex ]` 比 `score[ i ][ j ]` 更有信息量。

如果你必须使用 `i`、`j` 和 `k`，不要将它们用于简单循环索引以外的其他任何用途——这个约定太成熟了，打破它用于其他目的是令人困惑的。避免这类问题的最简单方法就是想出比 `i`、`j` 和 `k` 更有描述性的名字。

## 命名状态变量

状态变量描述程序的状态。这里有一个命名指导原则：

为状态变量想一个比 `flag` 更好的名字。最好把标志看作是状态变量。标志的名字里永远不应该有 `flag`，因为那不会给你任何关于标志做什么的线索。为了清晰，应该为标志赋值，并且应该用枚举类型、命名常量或充当命名常量的全局变量来测试它们的值。以下是一些名称糟糕的标志示例：

**含义模糊的标志的 C++ 示例**

示例 11-7.

```cpp
if ( flag ) ...
if ( statusFlag & 0x0F ) ...
if ( printFlag == 16 ) ...
if ( computeFlag == 0 ) ...

flag = 0x1;
statusFlag = 0x80;
printFlag = 16;
computeFlag = 0;
```

像 `statusFlag = 0x80` 这样的语句，除非你编写了代码或者有文档告诉你 `statusFlag` 是什么以及 `0x80` 代表什么，否则不会给你任何关于代码做什么的线索。以下是更清晰的等效代码示例：

示例 11-8. 更好地使用状态变量的 C++ 示例

```cpp
if ( dataReady ) ...
if ( characterType & PRINTABLE_CHAR ) ...
if ( reportType == ReportType_Annual ) ...
if ( recalcNeeded == True ) ...

dataReady = true;
characterType = CONTROL_CHARACTER;
reportType = ReportType_Annual;
recalcNeeded = false;
```

显然，`characterType = CONTROL_CHARACTER` 比 `statusFlag = 0x80` 更有意义。同样，条件 `if ( reportType == ReportType_Annual )` 比 `if ( printFlag == 16 )` 更清晰。第二个示例表明，你可以将这种方法与枚举类型以及预定义命名常量一起使用。以下是你如何用法用命名常量和枚举类型来设置示例中使用的值：

示例 11-9. 在 C++ 中声明状态变量

```cpp
// values for CharacterType
const int LETTER = 0x01;
const int DIGIT = 0x02;
const int PUNCTUATION = 0x04;
const int LINE_DRAW = 0x08;
const int PRINTABLE_CHAR = ( LETTER | DIGIT | PUNCTUATION | LINE_DRAW );

const int CONTROL_CHARACTER = 0x80;

// values for ReportType
enum ReportType {
   ReportType_Daily,
   ReportType_Monthly,
   ReportType_Quarterly,
   ReportType_Annual,
   ReportType_All
};
```

当你发现自己正在"琢磨"一段代码时，考虑重命名变量。琢磨侦探小说是可以的，但你不应该需要琢磨代码。你应该能够阅读它。

## 命名临时变量

临时变量用于保存计算的中间结果，作为临时占位符，以及保存内部值。它们通常被称为 `temp`、`x` 或其他模糊且不具描述性的名称。一般来说，临时变量表明程序员还没有完全理解问题。此外，由于这些变量被正式给予"临时"状态，程序员倾向于比其他变量更随意地对待它们，从而增加了错误的机会。

对"临时"变量保持警惕。临时保存值通常是必要的。但在某种程度上，你程序中的大多数变量都是临机的。称其中几个为临时的可能表明你不确定它们的真正用途。考虑以下示例：

示例 11-10. 含义不明确的"临时"变量名的 C++ 示例

```cpp
// Compute roots of a quadratic equation.
// This assumes that (b^2-4*a*c) is positive.
temp = sqrt( b^2 - 4*a*c );
root[0] = ( -b + temp ) / ( 2 * a );
root[1] = ( -b - temp ) / ( 2 * a );
```

将表达式 `sqrt( b^2 - 4 * a * c )` 的值存储在变量中是可以的，特别是因为它在后面两个地方使用。但名称 `temp` 不会告诉你变量做了什么。更好的方法如下例所示：

示例 11-11. 用真实变量名替换"临时"变量名的 C++ 示例

```cpp
// Compute roots of a quadratic equation.
// This assumes that (b^2-4*a*c) is positive.
discriminant = sqrt( b^2 - 4*a*c );
root[0] = ( -b + discriminant ) / ( 2 * a );
root[1] = ( -b - discriminant ) / ( 2 * a );
```

这本质上是相同的代码，但通过使用准确、描述性的变量名得到了改进。

## 命名布尔变量

以下是命名布尔变量的一些指导原则：

记住常用的布尔变量名。以下是一些特别有用的布尔变量名：

| 名称 | 说明 |
|------|------|
| done | 使用 `done` 来指示某事是否完成。变量可以指示循环是否完成或其他操作是否完成。在做某事之前将 `done` 设为 `false`，在完成某事时将其设为 `true`。 |
| error | 使用 `error` 来指示是否发生了错误。在没有错误发生时将变量设为 `false`，在发生错误时将其设为 `true`。 |
| found | 使用 `found` 来指示是否找到了某个值。在未找到值时将 `found` 设为 `false`，一旦找到值就将其设为 `true`。在数组中搜索值时、在文件中搜索员工 ID 时、在工资支票列表中搜索特定金额时等使用 `found`。 |
| success 或 ok | 使用 `success` 或 `ok` 来指示操作是否成功。在操作失败时将变量设为 `false`，在操作成功时将其设为 `true`。如果可以，用更具体的名称替换 `success`，描述成功的确切含义。如果程序在处理完成时成功，你可能使用 `processingComplete`。如果程序在找到值时成功，你可能使用 `found`。 |

给布尔变量起暗示 true 或 false 的名称。像 `done` 和 `success` 这样的名称是好的布尔名称，因为状态要么是 true 要么是 false；某事完成了或没有；它成功了或没有。另一方面，像 `status` 和 `sourceFile` 这样的名称是糟糕的布尔名称，因为它们不是明显的是或否。如果 `status` 是 true 意味着什么？是否意味着某物有状态？万物都有状态。true 是否意味着某物的状态是 OK？或者 false 是否意味着没有问题？用像 `status` 这样的名字，你无法判断。

为了更好的结果，用像 `error` 或 `statusOK` 这样的名称替换 `status`，用 `sourceFileAvailable` 或 `sourceFileFound` 或任何变量代表的东西来替换 `sourceFile`。

一些程序员喜欢在布尔名称前面加 `Is`。然后变量名变成了一个问题：`isdone?`、`isError?`、`isFound?`、`isProcessingComplete?` 用 true 或 false 回答这个问题提供了变量的值。这种方法的一个好处是它不适用于模糊的名称：`isStatus?` 完全没有意义。一个缺点是它使简单的逻辑表达式不太可读：`if ( isFound )` 比 `if ( found )` 稍微不那么可读。

使用肯定的布尔变量名。像 `notFound`、`notdone` 和 `notSuccessful` 这样的否定名称在被否定时难以阅读，例如：

```
if not notFound
```

这样的名称应该用 `found`、`done` 或 `processingComplete` 替换，然后根据需要用运算符否定。如果你要找的东西被找到了，就用"找到了"（found it）而不是"没找不到"（not notFound）。

## 命名枚举类型

交叉参考

关于使用枚举类型的详细信息，请参见"枚举类型"。

当你使用枚举类型时，你可以通过使用组前缀（如 `Color_`、`Planet_` 或 `Month_`）来确保类型的所有成员都清楚属于同一组。以下是使用前缀识别枚举类型元素的示例：

示例 11-12. 用于枚举类型的带有前缀命名约定的 Visual Basic 示例

```vb
Public Enum Color
   Color_Red
   Color_Green
   Color_Blue
End Enum

Public Enum Planet
   Planet_Earth
   Planet_Mars
   Planet_Venus
End Enum

Public Enum Month
   Month_January
   Month_February
   ...
   Month_December
End Enum
```

此外，枚举类型本身（Color、Planet 或 Month）可以通过各种方式识别，包括全大写或前缀（`e_Color`、`e_Planet` 或 `e_Month`）。有人可能会争辩说，枚举本质上是用户定义的类型，所以枚举的名称应该与其他用户定义类型（如类）的格式相同。另一个论点是，枚举是类型，但它们也是常量，所以枚举类型名应该格式化为常量。本书使用混合大小写的约定来命名枚举类型。

在某些语言中，枚举类型更像类，枚举的成员总是以前缀枚举名称为前缀，如 `Color.Color_Red` 或 `Planet.Planet_Earth`。如果你在那种语言中工作，重复前缀没有意义，所以你可以将枚举类型本身的名称视为前缀，并将名称简化为 `Color.Red` 和 `Planet.Earth`。

## 命名约定的力量

一些程序员抵制标准和约定——而且有充分的理由。一些标准和约定是死板的和无效的——对创造性和程序质量具有破坏性。不幸的是，有效的标准是你可以使用的最强大工具之一。本节讨论为什么、何时以及如何创建你自己的变量命名标准。

---

## 为什么要制定约定？

约定提供了几个具体的好处：

- **它们让你可以假设更多的东西**。通过做一个全局决策而不是许多本地决策，你可以专注于代码更重要的特性。
- **它们帮助你跨项目传递知识**。名称的相似性让你更容易、更自信地理解不熟悉的变量应该做什么。
- **它们帮助你更快地学习新项目的代码**。而不是学习 Anita 的代码是这样的，Julia 的是那样的，Kristin 的是别的什么，你可以使用更一致的代码。
- **它们减少名称的泛滥**。没有命名约定，你很容易用两个不同的名字称呼同一个东西。例如，你可能同时称总分为 `pointTotal` 和 `totalPoints`。当你编写代码时，这对你来说可能不会混淆，但对后来阅读代码的新程序员来说可能会非常混乱。
- **它们补偿语言的弱点**。你可以使用约定来模拟命名常量和枚举类型。约定可以区分局部、类和全局数据，并且可以合并编译器不支持的类型的信息。
- **它们强调相关项目之间的关系**。如果你使用对象数据，编译器会自动处理这个问题。如果你的语言不支持对象，你可以用命名约定来补充。像 `address`、`phone` 和 `name` 这样的名称没有表明变量是相关的。但是假设你决定所有员工数据变量都应该以 Employee 前缀开头。`employeeAddress`、`employeePhone` 和 `employeeName` 清楚地表明这些变量是相关的。编程约定可以弥补你正在使用的语言的弱点。

---

关键在于，任何约定通常都比没有约定好。约定可能是武断的。命名约定的力量不是来自选择的特定约定，而是来自约定存在的这一事实，为代码增加了结构，让你减少需要担心的事情。

---

## 你何时应该制定命名约定？

关于何时应该建立命名约定没有硬性规定，但以下是一些约定值得的情况：

- 当多个程序员在同一个项目上工作时
- 当你计划将程序移交给另一个程序员进行修改和维护时（这几乎是肯定的）
- 当你的程序在你的组织中由其他程序员审查时
- 当你的程序太大，你不能一次把整个东西记在脑子里，必须分块思考时
- 当程序寿命足够长，你可能把它搁置几周或几个月后再处理时
- 当你有许多在项目中常见的特殊术语，想在编码中有标准术语或缩写可以使用时

你总是从拥有某种命名约定中受益。上述考虑应该帮助你确定在特定项目上使用约定的程度。

## 正式程度的不同等级

不同的约定有不同的正式程度。一个非正式的约定可能简单到"使用有意义的名称"。其他非正式约定将在下一节描述。一般来说，你需要的正式程度取决于在程序上工作的人数、程序的大小以及程序的预期寿命。在微小的、可丢弃的项目上，严格的约定可能是不必要的开销。在涉及多人的更大项目上，无论是最初还是程序生命周期中，正式约定是提高可读性的不可缺少的辅助工具。

## 语言无关约定指南

以下是创建语言无关约定的一些指导原则：

**区分变量名和例程名**。本书使用的约定是以小写字母开头命名变量和对象，以大写字母开头命名例程：`variableName` vs. `RoutineName()`。

**区分类和对象**。类名和对象名之间的对应关系——或者类型和这些类型的变量之间的对应关系——可能变得棘手。有几个标准选项存在，如下例所示：

示例 11-13. 选项 1：通过首字母大写区分类型和变量

```cpp
Widget widget;
LongerWidget longerWidget;
```

示例 11-14. 选项 2：通过全大写区分类型和变量

```cpp
WIDGET widget;
LONGERWIDGET longerWidget
```

示例 11-15. 选项 3：通过类型的前缀 "t_" 区分

```cpp
t_Widget Widget;
t_LongerWidget LongerWidget;
```

示例 11-16. 选项 4：通过变量的前缀 "a" 区分

```cpp
Widget aWidget;
LongerWidget aLongerWidget;
```

示例 11-17. 选项 5：通过为变量使用更具体的名称区分

```cpp
Widget employeeWidget;
LongerWidget fullEmployeeWidget;
```

这些选项各有优缺点。选项 1 是大小写敏感语言（包括 C++ 和 Java）中的常见约定，但一些程序员不愿意仅基于大小写来区分名称。事实上，创建仅在名称第一个字母大小写不同的名称似乎提供了太少的"心理距离"，两个名称之间的视觉差异太小。

如果任何一种语言是大小写不敏感的，选项 1 方法在混合语言环境中无法一致应用。例如，在 Microsoft Visual Basic 中，`Dim widget as Widget` 会产生语法错误，因为 `widget` 和 `Widget` 被视为同一个标记。

选项 2 在类型名和变量名之间创建了更明显的区别。然而，由于历史原因，在 C++ 和 Java 中全大写用于表示常量，而且该方法在混合语言环境中存在与选项 1 相同的问题。

选项 3 在所有语言中都适用，但一些程序员出于美学原因不喜欢前缀的想法。

选项 4 有时用作选项 3 的替代方案，但它有一个缺点，即改变了类的每个实例的名称，而不仅仅是类名。

选项 5 需要在逐个变量的基础上进行更多思考。在大多数情况下，被迫为变量想出一个特定的名字会产生更易读的代码。但有时一个小部件确实只是一个通用的小部件，在这些情况下，你会发现自己在想出不那么明显的名字，比如 `genericWidget`，这些名字可以说更不易读。

简而言之，每个可用选项都涉及权衡。本书中的代码使用选项 5，因为它是阅读代码的人不一定熟悉不那么直观的命名约定的情况下最容易理解的。

**识别全局变量**。一个常见的编程问题是全局变量的滥用。例如，如果你给所有全局变量名加上 `g_` 前缀，看到变量 `g_RunningTotal` 的程序员会知道它是一个全局变量，并相应地对待它。

**识别成员变量**。识别类的成员数据。明确变量既不是局部变量也不是全局变量。例如，你可以用 `m_` 前缀标识类成员变量，以表明它是成员数据。

**识别类型定义**。类型的命名约定有两个目的：它们明确地将名称标识为类型名称，并且避免与变量名称冲突。为了满足这些考虑，前缀或后缀是一个好的方法。在 C++ 中，自定义方法是使用全大写字母表示类型名称——例如，`COLOR` 和 `MENU`。（此约定适用于 typedef 和结构体，不适用于类名。）但这会产生与命名预处理器常量混淆的可能性。为了避免混淆，你可以在类型名前加上 `t_` 前缀，如 `t_Color` 和 `t_Menu`。

**识别命名常量**。命名常量需要被识别，这样你就可以判断你是从另一个变量（其值可能会改变）赋值给变量，还是从命名常量赋值。在 Visual Basic 中，还有一个额外的可能性，即值可能来自函数。Visual Basic 不要求函数名使用括号，而在 C++ 中，即使是没有参数的函数也使用括号。

命名常量的一种方法是在常量名前使用 `c_` 前缀。那会给你像 `c_RecsMax` 或 `c_LinesPerPageMax` 这样的名称。在 C++ 和 Java 中，约定是使用全大写字母，可能用下划线分隔单词，如 `RECSMAX` 或 `RECS_MAX` 以及 `LINESPERPAGEMAX` 或 `LINES_PER_PAGE_MAX`。

**识别枚举类型的元素**。枚举类型的元素需要被识别，原因与命名常量相同——为了容易判断名称是用于枚举类型而不是变量、命名常量或函数。标准方法适用：你可以在类型名本身使用全大写或 `e_` 或 `E_` 前缀，并在类型的特定前缀基础上为类型的成员使用前缀，如 `Color_` 或 `Planet_`。

**在语言不强制执行的情况下识别仅输入参数**。有时输入参数被意外修改。在 C++ 和 Visual Basic 等语言中，你必须明确指出你是否希望将被修改的值返回给调用例程。这在 C++ 中用 `*`、`&` 和 `const` 修饰符表示，在 Visual Basic 中用 `ByRef` 和 `ByVal` 表示。

在其他语言中，如果你修改了一个输入变量，无论你是否喜欢，它都会被返回。传递对象时尤其如此。例如，在 Java 中，所有对象都按"值"传递，所以当你将对象传递给例程时，对象的内容可以在被调用的例程内被更改（Arnold, Gosling, Holmes 2000）。

交叉参考

用命名约定来增强语言以弥补语言本身的局限性，是"编程进入语言"而不是"在语言中编程"的一个例子。关于编程进入语言的更多细节，请参见"编程进入你的语言，而不是在语言中编程"。

在这些语言中，如果你建立一个命名约定，其中仅输入参数被给予 `const` 前缀（或 `final`、`nonmodifiable` 或类似的），那么当你看到任何带有 `const` 前缀的内容在等号左边时，你就会知道发生了错误。如果你看到 `constMax.SetNewMax( … )`，你就会知道这是个错误，因为 `const` 前缀表明该变量不应该被修改。

**格式化名称以增强可读性**。提高可读性的两个常用技巧是使用大小写和分隔字符来分隔单词。例如，`GYMNASTICSPOINTTOTAL` 比 `gymnasticsPointTotal` 或 `gymnastics_point_total` 可读性差。C++、Java、Visual Basic 和其他语言都允许使用混合大小写字母。C++、Java、Visual Basic 和其他语言也允许使用下划线（`_`）分隔符。

尽量不要混合这些技巧；那会使代码难以阅读。然而，如果你真诚地尝试一致地使用这些可读性技巧中的任何一个，它都会改进你的代码。人们曾经就细节问题进行过激烈的争论，比如名字的第一个字母是否应该大写（`TotalPoints` vs. `totalPoints`），但只要你和你的团队保持一致，差别就不大。本书使用小写开头，因为 Java 实践的强势以及为了在几种语言中保持风格相似性。

## 语言特定约定指南

遵循你正在使用的语言的命名约定。你可以为大多数语言找到描述风格指南的书籍。以下各节提供了 C、C++、Java 和 Visual Basic 的指导原则。

### C 约定

延伸阅读

关于 C 编程风格的经典著作是《C Programming Guidelines》（Plum 1984）。

有几个命名约定专门适用于 C 编程语言：

| 约定 | 说明 |
|------|------|
| c 和 ch | 字符变量 |
| i 和 j | 整数索引 |
| n | 某物的数量 |
| p | 指针 |
| s | 字符串 |
| 前处理器宏 | 全大写。这通常扩展为包括类型定义。 |
| 变量和例程名 | 全小写 |
| 下划线（_）字符 | 用作分隔符：`letters_in_lowercase` 比 `lettersinlowercase` 更易读 |

这些是通用 UNIX 风格和 Linux 风格 C 编程的约定，但 C 约定在不同的环境中是不同的。在 Microsoft Windows 中，C 程序员倾向于使用匈牙利命名约定的一种形式和混合大小写字母作为变量名。在 Macintosh 上，C 程序员倾向于为例程使用混合大小写名称，因为 Macintosh 工具箱和操作系统例程最初是为 Pascal 接口设计的。

### C++ 约定

延伸阅读

关于 C++ 编程风格的更多信息，请参见《The Elements of C++ Style》（Misfeldt, Bumgardner, and Gray 2004）。

以下是围绕 C++ 编程发展起来的约定：

| 约定 | 说明 |
|------|------|
| i 和 j | 整数索引 |
| p | 指针 |
| 常量、typedef 和前处理器宏 | 全大写 |
| 类和其他类型名称 | 混合大小写（MixedUpperAndLowerCase） |
| 变量和函数名 | 第一个词小写，后续每个词的首字母大写——例如，`variableOrRoutineName` |
| 下划线 | 不用作名称内的分隔符，除非在全大写的名称和某些前缀中（如用于识别全局变量的前缀） |

与 C 编程一样，这个约定远非标准，不同的环境已经标准化了不同的约定细节。

## 标准化前缀

延伸阅读

关于匈牙利命名约定的更多细节，请参见"The Hungarian Revolution"（Simonyi and Heller 1991）。

为常见含义标准化前缀提供了一种简洁但一致且可读的命名数据方法。最著名的标准化前缀方案是匈牙利命名约定，它是一套详细的变量和例程命名指导原则（不是匈牙利人！），曾在微软 Windows 编程中广泛使用。虽然匈牙利命名约定不再广泛使用，但标准化简洁、精确缩写的基本思想继续具有价值。

标准化前缀由两部分组成：用户定义类型（UDT）缩写和语义前缀。

---

## 用户定义类型缩写

UDT 缩写标识所命名的对象或变量的数据类型。UDT 缩写可能指的是窗口、屏幕区域和字体等实体。UDT 缩写通常不指编程语言提供的任何预定义数据类型。

UDT 用短代码描述，这些代码你为特定程序创建，然后在程序中标准化使用。代码是助记符，如 `wn` 代表窗口，`scr` 代表屏幕区域。表 11-6 提供了你可能在文字处理程序中使用的 UDT 示例列表。

**表 11-6. 文字处理程序的 UDT 示例**

| UDT 缩写 | 含义 |
|----------|------|
| ch | 字符（C++ 意义上的字符不是字符，而是文字处理程序用来表示文档中字符的数据类型） |
| doc | 文档 |
| pa | 段落 |
| scr | 屏幕区域 |
| sel | 选择 |
| wn | 窗口 |

当你使用 UDT 时，你还定义了使用与 UDT 相同缩写的编程语言数据类型。因此，如果你有表 11-6 中的 UDT，你会看到如下数据声明：

```cpp
CH    chCursorPosition;
SCR   scrUserWorkspace;
DOC   docActive
PA    firstPaActiveDocument;
PA    lastPaActiveDocument;
WN    wnMain;
```

同样，这些示例与文字处理程序相关。对于你自己的项目，你会为你环境中使用的最常见的 UDT 创建 UDT 缩写。

## 语义前缀

语义前缀比 UDT 更进一步，描述变量或对象的使用方式。与 UDT 不同，UDT 因项目而异，语义前缀在项目中是相当标准的。表 11-7 显示了标准语义前缀列表。

**表 11-7. 语义前缀**

| 语义前缀 | 含义 |
|----------|------|
| c | 计数（例如，记录数、字符数等） |
| first | 数组中需要处理的第一个元素。`first` 类似于 `min`，但是相对于当前操作而不是数组本身。 |
| g | 全局变量 |
| i | 数组索引 |
| last | 数组中需要处理的最后一个元素。`last` 是 `first` 的对应词。 |
| lim | 数组中需要处理的元素的上限。`lim` 不是有效索引。像 `last` 一样，`lim` 用作 `first` 的对应词。与 `last` 不同，`lim` 表示数组的非包含上限；`last` 表示最终的有效元素。一般来说，`lim` 等于 `last + 1`。 |
| m | 类级变量 |
| max | 数组或其他类型列表中的绝对最后一个元素。`max` 指的是数组本身，而不是对数组的操作。 |
| min | 数组或其他类型列表中的绝对第一个元素。 |
| p | 指针 |

语义前缀格式为小写或混合大小写，并根据需要与 UDT 和其他语义前缀组合。例如，文档中的第一个段落将命名为 `pa` 以表明它是一个段落，加上 `first` 以表明它是第一个段落：`firstPa`。段落集合的索引将命名为 `iPa`；`cPa` 是计数，或段落数；`firstPaActiveDocument` 和 `lastPaActiveDocument` 是当前活动文档中的第一个和最后一个段落。

## 标准化前缀的优点

标准化前缀给了你拥有命名约定的所有一般优势，以及其他几个优势。因为如此多的名称是标准的，你在任何单个程序或类中需要记住的名称更少。

标准化前缀为命名中容易不精确的几个领域增加了精确性。`min`、`first`、`last` 和 `max` 之间精确的区别特别有帮助。

标准化前缀使名称更紧凑。例如，你可以使用 `cpa` 表示段落计数，而不是 `totalParagraphs`。你可以使用 `ipa` 来标识段落数组的索引，而不是 `indexParagraphs` 或 `paragraphsIndex`。

最后，标准化前缀允许你在使用抽象数据类型时准确地检查类型，而编译器不一定能检查：`paReformat = docReformat` 可能错误，因为 `pa` 和 `doc` 是不同的 UDT。

标准化前缀的一个主要陷阱是程序员忽略给变量一个有意义的名称，除了它的前缀。如果 `ipa` 明确表示段落数组的索引，很容易不把名字变得更丰富，比如 `ipaActiveDocument`。为了可读性，闭合循环，想出一个描述性的名字。

## 创建可读性强短名称

使用短变量名的愿望在某种程度上是计算机早期时代的遗留物。像汇编器、通用 Basic 和 Fortran 这样的旧语言将变量名限制为 2-8 个字符，迫使程序员创建短名称。早期计算与数学的联系更紧密，数学在求和和其他方程中使用 `i`、`j` 和 `k` 这样的变量。在 C++、Java 和 Visual Basic 等现代语言中，你可以创建几乎任意长度的名称；你几乎没有理由缩短有意义的名称。

如果情况确实需要你创建短名称，注意一些缩短名称的方法比其他方法更好。你可以通过消除不必要的词、使用短同义词以及使用多种缩写策略中的任何一种来创建好的短变量名。熟悉多种缩写技巧是个好主意，因为没有一种技术在所有情况下都能很好地工作。

---

## 一般缩写指南

以下是创建缩写的一些指导原则。其中一些是相互矛盾的，所以不要试图同时使用所有。

- 使用标准缩写（常用的，在词典中列出的）。
- 删除所有非首元音。（computer 变成 cmptr，screen 变成 scrn。apple 变成 appl，integer 变成 intgr。）
- 删除冠词：and、or、the 等。
- 使用每个词的首字母或前几个字母。
- 一致地从每个词的第一、第二或第三个字母（ whichever is appropriate）开始截断。
- 保留每个词的首字母和末字母。
- 在名称中使用每一个重要的词，最多三个词。
- 删除无用的后缀——ing、ed 等。
- 保留每个音节中最明显的音。
- 确保不要改变变量的含义。
- 重复这些技巧，直到将每个变量名缩写为 8 到 20 个字符之间，或者你的语言限制变量名的字符数。

## 关于缩写的评论

创建缩写时你可能会陷入几个陷阱。以下是避免陷阱的一些规则：

**不要通过删除一个字符来缩写**。输入一个字符几乎没有额外的工作，一个字符的节省几乎不值得可读性的损失。这就像日历上有"Jun"和"Jul"一样。你一定很匆忙才会把 June 拼写成"Jun"。对于大多数单字母删除，很难记住你是否删除了字符。要么删除一个以上的字符，要么拼写出完整的单词。

**一致地缩写**。始终使用相同的缩写。例如，到处使用 Num 或到处使用 No，但不要两者都使用。同样，不要在某些名字中缩写一个词而在其他名字中不缩写。例如，不要在一些地方使用完整单词 Number，在其他地方使用缩写 Num。

**创建你可以发音的名字**。使用 `xPos` 而不是 `xPstn`，使用 `needsComp` 而不是 `ndsCmptg`。应用电话测试——如果你不能通过电话向某人读你的代码，重命名你的变量使其更具特色（Kernighan and Plauger 1978）。

**避免导致误读或误发音的组合**。为了引用 B 的末尾，使用 `ENDB` 而不是 `BEND`。如果你使用好的分隔技术，你就不需要这个指导原则，因为 `B-END`、`BEnd` 或 `b_end` 不会被误发音。

**使用同义词库解决命名冲突**。创建短名称的一个问题是命名冲突——缩写为相同内容的名称。例如，如果限制为三个字符，你需要在地毯的同一区域使用 fired 和 full revenue disbursal，你可能会无意中把两者都缩写为 `frd`。

避免命名冲突的一个简单方法是使用具有相同含义的不同单词，所以同义词库很方便。在这个例子中，`dismissed` 可能替代 `fired`，`complete revenue disbursal` 可能替代 `full revenue disbursal`。三个字母的缩写变成 `dsm` 和 `crd`，消除了命名冲突。

**在代码中用翻译表记录非常短的名称**。在只允许非常短名称的语言中，包含一个翻译表来提醒变量的助记内容。将表格作为代码块开头的注释包含。这里有一个例子：

示例 11-18. 好翻译表的 Fortran 示例

```fortran
C *******************************************************************
C    Translation Table
C
C    Variable    Meaning
C    --------    -------
C    XPOS        x-Coordinate Position (in meters)
C    YPOS        Y-Coordinate Position (in meters)
C    NDSCMP      Needs Computing (=0 if no computation is needed;
C                                 =1 if computation is needed)
C    PTGTTL      Point Grand Total
C    PTVLMX      Point Value Maximum
C    PSCRMX      Possible Score Maximum
C *****************************************************************
```

你可能认为这种技术已经过时了，但就在 2003 年中期，我与一个客户合作，该客户有数万行用 RPG 编写的代码，变量名限制为 6 个字符。这些问题仍然不时出现。

**在项目级"标准缩写"文档中记录所有缩写**。代码中的缩写会产生两个一般风险：

- 代码的读者可能不理解缩写。
- 其他程序员可能使用多个缩写来指代同一个词，这会造成不必要的混淆。

为了解决这两个潜在问题，你可以创建一个"标准缩写"文档，记录项目上使用的所有编码缩写。文档可以是文字处理文档或电子表格。在非常大的项目上，它可以是一个数据库。文档签入版本控制，任何人在代码中创建新缩写时都要签出。文档中的条目应该按完整单词排序，而不是按缩写排序。

这看起来可能有很多开销，但除了少量启动开销外，它实际上只是建立了一个帮助项目有效使用缩写的机制。它通过记录所有正在使用的缩写来解决上述两个一般风险中的第一个。程序员在代码中创建新缩写时，必须办理签出标准缩写文档、输入缩写并重新签入的手续，这是件好事。这意味着除非缩写如此常见以至于值得麻烦地记录它，否则不会被创建。

这种方法通过减少程序员创建冗余缩写的可能性来解决第二个风险。想要缩写某物的程序员会签出缩写文档并输入新的缩写。如果程序员想要缩写的词已经有缩写，程序员会注意到这一点，然后使用现有缩写而不是创建新的。

这个指导原则所说明的一般问题是写时方便和读时方便之间的区别。这种方法显然会造成写时不便，但程序员在系统生命周期内阅读代码的时间远远多于编写代码的时间。这种方法增加了读时的方便性。当所有尘埃落定在一个项目上时，它也可能改善写时的方便性。

记住，名字对代码的读者比对作者更重要。阅读你自己至少六个月没见过的代码，注意你在哪里必须努力理解名字的含义。下定决心改变导致这种混乱的做法。

## 应避免的命名

以下是关于变量名应避免的一些指导原则：

**避免误导性的名字或缩写**。确保名称是明确的。例如，`FALSE` 通常是 `TRUE` 的反义词，用它作为"无花果和杏仁季节"的缩写将是一个糟糕的选择。

**避免含义相似的名字**。如果你在不损害程序的情况下可以交换两个变量的名字，你需要重命名这两个变量。例如，`input` 和 `inputValue`、`recordNum` 和 `numRecords`、`fileNumber` 和 `fileIndex` 在语义上非常相似，如果你在同一段代码中使用它们，你会很容易混淆它们，并安装一些难以发现的微妙错误。

**避免含义不同但名字相似的变量**。如果你有两个名字相似但含义不同的变量，尝试重命名其中一个或更改你的缩写。避免像 `clientRecs` 和 `clientReps` 这样的名字。它们彼此只有一个字母不同，而这个字母很难注意到。在名字之间至少有两个字母的不同，或者把区别放在开头或结尾。`clientRecords` 和 `clientReports` 比原来的名字好。

交叉参考

这种类似变量名之间差异的技术术语是"心理距离"。详细信息请参见调试中的心理学考虑中的""心理距离"如何提供帮助"。

**避免听起来相似的名字**，如 `wrap` 和 `rap`。当你试图与他人讨论你的代码时，同音异义词会成为障碍。我对极限编程（Beck 2000）过度巧妙地使用"Goal Donor"和"Gold Owner"这两个术语感到恼火，这两个词在说出来时几乎无法区分。你最终会进行这样的对话：

> "我刚才正在和 Goal Donor 说话——"
> 
> 你说的是 "Gold Owner" 还是 "Goal Donor"？
> 
> 我说的是 "Goal Donor"。
> 
> 什么？
> 
> GOAL - - - DONOR！
> 
> 好吧，Goal Donor。你不用喊，Goll' Darn it。
> 
> 你说的是 "Gold Donut"？

记住，电话测试同样适用于听起来相似的名字，就像适用于奇怪缩写的名字一样。

**避免在名字中使用数字**。如果名字中的数字真的很重要，使用数组而不是单独的变量。如果数组不合适，数字更不合适。例如，避免 `file1` 和 `file2`，或 `total1` 和 `total2`。你几乎总是能想出比在名字末尾加上 1 或 2 更好的方法来区分两个变量。我不能说从不使用数字。一些现实世界的实体（如 Route 66 或 Interstate 405）在其中嵌入了数字。但在创建包含数字的名字之前，考虑是否有更好的选择。

**避免在名字中拼写错误的单词**。记住单词应该如何拼写已经够难了。要求人们记住"正确"的拼写错误简直太过分了。例如，为了节省三个字符而将 highlight 拼写错误为 hilite，会使读者很难记住 highlight 是如何拼写的。是 highlite？hilite？hilight？hilit？jai-a-lai-t？谁知道呢？

**避免在英语中经常拼错的单词**。Absense、acummulate、acsend、calender、concieve、defferred、definate、independance、occassionally、prefered、reciept、superseed 以及许多其他是英语中常见的拼写错误。大多数英语手册都包含常用拼写错误列表。避免在你的变量名中使用这样的词。

**不要仅通过大小写来区分变量名**。如果你在 C++ 等大小写敏感的语言中编程，你可能会想把 fired 缩写为 `frd`，final review duty 缩写为 `FRD`，full revenue disbursal 缩写为 `Frd`。避免这种做法。虽然名字是独特的，但每个名字与特定含义的关联是任意的，令人困惑。`Frd` 可以同样容易地与 final review duty 相关联，而 `FRD` 与 full revenue disbursal 相关联，没有逻辑规则能帮助你或任何其他人记住哪个是哪个。

**避免多种自然语言**。在跨国项目中，强制所有代码使用单一自然语言，包括类名、变量名等。阅读另一个程序员的代码可能是一种挑战；阅读另一个程序员用 Southeast Martian 写的代码是不可能的。

一个更微妙的问题发生在英语的变体中。如果一个项目在多个英语国家进行，标准化为一种英语版本，这样你就不会不断想知道代码应该用"color"还是"colour"，"check"还是"cheque"等等。

**避免使用标准类型、变量和例程的名称**。所有编程语言指南都包含语言保留和预定义名称的列表。偶尔阅读这些列表，以确保你没有踩到你正在使用的语言的脚趾。例如，以下代码片段在 PL/I 中是合法的，但使用它的人绝对是傻瓜：

```pl
if if = then then
   then = else;
else else = if;
```

**不要使用与变量所代表的东西完全无关的名字**。在你的程序中散布 margaret 和 pookie 这样的名字几乎保证了其他人无法理解它。避免使用你男朋友的名字、你妻子的名字、你最喜欢的啤酒的名字或其他聪明（即愚蠢）的变量名，除非程序真的是关于你的男朋友、妻子或最喜欢的啤酒。即使是这样，你也应该明智地认识到，这些中的每一个都可能改变，因此男朋友、妻子和 `favoriteBeer` 这样的通用名称是更好的选择！

**避免包含难以阅读字符的名字**。注意一些字符看起来如此相似，很难区分它们。如果两个名字之间的唯一区别是这样的字符，你可能很难区分这些名字。例如，尝试在以下每一组中圈出不属于的名字：

```
eyeChartl      eyeChartI      eyeChartl
TTLCONFUSION   TTLCONFUSION   TTLC0NFUSION
hard2Read      hardZRead      hard2Read
GRANDTOTAL     GRANDTOTAL     6RANDTOTAL
ttl5           ttlS           ttlS
```

难以区分的配对包括 (1 和 l)、(1 和 I)、(. 和 ,)、(0 和 O)、(2 和 Z)、(; 和 :)、(S 和 5) 以及 (G 和 6)。

这些细节真的重要吗？确实如此！Gerald Weinberg 报告说，在 20 世纪 70 年代，在 Fortran FORMAT 语句中应该使用句号的地方使用了逗号。结果是科学家们错误计算了航天器的轨迹，损失了一个太空探测器——损失了 16 亿美元（Weinberg 1983）。

## 命名变量检查表

### 一般命名考虑

- [ ] 名称是否完全准确地描述了变量代表什么？
- [ ] 名称是否指的是现实世界的问题而不是编程语言解决方案？
- [ ] 名称是否足够长，不需要你琢磨出来？
- [ ] 计算值修饰符（如果有）是否在名称的末尾？
- [ ] 名称是否使用 Count 或 Index 而不是 Num？

### 命名特定类型的数据

- [ ] 循环索引名称是否有意义（如果循环超过一到两行或者是嵌套的，使用除了 i、j 或 k 之外的名称）？
- [ ] 所有"临时"变量是否已重命名为更有意义的名称？
- [ ] 布尔变量命名是否使它们为真时的含义清晰？
- [ ] 枚举类型名称是否包含表示类别的前缀或后缀——例如，Color_Red、Color_Green、Color_Blue 等的 Color_？
- [ ] 命名常量是否为它们代表的抽象实体命名，而不是它们指向的数字？

### 命名约定

- [ ] 约定是否区分了局部、类和全局数据？
- [ ] 约定是否区分了类型名称、命名常量、枚举类型和变量？
- [ ] 约定是否在语言不强制执行的情况下识别例程的仅输入参数？
- [ ] 约定是否尽可能与语言的标准约定兼容？
- [ ] 名称是否格式化为可读性？

### 短名称

- [ ] 代码是否使用长名称（除非有必要使用短名称）？
- [ ] 代码是否避免仅节省一个字符的缩写？
- [ ] 所有单词是否一致缩写？
- [ ] 名称是否可发音？
- [ ] 是否避免可能误读或误发音的名称？
- [ ] 短名称是否在翻译表中记录？

## 关键点

- 好的变量名是程序可读性的关键要素。特定类型的变量（如循环索引和状态变量）需要特定的考虑。
- 名称应尽可能具体。足够模糊或通用以至于可用于多个目的的名称通常是坏名字。
- 命名约定区分局部、类和全局数据。它们区分类型名称、命名常量、枚举类型和变量。
- 无论你正在处理什么样的项目，你都应该采用变量命名约定。你采用的约定类型取决于程序的大小和在上面工作的人数。
- 使用现代编程语言时很少需要缩写。如果你确实使用缩写，在项目字典中跟踪缩写或使用标准化前缀方法。
- 代码被阅读的次数远远多于被编写的次数。确保你选择的名称有利于读时方便而不是写时方便。
