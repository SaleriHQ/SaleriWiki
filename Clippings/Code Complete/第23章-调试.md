## 调试在软件质量中的角色

像测试一样，调试本身并不是提高软件质量的方式；它是诊断缺陷的一种方式。软件质量必须从一开始就内置。构建高质量产品的最佳方式是仔细开发需求、设计良好并使用高质量的编码实践。调试是最后的手段。

## 调试

性能差异

为什么要谈论调试？难道不是每个人都知道如何调试吗？

不，并不是每个人都知道如何调试。对经验丰富的程序员的研究发现，经验丰富的程序员找到同一组缺陷所需的时间与经验不足的程序员相比大约是1:20的差异。此外，一些程序员能找到更多的缺陷，并且更准确地修正它们。以下是一项经典研究的结果，该研究检验了至少四年经验的专业程序员如何有效地调试一个有12个缺陷的程序：

| | 最快的三位程序员 | 最慢的三位程序员 |
|---|---|---|
| 平均调试时间（分钟） | 5.0 | 14.1 |
| 平均未发现的缺陷数 | 0.7 | 1.7 |
| 修复缺陷时引入的平均新缺陷数 | 3.0 | 7.7 |

资料来源："关于人们如何调试计算机程序的一些心理学证据"（Gould 1975）

调试能力最强的三位程序员能够在大约三分之一的时间内找到缺陷，并且在修正时引入的新缺陷只有调试能力最弱的三位程序员的五分之二。最优秀的程序员找到了所有缺陷，并且在修正时没有引入任何新缺陷。最差的程序员漏掉了12个缺陷中的4个，并且在修正他找到的8个缺陷时引入了11个新缺陷。

## 调试

性能差异

但这项研究并没有真正讲述全部故事。在第一轮调试之后，最快的三位程序员的代码中仍然有3.7个缺陷，而最慢的仍然有9.4个缺陷。两组都还没有完成调试。我想知道如果我将相同的发现和错误修复比率应用到额外的调试轮次会怎样。我的结果在统计上不是有效的，但它们仍然很有趣。当我将相同的发现和错误修复比率应用到连续的调试轮次，直到每组剩余的缺陷少于一半时，最快的一组需要总共三轮调试，而最慢的一组需要14轮调试。记住最慢的一组每轮花费的时间几乎是最快一组的三倍，根据我对这项研究的非科学推断，最慢的一组完全调试其程序所需的时间大约是最快一组所需时间的13倍。其他研究也证实了这种巨大差异（Gilb 1977, Curtis 1981）。

除了提供对调试的洞察外，证据支持软件质量的一般原则：提高质量会降低成本。最好的程序员找到了最多的缺陷，最快地找到缺陷，并且最经常做出正确的修改。你不必在质量、成本和时间之间选择——它们是齐头并进的。

## 缺陷即机会

有缺陷意味着什么？假设你不希望程序有缺陷，这意味着你不完全理解程序在做什么。不理解程序在做什么是令人不安的。毕竟，如果你创建了程序，它应该服从你的命令。如果你不确切知道你在告诉计算机做什么，你离仅仅尝试不同的事情直到某些东西似乎起作用只有一小步——也就是说，通过试错进行编程。如果你通过试错进行编程，缺陷是必然的。你不需要学习如何修复缺陷；你需要学习如何首先避免它们。

然而，大多数人都有一些易犯错的，你可能是一位优秀的程序员，只是犯了一个小错误。如果是这样的话，程序中的错误为你提供了一个强大的学习机会。你可以学到很多东西：

了解你正在处理的程序。你需要了解这个程序，因为如果你已经完美地了解它，它就不会有缺陷。你早就纠正它了。

进一步阅读

有关帮助你了解你所犯错误类型的一些实践的详细信息，请参见《软件工程学科》（Humphrey 1995）。

了解你所犯的错误类型。如果你编写了程序，你就插入了缺陷。并不是每天都有一个聚光灯如此清楚地暴露出一个弱点，但这样的日子是一个机会，所以要好好利用它。一旦你找到了错误，问问自己你是如何以及为什么犯了这个错误。你怎样才能更快地找到它？你怎样才能防止它？代码中是否有其他类似的错误？你能否在它们导致各自的问题之前纠正它们？

从必须阅读代码的人的角度了解你代码的质量。你必须阅读你的代码来找到缺陷。这是一个机会，可以批判性地看待你代码的质量。它容易阅读吗？怎样才能更好？用你的发现来重构你当前的代码或改进你下次编写的代码。

了解你如何解决问题。解决调试问题的方法是否能给你信心？你的方法有效吗？你能很快找到缺陷吗？或者你的调试方法很弱？你感到痛苦和沮丧吗？你随机猜测吗？你需要改进吗？考虑到许多项目在调试上花费的时间，如果你观察自己如何调试，你肯定不会浪费时间。花时间分析和改变你的调试方式可能是减少开发程序所需总时间的最快方法。

了解你如何修复缺陷。除了了解你如何找到缺陷之外，你还可以了解你如何修复它们。你是通过应用goto绷带和特殊情况的化妆品做出最简单的修正，从而改变症状而不是解决问题？还是做出系统性的修正，要求准确的诊断并为问题的核心开出治疗方案？

综合考虑，调试是为你自己的改进播种的异常肥沃的土壤。这是所有构建道路交汇的地方：可读性、设计、代码质量——只要你叫得出名字。这是你编写好代码获得回报的地方，尤其是当你做得足够好以至于不必经常调试的时候。

## 无效的方法

不幸的是，学院和大学的编程课程几乎从不提供调试指导。如果你是在大学里学的编程，你可能有一个专门讨论调试的讲座。虽然我的计算机科学教育非常好，但我收到的调试建议的程度是"在程序中放置打印语句来找到缺陷"。这是不够的。如果其他程序员的教育经历和我的一样，那么很多程序员被迫自己重新发明调试概念。真是浪费！

### 调试的魔鬼指南

程序员并不总是使用可用数据来约束他们的推理。他们进行轻微和不合理的修复，而且他们经常不撤销不正确的修复。
—— Iris Vessey

在但丁的地狱愿景中，最低的圈子是留给撒旦本人的。在现代，Old Scratch同意与不能有效学习调试的程序员分享最低的圈子。他通过让程序员使用这些常见的调试方法来折磨他们：

通过猜测找到缺陷。要找到缺陷，在程序中随机散布打印语句。检查输出，看看缺陷在哪里。如果你用打印语句找不到缺陷，试着在程序中更改一些东西，直到某些东西似乎起作用。不要备份程序的原始版本，也不要记录你所做的更改。当你不完全确定程序在做什么时，编程会更令人兴奋。储备可乐和糖果，因为你要在终端前度过一个漫长的夜晚。

不要浪费时间去理解问题。问题很可能是微不足道的，你不需要完全理解它就可以修复它。简单地找到它就够了。

用最明显的修复来修复错误。通常最好只修复你看到的具体问题，而不是浪费大量时间做一些会影响整个程序的大的、雄心勃勃的修正。这是一个完美的例子：

```cpp
x = Compute( y )
if ( y = 17 )
   x = $25.15      -- Compute()对y = 17不工作，所以修复它
```

谁需要深入Compute()处理一个与值17相关的模糊问题，当你可以只是在一个显而易见的地方为它写一个特殊情况？

### 迷信式调试

撒旦已经把地狱的一部分租给了用迷信方式调试的程序员。每个小组都有一个程序员，他有无尽的恶魔机器问题、神秘的编译器缺陷、在满月时出现的隐藏语言缺陷、坏数据、丢失重要更改、一个possessed编辑器错误保存程序——你能想到的都有。这就是"迷信式编程"。

如果你写的程序有问题，那是你的错。不是电脑的错，也不是编译器的错。程序不会每次都做不同的事情。它不是自己写的；是你写的，所以对它负责。

即使错误最初看起来不是你的错，假设它是你的错对你非常有利。这个假设能帮助你调试。当你在寻找缺陷时，很难在你的代码中找到它；当你假设你的代码没有错误时，就更难了。假设错误是你的也能提高你的可信度。如果你声称错误来自于别人的代码，其他程序员会相信你已经仔细检查了问题。如果你假设错误是你的，你就能避免以后当你自己发现它确实是你的缺陷时不得不公开收回的尴尬。

## 寻找缺陷

调试包括找到缺陷并修复它。找到缺陷——并理解它——通常是90%的工作。

幸运的是，你不需要与撒旦签订契约来找到比随机猜测更好的调试方法。通过思考问题进行调试比用蝾螈的眼睛和青蛙耳朵的灰尘进行调试要有效得多，也更有趣。

假设你被要求解决一个谋杀之谜。哪个更有趣：走遍全县，挨家挨户检查每个人在10月17日晚上的不在场证明，还是找到一些线索并推断出凶手的身份？大多数人会选择推断出那个人的身份，大多数程序员也觉得调试的智力方法更令人满意。更重要的是，在二十分之一的时间内有效调试的程序员并不是随机猜测如何修复程序。他们使用的是科学方法——也就是说，科学研究所需的发现和演示过程。

### 调试的科学方法

以下是你使用经典科学方法时经历的步骤：

1. 通过可重复的实验收集数据。
2. 形成一个解释相关数据的假设。
3. 设计一个实验来证明或反驳这个假设。
4. 证明或反驳这个假设。
5. 根据需要重复。

科学方法在调试中有许多相似之处。以下是找到缺陷的有效方法：

1. 稳定错误。
2. 定位错误源（"故障"）。

   a. 收集产生缺陷的数据。
   
   b. 分析收集到的数据，形成关于缺陷的假设。
   
   c. 确定如何通过测试程序或检查代码来证明或反驳这个假设。
   
   d. 使用在2(c)中确定的程序证明或反驳假设。

3. 修复缺陷。
4. 测试修复。
5. 寻找类似的错误。

第一步类似于科学方法的第一步，因为它依赖于可重复性。如果你能稳定缺陷——也就是说，让它可靠地发生——缺陷更容易诊断。第二步使用科学方法的步骤。你收集揭露缺陷的测试数据，分析产生的数据，并形成关于错误源的假设。然后你设计一个测试用例或检查来评估假设，并相应地宣布成功（关于证明你的假设）或更新你的努力。当你证明了你的假设后，你修复缺陷，测试修复，并在代码中搜索类似的错误。

让我们通过一个例子来看看每个步骤。假设你有一个员工数据库程序，有一个间歇性错误。程序应该按字母顺序打印员工及其所得税扣缴列表。以下是部分输出：

```
Formatting, Fred Freeform    $5,877
Global, Gary                 $1,666
Modula, Mildred             $10,788
Many-Loop, Mavis             $8,889
Statement, Sue Switch        $4,000
Whileloop, Wendy             $7,860
```

错误是Many-Loop, Mavis和Modula, Mildred的顺序不对。

### 稳定错误

如果缺陷不能可靠地发生，几乎不可能诊断。使间歇性缺陷可预测地发生是调试中最具挑战性的任务之一。

不能可预测地发生的错误通常是初始化错误、时序问题或悬空指针问题。如果求和的计算有时正确有时错误，那么参与计算的变量可能没有正确初始化——大多数时候它只是碰巧从0开始。如果问题是一个奇怪且不可预测的现象，而你使用的是指针，你几乎肯定有一个未初始化的指针，或者在使用指向已释放内存的指针后继续使用它。

稳定错误通常需要的不只是找到一个产生错误的测试用例。它包括将测试用例简化到仍然产生错误的最简单的那个。简化测试用例的目标是使其如此简单，以至于改变它的任何方面都会改变错误的行为。然后，通过仔细更改测试用例并在受控条件下观察程序的行为，你可以诊断问题。如果你在一个拥有独立测试团队的组织中工作，有时让他们简化测试用例是团队的工作。大多数时候，这是你的工作。

要简化测试用例，你需要再次运用科学方法。假设你有10个因素组合使用产生了错误。形成一个关于哪些因素与产生错误无关的假设。更改假设中无关的因素，重新运行测试用例。如果你仍然得到错误，你可以消除那些因素，你就简化了测试。然后你可以尝试进一步简化测试。如果你没有得到错误，你就反驳了那个特定的假设，你比之前知道了更多。可能某些微妙不同的更改仍然会产生错误，但你至少知道一个特定的更改不会。

在员工扣缴示例中，当程序最初运行时，Many-Loop, Mavis列在Modula, Mildred之后。然而，当程序第二次运行时，列表是正确的：

```
Formatting, Fred Freeform    $5,877
Global, Gary                 $1,666
Many-Loop, Mavis             $8,889
Modula, Mildred             $10,788
Statement, Sue Switch        $4,000
Whileloop, Wendy             $7,860
```

直到Fruit-Loop, Frita被输入并显示在错误位置时，你才想起Modula, Mildred之前也是刚输入就显示在错误位置的。这两个案例的奇怪之处在于它们都是单独输入的。通常，员工是成组输入的。

你的假设：问题与输入单个新员工有关。如果这是真的，再次运行程序应该会把Fruit-Loop, Frita放在正确的位置。以下是第二次运行的结果：

```
Formatting, Fred Freeform    $5,877
Fruit-Loop, Frita            $5,771
Global, Gary                 $1,666
Many-Loop, Mavis             $8,889
Modula, Mildred             $10,788
Statement, Sue Switch        $4,000
Whileloop, Wendy             $7,860
```

这次成功的运行支持了这个假设。为了确认它，你想尝试一次添加几个新员工，看看它们是否显示在错误的顺序中，以及顺序是否在第二次运行时改变。

### 定位错误源

定位错误源也需要使用科学方法。你可能怀疑缺陷是特定问题的结果，比如一个边界差一错误。然后你可以改变你怀疑导致问题的参数——边界以下一个、边界上、边界以上一个——并确定你的假设是否正确。

在运行的示例中，问题的根源可能是当你添加一个新员工时发生的边界差一错误，而不是当你添加两个或更多时。检查代码后，你没有发现明显的边界差一错误。转向B计划，你用一个新员工运行一个测试用例，看看这是否就是问题。你把Hardcase, Henry添加为单一员工，并假设他的记录将会顺序错误。这是你的发现：

```
Formatting, Fred Freeform    $5,877
Fruit-Loop, Frita            $5,771
Global, Gary                 $1,666
Hardcase, Henry                $493
Many-Loop, Mavis             $8,889
Modula, Mildred             $10,788
Statement, Sue Switch        $4,000
Whileloop, Wendy             $7,860
```

Hardcase, Henry的行恰恰在它应该的位置，这意味着你的第一个假设是错误的。问题不是简单地一次添加一个员工造成的。它要么是一个更复杂的问题，要么是完全不同的东西。

再次检查测试运行输出，你注意到Fruit-Loop, Frita和Many-Loop, Mavis是唯一包含连字符的名字。Fruit-Loop在她第一次输入时顺序错误，但Many-Loop没有，不是吗？虽然你没有原始输入的打印输出，但在原始错误中Modula, Mildred似乎顺序错误，但她是在Many-Loop旁边。也许Many-Loop顺序错误，而Modula实际上是正确的。

你再次假设：问题源于带连字符的名字，而不是单独输入的名字。

但这如何解释问题只在新员工第一次输入时出现？你查看代码，发现使用了两种不同的排序例程。一个是在输入员工时使用的，另一个是在保存数据时使用的。仔细查看在员工第一次输入时使用的例程，发现它不应该完全排序数据。它只是将数据大致排序以加快保存例程的排序。因此，问题是在排序之前就打印了数据。带连字符名字的问题是因为粗排序例程没有处理标点字符等细节。现在你可以进一步完善假设了。

你最后一次假设：带有标点字符的名字在保存之前不会被正确排序。

后来你通过额外的测试用例确认了这个假设。

### 寻找缺陷的技巧

一旦你稳定了一个错误并完善了产生它的测试用例，找到它的根源可能是简单的或有挑战性的，这取决于你代码的编写质量。如果你很难找到一个缺陷，可能是因为代码写得不够好。你可能不想听到这个，但这是真的。如果你遇到困难，考虑以下技巧：

使用所有可用的数据来形成你的假设。在创建关于缺陷根源的假设时，在假设中尽可能多地解释数据。在示例中，你可能注意到Fruit-Loop, Frita顺序错误，并形成一个假设，即以"F"开头的名字被错误排序。这是一个糟糕的假设，因为它没有解释Modula, Mildred顺序错误的事实，也没有解释名字第二次运行时被正确排序的事实。如果数据不符合假设，不要丢弃数据——问问为什么它不符合，并创建一个新的假设。

示例中的第二个假设——问题源于带连字符的名字而不是单独输入的名字——最初似乎也没有解释名字第二次运行时被正确排序的事实。然而，在这种情况下，第二个假设引向了一个更完善的假设，并被证明是正确的。假设一开始不能解释所有数据是可以的，只要你继续完善假设，最终它确实能解释所有数据。

完善产生错误的测试用例。如果你找不到错误的根源，尝试比你已经做的更进一步完善测试用例。你可能能够比你假设的更改变一个参数，而关注其中一个参数可能会提供关键的突破。

在单元测试套件中执行代码。缺陷在代码的小片段中往往比在大型集成程序中更容易找到。使用你的单元测试来隔离测试代码。

使用可用的工具。有许多工具可用于支持调试会话：交互式调试器、挑剔的编译器、内存检查器、语法导向编辑器等。正确的工具可以将困难的工作变得容易。例如，对于一个难以找到的错误，程序的一部分正在覆盖另一部分的内存。使用传统调试实践很难诊断这个错误，因为程序员无法确定程序错误地覆盖内存的具体点。程序员使用内存断点来监视特定的内存地址。当程序写入该内存位置时，调试器停止代码，有问题的代码就被暴露了。

使用阴性测试的结果。假设你创建了一个假设并运行了一个测试用例来证明它。进一步假设测试用例反驳了这个假设，所以你仍然不知道错误的根源。你确实知道你以前不知道的一些东西——也就是说，缺陷不在你认为的区域。这缩小了你的搜索范围和剩余可能假设的集合。

头脑风暴可能的假设。不要将自己限制在你想到的第一个假设，试着想出几个。一开始不要分析它们——只要在几分钟内尽可能多地想出来。然后看看每个假设，想想能证明或反驳它的测试用例。这种心理练习有助于打破由于过度集中于一条推理线而导致的调试僵局。

在办公桌旁放一本记事本，列出要尝试的事情。程序员在调试会话中陷入困境的原因之一是他们走得太远进入死胡同。列出一个要尝试的事情清单，如果一种方法不起作用，就转向下一个方法。

缩小代码的可疑区域。如果你一直在测试整个程序或整个类或例程，测试一个更小的部分。使用打印语句、日志记录或追踪来识别产生错误的代码部分。

如果你需要更强大的技术来缩小代码的可疑区域，系统地移除程序的部分，看看错误是否仍然发生。如果不发生，你就知道它在被你移除的部分中。如果仍然发生，你就知道它在保留的部分中。

不要胡乱移除区域，而是使用分而治之。使用二分搜索算法来集中搜索。第一次尝试移除大约一半的代码。确定缺陷所在的半部分，然后划分那一部分。再一次，确定哪一半包含缺陷，再把那一部分对半砍。继续直到找到缺陷。

如果你使用许多小程序，你就可以通过注释掉对例程的调用来切出代码部分。否则，你可以使用注释或预处理器命令来移除代码。

如果你使用的是调试器，你不一定需要移除代码部分。你可以在程序中途设置一个断点，通过这种方式检查缺陷。如果你的调试器允许你跳过对例程的调用，通过跳过某些例程的执行并看看错误是否仍然发生来消除嫌疑。否则，调试器中的过程类似于物理移除程序部分的过程。

警惕以前有缺陷的类和例程。以前有缺陷的类很可能继续有缺陷。过去有问题的类比没有缺陷的类更有可能包含新的缺陷。重新检查容易出错的类和例程。

暂时放下问题。有时候你太专心以至于无法思考。你有多少次在喝咖啡的途中想通了问题？或者在午餐中间？或者在回家的路上？或者第二天早上洗澡的时候？如果你在调试且没有进展，一旦你尝试了所有选项，就让它休息一下。去散散步。做些别的事情。回家休息一天。让你的潜意识从问题中理出解决方案。

### 蛮力调试

蛮力是调试软件问题时经常被忽视的方法。通过"蛮力"，我指的是一种可能乏味、艰巨且耗时的技术，但保证能解决问题。哪些具体技术保证能解决问题是上下文相关的，但这里有一些通用候选方法：

对损坏的代码进行完整的设计和/或代码审查。
扔掉代码部分并从头重新设计/重新编码。
扔掉整个程序并从头重新设计/重新编码。
用完整的调试信息编译代码。
用最挑剔的警告级别编译代码并修复所有挑剔的编译器警告。
套上单元测试工具并隔离测试新代码。
创建一个自动化测试套件并整夜运行它。
在调试器中手动单步执行一个大循环，直到到达错误条件。
用print、display或其他日志语句检测代码。
用不同的编译器编译代码。
在不同的环境中编译和运行程序。
链接或运行代码时使用在代码使用不正确时会产生警告的特殊库或执行环境。
复制最终用户的完整机器配置。
以小增量集成新代码，在集成时完全测试每个部分。

为快速而肮脏的调试设置最长时间。对于每种蛮力技术，你的反应很可能是，"我做不到——工作太多了！"关键是，只有当它比你所说的"快速而肮脏的调试"花费更多时间时，它才太多工作。尝试快速猜测而不是系统地检测代码并让缺陷无处藏身总是很诱人的。我们每个人心中的赌徒宁愿使用可能有风险的尝试在五分钟内找到缺陷的方法，而不是保证在半小时内找到缺陷的方法。风险是，如果五分钟的方法不起作用，你就会固执己见。用"简单"的方法找到缺陷变成了原则问题，时间无情地流逝，一天、几天、几周、几个月……你有多少次花了两个小时调试只花了30分钟编写的代码？这是糟糕的劳动力分配，与其调试坏代码，不如重写代码。

当你决定争取快速胜利时，为尝试快速方法设定一个最长时间限制。如果你超过时间限制，就接受这个想法，即缺陷将比你最初认为的更难诊断，然后用艰难的方式找出它。这种方法允许你立即获得容易的缺陷，然后在稍长的时间后获得难的缺陷。

在开始调试一个困难的错误之前，制作一份蛮力技术的清单。问问你自己，"如果我在调试这个问题时陷入困境，有什么方法能保证我能够修复问题？"如果你能识别至少一种能修复问题的蛮力技术——包括重写相关代码——当有更快的替代方案时，你就不太可能浪费数小时或数天。

### 语法错误

语法错误问题正在走向毛茸茸的猛犸象和剑齿虎的道路。编译器在诊断消息方面越来越好，你不得不在Pascal列表中找到放错位置的分号的日子几乎结束了。以下是你可以用来加速这个濒危物种灭绝的指南：

不要相信编译器消息中的行号。当你的编译器报告一个神秘的语法错误时，立即查看错误之前和之后——编译器可能误解了问题，或者可能只是诊断信息不好。一旦你找到真正的缺陷，试着确定编译器为什么把消息放在错误的语句上。更好地理解你的编译器可以帮助你找到未来的缺陷。

不要相信编译器消息。编译器试图准确地告诉你到底哪里出了问题，但编译器是狡辩的小流氓，你经常需要在字里行间理解它们的真正意思。例如，在UNIX C中，你可能收到一条"浮点异常"的消息来处理整数除以0。对于C++的标准模板库，你会得到一对错误消息：第一条消息是STL使用中的真正错误；第二条消息是编译器的一条消息，说"错误消息太长无法打印；消息被截断"。你可能可以提出很多你自己的例子。

不要相信编译器的第二条消息。一些编译器在检测多个错误方面比其他的好。有些编译器在检测到第一个错误后变得过于兴奋和自信；它们会絮絮叨叨地说出几十条毫无意义的错误消息。其他编译器则更冷静，虽然在检测到错误时一定会有成就感，但它们忍住不喷出大量不准确的消息。当你的编译器产生一连串级联的错误消息时，不要担心你不能很快找到第二条或第三条错误消息的根源。修复第一个错误并重新编译。

分而治之。将程序分成几部分来帮助检测缺陷的想法对语法错误特别有效。如果你有一个麻烦的语法错误，移除部分代码并重新编译。你要么得不到错误（因为错误在你移除的部分），得到相同的错误（意味着你需要移除不同的部分），要么得到不同的错误（因为你将欺骗编译器产生一个更有意义的错误消息）。

## 修复缺陷

调试中最难的部分是找到缺陷。修复缺陷是容易的部分。但像许多简单任务一样，正因为它容易，所以特别容易出错。至少有一项研究发现，缺陷修正第一次尝试时有超过50%的可能是错误的（Yourdon 1986b）。以下是减少错误机会的一些指南：

在修复之前理解问题。"调试的魔鬼指南"是对的：在不完全理解问题的情况下修复问题是让你的生活困难和腐蚀程序质量的最佳方式。在修复问题之前，确保你理解它的核心。用应该重现错误的用例和不应该重现错误的用例来三角测量缺陷。继续做直到你足够好地理解问题，每次都能正确预测它的发生。

理解程序，而不仅仅是问题。如果你理解问题发生的上下文，你更有可能完全解决问题，而不仅仅是解决问题的一个方面。一项对短程序进行的研究发现，对自己程序行为有全局理解的程序员比只关注局部行为的程序员更有可能成功地修改它（程序），后者只在需要时才了解程序（Littman et al. 1986）。因为这项研究中的程序很小（280行），它并不能证明你在修复缺陷之前应该尝试完全理解一个50000行的程序。它确实建议你应该至少理解缺陷修正附近的代码——"附近"不是几行而是几百行。

确认缺陷诊断。在急于修复缺陷之前，确保你已经正确诊断了问题。花时间运行测试用例来证明你的假设并反驳相互竞争的假设。如果你只证明了错误可能是几个原因之一的结果，你还没有足够的证据来处理那一个原因；首先排除其他的。

放松。一个程序员正准备去滑雪旅行。他的产品已经准备好发货了，他已经迟到了，而且只剩下一个缺陷需要修正。他更改了源文件并将其签入版本控制。他没有重新编译程序，也没有验证更改是否正确。

永远不要站着调试。
—— Gerald Weinberg

事实上，更改是不正确的，他的经理很愤怒。他怎么可能在准备发货的产品中更改代码而不检查它？还有什么比这更糟糕的？难道这不是专业鲁莽的顶峰吗？

如果不是鲁莽的顶峰，它也接近了，而且很常见。急于解决问题是你能做的最浪费时间的事情之一。它导致仓促的判断、不完整的缺陷诊断和不完整的修正。痴心妄想会让你在不存在解决方案的地方看到解决方案。压力——通常是自我施加的——鼓励草率的试错解决方案，以及在未经验证的情况下假设解决方案有效。

与此形成鲜明对比的是，在Microsoft Windows 2000开发的最后几天，一位开发人员需要修复一个缺陷，这是创建候选发布版本之前剩下的最后一个缺陷。开发人员更改了代码，检查了他的修复，并在本地构建上测试了他的修复。但他在那一点上没有将修复签入版本控制。相反，他去打篮球了。他说，"我现在感觉压力太大，无法确定我已经考虑到了所有应该考虑的事情。我要让我的头脑清醒一个小时，然后我会回来签入代码——一旦我说服自己修复确实是正确的。"

放松足够长的时间以确保你的解决方案是正确的。不要被诱惑走捷径。它可能需要更多时间，但可能需要更少。如果没有别的，你会正确地解决问题，你的经理不会在你滑雪旅行时把你叫回来。

保存原始源代码。在开始修复缺陷之前，确保存档一个你以后可以返回的代码版本。在一组更改中忘记哪个更改是重要的很容易。如果你有原始源代码，至少你可以比较旧文件和新文件，看看更改在哪里。

for ( claimNumber = 0; claimNumber < numClaims[ client ]; claimNumber++ ) {

if ( client == 45 ) {

这些修复大多数时候不会起作用。这些问题看起来是初始化缺陷的结果。初始化缺陷，顾名思义，是不可预测的，所以client 45今天的sum偏差$3.45并不能告诉你明天的任何信息。它可能偏差$10,000.02，或者可能是正确的。这是初始化缺陷的本质。
它不可维护。当代码被特例化来规避错误时，特殊情况成为代码最突出的特征。$3.45不会总是$3.45，稍后会显示另一个错误。代码将再次被修改以处理新的特殊情况，而$3.45的特殊情况不会被删除。代码将变得越来越像藤壶一样布满特殊情况。最终藤壶太重，代码无法承受，代码将沉入海底——这是它合适的归宿。
它把电脑用于更适合手工完成的事情。电脑擅长可预测的、系统化的计算，但人类更擅长创造性地伪造数据。你用修正液和打字机处理输出比折腾代码更明智。

只为好的理由更改代码。与修复症状相关的是随机更改代码直到它似乎工作的技术。典型的推理过程是这样的："这个循环似乎包含一个缺陷。它可能是一个边界差一错误，所以我就在这里放一个-1试试。好吧，那不行，所以我就在里面放一个+1试试。好吧，那似乎可以了。我就说它修复了。"

尽管这种做法很流行，但它并不有效。随机更改代码就像转动Pontiac Aztek的轮胎来修复引擎问题一样。你没有学到任何东西；你只是在瞎搞。通过随机更改程序，你实际上是在说，"我不知道这里发生了什么，但我会试试这个更改，希望它能工作。"不要随机更改代码。那是伏都教编程。你让它越不同而没有理解它，你就越不相信它能正确工作。

在你做出更改之前，要确信它会起作用。对更改的错误应该让你惊讶。它应该引起自我怀疑、个人重新评估和深刻的灵魂探索。这种情况应该很少发生。

一次只做一个更改。更改本身已经够棘手的了。当一次做两个时，它们可能引入看起来像原始错误的新错误。然后你就处于尴尬的境地，不知道你是没有纠正错误，还是纠正了错误但引入了看起来类似的新错误，还是既没有纠正错误又引入了类似的新错误。保持简单：一次只做一个更改。

检查你的修复。亲自检查程序，让别人帮你检查，或者和另一个人一起检查。使用你用来诊断问题的相同的三角测量测试用例，确保问题的所有方面都已解决。如果你只解决了问题的一部分，你会发现你还有工作要做。

## 调试中的心理因素

调试像任何其他软件开发活动一样对智力要求很高。你的自我告诉你，你的代码是好的，没有缺陷，即使你已经看到它有一个。你必须精确思考——形成假设、收集数据、分析假设、系统地拒绝它们——这种正式性对许多人来说是不自然的。如果你既在构建代码又在调试它，你必须快速地在与设计相关的流畅、创造性思维和与调试相关的僵化、批判性思维之间切换。当你阅读代码时，你必须与代码的熟悉感作斗争，防止看到你期望看到的东西。

进一步阅读

有关调试中心理学问题的优秀讨论以及软件开发的许多其他领域，请参见《计算机编程心理学》（Weinberg 1998）。

### "心理定势"如何导致调试盲区

当你在程序中看到一个标记为Num的符号时，你看到了什么？你看到的是单词"Numb"的拼写错误？还是"Number"的缩写？最可能的是，你看到的是"Number"的缩写。这就是"心理定势"现象——看到你期望看到的东西。这个标志说什么？

在这个经典的谜题中，人们经常只看到一个"the"。人们看到他们期望看到的东西。考虑一下：

学习while循环的学生经常期望循环被持续评估；也就是说，他们期望一旦while条件变为false，循环就终止，而不是只在顶部或底部终止（Curtis et al. 1986）。他们期望while循环像自然语言中的"while"一样运作。
一个程序员无意中同时使用了变量SYSTSTS和SYSSTSTS，以为他使用的是单个变量。直到程序已经运行了数百次并且一本包含错误结果的书被写出来，他才发现问题（Weinberg 1998）。
一个程序员看这样的代码：
```cpp
if ( x < y )
   swap = x;
   x = y;
   y = swap;
```

有时看到的代码是这样的：
```cpp
if ( x < y ) {
   swap = x;
   x = y;
   y = swap;
}
```

人们期望一个新现象类似于他们之前见过的类似现象。他们期望一个新的控制结构与旧的结构相同；期望编程语言的while语句像真实生活中的"while"语句一样工作；期望变量名与之前相同。你看到你期望看到的东西，因此忽略了差异，比如前一句中单词"language"的拼写错误。

心理定势与调试有什么关系？首先，它强调了良好编程实践的重要性。良好的格式化、注释、变量名、例程名称和编程风格的其他元素有助于构建编程背景，使可能的缺陷表现为变化而凸显出来。

心理定势的第二个影响是在发现错误时选择要检查的程序部分。研究表明，调试最有效的程序员在调试时会心理上切掉程序的不相关部分（Basili, Selby, and Hutchens 1986）。一般来说，这种做法允许优秀的程序员缩小搜索范围，更快地找到缺陷。然而，有时候包含缺陷的程序部分被错误地切掉了。你花时间在一段代码中搜索缺陷，却忽略了包含缺陷的部分。你在岔路口走错了方向，需要退后一步才能继续前进。寻找缺陷的技巧中的一些建议旨在克服这种"调试盲区"。

### "心理距离"如何提供帮助

心理距离可以定义为两个项目可以区分的容易程度。如果你正在看一长串单词，并且被告知它们都与鸭子有关，你很容易把"Queck"误认为"Quack"，因为这两个词看起来很相似。单词之间的心理距离很小。你把"Tuack"误认为"Quack"的可能性要小得多，尽管差异也是只有一个字母。"Tuack"比"Queck"更不像"Quack"，因为单词的第一个字母比中间的字母更突出。

### 编译器警告消息

最简单和最有效的调试工具之一是你自己的编译器。

将编译器的警告级别设置为可能的最高、最挑剔级别，并修复它报告的错误。忽略编译器错误是草率的。把警告关掉以至于你甚至看不到它们更是草率。孩子有时认为如果他们闭上眼睛看不到你，你就已经离开了。把编译器上的开关设置为关闭警告只是意味着你看不到错误。它并不会让它们消失，就像闭上眼睛不会让大人消失一样。

假设编写编译器的人对你的语言了解得比你多得多。如果他们在警告你某事，通常意味着你有机会了解关于你的语言的新知识。花点力气去理解警告真正意味着什么。

把警告当作错误对待。一些编译器允许你把警告当作错误。使用这个特性的一个原因是它提高了警告的表面重要性。就像把表调快五分钟会让你以为时间已经过了五分钟一样，把编译器设置为把警告当作错误会让你更认真地对待它们。把警告当作错误的另一个原因是它们经常影响你的程序如何编译。当你编译和链接一个程序时，警告通常不会阻止程序链接，但错误通常会。如果你想在链接之前检查警告，设置编译器开关把警告当作错误。

为编译时设置启动项目范围的标准。设定一个标准，要求团队中的每个人都使用相同的编译器设置编译代码。否则，当你尝试集成由不同人用不同设置编译的代码时，你会得到大量错误消息和集成的噩梦。如果你使用项目标准的make文件或构建脚本，这很容易执行。

### 调试器

商用调试器多年来稳步发展，今天可用的功能可以改变你的编程方式。好的调试器允许你设置断点，在执行到达特定行时中断，或者在它第n次到达特定行时中断，或者当全局变量改变时，或者当变量被赋予特定值时。它们允许你逐行单步执行代码，单步跳过或跳过例程。它们允许程序向后执行，逐步回到缺陷起源的点。它们允许你记录特定语句的执行——类似于在程序中散布"I'm here!"打印语句。

好的调试器允许完全检查数据，包括结构化和动态分配的数据。它们使查看指针链表或动态分配数组的内容变得容易。它们对用户定义的数据类型很智能。它们允许你对数据做出特别查询、分配新值并继续程序执行。

你可以查看高级语言或编译器生成的汇编语言。如果你使用多种语言，调试器会自动为每段代码显示正确的语言。你可以查看例程的调用链并快速查看任何例程的源代码。你可以在调试器环境中更改程序的参数。

当今最好的调试器还会记住每个单独程序的调试参数（断点、正在监视的变量等），这样你就不必为你调试的每个程序重新创建它们。

系统调试器在系统级别而不是应用程序级别运行，这样它们就不会干扰被调试程序的执行。当你要调试对时序或可用内存量敏感的的程序时，它们是必不可少的。

考虑到现代调试器提供的巨大功能，你可能会惊讶有人会批评它们。但计算机科学中最受尊敬的一些人建议不要使用它们。他们建议使用你的大脑，完全避免调试工具。他们的论点是调试工具是拐杖，通过思考问题比依赖工具能更快、更准确地发现问题。他们认为，你应该而不是调试器，应该在精神上执行程序来清除缺陷。

交互式调试器是不需要的东西的一个杰出例子——它鼓励试错式黑客攻击而不是系统性设计，也隐藏了勉强胜任精密编程的边缘人物。
—— Harlan Mills

不管经验证据如何，反调试器的基本论点并不成立。工具可以被滥用这一事实并不意味着它应该被拒绝。你不会仅仅因为可能过量服用就不服用阿司匹林。你不会仅仅因为可能割伤自己就不用电动割草机割草。任何其他强大的工具都可以被使用或滥用，调试器也不例外。

调试器不是好思考的替代品。但有时候，好思考也不是好调试器的替代品。最有效的组合是好思考加好调试器。

## 关键要点

调试是软件开发中成败攸关的方面。最好的方法是使用本书中描述的其他技术来首先避免缺陷。然而，提高你的调试技能仍然是值得的，因为好的和差的调试性能之间的差异至少是10比1。
系统化的寻找和修复错误方法对成功至关重要。集中你的调试，使每次测试都让你向前迈出一步。使用调试的科学方法。
在修复程序之前理解根本问题。对错误来源和随机修正的随机猜测会让程序处于比开始时更糟糕的状态。
将编译器警告设置为可能的最高级别，并修复它报告的错误。如果你忽略明显的错误，就很难修复微妙的错误。
调试工具是软件开发的强大辅助。找到它们并使用它们，并记住同时使用你的大脑。
