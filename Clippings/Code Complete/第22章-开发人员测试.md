测试是最受欢迎的质量改进活动——一种被丰富的工业和学术研究以及商业经验支持的做法。软件以多种方式测试，其中一些通常由开发人员执行，一些更常见地由专业测试人员执行：

**单元测试** 是由单个程序员或程序员团队编写的完整类、例程或小程序的执行，与更完整的系统隔离测试。

**组件测试** 是涉及多个程序员或编程团队工作的类、包、小程序或其他程序元素的执行，与更完整的系统隔离测试。

**集成测试** 是由多个程序员或编程团队创建的类、包、组件或子系统的组合执行。这种测试通常在有两个类要测试时就开始，一直持续到整个系统完成。

**回归测试** 是重复先前执行的测试用例，目的是在以前通过相同测试集的软件中发现缺陷。

**系统测试** 是软件在其最终配置中的执行，包括与其他软件和硬件系统的集成。它测试安全性、性能、资源损失、时序问题和其他无法在较低集成级别测试的问题。

在本章中，"测试"指的是开发人员进行的测试，这通常包括单元测试、组件测试和集成测试，但有时也可以包括回归测试和系统测试。由专业测试人员执行且开发人员很少执行的众多其他类型的测试，包括 beta 测试、客户验收测试、性能测试、配置测试、平台测试、压力测试、可用性测试等。这些类型的测试在本章中不再进一步讨论。

测试通常分为两大类：黑盒测试和白盒测试。"黑盒测试"指的是测试人员看不到被测试项目内部工作的测试。当测试你编写的代码时，这显然不适用！"白盒测试"指的是测试人员了解被测试项目内部工作的测试。这是你作为开发人员用来测试自己的代码的测试类型。黑盒和白盒测试都有优点和缺点；本章专注于白盒测试，因为这是开发人员执行的测试类型。

一些程序员交替使用"测试"和"调试"这两个术语，但仔细的程序员区分这两种活动。测试是检测错误的一种手段。调试是诊断和纠正已检测到的错误的根本原因的一种手段。本章专门处理错误检测。错误纠正将在第23章详细讨论。

测试这个主题比构建期间的测试主题要大得多。系统测试、压力测试、黑盒测试和其他测试专家的主题在本章末尾的"额外资源"部分讨论。

## 开发人员测试在软件质量中的作用

测试是任何软件质量计划的重要组成部分，在许多情况下它是唯一的部分。这是不幸的，因为各种形式的协作开发实践已被证明比测试发现更高比例的错误，而且每次发现错误的成本不到测试的一半（Card 1987, Russell 1991, Kaplan 1995）。单独的测试步骤（单元测试、组件测试和集成测试）通常发现的错误不到 50%。测试步骤的组合通常发现不到 60% 的错误（Jones 1998）。

程序不像人们获得细菌那样获得错误——通过在有错误的程序周围闲逛。程序员必须插入它们。
— Harlan Mills

如果你要在"Sesame Street"上列出一些软件开发活动并问："这些中哪一个不像其他的？"答案是"测试"。测试是大多数开发人员难以接受的一种活动，原因有几个：

**测试的目标与其他开发活动的目标背道而驰**。目标是发现错误。成功的测试是破坏软件的测试。每个其他开发活动的目标是防止错误并保持软件不崩溃。

**测试永远不能完全证明错误的缺失**。如果你已经广泛测试并发现了数千个错误，这是否意味着你发现了所有错误还是有数千个要发现？错误的缺失可能意味着无效或不完整的测试用例，就像可能意味着完美的软件一样容易。

**仅靠测试不能提高软件质量**。测试结果是质量的指标，但它们本身并不能改进它。试图通过增加测试量来提高软件质量就像试图通过更频繁地称重来减肥。你在踏上秤之前吃的东西决定了你会有多重，你使用的软件开发技术决定测试会发现多少错误。如果你想减肥，不要买新秤；改变你的饮食。如果你想改进你的软件，不要只测试更多；要更好地开发。

**测试要求你假设你会在代码中发现错误**。如果你假设你不会，你可能不会，但只是因为你建立了一个自我实现的预言。如果你执行程序希望它不会有任何错误，就太容易忽略你发现的错误。在一项已成为经典的研究中，Glenford Myers 让一组有经验的程序员测试一个包含 15 个已知缺陷的程序。普通程序员只发现了 15 个错误中的 5 个。最好的只发现了 9 个。未检测到错误的主要来源是错误输出没有被足够仔细地检查。错误是可见的，但程序员没有注意到它们（Myers 1978）。

**你必须希望在你的代码中发现错误**。这样的希望似乎是一种不自然的行为，但你应该希望是你发现了错误而不是别人。

一个关键问题是：在一个典型项目上应该花多少时间在开发人员测试上？通常引用的所有测试的数字是项目时间的 50%，但这是误导性的。首先，这个数字结合了测试和调试；仅测试需要更少的时间。其次，这个数字代表的是通常花费的时间而不是应该花费的时间。第三，这个数字包括独立测试和开发人员测试。

如图 22-1 所示，根据项目的大小和复杂性，开发人员测试可能应该占用总项目时间的 8% 到 25%。这与已报告的大部分数据一致。

> **图 22-1**。随着项目规模的增加，开发人员测试占总开发时间的比例较小。程序规模的影响在第27章中有更详细的描述。

第二个问题是：你用开发人员测试的结果做什么？最直接的是，你可以用结果来评估正在开发产品的可靠性。即使你从不纠正测试发现的缺陷，测试也描述了软件的可靠性。另一个用途是它们可以而且通常确实指导对软件的纠正。最后，随着时间的推移，通过测试发现的缺陷记录有助于揭示最常见的错误类型。你可以用这些信息选择适当的培训课程、指导未来的技术评审活动，以及设计未来的测试用例。

## 构建期间的测试

测试的大千世界有时忽略了本章的主题："白盒"或"玻璃盒"测试。你通常希望设计一个类作为一个黑盒——类的用户不需要看接口之外就能知道类做什么。然而，在测试类时，将其视为玻璃盒是有优势的，可以看到类的内部源代码以及其输入和输出。如果你知道盒子里有什么，你可以更彻底地测试类。当然，你在编写类时也有相同的盲点在测试它时，所以黑盒测试也有优势。

在构建期间，你通常编写一个例程或类，在心里检查它，然后评审或测试它。无论你的集成或系统测试策略是什么，在将每个单元与任何其他单元组合之前，你应该彻底测试它。如果你正在编写几个例程，你应该一次测试一个。例程单独测试不一定更容易，但调试起来更容易。如果你一次把几个未经测试的例程扔在一起并发现错误，几个例程中的任何一个都可能是罪魁祸首。如果你一次一个地将一个新例程添加到先前测试的例程集合中，你就知道任何新错误都是新例程或与新例程交互的结果。调试工作更容易。

协作构建实践有许多测试无法匹配的优点。但测试的部分问题在于测试通常没有尽可能好地执行。开发人员可以执行数百次测试，但仍然只达到部分代码覆盖率。良好的测试覆盖率感觉并不意味着实际的测试覆盖率足够。理解基本测试概念可以支持更好的测试并提高测试的有效性。

## 开发人员测试的推荐方法

开发人员测试的系统方法可以最大限度地提高你以最少努力检测各种错误的能力。一定要覆盖这个领域：

- 测试每个相关需求以确保需求已被实现。在需求阶段或尽可能早地计划此步骤的测试用例——最好在你开始编写要测试的单元之前。考虑测试需求中常见的遗漏。安全级别、存储、安装程序和系统可靠性都是测试的好目标，在需求时经常被忽视。
- 测试每个相关设计关注点以确保设计已被实现。在设计阶段或尽可能早地计划此步骤的测试用例——在你开始详细编码要测试的例程或类之前。
- 使用"基础测试"添加详细测试用例到测试需求和设计的那些用例。添加数据流测试，然后添加需要全面练习代码的其余测试用例。至少，你应该测试每一行代码。基础测试和数据流测试在本章后面描述。
- 使用一个检查表列出你到目前为止在项目上或以前项目上所犯的错误类型。

与产品一起设计测试用例。这有助于避免需求和设计中的错误，这些往往比编码错误更昂贵。计划尽早测试和发现缺陷，因为早期修复缺陷更便宜。

## 先测试还是后测试？

开发人员有时想知道是在编写代码之后还是之前编写测试用例更好（Beck 2003）。缺陷成本增加图表——见图 3-1——表明先编写测试用例将最小化将缺陷插入代码与检测和移除缺陷之间的时间量。这被证明是先编写测试用例的众多原因之一：

- 在编写代码之前编写测试用例并不会比在代码之后编写测试用例花费更多努力；它只是重新排序了测试用例编写活动。
- 当你先编写测试用例时，你可以更早发现缺陷并且更容易纠正它们。
- 先编写测试用例迫使你在编写代码之前至少稍微思考需求和设计，这往往会产生更好的代码。
- 先编写测试用例会在代码编写之前尽早暴露需求问题，因为为一个糟糕的需求编写测试用例是困难的。
- 如果你保存你的测试用例（你应该这样做），你仍然可以在后测试之外添加先测试。

总的来说，我认为测试优先编程是过去十年出现的最有价值的软件实践之一，是一个好的通用方法。但这并不是测试的万能药，因为它受到开发人员测试的一般限制的约束，接下来将描述这些限制。

## 开发人员测试的限制

注意开发人员测试的以下限制：

**开发人员测试往往是"干净测试"**。开发人员倾向于测试代码是否工作（干净测试），而不是测试代码以各种方式崩溃（脏测试）。不成熟的测试组织往往对每个脏测试有大约五个干净测试。成熟的测试组织对每个干净测试有大约五个脏测试。这个比例不会通过减少干净测试来逆转；它是通过创建 25 倍多的脏测试来完成的（Boris Beizer in Johnson 1994）。

**开发人员测试往往对测试覆盖率有乐观看法**。普通程序员认为他们达到了 95% 的测试覆盖率，但通常他们最多达到约 80% 的测试覆盖率，最坏情况下约 30%，平均约 50-60%（Boris Beizer in Johnson 1994）。

**开发人员测试往往跳过更复杂的测试覆盖率类型**。大多数开发人员认为称为"100% 语句覆盖率"的测试覆盖率是足够的。这是一个好的开始，但远远不够。更好的覆盖率标准是满足称为"100% 分支覆盖率"的标准，即每个谓词项至少测试一个真值和一个假值。测试技巧集合提供了如何完成此操作的更多详细信息。

这些要点都不会降低开发人员测试的价值，但它们确实有助于将开发人员测试置于适当的视角。尽管开发人员测试有价值，但它本身不足以提供充分的质量保证，应该用其他实践补充，包括独立测试和协作构建技术。

## 测试技巧集合

为什么不可能通过测试来证明程序是正确的？要使用测试来证明程序工作，你必须测试程序的每个可想象的输入值和输入值的每个可想象的组合。即使对于简单的程序，这样的任务也会变得非常禁止。例如，假设你有一个接受姓名、地址和电话号码并将其存储在文件中的程序。这当然是一个简单的程序，比任何你真正担心的正确性问题都简单得多。再假设每个可能的姓名和地址长度为 20 个字符，每个可以使用 26 个可能的字符。这将是可能的输入数量：

```
姓名：26^20（20个字符，每个有26种可能选择）
地址：26^20（20个字符，每个有26种可能选择）
电话号码：10^10（10个数字，每个有10种可能选择）

总可能性 = 26^20 × 26^20 × 10^10 ≈ 10^66
```

即使有这相对较少的输入，你有 66 个零前有一个的可能测试用例。为了正确看待这一点，如果诺亚离开方舟并开始以每秒一万亿个测试用例的速度测试这个程序，他将远远不到今天完成 1% 的进度。显然，如果你添加更现实的输入数据量，彻底测试所有可能性的任务将变得更加不可能。

## 不完整测试

由于穷举测试是不可能的，实际上测试的艺术是挑选最有可能发现错误的测试用例。在 10^66 个可能的测试用例中，只有少数可能揭示其他测试用例没有发现的错误。你需要专注于挑选一些告诉你不同内容的测试用例，而不是一遍又一遍告诉你相同内容的测试用例。

## 结构化基础测试

尽管名字有些古怪，结构化基础测试是一个相当简单的概念。这个想法是你需要至少测试程序中的每个语句。如果语句是一个逻辑语句——例如 if 或 while——你需要根据 if 或 while 内部的表达式有多复杂来改变测试，以确保语句被完全测试。确保你覆盖所有基础的最简单方法是计算通过程序的路径数，然后开发将练习程序每个路径所需的最小测试用例数。

你可能听说过"代码覆盖率"测试或"逻辑覆盖率"测试。它们是你测试程序所有路径的方法。由于它们覆盖所有路径，它们类似于结构化基础测试，但它们不包含用最小测试用例集覆盖所有路径的想法。如果你使用代码覆盖率或逻辑覆盖率测试，你可能会创建比你用结构化基础测试覆盖相同逻辑所需的更多的测试用例。

交叉参考

此过程类似于在控制结构和复杂度中的"如何测量复杂度"中测量复杂度的过程。

你可以用这种直接方式计算基础测试所需的最小用例数：

1. 从例程本身的直线路径开始加 1。
2. 为每个以下关键字或其等价物加 1：`if`、`while`、`repeat`、`for`、`and`、`or`。
3. 为 case 语句中的每个 case 加 1。如果 case 语句没有 default case，再加 1。

这里有一个例子：

示例 22-1. 计算 Java 程序路径数量的简单示例

```java
Statement1;       <-- 1
Statement2;
if ( x < 10 ) {       <-- 2
   Statement3;
}
Statement4;
```

(1)计算"1"作为例程本身。

(2)计算"2"作为 if。

在这个例子中，你从一开始，数一次 if，总计二。这意味着你需要至少两个测试用例来覆盖程序的所有路径。在本例中，你需要以下测试用例：

- 由 if 控制的语句被执行（x < 10）。
- 由 if 控制的语句未被执行（x >= 10）。

## 数据流测试

考虑到上一节和这一节一起为你提供了另一个说明控制流和数据流在计算机编程中同等重要的例子。

数据流测试基于这样的想法：数据使用至少与控制流一样容易出错。Boris Beizer 声称至少一半的代码由数据声明和初始化组成（Beizer 1990）。

数据可以处于三种状态之一：

**定义** 数据已被初始化，但尚未使用。

**使用** 数据已被用于计算，作为例程的参数，或用于其他事情。

**销毁** 数据曾经被定义，但以某种方式被取消定义。例如，如果数据是一个指针，也许指针已被释放。如果它是 for 循环索引，也许程序已超出循环，而编程语言没有定义循环外 for 循环索引的值。如果它是指向文件中记录的指针，也许文件已关闭，记录指针不再有效。

除了"定义"、"使用"和"销毁"这些术语外，用描述在做什么之前或之后立即进入或退出例程的术语也很方便：

**进入** 控制流在变量被操作之前立即进入例程。例如，一个工作变量在例程顶部被初始化。

**退出** 控制流在变量被操作之后立即离开例程。例如，在例程末尾将返回值分配给状态变量。

---

### 数据状态的组合

数据状态的正常组合是变量被定义，使用一次或多次，也许被销毁。要怀疑地查看以下模式：

**定义-定义** 如果你必须定义变量两次才能值才有效，你不需要更好的程序，你需要更好的计算机！这是浪费和容易出错的，即使实际上不是错误的。

**定义-退出** 如果变量是局部变量，定义它并在不使用的情况下退出是没有意义的。如果它是例程参数或全局变量，可能没问题。

**定义-销毁** 定义一个变量然后销毁它表明要么变量是多余的，要么应该使用变量的代码缺失。

**进入-销毁** 如果变量是局部变量，这是一个问题。如果它没有被定义或使用，它就不需要被销毁。另一方面，如果是例程参数或全局变量，只要变量在销毁之前在其他地方被定义，这个模式就没问题。

**进入-使用** 同样，如果是局部变量，这是一个问题。变量需要在使用之前被定义。另一方面，如果是例程参数或全局变量，只要变量在使用之前在其他地方被定义，这个模式就没问题。

**销毁-销毁** 变量不应该需要被销毁两次。变量不会复活。复活的变量表明编程草率。双重销毁对指针也是致命的——悬挂机器的最佳方式之一是两次销毁（释放）一个指针。

**销毁-使用** 在变量被销毁后使用它是逻辑错误。如果代码无论如何似乎都工作（例如，一个仍然指向已被释放的内存的指针），那是一个意外，墨菲定律说代码会在造成最大损害的时候停止工作。

**使用-定义** 使用然后定义一个变量可能或可能不是问题，取决于变量在使用之前是否也被定义。当然，如果你看到一个使用-定义模式，值得检查是否有先前的定义。

在测试开始之前检查这些异常数据状态序列。在检查了异常序列之后，编写数据流测试用例的关键是练习所有可能的定义-使用路径。你可以用不同程度的彻底性做到这一点：

**所有定义**。测试每个变量的每个定义——即任何变量接收值的每个地方。这是一个弱的策略，因为如果你试图练习每一行代码，你会默认这样做。

**所有定义-使用组合**。测试在某个地方定义变量并在另一个地方使用的每个组合。这是一个比测试所有定义更强的策略，因为仅仅执行每一行代码并不能保证每个定义-使用组合都被测试。

这里有一个例子：

示例 22-3. 要测试其数据流的 Java 程序示例

```java
if ( Condition 1 ) {
   x = a;
}
else {
   x = b;
}
if ( Condition 2 ) {
   y = x + 1;
}
else {
   y = x - 1;
}
```

要覆盖程序中的每个路径，你需要 Condition 1 为真和为假各一个测试用例。你还需要 Condition 2 为真和为假各一个测试用例。这可以通过两个测试用例处理：用例 1（Condition 1=True, Condition 2=True）和用例 2（Condition 1=False, Condition 2=False）。这两个用例是你进行结构化基础测试所需的全部。它们也是练习每个定义变量的代码行所需的全部；它们自动为你提供弱形式的数据流测试。

然而，要覆盖每个定义-使用组合，你需要添加更多用例。现在你有用例，其中 Condition 1 和 Condition 2 同时为真，Condition 1 和 Condition 2 同时为假：

```
x = a
...
y = x + 1
```

和

```
x = b
...
y = x - 1
```

但你还需要两个更多用例来测试每个定义-使用组合：(1) x = a 然后 y = x - 1 和 (2) x = b 然后 y = x + 1。在本例中，你可以通过添加两个更多用例来获得这些组合：用例 3（Condition 1=True, Condition 2=False）和用例 4（Condition 1=False, Condition 2=True）。

开发测试用例的好方法是首先从结构化基础测试开始，它给你一些（如果不是全部）定义-使用数据流。然后添加你仍然需要的用例，以获得完整的定义-使用数据流测试用例集。

## 好数据类

当你试图在程序中找到错误时，很容易忽视名义情况可能包含错误的事实。通常基础测试部分描述的名义情况代表一类好数据。以下是值得检查的其他类型的好数据。检查这些数据类型中的每一个都可以揭示错误，取决于被测试的项目。

## 哪些类包含最多的错误？

自然会假设缺陷均匀分布在你的源代码中。如果你平均每 1000 行代码有 10 个缺陷，你可能假设在一个包含 100 行代码的类中会有一个缺陷。这是一个自然的假设，但它是错误的。

Capers Jones 报告说，IBM 的一项集中质量改进计划识别出 IMS 系统中 425 个类中的 31 个是容易出错的。31 个类被修复或完全重新开发，在不到一年的时间里，客户报告的 IMS 缺陷减少了十比一。总维护成本降低了约 45%。客户满意度从"不可接受"提高到"良好"（Jones 2000）。

大多数错误往往集中在几个高度缺陷的例程中。以下是错误和代码之间的一般关系：

- 80% 的错误发现在 20% 的项目类或例程中（Endres 1975, Gremillion 1984, Boehm 1987b, Shull et al 2002）。
- 50% 的错误发现在 5% 的项目类中（Jones 2000）。

这些关系在你认识到几个推论之前可能看起来不那么重要。首先，20% 的项目例程贡献了 80% 的开发成本（Boehm 1987b）。这不一定意味着成本最多的 20% 与缺陷最多的 20% 相同，但这是相当有暗示性的。

其次，无论高度缺陷例程贡献的确切成本比例是多少，高度缺陷例程都是极其昂贵的。在 1960 年代的一项经典研究中，IBM 对其 OS/360 操作系统进行了分析，发现错误并非均匀分布在所有例程中，而是集中在一小部分中。那些容易出错的例程被发现是"编程中最昂贵的实体"（Jones 1986a）。它们包含高达每 1000 行代码 50 个缺陷，修复它们通常花费开发整个系统成本的 10 倍。（成本包括客户支持和现场维护。）

第三，高成本例程对开发的含义是清楚的。正如古老的表达所说，"时间就是金钱。"推论是"金钱就是时间"，如果你能通过避免麻烦的例程削减近 80% 的成本，你也可以削减相当大一部分进度。这是软件质量一般原则的清晰例证：提高质量可以改善开发进度并降低开发成本。

交叉参考

容易包含大量错误的另一类例程是过度复杂的例程类。有关识别和简化例程的详细信息，请参见控制结构和复杂度中的"减少复杂度的通用指导原则"。

第四，避免麻烦例程对维护的含义同样清楚。维护活动应集中在识别、重新设计和从头重写那些已被识别为容易出错的例程。在前面提到的 IMS 项目中，IMS 版本的 productivity 在替换容易出错的类后提高了约 15%（Jones 2000）。

## 按分类的错误

几位研究人员试图按类型对错误进行分类并确定每种错误发生的程度。每个程序员都有一个特别麻烦的错误列表：差一错误、忘记重新初始化循环变量等等。本书中提供的检查表提供了更多详细信息。

交叉参考

有关书中所有检查表的列表，请参见目录后面的列表。

Boris Beizer 组合了来自几项研究的数据，得出了异常详细的错误分类法（Beizer 1990）。以下是他的结果的摘要：

| 错误类型 | 百分比 |
|---------|-------|
| 结构 | 25.18% |
| 数据 | 22.44% |
| 实现的功能 | 16.19% |
| 构建 | 9.88% |
| 集成 | 8.98% |
| 功能需求 | 8.12% |
| 测试定义或执行 | 2.76% |
| 系统、软件架构 | 1.74% |
| 未指定 | 4.71% |

Beizer 将他的结果报告到精确的两位小数，但对错误类型的研究通常没有结论性。不同的研究报告了截然不同的错误类型，报告类似错误类型的研究得出了截然不同的结果，结果相差 50% 而不是百分点。

鉴于报告中广泛的差异，组合多项研究的结果（就像 Beizer 所做的那样）可能不会产生有意义的数据。但即使数据没有结论性，其中一些也是有暗示性的。以下是可以从数据中得出的一些建议：

**大多数错误的作用域相当有限**。一项研究发现 85% 的错误可以在不修改超过一个例程的情况下纠正（Endres 1975）。

**许多错误在构建领域之外**。进行 97 次访谈的一系列研究人员发现，错误最常见的三个来源是薄弱的应用程序领域知识、波动和冲突的需求，以及沟通和协调故障（Curtis, Krasner, and Iscoe 1988）。

**大多数构建错误是程序员的错**。多年前进行的一对研究发现，在报告的总数错误中，约 95% 是由程序员造成的，2% 由系统软件（编译器和操作系统）造成，2% 由其他软件造成，1% 由硬件造成（Brown and Sampson 1973, Ostrand and Weyuker 1984）。系统软件和开发工具今天被比 1970 年代和 1980 年代多得多的使用，所以我的最佳猜测是，今天更高比例的错误是程序员的错。

如果你看到蹄印，想想马——而不是斑马。操作系统可能没有坏。数据库可能没问题。
— Andy Hunt Dave Thomas

**文书错误（打字错误）是一个令人惊讶的常见问题来源**。一项研究发现 36% 的所有构建错误是文书错误（Weiss 1975）。1987 年对近 300 万行飞行动力学软件的研究发现 18% 的所有错误是文书错误（Card 1987）。另一项研究发现 4% 的所有错误是消息中的拼写错误（Endres 1975）。在我的一个程序中，一位同事发现了几处拼写错误，只是通过将可执行文件中的所有字符串通过拼写检查器运行。注意细节很重要。如果你不相信，考虑有史以来最昂贵的三个软件错误——成本 16 亿美元、9 亿美元和 2.45 亿美元——涉及在以前正确的程序中将单个字符更改为以前正确的程序（Weinberg 1983）。

**误解设计是程序员错误研究中反复出现的主题**。Beizer 的汇编研究（无论其价值如何）发现 16% 的错误源于设计的误解（Beizer 1990）。另一项研究发现 19% 的错误是由于误解设计（Weiss 1975）。值得花你需要的时间彻底理解设计。这样的时间不会产生直接红利——你不一定看起来在工作——但会在项目生命周期中收回成本。

**大多数错误很容易修复**。约 85% 的错误可以在不到几小时内修复。约 15% 可以用几小时到几天修复。约 1% 需要更长时间（Weiss 1975, Ostrand and Weyuker 1984, Grady 1992）。这个结果支持了 Barry Boehm 的观察，即约 20% 的错误需要约 80% 的资源来修复（Boehm 1987b）。通过在上游进行需求和设计评审，尽可能避免许多困难的错误。以尽可能高效的方式处理大量小错误。

衡量你自己的组织与错误的经历是个好主意。本节引用的结果多样性表明，不同组织的人有截然不同的经历。这使得将其他组织的经历应用到你自己的组织变得困难。有些结果与常识相反；你可能需要用其他工具补充你的直觉。一个好的第一步是开始衡量你的开发过程，这样你就知道问题在哪里。

## 由故障构建导致的错误比例

如果分类错误的数据没有结论性，将错误归因于各种开发活动的数据也有很多没有结论性。一个确定的是，构建总是导致大量错误。有时人们争论说构建导致的错误比需求或设计导致的错误便宜。修复单个构建错误可能更便宜，但证据不支持关于总成本的这种说法。

以下是我的结论：

- **在小项目上，构建缺陷构成了所有错误的绝大部分**。在一项对小项目（1000 行代码）编码错误的研究中，75% 的缺陷来自编码，而 10% 来自需求，15% 来自设计（Jones 1986a）。这种错误细分似乎代表了许多小项目。
- **无论项目规模如何，构建缺陷至少占所有缺陷的 35%**。虽然大型项目上构建缺陷的比例较小，但它们仍然至少占所有缺陷的 35%（Beizer 1990, Jones 2000）。一些研究报告即使在非常大的项目上也高达 75%（Grady 1987）。一般来说，应用程序领域被理解得越好，整体架构就越好。错误然后倾向于集中在详细设计和编码中（Basili and Perricone 1984）。
- **构建错误虽然比需求和设计错误便宜，但仍然昂贵**。对惠普两个非常大的项目的一项研究发现，平均构建缺陷修复成本是平均设计错误修复成本的 25-50%（Grady 1987）。当将更多数量的构建缺陷计入整体方程时，修复构建缺陷的总成本是设计缺陷成本的一到两倍。

## 你应该期望发现多少错误？

你应该期望发现的错误数量因你使用的开发过程的质量而异。以下是可能性范围：

- **行业平均经验**是交付软件每 1000 行代码 1-25 个错误。软件通常使用各种技术的组合开发（Boehm 1981, Gremillion 1984, Yourdon 1989a, Jones 1998, Jones 2000, Weber 2003）。错误只有十分之一的案例很少；错误多十倍的案例往往不被报告。（他们可能永远没有完成！）
- **微软应用程序部门**在内部测试期间经历每 1000 行代码 10-20 个缺陷，在发布产品中每 1000 行代码 0.5 个缺陷（Moore 1992）。实现这一水平使用的技术是第21章"其他类型的协作开发实践"中描述的代码阅读技术和独立测试的组合。
- **Harlan Mills 开创了"洁净室开发"**，这是一种能够达到每 1000 行代码仅 3 个缺陷（在内部测试期间）和每 1000 行代码 0.1 个缺陷（在发布产品中）的技术（Cobb and Mills 1990）。一些项目——例如，航天飞机软件——通过使用正式开发方法、同行评审和统计测试的系统达到了 50 万行代码中 0 个缺陷的水平（Fishman 1996）。

Watts Humphrey 报告说，使用团队软件过程（TSP）的团队达到了每 1000 行代码约 0.06 个缺陷的水平。TSP 专注于培训开发人员首先不要创建缺陷（Weber 2003）。

TSP 和洁净室项目的结果证实了软件质量一般原则的另一个版本：构建高质量软件比构建和修复低质量软件更便宜。对于完全检出的 80,000 行洁净室项目，productivity 是每个工作月 740 行代码。行业平均完全检出代码的 productivity 更接近每工作月 250-300 行，包括所有非编码开销（Cusumano et al 2003）。成本节约和 productivity 来自这样一个事实：TSP 或洁净室项目上几乎不花时间在调试上。不花时间在调试上？这真是一个值得追求的目标！

## 测试本身的错误

你可能有过这样的经历：软件被发现有错误。你对哪部分代码可能错误有几个直接猜测，但所有那些代码似乎都是正确的。你运行了几个更多的测试用例来尝试细化错误，但所有新的测试用例都产生了正确的结果。你花了好几个小时阅读和重读代码以及手工计算结果。它们都检查无误。再多几个小时后，某事让你重新检查测试数据。找到了！错误在测试数据中！花了数小时追踪测试数据中的错误而不是代码中的错误，这是多么愚蠢！

这是一个常见的经历。测试用例往往和被测试的代码一样有可能或更容易包含错误（Weiland 1983, Jones 1986a, Johnson 1994）。原因很容易找到——特别是当开发人员编写测试用例时。测试用例往往是在飞行中创建的，而不是通过仔细的设计和构建过程。它们通常被视为一次性测试，并以与一次性使用的东西相称的护理开发。

你可以做几件事来减少测试用例中的错误数量：

**检查你的工作**。和开发代码一样仔细地开发测试用例。这种护理肯定包括仔细检查你自己的测试。在调试器中逐行步进测试代码，就像你处理生产代码一样。走查和测试数据的检查是适当的。

**在开发软件时计划测试用例**。有效的测试计划应该从需求阶段开始，或者在你得到程序分配时就开始。这有助于避免基于错误假设的测试用例。

**保留你的测试用例**。花一些优质时间与你的测试用例在一起。为回归测试和版本 2 工作保存它们。如果你知道你将保留它们而不是扔掉它们，很容易证明麻烦是值得的。

**将单元测试插入测试框架**。首先为单元测试编写代码，但在完成每个测试时将它们集成到系统范围的测试框架（如 JUnit）中。拥有一个集成的测试框架可以防止刚才提到的扔掉测试用例的趋势。

## 构建脚手架以测试单个类

术语"脚手架"来自建筑施工。脚手架是为了让工人能够到达否则无法到达的建筑部分而建造的。软件脚手架是为了便于执行代码而单独构建的。

**进一步阅读**

有关脚手架的几个好例子，请参见 Jon Bentley 的文章"Programming Pearls, 2d ed."（2000）中的"A Small Matter of Programming"。

一种脚手架是一个被伪造使用的类，以便它可以被正在测试的另一个类使用。这样的类称为"模拟对象"或"存根对象"（Mackinnon, Freemand, and Craig 2000; Thomas and Hunt 2002）。类似的方法可以用于低级例程，称为"存根例程"。你可以或多或少地使模拟对象或存根例程变得现实，取决于你需要多少真实性。在这些情况下，脚手架可以：

- 立即返回控制，不采取任何行动。
- 测试输入到它的数据。
- 打印诊断消息，也许是输入参数的回显，或将消息记录到文件。
- 从交互式输入获取返回值。
- 无论输入是什么都返回一个标准答案。
- 燃烧分配给真实对象或例程的时钟周期数。
- 作为真实对象或例程的慢、胖、简单或不太准确的版本运行。

另一种脚手架是调用正在测试的真实例程的假例程。这称为"驱动程序"或有时称为"测试工具"。这个脚手架可以：

- 使用一组固定输入调用对象。
- 交互式提示输入并用它调用对象。
- 从命令行获取参数（在支持的操作系统中）并用参数调用对象。
- 从文件读取参数并用参数调用对象。
- 在对对象的多次调用中运行预定义的输入数据集。

最后一种脚手架是假文件，是真实事物的小版本，具有与全尺寸文件相同的组件类型。小的假文件提供了一些优势。因为它很小，你可以知道它的确切内容，并且可以合理地确信文件本身没有错误。而且因为你专门为测试创建它，你可以设计其内容，使得使用它的任何错误都是显眼的。

交叉参考

测试工具和调试工具之间的界限是模糊的。有关调试工具的详细信息，请参见调试工具——明显的和不那么明显的。

显然，构建脚手架需要一些工作，但如果在类中检测到错误，你可以重用脚手架。有许多工具存在以简化模拟对象和其他脚手架的创建。如果你使用脚手架，类也可以在不冒与其他类交互影响的风险的情况下进行测试。当涉及微妙算法时，脚手架特别有用。当被执行的代码嵌入其他代码中时，每次执行测试用例可能需要几分钟，很容易陷入困境。脚手架允许你直接执行代码。你花在构建脚手架上以执行深埋代码的几分钟可以节省数小时的调试时间。

你可以使用任何众多可用的测试框架为你的程序提供脚手架（JUnit、CppUnit、NUnit 等）。如果你的环境不被现有测试框架之一支持，你可以在一类中编写几个例程，并在文件中包含一个 main() 脚手架例程来测试类，即使被测试的例程不是打算独立存在的。main() 例程可以从命令行读取参数并将它们传递给正在测试的例程，以便你可以在将其与程序其余部分集成之前单独执行例程。当你集成代码时，将例程和练习它们的脚手架代码留在文件中，并使用预处理器命令或注释来停用脚手架代码。由于它被预处理掉了，它不影响可执行代码，而且它在文件底部，不会视觉上妨碍。留下它没有害处。如果你需要它，它就在那里，它不会消耗移除和归档它本来会花费的时间。

## 测试数据生成器

你也可以编写代码系统地练习程序的选定部分几年前，我开发了一个专有的加密算法并编写了一个使用它的文件加密程序。程序的目的是编码文件，使其只能用正确的密码解码。加密不仅仅是肤浅地改变文件；它改变了整个内容。程序能够正确解码文件是关键的，否则文件将被毁坏。

我设置了一个测试数据生成器，完全练习程序的加密和解密部分。它生成随机大小的随机字符文件，从 0K 到 500K。它生成随机长度（1 到 255）的随机字符密码。对于每个随机情况，它生成随机文件的两个副本，加密一个副本，重新初始化自己，解密副本，然后将解密的副本中的每个字节与未更改的副本进行比较。如果任何字节不同，生成器打印我需要重现错误的所有信息。

我加权测试用例偏向我的文件的平均长度 30K，这大大短于 500K 的最大长度。如果我没有将测试用例加权偏向较短的长度，文件长度将在 0K 和 500K 之间均匀分布。平均测试文件长度将是 250K。较短的平均长度意味着我可以在比均匀随机长度更多的情况下测试更多文件、密码、文件结束条件、奇数文件长度和其他可能产生错误的情况。

结果令人满意。在仅运行约 100 个测试用例后，我在程序中发现了两个错误。两者都源于可能在实践中永远不会出现的特殊情况，但它们仍然是错误，我很乐意发现它们。修复它们后，我运行程序数周，加密和解密超过 100,000 个文件而没有错误。鉴于我测试的文件内容、长度和密码范围，我可以自信地断言程序是正确的。

以下是这个故事的教训：

- 正确设计的随机数据生成器可以生成你不会想到的异常测试数据组合。
- 随机数据生成器可以比你更彻底地练习你的程序。
- 你可以随着时间的推移细化随机生成的测试用例，以便它们强调现实的输入范围。这将测试集中在用户最有可能使用的区域，在这些区域最大化可靠性。
- 模块化设计在测试期间有回报。我能够拉出加密和解密代码并在用户界面代码之外独立使用它，使编写测试驱动程序的工作变得简单。
- 如果代码必须更改，你可以重用测试驱动程序。一旦我纠正了两个早期错误，我就能够立即开始重新测试。

## 覆盖率监视器

cc2e.com/2282

Karl Wiegers 报告说，没有测量代码覆盖率的测试通常只练习约 50-60% 的代码（Wiegers 2002）。覆盖率监视器是一种跟踪被练习的代码和未被练习的代码的工具。覆盖率监视器对于系统测试特别有用，因为它告诉你一组测试用例是否完全练习了代码。如果你运行完整的测试用例集而覆盖率监视器指示一些代码仍未被执行，你就知道你需要更多测试。

## 数据记录器/日志

一些工具可以监视你的程序并在发生故障时收集关于程序状态的信息——类似于飞机用来诊断坠机结果的"黑匣子"。强大的日志记录有助于错误诊断，并支持软件发布后的有效服务。

你可以通过将重要事件记录到文件来构建自己的数据记录器。记录错误之前的系统状态和确切错误条件的详细信息。此功能可以编译到代码的开发版本中，并从发布版本中编译出来。或者，如果你使用自修剪存储和周到的错误消息放置和内容实现日志记录，你可以在发布版本中包含日志记录功能。

## 符号调试器

符号调试器是代码走查和检查的技术补充。调试器有能力逐行步进代码、跟踪变量值，并且总是以与计算机相同的方式解释代码。在调试器中步进一段代码并观察它工作的过程非常有价值。

交叉参考

调试器的可用性因技术环境的成熟度而异。有关此现象的更多信息，请参见技术波上的位置。

在调试器中走查代码在许多方面与让其他程序员在评审中步进你的代码的过程相同。你的同行和调试器都没有和你相同的盲点。调试器的额外好处是它不如团队评审那么劳动密集。在各种输入数据集下观看代码执行是对你已经实现了你想要的代码的好保证。

好的调试器甚至是一个很好的学习语言工具，因为你可以看到代码如何精确执行。你可以在高级语言代码视图和汇编代码视图之间切换，以查看高级代码如何被翻译成汇编。你可以观察寄存器和堆栈以查看参数如何传递。你可以看看编译器优化过的代码以查看执行的优化类型。这些好处都不太与调试器的预期用途——诊断已检测到的错误——有太大关系，但对调试器的创造性使用产生的收益远远超出其最初的任务。

## 系统扰动器

另一类测试支持工具旨在扰动系统。许多人都有关于程序 100 次中有 99 次工作的故事，但在第 100 次运行时使用相同数据失败。问题几乎总是一样：在某处未能初始化变量，而且通常很难重现，因为 100 次中有 99 次未初始化的变量恰好为 0。

这一类的测试支持工具具有各种能力：

**内存填充** 你想确保你没有未初始化的变量。一些工具在运行程序之前用任意值填充内存，以便未初始化的变量不会被意外设置为 0。在某些情况下，内存可能设置为特定值。例如，在 x86 处理器上，值 0xCC 是断点中断的机器语言代码。如果你用 0xCC 填充内存并且有一个错误导致你执行你不应该执行的东西，你将在调试器中遇到一个断点并检测到错误。

**内存摇动** 在多任务系统中，一些工具可以在程序运行时重新排列内存，以便你可以确保你没有编写任何依赖于数据在绝对位置而不是相对位置的代码。

**选择性内存故障** 内存驱动程序可以模拟程序可能运行内存不足的低内存条件，在内存请求上失败，在失败之前授予任意数量的内存请求，或者在授予一个之前在任意数量的请求上失败。这对于测试与动态分配内存一起工作的复杂程序特别有用。

**内存访问检查（边界检查）** 边界检查器观察指针操作以确保你的指针表现良好。这样的工具对于检测未初始化或悬空指针很有用。

## 改进你的测试

改进测试的步骤类似于改进任何其他过程的步骤。你必须准确知道过程做什么，这样你才能稍微改变它并观察变化的效果。当你观察到有积极影响的变化时，你修改过程使其变得更好。以下各节描述如何用测试做到这一点。

## 测试计划

有效测试的一个关键是从事项目开始就计划测试。将测试与设计或编码放在同等重要的位置意味着会为它分配时间，它会被视为重要，它将是一个高质量的过程。测试计划也是使测试过程可重复的因素之一。如果你不能重复它，你就不能改进它。

## 重新测试（回归测试）

假设你已经彻底测试了一个产品并发现了错误。假设然后产品在某个区域被更改，你想确保它仍然通过更改之前通过的所有测试——即更改没有引入任何新缺陷。设计来确保软件没有倒退或"回归"的测试称为"回归测试"。

除非你可以在更改后系统地重新测试，否则几乎不可能产生高质量的软件产品。如果在每次更改后运行不同的测试，你无法确定没有引入新缺陷。因此，回归测试必须每次运行相同的测试。有时随着产品成熟会添加新测试，但旧的测试也会保留。

---

## 自动化测试

管理回归测试的唯一实用方法是自动化它。人们会因为多次运行相同的测试并看到相同的测试结果而变得麻木。看到错误变得太容易，这违背了回归测试的目的。测试大师 Boris Beizer 报告说，手动测试中的错误率与被测试代码中的错误率相当。他估计在手动测试中，只有大约一半的测试被正确执行（Johnson 1994）。

自动化测试的好处包括：

- 自动化测试比手动测试出错的概率更低。
- 一旦你自动化了一个测试，它就可以在项目的剩余部分随时可用，只需很少的增量努力。
- 如果测试是自动化的，它们可以频繁运行以查看代码签入是否破坏了代码。测试自动化是密集测试实践（如每日构建和冒烟测试以及极限编程）基础的一部分。
- 自动化测试提高了在任何给定时刻尽早检测任何给定问题的机会，这往往会最小化诊断和纠正问题所需的工作。
- 自动化测试为大规模代码更改提供了安全网，因为它们增加了在修改期间快速检测缺陷的机会。
- 自动化测试在新的、不稳定的技术环境中特别有用，因为它们会及早发现环境中的变化。

交叉参考

有关技术成熟度与开发实践之间关系的更多信息，请参见技术波上的位置。

支持自动化测试的主要工具提供测试脚手架、生成输入、捕获输出以及将实际输出与预期输出进行比较。前一节讨论的各种工具将执行部分或全部这些功能。

## 保留测试记录

除了使测试过程可重复外，你还需要衡量项目，以便你可以确定更改是否改进或降低了它。以下是你可以收集来衡量项目的一些数据类型：

- 错误的行政描述（报告日期、报告人、标题或描述、构建编号、修复日期）
- 问题的完整描述
- 重复问题要采取的步骤
- 问题的建议变通方法
- 相关缺陷
- 问题的严重程度——例如，致命的、烦人的或 cosmetic
- 缺陷的来源：需求、设计、编码或测试
- 编码缺陷的子类：差一、错误赋值、错误数组索引、错误例程调用等
- 修复更改的类和例程
- 缺陷影响的代码行数
- 发现缺陷的小时数
- 修复缺陷的小时数

一旦你收集了数据，你可以计算一些数字以确定你的项目是变得更糟还是更健康：

- 每个类的缺陷数，从最差类到最优类排序，可能按类大小归一化
- 每个例程的缺陷数，从最差例程到最优例程排序，可能按例程大小归一化
- 每个发现缺陷的平均测试小时数
- 每个测试用例的平均发现缺陷数
- 每个修复缺陷的平均编程小时数
- 测试用例覆盖的代码百分比
- 每个严重性分类中的未完成缺陷数

## 测试用例检查表

- [ ] 每个适用于类或例程的需求是否有其自己的测试用例？
- [ ] 每个适用于类或例程的设计元素是否有其自己的测试用例？
- [ ] 是否每个代码行都至少有一个测试用例？是否已通过计算练习每个代码行所需的最少测试数来验证？
- [ ] 是否所有定义-使用数据流路径都至少有一个测试用例？
- [ ] 是否已检查可能不正确的异常数据流模式，如定义-定义、定义-退出和定义-销毁？
- [ ] 是否已使用常见错误列表编写测试用例来检测过去经常发生的错误？
- [ ] 是否已测试所有简单边界：最大值、最小值和差一边界？
- [ ] 是否已测试复合边界——即可能导致计算变量太小或太大的输入数据组合？
- [ ] 测试用例是否检查错误类型的数据——例如，工资单程序中员工人数为负数？
- [ ] 是否测试了典型的、中间道路的值？
- [ ] 是否测试了最小正常配置？
- [ ] 是否测试了最大正常配置？
- [ ] 是否测试了与旧数据的兼容性？是否测试了旧硬件、旧操作系统版本以及与其他软件旧版本的接口？
- [ ] 测试用例是否便于手工检查？

## 关键点

- 开发人员进行的测试是完整测试策略的关键部分。
- 独立测试也很重要，但不在本书范围内。
- 在代码之前编写测试用例与在代码之后编写测试用例花费的时间和精力相同，但它缩短了缺陷检测-调试-纠正循环。
- 即使考虑到可用的众多测试类型，测试只是一个好的软件质量计划的一部分。高质量开发方法，包括最小化需求和设计中的缺陷，至少和测试一样重要。协作构建实践在检测错误方面也至少和测试一样有效，而且这些实践检测不同类型的错误。
- 你可以通过使用基础测试、数据流分析、边界分析、坏数据类和好数据类来确定性地生成许多测试用例。你可以用错误猜测生成额外的测试用例。
- 错误倾向于聚集在少数容易出错的类和例程中。找到那些容易出错的代码，重新设计它，并重写它。
- 测试数据往往比被测试的代码有更高的错误密度。因为追踪此类错误是浪费时间而不改进代码，所以测试数据错误比编程错误更烦人。通过和代码一样仔细地开发你的测试来避免它们。
- 自动化测试 generally 有用，对于回归测试是必不可少的。
- 从长远来看，改进测试过程的最好方法是使其规律化、衡量它，并用你所学到的来改进它。
