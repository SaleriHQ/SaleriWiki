表驱动方法是一种允许你查找表中的信息而不是使用逻辑语句（if 和 case）来推断的方案。几乎任何你可以用逻辑语句选择的东西，你都可以用表来选择。在简单的情况下，逻辑语句更容易和更直接。随着逻辑链变得越来越复杂，表变得越来越有吸引力。

如果你已经熟悉表驱动方法，这一章可能只是一个复习。在这种情况下，你可能想检查直接访问表中的"灵活消息格式示例"，作为面向对象设计不一定是比其他设计更好的设计的很好例子（仅仅因为它是面向对象的），然后你可以继续阅读第19章关于一般控制问题的讨论。

## 使用表驱动方法的一般考虑

在适当的情况下使用，表驱动的代码比复杂逻辑更简单，更容易修改，而且更高效。假设你想将字符分类为字母、标点符号和数字；你可能会使用像这样的复杂逻辑链：

示例 18-1. 使用复杂逻辑对字符进行分类的 Java 示例

```java
if ( ( ( 'a' <= inputChar ) && ( inputChar <= 'z' ) ) ||
   ( ( 'A' <= inputChar ) && ( inputChar <= 'Z' ) ) ) {
   charType = CharacterType.Letter;
}
else if ( ( inputChar == ' ' ) || ( inputChar == ',' ) ||
   ( inputChar == '.' ) || ( inputChar == '!' ) || ( inputChar == '(' ) ||
   ( inputChar == ')' ) || ( inputChar == ':' ) || ( inputChar == ';' ) ||
   ( inputChar == '?' ) || ( inputChar == '-' ) ) {
   charType = CharacterType.Punctuation;
}
else if ( ( '0' <= inputChar ) && ( inputChar <= '9' ) ) {
   charType = CharacterType.Digit;
}
```

如果你使用查找表，你会将每种字符的类型存储在一个通过字符代码访问的数组中。刚才显示的复杂代码片段将被替换为：

示例 18-2. 使用查找表对字符进行分类的 Java 示例

```java
charType = charTypeTable[ inputChar ];
```

这个片段假设 charTypeTable 数组已经在前面设置好了。你把程序的知识放入其数据而不是其逻辑——放在表中而不是 if 测试中。

## 使用表驱动方法的两个问题

当你使用表驱动方法时，你必须解决两个问题。首先，你必须解决如何在表中查找条目的问题。你可以使用一些数据直接访问表。例如，如果你需要按月份分类数据，使用月份键入月份表是直接的。你可以使用索引 1 到 12 的数组。

其他数据太别扭，不能直接用于查找表条目。例如，如果你需要按社会安全号码分类数据，你不能使用社会安全号码直接键入表，除非你能负担得起在表中存储 999-99-9999 个条目。你被迫使用更复杂的方法。以下是在表中查找条目的方法列表：

- 直接访问
- 索引访问
- 阶梯访问

这些访问类型中的每一个都在本章后面的小节中更详细地描述。

如果你使用表驱动方法，第二个问题是应该将什么存储在表中。在某些情况下，表查找的结果是数据。如果是这样，你可以将数据存储在表中。在其他情况下，表查找的结果是一个动作。在这种情况下，你可以存储描述动作的代码，或者在某些语言中，存储实现动作的子程序的引用。在这两种情况下，表都变得更复杂。

## 每月天数示例

假设你需要确定每个月的天数（为了讨论，暂时忽略闰年）。当然，一个笨拙的方法是编写一个大的 if 语句：

示例 18-3. 笨拙地确定每月天数的 Visual Basic 示例

```vb
If ( month = 1 ) Then
   days = 31
ElseIf ( month = 2 ) Then
   days = 28
ElseIf ( month = 3 ) Then
   days = 31
ElseIf ( month = 4 ) Then
   days = 30
ElseIf ( month = 5 ) Then
   days = 31
ElseIf ( month = 6 ) Then
   days = 30
ElseIf ( month = 7 ) Then
   days = 31
ElseIf ( month = 8 ) Then
   days = 31
ElseIf ( month = 9 ) Then
   days = 30
ElseIf ( month = 10 ) Then
   days = 31
ElseIf ( month = 11 ) Then
   days = 30
ElseIf ( month = 12 ) Then
   days = 31 End If
```

执行相同功能的一个更容易和更可修改的方法是将数据放入表中。在 Microsoft Visual Basic 中，你首先设置表：

示例 18-4. 优雅地确定每月天数的 Visual Basic 示例

```vb
' 初始化"每月天数"数据表
Dim daysPerMonth() As Integer = _
   { 31, 28, 31, 30, 31, 30, 31, 31, 30, 31, 30, 31 }
```

现在，你可以简单地使用简单的数组访问来查找一个月中的天数，而不是使用长的 if 语句：

示例 18-5. 优雅地确定每月天数的方法（续）

```vb
days = daysPerMonth( month-1 )
```

如果你想在表查找版本中考虑闰年，代码仍然很简单，假设 LeapYearIndex() 的值为 0 或 1：

示例 18-6. 优雅地确定每月天数的方法（续）

```vb
days = daysPerMonth( month-1, LeapYearIndex() )
```

在 if 语句版本中，如果考虑闰年，一长串 if 会变得更加复杂。

确定每月天数是一个方便的例子，因为你可以使用月份变量来查找表中的条目。你经常可以使用本来会控制大量 if 语句的数据来直接访问表。

## 保险费率示例

假设你正在编写一个计算医疗保险费率的程序，你有按年龄、性别、婚姻状况和一个人是否吸烟而不同的费率。如果你必须为费率编写逻辑控制结构，你会得到类似这样的东西：

示例 18-7. 笨拙地确定保险费率的 Java 示例

```java
if ( gender == Gender.Female ) {
   if ( maritalStatus == MaritalStatus.Single ) {
      if ( smokingStatus == SmokingStatus.NonSmoking ) {
         if ( age < 18 ) {
            rate = 200.00;
         }
         else if ( age == 18 ) {
            rate = 250.00;
         }
         else if ( age == 19 ) {
            rate = 300.00;
         }
         ...
         else if ( 65 < age ) {
            rate = 450.00;
      }
      else {
         if ( age < 18 ) {
            rate = 250.00;
         }
         else if ( age == 18 ) {
            rate = 300.00;
         }
         else if ( age == 19 ) {
            rate = 350.00;
         }
         ...
         else if ( 65 < age ) {
            rate = 575.00;
         }
      }
   else if ( maritalStatus == MaritalStatus.Married )
   ...
}
```

逻辑结构的缩写版本应该足以让你了解这种东西可以变得多么复杂。它没有显示已婚女性、任何男性或 18 到 65 岁之间的大多数年龄。你可以想象当你对整个费率表进行编程时会变得多么复杂。

你可能会说，"是的，但你为什么对每个年龄都进行测试？为什么不简单地把费率放入每个年龄的数组中？"这是一个好问题，一个明显的改进是将费率放入每个年龄的单独数组中。

然而，一个更好的解决方案是将所有因素的费率放入数组，而不仅仅是年龄。以下是你如何在 Visual Basic 中声明数组：

示例 18-8. 声明数据以设置保险费率表的 Visual Basic 示例

```vb
Public Enum SmokingStatus
   SmokingStatus_First = 0
   SmokingStatus_Smoking = 0
   SmokingStatus_NonSmoking = 1
   SmokingStatus_Last = 1
End Enum
 Public Enum Gender
   Gender_First = 0
   Gender_Male = 0
   Gender_Female = 1
   Gender_Last = 1
End Enum
Public Enum MaritalStatus
   MaritalStatus_First = 0
   MaritalStatus_Single = 0
   MaritalStatus_Married = 1
   MaritalStatus_Last = 1
End Enum

Const MAX_AGE As Integer = 125

Dim rateTable ( SmokingStatus_Last, Gender_Last, MaritalStatus_Last, _
   MAX_AGE ) As Double
```

一旦你声明了数组，你必须想出某种方法将数据放入其中。你可以使用赋值语句、从磁盘文件读取数据、计算数据，或者做任何合适的事情。设置数据后，当你需要计算费率时，你就已经成功了。前面显示的复杂逻辑被一个像这样的简单语句替换：

## 基于逻辑的方法

如果你使用基于逻辑的方法，你可能会读取每条消息，检查 ID，然后调用一个设计用来读取、解释和打印每种消息的例程。如果你有 20 种消息，你就需要 20 个例程。你还会有谁知道有多少低级例程来支持它们——例如，你会有一个 PrintBuoyTemperatureMessage() 例程来打印浮标温度消息。面向对象的方法不会好太多：你通常会使用一个抽象消息对象和每种消息类型的一个子类。

每次任何消息的格式改变时，你都必须改变负责该消息的例程或类中的逻辑。在前面详细的消息中，如果平均温度字段从浮点数变为其他东西，你就必须改变 PrintBuoyTemperatureMessage() 的逻辑。（如果浮标本身从"浮点数"变为其他东西，你就必须得到一个新的浮标！）

在基于逻辑的方法中，消息读取例程由一个循环组成，该循环读取每条消息、解码 ID，然后根据消息 ID 调用 20 个例程之一。以下是基于逻辑方法的伪代码：

```
当有更多消息要读取时
   读取消息头
   从消息头解码消息 ID
   如果消息头是类型 1，则
      打印类型 1 消息
   否则如果消息头是类型 2，则
      打印类型 2 消息
   ...
   否则如果消息头是类型 19，则
      打印类型 19 消息
   否则如果消息头是类型 20，则
      打印类型 20 消息
```

交叉参考

这个低级伪代码的用途不同于你用于例程设计的伪代码。关于用伪代码设计的详细信息，请参见第9章。

伪代码是缩写版本，因为你可以不用看所有 20 种情况就明白这个想法。

## 面向对象方法

如果你使用死记硬背的面向对象方法，逻辑将隐藏在对象继承结构中，但基本结构同样复杂：

```
当有更多消息要读取时
   读取消息头
   从消息头解码消息 ID
   如果消息头是类型 1，则
      实例化类型 1 消息对象
   否则如果消息头是类型 2，则
      实例化类型 2 消息对象
   ...
   否则如果消息头是类型 19，则
      实例化类型 19 消息对象
   否则如果消息头是类型 20，则
      实例化类型 20 消息对象
   结束如果
结束循环
```

无论逻辑是直接编写还是包含在专门的类中，20 种消息中的每一种都会有自己的例程来打印其消息。每个例程也可以用伪代码表示。这是读取和打印浮标温度消息的例程的伪代码：

```
打印"浮标温度消息"

读取浮点值
打印"平均温度"
打印浮点值

读取浮点值
打印"温度范围"
打印浮点值

读取整数值
打印"样本数量"
打印整数值

读取字符串
打印"位置"
打印字符串

读取时间
打印"测量时间"
打印时间
```

这只是一种消息的代码。其他 19 种消息中的每一种都需要类似的代码。而且如果添加了第 21 种消息，需要添加第 21 个例程或第 21 个子类——无论哪种方式，新消息类型都需要更改代码。

## 表驱动方法

表驱动方法比之前的方法更经济。消息读取例程由一个循环组成，该循环读取每条消息头、解码 ID、在消息数组中查找消息描述，然后每次调用相同的例程来解码消息。使用表驱动方法，你可以用表而不是硬编码在程序逻辑中描述每条消息的格式。这使得最初编码更容易，生成的代码更少，并且更容易维护而无需更改代码。

要使用这种方法，你首先列出消息的种类和字段的类型。在 C++ 中，你可以这样定义所有可能的字段类型：

示例 18-10. C++ 中定义消息数据类型的示例

```cpp
enum FieldType {
   FieldType_FloatingPoint,
   FieldType_Integer,
   FieldType_String,
   FieldType_TimeOfDay,
   FieldType_Boolean,
   FieldType_BitField,
   FieldType_Last = FieldType_BitField
};
```

不要硬编码每种 20 种消息的打印例程，你可以创建少数几个打印每种主要数据类型——浮点数、整数、字符串等——的例程。你可以在表中描述每种消息的内容（包括每个字段的名称），然后基于表中的描述解码每条消息。描述一种消息的表条目可能看起来像这样：

示例 18-11. 定义消息表条目的示例

```
消息开始
   NumFields 5
   MessageName "浮标温度消息"
   字段 1, 浮点数, "平均温度"
   字段 2, 浮点数, "温度范围"
   字段 3, 整数, "样本数量"
   字段 4, 字符串, "位置"
   字段 5, 时间, "测量时间"
消息结束
```

这个表可以在程序中硬编码（在这种情况下，显示的每个元素将被分配给变量），或者可以在程序启动时或之后从文件读取。

一旦消息定义被读入程序，你就不把信息嵌入程序的逻辑中，而是将其嵌入数据中。数据往往比逻辑更灵活。当消息格式改变时，数据很容易更改。如果你必须添加一种新消息，你只需向数据表添加另一个元素。

以下是表驱动方法中顶层循环的伪代码：

示例 18-12. 表驱动方法的伪代码

```
当有更多消息要读取时       <-- 1
   读取消息头            |
   从消息头解码消息 ID       <-- 1
   在消息描述表中查找消息描述
   根据消息描述读取消息字段并打印它们
结束循环
```

(1)这里的这三行与基于逻辑的方法相同。

与基于逻辑方法的伪代码不同，在这种情况下伪代码不是缩写，因为逻辑要简单得多。在这个级别以下的逻辑中，你会发现一个能够解释消息描述表中的消息描述、读取消息数据和打印消息的例程。那个例程比任何基于逻辑的消息打印例程都更通用，但不会复杂多少，而且它将是一个例程而不是 20 个：

```
当有更多字段要打印时
   从消息描述中获取字段类型
   case (字段类型)
      浮点数
         读取浮点值
         打印字段标签
         打印浮点值
      整数
         读取整数值
         打印字段标签
         打印整数值
      字符串
         读取字符串
         打印字段标签
         打印字符串
      时间
         读取时间
         打印字段标签
         打印时间
      布尔
         读取单个标志
         打印字段标签
         打印单个标志
      位字段
         读取位字段
         打印字段标签
         打印位字段
   结束 case
结束循环
```

诚然，这个带有六个 case 的例程比打印浮标温度消息所需的单个例程要长。但这是你唯一需要的例程。你不需要其他 19 种消息的 19 个其他例程。这个例程处理六种字段类型，并照顾所有种类的消息。

这个例程也展示了实现这种表查找的最复杂方式，因为它使用了 case 语句。另一种方法是创建一个抽象类 AbstractField，然后为每种字段类型创建子类。你不需要 case 语句；你可以调用适当类型对象的成员例程。

以下是你如何在 C++ 中设置对象类型：

示例 18-13. C++ 中设置对象类型的示例

```cpp
class AbstractField {
   public:
   virtual void ReadAndPrint( string, FileStatus & ) = 0;
}

class FloatingPointField : public AbstractField {
   public:
   virtual void ReadAndPrint( string, FileStatus & ) {
   ...
   }
 }

class IntegerField ...
class StringField ...
...
```

这个代码片段声明了一个成员例程用于每个类，该例程有一个字符串参数和一个 FileStatus 参数。

下一步是声明一个数组来保存对象集。数组是查找表，以下是它的样子：

示例 18-14. C++ 中设置保存每种类型对象的表的示例

```cpp
AbstractField* field[ Field_Last ];
```

设置对象表所需的最后一步是将特定对象的名称分配给 Field 数组：

示例 18-15. C++ 中设置对象列表的示例

```cpp
field[ Field_FloatingPoint ] = new FloatingPointField();
field[ Field_Integer ] = new IntegerField();
field[ Field_String ] = new StringField();
field[ Field_TimeOfDay ] = new TimeOfDayField();
field[ Field_Boolean ] = new BooleanField();
field[ Field_BitField ] = new BitFieldField();
```

这个代码片段假设 FloatingPointField 和赋值语句右侧的其他标识符是 AbstractField 类型对象的名称。将对象分配给数组中的元素意味着你可以通过对数组元素的引用来调用正确的 ReadAndPrint() 例程，而不是直接使用特定类型的对象。

一旦例程表设置好了，你就可以简单地通过访问对象表并调用表中的成员例程来处理消息中的字段。代码看起来像这样：

示例 18-16. C++ 中在表中查找对象和成员例程的示例

```cpp
fieldIdx = 1;       <-- 1
while ( ( fieldIdx <= numFieldsInMessage ) and ( fileStatus == OK ) ) {       <-- 1
   fieldType = fieldDescription[ fieldIdx ].FieldType;
   fieldName = fieldDescription[ fieldIdx ].FieldName;
   field[ fieldType ].ReadAndPrint( fieldName, fileStatus );       <-- 2
}
```

(1)这只是消息中每个字段的例行工作。

(2)这是表查找，根据字段类型调用例程——只是通过在对象表中查找。

还记得包含 case 语句的原始 34 行表查找伪代码吗？如果你用对象表替换 case 语句，这就是你需要提供的所有代码来实现相同的功能。令人难以置信的是，这也是替换基于逻辑方法中所有 20 个单独例程所需的全部代码。此外，如果消息描述是从文件读取的，新消息类型不需要代码更改，除非有一种新的字段类型。

你可以在任何面向对象语言中使用这种方法。它比长的 if 语句、case 语句或大量子类更不容易出错、更容易维护和更高效。

设计使用继承和多态这一事实并不能使它成为一个好的设计。前面"面向对象方法"部分描述的死记硬背面向对象设计需要与死记硬背函数设计一样多的代码——甚至更多。这种方法使解决方案空间变得更复杂，而不是更简单。在这种情况下，关键的设计洞察既不是面向对象也不是函数式导向——而是对经过深思熟虑的查找表的使用。

## 伪造查找键

在前面的三个示例中，你可以直接使用数据键入表。也就是说，你可以直接使用 messageID 作为键，就像在每月天数示例中使用 month，以及在保险费率示例中使用 gender、maritalStatus 和 smokingStatus 一样。

你总是希望直接键入表，因为它简单且快速。然而，有时数据不配合。在保险费率示例中，年龄表现不好。原始逻辑对 18 岁以下的人有一种费率，对 18 到 65 岁有单独的费率，对 65 岁以上有一种费率。这意味着对于 0 到 17 岁和 66 岁及以上，你不能使用年龄直接键入只存储一组费率的表。

这引出了伪造表查找键的话题。你可以用几种方式伪造键：

**复制信息以使键直接工作**。使年龄作为键进入费率表的一种直接方法是，为 0 到 17 岁的每个年龄复制 18 岁以下费率，然后使用年龄直接键入表。你可以对 66 岁及以上的人做同样的事情。这种方法的好处是表结构本身是直接的，表访问也是直接的。如果你需要为 17 岁及以下添加年龄特定费率，你可以只更改表。缺点是重复会浪费冗余信息的空间，并增加表中错误的可能性——仅仅因为表会包含冗余数据。

**转换键使其直接工作**。使年龄作为直接键的第二种方法是对年龄应用一个函数，使其工作良好。在这种情况下，函数必须将 0 到 17 岁的所有年龄转换为一个键，比如 17，将 66 岁以上的所有年龄转换为另一个键，比如 66。这个特定范围表现良好，你可以使用 min() 和 max() 函数进行转换。例如，你可以使用表达式

```cpp
max( min( 66, Age ), 17 )
```

来创建一个从 17 到 66 的表键。

创建转换函数需要你认识到你要用作键的数据中的模式，而这并不总是像使用 min() 和 max() 例程那么简单。假设在这个例子中，费率为五年年龄段而不是一年年龄段。除非你想把数据重复五次，否则你必须想出一个函数正确地将年龄除以 5 并使用 min() 和 max() 例程。

**在自己的例程中隔离键转换**。如果你必须伪造数据以使其作为表键工作，将将数据转换为键的操作放入它自己的例程中。一个例程消除了在不同地方使用不同转换的可能性。当转换改变时，它使修改更容易。一个好的例程名称，比如 KeyFromAge()，也澄清并记录了数学运算的目的。

如果你的环境提供了现成的键转换，使用它们。例如，Java 提供 HashMap，可用于关联键/值对。

## 索引访问表

有时候简单的数学转换不够强大，无法从像年龄这样的数据跳转到表键。有些这样的情况适合使用索引访问方案。

当你使用索引时，你使用主要数据在索引表中查找键，然后使用索引表中的值来查找你主要感兴趣的主数据。

假设你经营一个仓库，有大约 100 种物品。假设每种物品有一个从 0000 到 9999 的四位数零件编号。在这种情况下，如果你想使用零件编号直接键入描述每种物品某个方面的表，你设置一个包含 10,000 个条目（从 0 到 9999）的索引数组。该数组除了对应于仓库中 100 种物品的零件编号的 100 个条目外都是空的。如图 18-4 所示，那些条目指向一个条目少得多的物品描述表。

> **图 18-4**。索引访问表不是直接访问的，而是通过中间的索引访问的

索引访问方案提供两个主要优势。首先，如果主查找表中的每个条目都很大，创建一个有很多浪费空间的索引数组比创建一个有很多浪费空间的主查找表节省更多空间。例如，假设主表每个条目占用 100 字节，索引数组每个条目占用 2 字节。假设主表有 100 个条目，用于访问它的数据有 10,000 个可能的值。在这种情况下，选择是在有 10,000 个条目的索引和有 10,000 个条目的主数据成员之间。如果你使用索引，你的总内存使用是 30,000 字节。如果你放弃索引结构并在主表中浪费空间，你的总内存使用是 1,000,000 字节。

第二个优势，即使你通过使用索引不节省空间，是有时操作索引中的条目比操作主表中的条目更便宜。例如，如果你有一个包含员工姓名、入职日期和薪水的表，你可以创建一个按员工姓名访问表的索引，另一个按入职日期访问，第三个按薪水访问。

索引访问方案的最后一个优势是通用表查找的可维护性优势。编码在表中的数据比编码在代码中的数据更容易维护。为了最大化灵活性，将索引访问代码放在它自己的例程中，并在你需要从零件编号获取表键时调用该例程。当需要更改表时，你可能决定切换索引访问方案或完全切换到另一个表查找方案。如果你不在整个程序中分散索引访问，访问方案会更容易更改。

## 阶梯访问表

另一种表访问是阶梯方法。这种访问方法不像索引结构那样直接，但也不会浪费那么多数据空间。

阶梯结构的一般思想（如图 18-5 所示）是表中的条目对数据范围有效，而不是对不同的数据点有效。

> **图 18-5**。阶梯方法通过确定条目在"楼梯"的哪个级别命中来对每个条目进行分类。它击中的"台阶"决定了它的类别

例如，如果你正在编写一个评分程序，"B"条目范围可能是 75% 到 90%。以下是你可能必须编程的等级范围：

```
≥ 90.0% A
< 90.0% B
< 75.0% C
< 65.0% D
< 50.0% F
```

这对表查找来说是一个丑陋的范围，因为你不能使用简单的数据转换函数来键入字母 A 到 F。索引方案会很尴尬，因为数字是浮点数。你可能考虑将浮点数转换为整数，在这种情况下这将是一个有效的设计选择，但为了说明，这个例子将坚持使用浮点数。

要使用阶梯方法，你将每个范围的上限放入一个表，然后编写一个循环来检查分数是否超过每个范围的上限。当你找到分数第一次超过某个范围顶部的点时，你就知道等级是什么。使用阶梯技术，你必须小心正确处理范围的端点。以下是 Visual Basic 中基于此示例给学生分等级的代码：

示例 18-17. 阶梯表查找的 Visual Basic 示例

```vb
' 设置评分表数据
Dim rangeLimit() As Double = { 50.0, 65.0, 75.0, 90.0, 100.0 }
Dim grade() As String = { "F", "D", "C", "B", "A" }
maxGradeLevel = grade.Length -- 1
...

' 根据学生的分数给学生分配等级
gradeLevel = 0
studentGrade = "A"
While ( ( studentGrade = "A" ) and ( gradeLevel < maxGradeLevel ) )
   If ( studentScore < rangeLimit( gradeLevel ) ) Then
      studentGrade = grade( gradeLevel )
   End If
   gradeLevel = gradeLevel + 1
Wend
```

虽然这是一个简单的例子，你可以很容易地将其推广到处理多个学生、多种评分方案（例如，不同作业的不同分数等级）和评分方案的更改。

这种方法的优势在于它适用于不规则数据。评分示例很简单，因为虽然等级是在不规则间隔分配的，但数字是"整齐的"，以 5 和 0 结尾。阶梯方法同样适用于不以 5 和 0 整齐结尾的数据。你可以在统计工作中使用阶梯方法处理如下所示的概率分布的数字：

| 概率 | 保险索赔金额 |
|------|-------------|
| 0.458747 | $0.00 |
| 0.547651 | $254.32 |
| 0.627764 | $514.77 |
| 0.776883 | $747.82 |
| 0.893211 | $1,042.65 |
| 0.957665 | $5,887.55 |
| 0.976544 | $12,836.98 |
| 0.987889 | $27,234.12 |
| ... | ... |

像这样的丑陋数字无法尝试任何函数来将它们整齐地转换为表键。阶梯方法是答案。

这种方法也享有表驱动方法的一般优势：它是灵活和可修改的。如果评分示例中的评分范围要更改，程序可以很容易地通过修改 RangeLimit 数组中的条目来适应。你可以很容易地推广程序的等级分配部分，使其接受等级表和相应的截止分数。程序的等级分配部分不必使用百分比表示的分数；它可以使用原始分数而不是百分比，程序不必改变太多。

以下是使用阶梯技术时需要考虑的一些微妙之处：

**注意端点**。确保你已经覆盖了每个阶梯范围顶端的情况。运行阶梯搜索，使其找到映射到任何范围而不是最顶层范围的项，然后让其余的落入最顶层范围。有时这需要为最顶层范围的顶部创建一个假值。

**小心不要混淆 < 和 <=**。确保循环正确终止于落入顶部范围的值，并且范围边界被正确处理。

**考虑使用二分搜索而不是顺序搜索**。在评分示例中，分配等级的循环按顺序搜索评分限制列表。如果你有一个更大的列表，顺序搜索的成本可能会变得过高。如果是这样，你可以用准二分搜索替换它。这是一个"准"二分搜索，因为大多数二分搜索的目的是找到一个值。在这种情况下，你不期望找到值；你期望找到值的正确类别。二分搜索算法必须正确确定值应该放在哪里。还要记住将端点视为特殊情况。

**考虑使用索引访问而不是阶梯技术**。诸如索引访问表中所述的索引访问方案可能是阶梯技术的一个很好的替代方案。阶梯方法所需的搜索可能会累加，如果执行速度是一个问题，你可能愿意用额外的索引结构占用的空间来换取更直接访问方法的时间优势。

显然，这种替代方案并非在所有情况下都是好选择。在评分示例中，你可能可以使用它；如果你只有 100 个离散的百分比点，设置索引数组的内存成本不会太高。另一方面，如果你有前面列出的概率数据，你就无法设置索引方案，因为你不能用像 0.458747 和 0.547651 这样的数字键入条目。

在某些情况下，任何几种选项都可能有效。设计的要点是为你的情况选择几种好选项中的一个。不要太担心选择最好的。正如微软杰出工程师 Butler Lampson 所说，追求好的解决方案并避免灾难比试图找到最佳解决方案要好（Lampson 1984）。

## 关键点

- 表提供了复杂逻辑和继承结构的替代方案。如果你发现你被程序的逻辑或继承树弄糊涂了，问问自己是否可以用查找表简化。
- 使用表的一个关键考虑是决定如何访问表。你可以通过直接访问、索引访问或阶梯访问来访问表。
- 另一个关键考虑是决定具体将什么放入表中。
