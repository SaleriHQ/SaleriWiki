## 布尔表达式

除了最简单的控制结构——按顺序执行语句的控制结构——所有控制结构都依赖于布尔表达式的求值。

## 在布尔测试中使用 true 和 false

在布尔表达式中使用 true 和 false 标识符，而不是使用像 0 和 1 这样的值。大多数现代语言都有布尔数据类型，并为 true 和 false 提供预定义的标识符。它们使事情变得简单——它们甚至不允许你为布尔变量赋 true 或 false 以外的值。不提供布尔数据类型的语言要求你有更多的纪律来使布尔表达式可读。以下是一个问题的例子：

示例 19-1. 在 Visual Basic 中使用模糊标志表示布尔值的示例

```vb
Dim printerError As Integer
Dim reportSelected As Integer
Dim summarySelected As Integer
...
If printerError = 0 Then InitializePrinter()
If printerError = 1 Then NotifyUserOfError()

If reportSelected = 1 Then PrintReport()
If summarySelected = 1 Then PrintSummary()

If printerError = 0 Then CleanupPrinter()
```

如果使用像 0 和 1 这样的标志是常见做法，有什么问题？从阅读代码来看，函数调用是在测试为真时执行还是为假时执行并不清楚。代码片段本身没有任何东西告诉你 1 代表真 0 代表假，或者相反的情况是真的。甚至不清楚值 1 和 0 是用来表示真假。例如，在 `If reportSelected = 1` 行，1 很容易代表第一份报告，2 代表第二份，3 代表第三份；代码中没有任何东西告诉你 1 代表真或假。在写 0 时你意思是 1，反之亦然，也很容易出错。

---

**使用名为 true 和 false 的术语进行布尔表达式测试**。如果你的语言不直接支持这些术语，使用预处理器宏或全局变量创建它们。以下是使用 Microsoft Visual Basic 内置的 True 和 False 重写的前面代码示例：

示例 19-2. 好的但不是最好的 Visual Basic 示例：使用 True 和 False 进行测试而不是数值

```vb
Dim printerError As Boolean
Dim reportSelected As ReportType
Dim summarySelected As Boolean
...
If ( printerError = False ) Then InitializePrinter()
If ( printerError = True ) Then NotifyUserOfError()

If ( reportSelected = ReportType_First ) Then PrintReport()
If ( summarySelected = True ) Then PrintSummary()

If ( printerError = False ) Then CleanupPrinter()
```

交叉参考

有关使这些相同测试更好的方法，请参见下一个代码示例。

使用 True 和 False 常量使意图更清晰。你不必记住 1 和 0 代表什么，而且你不会意外地将它们颠倒。此外，在重写的代码中，现在清楚了原始 Visual Basic 示例中的一些 1 和 0 并没有用作布尔标志。`If reportSelected = 1` 行根本不是布尔测试；它测试是否选择了第一份报告。

这种方法告诉读者你正在进行布尔测试。当你意思是假但写真时，比当你意思是 0 但写 1 时更难写，而且你避免了将魔法数字 0 和 1 散布在整个代码中。以下是定义布尔测试中 true 和 false 的一些提示：

**隐式比较布尔值**。你可以通过将表达式视为布尔表达式来编写更清晰的测试。例如，写

```cpp
while ( not done ) ...
while ( a > b ) ...
```

而不是

```cpp
while ( done = false ) ...
while ( (a > b) = true ) ...
```

使用隐式比较减少了阅读代码的人必须记住的术语数量，生成的表达式更像对话式英语。前面示例可以这样重写，样式更好：

示例 19-3. 隐式测试 True 和 False 的更好的 Visual Basic 示例

```vb
Dim printerError As Boolean
Dim reportSelected As ReportType
Dim summarySelected As Boolean
...
If ( Not printerError ) Then InitializePrinter()
If ( printerError ) Then NotifyUserOfError()

If ( reportSelected = ReportType_First ) Then PrintReport()
If ( summarySelected ) Then PrintSummary()

If ( Not printerError ) Then CleanupPrinter()
```

如果你的语言不支持布尔变量，你必须模拟它们，你可能无法使用这种技术，因为对 true 和 false 的模拟不能总是用像 `while ( not done )` 这样的语句来测试。

## 简化复杂表达式

你可以采取几个步骤来简化复杂表达式：

**将复杂测试分解为带有新布尔变量的部分测试**。不要创建一个带有六个术语的巨大测试，而是将中间值赋给允许你执行更简单测试的术语。

**将复杂表达式移入布尔函数**。如果测试经常重复或分散了程序的主流，将测试的代码移入一个函数并测试函数的值。例如，这里有一个复杂的测试：

示例 19-4. Visual Basic 中复杂测试的示例

```vb
If ( ( document.AtEndOfStream ) And ( Not inputError ) ) And _
   ( ( MIN_LINES <= lineCount ) And ( lineCount <= MAX_LINES ) ) And _
   ( Not ErrorProcessing( ) ) Then
   ' 做某事或其他
   ...
End If
```

如果你对测试本身不感兴趣，这是一个丑陋的测试，必须阅读。通过将其放入布尔函数，你可以隔离测试，让读者忽略它，除非它很重要。以下是你如何将 if 测试放入函数：

示例 19-5. Visual Basic 中复杂测试移入布尔函数的示例，以及新的中间变量使测试更清晰

```vb
Function DocumentIsValid( _
   ByRef documentToCheck As Document, _
   lineCount As Integer, _
   inputError As Boolean _
   ) As Boolean

   Dim allDataRead As Boolean
   Dim legalLineCount As Boolean

   allDataRead = ( documentToCheck.AtEndOfStream ) And ( Not inputError )       <-- 1
   legalLineCount = ( MIN_LINES <= lineCount ) And ( lineCount <= MAX_LINES )       <-- 1
   DocumentIsValid = allDataRead And legalLineCount And ( Not ErrorProcessing() )

End Function
```

(1)这里引入了中间变量来澄清下面最后一行的测试。

交叉参考

有关使用中间变量来澄清布尔测试的技术的详细信息，请参见布尔变量中的"使用布尔变量记录你的程序"。

这个例子假设 ErrorProcessing() 是一个指示当前处理状态的布尔函数。现在，当你在代码的主流中阅读时，你不必阅读复杂的测试：

示例 19-6. Visual Basic 中没有复杂测试的代码主流示例

```vb
If ( DocumentIsValid( document, lineCount, inputError ) ) Then
   ' 做某事或其他
   ...
End If
```

如果你只使用一次测试，你可能认为不值得将其放入例程。但将测试放入一个命名良好的函数会提高可读性，使你更容易看到代码在做什么，这是一个足够的理由这样做。

新函数名在程序中引入了一个抽象，在代码中记录了测试的目的。这比用注释记录测试更好，因为代码比注释更有可能被阅读，而且也更有可能保持最新。

**使用决策表替换复杂条件**。有时你有一个涉及几个变量的复杂测试。使用决策表执行测试而不是使用 if 或 case 可能会有帮助。决策表查找最初更容易编码，只有几行代码，没有棘手的控制结构。这种复杂度最小化最大限度地减少了犯错的机会。如果你的数据改变，你可以更改决策表而不改变代码；你只需要更新数据结构的內容。

交叉参考

有关使用表作为复杂逻辑替代品的详细信息，请参见第18章。

## 正面形成布尔表达式

我不傻不笨。
— 霍默·辛普森

不少人对一串非正数——也就是说，大多数人对大量负数理解有困难。你可以采取几个措施来避免程序中复杂的负布尔表达式：

示例 19-7. Java 中令人困惑的负布尔测试示例

```java
if ( !statusOK ) {       <-- 1
   // 做某事
   ...
}
else {
   // 做其他事情
   ...
}
```

(1)这里是负的 not。

你可以将其更改为以下正面表达的测试：

示例 19-8. Java 中更清晰的正面布尔测试示例

```java
if ( statusOK ) {       <-- 1
   // 做其他事情
   ...       <-- 2
}
else {
   // 做某事       <-- 3
   ...
}
```

(1)

(2)名义情况。

(3)名义情况。

第二个代码片段在逻辑上与第一个相同，但由于负表达式已变为正，所以更容易阅读。

交叉参考

对布尔表达式进行正面框架的建议有时与在 if 之后而不是 else 之后编写名义情况的建议相矛盾。（参见 if 语句。）在这种情况下，你必须考虑每种方法的好处，并决定哪种对你的情况更好。

或者，你可以选择一个不同的变量名，一个会颠倒测试真值的变量。在示例中，你可以用 ErrorDetected 替换 statusOK，这在 statusOK 为假时为真。

**应用德摩根定律简化带否定的布尔测试**。德摩根定律让你利用一个表达式和它的双重否定版本之间的逻辑关系，因为它们意味着相同的东西。例如，你可能有一个包含以下测试的代码片段：

示例 19-9. Java 中否定测试的示例

```java
if ( !displayOK || !printerOK ) ...
```

这在逻辑上等价于以下内容：

示例 19-10. 应用德摩根定律后的 Java 示例

```java
if ( !( displayOK && printerOK ) ) ...
```

在这里你不必翻转 if 和 else 子句；最后两个代码片段中的表达式在逻辑上是等价的。要将德摩根定律应用于逻辑运算符 and 或逻辑运算符 or 和一对操作数，你需要否定每个操作数，切换 and 和 or，并否定整个表达式。表 19-1 总结了德摩根定律下可能的转换。

> **表 19-1**。德摩根定律下逻辑表达式的转换

| 初始表达式 | 等价表达式 |
|-----------|-----------|
| not A and not B | not ( A or B ) |
| not A and B | not ( A or not B ) |
| A and not B | not ( not A or B ) |
| A and B | not ( not A or not B ) |
| not ( A and B ) | not A or not B[*] |
| not ( A or B ) | not A and not B |
| not A or B | A or not B |
| A or B | not ( not A and not B ) |
| not ( A and not B ) | not ( not A and B ) |
| not ( not A and B ) | not ( not A and not B ) |

[*] 这是示例中使用的表达式。

## 使用括号澄清布尔表达式

如果你有一个复杂的布尔表达式，不要依赖语言的求值顺序，而是使用括号使你的意思清晰。使用括号对你的读者要求更少，读者可能不理解你的语言如何求值布尔表达式的微妙之处。如果你聪明，你就不会依赖你自己或读者对求值优先级的深入记忆——尤其是当你必须在两种或更多语言之间切换时。使用括号不像发电报：你不会为每个字符收费——额外的字符是免费的。

交叉参考

有关使用括号澄清其他类型表达式的示例，请参见布局技术中的"括号"。

这里是一个括号太少的表达式：

示例 19-11. Java 中包含太少括号的表达式示例

```java
if ( a < b == c == d ) ...
```

这是一个从一开始就令人困惑的表达式，由于不清楚编码者是要测试 `(a < b ) == ( c == d )` 还是 `(( a < b ) == c ) == d`，所以更加令人困惑。以下是表达式更好的版本，仍然有点令人困惑，但括号有帮助：

示例 19-12. Java 中更好加括号的表达式示例

```java
if ( ( a < b ) == ( c == d ) ) ...
```

在这种情况下，括号有助于可读性和程序的正确性——编译器不会像第一个代码片段那样解释这个代码。当有疑问时，加括号。

**使用简单的计数技术来平衡括号**。如果你难以判断括号是否平衡，这里有一个有帮助的简单计数技巧。从说"零"开始。从左到右沿着表达式移动。当你遇到左括号时，说"一"。每次遇到另一个左括号，增加你说的数字。每次遇到右括号，减少你说的数字。如果在表达式结束时你回到 0，你的括号就是平衡的。

交叉参考

许多面向程序员的文本编辑器都有匹配括号、方括号和大括号的命令。有关编程编辑器的详细信息，请参见源代码工具中的"编辑"。

示例 19-13. Java 中平衡括号的示例

```java
if ( ( ( a < b ) == ( c == d ) ) && !done ) ...       <-- 1
   | | |          |    |        | |                  |
0  1 2 3          2    3        2 1                  0       <-- 2
```

(1)读这个。

(2)说这个。

在这个例子中，你以 0 结束，所以括号是平衡的。在下一个例子中，括号不平衡：

示例 19-14. Java 中不平衡括号的示例

```java
if ( ( a < b ) == ( c == d ) ) && !done ) ...       <-- 1
   | |               |        | |                  |
0  1 2 3          2    3        2 1                  0       <-- 2
```

(1)读这个。

(2)说这个。

在到达最后一个右括号之前你得到 0 是一个提示，在那一点之前缺少一个括号。你不应该在表达式的最后一个括号之前得到 0。

**完全为逻辑表达式加括号**。括号很便宜，它们有助于可读性。习惯性地为逻辑表达式完全加括号是好习惯。

## 了解布尔表达式如何求值

许多语言在布尔表达式求值中都有一种隐含的控制形式。一些语言的编译器先求值布尔表达式中的每个项，然后再组合项并求值整个表达式。其他语言的编译器有"短路"或"惰性"求值，只求值必要的部分。当取决于第一个测试的结果，你可能不想执行第二个测试时，这尤其重要。例如，假设你正在检查数组的元素，你有以下测试：

示例 19-15. 伪代码中错误测试的示例

```
while ( i < MAX_ELEMENTS and item[ i ] <> 0 ) ...
```

如果这个整个表达式被求值，在循环的最后一次遍历时你会得到一个错误。变量 i 等于 maxElements，所以表达式 item[ i ] 等同于 item[ maxElements ]，这是一个数组索引错误。你可能会争辩说，既然你只是在看值而不是改变它，这并不重要。但这是草率的编程实践，可能会让阅读代码的人感到困惑。在许多环境中，它也会产生运行时错误或保护违规。

在伪代码中，你可以重构测试以避免错误：

示例 19-16. 正确重构测试的伪代码示例

```
while ( i < MAX_ELEMENTS )
   if ( item[ i ] <> 0 ) then
      ...
```

这是正确的，因为 item[ i ] 除非 i 小于 maxElements，否则不会被求值。

许多现代语言提供了防止此类错误首先发生的设施。例如，C++ 使用短路求值：如果 and 的第一个操作数为假，则不求值第二个，因为无论如何整个表达式都为假。换句话说，在 C++ 中

```cpp
if ( SomethingFalse && SomeCondition ) ...
```

唯一被求值的是 SomethingFalse。一旦 SomethingFalse 被识别为假，求值就停止。

或运算符同样被短路。在 C++ 和 Java 中

```cpp
if ( somethingTrue || someCondition ) ...
```

唯一被求值的是 somethingTrue。一旦 somethingTrue 被识别为真，求值就停止，因为如果任何部分为真，整个表达式总是为真。由于这种求值方法，以下语句是一个好的、合法的语句。

示例 19-17. 由于短路求值而有效的 Java 测试示例

```java
if ( ( denominator != 0 ) && ( ( item / denominator ) > MIN_VALUE ) ) ...
```

如果当 denominator 等于 0 时这个完整表达式被求值，第二个操作数中的除法将产生除以零错误。但由于除非第一个部分为真，否则不求值第二个部分，所以在 denominator 等于 0 时永远不会被求值，因此不会发生除以零错误。

另一方面，因为 &&（and）是按从左到右求值的，以下逻辑等价的语句不起作用：

示例 19-18. 短路求值无法拯救的 Java 测试示例

```java
if ( ( ( item / denominator ) > MIN_VALUE ) && ( denominator != 0 ) ) ...
```

在这种情况下，item / denominator 在 denominator != 0 之前被求值。因此，这段代码犯了除以零错误。

Java 通过提供"逻辑"运算符进一步使这个情况复杂化。Java 的逻辑 & 和 | 运算符保证所有项将被完全求值，无论是否可以通过完整求值来确定表达式的真假。换句话说，在 Java 中，这是安全的：

示例 19-19. 由于短路（条件）求值而有效的 Java 测试示例

```java
if ( ( denominator != 0 ) && ( ( item / denominator ) > MIN_VALUE ) ) ...
```

但这不是安全的：

示例 19-20. 由于不能保证短路求值而无效的 Java 测试示例

```java
if ( ( denominator != 0 ) & ( ( item / denominator ) > MIN_VALUE ) ) ...
```

不同的语言使用不同种类的求值，语言实现者倾向于在表达式求值方面随意行事，所以请查阅你正在使用的特定版本语言的手册，找出你的语言使用哪种求值。更好的是，因为阅读代码的人可能不如你聪明，使用嵌套测试来澄清你的意图，而不是依赖求值顺序和短路求值。

## 按数线顺序编写数值表达式

组织数值测试，使其遵循数线上的点。一般来说，组织你的数值测试，使你有这样的比较：

```cpp
MIN_ELEMENTS <= i && i <= MAX_ELEMENTS
i < MIN_ELEMENTS || MAX_ELEMENTS < i
```

这个想法是从小到大从左到右排序元素。在第一行，MIN_ELEMENTS 和 MAX_ELEMENTS 是两个端点，所以它们放在两端。变量 i 应该在它们之间，所以它放在中间。在第二个例子中，你测试 i 是否在范围之外，所以 i 在任一端的测试的外部，MIN_ELEMENTS 和 MAX_ELEMENTS 在内部。这种方法很容易映射到图 19-1 中比较的可视图像：

> **图 19-1**。在布尔测试中使用数线排序的示例

如果你只针对 MIN_ELEMENTS 测试 i，i 的位置根据测试成功时 i 在哪里而变化。如果 i 应该更小，你会有这样的测试：

```cpp
while ( i < MIN_ELEMENTS ) ...
```

但如果 i 应该更大，你会有这样的测试：

```cpp
while ( MIN_ELEMENTS < i ) ...
```

这种方法比这样的测试更清晰：

```cpp
( i > MIN_ELEMENTS ) && ( i < MAX_ELEMENTS )
```

后者没有帮助读者可视化正在测试什么。

## 与 0 比较的指导原则

编程语言将 0 用于多个目的。它是一个数值。它是字符串中的空终止符。它是空指针的值。它是枚举中的第一项的值。它是逻辑表达式中的假。因为它被用于这么多目的，你应该编写突出 0 被使用的特定方式的代码。

**隐式比较逻辑变量**。如前所述，写逻辑表达式如 `while ( !done ) ...` 是合适的。这种对 0 的隐式比较是适当的，因为比较在逻辑表达式中。

**将数字与 0 显式比较**。虽然隐式比较逻辑表达式是适当的，但你应该显式比较数值表达式。对于数字，写

```cpp
while ( balance != 0 ) ...
```

而不是

```cpp
while ( balance ) ...
```

**在 C 中将字符与空终止符（'\0'）显式比较**。字符和数字一样，不是逻辑表达式。因此，对于字符，写

```cpp
while ( *charPtr != '\0' ) ...
```

而不是

```cpp
while ( *charPtr ) ...
```

这个建议与常见的 C 约定相反（就像这里的第二个例子一样处理字符数据），但它强化了表达式处理字符数据而不是逻辑数据的想法。一些 C 约定并不是基于最大化可读性或可维护性，这就是一个例子。幸运的是，随着越来越多的代码用 C++ 和 STL 字符串编写，整个问题正在逐渐消失。

**将指针与 NULL 比较**。对于指针，写

```cpp
while ( bufferPtr != NULL ) ...
```

而不是

```cpp
while ( bufferPtr ) ...
```

就像字符的建议一样，这个建议与既定的 C 约定相反，但可读性的提高证明了这一点。

## 布尔表达式的常见问题

布尔表达式在一些特定于语言的问题上有一些额外的陷阱：

**在 C 派生语言中，将常量放在比较的左侧**。C 派生语言在布尔表达式方面有一些特殊问题。如果你容易将 = 误写为 ==，考虑将常量和字面值放在表达式左侧的编程约定，像这样：

示例 19-21. C++ 中将常量放在表达式左侧的示例——编译器会捕获的错误

```cpp
if ( MIN_ELEMENTS = i ) ...
```

在这个表达式中，编译器应该将单个 = 标记为错误，因为给常量赋值是无效的。相反，在以下表达式中，编译器只会将其标记为警告，而且仅当你有完全打开的编译器警告时：

示例 19-22. C++ 中将常量放在表达式右侧的示例——编译器可能不会捕获的错误

```cpp
if ( i = MIN_ELEMENTS ) ...
```

这个建议与使用数线排序的建议相冲突。我个人偏好是使用数线排序，让编译器警告我意外的赋值。

**在 C++ 中，考虑为 &&、|| 和 == 创建预处理器宏替换（但仅作为最后手段）**。如果你有这个问题，可以为布尔 and 和 or 创建 #define 宏，并使用 AND 和 OR 而不是 && 和 ||。同样，当你要 == 时使用 = 是一个容易犯的错误。如果你经常被这个问题困扰，你可能创建一个像 EQUALS 这样的宏用于逻辑等于（==）。

许多有经验的程序员认为这种方法对不能保持编程语言细节的程序员有帮助，但对更精通语言的程序员来说会降低可读性。此外，大多数编译器会对看起来像错误的赋值和按位运算符的使用提供错误警告。打开完全编译器警告通常比创建非标准宏更好的选择。

**在 Java 中，知道 a==b 和 a.equals(b) 之间的区别**。在 Java 中，a==b 测试 a 和 b 是否引用同一个对象，而 a.equals(b) 测试对象是否具有相同的逻辑值。一般来说，Java 程序应该使用像 a.equals(b) 这样的表达式而不是 a==b。

## 复合语句（语句块）

"复合语句"或"语句块"是被视为控制程序流程的单一语句的一组语句。复合语句在 C++、C#、C 和 Java 中通过在一组语句周围写 { 和 } 来创建。有时它们由命令的关键字暗示，比如 Visual Basic 中的 For 和 Next。以下是有效使用复合语句的指导原则：

**成对写大括号**。写完块的开始和结束部分后再填写中间。人们经常抱怨匹配大括号或 begin-end 对有多难，这是一个完全不必要的问题。如果你遵循这个指导原则，你将永远不会在匹配这类配对时遇到麻烦。

首先写这个：

```cpp
for ( i = 0; i < maxLines; i++ )
```

然后写这个：

```cpp
for ( i = 0; i < maxLines; i++ ) { }
```

最后写这个：

```cpp
for ( i = 0; i < maxLines; i++ ) {
   // 这里放任何内容 ...
}
```

这适用于所有阻塞结构，包括 C++ 和 Java 中的 if、for 和 while，以及 Visual Basic 中的 If-Then-Else、For-Next 和 While-Wend 组合。

**使用大括号澄清条件**。条件已经够难阅读了，不需要确定哪些语句与 if 测试配对。在 if 测试后放一个语句在美学上很有吸引力，但在维护期间这样的语句往往会成为更复杂的块，而且当这种情况发生时，单个语句容易出错。

无论块内的代码是一行还是 20 行，都使用块来澄清你的意图。

## 驯服危险的深度嵌套

过度缩进或"嵌套"在计算文献中已经被批评了 25 年，仍然是混淆代码的主要罪魁祸首之一。Noam Chomsky 和 Gerald Weinberg 的研究表明，很少有人能理解超过三层的嵌套 if（Yourdon 1986a），许多研究人员建议避免超过三或四层的嵌套（Myers 1976, Marca 1981, Ledgard and Tauer 1987a）。深度嵌套违背了第5章中描述的软件主要技术 imperative：管理复杂度。这是一个足够避免深度嵌套的理由。

避免深度嵌套并不难。如果你有深度嵌套，你可以重新设计 if 和 else 子句中执行的测试，或者将代码重构为更简单的例程。以下小节介绍了几种减少嵌套深度的方法：

**通过重新测试部分条件来简化嵌套 if**。如果嵌套变得太深，你可以通过重新测试一些条件来减少嵌套级别。这个代码示例的嵌套已经深到需要重构：

示例 19-27. C++ 中糟糕的深度嵌套代码示例

```cpp
if ( inputStatus == InputStatus_Success ) {
   // 大量代码

   ...
   if ( printerRoutine != NULL ) {

     // 大量代码
     ...
     if ( SetupPage() ) {
        // 大量代码
        ...
        if ( AllocMem( &printData ) ) {
           // 大量代码
           ...
        }
     }
   }
}
```

交叉参考

重新测试部分条件以减少复杂度类似于重新测试状态变量。该技术在 goto 中的"错误处理和 goto"中演示。

这个例子是为了显示嵌套级别而设计的。// lots of code 部分旨在暗示例程有足够的代码延伸数个屏幕或跨页打印边界。以下是使用重新测试而不是嵌套重构的代码：

示例 19-28. C++ 中通过重新测试仁慈地去嵌套的代码示例

```cpp
if ( inputStatus == InputStatus_Success ) {
   // 大量代码
   ...
   if ( printerRoutine != NULL ) {
      // 大量代码
      ...
   }
}

if ( ( inputStatus == InputStatus_Success ) &&
   ( printerRoutine != NULL ) && SetupPage() ) {
   // 大量代码
   ...
   if ( AllocMem( &printData ) ) {
      // 大量代码
      ...
   }
}
```

这是一个特别现实的例子，因为它显示了你不能免费减少嵌套级别；为了减少嵌套级别，你必须忍受更复杂的测试。然而，从四级减少到两级对可读性是很大的改进，值得考虑。

**通过使用 break 块来简化嵌套 if**。刚才描述的方法的替代方案是定义一段将作为块执行的代码。如果块中间的某个条件失败，执行会跳转到块的末尾。

示例 19-29. C++ 中使用 break 块的示例

```cpp
do {
   // 开始 break 块
   if ( inputStatus != InputStatus_Success ) {
      break; // 跳出块
   }
   // 大量代码
   ...
   if ( printerRoutine == NULL ) {
      break; // 跳出块
   }

   // 大量代码
   ...
   if ( !SetupPage() ) {
      break; // 跳出块
   }

   // 大量代码
   ...
   if ( !AllocMem( &printData ) ) {
      break; // 跳出块
   }

   // 大量代码
   ...
} while (FALSE); // 结束 break 块
```

这种技术足够不常见，只有当你的整个团队都熟悉它并且它已被团队采用作为可接受的编码实践时才应该使用。

**将嵌套 if 转换为 if-then-else 集**。如果你批判性地思考嵌套 if 测试，你可能会发现你可以重新组织它，使其使用 if-then-else 而不是嵌套 if。假设你有一个灌木决策树像这样：

示例 19-30. Java 中过度生长的决策树示例

```java
if ( 10 < quantity ) {
   if ( 100 < quantity ) {
      if ( 1000 < quantity ) {
         discount = 0.10;
      }
      else {
         discount = 0.05;
      }
   }
   else {
      discount = 0.025;
   }

}
else {
   discount = 0.0;
}
```

这个测试在几个方面组织得很差，其中之一是测试是冗余的。当你测试数量是否大于 1000 时，你不需要同时测试它是否大于 100 和大于 10。因此，你可以重新组织代码：

示例 19-31. Java 中嵌套 if 转换为 if-then-else 集的示例

```java
if ( 1000 < quantity ) {
   discount = 0.10;
}
else if ( 100 < quantity ) {
   discount = 0.05;
}
else if ( 10 < quantity ) {
   discount = 0.025;
}
else {
   discount = 0;
}
```

这个解决方案比其他一些更容易，因为数字整齐地增加。以下是你如何重做数字不那么整齐的嵌套 if：

示例 19-32. Java 中当数字"混乱"时嵌套 if 转换为 if-then-else 集的示例

```java
if ( 1000 < quantity ) {
   discount = 0.10;
}
else if ( ( 100 < quantity ) && ( quantity <= 1000 ) ) {
   discount = 0.05;
}
else if ( ( 10 < quantity ) && ( quantity <= 100 ) ) {
   discount = 0.025;
}
else if ( quantity <= 10 ) {
   discount = 0;
}
```

这段代码和前面代码的主要区别是 else-if 子句中的表达式不依赖先前的测试。这段代码不需要 else 子句就能工作，实际上测试可以按任何顺序执行。代码可以由四个 if 和没有 else 组成。else 版本更可取的唯一原因是它避免不必要的重复测试。

**将嵌套 if 转换为 case 语句**。你可以将某些类型的测试（特别是那些涉及整数的）重新编码为使用 case 语句而不是 if 和 else 链。你不能在某些语言中使用这种技术，但对于那些可以使用的语言来说，这是一种强大的技术。以下是你如何在 Visual Basic 中重新编码这个例子：

示例 19-33. Visual Basic 中将嵌套 if 转换为 case 语句的示例

```vb
Select Case quantity
   Case 0 To 10
      discount = 0.0
   Case 11 To 100
      discount = 0.025
   Case 101 To 1000
      discount = 0.05
   Case Else
      discount = 0.10
End Select
```

这个例子读起来像一本书。当你将它与几页前的两个多重缩进例子进行比较时，它似乎是一个特别干净的解决方案。

**将深度嵌套的代码分解为它自己的例程**。如果深度嵌套发生在循环内部，你通常可以通过将循环内部放入自己的例程来改善情况。如果嵌套是条件和迭代共同造成的结果，这尤其有效。让 if-then-else 分支留在主循环中以显示决策分支，然后将分支内的语句移到它们自己的例程中。这段代码需要这样的修改：

示例 19-34. C++ 中需要分解为例程的嵌套代码示例


新例程中的代码只是从原始例程中提出来形成新例程的。（新例程这里没有显示。）新代码有几个优点。首先，两级嵌套使结构更简单、更容易理解。其次，你可以在一个屏幕上阅读、修改和调试更短的 while 循环——它不需要跨屏幕或打印页边界拆分。第三，将 ProcessDeposit() 和 ProcessWithdrawal() 的功能放入例程会带来模块化的所有其他一般优势。第四，现在很容易看到代码可以分解为 case 语句，这会使它更容易阅读，如下所示：

**使用更面向对象的方法**。在面向对象环境中简化这个特定代码的直接方法是创建一个抽象 Transaction 基类和 Deposit、Withdrawal 和 Transfer 的子类。

示例 19-37. C++ 中使用多态的好代码示例

```cpp
TransactionData transactionData;

while ( !TransactionsComplete() ) {
   // 读取交易记录
   transactionData = ReadTransaction();

   // 根据交易类型创建交易对象
   switch ( transactionData.Type ) {
      case ( TransactionType_Deposit ):

}
```

```cpp
   TransactionData transactionData
   ) {

   // 根据交易类型创建交易对象
   switch ( transactionData.Type ) {
      case ( TransactionType_Deposit ):
         return new Deposit( transactionData );
         break;
      case ( TransactionType_Withdrawal ):
         return new Withdrawal( transactionData );
         break;
      case ( TransactionType_Transfer ):
         return new Transfer( transactionData );
         break;

      default:
         // 处理未知交易类型
         LogTransactionError( "Unknown Transaction Type", transactionData );
         return NULL;
   }
}
```

**重新设计深度嵌套的代码**。一些专家认为，在面向对象编程中，case 语句几乎总是表示代码分解不佳，而且几乎不需要（Meyer 1997）。从调用例程的 case 语句到带有多态方法调用的对象工厂的这种转换就是这样一个例子。

更一般地说，复杂的代码表明你对自己的程序理解得不够好，无法使其简单。深度嵌套是一个警告信号，表明需要分解一个例程或重新设计代码中复杂的部分。这并不意味着你必须修改例程，但如果你不这样做，你应该有一个好的理由。

- 重新测试部分条件（本节）

## 编程基础：结构化编程

"结构化编程"一词源于 Edsger Dijkstra 在1969年 NATO 软件工程会议上发表的里程碑论文"Structured Programming"（Dijkstra 1969）。到结构化编程出现并成为过去时，"结构化"一词已被应用于每个软件开发活动，包括结构化分析、结构化设计和结构化闲聊。各种结构化方法没有任何共同线索，除了它们都是在"结构化"一词给它们额外价值的时代创建的。

结构化编程的核心是一个简单的想法：程序应该只使用一进一出控制构造（也称为单入口、单出口控制构造）。一进一出控制构造是一个代码块，它只有一个可以开始的地方和一个可以结束的地方。它没有其他入口或出口。结构化编程与结构化、自顶向下的设计不相同。它仅适用于详细的编码级别。

结构化程序以有序、纪律严明的方式进行，而不是不可预测地跳来跳去。你可以自上而下阅读它，它的执行方式大致相同。不那么有纪律的方法导致源代码提供对程序在机器中如何执行的不太有意义、更不易读的图片。更少的可读性意味着更少的理解，最终意味着更低的程序质量。

结构化编程的核心概念今天仍然有用，适用于 break、continue、throw、catch、return 和其他主题的考虑。

### 顺序

### 选择

选择是一种控制结构，导致语句被选择性地执行。if-then-else 语句是一个常见例子。要么执行 if-then 子句，要么执行 else 子句，但不是两者都执行。其中一个子句被"选择"执行。

有关使用选择的详细信息，请参见第15章。

case 语句是选择控制的另一个例子。C++ 和 Java 中的 switch 语句和 Visual Basic 中的 select 语句都是 case 的例子。在每种情况下，几个 case 中的一个被选择执行。从概念上讲，if 语句和 case 语句是相似的。如果你的语言不支持 case 语句，你可以用 if 语句模拟它们。这里有两个选择示例：

示例 19-41. Java 中选择的示例

```java
// if 语句中的选择
if ( totalAmount > 0.0 ) {
   // 做某事
   ...
}
else {
   // 做其他事情
   ...
}

// case 语句中的选择
switch ( commandShortcutLetter ) {
   case 'a':
      PrintAnnualReport();
      break;
   case 'q':
      PrintQuarterlyReport();
      break;
   case 's':
      PrintSummaryReport();
      break;
   default:
      DisplayInternalError( "Internal Error 905: Call customer support." );
}
```

```vb

   DoSomething( index )

   DoSomething ( index )
```

结构化编程的核心论点是，任何控制流都可以由顺序、选择和迭代这三种构造创建（Böhm Jacopini 1966）。程序员有时倾向于增加便利性的语言结构，但编程的发展似乎主要是通过限制我们被允许用编程语言做什么来实现的。在结构化编程之前，使用 goto 提供了控制流便利性的极致，但以这种方式编写的代码结果是不可理解和不可维护的。我的信念是，除了三种标准结构化编程构造之外的任何控制结构的使用——即 break、continue、return、throw-catch 等的使用——应该带着批判的眼光看待。

## 控制结构和复杂度

对控制结构给予如此多关注的原因之一是它们是整体程序复杂度的重大贡献者。控制结构的使用不当会增加复杂度；良好的使用会减少它。

"编程复杂度"的一个衡量标准是你必须同时在脑海中记住多少个心理对象才能理解一个程序。这种心理杂耍是编程中最困难的方面之一，也是编程比其他活动需要更多专注的原因。这就是程序员对"快速打断"感到不安的原因——这种打断等于要求一个杂耍演员在空中保持三个球的同时帮你拿杂货。

让事情尽可能简单——但不要过于简单。
— 阿尔伯特·爱因斯坦

直观地说，程序的复杂度似乎在很大程度上决定了理解它所需的努力。Tom McCabe 发表了一篇有影响力的论文，认为程序的复杂度由其控制流定义（1976）。其他研究人员已经确定了 McCabe 的圈复杂度度量之外的因素（比如例程中使用的变量数量），但他们同意控制流至少是复杂度的最大贡献者之一（如果不是最大的话）。

---

## 复杂度有多重要？

计算机科学研究人员至少在二十年前就已经意识到复杂度的重要性。很多年前，Edsger Dijkstra 警告了复杂度的危险："称职的程序员完全意识到自己头骨的严格有限大小；因此，他以完全谦逊的态度接近编程任务"（Dijkstra 1972）。这并不意味着你应该增加头骨的容量来处理巨大的复杂度。它意味着你永远无法处理巨大的复杂度，必须尽一切可能采取步骤减少它。

交叉参考

有关复杂度的更多信息，请参见关键设计概念中的"软件的主要技术 imperative：管理复杂度"。

控制流复杂度很重要，因为它与低可靠性和频繁错误相关（McCabe 1976, Shen et al. 1985）。William T. Ward 报告了在惠普使用 McCabe 复杂度度量导致软件可靠性显著提高（1989b）。McCabe 的度量被用于一个 77,000 行的程序来识别问题区域。该程序的发布后缺陷率为每千行代码 0.31 个缺陷。一个 125,000 行的程序的发布后缺陷率为每千行代码 0.02 个缺陷。Ward 报告说，由于它们的较低复杂度，两个程序在惠普的其他程序中缺陷都明显更少。我自己的公司 Construx Software 在 2000 年代使用复杂度度量来识别有问题的例程时也有类似的结果。

## 减少复杂度的通用指导原则

你可以通过两种方式之一更好地处理复杂度。首先，你可以通过做心理练习来提高自己的心理杂耍能力。但编程本身通常已经是足够的练习，而且人们似乎很难同时处理超过大约五到九个心理实体（Miller 1956）。改进的潜力很小。其次，你可以减少程序的复杂度和理解它们所需的专注力。

## 如何测量复杂度

你可能对你认为什么使例程或多或少复杂有直观的理解。研究人员试图将他们的直观感觉形式化，想出了几种测量复杂度的方法。也许最有影响力的数值技术是 Tom McCabe 的，其中复杂度通过计算例程中"决策点"的数量来测量。表 19-2 描述了一种计算决策点的方法。

> **表 19-2**。计算例程中决策点的技术

1. 从例程的直线路径开始加 1。
2. 为每个以下关键字或其等价物加 1：`if`、`while`、`repeat`、`for`、`and`、`or`
3. 为 case 语句中的每个 case 加 1。

**进一步阅读**

这里描述的方法基于 Tom McCabe 的有影响力的论文"A Complexity Measure"（1976）。

这里有一个例子：

```cpp
if ( ( (status = Success) and done ) or
     ( not done and ( numLines >= maxLines ) ) ) then ...
```

在这个片段中，你数 1 开始，数 2 代表 if，数 3 代表 and，数 4 代表 or，数 5 代表 and。因此，这个片段包含总共五个决策点。

## 如何处理你的复杂度测量

在你数完决策点后，你可以用这个数字来分析例程的复杂度：

| 决策点数量 | 建议 |
|-----------|------|
| 0–5 | 例程可能没问题。 |
| 6–10 | 开始考虑简化例程的方法。 |
| 10+ | 将部分例程分解为第二个例程并从第一个例程调用它。 |

将部分例程移入另一个例程并不会减少程序的总体复杂度；它只是移动决策点。但它减少了你必须同时处理的复杂度数量。既然重要的目标是最小化你必须同时心理杂耍的项目数量，减少给定例程的复杂度是值得的。

10 个决策点的最大值不是绝对限制。使用决策点数量作为警告标志，表明例程可能需要重新设计。不要将其用作僵化的规则。具有许多 case 的 case 语句可能超过 10 个元素长，而且根据 case 语句的目的，将其拆分可能是愚蠢的。

## 其他类型的复杂度

McCabe 的复杂度度量不是唯一合理的度量，但它是计算文献中讨论最多的度量，当你考虑控制流时它特别有帮助。其他度量包括使用的数据量、控制构造中嵌套级别的数量、代码行数、变量连续引用之间的行数（"跨度"）、变量使用中的行数（"存活时间"）以及输入和输出的数量。一些研究人员开发了基于这些更简单度量组合的复合度量。

## 控制结构问题检查表

- [ ] 表达式是否使用 true 和 false 而不是 1 和 0？
- [ ] 布尔值是否隐式与 true 和 false 比较？
- [ ] 数值是否显式与其测试值比较？
- [ ] 表达式是否通过添加新布尔变量、使用布尔函数和决策表来简化？
- [ ] 布尔表达式是否以正面形式陈述？
- [ ] 成对的大括号是否平衡？
- [ ] 是否在大括号需要的地方都使用了它们以保持清晰？
- [ ] 逻辑表达式是否完全加括号？
- [ ] 测试是否以数线顺序编写？
- [ ] Java 测试是否在适当的时候使用 a.equals(b) 样式而不是 a == b？
- [ ] 空语句是否显而易见？
- [ ] 嵌套语句是否通过重新测试部分条件、转换为 if-then-else 或 case 语句、将嵌套代码移入自己的例程、转换为更面向对象的设计或以其他方式改进来简化？
- [ ] 如果例程的决策数超过 10，是否有好的理由不重新设计它？

- 使布尔表达式简单易读对代码质量有很大贡献。
- 深度嵌套使例程难以理解。幸运的是，你可以相对容易地避免它。
- 结构化编程是一个今天仍然相关的简单想法：你可以用顺序、选择和迭代的组合构建任何程序。
- 最小化复杂度是编写高质量代码的关键。
