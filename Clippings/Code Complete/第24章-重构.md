神话：管理良好的软件项目进行有条理的需求开发，并定义程序职责的稳定列表。设计遵循需求，并且做得仔细，以便编码可以线性进行，从头到尾，这意味着大部分代码可以编写一次、测试一次然后遗忘。按照神话的说法，唯一对代码进行重大修改的时间是在软件维护阶段，这是发生在系统初始版本交付之后的事情。

所有成功的软件都会发生变化。

## 现实

现实：代码在初始开发期间会大量演化。许多初始编码期间看到的变化至少与维护期间看到的变化一样引人注目。编码、调试和单元测试在典型项目上消耗30%到65%的努力，取决于项目的大小。（详情见第27章。）如果编码和单元测试是直接的过程，它们在项目总努力中消耗不超过20-30%。然而，即使在管理良好的项目上，需求也会每月变化大约1%到4%（Jones 2000）。需求变化不可避免地会导致相应的代码变化——有时是大量的代码变化。

另一个现实：现代开发实践增加了构建期间代码变化的可能性。在旧的生命周期中，重点——成功与否——是避免代码变化。更现代的方法远离编码可预测性。当前的方法更加以代码为中心，在项目的生命周期中，你可以预期代码比以前更多地演化。

## 软件演化的类型

软件演化类似于生物演化，因为一些突变是有益的，而许多突变不是。良好的软件演化产生的代码，其开发模仿了从猴子到尼安德特人到我们作为软件开发人员当前崇高状态的进化。然而，进化力量有时会从另一个方向猛击程序，将程序打入去进化螺旋。

软件演化类型之间的关键区别在于程序的品质在修改下是提高还是降低。如果你用逻辑胶带和迷信修复错误，质量就会下降。如果你把修改视为收紧程序原始设计的机会，质量就会提高。如果你看到程序质量正在下降，这就像我之前提到的矿井中那只沉默的金丝雀。它警告程序正在错误的方向上演化。

软件演化的第二类区别是构建期间做出的更改和维护期间做出的更改之间的区别。这两类演化在几个方面不同。构建更改通常由原始开发人员做出，通常是在程序被完全遗忘之前。系统还没有上线，所以完成更改的压力只是进度压力——而不是500个愤怒的用户想知道为什么他们的系统宕机了。由于同样的原因，构建期间的更改可以更加自由奔放——系统处于更动态的状态，犯错的代价很低。这些情况意味着一种不同于软件维护期间发现的软件演化风格。

### 软件演化的哲学

程序员处理软件演化的方法中一个常见弱点是它作为一个无意识的过程进行。如果你认识到开发期间的演化是不可避免的和重要的现象并为之计划，你可以利用它为你服务。

代码没有大到、扭曲到或复杂到维护不能使其更糟。
—— Gerald Weinberg

演化既是危险的，也是接近完美的机会。当你必须做出更改时，努力改进代码，使未来的更改更容易。当你开始编写程序时，你知道的比你后来知道的少。当你有机会修改程序时，用你学到的来改进它。在编写你的初始代码和你的更改时都要考虑到未来的更改。

软件演化的基本规则是演化应该改善程序的内部质量。以下章节描述了如何实现这一点。

## 重构简介

实现软件演化的基本规则的关键策略是重构，Martin Fowler将其定义为"对软件内部结构的更改，使其更易于理解和修改而不改变其可观察行为"（Fowler 1999）。现代编程中"重构"一词源于Larry Constantine在结构化编程中最初使用"factoring"一词，指的是尽可能将程序分解为其组成部分（Yourdon and Constantine 1979）。

### 重构的理由

有时代码在维护下会退化，有时代码一开始就不是很好。在任何一种情况下，以下是一些警告信号——有时称为"味道"（Fowler 1999）——表明需要重构的地方：

**代码重复。** 重复的代码几乎总是代表首先没有完全分解设计的失败。重复代码让你陷入平行修改——每当你在一个地方进行更改时，你必须在另一个地方进行平行修改。它还违反了Andrew Hunt和Dave Thomas所说的"DRY原则"：不要重复自己（2000）。我认为David Parnas说得最好："复制和粘贴是一个设计错误"（McConnell 1998b）。

**一个例程太长。** 在面向对象编程中，超过一屏的例程很少需要，通常代表了将结构化编程的脚强行塞入面向对象的鞋子的尝试。

我的一位客户被分配的任务是分解一个遗留系统中最长的例程，该例程超过12000行。经过努力，他能够将最大例程的大小减少到大约4000行。

改进系统的一种方法是增加其模块化——增加数量更多、定义更好、命名更好的例程，每个例程做一件事并做好。当更改导致你重新访问代码的一部分时，抓住机会检查该部分例程的模块化。如果一个例程如果将其一部分做成单独的例程会更干净，那就创建一个单独的例程。

**循环太长或嵌套太深。** 循环内部往往是转换为例程的良好候选，这有助于更好地分解代码并减少循环的复杂性。

**一个类的内聚性差。** 如果你发现一个类承担了一堆不相关职责的所有权，那个类应该被分解成多个类，每个类对一组内聚的职责负责。

**类的接口没有提供一致的抽象级别。** 即使以一致接口开始的类也会随着时间的推移失去其原始一致性。类接口往往会随着在热烈时刻做出的有利于便利性而非接口完整性的修改而改变。最终，类接口变成了一个弗兰肯斯坦式的维护怪物，几乎无助于提高程序的知识可管理性。

**参数列表有太多参数。** 良好分解的程序往往有许多小的、定义良好的例程，不需要大量的参数列表。长的参数列表是一个警告，表明例程接口的抽象没有被充分考虑。

**类内的更改往往是分区的。** 有时一个类有两个或更多不同的职责。当这种情况发生时，你发现自己要么更改类的一部分，要么更改类的另一部分——但很少有更改同时影响类的两部分。这是一个信号，表明类应该沿着独立职责的界限分割成多个类。

**更改需要对多个类进行平行修改。** 我见过一个项目，有一个大约15个类的检查清单，每当添加一种新的输出时都必须修改这些类。当你发现自己经常更改相同的类集时，这表明那些类中的代码可以被重新排列，以便更改只影响一个类。根据我的经验，这是一个难以实现的理想，但它仍然是一个好的目标。

**继承层次结构需要平行修改。** 每次创建一个类的子类时，你都发现自己要创建另一个类的子类，这是一种特殊类型的平行修改，应该加以处理。

**case语句需要平行修改。** 虽然case语句本身并不坏，但如果你发现自己在程序的多个部分对相似的case语句进行平行修改，你应该问一问继承是否可能是更好的方法。

**一起使用的相关数据项没有组织成类。** 如果你发现自己反复操作相同的数据项集，你应该问这些操作是否应该被组合成它们自己的类。

**一个例程使用另一个类的特性比使用它自己类的特性更多。** 这表明例程应该被移动到另一个类中，然后由它的旧类调用。

**原始数据类型被重载。** 原始数据类型可以用来表示无限数量的真实世界实体。如果你的程序使用像整数这样的原始数据类型来表示像money这样的常见实体，考虑创建一个简单的Money类，这样编译器就可以对Money变量执行类型检查，这样你就可以对分配给money的值添加安全检查，等等。如果Money和Temperature都是整数，编译器不会警告你像`bankBalance = recordLowTemperature`这样的错误赋值。

**一个类没有做很多事情。** 有时重构代码的结果是一个旧类没有太多事情要做。如果一个类似乎没有承担它的重量，问问你是否应该将那个类的所有职责分配给其他类并完全消除该类。

**例程链传递跳板数据。** 如果你发现自己将数据传递给一个例程只是为了那个例程可以将它传递给另一个例程，这被称为"跳板数据"（Page-Jones 1988）。这可能是可以的，但要问问你自己传递所讨论的具体数据是否与每个例程接口呈现的抽象一致。如果每个例程的抽象都是可以的，传递数据就是可以的。如果不是，找到一些方法使每个例程的接口更加一致。

**中间人对象没有做任何事情。** 如果你发现一个类中的大部分代码只是将调用转交给其他类中的例程，考虑你是否应该消除中间人并直接调用那些其他类。

**一个类与另一个类过于亲密。** 封装（信息隐藏）可能是让你的程序在知识上可管理并最小化代码更改的连锁反应方面最强大的工具。每当你看到一个类对另一个类的了解超过它应该了解的程度——包括派生类对其父类了解太多——就要在更强封装而非更弱封装方面犯错误。

**一个例程的名称很差。** 如果一个例程的名称很差，在你定义例程的地方更改例程的名称，在你调用它的所有地方更改名称，然后重新编译。虽然现在这样做可能很难，但以后会更难，所以在注意到这是个问题时就尽快做。

**数据成员是公开的。** 在我看来，公开数据成员总是一个坏主意。它们模糊了接口和实现之间的界限，并且固有地违反封装并限制未来的灵活性。强烈考虑将公开数据成员隐藏在访问例程后面。

**子类只使用其父类的例程的一小部分。** 通常这表明创建子类是因为父类恰好包含了它需要的例程，而不是因为子类在逻辑上是父类的后代。考虑通过将子类与其父类的关系从is-a关系切换到has-a关系来实现更好的封装；将父类转换为前子类的成员数据，并且只公开前子类中真正需要的例程。

**注释被用来解释困难的代码。** 注释有重要的角色要扮演，但它们不应该被用作解释糟糕代码的拐杖。古老的智慧是正确的："不要为糟糕的代码编写文档——重写它"（Kernighan and Plauger 1978）。

**使用了全局变量。** 当你重新访问使用全局变量的代码部分时，花时间重新检查它们。自上次访问代码的那部分以来，你可能已经想到了一种避免使用全局变量的方法。因为你现在比最初编写它时对代码不太熟悉，你可能现在觉得全局变量足够令人困惑，你愿意开发一种更清晰的方法。你可能也有更好的感觉如何将全局变量隔离在访问例程中，以及对不这样做的痛苦的更敏锐感觉。下定决心，做出有益的修改。初始编码已经过去足够长的时间，你可以对你的工作保持客观，但又足够近，你将记住你需要正确进行修改的大部分内容。早期修改期间是改进代码的最佳时机。

有关全局变量使用的指南，请参见全局数据。有关全局数据和类数据之间区别的解释，请参见设计构建块：启发式方法中的"被误认为全局数据的类数据"。

**例程在调用前使用设置代码或在调用后使用清理代码。** 像这样的代码应该是一个警告：

```cpp
// 例程调用的糟糕设置和清理代码示例
WithdrawalTransaction withdrawal;       // 1
withdrawal.SetCustomerId( customerId );   |
withdrawal.SetBalance( balance );         |
withdrawal.SetWithdrawalAmount( withdrawalAmount );
withdrawal.SetWithdrawalDate( withdrawalDate );       // 1

ProcessWithdrawal( withdrawal );

customerId = withdrawal.GetCustomerId();       // 2
balance = withdrawal.GetBalance();               |
withdrawalAmount = withdrawal.GetWithdrawalAmount();
withdrawalDate = withdrawal.GetWithdrawalDate();       // 2
```

(1)这个设置代码是一个警告。

(2)这个清理代码是另一个警告。

一个类似的警告信号是当你发现自己为WithdrawalTransaction类创建一个特殊的构造函数，它接受其正常初始化数据的一个子集，这样你就可以编写这样的代码：

```cpp
// 方法调用的糟糕设置和清理代码示例
withdrawal = new WithdrawalTransaction( customerId, balance,
   withdrawalAmount, withdrawalDate );
withdrawal.ProcessWithdrawal();
delete withdrawal;
```

每当你看到为调用例程而设置代码或在调用例程后进行清理的代码时，问问例程接口是否呈现了正确的抽象。在这种情况下，也许应该修改ProcessWithdrawal的参数列表以支持这样的代码：

```cpp
// 不需要设置或清理代码的良好例程示例
ProcessWithdrawal ( customerId, balance, withdrawalAmount, withdrawalDate );
```

请注意，这个例子的反面也呈现了类似的问题。如果你发现自己通常手头有一个WithdrawalTransaction对象，但需要将它的几个值传递给像这里所示的例程，你也应该考虑重构ProcessWithdrawal接口，使其需要WithdrawalTransaction对象而不是其各个字段：

```cpp
// 需要多个方法调用的代码示例
ProcessWithdrawal( withdrawal.GetCustomerId(), withdrawal.GetBalance(),
   withdrawal.GetWithdrawalAmount(), withdrawal.GetWithdrawalDate() );
```

这些方法中的任何一种都是对的，也都是错的——这取决于ProcessWithdrawal()接口的抽象是期望有四个不同的数据片段还是期望有一个WithdrawalTransaction对象。

**程序包含似乎有一天可能需要的代码。** 程序员在猜测什么功能可能有一天需要方面是出了名的糟糕。"提前设计"存在许多可预测的问题：

"提前设计"代码的需求还没有完全开发，这意味着程序员可能会对这些未来需求猜错。"代码超前"工作最终将被丢弃。
如果程序员对未来需求的猜测非常接近，程序员仍然通常不会预期到未来需求的所有复杂性。这些复杂性破坏了程序员的基本设计假设，这意味着"提前设计"工作将不得不被丢弃。
使用"提前设计"代码的未来程序员不知道它是"提前设计"代码，或者他们假设代码比实际工作得更好。他们假设代码已经按照与其他代码相同的级别进行了编码、测试和审查。他们浪费了大量时间构建使用"提前设计"代码的代码，最终才发现"提前设计"代码实际上并不能工作。
额外的"提前设计"代码产生了额外的复杂性，这需要额外的测试、额外的缺陷修正等等。总体效果是减慢项目进度。

专家们一致认为，为未来需求做准备的最好方式不是编写投机性代码；而是使当前需要的代码尽可能清晰和直接，这样未来程序员就会知道它做什么和不做什么，并相应地进行更改（Fowler 1999, Beck 2000）。

- 一个例程太长。

## 不重构的理由

在普通用语中，"重构"被宽松地用来指修复缺陷、添加功能、修改设计——本质上是作为对代码进行任何更改的同义词。不幸的是，这个术语含义的这种常见稀释。改变本身不是美德，但有目的的改变，应用一茶匙的纪律，可以是支持程序在维护下品质稳步提高并防止过于熟悉的软件熵死亡螺旋的关键策略。

## 特定重构

在本节中，我呈现了一个重构目录，其中许多我通过总结《重构》（Fowler 1999）中呈现的更详细描述来描述。然而，我没有试图使这个目录详尽无遗。从某种意义上说，本书中每个展示"糟糕代码"示例和"好代码"示例的案例都是成为重构的候选者。为了节省空间，我专注于我个人发现最有用的重构。

### 数据级重构

以下是可以改善变量和其他数据类型使用的重构。

**用命名常量替换魔术数字。** 如果你使用像3.14这样的数字或字符串字面量，用像PI这样的命名常量替换该字面量。

**用更清晰或更有信息的名称重命名变量。** 如果变量的名称不清楚，将其更改为更好的名称。当然，相同的建议适用于重命名常量、类和例程。

**内联移动表达式。** 用表达式本身替换被赋予表达式结果的中间变量。

**用例程替换表达式。** 用例程替换表达式（通常是为了不在代码中重复该表达式）。

**引入中间变量。** 将表达式赋值给一个中间变量，其名称总结表达式的目的。

**将多用途变量转换为多个单用途变量。** 如果一个变量用于多个目的——常见的罪魁祸首是i、j、temp和x——为每个用法创建单独的变量，每个变量都有一个更具体的名称。

**使用局部变量而不是参数用于局部目的。** 如果一个仅输入的例程参数被用作局部变量，创建一个局部变量并使用它代替。

**将数据原语转换为类。** 如果一个数据原语需要额外行为（包括更严格的类型检查）或额外数据，将其转换为对象并添加你需要的行为。这可以应用于像Money和Temperature这样的简单数值类型。它也可以应用于像Color、Shape、Country或OutputType这样的枚举类型。

**将一组类型码转换为类或枚举。** 在旧程序中，通常会看到这样的关联：

```cpp
const int SCREEN = 0;
const int PRINTER = 1;
const int FILE = 2;
```

不是定义独立常量，而是创建一个类，这样你就可以获得更严格的类型检查的好处，并让自己准备好在需要时为OutputType提供更丰富的语义。创建枚举有时是创建类的良好替代方案。

**将一组类型码转换为带子类的类。** 如果与不同类型关联的不同元素可能有不同的行为，考虑为该类型创建一个基类，并为每个类型码创建子类。对于OutputType基类，你可以创建像Screen、Printer和File这样的子类。

**将数组更改为对象。** 如果你使用的数组中不同元素是不同类型的，创建一个对象，该对象为数组的每个前元素都有一个字段。

**封装集合。** 如果一个类返回一个集合，集合的多个实例四处漂浮会产生同步困难。考虑让类返回一个只读集合，并提供例程来从集合中添加和删除元素。

**用数据类替换传统记录。** 创建一个包含记录成员的类。创建类允许你集中错误检查、持久性和与记录有关的其他操作。

### 语句级重构

以下是可以改善单个语句使用的重构。

**分解布尔表达式。** 引入命名良好的中间变量来简化布尔表达式，这些变量有助于记录表达式的含义。

**将复杂布尔表达式移动到命名良好的布尔函数中。** 如果表达式足够复杂，这个重构可以提高可读性。如果表达式被使用多次，它消除了平行修改的需要并减少了使用表达式时出错的机会。

**合并条件不同部分中重复的片段。** 如果你在else块的末尾有与if块的末尾相同的代码行，将那些代码行移动到它们出现在整个if-then-else块之后。

**使用break或return而不是循环控制变量。** 如果你在循环中有一个像done这样的变量用于控制循环，使用break或return退出循环代替。

**一旦知道答案就返回，而不是在嵌套的if-then-else语句中赋值返回值。** 如果你一旦知道返回值就退出例程，代码通常最容易阅读且最不容易出错。在知道你已经知道答案后设置返回值然后通过大量逻辑展开的方式的替代方案可能更难遵循。

**用多态性替换条件（特别是重复的case语句）。** 结构化程序中曾经包含在case语句中的大量逻辑可以烘焙到继承层次结构中，并通过多态例程调用来实现。

**创建并使用空对象而不是测试空值。** 有时一个空对象会有与之关联的通用行为或数据，比如将不知道姓名的居民称为"occupant"。在这种情况下，考虑将处理空值的责任从客户端代码转移到类中——也就是说，让Customer类将未知居民定义为"occupant"，而不是让Customer的客户端代码反复测试客户的姓名是否已知并在不知道时替换"occupant"。

### 例程级重构

以下是可以改善单个例程级别代码的重构。

**提取例程/提取方法。** 从一个例程中移除内联代码，并将其变成它自己的例程。

**内联移动例程的代码。** 从一个身体简单且不言自明的例程中取出代码，并将该例程的代码内联到它被使用的地方。

**将长例程转换为类。** 如果一个例程太长，有时将其转换为一个类，然后将前例程进一步分解为多个例程将提高可读性。

**用简单算法替换复杂算法。** 用更简单的算法替换复杂的算法。

**添加参数。** 如果例程需要从其调用者那里获取更多信息，添加一个参数以便可以提供该信息。

**移除参数。** 如果例程不再使用参数，将其移除。

**分离查询操作和修改操作。** 通常，查询操作不会改变对象的状态。如果像GetTotals()这样的操作改变了对象的状态，将查询功能与状态更改功能分离，并提供两个单独的例程。

**通过参数化合并相似的例程。** 两个相似的例程可能只是在例程内使用的常量值方面不同。将例程合并为一个例程，并将要使用的值作为参数传入。

**分离行为依赖于传入参数的例程。** 如果一个例程根据输入参数的值执行不同的代码，考虑将例程分解为可以单独调用而无需传入那个特定输入参数的单独例程。

**传递整个对象而不是特定字段。** 如果你发现自己将同一对象的几个值传递给一个例程，考虑更改例程的接口使其接收整个对象代替。

**传递特定字段而不是整个对象。** 如果你发现自己创建一个对象只是为了将它传递给一个例程，考虑修改例程使其接收特定字段而不是整个对象。

**封装向下转换。** 如果一个例程返回一个对象，它通常应该返回它所知道的最具体的对象类型。这特别适用于返回迭代器、集合、集合元素等的例程。

### 类实现重构

以下是在类级别改进的重构。

**将值对象更改为引用对象。** 如果你发现自己创建和维护大量大型或复杂对象的副本，更改你对这些对象的使用，使只有一个主副本存在（值对象），其余代码使用对该对象的引用（引用对象）。

**将引用对象更改为值对象。** 如果你发现自己对小型或简单对象执行大量引用管理，更改你对这些对象的使用，使所有对象都是值对象。

**用数据初始化替换虚例程。** 如果你有一组仅根据它们返回的常量值而不同的子类，与其不是在派生类中覆盖成员例程，不如让派生类用适当的常量值初始化类，然后在基类中有使用这些值的通用代码。

**更改成员例程或数据放置。** 考虑在继承层次结构中进行几个一般的更改。这些更改通常是为了消除派生类中的重复：

- 将例程拉到其超类中。
- 将字段拉到其超类中。
- 将构造函数体拉到其超类中。

其他更改通常是为了支持派生类中的专门化：

- 将例程下推到其派生类中。
- 将字段下推到其派生类中。
- 将构造函数体下推到其派生类中。

**将专门化代码提取到子类中。** 如果一个类的代码仅被其实例的一个子集使用，将该专门化代码移动到它自己的子类中。

**将相似代码合并到超类中。** 如果两个子类有相似代码，将该代码合并并移动到超类中。

### 类接口重构

以下是可以使类接口更好的重构。

**将例程移动到另一个类中。** 在目标类中创建一个新例程，并将例程的主体从源类移动到目标类。然后你可以从旧例程调用新例程。

**将一个类转换为两个。** 如果一个类有两个或更多不同的职责区域，将类分解为多个类，每个类都有明确定义的职责。

**消除一个类。** 如果一个类没有做很多事情，将其代码移动到其他更有内聚性的类中并消除该类。

**隐藏委托。** 有时类A调用类B和类C，而实际上类A应该只调用类B，类B应该调用类C。问问你自己A与B交互的正确抽象是什么。如果B应该负责调用C，让B调用C。

**移除中间人。** 如果类A调用类B，类B调用类C，有时候让类A直接调用类C效果更好。你是否应该委托给类B取决于什么最能维护类B接口的完整性。

**用委托替换继承。** 如果一个类需要使用另一个类但想对其接口有更多控制，将超类作为前子类的字段，然后暴露一组将提供内聚抽象的例程。

**用继承替换委托。** 如果一个类暴露了委托类（成员类）的每个公共例程，改为从委托类继承而不仅仅是使用该类。

**引入外来例程。** 如果一个类需要一个额外的例程而你无法修改该类来提供它，你可以在客户端类中创建一个提供该功能的新例程。

**引入扩展类。** 如果一个类需要几个额外的例程而你无法修改该类，你可以创建一个新类，将不可修改类的功能与额外功能组合在一起。你可以通过子类化原始类并添加新例程，或者通过包装类并暴露你需要的例程来做到这一点。

**封装公开的成员变量。** 如果成员数据是公开的，将成员数据更改为私有，并通过例程暴露成员数据的值代替。

**移除不可更改字段的Set()例程。** 如果一个字段应该在对象创建时设置且之后不应更改，在对象的构造函数中初始化该字段，而不是提供一个误导性的Set()例程。

**隐藏不打算在类外使用的例程。** 如果没有某个例程，类接口会更加内聚，那就隐藏该例程。

**封装未使用的例程。** 如果你发现自己通常只使用类接口的一部分，为该类创建一个新接口，只暴露那些必要的例程。确保新接口提供了一个内聚的抽象。

**如果超类和子类的实现非常相似，则合并它们。** 如果子类没有提供太多专门化，将其合并到其超类中。

### 系统级重构

以下是在整个系统级别改进的重构。

**为你无法控制的数据创建明确的参考源。** 有时你有由系统维护的数据，你无法方便或一致地从其他需要了解该数据的对象访问。常见示例是在GUI控件中维护的数据。在这种情况下，你可以创建一个镜像GUI控件中数据的类，然后让GUI控件和其他代码都将该类作为该数据的明确来源。

**将单向类关联更改为双向类关联。** 如果你有两个类需要相互使用彼此的功能，但只有一个类可以知道另一个类，更改类使它们都知道彼此。

**将双向类关联更改为单向类关联。** 如果你有两个类知道彼此的功能，但只有一个类真正需要知道另一个，更改类使一个知道另一个但反之不然。

**提供工厂方法而不是简单构造函数。** 当你需要基于类型码创建对象时，或者当你想使用引用对象而不是值对象时，使用工厂方法（例程）。

**用异常替换错误码或反之亦然。** 根据你的错误处理策略，确保代码使用标准方法。

## 安全重构

重构是提高代码质量的强大技术。像所有强大的工具一样，如果使用不当，重构弊大于利。一些简单的指南可以防止重构失误。

打开一个工作的系统更像是打开人脑并更换神经，而不是打开水槽并更换垫圈。如果维护被称为"软件脑外科手术"会不会更容易？
—— Gerald Weinberg

**保存你开始的代码。** 在你开始重构之前，确保你可以回到你开始的代码。在修订控制系统中保存一个版本，或将正确的文件复制到备份目录。

**保持重构小。** 有些重构比其他的大，"一次重构"的确切构成可能有点模糊。保持重构小，这样你就完全理解你所做更改的所有影响。《重构》（Fowler 1999）中描述的详细重构提供了许多如何做到这一点的好例子。

**一次做一个重构。** 有些重构比其他更复杂。对于除了最简单的重构之外的所有重构，一次做一个重构，在重构后重新编译和重新测试，然后再做下一个。

**列出你打算采取的步骤。** 伪代码编程过程的自然扩展是列出将让你从A点到达B点的重构列表。制作列表帮助你将每个更改保持在上下文中。

**制作一个停车场。** 当你在一个重构中途时，你有时会发现你需要另一个重构。在那个重构中途，你发现第三个重构是有益的。对于不需要立即进行的更改，制作一个"停车场"，一个你希望在某个时候进行但不需要现在进行的更改列表。

**经常制作检查点。** 当你重构时，代码很容易突然走向歪路。除了保存你开始的代码之外，在重构会话的各个步骤保存检查点，这样如果代码把你引入了死胡同，你可以回到一个工作的程序。

**使用编译器警告。** 小的错误很容易溜过编译器。将编译器设置为可能的最高警告级别将有助于在你输入它们后几乎立即捕获许多错误。

**重新测试。** 对更改代码的审查应该辅以重新测试。当然，这取决于首先有好的测试用例集。回归测试和其他测试主题在第22章中有更详细的描述。

**添加测试用例。** 除了用旧测试重新测试之外，添加新的单元测试来执行新代码。移除任何已被重构使过时的测试用例。

**审查更改。** 如果审查在第一次时很重要，它们在后续修改期间就更重要了。Ed Yourdon报告说，程序员在第一次尝试进行更改时通常有超过50%的出错机会（Yourdon 1986b）。有趣的是，如果程序员处理代码的大部分而不是仅仅几行，进行正确修改的机会会提高，如图24-1所示。具体地说，当更改的行数从一行增加到五行时，做坏更改的机会增加。之后，做坏更改的机会减少。

程序员随意对待小更改。他们不检查它们，他们不让别人审查它们，他们有时甚至不运行代码来验证修复是否正常工作。

寓意很简单：像对待复杂更改一样对待简单的更改。一个组织为一个更改引入审查，发现其错误率从审查前的55%降至审查后的2%（Freedman and Weinberg 1982）。一个电信组织从审查代码更改前的86%正确提高到审查后的99.6%（Perrott 2004）。

**根据重构的风险级别调整你的方法。** 有些重构比其他风险更大。像"用命名常量替换魔术数字"这样的重构相对无风险。涉及类或例程接口更改、数据库模式更改或布尔测试更改等的重构往往更有风险。对于更简单的重构，你可能会简化你的重构过程，一次做多个重构，简单地重新测试，而不经过正式审查。

对于风险更大的重构，要在谨慎方面犯错误。一次做一个重构。让别人审查重构或使用结对编程进行该重构，除了正常的编译器检查和单元测试。

### 不适合重构的时机

重构是一项强大的技术，但它不是万能药，而且会受到一些特定类型的滥用。

**不要使用重构作为代码和修复的掩护。** 重构最糟糕的问题是如何被滥用。程序员有时会说他们在重构，而他们真正做的只是调整代码，希望找到一种方法使其工作。重构指的是对工作代码的更改，不影响程序的行为。调整损坏代码的程序员不是在重构；他们是在黑客攻击。

不要部分编写一个功能并打算稍后通过重构使其完整。
—— John Manzo

**避免用重构代替重写。** 有时代码不需要小的更改——它需要被扔掉，这样你就可以重新开始。如果你发现自己处于一个大型重构会话中，问问你自己是否应该从底层重新设计和重新实现代码的那一部分。

大型重构是灾难的配方。
—— Kent Beck

## 重构策略

对任何特定程序有益的重构数量本质上是无限的。重构与编程活动的回报递减规律相同，80/20规则适用。将你的时间花在提供80%收益的重构的20%上。在决定哪些重构最重要时，考虑以下指南：

**当你添加例程时重构。** 当你添加例程时，检查相关例程是否组织良好。如果没有，重构它们。

**当你添加类时重构。** 添加一个类通常会将现有代码的问题推到前面。利用这段时间作为机会重构与你添加的类密切相关的其他类。

**当你修复缺陷时重构。** 利用你从修复bug中学到的知识来改进可能有类似缺陷的其他代码。

**针对易出错的模块。** 有些模块比其他模块更容易出错和脆弱。是否有你和团队中其他人都害怕的代码部分？那可能是一个易出错的模块。虽然大多数人的自然倾向是避免这些具有挑战性的代码部分，但针对这些部分进行重构可能是更有效的策略之一（Jones 2000）。

**针对高复杂度模块。** 另一种方法是关注复杂度评级最高的模块。（参见控制结构和复杂度中的"如何测量复杂度"以获取有关这些指标的详细信息。）一项经典研究发现，当维护程序员将改进工作集中在复杂度最高的模块上时，程序质量显著提高（Henry and Kafura 1984）。

**在维护环境中，改进你接触的部分。** 从未修改的代码不需要重构。但当你确实接触一段代码时，确保你离开时它比你发现时更好。

**在干净代码和丑陋代码之间定义一个接口，然后移动代码穿过接口。** "现实世界"往往比你想要的更混乱。混乱可能来自复杂的业务规则、硬件接口或软件接口。老化系统的一个常见问题是写得糟糕的生产代码，必须始终保持运行。

恢复老化生产系统的一种有效策略是将一些代码指定为在混乱的现实世界中，一些代码指定为在理想的新世界中，一些代码指定为两者之间的接口。图24-2说明了这个想法。

图24-2。你的代码不必仅仅因为现实世界是混乱的就变得混乱。将你的系统构想为理想代码、从理想代码到混乱现实世界的接口以及混乱现实世界的组合。

当你使用系统时，你可以开始将代码穿过"现实世界接口"移动到更有组织的理想世界。当你开始使用遗留系统时，写得糟糕的遗留代码可能几乎构成整个系统。一个效果很好的策略是，每当你接触一段混乱的代码，你就必须将其提升到当前编码标准，给它清晰的变量名，等等——有效地将其移动到理想世界。随着时间的推移，这可以提供代码库的快速改进，如图24-3所示。

图24-3。改进生产代码的一种策略是，在你接触时重构写得糟糕的遗留代码，将其移动到"混乱现实世界接口"的其他一侧。

## 安全重构

- 每个更改都是系统更改策略的一部分吗？
- 在开始重构之前，你保存了开始的代码吗？
- 你保持每个重构小吗？
- 你一次做一个重构吗？
- 你列出了你打算在重构期间采取的步骤清单吗？
- 你有一个停车场，以便记住在重构中途想到的想法吗？
- 你在每个重构后重新测试了吗？
- 如果更改复杂或影响关键任务代码，是否已审查？
- 你是否考虑了特定重构的风险性并相应地调整了方法？
- 更改是否增强了程序的内部质量而不是降低它？
- 你是否避免了将重构用作代码和修复的掩护或作为不重写糟糕代码的借口？

## 关键要点

- 程序更改在初始开发期间和初始发布后都是不可避免的。
- 软件在更改时可以改进或退化。软件演化的基本规则是内部质量应该随着代码演化而提高。
- 重构成功的关键之一是学会关注表明需要重构的众多警告信号或"味道"。
- 重构成功的另一个关键是学习许多特定的重构。
- 成功的最后一个关键是拥有安全重构的策略。有些重构方法比其他方法更好。
- 开发期间的重构是你改进程序的机会，是你会希望你第一次就做的所有更改的机会。在开发期间利用这些机会！
