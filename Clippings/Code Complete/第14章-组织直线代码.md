本章从以数据为中心的编程视角转向以语句为中心的视角。它介绍最简单的控制流类型：按顺序放置语句和语句块。

虽然组织直线代码是一个相对简单的任务，但一些组织细微之处会影响代码质量、正确性、可读性和可维护性。

## 必须按特定顺序排列的语句

最容易排序的顺序语句是顺序重要的那些。以下是一个例子：

示例 14-1. 语句顺序重要的 Java 示例

```java
data = ReadData();
results = CalculateResultsFromData( data );
PrintResults( results );
```

除非这段代码发生了什么神秘的事情，否则语句必须按所示顺序执行。必须先读取数据才能计算结果，必须先计算结果才能打印结果。

这个例子背后的概念是依赖性。第三条语句依赖第二条，第二条依赖第一条。在这个例子中，一条语句依赖另一条的事实从例程名称可以明显看出。在下面的代码片段中，依赖关系不太明显：

示例 14-2. 语句顺序重要但不太明显的 Java 示例

```java
revenue.ComputeMonthly();
revenue.ComputeQuarterly();
revenue.ComputeAnnual();
```

在这种情况下，季度收入计算假设月度收入已经被计算过了。对会计的熟悉——或者甚至常识——可能会告诉你，季度收入必须在年度收入之前计算。这里有依赖关系，但它从阅读代码来看并不明显。而且在这里，依赖关系不明显——它们是字面上隐藏的：

示例 14-3. 顺序依赖被隐藏的 Visual Basic 示例

```vb
ComputeMarketingExpense
ComputeSalesExpense
ComputeTravelExpense
ComputePersonnelExpense
DisplayExpenseSummary
```

假设 `ComputeMarketingExpense()` 初始化了所有其他例程放入数据的类成员变量。在这种情况下，它需要在其他例程之前被调用。从阅读这段代码中你怎么知道呢？因为例程调用没有任何参数，你可能可以猜测这些例程中的每一个都访问类数据。但是从阅读这段代码你无法确定。

当语句有需要你按特定顺序放置的依赖关系时，采取步骤使依赖关系明显。以下是排序语句的一些简单指导原则：

**组织代码使依赖关系明显**。在刚才呈现的 Microsoft Visual Basic 示例中，`ComputeMarketingExpense()` 不应该初始化类成员变量。例程名称表明 `ComputeMarketingExpense()` 与 `ComputeSalesExpense()`、`ComputeTravelExpense()` 和其他例程相似，只是它处理营销数据而不是销售数据或其他数据。让 `ComputeMarketingExpense()` 初始化成员变量是你应该避免的任意做法。为什么初始化应该在那个例程而不是其他两个例程中完成？除非你能想出一个好的理由，否则你应该编写另一个例程 `InitializeExpenseData()` 来初始化成员变量。例程名称清楚地表明它应该在其他费用例程之前被调用。

**命名例程使依赖关系明显**。在 Visual Basic 示例中，`ComputeMarketingExpense()` 被错误命名，因为它做的不仅仅是计算营销费用；它还初始化成员数据。如果你反对创建额外的例程来初始化数据，至少给 `ComputeMarketingExpense()` 一个描述它执行的所有功能的名称。在这种情况下，`ComputeMarketingExpenseAndInitializeMemberData()` 是一个合适的名称。你可能会说它是一个很糟糕的名称，因为它太长了，但名称描述了例程做什么，并不糟糕。例程本身才是糟糕的！

**使用例程参数使依赖关系明显**。同样在 Visual Basic 示例中，因为没有数据在例程之间传递，你不知道这些例程中的任何一个是否使用相同的数据。通过重写代码使数据在例程之间传递，你建立了一个执行顺序重要的线索。新的代码看起来是这样的：

示例 14-4. 暗示顺序依赖的数据的 Visual Basic 示例

```vb
InitializeExpenseData( expenseData )
ComputeMarketingExpense( expenseData )
ComputeSalesExpense( expenseData )
ComputeTravelExpense( expenseData )
ComputePersonnelExpense( expenseData )
DisplayExpenseSummary( expenseData )
```

因为所有例程都使用 `expenseData`，你有了一个它们可能处理相同数据的暗示，并且语句的顺序可能很重要。

在这个特定的例子中，一个更好的方法可能是将例程转换为将 `expenseData` 作为输入并返回更新后的 `expenseData` 作为输出的函数，这使得代码包含顺序依赖更加清楚。

示例 14-5. 暗示顺序依赖的数据和例程调用的 Visual Basic 示例

```vb
expenseData = InitializeExpenseData( expenseData )
expenseData = ComputeMarketingExpense( expenseData )
expenseData = ComputeSalesExpense( expenseData )
expenseData = ComputeTravelExpense( expenseData )
expenseData = ComputePersonnelExpense( expenseData )
DisplayExpenseSummary( expenseData )
```

数据也可以表明执行顺序不重要，如下例：

示例 14-6. 不暗示顺序依赖的数据的 Visual Basic 示例

```vb
ComputeMarketingExpense( marketingData )
ComputeSalesExpense( salesData )
ComputeTravelExpense( travelData )
ComputePersonnelExpense( personnelData )
DisplayExpenseSummary( marketingData, salesData, travelData, personnelData )
```

因为前四行的例程没有任何共同数据，代码暗示它们被调用的顺序不重要。因为第五行的例程使用前四个例程中每个的数据，你可以假设它需要在前四个例程之后执行。

**用注释文档化不明确的依赖关系**。首先尝试编写没有顺序依赖的代码。第二个尝试编写使依赖关系明显的代码。如果你仍然担心顺序依赖不够明确，请用注释记录。文档化不明确的依赖关系是文档化编码假设的一个方面，这对编写可维护、可修改的代码至关重要。在 Visual Basic 示例中，沿这些行的注释会有所帮助：

示例 14-7. 顺序依赖被隐藏但用注释澄清的 Visual Basic 示例

```vb
' Compute expense data. Each of the routines accesses the
' member data expenseData. DisplayExpenseSummary
' should be called last because it depends on data calculated
' by the other routines.
InitializeExpenseData
ComputeMarketingExpense
ComputeSalesExpense
ComputeTravelExpense
ComputePersonnelExpense
DisplayExpenseSummary
```

这段代码没有使用使顺序依赖明显的技巧。最好依赖这种技巧而不是注释，但如果你维护严格控制的代码或者由于其他原因无法改进代码本身，使用文档来弥补代码弱点。

**用断言或错误处理代码检查依赖关系**。如果代码足够关键，你可能使用状态变量和错误处理代码或断言来文档化关键的顺序依赖关系。例如，在类的构造函数中，你可以将类成员变量 `isExpenseDataInitialized` 初始化为 false。然后在 `InitializeExpenseData()` 中，你可以将 `isExpenseDataInitialized` 设置为 true。然后每个依赖 `expenseData` 被初始化的函数可以在对 `expenseData` 执行其他操作之前检查 `isExpenseDataInitialized` 是否已设置为 true。根据依赖关系的范围，你还可能需要像 `isMarketingExpenseComputed`、`isSalesExpenseComputed` 等变量。

这种技术创建新变量、新初始化代码和新错误检查代码，所有这些都创造了额外的错误可能性。这项技术的好处应该与其创造的额外复杂性和次要错误增加的可能性相权衡。

## 顺序无关的语句

你可能遇到一些情况，似乎几条语句或几个代码块的顺序完全无关。一条语句不依赖另一条语句，或者逻辑上不跟随另一条语句。但是排序会影响可读性、性能和可维护性，在没有执行顺序依赖的情况下，你可以使用次要标准来确定语句或代码块的顺序。指导原则是邻近原则：保持相关操作在一起。

---

## 使代码从上到下阅读

作为一个一般原则，使程序从上到下阅读而不是跳来跳去。专家一致认为，自上而下的顺序最能促进可读性。简单地使控制流在运行时从上到下是不够的。如果阅读你代码的人必须搜索整个程序才能找到需要的信息，你应该重新组织代码。以下是一个例子：

示例 14-8. 跳来跳去的糟糕代码的 C++ 示例

```cpp
MarketingData marketingData;
SalesData salesData;
TravelData travelData;

travelData.ComputeQuarterly();
salesData.ComputeQuarterly();
marketingData.ComputeQuarterly();

salesData.ComputeAnnual();
marketingData.ComputeAnnual();
travelData.ComputeAnnual();

salesData.Print();
travelData.Print();
marketingData.Print();
```

假设你想确定 `marketingData` 是如何计算的。你必须从最后一行开始，将所有 `marketingData` 引用跟踪回第一行。`marketingData` 只在少数其他地方使用，但你必须记住在它第一次和最后一次引用之间所有地方如何使用 `marketingData`。换句话说，你必须查看并思考这个片段中的每一行代码才能弄清楚 `marketingData` 是如何计算的。当然，这个例子比你在实际系统中看到的代码简单。以下是组织得更好的相同代码：

示例 14-9. 从上到下阅读的好的顺序代码的 C++ 示例

```cpp
MarketingData marketingData;
marketingData.ComputeQuarterly();
marketingData.ComputeAnnual();
marketingData.Print();

SalesData salesData;
salesData.ComputeQuarterly();
salesData.ComputeAnnual();
salesData.Print();

TravelData travelData;

travelData.ComputeQuarterly();
travelData.ComputeAnnual();
travelData.Print();
```

这段代码在几个方面更好。每个对象的引用保持在一起；它们是"本地化的"。对象"存活"的代码行数很少。也许最重要的是，代码现在看起来可以分解为用于营销、销售和旅行数据的单独例程。第一个代码片段没有暗示这样的分解是可能的。

## 将相关语句分组

将相关语句放在一起。它们可能相关，因为它们操作相同的数据、执行类似的任务，或者依赖于彼此按顺序执行。

交叉参考

如果你遵循伪代码编程过程，你的代码将自动分组为相关语句。关于该过程的详细信息，请参见第9章。

测试相关语句是否分组良好的一个简单方法是打印出例程列表，然后在相关语句周围画方框。如果语句排序良好，你会得到如图 14-1 所示的图片，方框不重叠。

> **图 14-2**。如果代码组织不好，方框画在相关部分周围会重叠

## 关键点

- 组织直线代码最强的原则是排序依赖关系。
- 依赖关系应该通过使用好的例程名称、参数列表、注释来明显化——如果代码足够关键——以及内部变量。
- 如果代码没有顺序依赖，将相关语句尽可能放在一起。
