## 鼓励良好编码

因为代码是构建的主要输出，管理构建的一个关键问题是"你如何鼓励良好的编码实践？"通常，从管理职位强制执行一套严格的技术标准并不是一个好主意。程序员倾向于将管理者视为处于技术进化的较低层次，位于单细胞生物和冰河时代灭绝的长毛象之间的某个位置，而且如果要有编程标准，程序员需要认同这些标准。

如果项目上有人要定义标准，让一位受尊敬的架构师来定义标准，而不是经理。软件项目既是按"权威层级"运作，也是按"专业层级"运作。如果架构师被视为项目的思想领袖，项目团队通常会遵循该人设定的标准。

如果你选择这种方法，确保架构师确实受到尊敬。有时候项目架构师只是一个资历太深、与生产编码问题脱节的资深人员。程序员会厌恶那种与他们的工作脱节的"架构师"定义的标准。

### 设定标准时的注意事项

标准在某些组织中比其他组织更有用。一些开发者欢迎标准，因为它们减少了项目中的任意差异。如果你的团队抵制采用严格的标准，考虑一些替代方案：灵活的指南，而不是指南的建议集合，或体现最佳实践的示例集合。

### 鼓励良好编码的技术

本节描述了几种实现良好编码实践的技术，这些技术比设定 rigid 编码标准要轻量得多：

为项目的每个部分指派两个人。如果两个人必须在每一行代码上合作，你将保证至少有两个人认为它能工作且可读。双人合作机制可以从结对编程到师徒对到伙伴系统审查。

审查每一行代码。代码审查通常涉及程序员和至少两名审查者。这意味着至少有三人阅读每一行代码。同行评审的另一个名称是"同伴压力"。除了在原始程序员离开项目时提供安全网外，审查还会提高代码质量，因为程序员知道代码会被其他人阅读。即使你的团队没有创建明确的编码标准，审查也提供了一种向小组编码标准靠拢的微妙方式——决策由小组在审查期间做出，随着时间推移，小组会形成自己的标准。

要求代码签核。在其他领域，技术图纸由管理工程师批准和签署。签名意味着，据工程师所知，图纸在技术上是胜任且无错误的。一些公司以同样的方式对待代码。在代码被认为完成之前，资深技术人员必须在代码清单上签字。

路由良好代码示例以供审查。良好管理的一个重要部分是清楚地传达你的目标。传达你的目标的一种方式是向你的程序员传阅良好代码或将其公开展示。这样做时，你提供了你所追求的质量的明确示例。同样，编码标准手册可以主要由一组"最佳代码清单"组成。将某些清单标识为"最佳"为其他人树立了追随的榜样。这样的手册比英语标准手册更容易更新，而且它毫不费力地呈现了编码风格中难以在散文描述中逐点捕捉的细微之处。

强调代码清单是公共资产。程序员有时会感觉他们编写的代码是"他们的代码"，就好像它是私有财产。虽然它是他们工作的结果，但代码是项目的一部分，应该可供项目中任何其他需要它的人自由使用。它应该在审查和维护期间被其他人看到，即使在其他时候没有。

有史以来最成功的项目之一在11人年的努力中开发了83000行代码。在运行的前13个月中，只检测到一个导致系统失败的错误。当你知道该项目是在20世纪60年代末完成的，没有在线编译或交互式调试时，这一成就更加引人注目。该项目的生产力——在20世纪60年代末每年7500行代码——按今天的标准仍然令人印象深刻。该项目的首席程序员报告说，项目成功的关键之一是将所有计算机运行（包括错误和其他）识别为公共资产而非私有资产（Baker and Mills 1973）。这个想法已经延伸到现代背景，包括开源软件（Raymond 2000）和极限编程的集体所有权理念（Beck 2000），以及其他背景。

奖励良好代码。使用你组织的奖励系统来强化良好的编码实践。在开发你的强化系统时牢记以下注意事项：

奖励应该是程序员想要的东西。（许多程序员觉得"表扬"类的奖励令人反感，特别是当它们来自非技术经理时。）

获得奖励的代码应该非常好。如果你给一个每个人都知道工作做得不好的程序员奖励，你看起来就像 Homer Simpson 试图经营核反应堆。程序员有合作态度或总是准时上班并不重要。如果你的奖励与情况的技术优点不匹配，你就会失去信誉。如果你的技术不够好到可以做出良好代码的判断，就不要做！完全不要颁奖，或者让你的团队选择获奖者。

一个简单的标准。如果你正在管理一个编程项目，而你又有编程背景， elicit 良好工作的一种简单有效的方法是说"我必须能够阅读和理解为项目编写的任何代码。"经理不是最热门的的技术高手这一点可能是一个优势，因为它可能会阻止"聪明"或棘手的代码。

### 本书的作用

本书的大部分内容都是关于良好编程实践的讨论。它不是用来支持 rigid 标准的，也不打算用作一套 rigid 标准。将本书用作讨论的基础、良好编程实践的参考书，以及识别可能对你的环境有益的实践。

## 配置管理

软件项目是动态的。代码会变，设计会变，需求也会变。而且，需求的变化会导致设计中更多的变化，设计的变化会导致代码和测试用例中更多的变化。

### 什么是配置管理？

配置管理是识别项目工件并系统处理变更的实践，以便系统能够随时间保持其完整性。它的另一个名称是"变更控制"。它包括评估拟议变更、跟踪变更以及在系统存在的各个时间点保留副本的技术。

如果你不控制需求的变更，你可能会最终为最终被淘汰的系统部分编写代码。你可以编写与系统新部分不兼容的代码。你可能不会检测到许多不兼容性，直到集成阶段，那将变成互相推诿的时候，因为没有人真正知道发生了什么。

如果代码变更不受控制，你可能会更改某人同时正在更改的例程；成功地合并你的变更和他们的变更将是有问题的。不受控制的代码变更可能使代码看起来比实际经过更多测试。已经测试的版本可能是旧的、未更改的版本；修改后的版本可能没有经过测试。如果没有良好的变更控制，你可以更改例程，发现新错误，并且无法恢复到旧的、可工作的例程。

问题会无限期地继续下去。如果变更不是系统处理的，你就是在雾中随机迈步，而不是直接走向清晰的目的地。没有良好的变更控制，你不是在开发代码，而是在浪费时间。配置管理帮助你有效地利用时间。

尽管配置管理有明显需要，但许多程序员几十年来一直在回避它。20多年前的一项调查发现，超过三分之一的程序员甚至不熟悉这个想法（Beck and Perkins 1983），几乎没有迹象表明这种情况已经改变。软件工程研究所的一项更近期研究发现，在使用非正式软件开发实践的组织中，只有不到20%有足够的配置管理（SEI 2003）。

配置管理不是由程序员发明的，但由于编程项目非常不稳定，它对程序员特别有用。应用于软件项目时，配置管理通常被称为"软件配置管理"（SCM）。SCM专注于程序的需求、源代码、文档和测试数据。

SCM的系统问题是过度控制。阻止汽车事故的最确定方法是阻止所有人驾驶，而阻止软件开发问题的一个确定方法是停止所有软件开发。虽然这是控制变更的一种方式，但这是一种可怕的软件开发方式。你必须仔细计划 SCM，使其成为资产而不是脖子上的海鸟。

在小型单人项目上，除了计划非正式的定期备份外，你可能不需要任何 SCM 就做得很好。尽管如此，配置管理仍然有用（事实上，我在创建这份手稿时使用了配置管理）。在大型50人项目上，你可能需要一个全面的 SCM 方案，包括相当正式的备份程序、需求和设计的变更控制，以及对文档、源代码、内容、测试用例和其他项目工件的控制。如果你的项目既不是非常大也不是非常小，你将不得不在两个极端之间确定一定程度的正式性。以下小节描述了实现 SCM 的一些选项。

在开发期间，你一定会对如何改进系统有很多想法。如果你每次想到什么就实现每个变更，你会很快发现自己走在软件跑步机上——尽管系统一直在变化，但它不会更接近完成。以下是一些控制设计变更的指南：

遵循系统化的变更控制程序。如构建前提条件所述，当你有很多变更请求时，系统化的变更控制程序是一个天赐之物。通过建立系统化的程序，你可以清楚地表明变更将在对整个项目最有利的情况下加以考虑。

成批处理变更请求。处理想法时很容易在它们出现时实现简单的变更。的问题是，以这种方式处理变更会导致好的变更丢失。如果你想到一个简单的变更是在项目进行到25%的时候，而你正在按计划进行，你会做这个变更。如果你在项目进行到50%的时候想到另一个简单的变更，而你已经开始落后了，你就不会做。当你在项目结束时开始用完时间时，第二个变更是第一个的10倍好也没关系——你将无法进行任何非必要的变更。一些最好的变更可能只是因为你想到了它们而不是更早，而从缝隙中溜走。

这个问题的解决方案是写下所有的想法和建议，无论它们实现起来多么容易，并保存它们直到你有时间处理它们。然后，将它们作为一个组来查看，选择那些最有价值的。

估算每个变更的成本。每当你的客户、你的老板或你被诱惑变更系统时，估算进行变更所需的时间，包括审查变更的代码和重新测试整个系统。在你的估算中包括处理变更的涟漪效应所需的时间——从需求到设计到代码到测试再到用户文档的变更。让所有相关方都知道软件是错综复杂地交织在一起的，即使变更乍看之下很小，时间估算也是必要的。

无论你在变更首次提出时感觉多么乐观，不要给出即兴估算。这样的估算经常被错误地估计两倍或更多。

警惕高变更量。虽然一定程度的变更是不可避免的，但大量变更请求是需求、架构或顶层设计做得不够好以支持有效构建的关键警告信号。回去处理需求或架构似乎很昂贵，但不会像多次构造软件或为真正不需要的功能丢弃代码那样昂贵。

建立变更控制委员会或其等价物，以对你的项目有意义的方式。变更控制委员会的工作是将变更请求中的精华与糟粕分开。任何想要提出变更的人将变更请求提交给变更控制委员会。"变更请求"一词指的是任何会变更软件的请求：新功能的想法、对现有功能的变更、"错误报告"，可能是也可能不是报告真正的错误，等等。委员会定期开会审查拟议的变更。它批准、否决或推迟每个变更。变更控制委员会被认为是优先级排序和控制需求变更的最佳实践；然而，它们在商业环境中仍然相当少见（Jones 1998, Jones 2000）。

警惕官僚主义，但不要让对官僚主义的恐惧阻碍有效的变更控制。缺乏严格的变更控制是当今软件行业面临的最大管理问题之一。如果被认为迟到的项目中很大一部分如果考虑到未跟踪但已同意的变更的影响，实际上会按时完成。糟糕的变更控制允许变更累积在账外，这破坏了状态可见性、长期可预测性、项目规划、风险管理和项目管理的普遍性。

变更控制往往会漂向官僚主义，因此寻找简化变更控制过程的方法很重要。如果你不想使用传统的变更请求，设置一个简单的"变更板"电子邮件别名，让人们将变更请求发送到该地址。或者让人们在变更委员会会议上以交互方式提出变更建议。一个特别强大的方法是将变更请求记录为缺陷跟踪软件中的缺陷。纯粹主义者会将这类变更归类为"需求缺陷"，或者你可以将它们归类为变更而非缺陷。

你可以正式实现变更控制委员会本身，或者你可以定义一个产品规划组或战争委员会来承担变更控制委员会的传统职责。或者你可以指定一个人作为变更沙皇。但无论你怎么称呼它，都要这样做！

我偶尔会看到项目遭受笨拙的变更控制实现之苦。但10倍于此，我看到项目遭受根本没有有意义的变更控制之苦。变更控制的实质是重要的，所以不要让对官僚主义的恐惧阻止你实现它的许多好处。

### 软件代码变更

另一个配置管理问题是控制源代码。如果你更改了代码，而出现了一个似乎与你所做的变更无关的新错误，你可能想要将新版本的代码与你正在搜索错误源的旧版本进行比较。如果那没有告诉你什么，你可能想要查看一个更旧的版本。如果你有版本控制工具来跟踪源代码的多个版本，这种穿越历史的方式很容易。

版本控制软件。好的版本控制软件工作起来非常容易，你几乎不会注意到你正在使用它。它在团队项目中特别有帮助。一种版本的控制锁定源文件，以便一次只有一个人可以修改文件。通常，当你需要处理特定文件中的源代码时，你从版本控制中检查该文件。如果其他人已经检查了它，你会收到通知说你不能检查它。当你能够检查该文件时，你就像没有版本控制一样处理它，直到你准备好检查它。另一种样式允许多人同时处理文件，并在代码被检查进来时处理合并变更的问题。无论哪种情况，当你检查文件进来时，版本控制会询问你为什么更改它，你输入一个原因。

对于这种适度的努力投入，你得到几个大的好处：

- 你不会因为在别人工作时处理一个文件而踩到任何人的脚（或者至少如果你这样做了，你会知道的）。
- 你可以轻松地将项目所有文件的副本更新到当前版本，通常只需发出一个命令。
- 你可以回溯到曾经检查到版本控制的任何版本的任何文件。
- 你可以获得对任何版本的任何文件所做变更的列表。
- 你不必担心个人备份，因为版本控制副本是一个安全网。

版本控制在团队项目上是不可或缺的。当版本控制、缺陷跟踪和变更管理集成在一起时，它变得更加强大。微软的应用部门发现其专有版本控制工具是一个"主要竞争优势"（Moore 1992）。

### 工具版本

对于某些类型的项目，可能需要能够重建用于创建软件每个特定版本的确切环境，包括编译器、链接器、代码库等。在这种情况下，你也应该将所有这些工具放入版本控制。

### 机器配置

许多公司（包括我的公司）从创建标准化的开发机器配置中获得了良好的结果。标准开发工作站的磁盘映像被创建，包括所有常见的开发工具、办公应用程序等。该映像被加载到每个开发者的机器上。拥有标准化的配置有助于避免与略有不同的配置设置、不同版本工具使用等相关的许多问题。与单独安装每件软件相比，标准化的磁盘映像也大大简化了新机器的设置。

### 备份计划

备份计划不是一个引人注目的新概念；它是定期备份你的工作的想法。如果你用手写书，你不会把书页堆放在门廊上。如果你这样做，它们可能会被雨淋或被风吹走，或者你邻居的狗可能借它们进行一些睡前阅读。你会把它们放在安全的地方。软件不太有形，所以更容易忘记你在某台机器上有价值巨大的东西。

很多事情可能发生在计算机化数据上：磁盘可能发生故障；你或其他人可能意外删除关键文件；愤怒的员工可能破坏你的机器；或者你可能因盗窃、洪水或火灾而丢失机器。采取措施保护你的工作。你的备份计划应包括定期制作备份和定期将备份转移到异地存储，并且它应该涵盖项目上的所有重要材料——文档、图形和笔记——以及源代码。

制定备份计划时一个经常被忽视的方面是测试你的备份程序。在某个时候尝试进行恢复，以确保备份包含你需要的每样东西，并且恢复工作正常。

当你完成一个项目时，制作项目存档。保存每样东西的副本：源代码、编译器、工具、需求、设计、文档——重新创建产品所需的每样东西。将它们全部保存在安全的地方。

## 估算构建进度

管理软件项目是二十一世纪最艰巨的挑战之一，估算项目规模和完成所需努力是软件项目管理最具挑战性的方面之一。大型软件项目平均晚一年完成，预算超支100%（Standish Group 1994, Jones 1997, Johnson 1999）。在个人层面，对估算与实际进度的调查显示，开发人员的估算往往有20%到30%的乐观因素（van Genuchten 1991）。这与糟糕的规模和努力估算以及糟糕的开发工作有关。本节概述了估算软件项目所涉及的问题，并指出在哪里可以找到更多信息。

### 估算方法

你可以通过以下几种方式估算项目规模和完成所需努力：

进一步阅读

有关进度估算技术的更多阅读，请参见《快速开发》（McConnell 1996）第8章和《用Cocomo II估算软件成本》（Boehm et al. 2000）。

- 使用估算软件
- 使用算法方法，如 Cocomo II，Barry Boehm 的估算模型（Boehm et al. 2000）
- 让外部估算专家估算项目
- 举行走查会议进行估算
- 估算项目的各个部分，然后将各部分相加
- 让人们估算他们自己的任务，然后将任务估算相加
- 参考以往项目的经验
- 保留以前的估算，看看它们有多准确。用它们来调整新的估算

本节开头列出的估算方法列表中给出了这些方法的更多信息指针。以下是一种估算项目的好方法：

确定目标。你为什么需要估算？你在估算什么？你只估算构建活动，还是整个开发？你只估算项目的努力，还是你的项目加上假期、节日、培训和其他非项目活动？估算需要有多准确才能满足你的目标？需要与估算关联什么程度的确定性？乐观或悲观的估算会产生 substantially 不同的结果吗？

进一步阅读

这种方法改编自《软件工程经济学》（Boehm 1981）。

留出时间进行估算，并计划它。仓促的估算是不准确的估算。如果你在估算一个大项目，把估算当作一个小项目来处理，并花时间小规划估算，以便你能做好它。

明确软件需求。就像建筑师无法可靠地估算"相当大"的房子要花多少钱一样，你无法可靠地估算"相当大"的软件项目。当"某物"尚未被定义时，任何人期望你能够估算构建某物所需的工作量是不合理的。在进行估算之前定义需求或计划初步探索阶段。

交叉引用

有关软件需求的更多信息，请参见构建前提条件。

在较低细节级别进行估算。根据你确定的目标，基于对项目活动的详细检查来进行估算。一般来说，你的检查越详细，你的估算就越准确。大数定律说，一大块上的10%误差将是10%高或10%低。在50个小块上，一些块上的10%误差会高，一些会低，误差往往会相互抵消。

使用几种不同的估算技术，并比较结果。本节开头列出的估算方法列表确定了几个技术。它们不会都产生相同的结果，所以尝试其中几个。研究来自不同方法的不同的结果。孩子们很小就学会，如果他们分别向每个父母要第三碗冰淇淋，他们比只问一个父母有更好的机会获得至少一个"是"。有时父母会醒悟过来给出相同的答案；有时不会。看看你能从不同的估算技术中得到什么不同的答案。

交叉引用

很难找到软件开发中迭代没有价值的领域。估算是迭代有用的情况之一。对于迭代技术的总结，请参见迭代、重复、再来一次再来一次。

没有任何方法在所有情况下都是最好的，它们之间的差异可能是富有启发性的。例如，对于本书的第一版，我最初对书长度的目测估计是250-300页。当我最终进行深入估算时，估算结果是873页。"这不可能是对的，"我想。所以我用一种完全不同的技术重新估算。第二次估算得出828页。考虑到这些估算相差大约5%，我得出结论，这本书将更接近850页而不是250页，我能够相应地调整我的写作计划。

### 如果你落后了怎么办

如本章前面所述，平均项目超过其计划进度约100%。当你落后时，增加时间通常不是一个选择。如果是这样，就这样做。否则，你可以尝试以下一个或多个解决方案：

希望你会赶上来。对项目落后进度的乐观希望是一种常见反应。合理化通常是这样的："需求比我们预期的花费了更长时间，但现在它们是 solid 的，所以我们一定会在以后节省时间。我们将在编码和测试期间弥补短缺。"这几乎从来不是这种情况。一项对超过300个软件项目的调查得出结论，延迟和超支通常在项目结束时增加（van Genuchten 1991）。项目以后不会弥补失去的时间；它们会进一步落后。

扩大团队。根据 Fred Brooks 的法则，将人加入延迟的软件项目会使其更晚（Brooks 1995）。这就像往火上浇油。Brooks 的解释令人信服：新人们需要时间熟悉项目，然后才能变得高效。他们的培训占用了已经培训过的人的时间。而且，仅仅增加人数就增加了项目沟通的复杂性和数量。Brooks 指出，一个女人可以在九个月内生孩子这一事实并不意味着九个女人可以一个月内生孩子。

无疑，Brooks 法则中的警告应该比实际情况更经常被注意。很想把人扔向项目并希望他们能按时完成。经理们需要理解，开发软件不像铆接金属板：更多工人工作不一定意味着更多工作会完成。

然而，简单的陈述，即将程序员加入延迟的项目会使其更晚，掩盖了在某些情况下可以加入人以加速延迟项目的 fact。正如 Brooks 在他的法则分析中指出的，在任务不能分割和独立执行的项目中增加人没有帮助。但如果项目的任务是可分割的，你可以进一步分割它们并将它们分配给不同的人，即使是在项目后期加入的人。其他研究人员已经正式确定了你可以将人加入延迟项目而不会使其更晚的情况（Abdel-Hamid 1989, McConnell 1999）。

减少项目范围。减少项目范围这一强大技术经常被忽视。如果你消除一个功能，你就消除了该功能的设计、编码、调试、测试和文档。你消除了该功能与其他功能的接口。

进一步阅读

有关只构建最需要功能的原因的论证，请参见《快速开发》（McConnell 1996）第14章"功能集控制"。

当你在最初规划产品时，将产品的能力分为"必须有"、"最好有"和"可选"。如果你落后了，优先考虑"可选"和"最好有"，放弃最不重要的。

除了完全放弃一个功能，你可以提供相同功能的更便宜版本。你可能提供一个准时的版本，但没有为性能调优。你可能提供一个最不重要功能实现得很粗糙的版本。你可能决定放弃速度要求，因为提供一个慢版本要容易得多。你可能放弃空间要求，因为提供一个内存密集型版本更容易。

重新估算最不重要功能的开发时间。你能在两个小时、两天或两周内提供什么功能？通过构建两周版本而不是两天版本，或者两天版本而不是两小时版本，你获得了什么？

## 度量

软件项目可以通过多种方式度量。以下是度量你的过程的两个坚实理由：

对于任何项目属性，总是可以用比完全不度量更好的方式来度量该属性。度量可能不是完全精确的，可能难以进行，并且可能需要随着时间推移进行细化，但度量会让你掌握你没有的关于软件开发过程的信息（Gilb 2004）。

如果数据要在科学实验中使用，它必须被量化。你能想象一位科学家仅仅因为一组白鼠"看起来比另一组更容易生病"就建议禁止一种新食品吗？这是荒谬的。你会要求一个量化的理由，比如"吃了新食品的老鼠每月比没吃新食品的老鼠多生病3.7天。"要评估软件开发方法，你必须度量它们。像"这种新方法似乎更有生产力"这样的陈述是不够好的。

注意度量副作用。度量有激励效应。人们注意他们被度量的任何东西，假设它被用来评估他们。仔细选择你要度量的东西。人们倾向于专注于被度量的工作，而忽略不被度量的工作。

被度量的事情才会被完成。——Tom Peters

反对度量就是反对知道你的项目真正发生了什么。当你度量项目的一个方面时，你比之前更了解它。你可以看看该方面是变得更大、更小还是保持不变。度量至少为你提供了项目那个方面的一扇窗户。在你细化你的度量之前，这扇窗户可能又小又模糊，但它总比没有窗户好。因为一些度量没有结论就反对所有度量，就是反对窗户，因为有些窗户恰好是模糊的。

你可以度量软件开发过程的几乎任何方面。表28-2列出了一些其他从业者发现有用的度量。

- 每千行代码的平均缺陷数
- 故障之间的平均时间
- 编译器检测到的错误

### 可维护性

- 每个类的公共例程数
- 传递给每个例程的参数数
- 每个类的私有例程和/或变量数
- 每个例程使用的局部变量数
- 每个类或例程调用的例程数
- 每个例程中的决策点数
- 每个例程的控制流复杂性
- 每个类或例程的代码行数
- 每个类或例程中的注释行数
- 每个类或例程中的数据声明数
- 每个类或例程中的空行数
- 每个类或例程中的goto数
- 每个类或例程中的输入或输出语句数

你可以用目前可用的软件工具收集大部分这些度量。全书的讨论表明了每个度量有用的原因。目前，大多数度量对于在程序、类和例程之间进行细微区分没有用处（Shepperd and Ince 1989）。它们主要用于识别"异常值"的例程；例程中的异常度量是一个警告信号，你应该重新检查该例程，检查是否具有异常低的质量。

不要从收集所有可能度量的数据开始——你会用如此复杂的数据埋葬自己，以至于你无法弄清楚它们的含义。从一套简单的度量开始，如缺陷数量、工作月数、总美元数和总代码行数。标准化你项目之间的度量，然后随着你对想要度量的内容理解的提高来细化和添加它们（Pietrasanta 1990）。

确保你收集数据是有原因的。设定目标，确定你需要问什么问题来达到目标，然后度量来回答问题（Basili and Weiss 1984）。确保你只要求尽可能多地获取是可行的信息，并记住数据收集总是会排在截止日期之后（Basili et al. 2002）。

## 将程序员当人看

编程活动的抽象性要求办公环境有补偿性的自然性，以及与同事的丰富接触。高科技公司提供公园式的企业园区、有机组织结构、舒适办公室和其他"高触感"环境特征，以 balance 工作本身的强烈、有时是贫瘠的智力活动。最成功的技术公司结合了高科技和高触感的元素（Naisbitt 1982）。本节描述了程序员不仅仅是他们的硅替身的有机反映的方式。

### 程序员如何度过他们的时间？

程序员花时间编程，但他们也花时间开会、培训、阅读邮件，以及只是思考。1964年贝尔实验室的一项研究发现，程序员的时间分配如表28-3所示。

**表28-3。程序员如何度过时间的一种观点**

| 活动 | 来源 |
|------|------|
| 代码 | 35% |
| 商务/个人 | 29% |
| 会议/培训 | 13% |
| 邮件/杂项 | 7% |
| 技术手册 | 6% |
| 交谈或倾听 | 4% |
| 与经理交谈 | 4% |
| 电话 | 2% |
| 阅读 | 2% |
| 书写/记录 | 3% |
| 走开或外出 | 1% |
| 步行 | 1% |
| 杂项 | 1% |

来源："Research Studies of Programmers and Programming"（Bairdain 1964，报告于 Boehm 1981）。

这些数据基于对70名程序员的时间研究和动作研究。这些数据是旧的，不同活动的时间比例会因程序员而异，但结果仍然发人深省。约30%的程序员时间花在非直接帮助项目的非技术活动上：步行、个人事务等。这项研究中的程序员花了6%的时间步行；这大约是每周2.5小时，每年125小时。这似乎不多，直到你意识到程序员每年步行的时间与他们花在培训上的时间一样多，是他们阅读技术手册时间的三倍，是他们与经理交谈时间的六倍。就个人而言，我今天没有看到这种模式有太大变化。

### 绩效和质量的差异

个人程序员之间的才能和努力差异很大，就像在所有领域一样。一项研究发现，在各种职业中——写作、足球、发明、警察工作和飞机驾驶——前20%的人产生了大约50%的产出（Augustine 1979）。该研究的结果基于对生产力数据的分析，如触地得分、专利、解决的案例等。由于一些人根本没有做出有形的贡献，根本没有被考虑在研究中（四分卫没有触地得分，发明家没有专利，侦探没有结案，等等），数据可能低估了实际的生产力差异。

具体到编程，许多研究已经表明编写的程序质量、编写程序的大小和程序员生产力之间存在数量级差异。

### 个人差异

最初显示个人编程生产力巨大差异的研究是由 Sackman、Erikson 和 Grant（1968）在20世纪60年代末进行的。他们研究了平均有7年经验的专业程序员，发现最佳和最差程序员之间初始编码时间的比率约为20比1，调试时间的比率超过25比1，程序大小的比率为5比1，程序执行速度的比率为约10比1。他们发现程序员的经验数量与代码质量或生产力之间没有关系。

虽然诸如25比1之类的具体比率不是特别有意义，但更一般的陈述如"程序员之间存在数量级差异"是有意义的，并且已被许多其他专业程序员研究证实（Curtis 1981, Mills 1983, DeMarco and Lister 1985, Curtis et al. 1986, Card 1987, Boehm and Papaccio 1988, Valett and McGarry 1989, Boehm et al. 2000）。

### 团队差异

编程团队在软件质量和生产力方面也表现出相当大的差异。好的程序员倾向于聚集在一起，坏的程序员也是如此，这一观察已被对来自18个组织的166名专业程序员的研究证实（Demarco and Lister 1999）。

在一项对七个相同项目的研究中，投入的努力相差3.4比1，程序大小相差3比1（Boehm, Gray, and Seewaldt 1984）。尽管存在生产力范围，但这项研究中的程序员不是一个多样化的人群。他们都是有多年初经验的专业程序员，他们注册了计算机科学研究生课程。可以合理地假设，对一个同质性较低群体的研究会产生甚至更大的差异。

一项早期对编程团队的研究观察到程序大小5比1的差异和团队完成相同项目所需时间2.6比1的变化（Weinberg and Schulman 1974）。

在构建 Cocomo II 估算模型时回顾了20多年的数据后，Barry Boehm和其他研究人员得出结论，用排名为第15百分位的程序员团队开发程序通常比用排名为第90百分位的团队多花约3.5倍的工作月（Boehm et al. 2000）。Boehm和其他研究人员发现，80%的贡献来自20%的贡献者（Boehm 1987b）。

对招聘和雇用的含义是明确的。如果你必须支付更多才能得到前10%的程序员而不是后10%的程序员，抓住这个机会。你将在你雇用的程序员的素质和生产力上获得直接回报，并且由于好程序员倾向于聚集，你将能够在你的组织保留的其他程序员的素质和生产力上获得剩余效应。

### 宗教问题

编程项目的管理者并不总是意识到某些编程问题是宗教问题。如果你是一名经理，你试图要求遵守某些编程实践，你是在邀请程序员们的愤怒。以下是宗教问题列表：

- 编程语言
- 缩进风格
- 大括号放置
- IDE选择
- 注释风格
- 效率与可读性权衡
- 方法论选择——例如，Scrum vs. 极限编程 vs. 进化交付
- 编程工具
- 命名约定
- goto的使用
- 全局变量的使用
- 度量，特别是生产力度量，如每天代码行数

这些主题的共同点是，程序员在每个主题上的立场都反映了个人风格。如果你认为你需要控制任何这些宗教领域的程序员，考虑以下几点：

意识到你正在处理一个敏感区域。在全身投入之前，先听取程序员对每个情感主题的看法。

使用关于该领域的"建议"或"指南"。避免设定 rigid 的"规则"或"标准"。

通过回避明确指令来巧妙处理你可以通过的问题。要巧妙处理缩进风格或大括号放置，要求源代码在声明完成之前通过漂亮的格式化程序运行。让格式化程序做格式化。要巧妙处理注释风格，要求所有代码被审查，不清楚的代码被修改直到清晰。

让你的程序员自己制定标准。如其他地方所述，特定标准的细节通常不如存在某个标准这一事实重要。不要为你的程序员设定标准，但确实要坚持他们在对你重要的领域标准化。

哪些宗教话题重要到值得全力以赴？在任何领域的小事上的一致性可能不会产生足够的收益来抵消士气低落的影响。如果你发现 goto 或全局变量的 indiscriminate 使用、不可读的风格或其他影响整个项目的实践，准备好容忍一些摩擦以提高代码质量。如果你的程序员是认真的，这很少是问题。最大的战斗往往是关于编码风格的细微差别，你可以置身事外而不会对项目造成任何损失。

### 物理环境

这里有一个实验：去乡下，找一个农场，找一个农民，问他每个工人有多少设备资金。农民看着谷仓，看到几台拖拉机、一些 wagon、一台小麦收割机和一台豌豆去荚机，会告诉你每个员工超过10万美元。

接下来去城市，找一个编程工作室，找一个编程经理，问他每个工人有多少设备资金。编程经理看着办公室，看到一张桌子、一把椅子、几本书和一台计算机会告诉你每个员工不到2.5万美元。

物理环境对生产力有很大影响。DeMarco 和 Lister 询问了来自35个组织的166名程序员关于他们物理环境的质量。大多数员工将他们的工作场所评为不可接受。在随后的编程竞赛中，表现处于前25%的程序员拥有更大、更安静、更私密的办公室，以及更少的来自人和电话的中断。以下是最佳和最差表现者之间办公空间的差异摘要：

| 环境因素 | 前25% | 后25% |
|----------|-------|-------|
| 专用楼层空间 | 78平方英尺 | 46平方英尺 |
| 可接受的安静工作空间 | 57% 是 | 29% 是 |
| 可接受的私密工作空间 | 62% 是 | 19% 是 |
| 能够使电话静音 | 52% 是 | 10% 是 |
| 能够转接电话 | 76% 是 | 19% 是 |
| 频繁的无谓中断 | 38% 是 | 76% 是 |
| 让程序员感到被欣赏的工作空间 | 57% 是 | 29% 是 |

来源：《人件》（DeMarco and Lister 1999）。

数据显示生产力和工作场所质量之间有很强的相关性。前25%的程序员比后25%的程序员生产力高2.6倍。DeMarco 和 Lister 认为更好的程序员可能自然拥有更好的办公室，因为他们被提升了，但进一步检查发现事实并非如此。来自相同组织的程序员有类似的设施，无论其表现的差异如何。

大型软件密集型组织也有类似的经验。施乐、TRW、IBM 和贝尔实验室表示，他们通过每人1万到3万美元的资本投资获得了显著提高的生产力，这些金额在提高的生产力中得到了超额回报（Boehm 1987a）。在"生产力办公室"中，自我报告的估计范围从生产力提高39%到47%（Boehm et al. 1984）。

总之，如果你的工作场所是后25%的环境，你可以通过将其变成前25%的环境来实现生产力约100%的提高。如果你的工作场所是平均水平，通过将其变成前25%的环境，你仍然可以实现40%或更多的生产力提高。

## 管理你的经理

在软件开发中，非技术经理很常见，也有具有技术经验但落后10年的经理。技术称职、技术更新的经理很少见。如果你在这样的人手下工作，尽你所能保住你的工作。这是一个难得的享受。

在等级制度中，每个员工往往会晋升到他的无能力水平。——彼得原理

如果你的经理更典型，你面临着管理你的经理这个令人羡慕的任务。"管理你的经理"意味着你需要告诉你的经理该做什么，而不是反过来。诀窍是以一种允许你的经理继续相信你是被管理的那个人的方式来做。以下是一些处理你的经理的方法：

种下你想要做什么的想法，然后等待你的经理有一个关于做你想做的事的灵感（你的想法）。

教育你的经理了解做事的正确方式。这是一项持续的工作，因为经理经常被提升、调动或解雇。

关注你经理的兴趣，做他或她真正希望你做的事，不要用不必要的实现细节分散你经理的注意力。（把它想成是你工作的"封装"。）

拒绝做你经理告诉你的事，坚持用正确的方式做你的工作。

另找工作。

最好的长期解决方案是尝试教育你的经理。这并不总是一项容易的任务，但你可以准备的一种方式是阅读 Dale Carnegie 的《如何赢得朋友并影响人》。

## 关键要点

- 良好的编码实践可以通过强制执行标准或更轻量的方法来实现。
- 配置管理（在适当应用时）使程序员的工作更轻松。这尤其包括变更控制。
- 良好的软件估算是重大挑战。成功的关键是使用多种方法，随着你深入项目收紧你的估算，并利用数据来创建估算。
- 度量是成功构建管理的关键。你可以找到比完全不度量更好的方式来度量项目的任何方面。准确的度量是准确进度、质量控制和改进开发过程的关键。
- 程序员和管理者都是人，当被当作人来对待时，他们工作得最好。
