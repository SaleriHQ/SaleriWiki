### 创建类的步骤

构建类的关键步骤是：

**创建类的一般设计。** 类设计包括众多具体问题。定义类的特定职责，定义类将隐藏的"秘密"，精确定义类接口将捕获的抽象。确定类是否将从另一个类派生，以及是否允许其他类从它派生。识别类的关键公共方法，识别并设计类使用的任何非平凡数据成员。根据需要多次迭代这些主题以创建例程的直接设计。这些考虑和许多其他问题在第6章中有更详细的讨论。

**在类中构建每个例程。** 一旦在第一步中识别了类的主要例程，你必须构建每个特定例程。每个例程的构建通常会发掘对额外例程的需求——既有小的也有大的——而创建这些额外例程引起的问题经常反馈到整体类设计。

**整体审查和测试类。** 通常，每个例程在创建时都被测试。一旦类整体可运行，类整体应被审查和测试任何无法在单个例程级别测试的问题。

## 专业人士的伪代码

"伪代码"一词指的是用于描述算法、例程、类或程序如何工作的非正式、类似英语的符号。伪代码编程过程定义了使用伪代码简化例程内代码创建的特定方法。

因为伪代码类似于英语，很自然地假设任何类似英语的描述来收集你的想法，与任何其他描述大致相同。在实践中，你会发现某些伪代码风格比其他更有用。以下是有效使用伪代码的指导原则：

- 使用类似英语的语句精确描述特定操作。
- 避免目标编程语言的语法元素。伪代码允许你比代码本身在略高的层次上设计。当你使用编程语言构造时，你下沉到更低层次，消除了在更高层次设计的主要好处，并给自己添加了不必要的语法限制。
- 在意图层次编写伪代码。描述方法的意义而不是方法将如何在目标语言中实现。

> **交叉参考**
>
> 有关在意图层次注释的详细信息，请参见"有效注释的关键"中的"注释种类"。

- 在足够低的层次编写伪代码，以便从它生成代码几乎是自动的。如果伪代码层次太高，它可能掩盖代码中有问题的细节。将伪代码精炼得越来越详细，直到看起来直接写代码会更容易。

一旦伪代码写好，你围绕它构建代码，伪代码变成编程语言注释。这消除了大部分注释工作。如果伪代码遵循指导原则，注释将是完整且有意义的。

以下是一个违反刚才描述的所有原则的伪代码设计示例：

**示例9-1。** 糟糕伪代码示例

```
increment resource number by 1
allocate a dlg struct using malloc
if malloc() returns NULL then return 1
invoke OSrsrc_init to initialize a resource for the operating system
*hRsrcPtr = resource number
return 0
```

这段伪代码的意图是什么？因为它写得不好，很难说清楚。这个所谓的伪代码是糟糕的，因为它包含目标语言编码细节，如*hRsrcPtr（特定C语言指针符号）和malloc()（特定C语言函数）。这段伪代码关注的是代码将如何编写，而不是设计的意思。它涉及编码细节——例程返回1还是0。如果你从伪代码是否将变成好注释的角度思考，你将开始理解它没有多大帮助。

以下是同一操作的极大改进伪代码设计：

**示例9-2。** 好伪代码示例

```
Keep track of current number of resources in use
If another resource is available
   Allocate a dialog box structure
   If a dialog box structure could be allocated
      Note that one more resource is in use
      Initialize the resource
      Store the resource number at the location provided by the caller
   Endif
Endif
Return true if a new resource was created; else return false
```

这段伪代码比第一段更好，因为它完全用英语写成；它不使用目标语言的任何语法元素。在第一个示例中，伪代码只能用C实现。在第二个示例中，伪代码不限制语言选择。第二段伪代码也是在意图层次编写的。第二段伪代码的意思是什么？对你来说可能比第一段更容易理解。

尽管用清晰的英语写成，第二段伪代码足够精确和详细，可以很容易地用作编程语言代码的基础。当伪代码语句转换为注释时，它们将成为代码意图的良好解释。

以下是你可以从使用这种风格伪代码中期望的好处：

**伪代码使审查更容易。** 你可以在不检查源代码的情况下审查详细设计。伪代码使低级设计审查更容易，减少了审查代码本身的需要。

**伪代码支持迭代提炼的思想。** 你从高层设计开始，将设计精炼为伪代码，然后将伪代码精炼为源代码。这种在小步骤中的连续精炼允许你在将其驱动到更低细节层次时检查设计。结果是，你在最高层次捕获高层错误，在中间层次捕获中层错误，在最低层次捕获低层错误——在任何错误成为问题或污染更详细层次的工作之前。

**伪代码使更改更容易。** 几行伪代码比一页代码更容易更改。你宁愿在蓝图上改一行还是拆掉一堵墙在其他地方钉2x4？在软件中效果不如物理上那么戏剧性，但在产品最可塑时更改的原则是一样的。项目成功的关键之一是在"最低价值阶段"捕获错误，即投入努力最少的阶段。在伪代码阶段投入比完整编码、测试和调试之后少得多，所以在早期捕获错误在经济上是合理的。

> **进一步阅读**
>
> 有关在最低价值阶段进行更改的优势的更多信息，请参见Andy Grove的《High Output Management》（Grove 1983）。

**伪代码最小化注释工作。** 在典型编码场景中，你先写代码然后添加注释。在PPP中，伪代码语句变成注释，所以实际上删除注释比保留它们需要更多工作。

**伪代码比其他形式的设计文档更容易维护。** 使用其他方法，设计与代码分离，当一个改变时，两者变得不一致。使用PPP，伪代码语句成为代码中的注释。只要内联注释被维护，伪代码对设计的文档将是准确的。

作为一种详细设计工具，伪代码难以匹敌。一项调查发现，程序员喜欢伪代码，因为其简化编程语言构建的方式、帮助他们检测不够详细的设计的能力，以及它提供的文档和修改的便利（Ramsey, Atwood, and Van Doren 1983）。伪代码不是唯一用于详细设计的工具，但伪代码和PPP是有待在程序员工具箱中的有用工具。试试它们。下一节告诉你怎么做。

### 设计例程

一旦你识别了类的例程，构建任何较复杂例程的第一步就是设计它。假设你想编写一个根据错误代码输出错误消息的例程，假设你调用例程ReportErrorMessage()。以下是ReportErrorMessage()的非正式规范：

> **交叉参考**
>
> 有关设计其他方面的详细信息，请参见第5章到第8章。

ReportErrorMessage()将错误代码作为输入参数，并根据代码输出对应的错误消息。它负责处理无效代码。如果程序以交互方式运行，ReportErrorMessage()向用户显示消息。如果以命令行模式运行，ReportErrorMessage()将消息记录到消息文件。输出消息后，ReportErrorMessage()返回一个状态值，指示它是成功还是失败。

本章其余部分使用此例程作为运行示例。本节其余部分描述如何设计例程。

**检查前置条件。** 在对例程本身做任何工作之前，检查例程的作业是否定义良好并干净地适合整体设计。检查以确保例程实际上被项目需求调用，至少间接地被调用。

> **交叉参考**
>
> 有关检查前置条件的详细信息，请参见第3章和第4章。

**定义例程将解决的问题。** 详细陈述例程将解决的问题，以便允许创建例程。如果高层设计足够详细，作业可能已经完成。高级设计至少应指出：

- 例程将隐藏的信息
- 例程的输入
- 例程的输出
- 保证在调用例程之前为真的前置条件（输入值在某些范围内、流已初始化、文件已打开或关闭、缓冲区已填充或刷新等）

> **交叉参考**
>
> 有关前置条件和后置条件的详细信息，请参见"断言"中的"使用断言记录和验证前置条件和后置条件"。

- 例程保证在将控制权传递回调用者之前为真的后置条件（输出值在指定范围内、流已初始化、文件已打开或关闭、缓冲区已填充或刷新等）

以下是这些考虑在ReportErrorMessage()示例中如何处理：

- 例程隐藏两个事实：错误消息文本和当前处理方法（交互式或命令行）。
- 没有保证给例程的前置条件。
- 例程的输入是错误代码。
- 要求两种输出：第一是错误消息，第二是ReportErrorMessage()返回给调用例程的状态。
- 例程保证状态值要么是Success要么是Failure。

**命名例程。** 命名例程可能看起来微不足道，但好的例程名称是优越程序的标志之一，它们不容易想出来。一般来说，例程应该有一个清晰、明确的名称。如果你难以创建好名称，这通常表明例程的目的不清晰。模糊、摇摆不定的名称就像竞选期间的政客。它听起来好像在说什么，但当你仔细看时，你无法弄清楚它的意思。如果你能让名称更清晰，就这样去做。如果摇摆不定的名称源于摇摆不定的设计，要注意警告信号。退后并改进设计。

> **交叉参考**
>
> 有关命名例程的详细信息，请参见"良好的例程命名"。

在示例中，ReportErrorMessage()是明确的。它是一个好名称。

**决定如何测试例程。** 在编写例程时，考虑如何测试它。这对你进行单元测试以及独立测试你的例程的测试人员很有用。

> **进一步阅读**
>
> 有关首先关注编写测试用例的构建替代方法，请参见《测试驱动开发：示例》（Beck 2003）。

在示例中，输入很简单，所以你可能计划用所有有效错误代码和各种无效代码测试ReportErrorMessage()。

**研究标准库中可用的功能。** 提高代码质量和生产力的最大单一方法是重用好代码。如果你发现自己在设计似乎过于复杂的例程，问问自己例程的部分或全部功能是否可能已在使用的语言、平台或工具的库代码中可用。问自己代码是否可能在你公司维护的库代码中可用。许多算法已经被发明、测试、在行业文献中讨论、审查和改进。与其在你有人已经写了博士论文的东西上花费时间发明，不如花几分钟浏览已经编写的代码，确保你没有做不必要的工作。

**思考错误处理。** 思考例程中可能出错的所有事情。思考坏输入值、从其他例程返回的无效值等。

例程可以用众多方式处理错误，你应该有意识地选择如何处理错误。如果程序架构定义了程序的错误处理策略，你可以简单计划遵循该策略。在其他情况下，你必须决定什么方法最适合特定例程。

**思考效率。** 根据你的情况，你可以通过两种方式之一处理效率。在第一种情况中，在绝大多数系统中，效率并不关键。在这种情况下，要确保例程接口抽象良好且代码可读，以便以后需要时可以改进它。如果你有良好的封装，你可以在不影响其他例程的情况下，用更好的算法或快速、精简的低级语言实现替换慢的、消耗资源的高级语言实现。

在第二种情况中——在少数系统中——性能是关键的。性能问题可能与稀缺的数据库连接、有限的内存、稀少的可用句柄、雄心勃勃的时序约束或其他稀缺资源有关。架构应指示每个例程（或类）允许使用多少资源以及它应该以多快的速度执行其操作。

> **交叉参考**
>
> 有关效率的详细信息，请参见第25章和第26章。

设计你的例程使其满足其资源和速度目标。如果资源或速度似乎更关键，设计为用资源换速度或反之亦然。在例程初始构建期间将其调整到满足资源和速度预算是可以接受的。

除了处理这两种一般情况建议的方法外，在单个例程层次上处理效率通常是浪费精力。大的优化来自于精炼高层设计，而不是单个例程。你通常只在高层设计结果不支持系统性能目标时才使用微优化，而你直到整个程序完成后才知道这一点。在你知道需要之前，不要浪费时间寻求增量改进。

**研究算法和数据类型。** 如果功能在可用库中不可用，它可能仍在算法书中描述。在从头编写复杂代码之前，检查算法书看看已经有什么。如果你使用预定义算法，确保正确地适应你的编程语言。

**编写伪代码。** 完成前述步骤后，你可能没有太多书面内容。这些步骤的主要目的是在你实际编写例程时建立有用的心理方向。

完成初步步骤后，你可以开始将例程作为高层伪代码编写。继续使用你的编程编辑器或集成环境编写伪代码——伪代码将很快用作编程语言代码的基础。

> **交叉参考**
>
> 本讨论假设使用良好设计技术来创建例程的伪代码版本。设计详细信息请参见第5章。

从一般开始，向更具体的方向努力。例程最一般的部分是描述例程应该做什么的头部注释，所以首先写例程目的的简明陈述。写陈述将帮助你澄清对例程的理解。写一般注释有困难是警告信号，表明你需要更好地理解例程在程序中的作用。一般来说，如果难以总结例程的作用，你可能应该假设出了问题。

以下是描述例程的简明头部注释示例：

**示例9-3。** 例程头部注释示例

```cpp
This routine outputs an error message based on an error code
supplied by the calling routine. The way it outputs the message
depends on the current processing state, which it retrieves
on its own. It returns a value indicating success or failure.
```

写完一般注释后，为例程填充高层伪代码。以下是此示例的伪代码：

**示例9-4。** 例程伪代码示例

```
This routine outputs an error message based on an error code
supplied by the calling routine. The way it outputs the message
depends on the current processing state, which it retrieves
on its own. It returns a value indicating success or failure.

set the default status to "fail"
look up the message based on the error code

if the error code is valid
   if doing interactive processing, display the error message
   interactively and declare success

   if doing command line processing, log the error message to the
   command line and declare success

if the error code isn't valid, notify the user that an internal error
has been detected

return status information
```

再次注意，伪代码是在相当高的层次编写的。它当然不是用编程语言编写的。相反，它用精确的英语表达例程需要做什么。

**思考数据。** 你可以在过程中的几个不同点设计例程的数据。在这个示例中，数据很简单，数据操作不是例程的突出部分。如果数据操作是例程的突出部分，在考虑例程逻辑之前考虑主要数据片段是值得的。关键数据类型定义在设计例程逻辑时很有用。

> **交叉参考**
>
> 有关有效使用变量的详细信息，请参见第10章到第13章。

**检查伪代码。** 一旦你写了伪代码并设计了数据，花一分钟审查你写的伪代码。退后一步，思考你将如何向别人解释它。

> **交叉参考**
>
> 有关审查技术的详细信息，请参见第21章。

请别人看它或听你解释。你可能认为让别人看11行伪代码是愚蠢的，但你会惊讶。伪代码可以使你的假设和高层错误比编程语言代码更明显。人们也更愿意审查几行伪代码而不是审查35行C++或Java。

确保你对例程做什么和如何做有轻松舒适的理解。如果你在概念上、在伪代码层次不理解它，你在编程语言层次理解它的机会是什么？如果你不理解它，谁会理解？

**用伪代码尝试几个想法，保留最好的（迭代）。** 在你开始编码之前，用伪代码尝试尽可能多的想法。一旦你开始编码，你就情感上投入了代码，抛弃糟糕设计重新开始变得更难。

### 检查代码

在设计和实现例程之后，构建它的第三个大步骤是检查以确保你构建的是正确的。任何你在这个阶段错过的错误都不会被发现，直到以后的测试。发现和纠正它们更昂贵，所以你应该在这个阶段发现所有你能发现的。

问题可能直到例程完全编码后才出现，原因有几个。伪代码中的错误可能在详细实现逻辑中变得更明显。在伪代码中看起来优雅的设计在实现语言中可能变得笨拙。使用详细实现可能披露架构、高层设计或需求中的错误。最后，代码可能有老式的、杂种的编码错误——没有人是完美的！由于所有这些原因，在继续之前审查代码。

> **交叉参考**
>
> 有关检查架构和需求的错误的详细信息，请参见第3章。

**Mental检查例程是否有错误。** 例程的第一次正式检查是mental的。之前提到的清理和非正式检查步骤是两种mental检查。另一种是Mental执行每个路径。Mental执行例程是困难的，这种困难是保持例程小的一个原因。确保检查正常路径和端点以及所有异常条件。既要自己做这（称为"桌面检查"），也要与一个或多个同行做（称为"同行评审"、"走查"或"检查"，取决于你如何做）。

> 业余爱好者和专业程序员之间最大的区别之一是走出迷信进入理解的差异。"迷信"这个词在这个上下文中不是指给你 creepy感觉或在满月时产生额外错误的程序。它意味着用对代码的感觉代替理解。如果你经常发现自己怀疑编译器或硬件出了错，你仍在迷信领域。多年前进行的一项研究发现，只有约5%的所有错误是硬件、编译器或操作系统错误（Ostrand and Weyuker 1984）。今天，这个百分比可能更低。进入理解领域的程序员总是首先怀疑自己的工作，因为他们知道他们造成了95%的错误。理解每行代码的作用和为什么需要它。没有什么仅仅因为似乎工作就是正确的。如果你不知道为什么它工作，它可能就不工作——你只是还不知道。

底线：工作的例程不够。如果你不理解它为什么工作，研究它，讨论它，尝试替代设计直到你理解。

**编译例程。** 审查例程后，编译它。等待这么久才编译似乎效率低下，因为代码在几页前就完成了。诚然，你可能通过早点编译例程节省一些工作，让计算机检查未声明的变量、命名冲突等。

然而，在过程后期才编译有几个好处。主要原因是当你编译新代码时，内部秒表开始滴答作响。第一次编译后，你加大压力："再编译一次我就会正确。""再编译一次"综合症导致匆忙、容易出错的更改，从长远来看花费更多时间。通过直到你确信例程正确才编译来避免匆忙完成。

本书的目的是展示如何超越黑进去组合一些东西并运行它看是否工作的循环。在你确信程序工作之前编译通常是新手下心态的症状。如果你没有陷入黑客和编译循环，在适当的时候编译。但要注意大多数人感到的向"黑客、编译和修复"方式获得工作程序的拉力。

以下是充分利用编译例程的一些指导原则：

- 将编译器警告级别设置为尽可能严格的级别。你可以通过允许编译器检测它们来捕获大量微妙错误。
- 使用验证器。C等语言执行的编译器检查可以通过使用lint等工具来补充。甚至未编译的代码（如HTML和JavaScript）也可以通过验证工具检查。
- 消除所有错误消息和警告的原因。注意消息告诉你的关于代码的内容。大量警告通常表明代码质量低，你应该努力理解你获得的每个警告。实际上，你一次又一次看到的警告有两个可能的效果之一：要么忽略它们，它们伪装其他更重要警告；要么它们只是令人讨厌。重写代码解决潜在问题并消除警告通常更安全，痛苦更少。

**在调试器中单步执行代码。** 一旦例程编译，将其放入调试器并单步执行每行代码。确保每行代码按你期望的方式执行。通过遵循这个简单实践你可以发现许多错误。

**测试代码。** 使用在开发例程时计划或创建的测试用例测试代码。你可能需要开发脚手架来支持你的测试用例——即在测试期间用于支持例程且不包含在最终产品中的代码。脚手架可以是使用测试数据调用例程的测试工具例程，也可以是你的例程调用的存根。

**从例程中移除错误。** 一旦检测到错误，必须将其移除。如果你在此时开发的例程有缺陷，它很可能保持有缺陷。如果你发现例程异常有缺陷，重新开始。不要在其周围黑客——重写它。黑客通常表明理解不完整，保证现在和以后都有错误。为有缺陷的例程创建全新的设计是有回报的。没有什么比重写有问题的例程而不再在其中发现任何错误更令人满意的。

### 清理收尾

当你完成检查代码问题后，检查它是否有本书其他地方描述的一般特征。你可以采取几个清理步骤以确保例程的质量符合你的标准：

- 检查例程的接口。确保所有输入和输出数据都被考虑，所有参数都被使用。详细信息请参见"如何使用例程参数"。
- 检查一般设计质量。确保例程做一件事且做好它，它与其他例程松散耦合，它被防御性地设计。详细信息请参见第7章。
- 检查例程的变量。检查不准确的变量名称、未使用的对象、未声明的变量、未正确初始化的对象等。详细信息请参见使用变量的章节，第10章到第13章。
- 检查例程的语句和逻辑。检查差一错误、无限循环、不正确的嵌套和资源泄漏。详细信息请参见语句章节，第14章到第19章。
- 检查例程的布局。确保你使用空白来澄清例程的逻辑结构、表达式和参数列表。详细信息请参见第31章。
- 检查例程的文档。确保翻译成注释的伪代码仍然准确。检查算法描述、接口假设和非明显依赖项的文档、不清晰编码实践的理由等。详细信息请参见第32章。
- 移除冗余注释。有时伪代码注释与注释描述的代码冗余，特别是当PPP被递归应用且注释只是在一个命名良好的例程调用之前时。

## PPP的替代方案

以我的钱来看，PPP是创建类和例程的最佳方法。以下是其他专家推荐的一些不同方法。你可以使用这些方法作为PPP的替代方案或补充。

**测试先行开发。** 测试先行是一种流行的开发风格，测试用例在任何代码之前编写。这种方法在"开发者测试推荐方法"中的"首先测试还是最后测试？"中有更详细的描述。关于测试先行编程的一本好书是Kent Beck的《测试驱动开发：示例》（Beck 2003）。

**重构。** 重构是一种开发方法，你通过一系列语义保持转换来改进代码。程序员使用坏代码模式或"味道"来识别需要改进的代码部分。第24章详细描述了这种方法，Martin Fowler的《重构：改进现有代码的设计》（Fowler 1999）是关于这个主题的好书。

**契约式设计。** 契约式设计是一种开发方法，其中每个例程被认为有前置条件和后置条件。这种方法在"断言"中的"使用断言记录和验证前置条件和后置条件"中有描述。关于契约式设计信息来源最好的是Bertrand Meyer的《面向对象软件构造》（Meyer 1997）。

**黑客？** 一些程序员尝试黑客方式获得工作代码而不是使用像PPP这样的系统性方法。如果你曾经发现你在例程中把自己编码到了角落里，必须重新开始，那就是PPP可能更好的迹象。如果你发现在编码例程中间失去思路，那是另一个PPP有益的迹象。你曾经简单忘记写类的一部分或例程的一部分吗？如果你使用PPP，这几乎不会发生。如果你发现自己盯着电脑屏幕不知道从哪里开始，那是PPP将使你的编程生活更轻松的可靠迹象。

## 关键点

- 构建类和构建例程往往是一个迭代过程。在构建特定例程时获得的洞察往往会反馈到类的设计中。
- 写好伪代码需要使用可理解的英语、避免特定编程语言的特征，并在意图层次编写（描述设计做什么而不是如何做）。

- 不要满足于你想出的第一个设计。在开始写代码之前，用伪代码迭代多个方法并选择最好的方法。
- 在每一步检查你的工作，并鼓励其他人也检查。这样，你将在投入努力最少的阶段以最低成本捕获错误。
