### 隐式声明

一些语言有隐式变量声明。例如，如果你在Microsoft Visual Basic中使用变量而不声明它，编译器会自动为你声明它（取决于编译器设置）。

隐式声明是任何语言中最危险的特性之一。如果你在Visual Basic中编程，你知道尝试弄清楚为什么acctNo没有正确的值然后注意到acctNum是重新初始化为0的变量是多么令人沮丧。如果你的语言不要求你声明变量，这种错误很容易发生。

如果你在要求声明变量的语言中编程，你的程序咬你需要先犯两个错误。首先，你必须在例程体中放入acctNum和acctNo。然后你必须在例程中声明这两个变量。这是一个更难犯的错误，实际上消除了同义变量问题。要求你明确声明数据的语言本质上要求你更小心地使用数据，这是它们的主要优势之一。如果你在具有隐式声明的语言中编程怎么办？以下是一些建议：

- **关闭隐式声明。** 一些编译器允许你禁用隐式声明。例如，在Visual Basic中你可以使用Option Explicit语句，它强制你在使用所有变量之前声明它们。
- **声明所有变量。** 当你输入新变量时声明它，即使编译器不要求你。这不会捕获所有错误，但会捕获其中一些。
- **使用命名约定。** 为常见后缀如Option Explicit和No建立命名约定，这样当你意思使用一个变量时不会使用两个变量。

> **交叉参考**
>
> 有关缩写的标准化详细信息，请参见"创建可读短名称"中的"通用缩写指导原则"。

- **检查变量名。** 使用编译器生成或另一个实用程序生成的交叉引用列表。许多编译器列出例程中的所有变量，允许你同时发现acctNum和acctNo。它们还指出你声明但未使用的变量。

## 变量初始化指导原则

不正确的初始化是计算机编程中最多产的错误来源之一。开发有效避免初始化问题的技术可以节省大量调试时间。

不正确初始化的问题源于变量包含你不期望的初始值。这可能由于以下几个原因发生：

- **变量从未被赋值。** 它的值是程序开始时恰好在其内存区域中的任何位值。

> **交叉参考**
>
> 有关基于数据初始化和使用模式的测试方法，请参见"测试技巧包"中的"数据流测试"。

- **变量中的值已过时。** 变量在某个时候被赋值，但该值不再有效。
- **变量的部分被赋值而部分没有。**

最后一个主题有几种变体。你可以初始化对象的一些成员但不是全部。你可以忘记分配内存然后初始化指针指向的"变量"。这意味着你真的在选择一块随机计算机内存并为其分配某个值。它可能是包含数据的内存。它可能是包含代码的内存。它可能是操作系统。指针问题的症状可能以完全令人惊讶的方式表现出来，每次都不同——这就是调试指针错误比调试其他错误更困难的原因。

以下是避免初始化问题的指导原则：

**在声明时初始化每个变量。** 在声明时初始化变量是一种廉价的防御性编程形式。这是对抗初始化错误的好保险政策。以下示例确保studentGrades在你每次调用包含它的例程时将被重新初始化。

**示例10-1。** C++ 声明时初始化示例

```cpp
float studentGrades[ MAX_STUDENTS ] = { 0.0 };
```

**在首次使用附近初始化每个变量。** 一些语言（包括Visual Basic）不支持在声明时初始化变量。这可能导致像以下这样的编码风格，其中声明被分组在一起，然后初始化被分组在一起——所有这些都远离变量的第一次实际使用。

> **交叉参考**
>
> 检查输入参数是防御性编程的一种形式。防御性编程的详细信息，请参见第8章。

**示例10-2。** Visual Basic 糟糕初始化示例

```vb
' 声明所有变量
Dim accountIndex As Integer
Dim total As Double
Dim done As Boolean

' 初始化所有变量
accountIndex = 0
total = 0.0
done = False
...

' 使用accountIndex的代码
...

' 使用total的代码
...

' 使用done的代码
While Not done
   ...
```

更好的做法是在尽可能接近首次使用的地方初始化变量：

**示例10-3。** Visual Basic 好的初始化示例

```vb
Dim accountIndex As Integer
accountIndex = 0
' 使用accountIndex的代码
...

Dim total As Double
total = 0.0       // 1
' 使用total的代码
...

Dim done As Boolean
done = False       // 2
' 使用done的代码
While Not done
...
```

(1) total在它被使用附近声明和初始化。

(2) done也在它被使用附近声明和初始化。

第二个示例由于几个原因优于第一个。到达使用done的代码时，done可能已被修改。当你第一次编写程序时情况不是这样，但以后的修改可能使其如此。第一个方法的另一个问题是一次抛出所有初始化创造了所有变量在整个例程中使用的印象——而实际上done只在末尾使用。最后，随着程序被修改（它会修改的，至少通过调试），可能围绕使用done的代码构建循环，done将需要重新初始化。第二个示例中的代码在这种情况下需要很少修改。第一个示例中的代码更容易产生恼人的初始化错误。

这是邻近原则的一个例子：保持相关动作在一起。同一原则适用于保持注释接近它们描述的代码、保持循环设置代码接近循环、分组直连代码中的语句以及许多其他领域。

> **交叉参考**
>
> 有关保持相关动作在一起的更多信息，请参见"作用域"。

理想情况下，在首次使用附近声明和定义每个变量。声明建立变量的类型。定义赋给变量特定值。在支持的语言中（如C++和Java），变量应在首次使用附近声明和定义。理想情况下，每个变量应在声明时同时定义，如下所示：

**示例10-4。** Java 好的初始化示例

```java
int accountIndex = 0;
// 使用accountIndex的代码
...

double total = 0.0;       // 1
// 使用total的代码
...

boolean done = false;       // 2
// 使用done的代码
while ( ! done ) {
...
```

(1) total在它被使用附近初始化。

(2) done也在它被使用附近初始化。

**尽可能使用final或const。** 通过在Java中将变量声明为final或在C++中声明为const，你可以防止变量在初始化后被赋值。final和const关键字对定义类常量、仅输入参数以及任何打算在初始化后保持不变的值局部变量很有用。

> **交叉参考**
>
> 有关保持相关动作在一起的更多信息，请参见"顺序不重要的语句"。

**特别注意计数器和累加器。** 变量i、j、k、sum和total通常是计数器或累加器。常见错误是在下次使用前忘记重置计数器或累加器。

**在构造函数中初始化类的成员数据。** 就像例程的变量应在每个例程内初始化一样，类的数据应在构造函数中初始化。如果在构造函数中分配了内存，应在析构函数中释放。

**检查重新初始化的需要。** 问问自己变量是否需要重新初始化，可能因为例程中的循环多次使用变量，或者因为变量在例程调用之间保持其值且需要在调用之间重置。如果需要重新初始化，确保初始化语句在重复执行的代码部分内。

**初始化命名常量一次；用可执行代码初始化变量。** 如果你使用变量来模拟命名常量，一次（在程序开始时）初始化它们的代码是可以的。为此，在Startup()例程中初始化它们。在接近使用的地方在可执行代码中初始化真正的变量。最常见的程序修改之一是更改原本调用一次的例程使其被调用多次。在程序级Startup()例程中初始化的变量在第二次通过例程时不会重新初始化。

**使用自动初始化所有变量的编译器设置。** 如果你的编译器支持这样的选项，让编译器设置为自动初始化所有变量是一种依赖编译器的简单变体。但是，依赖特定编译器设置可能在将代码移动到另一台机器和另一个编译器时造成问题。确保记录你对编译器设置的使用；依赖特定编译器设置的假设否则难以发现。

**利用编译器的警告消息。** 许多编译器警告你正在使用未初始化的变量。

**检查输入参数的有效性。** 另一种有价值的初始化形式是检查输入参数的有效性。在将输入值赋给任何东西之前，确保值是合理的。

> **交叉参考**
>
> 有关检查输入参数的更多信息，请参见"保护程序免受无效输入"以及第8章的其余部分。

**使用内存访问检查器检查坏指针。** 在某些操作系统中，操作系统代码检查无效指针引用。在其他系统中，你自己处理。但是，你不必须独自处理，因为你可以购买检查程序指针操作的内存访问检查器。

**在程序开始时初始化工作内存。** 将工作内存初始化为已知值有助于暴露初始化问题。你可以采用几种方法：

- 你可以使用预编程内存填充器用可预测值填充内存。值0对某些目的是好的，因为它确保未初始化的指针指向低内存，使它们在使用时相对容易被检测到。在Intel处理器上，0xCC是一个好值，因为它是断点中断的机器码；如果你在调试器中运行代码并尝试执行数据而不是代码，你将陷入断点。0xCC值的另一个优点是它在内存转储中容易识别——它很少用于合法目的。或者Brian Kernighan和Rob Pike建议使用常量0xDEADBEEF作为在调试器中容易识别的内存填充器（1999）。
- 如果你使用内存填充器，你可以不时更改你用来填充内存的值。震动程序有时会发现如果环境背景从不变化就会隐藏的问题。
- 你可以让程序在启动时初始化其工作内存。使用预编程内存填充器的目的是暴露缺陷，而这种技术的目的是隐藏缺陷。通过每次用相同的值填充工作内存，你保证程序不会受到启动内存随机变化的影响。

## 作用域

"作用域"是思考变量名人身份的方式：它有多出名？作用域或可见性指的是变量在整个程序中被知道和引用的程度。作用域有限或小的变量只在程序的小区域中被知道——例如，只在一个小循环中使用的循环索引。作用域大的变量在程序中许多地方被知道——例如，在整个程序中使用的员工信息表。

不同语言以不同方式处理作用域。在一些原始语言中，所有变量都是全局的。因此你无法控制变量的作用域，这会造成很多问题。在C++和类似语言中，变量可以对块（用花括号括起来的代码段）、例程、类（可能及其派生类）或整个程序可见。在Java和C#中，变量也可以对包或命名空间（类的集合）可见。

以下各节提供适用于作用域的指导原则。

---

### 局部化对变量的引用

变量引用之间的代码是"脆弱窗口"。在窗口中，可能添加新代码，无意中更改变量，或者阅读代码的人可能忘记变量应该包含的值。始终局部化对变量的引用是个好主意，保持它们靠近在一起。

局部化对变量的引用是相当不言自明的，但它是一种适合正式测量的思想。测量变量引用靠近程度的一种方法是计算变量的"跨度"。以下是示例：

**示例10-5。** Java 变量跨度示例

```java
a = 0;
b = 0;
c = 0;
a = b + c;
```

在这种情况下，两行在a的第一次引用和第二次引用之间，所以a的跨度为2。一行在b的两次引用之间，所以b的跨度为1，c的跨度为0。以下是另一个示例：

**示例10-6。** 跨度为一和零的Java示例

```java
a = 0;
b = 0;
c = 0;
b = a + 1;
b = b / c;
```

在这种情况下，b的第一次引用和第二次引用之间有一行，跨度为1。b的第二次引用和第三次引用之间没有行，跨度为0。

> **进一步阅读**
>
> 有关变量跨度的更多信息，请参见《软件工程度量和模型》（Conte, Dunsmore, and Shen 1986）。

平均跨度是通过平均各个跨度计算的。在第二个示例中，对于b，(1+0)/2等于平均跨度0.5。当你保持对变量的引用靠近时，你使阅读代码的人能够一次专注于一个区域。如果引用相距很远，你强迫读者在程序中跳来跳去。因此，保持对变量引用在一起的主要好处是它改进了程序可读性。

---

### 保持变量"活跃"尽可能短的时间

与变量跨度相关的概念是变量"存活时间"，即变量存活的总语句数。变量的生命从第一次引用它的语句开始，在最后一次引用它的语句结束。

与跨度不同，存活时间不受变量在第一次和最后一次引用之间使用次数的影响。如果变量第一次在第1行被引用，最后一次在第25行被引用，它的存活时间是25个语句。如果这些是唯一使用它的两行，它的平均跨度是23个语句。如果变量在第1行到第25行的每一行都被使用，它的平均跨度是0个语句，但它仍然有25个语句的存活时间。图10-1说明了跨度和存活时间。

---

**图10-1。** "长存活时间"意味着变量在许多语句过程中存活。"短存活时间"意味着它只存活几个语句。"跨度"指的是变量引用靠得多近。

---

与跨度一样，关于存活时间的目标是保持数量低，保持变量存活时间尽可能短。与跨度一样，保持低数量的基本好处是它减少了脆弱窗口。你减少了在你打算更改变量的地方之间错误或无意更改变量的机会。

保持短存活时间的第二个好处是它给了你代码的准确图片。如果变量在第10行被赋值，第45行才再次使用，两次引用之间的空间意味着变量在第10到45行之间被使用。如果变量在第44行被赋值，第45行被使用，则不暗示变量的其他使用，当你考虑该变量时，你可以专注于更小的代码区域。

短存活时间也减少了初始化错误的机会。随着你修改程序，直线代码往往变成循环，你往往忘记远离循环的初始化。通过将初始化代码和循环代码保持更近，你减少了修改引入初始化错误的机会。

短存活时间使你的代码更易读。读者一次必须记住的代码行数越少，你的代码就越容易理解。同样，存活时间越短，在编辑和调试期间你想看到变量的所有引用时必须在屏幕上保留的代码就越少。

最后，短的存活时间在将大例程拆分为小例程时很有用。如果对变量的引用保持靠近，将相关代码部分重构为自己例程更容易。

**测量变量的存活时间**

你可以通过计算变量第一次和最后一次引用之间的行数（包括第一行和最后一行）来正式化存活时间的概念。以下是存活时间过长的示例：

**示例10-7。** Java 存活时间过长的变量示例

```java
1   // 初始化所有变量
2   recordIndex = 0;
3   total = 0;
4   done = false;
    ...

27  ...
28     recordIndex = recordIndex + 1;       // 1
       ...

64  while ( !done ) {
       ...
69     if ( total > projectedTotal ) {       // 2
70        done = true;       // 3
```

(1) recordIndex的最后引用。

(2) total的最后引用。

(3) done的最后引用。

以下是本例中变量的存活时间：

- recordIndex: (第28行 - 第2行 + 1) = 27
- total: (第69行 - 第3行 + 1) = 67
- done: (第70行 - 第4行 + 1) = 67

### 关于最小化作用域的评论

许多程序员最小化变量作用域的方法取决于他们对"便利性"和"智力可管理性"问题的看法。一些程序员使他们的许多变量成为全局的，因为全局作用域使变量访问方便，程序员不必在参数列表和类作用域规则周围浪费时间。在他们看来，能够随时访问变量的便利性超过了所涉及的风险。

其他程序员更喜欢尽可能保持变量局部，因为局部作用域有助于智力可管理性。你能隐藏的信息越多，你一次必须记住的越少。你必须记住的越少，因为你忘记需要记住的众多细节之一而犯错误的机会就越小。

> **交叉参考**
>
> 最小化作用域的思想与信息隐藏的思想相关。详细信息请参见"设计构建模块：启发法"中的"隐藏秘密（信息隐藏）"。

"便利性"哲学和"智力可管理性"哲学之间的差异归根结底是编写程序和阅读程序之间的重点差异。最大化作用域确实可能使程序易于编写，但任何例程可以在任何时候使用任何变量的程序比使用良好分解例程的程序更难理解。在这样的程序中，你不能只理解一个例程；你必须理解与那个例程共享全局数据的所有其他例程。这样的程序难以阅读、难以调试、难以修改。

因此，你应该声明每个变量使其只需要看到它的最小代码段可见。如果你可以将变量的作用域限制在单个循环或单个例程中，很好。如果你不能将作用域限制在一个例程中，将可见性限制在单个类的例程中。如果你不能将变量作用域限制在对变量最负责的类中，创建访问例程与其它类共享变量的数据。你会发现你很少（如果有的话）需要使用裸全局数据。

> **交叉参考**
>
> 有关使用访问例程的详细信息，请参见"全局数据"中的"使用访问例程而不是全局数据"。

## 持久性

"持久性"是数据生命周期的另一个词。持久性有几种形式。一些变量：

- **持续特定代码块或例程的生命。** C++或Java中在for循环内声明的变量是这种持久性的例子。
- **只要你允许就持续。** 在Java中，用new创建的变量持续到被垃圾收集。在C++中，用new创建的变量持续到你删除它们。
- **持续程序的生命。** 大多数语言中的全局变量符合这个描述，C++和Java中的静态变量也是。
- **永远持续。** 这些变量可能包括你在程序执行之间存储在数据库中的值。例如，如果你有一个交互式程序，用户可以自定义屏幕颜色，你可以将他们的颜色存储在文件中，然后在每次加载程序时读取它们。

持久性的主要问题出现在你假设变量比实际有更长持久性时。变量就像你冰箱里的那罐牛奶。它应该持续一周。有时它持续一个月，有时五天后就变酸了。变量可能同样不可预测。如果你在变量的正常生命周期结束后尝试使用它的值，它会保留其值吗？有时变量中的值是酸的，你知道你有了错误。其他时候，计算机在变量中保留旧值，让你以为你正确使用了它。

以下是避免这类问题你可以采取的几个步骤：

- **在程序中使用调试代码或断言检查关键变量的合理值。** 如果值不合理，显示警告让你寻找不正确的初始化。
- **在你用完变量后将变量设置为"不合理的值"。** 例如，你可以在删除后将指针设置为null。
- **编写假设数据不持久的代码。** 例如，如果变量在你退出例程时有某个值，不要假设下次进入例程时它有相同的值。这不适用于如果你使用保证值保持不变的特定语言特性，如C++和Java中的static。
- **养成在使用之前立即声明和初始化所有数据的习惯。** 如果你看到数据在没有附近初始化的情况下被使用，要警惕！

## 绑定时间

与程序维护和可修改性有深远影响的一个初始化主题是"绑定时间"：变量及其值绑定在一起的时间（Thimbleby 1988）。它们是在代码编写时绑定？编译时？加载时？程序运行时？还是其他时间？

使用尽可能晚的绑定时间可能对你有利。一般来说，你越晚进行绑定时间，你在代码中构建的灵活性就越大。下一个示例显示在最早可能的时间绑定，即代码编写时：

**示例10-12。** Java 在代码编写时绑定的变量示例

```java
titleBar.color = 0xFF; // 0xFF是颜色蓝色的十六进制值
```

值0xFF在代码编写时绑定到变量titleBar.color，因为0xFF是硬编码到程序中的字面量值。像这样的硬编码几乎总是一个坏主意，因为如果这个0xFF改变，它可能与你必须在其中使用相同值的代码中其他地方使用的0xFF失去同步。

以下是稍后绑定的一个示例，在编译时：

**示例10-13。** Java 在编译时绑定的变量示例

```java
private static final int COLOR_BLUE = 0xFF;
private static final int TITLE_BAR_COLOR = COLOR_BLUE;
...
titleBar.color = TITLE_BAR_COLOR;
```

TITLE_BAR_COLOR是一个命名常量，编译器在编译时替换其值的表达式。如果你的语言支持，这几乎总是比硬编码好。它提高了可读性，因为TITLE_BAR_COLOR告诉你比0xFF更多关于正在表示什么。它使更改标题栏颜色更容易，因为一次更改说明了所有出现。它不会产生运行时性能惩罚。

以下是较晚绑定的示例，在运行时：

**示例10-14。** Java 在运行时绑定的变量示例

```java
titleBar.color = ReadTitleBarColor();
```

ReadTitleBarColor()是一个在程序执行时读取值的例程，可能从Microsoft Windows注册表文件或Java属性文件读取。

代码比值被硬编码时更易读和灵活。你不需要更改程序来更改titleBar.color；你只需更改ReadTitleBarColor()读取的源内容。这种方法常用于用户可以自定义应用环境的交互式应用程序。

关于ReadTitleBarColor()例程何时被调用还有另一种绑定时间变化。那个例程可以在程序加载时调用一次，可以在每次创建窗口时调用，也可以在每次绘制窗口时调用——每个替代方案代表连续更晚的绑定时间。

总结，以下是本例中变量可以绑定到值的时间：（细节在其他情况下可能有所不同。）

- **编码时（使用幻数）**
- **编译时（使用命名常量）**
- **加载时（从外部源如Windows注册表文件或Java属性文件读取值）**
- **对象实例化时（例如，每次创建窗口时读取值）**
- **恰好及时（例如，每次绘制窗口时读取值）**

一般来说，越早绑定时间，越低灵活性、越低复杂性。对于前两个选项，使用命名常量优于使用幻数，所以通过使用良好编程实践你可以获得命名常量提供的灵活性。除此之外，期望的灵活性越大，支持该灵活性的代码复杂性越高，代码越容易出错。因为成功编程依赖于最小化复杂性，有技能的程序员将构建满足软件需求的尽可能多的灵活性，但不会添加超出需要的灵活性——以及相关复杂性。

## 使每个变量只用于一个目的

以几种微妙方式将变量用于多个目的是可能的。你最好没有这种微妙。

**每个变量只用于一个目的。** 有时在一个地方为两种不同活动使用一个变量是很诱人的。通常，变量对其中一个用途命名不当，或者"临时"变量在两种情况下都使用（通常名称无帮助如x或temp）。以下是显示用于两个目的的临时变量的示例：

**示例10-15。** C++ 一个变量用于两个目的示例——糟糕做法

```cpp
// 计算二次方程的根。
// 此代码假设(b*b-4*a*c)为正。
temp = Sqrt( b*b - 4*a*c );
root[0] = ( -b + temp ) / ( 2 * a );
root[1] = ( -b - temp ) / ( 2 * a );
...

// 交换根
temp = root[0];
root[0] = root[1];
root[1] = temp;
```

问题：temp在前几行和后几行之间的关系是什么？答案：两个temp没有关系。在两种情况下使用相同变量使它们看起来像相关的而实际上不相关。为每个目的创建唯一变量使你的代码更易读。以下是改进：

> **交叉参考**
>
> 例程参数也应该只用于一个目的。例程参数使用的详细信息，请参见"如何使用例程参数"。

**示例10-16。** C++ 两个变量用于两个目的示例——好做法

```cpp
// 计算二次方程的根。
// 此代码假设(b*b-4*a*c)为正。
discriminant = Sqrt( b*b - 4*a*c );
root[0] = ( -b + discriminant ) / ( 2 * a );
root[1] = ( -b - discriminant ) / ( 2 * a );
...

// 交换根
oldRoot = root[0];
root[0] = root[1];
root[1] = oldRoot;
```

**避免具有隐藏含义的变量。** 变量用于多个目的的另一种方式是让变量的不同值意味着不同的事情。例如：

- 变量pageCount可能代表打印的页数，除非它等于-1，在这种情况下它表示发生了错误。
- 变量customerId可能代表客户号，除非它的值大于500,000，在这种情况下你减去500,000得到违约账户的号码。
- 变量bytesWritten可能是写入输出文件的字节数，除非它的值为负，在这种情况下它表示用于输出的驱动器号码。

避免具有这些隐藏含义的变量。这种滥用的技术名称是"混合耦合"（Page-Jones 1988）。变量被延伸到两个工作，意味着变量对其中一个工作是错误的类型。在pageCount示例中，pageCount通常指示页数；它是一个整数。然而当pageCount是-1时，它指示发生了错误；整数兼职做布尔值！

即使双重使用对你清楚，对别人也不会。通过使用两个变量保存两种信息，你将实现的额外清晰会让你惊讶。没有人会抱怨你使用额外存储。

**确保所有声明的变量都被使用。** 与变量用于多个目的相反是完全不使用变量。Card、Church和Agresti进行的一项研究发现，未引用变量与更高错误率相关（1986）。养成检查确保声明的所有变量都被使用的习惯。一些编译器和实用程序（如lint）将未使用变量报告为警告。

## 关键点

- 数据初始化容易出错，因此使用本章描述的初始化技术来避免意外初始值造成的问题。
- 最小化每个变量的作用域。保持对变量的引用靠近。使其对例程或类局部。避免全局数据。
- 保持使用相同变量的语句尽可能靠近。
- 早绑定倾向于限制灵活性但最小化复杂性。晚绑定倾向于增加灵活性但以增加复杂性为代价。
- 每个变量只用于一个且仅一个目的。
