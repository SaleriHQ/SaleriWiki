## 类基础：抽象数据类型（ADT）

抽象数据类型是数据和操作这些数据的操作的集合。操作既向程序的其余部分描述数据，也允许程序其余部分更改数据。"抽象数据类型"中的"数据"一词使用得较为宽泛。ADT可能是带有影响它的所有操作的图形窗口、文件和文件操作、保险费率表及其上的操作，或其他内容。

理解ADT对于理解面向对象编程至关重要。如果不理解ADT，程序员创建的类只是名义上的"类"——实际上，它们不过是松散相关的数据和例程的便捷携带容器。有了对ADT的理解，程序员可以创建最初易于实现、随时间易于修改的类。

> **交叉参考**
>
> 首先考虑ADT然后考虑类是"编程进入语言而非在语言中编程"的一个例子。参见"你在技术浪潮中的位置"以及"编程进入你的语言，而非在语言中编程"。

传统上，当编程书籍到达抽象数据类型主题时会变得数学化。它们倾向于做出这样的陈述："人们可以将抽象数据类型视为具有在其上定义的操作集合的数学模型。"这样的书籍让人觉得，除了作为催眠助剂，你永远不会真正使用抽象数据类型。

这种对抽象数据类型的干巴巴解释完全偏离了要点。抽象数据类型之所以令人兴奋，是因为你可以用它们来处理真实世界的实体，而不是低级实现实体。你可以在电子表格中添加单元格、向窗口类型列表添加新类型的窗口、或向火车模拟添加另一个客车车厢——而不是在链表中插入节点。利用能够在问题域而非低级实现域工作的力量！

### ADT需求的示例

为了开始，这里有一个ADT有用的案例。我们将在有示例讨论之后了解细节。

假设你正在编写一个程序，使用多种字族、磅值大小和字体属性（如粗体和斜体）来控制屏幕文本输出。程序的一部分操作文本的字体。如果你使用ADT，你将有一组字体例程与它们操作的数据——字族名称、磅值大小和字体属性——捆绑在一起。字体例程和数据的集合就是一个ADT。

如果你不使用ADT，你将采用临时方法来操作字体。例如，如果你需要将字体大小设置为12磅（恰好是16像素高），你将有如下代码：

```cpp
currentFont.size = 16
```

如果你已经建立了库例程集合，代码可能稍微更易读：

```cpp
currentFont.size = PointsToPixels( 12 )
```

或者你可以为属性提供更具体的名称，比如：

```cpp
currentFont.sizeInPixels = PointsToPixels( 12 )
```

但你不能同时拥有currentFont.sizeInPixels和currentFont.sizeInPoints，因为如果两个数据成员都参与其中，currentFont将没有办法知道应该使用这两个中的哪一个。如果你需要在程序的多处更改大小，你将在整个程序中看到类似的代码行。

如果你需要将字体设置为粗体，你可能有如下使用逻辑或和十六进制常量0x02的代码：

```cpp
currentFont.attribute = currentFont.attribute or 0x02
```

如果你运气好，你会有比这更干净的代码，但使用临时方法你能得到的最好结果是：

```cpp
currentFont.attribute = currentFont.attribute or BOLD
```

或者可能是这样：

```cpp
currentFont.bold = True
```

与字体大小一样，限制在于客户端代码需要直接控制数据成员，这限制了currentFont的使用方式。

如果你以这种方式编程，你的程序中多处会有类似的代码行。

### 使用ADT的好处

问题不在于临时方法是不好的编程实践。而是你可以用更好的编程实践来替代这种方法，从而产生以下好处：

**你可以隐藏实现细节。** 隐藏字体数据类型的实现细节意味着，如果数据类型改变，你可以在一个地方更改它而不影响整个程序。例如，除非你在ADT中隐藏了实现细节，否则将数据类型从第一种粗体表示改为第二种将需要在每个设置粗体的地方更改程序，而不是只在一个地方。更改信息还可以在你决定将数据存储在外部存储而非内存中或用另一种语言重写所有字体操作例程时保护程序的其余部分。

**更改不会影响整个程序。** 如果字体需要更丰富并支持更多操作（如切换到小型大写字母、上标、删除线等），你可以在一个地方更改程序。更改不会影响程序的其余部分。

**你可以使接口更具信息性。** 像`currentFont.size = 16`这样的代码是模糊的，因为16可以是像素大小或磅值大小。上下文不会告诉你哪个是哪个。将所有类似操作收集到一个ADT中允许你完全用磅值定义整个接口，或用像素，或清楚地区分两者，这有助于避免混淆它们。

**更容易提高性能。** 如果你需要提高字体性能，你可以重新编写几个定义良好的例程，而不是遍历整个程序。

**程序更明显正确。** 你可以用验证`currentFont.SetBoldOn()`调用是否正确的更简单任务来替换验证`currentFont.attribute = currentFont.attribute or 0x02`这样的语句是否正确的更繁琐任务。对于第一条语句，你可能使用了错误的结构名称、错误的字段名称、错误的操作（用and而不是or），或错误的属性值（0x20而不是0x02）。在第二种情况下，`currentFont.SetBoldOn()`调用唯一可能出错的是调用了错误的例程名称，所以更容易判断它是否正确。

**程序变得更具自文档性。** 你可以用BOLD或0x02代表的任何内容替换`currentFont.attribute or 0x02`这样的语句来改进代码，但这与`currentFont.SetBoldOn()`这样的例程调用的可读性无法相比。

Woodfield、Dunsmore和Shen进行了一项研究，其中研究生和高年级计算机科学本科生回答了关于两个程序的问题：一个按功能线分为八个例程，另一个分为八个抽象数据类型例程（1981年）。使用抽象数据类型程序的学生比使用功能版本的学生得分高30%以上。

**你不必在整个程序中传递数据。** 在刚才展示的示例中，你必须直接更改currentFont或将其传递给每个使用字体的例程。如果你使用抽象数据类型，你不必在整个程序中传递currentFont，也不必将其变成全局数据。ADT有一个包含currentFont数据的结构。数据只能由作为ADT一部分的例程直接访问。不是ADT一部分的例程不必担心数据。

**你能够使用真实世界的实体而不是低级实现结构。** 你可以定义处理字体的操作，使程序的绝大部分仅用字体术语而不是数组访问、结构定义以及True和False来工作。

在这种情况下，要定义一个抽象数据类型，你要定义几个控制字体的例程——大概像这样：

```cpp
currentFont.SetSizeInPoints( sizeInPoints )
currentFont.SetSizeInPixels( sizeInPixels )
currentFont.SetBoldOn()
currentFont.SetBoldOff()
currentFont.SetItalicOn()
currentFont.SetItalicOff()
currentFont.SetTypeFace( faceName )
```

这些例程内部的代码可能很短——可能类似于你在前面字体问题的临时方法中看到的代码。区别在于你已经将字体操作隔离在一组例程中。这为程序的其余部分提供了一个更好的抽象层次来处理字体，并为你提供了一层保护以防止字体操作的更改。

coolingSystem.GetTemperature()
coolingSystem.SetCirculationRate( rate )

从这些示例的研究中，你可以得出几条指导原则；这些原则在以下小节中描述：

**将典型的低级数据类型构建或用作ADT，而不是用作低级数据类型。** 大多数关于ADT的讨论都集中在将典型低级数据类型表示为ADT。正如你从示例中看到的，你可以将栈、列表和队列以及几乎任何其他典型数据类型表示为ADT。

你需要问的问题是："这个栈、列表或队列代表什么？"如果栈代表一组员工，将ADT视为员工而不是栈。如果列表代表一组账单记录，将其视为账单记录而不是列表。如果队列代表电子表格中的单元格，将其视为单元格集合，而不是队列中的通用项。让自己享受尽可能高的抽象层次。

**将文件等常见对象视为ADT。** 大多数语言都包含一些你可能熟悉但可能不认为是ADT的抽象数据类型。文件操作就是一个很好的例子。在写入磁盘时，操作系统使你免于在特定物理地址定位读写头、在你用完旧扇区时分配新磁盘扇区、以及解释神秘错误代码的痛苦。操作系统提供了第一层抽象和该层的ADT。高级语言提供了更高层次的抽象和更高级别的ADT。高级语言保护你免受生成操作系统调用和操作数据缓冲区的混乱细节的影响。它允许你将一块磁盘空间视为"文件"。

你可以类似地分层ADT。如果你想在提供数据结构级别操作（如栈的压入和弹出）的级别使用ADT，那很好。你可以在其上创建另一个层次，以真实世界问题的层次工作。

**即使简单的项也要视为ADT。** 你不需要一个令人生畏的数据类型来证明使用抽象数据类型的合理性。示例列表中的一个ADT是一个只支持两个操作的灯——打开它和关闭它。你可能认为将简单的"开"和"关"操作隔离在自己的例程中是浪费，但即使简单的操作也可以从使用ADT中受益。将灯及其操作放入ADT使代码更具自文档性、更易于更改，将潜在更改影响限制在TurnLightOn()和TurnLightOff()例程中，并减少你必须传递的数据项数量。

**独立于存储介质引用ADT。** 假设你有一个保险费率表，它太大了，以至于总是存储在磁盘上。你可能很想将其称为"费率文件"并创建访问例程，如RateFile.Read()。然而，当你将其称为文件时，你暴露了比需要更多的数据信息。如果有一天你更改程序使表在内存中而不是磁盘上，那么将其称为文件的代码将是不正确的、误导性的和令人困惑的。尝试使类和访问例程的名称独立于数据的存储方式，而是引用抽象数据类型（如保险费率表）。这样你的类和访问例程名称将像rateTable.Read()或简单地rates.Read()。

### 在非面向对象环境中使用ADT处理多个数据实例

面向对象语言为处理ADT的多个实例提供自动支持。如果你只在面向对象环境中工作过，从未需要自己处理多个实例的实现细节，你应该庆幸！（你也可以继续阅读下一节"ADT和类"。）

如果你在非面向对象环境（如C）中工作，你需要手动构建对多个实例的支持。一般来说，这意味着包括ADT创建和删除实例的服务，并设计ADT的其他服务，使它们能够处理多个实例。

原始字体ADT提供这些服务：

```cpp
currentFont.SetSize( sizeInPoints )
currentFont.SetBoldOn()
currentFont.SetBoldOff()
currentFont.SetItalicOn()
currentFont.SetItalicOff()
currentFont.SetTypeFace( faceName )
```

在非面向对象环境中，这些函数不会附加到类，看起来更像这样：

```cpp
SetCurrentFontSize( sizeInPoints )
SetCurrentFontBoldOn()
SetCurrentFontBoldOff()
SetCurrentFontItalicOn()
SetCurrentFontItalicOff()
SetCurrentFontTypeFace( faceName )
```

如果你想一次处理多个字体，你需要添加创建和删除字体实例的服务——大概是这样：

```cpp
CreateFont( fontId )
DeleteFont( fontId )
SetCurrentFont( fontId )
```

fontId的概念被添加为跟踪创建的多个字体的方式。对于其他操作，你可以从三种处理ADT接口的方式中选择：

**选项1：每次使用ADT服务时显式标识实例。** 在这种情况下，你没有"当前字体"的概念。你将fontId传递给每个操作字体的例程。字体函数跟踪任何底层数据，客户端代码只需要跟踪fontId。这需要在每个字体例程中将fontId作为参数添加。

**选项2：显式提供ADT服务使用的数据。** 在这种方法中，你在每个使用ADT服务的例程中声明ADT使用的数据。换句话说，你创建一个字体数据类型，将其传递给每个ADT服务例程。你必须设计ADT服务例程，使它们每次调用时使用传递给它们的字体数据。如果你使用这种方法，客户端代码不需要字体ID，因为它自己跟踪字体数据。（即使数据可直接从字体数据类型获取，你也应该只使用ADT服务例程访问它。这称为保持结构"闭合"。）

这种方法的优点是ADT服务例程不必根据字体ID查找字体信息。缺点是它向程序的其余部分暴露了字体数据，这增加了客户端代码使用ADT实现细节的可能性，而这些细节本应隐藏在ADT内。

**选项3：使用隐式实例（谨慎使用）。** 设计一个新的服务来调用以使特定字体实例成为当前实例——比如SetCurrentFont(fontId)。设置当前字体使所有其他服务在调用时使用当前字体。如果你使用这种方法，你不需要将fontId作为其他服务的参数。对于简单的应用程序，这可以简化多个实例的使用。对于复杂的应用程序，这种系统范围的依赖状态意味着你必须在使用字体函数的整个代码中跟踪当前字体实例。复杂性往往会扩散，对于任何规模的应用程序，存在更好的替代方案。

在抽象数据类型内部，你有丰富的选项来处理多个实例，但在外部，如果你使用非面向对象语言，这就是选择总结。

### ADT和类

抽象数据类型为类的概念奠定了基础。在支持类的语言中，你可以将每个抽象数据类型实现为自己的类。类通常涉及继承和多态性的额外概念。思考类的一种方式是将其视为抽象数据类型加上继承和多态性。

## 良好的类接口

创建高质量类的第一步（也许是最重要的一步）是创建一个良好的接口。这包括为接口创建一个良好的抽象，并确保细节隐藏在抽象之后。

---

### 良好的抽象

正如"设计构建模块：启发法"中的"形成一致的抽象"所述，抽象是以简化形式查看复杂操作的能力。类接口提供了隐藏在接口之后的实现的抽象。类的接口应该提供一组明显属于一起的例程。

你可能有一个实现员工的类。它包含描述员工姓名、地址、电话号码等的数据。它提供初始化和使用员工的服务。以下是它的样子：

**示例6-1。** C++ 良好抽象的类接口示例

```cpp
class Employee {
public:
   // 公共构造函数和析构函数
   Employee();
   Employee(
      FullName name,
      String address,
      String workPhone,
      String homePhone,
      TaxId taxIdNumber,
      JobClassification jobClass
   );
   virtual ~Employee();
   
   // 公共例程
   FullName GetName() const;
   String GetAddress() const;
   String GetWorkPhone() const;
   String GetHomePhone() const;
   TaxId GetTaxIdNumber() const;
   JobClassification GetJobClassification() const;
   ...
private:
   ...
};
```

> **交叉参考**
>
> 本书中的代码示例使用强调多种语言风格相似性的编码约定。有关该约定（以及多种编码风格讨论）的详细信息，请参阅"非正式命名约定"中的"混合语言编程注意事项"。

在这个类的内部，可能有支持这些服务的额外例程和数据，但类的用户不需要知道任何关于它们的信息。类接口抽象很棒，因为接口中的每个例程都朝着一致的目标努力。

呈现糟糕抽象的类将包含一组杂乱的函数。以下是一个示例：

**示例6-2。** 糟糕抽象的类接口示例

```cpp
class Program {
public:
   ...
   // 公共例程
   void InitializeCommandStack();
   void PushCommand( Command command );
   Command PopCommand();
   void ShutdownCommandStack();
   void InitializeReportFormatting();
   void FormatReport( Report report );
   void PrintReport( Report report );
   void InitializeGlobalData();
   void ShutdownGlobalData();
   ...
private:
   ...
};
```

假设一个类包含处理命令栈、格式化报告、打印报告和初始化全局数据的例程。命令栈和报告例程或全局数据之间很难看到任何联系。类接口没有呈现一致的抽象，所以类的内聚性很差。例程应该被重新组织成更专注的类，每个类在其接口中提供更好的抽象。

如果这些例程是Program类的一部分，可以将它们修改为呈现一致的抽象，如下所示：

**示例6-3。** 更好抽象的类接口示例

```cpp
class Program {
public:
   ...
   // 公共例程
   void InitializeUserInterface();
   void ShutDownUserInterface();
   void InitializeReports();
   void ShutDownReports();
   ...
private:
   ...
};
```

这个接口的清理假设一些原始例程被移动到其他更合适的类，一些被转换为InitializeUserInterface()和其他例程使用的私有例程。

这个类抽象评估是基于类的公共例程集合——即类的接口。类内部的例程不一定仅仅因为整个类是好的就呈现良好的单个抽象，但它们也需要被设计为呈现良好的抽象。关于这方面的指导原则，请参见"例程级别的设计"。

追求良好的抽象接口引出了几条创建类接口的指导原则。

**在类接口中呈现一致的抽象层次** 思考类的好方法是将其作为实现"类基础：抽象数据类型（ADT）"中描述的抽象数据类型的机制。每个类应该只实现一个ADT。如果你发现一个类实现了多个ADT，或者如果你不能确定类实现了什么ADT，是时候将类重新组织成一个或多个定义良好的ADT了。

以下是一个类接口的示例，其抽象层次不一致：

**示例6-4。** 混合抽象层次的类接口示例

```cpp
class EmployeeCensus: public ListContainer {
public:
   ...
   // 公共例程
   void AddEmployee( Employee employee );       // 1
   void RemoveEmployee( Employee employee );       // 1

   Employee NextItemInList();       // 2
   Employee FirstItem();              |
   Employee LastItem();       // 2
   ...
private:
   ...
};
```

(1) 这些例程的抽象在"员工"层次。

(2) 这些例程的抽象在"列表"层次。

这个类呈现了两个ADT：Employee和ListContainer。这种混合抽象通常在程序员使用容器类或其他库类进行实现且没有隐藏使用了库类的事实时出现。问问你自己，使用容器类是否应该成为抽象的一部分。通常这是应该从程序其余部分隐藏的实现细节，像这样：

**示例6-5。** 一致抽象层次的类接口示例

```cpp
class EmployeeCensus: public ListContainer {
public:
   ...
   // 公共例程

   void AddEmployee( Employee employee );       // 1
   void RemoveEmployee( Employee employee );      |
   Employee NextEmployee();                       |
   Employee FirstEmployee();                      |
   Employee LastEmployee();       // 1
   ...

   ListContainer m_EmployeeList;       // 2
   ...
};
```

(1) 所有这些例程的抽象现在都在"员工"层次。

(2) 类使用ListContainer库的事实现在是隐藏的。

程序员可能会争辩说，从ListContainer继承很方便，因为它支持多态性，允许接受ListContainer对象的外部搜索或排序函数。这个论点无法通过继承的主要测试，即"继承只用于'is a'关系吗？"从ListContainer继承意味着EmployeeCensus"是一个"ListContainer，这显然不正确。如果EmployeeCensus对象的抽象是它可以被搜索或排序，那应该作为类接口的明确、一致的部分纳入。

如果你将类的公共例程视为防止水进入潜艇的气闸，不一致的公共例程就是类中的漏水面板。漏水面板可能不会像打开的气闸那样快地让水进来，但如果你给它们足够的时间，它们仍然会让船沉没。在实践中，这就是混合抽象层次时发生的情况。当程序被修改时，混合的抽象层次使程序越来越难以理解，它逐渐退化直到变得无法维护。

**确保你理解类正在实现的抽象。** 有些类足够相似，你必须小心理解类接口应该捕获哪个抽象。我曾经参与过一个需要允许以表格格式编辑信息的程序。我们想使用一个简单的网格控件，但可用的网格控件不允许我们对数据输入单元格着色，所以我们决定使用提供该功能的电子表格控件。

电子表格控件比网格控件复杂得多，向网格控件的15个例程提供了大约150个例程。既然我们的目标是使用网格控件而不是电子表格控件，我们指派一名程序员编写一个包装类来隐藏我们正在使用电子表格控件作为网格控件的事实。程序员抱怨了很多关于不必要的开销和官僚作风，走开了，几天后带着一个忠实地暴露了电子表格控件全部150个例程的包装类回来了。

这不是所需要的。我们想要一个封装了我们正在使用更复杂的电子表格控件这一事实的网格控制接口。程序员应该只暴露15个网格控件例程加上一个支持单元格着色的第16个例程。通过暴露全部150个例程，程序员创造了这种可能性：如果有一天我们想要更改底层实现，我们可能会发现自己支持150个公共例程。程序员未能实现我们想要的封装，也给自己创造了比必要更多的工作。

根据具体情况，正确的抽象可能是电子表格控件或网格控件。当你必须在两个相似的抽象之间选择时，确保你选择正确的。

**成对提供服务及其反面。** 大多数操作都有对应的、相等且相反的操作。如果你有一个打开灯的操作，你可能需要一个关闭它的操作。如果你有一个向列表添加项的操作，你可能需要一个从列表删除项的操作。如果你有一个激活菜单项的操作，你可能需要一个停用项的操作。当你设计一个类时，检查每个公共例程以确定你是否需要它的补充。不要无端创建反面，但一定要检查是否需要一个。

**将不相关的信息移动到另一个类。** 在某些情况下，你会发现类的一半例程处理一半的类数据，而另一半例程处理另一半数据。在这种情况下，你实际上有两个类伪装成一个。将它们拆分！

**尽可能使接口成为程序化的而非语义的。** 每个接口由程序化部分和语义部分组成。程序化部分由接口的数据类型和其他属性组成，这些属性可以由编译器强制执行。接口的语义部分包括关于接口将如何使用的前提假设，这些不能由编译器强制执行。语义接口包括诸如"RoutineA必须在RoutineB之前调用"或"如果dataMember1在传递给RoutineA之前没有初始化，RoutineA将崩溃"这样的考虑。语义接口应该在注释中记录，但尝试使接口最小化依赖文档。接口的任何不能由编译器强制执行的方面都是可能被误用的方面。寻找使用Assert或其他技术将语义接口元素转换为程序化接口元素的方法。

**注意修改时接口抽象的侵蚀。** 当一个类被修改和扩展时，你经常发现需要额外的功能，这些功能与原始类接口不太吻合，但似乎太难用其他方式实现了。例如，在Employee类中，你可能发现类演变成这样：

**将抽象和内聚一起考虑。** 抽象和内聚的思想密切相关——呈现良好抽象的类接口通常具有强内聚。具有强内聚的类往往呈现良好的抽象，尽管这种关系不那么强。

### 良好的封装

正如"设计构建模块：启发法"所述，封装是比抽象更强的概念。抽象通过提供允许你忽略实现细节的模型来帮助管理复杂性。封装是防止你即使想看也看不到细节的强制执行者。

> **交叉参考**
>
> 有关封装的更多信息，请参见"设计构建模块：启发法"中的"封装实现细节"。

这两个概念是相关的，因为如果没有封装，抽象往往会崩溃。根据我的经验，你要么同时拥有抽象和封装，要么两者都没有。没有中间立场。

**最小化类和成员的可达性。** 最小化可达性是旨在鼓励封装的几条规则之一。如果你疑惑特定例程应该是public、private还是protected，一种观点认为你应该倾向于最严格的、可行的隐私级别（Meyers 1998，Bloch 2001）。我认为这是一个很好的指导原则，但我认为更重要的指导原则是："什么最能保持接口抽象的完整性？"如果暴露例程与抽象一致，可能暴露它没问题。如果你不确定，隐藏更多通常比隐藏更少好。

> 区分设计良好的模块和设计不佳的模块的最重要因素是模块向其他模块隐藏其内部数据和其他实现细节的程度。
> —— Joshua Bloch

**不要在public中暴露成员数据。** 暴露成员数据违反封装，限制你对抽象的控制。正如Arthur Riel指出的，暴露的Point类

```cpp
float x;
float y;
float z;
```

违反了封装，因为客户端代码可以随意摆弄Point的数据，而Point甚至不一定知道其值何时被更改（Riel 1996）。然而，暴露的Point类

```cpp
float GetX();
float GetY();
float GetZ();
void SetX( float x );
void SetY( float y );
void SetZ( float z );
```

保持了完美的封装。你完全不知道底层实现是用浮点数x、y和z，还是Point将它们存储为双精度浮点数并在需要时转换为浮点数，或者Point将它们存储在月球上并从外层空间的卫星检索它们。

**避免将私有实现细节放入类的接口中。** 有了真正的封装，程序员根本不应该能够看到实现细节。它们在比喻上和字面上都是隐藏的。然而，在流行语言中，包括C++，语言的结构要求程序员在类接口中披露实现细节。以下是一个示例：

**示例6-7。** C++ 暴露类实现细节的示例

```cpp
class Employee {
public:
   ...
   Employee(
      FullName name,
      String address,
      String workPhone,
      String homePhone
      TaxId taxIdNumber
      JobClassification jobClass
   );
   ...
   FullName GetName() const;
   String GetAddress() const;
   ...
private:
   String m_name;       // 1
   String m_Address;      |
   int m_jobClass;       // 1
   ...
};
```

(1) 这里暴露了实现细节。

在类头文件中包含私有声明可能看起来是小小的违规，但它鼓励其他程序员检查实现细节。在这种情况下，客户端代码打算使用Address类型处理地址，但头文件暴露了地址存储为String的实现细节。

Scott Meyers在《Effective C++》第2版的第34条中描述了解决此问题的常见方法。你将类接口与类实现分开。在类声明中，包含指向类实现的指针，但不包含任何其他实现细节。

**示例6-8。** C++ 隐藏类实现细节的示例

```cpp
class Employee {
public:
   ...
   Employee( ... );
   ...
   FullName GetName() const;
   String GetAddress() const;
   ...
private:
   EmployeeImplementation *m_implementation;       // 1
};
```

(1) 实现细节在这里隐藏在指针之后。

现在你可以将实现细节放入EmployeeImplementation类中，它应该只对Employee类可见，而不对使用Employee类的代码可见。

如果你已经编写了很多不为项目使用这种方法的代码，你可能认为将大量现有代码转换为使用这种方法不值得。但当你阅读不隐藏实现细节的代码时，你可以抵制在类接口的私有部分中寻找实现线索的冲动。

**不要对类的用户做出假设。** 一个类应该被设计和实现以遵守类接口隐含的契约。它不应该对接口将如何使用或不会如何使用做出任何假设，除接口文档中记录的内容。像下面这样的注释表明一个类比它应该的更了解其用户：

```cpp
-- 将x、y和z初始化为1.0，因为DerivedClass如果初始化为0.0会崩溃
```

**避免friend类。** 在某些情况下，如状态模式，friend类可以以纪律严明的方式使用，有助于管理复杂性（Gamma等人1995）。但一般来说，friend类违反封装。它们扩大了你任何时候都必须考虑的代码量，从而增加复杂性。

**不要仅仅因为一个例程只使用公共例程就将其放入公共接口。** 一个例程只使用公共例程这一事实不是一个重要考虑因素。相反，要问暴露例程是否与接口呈现的抽象一致。

**倾向于读时方便而非写时方便。** 代码被读的次数远远多于被写的次数，即使在初始开发期间也是如此。倾向于以牺牲读时方便为代价来提高写时方便的技术是虚假的经济。这特别适用于类接口的创建。即使一个例程与接口抽象不太吻合，有时你很想为你当时正在处理的类的特定客户端添加一个方便的例程。但添加该例程是滑坡的第一步，最好不要迈出第一步。

**非常、非常警惕语义上的封装违反。** 有一次我认为当我学会如何避免语法错误时我就自由了。我很快发现，学会如何避免语法错误只是为我购买了一张进入全新编码错误剧院的门票，其中大多数比语法错误更难诊断和纠正。

> 如果你必须查看底层实现才能理解发生了什么，那它就不是抽象的。
> —— P. J. Plauger

与句法封装相比，语义封装的难度类似。句法上，相对容易避免将鼻子探入另一个类的内部工作，方法只是将类的内部例程和数据声明为私有。实现语义封装是另一回事。以下是类的用户可能语义上破坏封装的几种方式：

- 不调用Class A的InitializeOperations()例程，因为你知道Class A的PerformFirstOperation()例程会自动调用它。
- 在调用employee.Retrieve(database)之前不调用database.Connect()，因为你知道如果没有现有连接，employee.Retrieve()函数将连接到数据库。
- 不调用Class A的Terminate()例程，因为你知道Class A的PerformFinalOperation()例程已经调用了它。
- 即使在ObjectA超出范围后仍使用ObjectA创建的ObjectB的指针或引用，因为你知道ObjectA将ObjectB存储在静态存储中，ObjectB仍然有效。
- 使用Class B的MAXIMUM_ELEMENTS常量而不是使用ClassA.MAXIMUM_ELEMENTS，因为你知道它们都等于相同的值。

这些示例中每个问题在于，它们使客户端代码依赖于类的私有实现，而不是公共接口。每当你发现自己查看类的实现以弄清楚如何使用类时，你不是在编程面向接口；你是通过接口编程到实现。如果你正在通过接口编程，封装就被破坏了，一旦封装开始崩溃，抽象也即将随之而来。

如果你仅基于接口文档无法弄清楚如何使用类，正确的回应不是拉起源代码并查看实现。那是好的主动性但糟糕的判断。正确的回应是联系类的作者说"我无法弄清楚如何使用这个类。"类的作者正确的回应不是面对面回答你的问题。类的作者正确的回应是签出类接口文件，修改类接口文档，重新签入文件，然后说"看看现在你是否能理解它是如何工作的。"你希望这个对话发生在接口代码本身中，以便为未来的程序员保留。你不希望对话只发生在你自己的脑海中，那会将微妙的语义依赖烘焙到使用类的客户端代码中。你也不希望对话以人际方式进行，那样只使你的代码受益而使其他人的代码不受益。

**注意过于紧密的耦合。** "耦合"指的是两个类之间连接的紧密程度。一般来说，连接越松散越好。从这个概念引出几条通用指导原则：

- 最小化类和成员的可达性。
- 避免friend类，因为它们紧密耦合。
- 在基类中将数据设为private而非protected，使派生类与基类的耦合更松。
- 避免在类的公共接口中暴露成员数据。
- 警惕语义上的封装违反。
- 遵守"得墨忒耳法则"（见本章的"设计和实现问题"）。

耦合与抽象和封装齐头并进。当抽象泄漏或封装被破坏时，就会发生紧密耦合。如果一个类提供不完整的服务集，其他例程可能发现它们需要直接读取或写入其内部数据。这打开了类，使其成为玻璃盒而不是黑盒，几乎消除了类的封装。

## 设计和实现问题

定义良好的类接口对创建高质量程序有很大帮助。内部类设计和实现也很重要。本节讨论与包含、继承、成员函数和数据、类耦合、构造函数以及值对象与引用对象相关的问题。

---

### 包含（"has a"关系）

包含是一个简单的概念，即一个类包含一个原始数据元素或对象。关于继承的著作比包含多得多，但这是因为继承更棘手且更容易出错，不是因为它更好。包含是面向对象编程的主力技术。

**通过包含实现"has a"。** 思考包含的一种方式是"has a"关系。例如，一个员工"有一个"姓名、"有一个"电话号码、"有一个"税号等。你通常可以通过使姓名、电话号码和税号成为Employee类的成员数据来实现这一点。

**将私有继承作为最后的手段实现"has a"。** 在某些情况下，你可能发现无法通过使一个对象成为另一个对象的成员来实现包含。在这种情况下，一些专家建议私有继承自包含的对象（Meyers 1998，Sutter 2000）。你这样做的主要原因是设置包含类访问被包含类的protected成员函数或protected成员数据。在实践中，这种方法与祖先类创建了过于亲密的关系，违反封装。它往往指向应该通过私有继承以外的其他方式解决的设计错误。

**对包含超过约七个数据成员的类持批判态度。** 数字"7±2"被发现是一个人执行其他任务时可以记住的离散项目数量（Miller 1956）。如果一个类包含超过约七个数据成员，考虑该类是否应该分解为多个更小的类（Riel 1996）。如果数据成员是整数和字符串等原始数据类型，你可能更倾向于7±2的高端，如果数据成员是复杂对象，则更倾向于7±2的低端。

---

### 继承（"is a"关系）

继承是一个类是另一个类的专门化的概念。继承的目的是通过定义指定两个或多个派生类的公共元素的基类来创建更简单的代码。公共元素可以是例程接口、实现、数据成员或数据类型。继承有助于避免在多个位置重复代码和数据，方法是将其集中在一个基类中。

当你决定使用继承时，你必须做出几个决定：

- 对于每个成员例程，该例程对派生类可见吗？
- 它会有默认实现吗？
- 默认实现可被覆盖吗？
- 对于每个数据成员（包括变量、命名常量、枚举等），数据成员对派生类可见吗？

以下小节解释做出这些决定的细节：

**通过公共继承实现"is a"。** 当程序员决定通过继承现有类来创建新类时，该程序员是在说新类是其前身类的更专门化版本。基类设定期望派生类将如何运作，并施加派生类可以如何运作的约束（Meyers 1998）。

> C++面向对象编程的最重要规则是：公共继承意味着"is a"。将这条规则铭记在心。
> —— Scott Meyers

如果派生类不打算完全遵守基类定义的接口契约，继承就不是正确的实现技术。考虑包含或在继承层次结构中进一步向上做出更改。

**为继承设计或禁止继承。** 继承增加了程序的复杂性，因此它是一种危险的技术。正如Java专家Joshua Bloch所说，"为继承设计，否则就禁止。"如果一个类不是被设计为可继承的，使其成员在C++中成为非虚的，在Java中成为final，在Microsoft Visual Basic中成为non-overridable，这样你就不能继承它。

**遵守里氏替换原则（LSP）。** 在面向对象编程的开创性论文之一中，Barbara Liskov认为你不应该继承一个基类，除非派生类真正"是一个"基类的更具体版本（Liskov 1988）。Andy Hunt和Dave Thomas这样总结LSP："子类必须可以通过基类接口使用，而用户无需知道差异"（Hunt and Thomas 2000）。

换句话说，基类中定义的所有例程在每个派生类中使用时应该意味着相同的东西。

如果你有一个Account基类和CheckingAccount、SavingsAccount和AutoLoanAccount派生类，程序员应该能够对Account的任何子类型调用从Account派生的任何例程，而无需关心特定账户对象是哪种子类型。

如果一个程序被编写成使里氏替换原则成立，继承是降低复杂性的强大工具，因为程序员可以专注于对象的通用属性而不必担心细节。如果程序员必须不断思考子类实现中的语义差异，那么继承是在增加而不是减少复杂性。假设程序员必须这样想："如果我对CheckingAccount或SavingsAccount调用InterestRate()例程，它返回银行支付的利息，但如果我对AutoLoanAccount调用InterestRate()，我必须改变符号，因为它返回消费者支付给银行的利息。"根据LSP，在这个例子中AutoLoanAccount不应该从Account基类继承，因为InterestRate()例程的语义与基类InterestRate()例程的语义不相同。

**确保只继承你想要的。** 派生类可以继承成员例程接口、实现或两者。表6-1显示了例程如何实现和覆盖的变体。

**表6-1。继承例程的变体**

|  | **可覆盖** | **不可覆盖** |
|--|-----------|-------------|
| **提供实现：默认提供** | 可覆盖例程 | 不可覆盖例程 |
| **提供实现：无默认提供** | 抽象可覆盖例程 | 未使用（留下一个未定义的例程且不允许被覆盖没有意义） |

如表所示，继承的例程有三种基本风格：

- **抽象可覆盖例程** 意味着派生类继承例程的接口但不继承其实现。
- **可覆盖例程** 意味着派生类继承例程的接口和默认实现，并被允许覆盖默认实现。
- **不可覆盖例程** 意味着派生类继承例程的接口和其默认实现，并且不允许覆盖例程的实现。

当你选择通过继承实现一个新类时，要仔细考虑每个成员例程你想要什么样的继承。警惕仅仅因为继承接口就继承实现，也警惕仅仅因为想继承实现就继承接口。如果你想使用一个类的实现但不想使用其接口，使用包含而不是继承。

**不要"覆盖"不可覆盖的成员函数。** C++和Java都允许程序员覆盖不可覆盖的成员例程——某种程度上。如果一个函数在基类中是私有的，派生类可以创建同名函数。对于阅读派生类代码的程序员来说，这样的函数会造成混淆，因为它看起来应该是多态的，但实际上不是；它只是同名。表达这一指导原则的另一种方式是，"不要在派生类中重用不可覆盖的基类例程的名称。"

**尽可能将公共接口、数据和行为移动到继承树的更高位置。** 你将接口、数据和行为移动得越高，派生类就越容易使用它们。多高算太高？让抽象成为你的指南。如果你发现将例程移动更高会破坏更高对象的抽象，就不要这样做。

**警惕只有一个实例的类。** 单一实例可能表明设计混淆了对象和类。考虑你是否可以直接创建一个对象而不是一个新类。派生类的变体可以用数据表示而不是作为不同的类吗？单例模式是此指导原则的一个值得注意的例外。

**警惕只有一个派生类的基类。** 当我看到一个只有一个派生类的基类时，我怀疑一些程序员在"预先设计"——试图预见未来的需求，通常不完全理解那些未来需求是什么。准备未来工作的最好方式不是设计可能"某天需要"的额外基类层；而是使当前工作尽可能清晰、直接和简单。这意味着不要创建任何比绝对必要更多的继承结构。

**警惕覆盖例程但派生例程内部什么都不做的类。** 这通常表明基类的设计存在错误。例如，假设你有一个Cat类和一个Scratch()例程，并假设你最终发现有些猫被去爪了，不能抓挠。你可能很想创建一个从Cat派生的名为ScratchlessCat的类，并覆盖Scratch()例程使其什么都不做。这种方法有几个问题：

- 它通过更改其接口的语义违反了Cat类呈现的抽象（接口契约）。
- 当你将其扩展到其他派生类时，这种方法很快就会失控。如果你发现一只没有尾巴的猫怎么办？或者一只不抓老鼠的猫？或者一只不喝牛奶的猫？最终你会得到像ScratchlessTaillessMicelessMilklessCat这样的派生类。
- 随着时间的推移，这种方法会产生维护人员难以理解的代码，因为祖先类的接口和行为对其后代的行为几乎没有任何暗示。

修复这个问题的位置不在基类，而在原始Cat类。创建一个Claws类并将其包含在Cats类中。根本问题是假设所有猫都会抓挠，所以在源头修复这个问题，而不是只在目的地包扎。

**避免深层继承树。** 面向对象编程提供了大量管理复杂性的技术。但每个强大的工具都有其危害，一些面向对象技术倾向于增加而不是减少复杂性。

在他出色的书《面向对象设计启发法》（1996）中，Arthur Riel建议将继承层次限制在最多六层。Riel基于"魔法数字7±2"提出他的建议，但我认为这过于乐观。根据我的经验，大多数人一次在大脑中处理超过两三层继承会有困难。"魔法数字7±2"可能更好地用作基类的子类总数限制，而不是继承树的层次数限制。

深层继承树被发现与更高的错误率显著相关（Basili, Briand, and Melo 1996）。任何曾经尝试调试复杂继承层次结构的人都知道原因。深层继承树增加复杂性，这恰恰与继承应该完成的目标相反。记住主要的技术使命。确保你使用继承来避免重复代码并最小化复杂性。

**倾向于多态而非广泛类型检查。** 经常重复的case语句有时表明继承可能是更好的设计选择，尽管这并非总是如此。这里有一个经典示例，代码呼唤更面向对象的方法：

**示例6-9。** C++ 可能应该被多态替换的case语句示例

```cpp
switch ( shape.type ) {
   case Shape_Circle:
      shape.DrawCircle();
      break;
   case Shape_Square:
      shape.DrawSquare();
      break;
   ...
}
```

在这个例子中，对shape.DrawCircle()和shape.DrawSquare()的调用应该被一个名为shape.Draw()的单一例程替换，无论形状是圆形还是方形都可以调用。

另一方面，有时case语句用于分离真正不同种类的对象或行为。这里是一个面向对象程序中合适的case语句示例：

**示例6-10。** C++ 可能不应该被多态替换的case语句示例

```cpp
switch ( ui.Command() ) {
   case Command_OpenFile:
      OpenFile();
      break;
   case Command_Print:
      Print();
      break;
   case Command_Save:
      Save();
      break;
   case Command_Exit:
      ShutDown();
      break;
   ...
}
```

在这种情况下，创建具有派生类和一个用于每个命令的多态DoCommand()例程的基类是可能的（正如命令模式中所做的那样）。但在像这样简单的案例中，DoCommand()的含义将被稀释到毫无意义的程度，而case语句是更易理解的解决方案。

**将所有数据设为private，而不是protected。** 正如Joshua Bloch所说，"继承破坏封装"（2001）。当你继承一个对象时，你获得对该对象的protected例程和数据的特权访问。如果派生类真的需要访问基类的属性，提供protected访问函数代替。

### 多重继承

继承是一个强力工具。就像用链锯而不是手动横切锯砍树一样。它如果小心使用会非常有用，但在不了解适当预防措施的人手中是危险的。

> 关于C++中多重继承的一个无可争辩的事实是，它打开了一个在单继承下根本不存在的复杂性 Pandora盒子。
> —— Scott Meyers

如果继承是一把链锯，多重继承就是一把20世纪50年代风格的链锯，没有刀片护罩、没有自动关闭装置、引擎也不稳定。有时候这样的工具有价值；但大多数情况下，你最好把工具放在车库里，那里它不会造成损害。

尽管一些专家推荐广泛使用多重继承（Meyer 1997），但根据我的经验，多重继承主要用于定义"混入"——用于向对象添加一组属性的简单类。混入之所以被称为混入，是因为它们允许属性被"混入"派生类。混入可能是像Displayable、Persistent、Serializable或Sortable这样的类。混入几乎总是抽象的，不打算与其他对象独立实例化。

混入需要使用多重继承，但只要所有混入真正相互独立，它们就不会受到与多重继承相关的经典菱形继承问题的影响。它们还通过将属性"分块"在一起来使设计更容易理解。程序员将更容易理解一个对象使用Displayable和Persistent混入，而不是理解实现这两个属性所需的11个更具体的例程。

Java和Visual Basic通过允许多重继承接口但只允许单类继承来认可混入的价值。C++支持接口和实现的多重继承。程序员应该只在仔细考虑替代方案并权衡对系统复杂性和可理解性的影响后才使用多重继承。

### 为什么继承有这么多规则？

本节提出了许多避免继承麻烦的规则。所有这些规则的潜在信息是，继承倾向于与你作为程序员的主要技术使命背道而驰，即管理复杂性。为了控制复杂性，你应该对继承保持强烈的偏见。以下是何时使用继承和何时使用包含的总结：

- 如果多个类共享公共数据但不是行为，创建这些类可以包含的公共对象。
- 如果多个类共享公共行为但不是数据，从定义公共例程的公共基类派生它们。
- 如果多个类共享公共数据和行为，从定义公共数据和例程的公共基类继承。
- 当你想让基类控制你的接口时继承；当你想控制你的接口时使用包含。

### 成员函数和数据

以下是有效实现成员函数和成员数据的一些指导原则。

> **交叉参考**
>
> 有关例程的更多讨论，请参见第7章。

**保持类中例程的数量尽可能少。** 对C++程序的研究发现，每个类中更高数量的例程与更高的错误率相关（Basili, Briand, and Melo 1996）。然而，发现其他竞争因素更为重要，包括深层继承树、在类中调用的大量例程以及类之间的强耦合。在最小化例程数量和这些其他因素之间权衡。

**禁止你不需要的隐式生成的成员函数和运算符。** 有时你会发现自己想要禁止某些函数——也许你想要禁止赋值，或者你不希望允许构造对象。你可能认为既然编译器自动生成运算符，你就无法禁止访问。但在这种情况下，你可以通过将构造函数、赋值运算符或其他函数或运算符声明为private来禁止这些用途，从而阻止客户端访问。（将构造函数设为private是定义单例类的标准技术，本章稍后讨论。）

**最小化类调用的不同例程数量。** 一项研究发现，类中的错误数量与从类内部调用的例程总数统计相关（Basili, Briand, and Melo 1996）。同一研究发现，一个类使用的类越多，其错误率往往越高。这些概念有时被称为"扇出"。

**最小化对其他类的间接例程调用。** 直接连接已经足够危险。间接连接——如`account.ContactPerson().DaytimeContactInfo().PhoneNumber()`——往往更危险。研究人员制定了一条叫做"得墨忒耳法则"的规则（Lieberherr and Holland 1989），它本质上说明对象A可以调用它自己的任何例程。如果对象A实例化对象B，它可以调用对象B的任何例程。但它应该避免调用由对象B提供的对象上的例程。在account示例中，这意味着`account.ContactPerson()`可以，但`account.ContactPerson().DaytimeContactInfo()`不行。

> **进一步阅读**
>
> 可以在《实用程序员》（Hunt and Thomas 2000）、《应用UML和模式》（Larman 2001）以及《UML面向对象设计基础》（Page-Jones 2000）中找到得墨忒耳法则的良好说明。
>
> 这是简化解释。参见本章末尾的额外资源获取更多细节。

一般来说，最小化类与其他类协作的程度。尝试最小化以下所有内容：

- 实例化的对象种类数量
- 对实例化对象的不同直接例程调用数量
- 对其他实例化对象返回的对象的例程调用数量

### 构造函数

以下是专门适用于构造函数的一些指导原则。构造函数的指导原则在各种语言中相当相似（C++、Java和Visual Basic都可以）。析构函数差异更大，所以你应该查看本章"额外资源"部分列出的资料获取析构函数信息。

**如果可能，在所有构造函数中初始化所有成员数据。** 在所有构造函数中初始化所有数据成员是一种廉价的防御性编程实践。

**使用私有构造函数强制单例属性。** 如果你想定义一个只允许实例化一个对象的类，你可以通过隐藏类的所有构造函数然后提供一个静态GetInstance()例程来访问类的单一实例来强制执行此操作。以下是它如何工作的示例：

> **进一步阅读**
>
> 在C++中执行此操作的代码类似。详情请参见《More Effective C++》第26条（Meyers 1998）。

**示例6-11。** Java 使用私有构造函数强制单例示例

```java
public class MaxId {
   // 构造函数和析构函数
   private MaxId() {       // 1
      ...
   }
   ...

   // 公共例程
   public static MaxId GetInstance() {       // 2
      return m_instance;
   }
   ...

   // 私有成员
   private static final MaxId m_instance = new MaxId();       // 3
   ...
}
```

(1) 这里是私有构造函数。

(2) 这里是提供对单一实例访问的公共例程。

(3) 这里是单一实例。

私有构造函数只在静态对象m_instance初始化时被调用。在这种方法中，如果你想引用MaxId单例，你只需引用MaxId.GetInstance()。

**除非证明必要，否则优先深拷贝而非浅拷贝。** 关于复杂对象你要做出的主要决定之一是实现对象的深拷贝还是浅拷贝。对象的深拷贝是对象成员数据的成员方式拷贝；浅拷贝通常只指向或引用单个引用副本，尽管"深"和"浅"的具体含义有所不同。

创建浅拷贝的动机通常是为了提高性能。虽然创建多个大对象的副本可能在美学上令人反感，但它很少引起可测量的性能影响。少量对象可能引起性能问题，但程序员在猜测真正导致问题的代码方面声名狼藉。（详情请参见第25章。）因为为可疑的性能提升增加复杂性是一个糟糕的权衡，所以处理深拷贝与浅拷贝的好方法是除非证明必要，否则优先深拷贝。

深拷贝比浅拷贝更简单、更易于维护。除了任何一种对象将包含的代码外，浅拷贝还增加了代码来计数引用、确保安全的对象拷贝、安全比较、安全删除等。这段代码容易出错，除非有令人信服的理由创建它，否则应该避免。

如果你发现确实需要使用浅拷贝方法，Scott Meyers的《More Effective C++》第29条（1996）包含了对C++中问题的出色讨论。Martin Fowler的《重构》（1999）描述了从浅拷贝转换为深拷贝以及从深拷贝转换为浅拷贝所需的具体步骤。（Fowler称它们为引用对象和值对象。）

## 创建类的理由

如果你相信你读到的所有内容，你可能会认为创建类的唯一原因是模拟真实世界的对象。实际上，创建类的原因比这多得多。以下是创建类的好理由列表。

> **交叉参考**
>
> 创建类和例程的理由有重叠。参见"创建例程的有效理由"。

**模拟真实世界的对象。** 模拟真实世界的对象可能不是创建类的唯一理由，但它仍然是一个很好的理由！为程序建模的每个真实世界对象类型创建一个类。将对象所需的数据放入类中，然后构建模拟对象行为的服务的例程。参见"类基础：抽象数据类型（ADT）"中ADT的讨论获取示例。

> **交叉参考**
>
> 有关识别真实世界对象的更多信息，请参见"设计构建模块：启发法"中的"找到真实世界的对象"。

**模拟抽象对象。** 创建类的另一个好理由是模拟抽象对象——一个不是具体的、真实世界的对象但提供其他具体对象抽象的对象。一个很好的例子是经典的Shape对象。Circle和Square确实存在，但Shape是其他特定形状的抽象。

在编程项目中，抽象不像Shape那样现成可用，所以我们必须更努力地想出干净的抽象。从真实世界实体中提取抽象概念的过程是不确定的，不同的设计师会抽象出不同的共性。例如，如果我们不知道圆形、方形和三角形等几何形状，我们可能会提出像压扁形状、芜菁甘蓝形状和庞蒂亚克Aztec形状这样更不寻常的形状。提出适当的抽象对象是面向对象设计的主要挑战之一。

**降低复杂性。** 创建类最重要的理由是降低程序的复杂性。创建一个类来隐藏信息，这样你就不必考虑它。当然，你在编写类时需要考虑它。但编写之后，你应该能够忘记细节，在不了解其内部工作原理的情况下使用类。创建类的其他理由——最小化代码大小、提高可维护性和提高正确性——也是好理由，但没有类的抽象能力，复杂程序将不可能在智力上管理。

**隔离复杂性。** 各种形式的复杂性——复杂算法、大数据集、复杂通信协议等——容易出错。如果确实发生错误，如果错误没有散布在代码中而是局部化在类内，就更容易找到。修复错误产生的更改不会影响其他代码，因为只需要修复一个类——其他代码不会被触及。如果你发现了一个更好、更简单或更可靠的算法，如果它已被隔离到一个类中，替换旧算法会更容易。在开发期间，尝试几种设计并保留效果最好的一个会更容易。

**隐藏实现细节。** 隐藏实现细节的愿望是创建类的一个极好的理由，无论细节像复杂的数据库访问一样复杂，还是像特定数据成员是存储为数字还是字符串一样平凡。

**限制更改的影响。** 隔离可能更改的区域，使更改的影响限于单个类或少数几个类的范围内。设计使最可能更改的区域最容易更改。可能更改的区域包括硬件依赖、输入/输出、复杂数据类型和业务规则。"设计构建模块：启发法"中名为"隐藏秘密（信息隐藏）"的小节描述了几种常见的更改来源。

**隐藏全局数据。** 如果你需要使用全局数据，你可以将其实现细节隐藏在类接口后面。通过访问例程处理全局数据与直接处理全局数据相比有几个好处。你可以更改数据结构而不更改程序。你可以监视对数据的访问。使用访问例程的纪律也鼓励你考虑数据是否真的是全局的；通常很明显"全局数据"实际上只是对象数据。

> **交叉参考**
>
> 有关使用全局数据相关问题的讨论，请参见"全局数据"。

**简化参数传递。** 如果你在多个例程之间传递一个参数，这可能表明需要将这些例程及其共享的参数分解为一个将参数作为对象数据的类。简化参数传递本身不是目标，但到处传递大量数据表明不同的类组织可能效果更好。

**建立中央控制点。** 在一个地方控制每个任务是明智的。控制有多种形式。知道表中条目数量是一种形式。设备控制——文件、数据库连接、打印机等——是另一种。使用一个类来读写数据库是集中控制的一种形式。如果数据库需要转换为平面文件或内存数据，更改将只影响一个类。

> **交叉参考**
>
> 有关信息隐藏的详细信息，请参见"设计构建模块：启发法"中的"隐藏秘密（信息隐藏）"。

集中控制的思想类似于信息隐藏，但它具有独特的启发式力量，值得添加到你的编程工具箱中。

**促进代码重用。** 放入良好分解类的代码比嵌入一个更大类中的相同代码更容易在其他程序中重用。即使一段代码只从程序中的一个地方调用，并且作为更大类的一部分可以理解，如果该代码段可能在另一个程序中使用，将其放入自己的类中是有意义的。

美国宇航局软件工程实验室研究了十个积极追求重用的项目（McGarry, Waligora, and McDermott 1989）。在面向对象和面向功能的方法中，初始项目无法从前面的项目获取太多代码，因为前面的项目没有建立足够的代码库。随后，使用功能设计的项目能够从前面的项目获取约35%的代码。使用面向对象方法的项目能够从前面的项目获取超过70%的代码。如果你能够通过提前计划避免编写70%的代码，就去做！

值得注意的是，NASA创建可重用类的核心方法不涉及"为重用设计"。NASA在项目结束时识别重用候选者。然后，他们将制作类可重用所需的工作作为主项目结束时的特殊项目或新项目的第一步来执行。这种方法有助于防止"镀金"——创建不需要的功能和不必要地增加复杂性。

> **交叉参考**
>
> 有关实现所需的最少功能量的更多信息，请参见"重构简介"中的"一个程序包含看起来某天可能需要的代码"。

**为程序族做计划。** 如果你期望程序被修改，隔离你期望会更改的部分并将它们放入自己的类中是明智的。然后你可以修改类而不影响程序的其余部分，或者你可以完全放入新的类。思考不仅一个程序会是什么样子，而是整个程序族可能是什么样子，是预见整个类别更改的强大启发式方法（Parnas 1976）。

几年前，我管理一个为客户销售保险编写程序系列的团队。我们必须为每个客户的保险费率、报价报告格式等进行定制。但程序的许多部分是相似的：输入潜在客户信息的类、在客户数据库中存储信息的类、查找费率的类、为一组计算总费率的类等。团队分解了程序，以便每个因客户而异的部分都在自己的类中。初始编程可能需要大约三个月，但当我们得到一个新客户时，我们只需为新客户编写几个新类并将它们放入其余代码中。几天的工作——瞧！——定制软件！

**打包相关操作。** 在你无法隐藏信息、共享数据或为灵活性做计划的情况下，你仍然可以将操作集合打包成合理的组，如三角函数、统计函数、字符串操作例程、位操作例程、图形例程等。类是组合相关操作的一种手段。根据你使用的语言，你也可以使用包、命名空间或头文件。

**完成特定重构。** 第24章中描述的许多特定重构会导致新类——包括将一个类转换为两个、隐藏委托、移除中间人以及引入扩展类。这些新类可能出于在本节整个描述中更好实现任何目标的愿望而推动。

### 应避免的类

虽然类通常是好的，但你会遇到一些陷阱。以下是一些应避免的类。

**避免创建上帝类。** 避免创建无所不知、无所不能的全知类。如果一个类花时间通过Get()和Set()例程从其他类检索数据（即挖掘它们的业务并告诉它们该怎么做），问问自己该功能是否最好组织到那些其他类而不是上帝类中（Riel 1996）。

**消除无关的类。** 如果一个类只包含数据而没有行为，问问你自己它真的是一个类吗，考虑将其降级以便其成员数据只成为一个或多个其他类的属性。

> **交叉参考**
>
> 这种类通常称为结构。有关结构的更多信息，请参见"结构"。

**避免以动词命名的类。** 只有行为而没有数据的类通常真的不是类。考虑将像DatabaseInitialization()或StringBuilder()这样的类转换为其他类的例程。

## 语言特定问题

不同编程语言中的类方法以有趣的方式变化。考虑一下如何在派生类中覆盖成员例程以实现多态。在Java中，默认情况下所有例程都是可覆盖的，例程必须声明为final才能防止派生类覆盖它。在C++中，例程默认不可覆盖。例程必须在基类中声明为virtual才能被覆盖。在Visual Basic中，例程必须在基类中声明为overridable，派生类应使用overrides关键字。

以下是一些因语言不同而显著不同的类相关领域：

- 继承树中构造函数和析构函数的覆盖行为
- 异常处理条件下构造函数和析构函数的行为
- 默认构造函数（无参数构造函数）的重要性
- 调用析构函数或终结器的时间
- 覆盖语言内置运算符（包括赋值和相等）的智慧
- 创建和销毁对象时或在声明并超出范围时如何处理内存

这些问题的详细讨论超出了本书的范围，但"额外资源"部分指向了好的语言特定资源。

## 类之外：包

类是程序员实现模块化的当前最佳方式。但模块化是一个大主题，它超出了类。过去几十年，软件发展在很大程度上是通过增加我们必须处理的聚合粒度来推进的。我们拥有的第一个聚合是语句，它在当时看起来是从机器指令迈出的一大步。然后是子程序，后来是类。

> **交叉参考**
>
> 有关类和包之间区别的更多信息，请参见"关键设计概念"中的"设计层次"。

很明显，如果我们在聚合对象组方面有好工具，我们可以更好地支持抽象和封装的目标。Ada在十多年前就支持包的概念，Java今天支持包。如果你正在使用不支持包直接的语言，你可以创建你自己的程序员贫民版包，并通过包括以下内容的编程标准来强制执行：

- 命名约定，区分哪些类是公共的，哪些用于包的私有使用
- 命名约定、代码组织约定（项目结构）或两者，标识每个类属于哪个包
- 规则定义哪些包允许使用哪些其他包，包括使用是否可以继承、包含或两者

这些变通方法是"在语言中编程"与"编程进入语言"之间区别的很好例子。有关这种区别的更多信息，请参见"编程进入你的语言，而非在语言中编程"。

## 类质量检查清单

### 抽象数据类型

- 你是否将程序中的类视为抽象数据类型并从该角度评估它们的接口？

### 抽象

- 类是否有中心目的？
- 类是否被很好地命名，并描述其中心目的？
- 类的接口是否呈现一致的抽象？
- 类的接口是否明显说明你应该如何使用类？
- 类的接口是否足够抽象，你不必考虑其服务如何实现？你能将类视为黑盒吗？
- 类的服务是否足够完整，其他类不必干涉其内部数据？
- 不相关的信息是否已移出类？
- 你是否考虑过将类细分为组件类，并尽可能地细分？
- 在修改类时是否保持类接口的完整性？

### 封装

- 类是否最小化其成员的可达性？
- 类是否避免暴露成员数据？
- 类是否在编程语言允许的范围内向其他类隐藏其实现细节？
- 类是否避免对其用户做出假设，包括其派生类？
- 类是否独立于其他类？是否松耦合？

### 继承

- 继承是否只用于建模"is a"关系——即派生类是否遵守里氏替换原则？
- 类文档是否描述了继承策略？
- 派生类是否避免"覆盖"不可覆盖的例程？
- 公共接口、数据和行为是否在继承树中尽可能高？
- 继承树是否相当浅？
- 基类中的所有数据成员是否都是private而不是protected？

### 其他实现问题

- 类是否包含约七个或更少的数据成员？
- 类是否最小化对其他类的直接和间接例程调用？
- 类是否仅在绝对必要时与其他类协作？
- 所有成员数据是否在构造函数中初始化？
- 类是否被设计为使用深拷贝而不是浅拷贝，除非有测量过的理由创建浅拷贝？

### 语言特定问题

- 你是否调查了特定编程语言中类的语言特定问题？

## 关键点

- 类接口应该呈现一致的抽象。许多问题源于违反这一原则。
- 类接口应该隐藏一些东西——系统接口、设计决策或实现细节。
- 包含通常优于继承，除非你正在建模"is a"关系。
- 继承是一个有用的工具，但它增加了复杂性，这与"软件的主要技术使命：管理复杂性"背道而驰。
- 类是你管理复杂性的主要工具。给予它们的设计尽可能多的关注以完成该目标。
