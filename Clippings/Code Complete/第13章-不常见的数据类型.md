## 结构体

术语"结构体"指的是由其他类型构建的数据。因为数组是一种特殊情况，它们在第12章中单独处理。本节处理用户创建的结构化数据——C 和 C++ 中的 struct 以及 Microsoft Visual Basic 中的 Structure。在 Java 和 C++ 中，类有时也充当结构体的角色（当类完全由公共数据成员组成且没有公共例程时）。

一般来说，你会想创建类而不是结构体，这样你就可以利用类提供的隐私和功能，而不仅仅是结构体支持的公共数据。但有时直接操作数据块可能很有用，因此这里有一些使用结构体的原因：

**使用结构体来澄清数据关系**。结构体将相关项目组合在一起。有时理解程序最难的部分是弄清楚哪些数据与哪些其他数据相关。这就像去一个小镇问谁与谁有亲戚关系。你会发现每个人在某种程度上都与其他人有关，但又不是真正的亲戚关系，你永远得不到一个好的答案。

如果数据被仔细地结构化了，弄清楚什么与什么相关就容易多了。这里有一个未结构化数据的例子：

示例 13-1. 具有误导性的非结构化变量的 Visual Basic 示例

```vb
name = inputName
address = inputAddress
phone = inputPhone
title = inputTitle
department = inputDepartment
bonus = inputBonus
```

因为这些数据是非结构化的，看起来所有赋值语句都属同一组。实际上，`name`、`address` 和 `phone` 是与个人员工关联的变量，而 `title`、`department` 和 `bonus` 是与主管关联的变量。代码片段没有暗示有两种数据在工作。在下面的代码片段中，结构体的使用使关系更清晰：

示例 13-2. 信息更丰富、结构化的变量的 Visual Basic 示例

```vb
employee.name = inputName
employee.address = inputAddress
employee.phone = inputPhone

supervisor.title = inputTitle
supervisor.department = inputDepartment
supervisor.bonus = inputBonus
```

在使用结构化变量的代码中，很明显有些数据与员工相关，有些数据与主管相关。

**使用结构体来简化对数据块的操作**。你可以将相关元素组合成一个结构体，然后对结构体执行操作。对结构体进行操作比对其中的每个元素执行相同的操作更容易。它也更可靠，而且需要更少的代码行。

假设你有一组属于一起的数据项——例如，人事数据库中关于员工的数据。如果数据没有组合成结构体，仅仅是复制这组数据就可能涉及很多语句。以下是 Visual Basic 中的一个例子：

示例 13-3. 笨拙地复制一组数据项的 Visual Basic 示例

```vb
newName = oldName
newAddress = oldAddress
newPhone = oldPhone
newSsn = oldSsn
newGender = oldGender
newSalary = oldSalary
```

每次你想传输关于员工的信息时，你都必须有这整组语句。如果你添加一条新的员工信息——例如，`numWithholdings`——你必须在你有赋值块的每个地方添加 `newNumWithholdings = oldNumWithholdings`。

想象在两个员工之间交换数据会有多可怕。你不必想象——这里就是了：

**困难方式交换两组数据的 Visual Basic 示例**

示例 13-4.

```vb
' swap new and old employee data
previousOldName = oldName
previousOldAddress = oldAddress
previousOldPhone = oldPhone
previousOldSsn = oldSsn
previousOldGender = oldGender
previousOldSalary = oldSalary

oldName = newName
oldAddress = newAddress
oldPhone = newPhone
oldSsn = newSsn
oldGender = newGender
oldSalary = newSalary

newName = previousOldName
newAddress = previousOldAddress
newPhone = previousOldPhone
newSsn = previousOldSsn
newGender = previousOldGender
newSalary = previousOldSalary
```

处理问题的更简单方法是声明一个结构化变量：

示例 13-5. Visual Basic 中声明结构体的示例

```vb
Structure Employee
   name As String
   address As String
   phone As String
   ssn As String
   gender As String
   salary As Long
End Structure
Dim newEmployee As Employee
Dim oldEmployee As Employee
Dim previousOldEmployee As Employee
```

现在你可以用三条语句交换旧和新员工结构中的所有元素：

示例 13-6. 交换两组数据的更简单方法的 Visual Basic 示例

```vb
previousOldEmployee = oldEmployee
oldEmployee = newEmployee
newEmployee = previousOldEmployee
```

如果你想添加一个字段（如 `numWithholdings`），你只需将其添加到 Structure 声明中。上面的三条语句以及整个程序中类似的语句都不需要修改。C++ 和其他语言有类似的能力。

**使用结构体来简化参数列表**。你可以通过使用结构化变量来简化例程参数列表。这种技术与刚才展示的技术类似。与其分别传递每个需要的元素，不如将相关元素分组到一个结构体中，然后将整个包作为一组传递。以下是传递一组相关参数的困难方式的例子：

示例 13-7. 没有结构体的笨拙例程调用的 Visual Basic 示例

```vb
HardWayRoutine( name, address, phone, ssn, gender, salary )
```

以下是通过使用包含第一个参数列表元素的结构化变量来调用例程的简单方式的例子：

示例 13-8. 使用结构体的优雅例程调用的 Visual Basic 示例

```vb
EasyWayRoutine( employee )
```

如果你想在第一种调用中添加 `numWithholdings`，你必须深入你的代码并更改对 `HardWayRoutine()` 的每次调用。如果你向 `Employee` 添加一个 `numWithholdings` 元素，你根本不需要更改 `EasyWayRoutine()` 的参数。

你可以将这种技术发挥到极致，将程序中的所有变量放入一个大而多汁的变量中，然后到处传递。谨慎的程序员避免比逻辑上必要的情况下捆绑更多的数据。此外，谨慎的程序员避免在只需要结构体中的一个或两个字段时将其作为参数传递——他们传递需要的特定字段。这是信息隐藏的一个方面：一些信息隐藏在例程中，一些信息对例程隐藏。信息在需要知道的基础上传递。

**使用结构体来减少维护**。因为你在使用结构体时对相关数据进行了分组，改变结构体在整个程序中需要更少的更改。这在代码中与结构体变化没有逻辑关系的部分尤其如此。因为更改往往会产生错误，更少的更改意味着更少的错误。如果你的 `Employee` 结构体有一个 `title` 字段，而你决定删除它，你不需要更改任何使用整个结构体的参数列表或赋值语句。当然，你必须更改任何专门处理员工标题的代码，但那在概念上与删除标题字段相关，很难被忽略。

结构化数据的最大优势在于与 `title` 字段没有逻辑关系的代码部分。有时程序语句在概念上引用的是一组数据而不是单个组件。在这种情况下，单个组件（如 `title` 字段）被引用仅仅是因为它们是集合的一部分。这种代码部分没有具体处理 `title` 字段的逻辑理由，当更改 `title` 时，这些部分很容易被忽略。如果你使用结构体，忽略这些部分是可以的，因为代码引用的是相关数据的集合，而不是单独引用每个组件。

## 指针

指针使用是现代编程中最容易出错的领域之一程度如此之高，以至于 Java、C# 和 Visual Basic 等现代语言不提供指针数据类型。使用指针本质上是复杂的，正确使用它们需要你对编译器的内存管理方案有极好的理解。许多常见的安全问题，尤其是缓冲区溢出，可以追溯到指针的错误使用（Howard and LeBlanc 2003）。

即使你的语言不要求你使用指针，对指针的良好理解也会有助于你理解编程语言的工作原理。大量使用防御式编程实践会进一步提供帮助。

---

## 理解指针的范式

从概念上讲，每个指针由两部分组成：内存中的一个位置以及如何解释该位置内容的信息。

### 内存中的位置

内存中的位置是一个地址，通常用十六进制表示。32 位处理器上的地址是一个 32 位值，如 `0x0001EA40`。指针本身只包含这个地址。要使用指针指向的数据，你必须去那个地址并解释该位置内存的内容。如果你查看该位置的内存，它只是一组位。它必须被解释才有意义。

### 如何解释内容的知识

如何解释内存位置内容的知识由指针的基类型提供。如果指针指向一个整数，真正意思是编译器将指针给出的内存位置解释为整数。当然，你可以有一个整数指针、一个字符串指针和一个浮点指针都指向同一个内存位置。但只有其中一个指针正确解释了该位置的内容。

在思考指针时，记住内存本身没有任何固有的解释与之关联是很有帮助的。只有通过使用特定类型的指针，特定位置的位才会被解释为有意义的数据。

图 13-1 显示了同一内存位置的几种视图，以几种不同的方式解释。

> **图 13-1**。每种数据类型使用的内存量用双线表示

在图 13-1 的每种情况中，指针都指向包含十六进制值 `0x0A` 的位置。`0A` 之后使用的字节数取决于内存如何解释。内存内容的使用方式也取决于内存如何解释。（这也取决于你使用的处理器，所以如果你尝试在自己的桌面超级计算机上复制这些结果，请记住这一点。）相同的原始内存内容可以被解释为字符串、整数、浮点数或任何其他东西——这完全取决于指向内存的指针的基类型。

## 指针使用的一般技巧

对于许多类型的缺陷，定位错误是处理错误和纠正错误中最简单的部分，而纠正它是最困难的部分。指针错误是不同的。指针错误通常是指针指向不应该指向的地方的结果。当你给一个坏的指针变量赋值时，你会将数据写入你不应该写入的内存区域。这被称为"内存损坏"。有时内存损坏会产生可怕的、剧烈的系统崩溃；有时它会改变程序另一部分的计算结果；有时它会导致程序不可预测地跳过例程；有时它根本什么都不做。在最后一种情况下，指针错误是一个滴答作响的定时炸弹，等待在你向最重要的客户展示前五分钟摧毁你的程序。指针错误的症状往往与指针错误的原因无关。因此，纠正指针错误的大部分工作是定位原因。

成功使用指针需要双管齐下的策略。首先，首先避免安装指针错误。指针错误是如此难以发现，额外的预防措施是合理的。其次，在编码后尽快检测指针错误。指针错误的症状是如此不稳定，额外的措施使症状更可预测是合理的。以下是如何实现这些关键目标：

**将指针操作隔离在例程或类中**。假设你在程序的几个地方使用链表。与其在每个使用的地方手动遍历列表，不如编写访问例程，如 `NextLink()`、`PreviousLink()`、`InsertLink()` 和 `DeleteLink()`。通过最小化访问指针的位置数量，你最大限度地减少了可能蔓延到整个程序并永远找不到的粗心错误。因为代码然后相对独立于数据实现细节，你也提高了在其他程序中重用的机会。为指针分配编写例程是另一种集中控制数据的方法。

**同时声明和定义指针**。在声明附近给变量赋初始值通常是好的编程实践，在处理指针时更是如此。以下是一个不应该做什么的例子：

**指针初始化不佳的 C++ 示例**

示例 13-9.

```cpp
Employee *employeePtr;
// lots of code
...
employeePtr = new Employee;
```

即使这段代码最初正确工作，在修改时也容易出错，因为在声明指针的时刻和初始化它的时刻之间，有人可能会尝试使用 `employeePtr`。以下是更安全的方法：

示例 13-10. 良好指针初始化的 C++ 示例

```cpp
// lots of code
...
Employee *employeePtr = new Employee;
```

**在与分配相同的作用域级别删除指针**。保持指针的分配和释放对称。如果你在单个作用域内使用指针，在同一作用域内调用 `new` 分配和 `delete` 释放指针。如果你在例程内部分配指针，在姐妹例程内释放它。如果你在对象的构造函数内部分配指针，在对象的析构函数内释放它。分配内存然后期望客户端代码手动释放内存的例程创造了容易出错的不一致性。

**使用指针之前检查它们**。在你程序的关键部分使用指针之前，确保它指向的内存位置是合理的。例如，如果你期望内存位置在 `StartData` 和 `EndData` 之间，你应该对指向 `StartData` 之前或 `EndData` 之后的指针持怀疑态度。你必须确定在你的环境中 `StartData` 和 `EndData` 的值是什么。如果你通过访问例程使用指针而不是直接操作它们，你可以设置它自动工作。

**使用指针引用的变量之前检查它**。有时你可以对指针指向的值进行合理性检查。例如，如果你应该指向一个介于 0 和 1000 之间的整数值，你应该对超过 1000 的值持怀疑态度。如果你指向一个 C++ 风格的字符串，你可能会对长度超过 100 的字符串持怀疑态度。如果你通过访问例程处理指针，这也可以自动完成。

**使用狗牌字段检查损坏的内存**。"标签字段"或"狗牌"是你添加到结构体中的字段，仅用于错误检查。当你分配变量时，将其标签字段设置为一个应该保持不变的值。当你使用结构体时——尤其是在删除内存时——检查标签字段的值。如果标签字段没有预期值，数据已被损坏。

当你删除指针时，损坏该字段，这样如果你意外地尝试再次释放同一个指针，你会检测到损坏。例如，假设你需要分配 100 字节：

1. 首先，分配 104 字节，比请求的多 4 字节。
2. 将前 4 个字节设置为狗牌值，然后返回一个指向其后开始的内存的指针。
3. 当要删除指针时，检查标签。
4. 如果标签正常，将其设置为 0 或你和你的程序识别为无效标签值的其他值。你不希望该值在内存被释放后被误认为有效标签。由于同样的原因，将数据设置为 0、`0xCC` 或其他非随机值。
5. 最后，删除指针。

在你分配的内存块开头放置狗牌允许你检查重复释放内存块的尝试，而不需要维护你分配的所有内存块的列表。在内存块末尾放置狗牌允许你检查写入超出应该使用的位置的内存。你可以在块的开头和末尾使用标签来实现两个目标。

你可以将这种方法与前面建议的合理性检查结合使用——检查指针是否在 `StartData` 和 `EndData` 之间。为了确保指针指向合理的位置，而不是检查可能的内存范围，检查指针是否在已分配指针列表中。

你可以只在删除变量之前检查标签字段一次。损坏的标签会告诉你，在该变量生命中的某个时候其内容已被损坏。但是，你越频繁地检查标签字段，你就越接近检测到问题的根源。

**添加显式冗余**。使用标签字段的替代方法是某些字段使用两次。如果冗余字段中的数据不匹配，你就知道内存已被损坏。如果你直接操作指针，这可能导致大量开销。但是，如果你将指针操作隔离在例程中，它只在几个地方添加重复代码。

**使用额外的指针变量来提高清晰度**。无论如何，不要吝啬指针变量。在其他地方已经指出，变量不应该用于多个目的。这对于指针变量尤其正确。弄清楚某人对链表做了什么已经够难了，更不用说弄清楚为什么一个通用的 `link` 变量被反复使用或者 `pointer->next->last->next` 指向什么了。考虑这个代码片段：

示例 13-11. 传统节点插入代码的 C++ 示例

```cpp
void InsertLink(
   Node *currentNode,
   Node *insertNode
   ) {
   // insert "insertNode" after "currentNode"
   insertNode->next = currentNode->next;
   insertNode->previous = currentNode;
   if ( currentNode->next != NULL ) {
      currentNode->next->previous = insertNode;       // 1
   }
   currentNode->next = insertNode;
}
```

(1) 这一行不必要地难以理解。

这是用于在链表中插入节点的传统代码，理解起来不必要地困难。插入新节点涉及三个对象：当前节点、当前节点之后的当前节点以及要在它们之间插入的节点。代码片段只明确承认两个对象：`insertNode` 和 `currentNode`。它迫使你弄清楚并记住 `currentNode->next` 也涉及其中。如果你试图在最初跟随 `currentNode` 的节点旁边画图来说明发生了什么，你会得到类似这样的东西：

更好的图表会识别所有三个对象。它看起来是这样的：

以下是显式引用所涉及的所有三个对象的代码：

示例 13-12. 可读性更好的节点插入代码的 C++ 示例

```cpp
void InsertLink(
   Node *startNode,
   Node *newMiddleNode
   ) {
   // insert "newMiddleNode" between "startNode" and "followingNode"

   Node *followingNode = startNode->next;
   newMiddleNode->next = followingNode;
   newMiddleNode->previous = startNode;
   if ( followingNode != NULL ) {
      followingNode->previous = newMiddleNode;
   }
   startNode->next = newMiddleNode;
}
```

这个代码片段多了一行代码，但是没有第一个片段中的 `currentNode->next->previous`，更容易跟踪。

**简化复杂的指针表达式**。复杂的指针表达式很难阅读。如果你的代码包含像 `p->q->r->s.data` 这样的表达式，想想必须阅读表达式的人。以下是一个特别恶劣的例子：

**难以理解的指针表达式的 C++ 示例**

示例 13-13.

```cpp
for ( rateIndex = 0; rateIndex < numRates; rateIndex++ ) {
   netRate[ rateIndex ] = baseRate[ rateIndex ] * rates->discounts->factors->net;
}
```

像这个例子中的指针表达式这样的复杂表达式使代码必须被琢磨出来而不是被阅读。如果你的代码包含一个复杂的表达式，将其赋值给一个命名良好的变量来阐明操作的意图。以下是改进后的例子：

示例 13-14. 简化复杂指针表达式的 C++ 示例

```cpp
quantityDiscount = rates->discounts->factors->net;
for ( rateIndex = 0; rateIndex < numRates; rateIndex++ ) {
   netRate[ rateIndex ] = baseRate[ rateIndex ] * quantityDiscount;
}
```

通过这种简化，你不仅获得了可读性的提高，而且可能通过简化循环内的指针操作来提高性能。和往常一样，你必须在用真金白银打赌之前测量性能提升。

**以正确的顺序删除链表中的指针**。在使用动态分配的链表时的一个常见问题是先释放列表中的第一个指针，然后无法到达列表中的下一个指针。为了避免这个问题，确保在释放当前指针之前，你有一个指向列表中下一个元素的指针。

**分配内存备用降落伞**。如果你的程序使用动态内存，你需要避免突然耗尽内存的问题，让你的用户和用户的数据迷失在 RAM 空间中。给你的程序一个错误余地的一种方法是预分配一个内存降落伞。确定你的程序需要多少内存来保存工作、清理和优雅退出。在程序开始时分配该数量的内存作为备用降落伞，然后不要动它。当你用完内存时，释放备用降落伞，清理，然后关闭。

**粉碎你的垃圾**。指针错误难以调试，因为指针指向的内存变得无效的点是不确定的。有时内存内容在指针被释放后很长时间看起来都是有效的。其他时候，内存会立即改变。

延伸阅读

关于安全处理 C 中指针方法的优秀讨论，请参见《Writing Solid Code》(Maguire 1993)。

你可以通过在释放内存块之前用垃圾数据覆盖它们来强制使与使用已释放指针相关的错误更加一致。像许多其他操作一样，如果你使用访问例程，你可以自动完成此操作。在 C++ 中，每次你删除一个指针，你可以使用这样的代码：

示例 13-15. 强制已释放内存包含垃圾数据的 C++ 示例

```cpp
memset( pointer, GARBAGE_DATA, MemoryBlockSize( pointer ) );
delete pointer;
```

当然，这种技术要求你维护一个指针大小列表，可以通过 `MemoryBlockSize()` 例程检索，我稍后会讨论。

**删除或释放后将指针设置为 null**。一种常见的指针错误是"悬空指针"，即使用已被 `delete` 或 `free` 的指针。指针错误难以检测的原因之一是错误有时不会产生任何症状。通过在释放后将指针设置为 null，你不会改变你可以读取悬空指针指向的数据的事实。但是你确实确保了向悬空指针写入数据会产生错误。这可能是一个丑陋的、讨厌的、灾难性的错误，但至少你会找到它，而不是其他人找到它。

前一个例子的删除操作之前的代码可以扩充来处理这个问题：

示例 13-16. 删除后将指针设置为 null 的 C++ 示例

```cpp
memset( pointer, GARBAGE_DATA, MemoryBlockSize( pointer ) );
delete pointer;
pointer = NULL;
```

**删除变量之前检查坏指针**。搞垮程序的最佳方式之一是在指针已经被 `delete` 或 `free` 之后再 `delete` 或 `free` 它。不幸的是，很少有语言能检测到这种问题。

将释放的指针设置为 null 也允许你在使用它或再次尝试删除之前检查指针是否设置为 null；如果你不将释放的指针设置为 null，你就没有这个选项。这建议了对指针删除代码的另一个补充：

示例 13-17. 在删除之前断言指针不为 null 的 C++ 示例

```cpp
ASSERT( pointer != NULL, "Attempting to delete null pointer." );
memset( pointer, GARBAGE_DATA, MemoryBlockSize( pointer ) );
delete pointer;
pointer = NULL;
```

**跟踪指针分配**。维护你已分配的指针列表。这允许你在释放指针之前检查它是否在列表中。以下是标准指针删除代码如何修改以包含该功能的示例：

示例 13-18. 检查指针是否已被分配的 C++ 示例

```cpp
ASSERT( pointer != NULL, "Attempting to delete null pointer." );
if ( IsPointerInList( pointer ) ) {
   memset( pointer, GARBAGE_DATA, MemoryBlockSize( pointer ) );
   RemovePointerFromList( pointer );
   delete pointer;
   pointer = NULL;
}
else {
   ASSERT( FALSE, "Attempting to delete unallocated pointer." );
}
```

**编写包装例程来集中你的避免指针问题策略**。从这个例子中，你可以看到每次指针被 `new` 或 `delete` 时，你最终可能会有相当多的额外代码。本节描述的一些技术是相互排斥或冗余的，你不会想在同一代码库中使用多个相互冲突的策略。例如，如果你维护自己的有效指针列表，你不需要创建和检查狗牌值。

你可以通过为常见指针操作创建包装例程来最小化编程开销和减少错误机会。在 C++ 中，你可以使用这两个例程：

- **SAFE_NEW** 此例程调用 `new` 分配指针，将新指针添加到已分配指针列表中，并将新分配的指针返回给调用例程。它也可以在一个地方检查 `new` 的空返回（也就是"内存不足"错误），这简化了程序其他部分的错误处理。
- **SAFE_DELETE** 此例程检查传递给它的指针是否在已分配指针列表中。如果在列表中，它将指针指向的内存设置为垃圾值，从列表中移除指针，调用 C++ 的 `delete` 操作符释放指针，并将指针设置为 null。如果指针不在列表中，`SAFE_DELETE` 显示诊断消息并停止程序。

作为宏实现，`SAFE_DELETE` 例程如下：

示例 13-19. 将指针删除代码包装起来的 C++ 示例

```cpp
#define SAFE_DELETE( pointer ) { \
   ASSERT( pointer != NULL, "Attempting to delete null pointer."); \
   if ( IsPointerInList( pointer ) ) { \
      memset( pointer, GARBAGE_DATA, MemoryBlockSize( pointer ) ); \
      RemovePointerFromList( pointer ); \
      delete pointer; \
      pointer = NULL; \
   } \
   else { \
      ASSERT( FALSE, "Attempting to delete unallocated pointer." ); \
   } \
}
```

在 C++ 中，此例程将删除单个指针，但你还需要实现一个类似的 `SAFE_DELETE_ARRAY` 例程来删除数组。

交叉参考

关于计划移除用于调试的代码的详细信息，请参见调试辅助工具中的"计划移除调试辅助工具"。

通过在这两个例程中集中内存处理，你还可以让 `SAFE_NEW` 和 `SAFE_DELETE` 在调试模式和生产模式下表现不同。例如，当 `SAFE_DELETE` 在开发期间检测到尝试释放空指针时，它可能会停止程序，但在生产期间，它可能只是记录错误并继续处理。

你可以轻松地使这种方案适应 C 中的 `calloc` 和 `free` 以及其他使用指针的语言。

**使用非指针技术**。指针比一般情况更难理解，它们容易出错，而且往往需要特定于机器的、不可移植的代码。如果你能想到一个合理替代使用指针的方案，那就省去一些头疼，使用它吧。

## C++ 指针要点

C++ 引入了与使用指针和引用相关的一些具体问题。以下小节描述了适用于在 C++ 中使用指针的指导原则：

延伸阅读

关于使用指针的更多技巧，请参见《Effective C++, 2d ed.》(Meyers 1998) 和《More Effective C++》(Meyers 1996)。

**理解指针和引用之间的区别**。在 C++ 中，指针（`*`）和引用（`&`）都间接引用一个对象。对于外行人来说，唯一的区别似乎只是在将字段引用为 `object->field` vs. `object.field` 之间的纯粹表面区别。最显著的区别是引用必须始终引用一个对象，而指针可以指向 null，而且引用在初始化后不能更改它所引用的内容。

**使用指针进行"引用传递"参数，使用 const 引用进行"值传递"参数**。C++ 默认通过值而不是引用将参数传递给例程。当你通过值将对象传递给例程时，C++ 创建对象的副本，当对象被传回调用例程时，再次创建副本。对于大对象，这种复制会消耗时间和资源。因此，当将对象传递给例程时，你通常想避免复制对象，这意味着你想通过引用而不是值传递它。

然而，有时你希望具有值传递的语义——也就是说，传递的对象不应该被更改——但实现值传递——也就是说，传递实际对象而不是副本。

在 C++ 中，这个问题的解决方案是你使用指针进行引用传递，而——尽管术语听起来很奇怪——"const 引用"进行值传递！这里有一个例子：

示例 13-20. 通过引用和通过值传递参数的 C++ 示例

```cpp
void SomeRoutine(
   const LARGE_OBJECT &nonmodifiableObject,
   LARGE_OBJECT *modifiableObject
);
```

这种方法提供了额外的好处，在被调用的例程内，对象应该被修改和不应该被修改之间提供语法区分。对于可修改对象，对成员的引用将使用 `object->member` 表示法，而对于不可修改对象，对成员的引用将使用 `object.member` 表示法。

这种方法的局限性在于传播 const 引用的困难。如果你控制你自己的代码库，在尽可能多的地方使用 const 是好的纪律，你应该能够将值传递参数声明为 const 引用。对于你无法控制的库代码或其他代码，你会在使用 const 例程参数时遇到问题。退一步说，仍然对只读参数使用引用但不声明它们为 const。使用这种方法，你不会意识到编译器检查尝试修改不可修改参数的全部好处，但至少你会在 `object->member` 和 `object.member` 之间给自己视觉区分。

**使用 auto_ptr**。如果你还没有养成使用 auto_ptr 的习惯，养成它！通过在 auto_ptr 超出作用域时自动删除内存，auto_ptr 避免了与常规指针相关的许多内存泄漏问题。在 Scott Meyers 的《More Effective C++》中，Item #9 包含了对 auto_ptr 的良好讨论（Meyers 1996）。

**对智能指针要聪明**。智能指针是常规指针或"哑"指针的替代品（Meyers 1996）。它们的行为类似于常规指针，但提供了对资源管理、复制操作、赋值操作、对象构造和对象析构的更多控制。所涉及的问题是特定于 C++ 的。《More Effective C++》Item #28 包含完整讨论。

## C 指针要点

以下是专门适用于 C 语言的指针使用技巧：

**使用显式指针类型而不是默认类型**。C 允许你对任何类型的变量使用 char 或 void 指针。只要指针指向，语言并不真正关心它指向什么。然而，如果你对指针使用显式类型，编译器可以给你关于不匹配指针类型和不适当解引用的警告。如果你不这样做，它就不能。使用特定指针类型只要你能够。

这个规则的推论是当你必须进行类型转换时使用显式类型转换。例如，在这个片段中，很明显正在分配一个 `NODE_PTR` 类型的变量：

示例 13-21. 显式类型转换的 C 示例

```c
NodePtr = (NODE_PTR) calloc( 1, sizeof( NODE ) );
```

**避免类型转换**。避免类型转换与去表演学校或不再总是扮演"坏人"无关。它与避免将一种类型的变量塞入另一种类型的变量空间有关。类型转换关闭了编译器检查类型不匹配的能力，因此在你的防御式编程盔甲上戳了一个洞。如果程序需要很多类型转换，可能有一些需要重新审视的架构差距。如果可能，重新设计；否则，尽可能避免类型转换。

**遵循参数传递的星号规则**。在 C 中，只有在赋值语句中参数前有星号（`*`）时，你才能将参数从例程传回。许多 C 程序员难以确定 C 何时允许将值传回调用例程。只要你在赋值时在参数前有一个星号，值就会传回调用例程，这很容易记住。无论你在声明中堆叠了多少个星号，如果你想传回值，你必须在赋值语句中至少有一个。以下片段中，赋给参数的值不会传回调用例程，因为赋值语句没有使用星号：

示例 13-22. 不会工作的参数传递的 C 示例

```c
void TryToPassBackAValue( int *parameter ) {
   parameter = SOME_VALUE;
}
```

这里，赋给 `parameter` 的值会被传回，因为 `parameter` 前面有星号：

示例 13-23. 会工作的参数传递的 C 示例

```c
void TryToPassBackAValue( int *parameter ) {
   *parameter = SOME_VALUE;
}
```

**使用 sizeof() 来确定内存分配中变量的大小**。使用 sizeof() 比查手册容易，而且 sizeof() 适用于你自己创建的手册中没有的结构。因为它在编译时计算，sizeof() 不会带来性能惩罚。它是可移植的——在不同环境中重新编译会自动改变 sizeof() 计算的值。由于你可以更改你定义的类型，分配会自动调整，它几乎不需要维护。

## 全局数据

全局变量在程序中的任何地方都可以访问。这个术语有时也被松散地用来指比局部变量作用域更广的变量——比如在类内任何地方都可以访问的类变量。但是，在单个类内任何地方可访问并不本身就意味着变量是全局的。

交叉参考

关于全局数据和类数据之间区别的详细信息，请参见"被误认为全局数据的类数据"。

大多数有经验的程序员得出结论，使用全局数据比使用局部数据风险更大。大多数有经验的程序员也得出结论，从几个例程访问数据相当有用。

然而，即使全局变量并不总是产生错误，它们几乎从来不是编程的最佳方式。本节的其余部分充分探讨了所涉及的问题。

## 全局数据的常见问题

如果你不加区分地使用全局变量，或者你觉得不能使用它们是有限制的，你可能还没有完全领会信息隐藏和模块化的价值。模块化、信息隐藏和相关的好设计类的使用可能不是显而易见的真相，但它们在使大型程序易于理解和维护方面走得很远。一旦你领悟了，你就会想编写与全局变量和外部世界尽可能少联系的例程和类。

人们列举了使用全局数据的众多问题，但问题归结为少数几个主要问题：

**无意中更改全局数据**。你可能在一个地方更改变量 `theAnswer` 的值，却错误地认为它在其他地方保持不变。这样的问题被称为"副作用"。例如，在这个例子中，`theAnswer` 是一个全局变量：

示例 13-24. 副作用问题的 Visual Basic 示例

```vb
theAnswer = GetTheAnswer()       // 1
otherAnswer = GetOtherAnswer()       // 2
averageAnswer = (theAnswer + otherAnswer) / 2       // 3
```

(1) `theAnswer` 是一个全局变量。

(2) `GetOtherAnswer()` 更改了 `theAnswer`。

(3) `averageAnswer` 是错误的。

你可能假设对 `GetOtherAnswer()` 的调用不会更改 `theAnswer` 的值；如果它更改了，第三行的平均值将是错误的。而且，实际上，`GetOtherAnswer()` 确实更改了 `theAnswer` 的值，所以程序有一个需要修复的错误。

**全局数据出现奇特而令人兴奋的别名问题**。"别名"指的是通过两个或多个不同的名字调用同一个变量。当全局变量被传递给例程，然后例程既作为全局变量又作为参数使用它时，就会发生这种情况。这里有一个使用全局变量的例程：

**别名问题易发的例程的 Visual Basic 示例**

示例 13-25.

```vb
Sub WriteGlobal( ByRef inputVar As Integer )
   inputVar = 0
   globalVar = inputVar + 5
   MsgBox( "Input Variable: " & Str( inputVar ) )
   MsgBox( "Global Variable: " & Str( globalVar ) )
End Sub
```

以下是用全局变量作为参数调用该例程的代码：

示例 13-26. 用参数调用暴露别名问题的例程的 Visual Basic 示例

```vb
WriteGlobal( globalVar )
```

因为 `inputVar` 被初始化为 0，而 `WriteGlobal()` 向 `inputVar` 加 5 得到 `globalVar`，你期望 `globalVar` 比 `inputVar` 多 5。但这里有一个令人惊讶的结果：

示例 13-27. Visual Basic 中别名问题的结果

```
Input Variable:  5
Global Variable: 5
```

这里的微妙之处在于 `globalVar` 和 `inputVar` 实际上是同一个变量！因为 `globalVar` 通过调用例程传入 `WriteGlobal()`，它被两个不同的名字引用或"别名"。因此，`MsgBox()` 行的效果与预期大不相同：它们显示同一个变量两次，即使它们引用两个不同的名字。

**全局数据的可重入代码问题**。可以被多个控制线程进入的代码正变得越来越常见。多线程代码创造了全局数据不仅在例程之间共享，而且在同一程序的不同副本之间共享的可能性。在这样的环境中，你必须确保全局数据即使在程序的多份副本运行时也保持其含义。这是一个重要的问题，你可以通过本节后面建议的技术来避免。

**全局数据阻碍代码重用**。为了在一个程序中使用另一个程序的代码，你必须能够将其从第一个程序中拉出并插入第二个程序。理想情况下，你应该能够提取单个例程或类，将其插入另一个程序，然后继续愉快地前进。

全局数据使情况复杂化。如果你想重用的类读取或写入全局数据，你就不能简单地将其插入新程序。你必须修改新程序或旧类以使它们兼容。如果你走正路，你会修改旧类使其不使用全局数据。如果你这样做，下次你需要重用类时，你可以毫无困难地插入它。如果你走邪路，你会修改新程序以创建旧类需要使用的全局数据。这就像病毒；全局数据不仅影响原始程序，而且传播到使用旧程序类的任何新程序。

**全局数据的不确定初始化顺序问题**。在某些语言中（尤其是 C++），不同"翻译单元"（文件）之间数据初始化的顺序未定义。如果一个文件中全局变量的初始化使用了在不同文件中初始化的全局变量，除非你采取明确步骤确保两个变量按正确顺序初始化，否则第二个变量的值完全无法保证。

这个问题可以通过 Scott Meyers 在《Effective C++》Item #47 中描述的变通方法解决。但解决方案的棘手代表了使用全局数据引入的额外复杂性的典型。

**全局数据损害模块化和知识可管理性**。创建比几百行代码更大的程序的本质是管理复杂性。你能够知识管理大型程序的唯一方法是将其分解成小块，这样你就一次只需考虑一个部分。模块化是你可以用来将程序分解成小块的最有力工具。

全局数据会破坏你模块化的能力。如果你使用全局数据，你能一次集中在一个例程上吗？不能。你必须集中在一个例程以及使用相同全局数据的每个其他例程上。虽然全局数据不会完全破坏程序的模块化，但它会削弱它，这足以成为尝试为你的问题找到更好解决方案的理由。

## 使用全局数据的理由

数据纯粹主义者有时争论说程序员永远不应该使用全局数据，但如果从广义上解释，大多数程序使用"全局数据"。数据库中的数据是全局数据，配置文件（如 Windows 注册表）中的数据也是全局数据。命名常量是全局数据，只是不是全局变量。

有纪律地使用，全局变量在几种情况下很有用：

**保留全局值**。有时你有在概念上适用于整个程序的数据。这可能是一个反映程序状态的变量——例如，交互式 vs. 命令行模式，或正常 vs. 错误恢复模式。或者它可能是整个程序都需要的信息——例如，每个程序例程都使用的数据表。

**模拟命名常量**。虽然 C++、Java、Visual Basic 和大多数现代语言支持命名常量，但 Python、Perl、Awk 和 UNIX shell 脚本等一些语言仍然不支持。当你的语言不支持命名常量时，你可以使用全局变量作为命名常量的替代品。例如，你可以用设置为 1 和 0 的全局变量 `TRUE` 和 `FALSE` 替换字面值 1 和 0，或者你可以用 `LINES_PER_PAGE = 66` 替换每页 66 行作为行数。使用这种方法，以后更改代码更容易，代码往往更容易阅读。这种有纪律地使用全局数据是"编程进入语言"vs."在语言中编程"之间区别的典型例子，这在本章的"编程进入你的语言，而不是在语言中编程"中有更多讨论。

交叉参考

关于命名常量的更多细节，请参见"命名常量"。

**模拟枚举类型**。你也可以使用全局变量在 Python 等不直接支持枚举类型的语言中模拟枚举类型。

**简化极常见数据的使用**。有时你对变量的引用如此之多，以至于它出现在你编写的每个例程的参数列表中。与其将其包含在每个参数列表中，不如将其设为全局变量。然而，在变量似乎被到处访问的情况下，很少真的是这样。通常它只被一组有限的例程访问，你可以将它们打包成一个包含它们处理的数据的类。稍后会详细介绍。

**消除 tramp 数据**。有时你将数据传递给例程或类仅仅是为了将其传递给另一个例程或类。例如，你可能有一个在每个例程中使用的错误处理对象。当调用链中间的例程不使用该对象时，该对象被称为"tramp 数据"。使用全局变量可以消除 tramp 数据。

## 仅作为最后手段使用全局数据

在你诉诸使用全局数据之前，考虑几个替代方案：

**首先使每个变量成为局部变量，仅在需要时使变量成为全局变量**。最初将所有变量设为单个例程的局部变量。如果你发现它们在其他地方需要，在使其成为全局变量之前，先使其成为私有或受保护的类变量。如果你最终发现你必须使其成为全局变量，那就这样做，但只有当你确定你必须这样做时。如果你从一开始就使变量成为全局变量，你就永远不会使其成为局部变量，而如果你从使其成为局部变量开始，你可能永远不需要使其成为全局变量。

**区分全局变量和类变量**。有些变量真的是全局的，因为它们在整个程序中被访问。其他真的是类变量，仅在一组特定的例程中大量使用。在大量使用它的例程集中，以任何你喜欢的方式访问类变量是可以的。如果类外的例程需要使用它，通过访问例程提供变量的值。不要直接访问类值——就好像它们是全局变量一样——即使你的编程语言允许你。这等同于说"模块化！模块化！模块化！"

**使用访问例程**。创建访问例程是解决全局数据问题的主要方法。下一节有更多内容。

## 使用访问例程而不是全局数据

你可以用全局数据做的任何事情，用访问例程可以做得更好。使用访问例程是实现抽象数据类型和实现信息隐藏的核心技术。即使你不想使用完全成熟的抽象数据类型，你仍然可以使用访问例程来集中控制你的数据并保护自己免受更改。

### 访问例程的优点

使用访问例程有多个优点：

- **你可以集中控制数据**。如果你后来发现结构的实现更合适，你不必在数据被引用的每个地方更改代码。更改不会波及其整个程序。它们停留在访问例程内部。
- **你可以确保对变量的所有引用都被拦住**。如果你用像 `stack.array[ stack.top ] = newElement` 这样的语句将元素压入栈，你可以很容易地忘记检查栈溢出并犯一个严重错误。如果你使用访问例程——例如，`PushStack( newElement )`——你可以将栈溢出检查写入 `PushStack()` 例程。每次调用例程时都会自动进行检查，你可以忘记它。

交叉参考

关于拦路的更多细节，请参见"用拦障隔离程序以contain由错误造成的损害"。

- **你自动获得信息隐藏的一般好处**。访问例程是信息隐藏的一个例子，即使你没有为此目的设计它们。你可以更改访问例程的内部而不更改程序的其余部分。访问例程允许你重新装修房子的内部，而保持外观不变，这样你的朋友仍然能认出它。

交叉参考

关于信息隐藏的详细信息，请参见"隐藏秘密（信息隐藏）"。

- **访问例程易于转换为抽象数据类型**。使用访问例程的一个优势是你可以创建一个在使用全局数据时更难实现的抽象层级。例如，与其编写说 `if lineCount > MAX_LINES` 的代码，不如使用访问例程编写说 `if PageFull()` 的代码。这个小改动文档化了 if `lineCount` 测试的意图，而且是在代码中这样做的。这是可读性的一个小收益，但对这种细节的一致关注使精美制作的软件和只是拼凑在一起的代码产生了差异。

### 如何使用访问例程

以下是访问例程理论和实践的简短版本：将数据隐藏在类中。使用 `static` 关键字或其等效项声明该数据以确保只存在数据的单个实例。编写让你查看和更改数据的例程。要求类外的代码使用访问例程而不是直接处理数据。

例如，如果你有一个描述程序整体状态的全局状态变量 `g_globalStatus`，你可以创建两个访问例程：`globalStatus.Get()` 和 `globalStatus.Set()`，每个都做它听起来像的事情。这些例程访问隐藏在类内替换 `g_globalStatus` 的变量。程序的其余部分可以通过访问 `globalStatus.Get()` 和 `globalStatus.Set()` 来获得以前全局变量的所有好处。

如果你的语言不支持类，你仍然可以创建访问例程来操作全局数据，但你必须通过编码标准而不是内置编程语言强制执行来强制限制全局数据的使用。

交叉参考

即使你的语言不直接支持，限制对全局变量的访问也是"编程进入语言"vs."在语言中编程"的一个例子。更多细节请参见"编程进入你的语言，而不是在语言中编程"。

以下是当你的语言没有内置支持时，使用访问例程隐藏全局变量的一些详细指导原则：

**要求所有代码通过数据访问例程**。一个好的约定是要求所有全局数据以 `g_` 前缀开头，并进一步要求没有代码访问带有 `g_` 前缀的变量，除非通过该变量的访问例程。所有其他代码通过访问例程访问数据。

**不要把你的所有全局数据扔进同一个桶**。如果你把所有全局数据扔进一个大堆并为其编写访问例程，你消除了全局数据的问题，但你错过了信息隐藏和抽象数据类型的一些优势。只要你在编写访问例程，花点时间思考每个全局变量属于哪个类，然后将数据及其访问例程与该类中的其他数据和例程打包在一起。

**使用锁定控制对全局变量的访问**。类似于多用户数据库环境中的并发控制，锁定要求在可以使用或更新全局变量的值之前，必须"检出"该变量。变量使用后，它被检入。在它使用期间（已检出），如果程序的其他部分试图检出它，锁/解锁例程显示错误消息或触发断言。

这个锁定的描述忽略了许多完全支持并发的代码编写的微妙之处。由于这个原因，像这样简化的锁定方案在开发阶段最有用了。除非方案被非常好地思考过，否则它可能不够可靠，无法投入生产。当程序投入生产时，代码被修改为在检测到程序的多个部分试图锁定同一个全局变量时，做比显示错误消息更安全和更优雅的事情。例如，它可能将错误消息记录到文件。

交叉参考

关于计划程序开发和生产版本之间差异的详细信息，请参见"计划移除调试辅助工具"和"确定在生产代码中保留多少防御式编程"。

当你为全局数据使用访问例程时，这种开发时间保护措施相当容易实现，但如果你直接使用全局数据，它会很尴尬。

**在访问例程中构建一层抽象**。在问题域的层面构建访问例程，而不是在实现细节的层面。这种方法为你购买了改进的可读性以及对实现细节更改的保险。

比较表 13-1 中的配对语句：

**表 13-1. 直接使用全局数据与通过访问例程使用全局数据**

| 直接使用全局数据 | 通过访问例程使用全局数据 |
|-----------------|------------------------|
| `node = node.next` | `account = NextAccount( account )` |
| `node = node.next` | `employee = NextEmployee( employee )` |
| `node = node.next` | `rateLevel = NextRateLevel( rateLevel )` |
| `event = eventQueue[queueFront]` | `event = HighestPriorityEvent()` |
| `event = eventQueue[queueBack]` | `event = LowestPriorityEvent()` |

在前三个例子中，关键是抽象访问例程比通用结构告诉你更多。如果你直接使用结构，你同时做了太多事情：你既展示了结构本身在做什么（移动到链表中的下一个链接），也展示了关于它所代表的实体正在做什么（获取账户、下一个员工或费率级别）。这对简单的数据结构赋值来说是一个很大的负担。将信息隐藏在抽象访问例程后面让代码自己说话，使代码在问题域层面阅读，而不是在实现细节层面。

**将对数据的所有访问保持在相同的抽象层级**。如果你使用访问例程对结构做一个事情，你也应该使用访问例程对它做其他所有事情。如果你用访问例程读取结构，用访问例程写入它。如果你调用 `InitStack()` 初始化栈，用 `PushStack()` 将项目压入栈，你就创建了对数据的一致视图。如果你通过写入 `value = array[ stack.top ]` 来弹出栈，你就创建了对数据的不一致视图。不一致性使其他人更难理解代码。创建一个 `PopStack()` 例程，而不是写 `value = array[ stack.top ]`。

交叉参考

使用事件队列的访问例程表明需要创建一个类。详细信息请参见第6章。

在表 13-1 中语句对的例子中，两个事件队列操作是并行发生的。将事件插入队列比表中的两个操作中的任何一个都更棘手，需要几行代码来找到插入事件的位置、调整现有事件为新事件腾出空间，以及调整队列的前面或后面。从队列中移除事件同样复杂。在编码过程中，复杂的操作会被放入例程，其他操作会保留为直接数据操作。这会产生对结构丑陋的、非并行的使用。现在比较表 13-2 中的语句对：

**表 13-2. 复杂数据的非并行使用与并行使用**

|------------------|-----------------|
| `event = EventQueue[ queueFront ]` | `event = HighestPriorityEvent()` |

| `eventCount = eventCount - 1` | `AddEvent( event )` |

虽然你可能认为这些指导原则只适用于大型程序，但访问例程已被证明是避免全局数据问题的富有成效的方式。作为额外奖励，它们使代码更易读并增加了灵活性。

## 如何减少使用全局数据的风险

在大多数情况下，全局数据实际上是尚未很好设计或实现的类的类数据。在少数情况下，数据确实需要是全局的，但可以通过访问例程包装对它的访问以最小化潜在问题。在极少数剩余情况下，你确实需要使用全局数据。在这些情况下，你可以把遵循本节的指导原则想象成旅行到外国时为了喝水而打预防针：它们有点痛苦，但它们提高了保持健康的可能性。

**开发使全局变量显而易见的命名约定**。你可以通过使你正在处理全局数据这一点显而易见来避免一些错误。如果你将全局变量用于多个目的（例如，作为变量和作为命名常量的替代品），确保你的命名约定区分这些类型的用途。

交叉参考

关于全局变量命名约定的详细信息，请参见"识别全局变量"。

**创建所有全局变量的良好注释列表**。一旦你的命名约定表明变量是全局的，标明变量做什么是有帮助的。全局变量列表是在你的程序上工作的人可以拥有的最有用的工具之一。

**不要使用全局变量来包含中间结果**。如果你需要为全局变量计算新值，在计算结束时将最终值赋给全局变量，而不是用它来保存中间计算的结果。

**不要假装你通过将所有数据放入一个巨型对象并到处传递而不是使用全局数据**。将所有内容放入一个巨大的对象可能满足了法律条款，通过避免全局变量，但它只是开销，没有真正封装的任何好处。如果你使用全局数据，公开地做。不要用肥胖的对象来伪装它。

## 使用不常见数据类型的注意事项

### 结构体

- [ ] 你是否使用结构体而不是裸变量来组织和操作相关数据组？
- [ ] 你是否考虑过创建类作为使用结构体的替代方案？

### 全局数据

- [ ] 所有变量是否都是局部变量或类作用域，除非绝对需要是全局的？
- [ ] 变量命名约定是否区分局部、类和全局数据？
- [ ] 是否记录了所有全局变量？
- [ ] 代码是否没有伪全局数据——包含混乱数据混合物的大块对象传递给每个例程？
- [ ] 是否使用访问例程而不是全局数据？
- [ ] 访问例程和数据是否组织成类？
- [ ] 访问例程是否提供了超出底层数据类型实现的抽象层级？
- [ ] 所有相关访问例程是否在同一抽象层级？

### 指针

- [ ] 指针操作是否被隔离在例程中？
- [ ] 指针引用是否有效，或者指针可能是悬空的？
- [ ] 代码在使用指针之前是否检查指针的有效性？
- [ ] 使用指针引用的变量之前是否检查其有效性？
- [ ] 指针释放后是否设置为 null？
- [ ] 代码是否为可读性使用了所有需要的指针变量？
- [ ] 链表中的指针是否以正确顺序释放？
- [ ] 程序是否分配了备用内存降落伞，以便在内存耗尽时能够优雅关闭？
- [ ] 是否仅在没有办法时作为最后手段使用指针？

## 关键点

- 结构体可以帮助使程序不那么复杂、更易于理解和维护。
- 每当你考虑使用结构体时，考虑类是否可以更好地工作。
- 指针容易出错。通过使用访问例程或类以及防御式编程实践来保护自己。
- 避免全局变量，不仅因为它们危险，而且因为你可以用更好的东西替换它们。
- 如果你不能避免全局变量，通过访问例程处理它们。访问例程给你全局变量给你的所有东西，还有更多。
