防御式编程并不意味着对你的编程持防御态度——"它确实能工作！"这个想法基于防御性驾驶。在防御性驾驶中，你采用的心态是永远不确定其他司机会做什么。这样，你可以确保如果他们做出危险的事情，你不会受伤。即使可能是其他司机的错，你也承担保护自己的责任。在防御式编程中，主要思想是如果例程收到坏数据，它不会受损，即使坏数据是另一个例程的错。更一般地说，这是认识到程序会有问题和变更，聪明的程序员会相应地开发代码。

本章描述如何保护自己免受无效数据、"永远不会"发生的事件以及其他程序员错误等冷酷残酷世界的侵害。如果你是一名有经验的程序员，你可以跳过下一节关于处理输入数据的内容，从断言开始，这会回顾断言的使用。

## 保护程序免受无效输入

在学校你可能听过"垃圾进，垃圾出"这个表达。这个表达本质上是软件开发的"买者自负"版本：让用户小心。

对于生产软件，"垃圾进，垃圾出"是不够的。一个好的程序永远不会输出垃圾，无论它接受什么。一个好的程序使用"垃圾进，什么都不出"、"垃圾进，错误消息出"或"不允许垃圾进入"。按照今天的标准，"垃圾进，垃圾出"是草率、不安全程序的标志。

有三种处理垃圾的常用方法：

**检查来自外部源的所有数据的值。** 当从文件、用户、网络或其他外部接口获取数据时，检查以确保数据在允许范围内。确保数值在容差范围内，字符串足够短可以处理。如果字符串旨在表示受限范围的值（如金融交易ID或类似的东西），确保字符串对其预期目的是有效的；否则拒绝它。如果你正在处理安全应用程序，要特别警惕可能攻击你的数据：尝试缓冲区溢出、注入SQL命令、注入HTML或XML代码、整数溢出、传递给系统调用的数据等。

**检查所有例程输入参数的值。** 检查例程输入参数的值本质上与检查来自外部源的数据相同，只是数据来自另一个例程而不是外部接口。"用栅栏隔离程序以遏制错误造成的损害"中的讨论提供了一种实用方法来确定哪些例程需要检查其输入。

**决定如何处理坏输入。** 一旦检测到无效参数，你要怎么处理？根据情况，你可以选择十几种不同方法中的任何一种，这些方法在本章后面的"错误处理技术"中有详细描述。

防御式编程是本书描述的其他质量改进技术的有效辅助。防御性编码的最佳形式首先是不插入错误。使用迭代设计、在编码前编写伪代码、在编写代码前编写测试用例以及进行低级设计检查都是有助于防止插入缺陷的活动。因此，它们应该比防御式编程具有更高的优先级。幸运的是，你可以将防御式编程与其他技术结合使用。

如图8-1所示，保护自己免受看似小问题的影响可以产生比你想象的更大的影响。本章其余部分描述了检查外部数据、检查输入参数和处理坏输入的具体选项。

## 断言

断言是在开发期间使用的代码——通常是例程或宏——允许程序在运行时检查自身。当断言为真时，意味着一切按预期运行。当它为假时，意味着它检测到代码中的意外错误。例如，如果系统假设客户信息文件永远不会有超过50,000条记录，程序可能包含断言记录数小于或等于50,000。只要记录数小于或等于50,000，断言就会保持沉默。然而，如果遇到超过50,000条记录，它将大声"断言"程序中存在错误。

断言在大型、复杂程序和高可靠性程序中特别有用。它们使程序员能够更快地发现不匹配的接口假设、代码被修改时悄悄渗入的错误等。

断言通常接受两个参数：一个布尔表达式描述应该为真的假设，以及如果不为真时要显示的消息。如果期望变量denominator不为零，以下是Java断言的样子：

**示例8-1。** Java 断言示例

```java
assert denominator != 0 : "denominator is unexpectedly equal to 0.";
```

这个断言断言denominator不等于0。第一个参数denominator != 0是一个布尔表达式，评估为真或假。第二个参数是一个消息，如果第一个参数为假——即断言为假——则打印该消息。

**使用断言记录代码中的假设并发现意外情况。** 断言可用于检查这样的假设：

- 输入参数的值落在其预期范围内（或者输出参数的值如此）
- 当例程开始执行时（或执行结束时）文件或流是打开的（或关闭的）
- 当例程开始执行时（或执行结束时）文件或流在开头（或结尾）
- 文件或流以只读、只写或读写方式打开
- 仅输入变量的值不会被例程更改
- 指针非空
- 传入例程的数组或其他容器可以包含至少X个数据元素
- 表已初始化为包含真实值
- 当例程开始执行时（或完成时）容器是空的（或满的）
- 高度优化、复杂例程的结果与较慢但清晰编写的例程的结果匹配

当然，这些只是基础知识，你自己的例程将包含更多可以用断言记录的具体假设。

通常，你不希望用户在生产代码中看到断言消息；断言主要用于开发和维护期间。断言通常在开发时编译进代码，在生产时编译掉。在开发期间，断言会发现矛盾的假设、意外情况、传递给例程的坏值等。在生产期间，它们可以被编译掉，这样断言就不会降低系统性能。

### 使用断言的指导原则

以下是使用断言的指导原则：

**对你期望发生的情况使用错误处理代码；对绝不应该发生的情况使用断言。** 断言检查绝不应该发生的条件。错误处理代码检查不一定经常发生的非正常情况，但已被编写代码的程序员预料到，需要由生产代码处理。错误处理通常检查坏输入数据；断言检查代码中的错误。

如果错误处理代码用于处理异常情况，错误处理将使程序能够优雅地响应错误。如果为异常情况触发断言，纠正操作不仅仅是优雅地处理错误——纠正操作是更改程序的源代码、重新编译并发布新版本的软件。

考虑断言的好方法是将其作为可执行文档——你不能依赖它们使代码工作，但它们可以比程序语言注释更主动地记录假设。

**避免在断言中放置可执行代码。** 在断言中放入代码会导致编译器在你关闭断言时可能消除代码的可能性。假设你有这样的断言：

**示例8-3。** Visual Basic 断言危险使用示例

```vb
Debug.Assert( PerformAction() ) ' Couldn't perform action
```

> **交叉参考**
>
> 你可以将其视为在一行上放置多个语句相关的众多问题之一。更多示例请参见"布局单独语句"中的"每行只使用一条语句"。

这段代码的问题是，如果你不编译断言，你就不会编译执行该操作的代码。将可执行语句放在自己的行上，将结果赋给状态变量，然后测试状态变量。以下是断言安全使用的示例：

**示例8-4。** Visual Basic 断言安全使用示例

```vb
actionPerformed = PerformAction()
Debug.Assert( actionPerformed ) ' Couldn't perform action
```

**使用断言记录和验证前置条件和后置条件。** 前置条件和后置条件是称为"契约式设计"的程序设计和开发方法的一部分（Meyer 1997）。当使用前置条件和后置条件时，每个例程或类与程序的其余部分形成契约。

> **进一步阅读**
>
> 有关前置条件和后置条件的更多信息，请参见《面向对象软件构造》（Meyer 1997）。

前置条件是例程或类的客户代码承诺在调用例程或实例化对象之前为真的属性。前置条件是客户代码对其调用的代码的义务。

后置条件是例程或类承诺在完成执行时为真的属性。后置条件是例程或类对其使用代码的义务。

断言是记录前置条件和后置条件的有用工具。注释可用于记录前置条件和后置条件，但与注释不同，断言可以动态检查前置条件和后置条件是否为真。

在以下示例中，断言用于记录Velocity例程的前置条件和后置条件。

**示例8-5。** Visual Basic 使用断言记录前置条件和后置条件示例

```vb
Private Function Velocity ( _
   ByVal latitude As Single, _
   ByVal longitude As Single, _
   ByVal elevation As Single _
   ) As Single

   ' 前置条件
   Debug.Assert ( -90 <= latitude And latitude <= 90 )
   Debug.Assert ( 0 <= longitude And longitude < 360 )
   Debug.Assert ( -500 <= elevation And elevation <= 75000 )
   ...

   ' 后置条件
   Debug.Assert ( 0 <= returnVelocity And returnVelocity <= 600 )

   ' 返回值
   Velocity = returnVelocity
End Function
```

如果变量latitude、longitude和elevation来自外部源，则无效值应由错误处理代码检查和处理，而不是由断言处理。但是，如果变量来自可信的内部源，并且例程的设计基于这些值将在其有效范围内的假设，那么断言是合适的。

**对于高度健壮的代码，仍然断言然后处理错误。** 对于任何给定的错误情况，例程通常使用断言或错误处理代码，但不是两者兼用。一些专家认为只需要一种（Meyer 1997）。

> **交叉参考**
>
> 有关健壮性的更多信息，请参见本章后面的"错误处理技术"中的"健壮性vs正确性"。

但现实世界的程序和项目往往太混乱，无法仅依赖断言。在大型、长期运行的系统上，不同部分可能由不同设计人员在5-10年或更长时间内设计。设计人员将在时间上分离，跨越众多版本。他们的设计将集中在系统开发不同点的不同技术上。设计人员将在地理上分离，特别是如果系统部分从外部来源获取。程序员将在系统生命周期的不同时间点按照不同的编码标准工作。在大型开发团队中，一些程序员必然比其他程序员更认真，一些代码的审查比其他代码更严格。一些程序员的单元测试比其他程序员更彻底。随着测试团队在不同地理区域工作，并受到导致每次发布的测试覆盖率不同的业务压力影响，你也不能依赖全面的、系统级的回归测试。

在这种环境下，断言和错误处理代码都可能用于处理同一错误。例如，在Microsoft Word的源代码中，应该始终为真的条件会被断言，但此类错误也会由错误处理代码处理，以防断言失败。对于像Word这样极大、复杂、寿命长的应用程序，断言是有价值的，因为它们有助于在开发时发现尽可能多的错误。但应用程序如此复杂（数百万行代码），并经历了这么多代修改，假设每个可想象的错误都将在软件发布前被检测和纠正是不现实的，所以错误也必须在系统的生产版本中处理。

以下是这个方法在Velocity示例中如何工作的示例：

## 错误处理技术

断言用于处理代码中不应该发生的错误。如何处理你确实期望发生的错误？根据具体情况，你可能希望返回中立值、替换下一条有效数据、返回与上次相同的答案、替换最接近的合法值、记录警告消息到文件、返回错误代码、调用错误处理例程或对象、显示错误消息或关闭程序——或者你可能希望使用这些响应的组合。

以下是这些选项的更多细节：

**返回中立值。** 有时对坏数据最好的响应是继续运行并简单返回一个已知无害的值。数值计算可能返回0。字符串操作可能返回空字符串，或者指针操作可能返回空指针。在视频游戏中获取颜色坏输入值的绘图例程可能使用默认背景或前景颜色。然而，显示癌症患者X射线数据的绘图例程不会希望显示"中立值"。在这种情况下，关闭程序比显示错误的患者数据更好。

**替换下一条有效数据。** 在处理数据流时，某些情况只需要返回下一条有效数据。如果你在从数据库读取记录时遇到损坏的记录，你可以简单继续读取直到找到有效记录。如果你在以每秒100次的频率从温度计读取读数，而一次没有得到有效读数，你可以简单等待另外1/100秒并获取下一次读数。

**返回与上次相同的答案。** 如果温度计读取软件一次没有得到读数，它可以简单返回与上次相同的值。根据应用程序，温度在1/100秒内可能不会变化太多。在视频游戏中，如果你检测到将屏幕部分绘制为无效颜色的请求，你可以简单返回之前使用的颜色。但是，如果你在自动取款机上授权交易，你可能不想使用"与上次相同的答案"——那将是前一个用户的银行账号！

**替换最接近的合法值。** 在某些情况下，你可能选择返回最接近的合法值，如前面Velocity示例所示。当从校准仪器获取读数时，这通常是一种合理的方法。例如，温度计可以在0到100摄氏度之间校准。如果你检测到小于0的读数，你可以替换0，这是最接近的合法值。如果你检测到大于100的值，你可以替换100。对于字符串操作，如果报告字符串长度小于0，你可以替换0。我的车在我倒车时使用这种方法处理错误。由于我的速度计不显示负速度，当我倒车时它简单地显示0的速度——最接近的合法值。

**将警告消息记录到文件。** 检测到坏数据时，你可以选择将警告消息记录到文件然后继续。这种方法可以与其他技术结合使用，如替换最接近的合法值或替换下一条有效数据。如果你使用日志，考虑你是否能够安全地使其公开可用，或者你是否需要加密它或以其他方式保护它。

**返回错误代码。** 你可以决定只有系统的某些部分将处理错误。其他部分不会在本地处理错误；它们将只报告检测到错误，并信任调用层次结构中更高的一些其他例程将处理错误。通知系统其余部分发生错误的具体机制可以是以下任何一种：

- 设置状态变量的值
- 将状态作为函数的返回值返回
- 使用语言内置异常机制抛出异常

在这种情况下，具体的错误报告机制不如关于哪些部分将直接处理错误、哪些将只报告发生了错误的决定重要。如果安全性是个问题，确保调用例程始终检查返回代码。

**调用错误处理例程/对象。** 另一种方法是在全局错误处理例程或错误处理对象中集中错误处理。这种方法的优点是错误处理责任可以集中，这可以使调试更容易。权衡是整个程序将知道这个集中能力并将与它耦合。如果你曾经想在新系统中重用系统中任何代码，你将不得不与你重用的代码一起拖动错误处理机制。

这种方法有重要的安全含义。如果你的代码遇到缓冲区超限，攻击者可能已经破坏了处理程序例程或对象的地址。因此，一旦应用程序运行时发生缓冲区超限，使用这种方法就不再安全。

**在遇到错误的地方显示错误消息。** 这种方法最小化错误处理开销；但是，它确实有在整个应用程序中散布用户界面消息的潜力，这会在你需要创建一致的用户界面、尝试清晰分离UI和系统其余部分或尝试将软件本地化为不同语言时造成挑战。同时，警惕告诉潜在攻击者系统太多信息。攻击者有时使用错误消息来发现如何攻击系统。

**以最适合本地的方式处理错误。** 一些设计要求在本地处理所有错误——使用哪种特定错误处理方法的决定留给设计和实现遇到错误的系统部分程序员。

这种方法为个人开发人员提供了很大的灵活性，但它创造了系统整体性能可能无法满足其正确性或健壮性要求的重大风险（稍后会详细介绍）。根据开发人员最终如何处理特定错误，这种方法也有在整个系统中散布用户界面代码的潜力，这会使程序暴露与显示错误消息相关的所有问题。

**关闭程序。** 一些系统在检测到错误时关闭。这种方法在安全关键应用程序中很有用。例如，如果控制癌症患者放射设备放射剂量的软件收到放射剂量的坏输入数据，其最佳错误处理响应是什么？它应该使用与上次相同的值吗？它应该使用最接近的合法值吗？它应该使用中立值吗？在这种情况下，关闭是最好的选择。我们宁愿重启机器也不愿冒交付错误剂量的风险。

类似的方法可用于提高Microsoft Windows的安全性。默认情况下，即使安全日志已满，Windows也会继续运行。但你可以将Windows配置为在安全日志变满时停止服务器，这在安全关键环境中可能是适当的。

---

### 健壮性vs正确性

正如视频游戏和X射线示例向我们展示的，最合适的错误处理风格取决于错误发生的软件类型。这些示例还说明错误处理通常偏向于更高的正确性或更高的健壮性。开发人员倾向于非正式地使用这些术语，但严格来说，这些术语彼此处于规模的两端。正确性意味着永远不返回不准确的结果；不返回结果比返回不准确的结果更好。健壮性意味着始终尝试做一些允许软件继续运行的事情，即使这有时会导致不准确的结果。

安全关键应用程序倾向于偏向正确性而非健壮性。不返回结果比返回错误结果更好。放射机器是这一原则的好例子。

消费者应用程序倾向于偏向健壮性而非正确性。任何结果通常比软件关闭更好。我正在使用的文字处理器偶尔会在屏幕底部显示一行文本的一小部分。如果它检测到该情况，我是否希望文字处理器关闭？不。我知道下次我按Page Up或Page Down时，屏幕将刷新，显示将恢复正常。

---

### 错误处理的高层设计含义

有这么多选项，你需要小心在整个程序中以一致的方式处理无效参数。错误处理的方式影响软件满足与正确性、健壮性和其他非功能属性相关要求的能力。决定对坏参数的一般方法是架构或高层设计决策，应该在其中一个层面处理。

一旦你决定了方法，确保你一致地遵循它。如果你决定让高层代码处理错误而让低级代码只报告错误，确保高层代码实际处理错误！一些语言给你忽略函数返回错误代码这一事实的选项——在C++中，你不需要对函数返回值做任何事情——但不要忽略错误信息！测试函数返回值。如果你期望函数永远不会产生错误，还是要检查。不管怎样。防御式编程的全部意义在于防范你意想不到的错误。

这一指导原则对系统函数以及你自己的函数都适用。除非你设置了不检查系统调用错误这样的架构指导原则，否则每次调用后检查错误代码。如果你检测到错误，包括错误号和错误描述。

## 异常

异常是代码可以将错误或异常事件传递给它所调用的代码的特定方式。如果一个例程中的代码遇到它不知道如何处理的意外情况，它会抛出异常，本质上是举起手来喊道："我不知道该怎么办这个——我真希望其他人知道如何处理它！"对错误上下文没有感觉的代码可以将控制权返回给系统其他可能有更好能力解释错误并做出有用响应的部分。

异常也可用于理顺单个代码段内的纠缠逻辑，如goto中的"用try-finally重写"示例。异常的基本结构是例程使用throw抛出异常对象。调用层次结构中某个其他例程中的代码将在try-catch块中捕获异常。

流行语言在实现异常方面有所不同。表8-1总结了其中三种语言的主要区别：

**表8-1。流行语言的异常支持**

| 异常属性 | C++ | Java | Visual Basic |
|---------|-----|------|--------------|
| 支持try-catch | yes | yes | yes |
| 支持try-catch-finally | no | yes | yes |
| 可以抛出什么 | 异常对象或从Exception类派生的对象；对象指针；对象引用；字符串或int等数据类型 | 异常对象或从Exception类派生的对象 | 异常对象或从Exception派生的对象 |
| 未捕获异常的效果 | 调用std::unexpected()，默认调用std::terminate()，默认调用abort() | 如果异常是"已检查异常"则终止执行线程；如果是"运行时异常"则无影响 | 终止程序 |
| 抛出的异常必须在类接口中定义 | 否 | 是 | 否 |
| 捕获的异常必须在类接口中定义 | 否 | 是 | 否 |

---

异常与继承有一个共同属性：谨慎使用，它们可以减少复杂性。使用不谨慎，它们几乎不可能遵循。本节包含实现异常好处和避免通常与之相关困难的建议。

> 使用异常作为正常处理一部分的程序遭受经典意大利面条式代码的所有可读性和可维护性问题。
> —— Andy Hunt Dave Thomas

**使用异常通知程序其他部分不应被忽略的错误。** 异常的主要好处是它们能够以不能被忽略的方式发信号通知错误条件（Meyers 1996）。处理错误的其他方法会产生错误条件可能在代码库中未被检测到地传播的可能性。异常消除了这种可能性。

**只为真正例外的情况抛出异常。** 异常应保留用于真正例外的情况——换句话说，用于不能通过其他编码实践处理的情况。异常在类似于断言的情况下使用——对于不仅不频繁而且永远不应该发生的事件。

异常代表了一种处理意外情况的强大方式与增加复杂性之间的权衡。异常通过要求调用例程的代码知道被调用代码内部可能抛出哪些异常来削弱封装。这增加了代码复杂性，这与第5章称为"软件的主要技术使命：管理复杂性"背道而驰。

**不要使用异常来推卸责任。** 如果错误条件可以在本地处理，就在本地处理。如果你能本地处理错误，不要在代码段中抛出未捕获的异常。

**避免在构造函数和析构函数中抛出异常，除非你在同一位置捕获它们。** 当在构造函数和析构函数中抛出异常时，关于异常如何处理的规则很快就会变得非常复杂。例如在C++中，除非对象完全构造，否则不会调用析构函数，这意味着如果构造函数内的代码抛出异常，将不会调用析构函数，从而设置可能的资源泄漏（Meyers 1996, Stroustrup 1997）。类似复杂的规则也适用于析构函数中的异常。

语言律师可能会说记住这样的规则是"微不足道的"，但凡人程序员会很难记住它们。更好的编程实践是简单地通过首先不编写那种代码来避免这种代码造成的额外复杂性。

**在正确的抽象层次抛出异常。** 例程应该在接口中呈现一致的抽象，类也应该如此。抛出的异常是例程接口的一部分，就像特定数据类型一样。

> **交叉参考**
>
> 有关维护一致的接口抽象的更多信息，请参见"良好的类接口"中的"良好的抽象"。

当你选择将异常传递给调用者时，确保异常的抽象层次与例程接口的抽象一致。以下是不应该做的事情的示例：

**示例8-7。** 在不一致抽象层次抛出异常的类的糟糕Java示例

```java
class Employee {
  ...
  public TaxId GetTaxId() throws EOFException {       // 1
    ...
  }
  ...
}
```

(1) 这里是声明处于不一致抽象层次的异常。

GetTaxId()代码将较低级的EOFException异常传递回其调用者。它不接管异常本身；它通过将较低级异常传递给调用者来暴露其实现的一些细节。这有效地将例程客户代码耦合到不是Employee类的代码而是Employee类下面抛出EOFException异常的代码。封装被破坏，智力可管理性开始下降。

相反，GetTaxId()代码应该传递与其所属类接口一致的异常，像这样：

**示例8-8。** 在一致抽象层次抛出异常的类的好Java示例

```java
class Employee {
  ...
  public TaxId GetTaxId() throws EmployeeDataNotAvailable {       // 1
    ...
  }
  ...
}
```

(1) 这里是声明有助于一致抽象层次的异常。

GetTaxId()内部的异常处理代码可能只是将io_disk_not_ready异常映射到EmployeeDataNotAvailable异常，这很好，因为这足以保留接口抽象。

**在异常消息中包含导致异常的所有信息。** 每个异常都发生在代码抛出异常时检测到的特定情况下。这些信息对阅读异常消息的人非常有价值。确保消息包含理解为什么抛出异常所需的信息。如果因为数组索引错误而抛出异常，确保异常消息包括数组的上下限和非法索引的值。

**避免空的catch块。** 有时你想忽略你不知道该怎么处理的异常，像这样：

**示例8-9。** 忽略异常的糟糕Java示例

```java
try {
   ...
   // lots of code
  ...
} catch ( AnException exception ) {
}
```

这种方法表明，要么try块中的代码因为没有原因引发异常是错误的，要么catch块中的代码不处理有效异常是错误的。确定哪个是问题的根本原因，然后修复try块或catch块。

你可能偶尔会发现罕见的情况，其中较低层次的异常真正不代表调用例程抽象层次的异常。如果是这样，至少记录为什么空的catch块是适当的。你可以用注释记录该情况，或者通过将消息记录到文件，如下所示：

**示例8-10。** 忽略异常的好Java示例

```java
try {
   ...
   // lots of code
  ...
} catch ( AnException exception ) {
   LogError( "Unexpected exception" );
}
```

**了解你的库代码抛出的异常。** 如果你在一种不要求例程或类定义其抛出异常的语言中工作，确保你知道你使用的任何库代码抛出的异常。未捕获库代码生成的异常会像未捕获你自己生成的异常一样快地使程序崩溃。如果库代码不记录它抛出的异常，创建原型代码来行使库并发现异常。

**考虑构建集中式异常报告器。** 确保异常处理一致的一种方法是使用集中式异常报告器。集中式异常报告器提供关于存在哪些异常、如何处理每个异常、异常消息格式等方面的知识集中存储库。

以下是简单打印诊断消息的简单异常处理程序示例：

**示例8-11。** Visual Basic 集中式异常报告器示例，第1部分

```vb
Sub ReportException( _
   ByVal className, _
   ByVal thisException As Exception _
)
   Dim message As String
   Dim caption As String

   message = "Exception: " & thisException.Message & "." & ControlChars.CrLf & _
      "Class: " & className & ControlChars.CrLf & _
      "Routine: " & thisException.TargetSite.Name & ControlChars.CrLf
   caption = "Exception"
   MessageBox.Show( message, caption, MessageBoxButtons.OK, _
      MessageBoxIcon.Exclamation )

End Sub
```

> **进一步阅读**
>
> 有关此技术更详细的解释，请参见《Microsoft Visual Basic .NET实用标准》（Foxall 2003）。

你可以将这个通用异常处理程序与这样的代码一起使用：

**示例8-12。** Visual Basic 集中式异常报告器示例，第2部分

```vb
Try
  ...
Catch exceptionObject As Exception
  ReportException( CLASS_NAME, exceptionObject )
End Try
```

这个版本的ReportException()代码很简单。在实际应用程序中，你可以根据需要使代码简单或复杂以满足你的异常处理需求。

如果你确实决定构建集中式异常报告器，一定要考虑集中式错误处理涉及的一般问题，这些在"错误处理技术"中的"调用错误处理例程/对象"中讨论。

**标准化你的项目对异常的使用。** 为使异常处理尽可能在智力上可管理，你可以通过几种方式标准化异常的使用：

- 如果你在像C++这样的语言中工作，允许你抛出各种对象、数据和指针，具体标准化你要抛出的内容。为与其他语言兼容，考虑只抛出从Exception基类派生的对象。
- 考虑创建你自己特定于项目的异常类，它可以作为项目上抛出所有异常的基类。这支持集中化和标准化日志记录、错误报告等。
- 定义允许代码使用throw-catch语法在本地执行错误处理的具体情况。
- 定义允许代码抛出不会在本地处理异常的具体情况。
- 确定是否将使用集中式异常报告器。
- 定义是否允许在构造函数和析构函数中使用异常。

**考虑异常的替代方案。** 几种编程语言已经支持异常5-10年或更长时间，但没有形成关于如何安全使用异常的传统智慧。

> **交叉参考**
>
> 有关众多替代错误处理方法，请参见本章前面的"错误处理技术"。

一些程序员仅仅因为他们的语言提供了特定的错误处理机制就使用异常处理错误。你应该始终考虑完整的错误处理替代方案：在本地处理错误、使用错误代码传播错误、将调试信息记录到文件、关闭系统或使用其他方法。仅仅因为你的语言提供异常处理就用异常处理错误是"在语言中编程"而不是"编程进入语言"的经典例子。（有关这种区别的详细信息，请参见"你在技术浪潮中的位置"以及"编程进入你的语言，而非在语言中编程"。）

最后，考虑你的程序是否真的需要处理异常。Bjarne Stroustrup指出，对严重运行时错误的最佳响应有时是释放所有获取的资源并中止。让用户用正确的输入重新运行程序（Stroustrup 1997）。

## 用栅栏隔离程序以遏制错误造成的损害

栅栏是一种损害遏制策略。原因类似于在船体中有隔离舱。如果船撞上冰山并打开船体，该舱被关闭，船只的其余部分不受影响。它们也类似于建筑物中的防火墙。建筑物的防火墙防止火灾从建筑物的一部分蔓延到另一部分。（用于防御式编程目的的栅栏过去常被称为"防火墙"，但"防火墙"一词现在通常指阻止恶意网络流量。）

防御式编程中用栅栏隔离的一种方式是将某些接口指定为"安全"区域的边界。检查跨越安全区域边界的数据的有效性，如果数据无效则做出合理响应。图8-2说明了这个概念。

---

**图8-2。** 定义处理脏数据的一些软件部分和处理干净数据的一些部分可以是有效的方法，使大部分代码免于承担检查坏数据的责任。

这种方法可以在类级别使用。类的公共方法假设数据不安全，它们负责检查数据并清理数据。一旦数据被类的公共方法接受，类的私有方法可以假设数据是安全的。

思考这种方法的其他方式是将其视为手术室技术。数据在被允许进入手术室之前被消毒。进入手术室的任何东西都假定是安全的。关键设计决策是决定什么放在手术室里、什么放在外面、门放在哪里——哪些例程被认为在安全区域内、哪些在外面、哪些消毒数据。最简单的方法通常是在外部数据到达时消毒，但数据通常需要在多个层次消毒，所以有时需要多个级别的消毒。

**在输入时将输入数据转换为适当类型。** 输入通常以字符串或数字形式到达。有时值将映射到布尔类型如"是"或"否"。有时值将映射到枚举类型如Color_Red、Color_Green和Color_Blue。在程序中以可疑类型携带数据任何长度都会增加复杂性，并增加某人通过输入"是"这样的颜色使程序崩溃的机会。在输入后尽快将输入数据转换为适当形式。

---

### 栅栏与断言的关系

栅栏的使用使断言和错误处理之间的区别变得清晰。栅栏外的例程应使用错误处理，因为对数据做出任何假设都是不安全的。栅栏内的例程应使用断言，因为传递给它们的数据应该在跨越栅栏传递之前被清理。如果栅栏内的例程之一检测到坏数据，那是程序中的错误而不是数据中的错误。

栅栏的使用也说明在架构层面决定如何处理错误的价值。决定哪些代码在栅栏内和哪些在栅栏外是架构层面的决策。

## 调试辅助

防御式编程的另一个关键方面是使用调试辅助，这可以成为快速检测错误的强大盟友。

---

### 不要自动将生产约束应用于开发版本

一个常见的程序员盲点是假设生产软件的限制适用于开发版本。生产版本必须运行快。开发版本可能可以运行慢。生产版本必须吝啬资源。开发版本可能允许奢侈地使用资源。生产版本不应向用户暴露危险操作。开发版本可以有你可以无安全网使用的额外操作。

> **进一步阅读**
>
> 有关使用调试代码支持防御式编程的更多信息，请参见《编写可靠代码》（Maguire 1993）。

我参与的一个程序大量使用四重链表。链表代码容易出错，链表往往被破坏。我添加了一个菜单选项来检查链表的完整性。

在调试模式下，Microsoft Word在空闲循环中包含每几秒检查一次Document对象完整性的代码。这有助于快速检测数据损坏，并使错误诊断更容易。

愿意在开发期间以速度和资源使用换取可能使开发更顺利的内置工具。

### 尽早引入调试辅助

你越早引入调试辅助，它们就会越有帮助。通常，在你被问题困扰几次之前，你不会费力编写调试辅助。但是，如果你第一次后就编写辅助，或从前一个项目使用，它将在整个项目中提供帮助。

### 使用进攻式编程

异常情况应该以在开发期间使其显而易见、在生产代码运行时可恢复的方式处理。Michael Howard和David LeBlanc称这种方法为"进攻式编程"（Howard and LeBlanc 2003）。

> **交叉参考**
>
> 有关处理意外情况的更多详细信息，请参见"case语句技巧"中的"处理意外情况"。

假设你有一个case语句，期望只处理五种事件。在开发期间，default case应该用于生成警告说"嘿！这里有另一种情况！修复程序！"然而在生产期间，default case应该做更优雅的事情，比如将消息写入错误日志文件。

以下是一些你可以进攻式编程的方式：

> 死掉的程序通常比残缺的程序造成的损害要小得多。
> —— Andy Hunt Dave Thomas

- 确保断言中止程序。不允许程序员养成仅仅敲击回车键绕过已知问题的习惯。让问题足够痛苦以至于它将被修复。
- 完全填充任何分配的内存，以便你可以检测内存分配错误。
- 完全填充任何分配的文件或流，以发现任何文件格式错误。
- 确保每个case语句的default或else子句中的代码失败严重（中止程序）或其他方式不可忽视。
- 在删除对象之前用垃圾数据填充它。
- 设置程序将错误日志文件通过电子邮件发送给你自己，这样你就可以看到发布的软件中正在发生的错误类型，如果这对你的软件类型合适的话。

> 有时最好的防御是良好的进攻。在开发期间失败困难，这样你就可以在生产期间较软地失败。

### 计划移除调试辅助

如果你为自用编写代码，将所有调试代码保留在程序中可能没问题。如果你为商业用途编写代码，大小和速度方面的性能损失可能是禁止的。计划避免在程序中来回调整调试代码。以下是几种方法：

**使用版本控制工具和构建工具如ant和make。** 版本控制工具可以从相同源文件构建不同版本的程序。在开发模式下，你可以将构建工具设置为包含所有调试代码。在生产模式下，你可以将其设置为排除你不想在商业版本中的任何调试代码。

> **交叉参考**
>
> 有关版本控制的详细信息，请参见"配置管理"。

**使用内置预处理器。** 如果你的编程环境有预处理器——如C++——你可以用编译器开关打开或关闭调试代码。你可以直接使用预处理器，或编写与预处理器定义一起工作的宏。以下是直接使用预处理器编写代码的示例：

**示例8-13。** C++ 直接使用预处理器控制调试代码示例

```cpp
#define DEBUG       // 1
...

#if defined( DEBUG )
// debugging code
...

#endif
```

(1) 要包含调试代码，使用#define定义符号DEBUG。要排除调试代码，不要定义DEBUG。

这个主题有几种变体。不仅仅定义DEBUG，你可以为其赋值然后测试值而不是测试是否定义。这样你可以区分不同级别的调试代码。你可能有一些调试代码你一直想在程序中，所以用`#if DEBUG > 0`包围。其他调试代码可能仅用于特定目的，所以你可以用`#if DEBUG == POINTER_ERROR`包围。在其他地方，你可能想设置调试级别，所以你可以有`#if DEBUG > LEVEL_A`这样的语句。

如果你不喜欢`#if defined()`分散在你的代码中，你可以编写一个预处理器宏来完成相同的任务。以下是一个示例：

**示例8-14。** C++ 使用预处理器宏控制调试代码示例

```cpp
#define DEBUG
#if defined( DEBUG )
#define DebugCode( code_fragment ) { code_fragment }
#else
#define DebugCode( code_fragment )
#endif
...

DebugCode(
   statement 1;       // 1
   statement 2;         |
   ...                  |
   statement n;       // 1
);
...
```

(1) 这段代码根据DEBUG是否被定义而被包含或排除。

与第一个使用预处理器的示例一样，这种技术可以以各种方式改变，使其比完全包含所有调试代码或完全排除所有调试代码更复杂。

**编写你自己的预处理器。** 如果语言不包含预处理器，为包含和排除调试代码编写一个预处理器是相当容易的。建立指定调试代码的约定，并编写预编译器遵循该约定。例如，在Java中你可以编写预编译器来响应关键字//#BEGIN DEBUG和//#END DEBUG。编写调用预处理器的脚本，然后编译处理后的代码。从长远来看你会节省时间，你也不会错误地编译未预处理的代码。

> **交叉参考**
>
> 有关预处理器的更多信息和关于编写自己的预处理器信息来源的方向，请参见"可执行代码工具"中的"宏预处理器"。

**使用调试存根。** 在许多情况下，你可以调用例程进行调试检查。在开发期间，例程可能在控制返回调用者之前执行几个操作。对于生产代码，你可以用只立即将控制返回给调用者的存根例程替换复杂例程，或者在返回控制之前执行几个快速操作。这种方法只产生很小的性能损失，而且比编写自己的预处理器更快。在开发和生产版本中都保留例程，这样你就可以在未来开发和生产之间来回切换。

> **交叉参考**
>
> 有关存根的详细信息，请参见"测试支持工具"中的"构建脚手架来测试单独例程"。

你可以从一个旨在检查传递给它的指针的例程开始：

**示例8-15。** C++ 使用调试存根的例程示例

```cpp
void DoSomething(
   SOME_TYPE *pointer;
   ...
   ) {

   // check parameters passed in
   CheckPointer( pointer );       // 1
   ...
}
```

(1) 这行调用检查指针的例程。

在开发期间，CheckPointer()例程将对指针执行完整检查。它会慢但有效，可能看起来像这样：

**示例8-16。** C++ 开发期间检查指针的例程示例

```cpp
void CheckPointer( void *pointer ) {       // 1
   // 执行检查1——也许检查它不是NULL
   // 执行检查2——也许检查它的狗牌是合法的

   // 执行检查3——也许检查它指向的东西没有被破坏
   ...
   // 执行检查n——...
}
```

(1) 这个例程检查传递给它的任何指针。它可以在开发期间使用来执行你能承受的尽可能多的检查。

当代码准备好生产时，你可能不希望所有与指针检查相关的开销。你可以用这个例程交换前面的例程：

**示例8-17。** C++ 生产期间检查指针的例程示例

```cpp
void CheckPointer( void *pointer ) {       // 1
   // 没有代码；只返回给调用者
}
```

(1) 这个例程只立即返回给调用者。

这不是你可以计划移除调试辅助的所有方式的详尽调查，但它应该足以让你了解在你的环境中会起作用的一些东西。

## 决定在生产代码中保留多少防御式编程

防御式编程的矛盾之一是，在开发期间，你希望错误是明显的——你宁愿它令人讨厌也不要冒忽略它的风险。但在生产期间，你宁愿错误尽可能不引人注目，让程序恢复或优雅失败。以下是关于决定将哪些防御式编程工具留在生产代码中、哪些不留在生产代码中的一些指导原则：

**保留检查重要错误的代码。** 决定程序的哪些区域可以承受未检测到的错误，哪些区域不能。例如，如果你正在编写电子表格程序，你可以承受程序屏幕更新区域中未检测到的错误，因为错误的唯一主要惩罚是屏幕混乱。你不能承受计算引擎中未检测到的错误，因为这样的错误可能导致某人电子表格中的结果轻微错误。大多数用户宁愿忍受混乱的屏幕也不愿忍受错误的税务计算和IRS审计。

**移除检查无关紧要错误的代码。** 如果错误确实后果微不足道，移除检查它的代码。在前面的示例中，你可能移除检查电子表格屏幕更新的代码。"移除"不是物理移除代码。它意味着使用版本控制、预编译器开关或其他技术来编译不包括该特定代码的程序。如果空间不是问题，你可以保留错误检查代码但让它不显眼地将消息记录到错误日志文件。

**移除导致硬崩溃的代码。** 正如我提到的，在开发期间，当程序检测到错误时，你希望错误尽可能明显，这样你就可以修复它。通常，实现该目标的最好方式是让程序在检测到错误时打印调试消息并崩溃。这即使对次要错误也很有用。

在生产期间，你的用户需要在程序崩溃前有机会保存他们的工作，他们可能愿意容忍一些异常以换取让程序继续运行足够长时间来完成工作。

> 用户不感激任何导致丢失工作的事情，无论它对调试有多大帮助，最终会提高程序质量。如果你的程序包含可能导致数据丢失的调试代码，请将其从生产版本中取出。

**保留允许程序优雅崩溃的代码。** 如果你的程序包含检测潜在致命错误的调试代码，保留允许程序优雅崩溃的代码。例如在火星探路者号上，工程师故意保留了一些调试代码。探路者号着陆后发生了错误。通过使用保留的调试辅助，JPL的工程师能够诊断问题并将修改后的代码上传到探路者号，探路者号完美完成了任务（1999年3月）。

**记录错误给你的技术支持人员。** 考虑将调试辅助保留在生产代码中但更改其行为以适合生产版本。如果你用断言加载了代码，这些断言在开发期间会停止程序，你可能考虑在生产期间将断言例程更改为将消息记录到文件而不是完全消除它们。

**确保你保留的错误消息是友好的。** 如果你在程序中保留内部错误消息，验证它们使用的是对用户友好的语言。在我早期的一个程序中，我接到一个用户的电话，她报告说她收到了一条写着"你有一个坏的指针分配，狗呼吸！"的消息。对我来说幸运的是，她有幽默感。常见且有效的方法是通知用户发生了"内部错误"，并列出用户可以用于报告的电子邮件地址或电话号码。

## 对防御式编程保持防御态度

过多的防御式编程会造成自己的问题。如果你在每个可想象的地方以每种可想象的方式检查作为参数传递的数据，你的程序会变得臃肿和缓慢。更糟糕的是，防御式编程所需的额外代码增加了软件的复杂性。为防御式编程安装的代码也不能免于缺陷，你和其他代码一样可能在防御式编程代码中发现缺陷——如果你随意编写代码，可能性更大。思考你需要在哪里保持防御态度，并相应地设置你的防御式编程优先级。

> 任何事情太多都不好，但太多威士忌刚刚好。
> —— Mark Twain

## 防御式编程检查清单

### 一般防御式编程

- 例程是否保护自己免受坏输入数据的影响？
- 你是否使用断言记录假设，包括前置条件和后置条件？
- 断言是否仅用于记录绝不应该发生的情况？
- 架构或高层设计是否指定了特定的错误处理技术集合？
- 架构或高层设计是否指定错误处理偏向健壮性还是正确性？
- 是否创建了栅栏来遏制错误造成的损害并减少必须关注错误处理的代码量？
- 代码中是否使用了调试辅助？
- 调试辅助是否安装成可以激活或停用而不会大惊小怪？
- 防御式编程代码的数量是否适当——不过多也不过少？
- 你是否使用进攻式编程技术使错误在开发期间难以被忽视？

### 异常

- 你的项目是否定义了异常处理的标准化方法？
- 你是否考虑了使用异常的替代方案？
- 如果可能，错误是否在本地处理而不是抛出非局部异常？
- 代码是否避免在构造函数和析构函数中抛出异常？
- 所有异常是否都在抛出它们的例程的适当抽象层次？
- 每个异常是否包含所有相关异常背景信息？
- 代码是否没有空的catch块？（或者如果空的catch块确实适当，是否已记录？）

### 安全问题

- 检查坏输入数据的代码是否检查了尝试缓冲区溢出、SQL注入、HTML注入、整数溢出和其他恶意输入？
- 是否检查了所有错误返回代码？
- 是否捕获了所有异常？
- 错误消息是否避免提供有助于攻击者闯入系统的信息？

## 关键点

- 生产代码处理错误的方式应该比"垃圾进，垃圾出"更复杂。
- 防御式编程技术使错误更容易发现、更容易修复，对生产代码的损害更小。
- 断言可以帮助早期检测错误，特别是在大型系统、高可靠性系统和快速变化的代码库中。
- 关于如何处理坏输入的决定是关键错误处理决策，也是关键高层设计决策。
- 异常提供了一种与代码正常流程不同维度的错误处理手段。当谨慎使用时，它们是程序员智力工具箱中的有价值补充，应该与其他错误处理技术权衡。
- 适用于生产系统的约束不一定适用于开发版本。你可以利用这一点，在开发版本中添加帮助快速发现错误的代码。
