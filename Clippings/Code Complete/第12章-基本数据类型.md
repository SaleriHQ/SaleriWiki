## 数字概述

以下是使你对数字的使用减少出错的一些指导原则：

交叉参考

关于使用命名常量代替幻数的更多细节，请参见本章后面的"命名常量"。

**避免"幻数"**。幻数是没有解释就出现在程序中间的字面数字，如 100 或 47524。如果你使用的语言支持命名常量，用它们代替。如果不能使用命名常量，在可行的情况下使用全局变量。

避免幻数产生三个优点：

- 更改可以更可靠地进行。如果你使用命名常量，你不会忽略某个 100 或更改引用其他东西的 100。
- 更改可以更容易地进行。当最大条目数从 100 更改为 200 时，如果你使用幻数，你必须找到所有的 100 并将它们改为 200。如果你使用 100+1 或 100-1，你还必须找到所有的 101 和 99 并将它们改为 201 和 199。如果你使用命名常量，你只需在一个地方将常量的定义从 100 更改为 200。
- 你的代码更易读。当然，在表达式

```cpp
for i = 0 to 99 do ...
```

中，你可以猜测 99 是指最大条目数。但在表达式

```cpp
for i = 0 to MAX_ENTRIES-1 do ...
```

中毫无疑问。即使你确定一个数字永远不会改变，如果你使用命名常量，你也会获得可读性的好处。

---

**必要时使用硬编码的 0 和 1**。值 0 和 1 用于递增、递减和从数组的第一个元素开始循环。以下表达式中的 0 是可以的：

```cpp
for i = 0 to CONSTANT do ...
```

以及

```cpp
total = total + 1
```

中的 1。一个好的经验法则是，程序体中应该出现的唯一字面量是 0 和 1。任何其他字面量都应该用更具描述性的东西替换。

**防止除以零错误**。每次使用除法符号（大多数语言中为 `/`）时，都要考虑表达式的分母是否可能为 0。如果存在这种可能性，编写代码以防止除以零错误。

**使类型转换显而易见**。确保阅读你代码的人在发生不同数据类型之间的转换时会注意到它。在 C++ 中你可以说：

```cpp
y = x + (float) i
```

在 Microsoft Visual Basic 中你可以说：

```vb
y = x + CSng( i )
```

这种做法也有助于确保转换是你想要发生的——不同的编译器做不同的转换，否则你就只能碰运气。

**避免混合类型比较**。如果 x 是浮点数而 i 是整数，那么测试：

```cpp
if ( i = x ) then ...
```

几乎肯定不能工作。当编译器弄清楚它想用什么类型进行比较、将一种类型转换为另一种类型、做一堆四舍五入、然后确定答案时，如果你的程序能运行的话，你就是幸运的。手动进行转换，这样编译器就可以比较两个相同类型的数字，而你确切地知道正在比较什么。

**注意编译器的警告**。许多现代编译器会告诉你什么时候在同一表达式中有不同的数字类型。注意！每个程序员都曾被要求帮助某人追踪一个恼人的错误，结果却发现编译器一直都在警告这个错误。顶级程序员会修复代码以消除所有编译器警告。让编译器做这项工作比你自己做更容易。

## 整数

使用整数时要记住这些考虑事项：

**检查整数除法**。当你使用整数时，7/10 不等于 0.7。它通常等于 0，或负无穷大，或最接近的整数，或——你明白我的意思了。它等于什么因语言而异。这同样适用于中间结果。在现实世界中 10 * (7/10) = (10*7) / 10 = 7。在整数算术的世界里不是这样。10 * (7/10) 等于 0，因为整数除法 (7/10) 等于 0。解决这个问题的最简单方法是重新排序表达式，使除法最后执行：(10*7) / 10。

**检查整数溢出**。在做整数乘法或加法时，你需要知道最大可能的整数。最大可能的无符号整数通常是 2^32-1，有时是 2^16-1，即 65,535。当乘以两个产生比最大整数更大的数字时，问题就出现了。例如，如果你乘以 250 * 300，正确答案是 75,000。但是如果最大整数是 65,535，由于整数溢出，你得到的答案可能是 9,464（75,000 - 65,536 = 9,464）。表 12-1 显示了常见整数类型的范围。

**表 12-1. 不同类型整数的范围**

| 整数类型 | 范围 |
|---------|------|
| 有符号 8 位 | -128 到 127 |
| 无符号 8 位 | 0 到 255 |
| 有符号 16 位 | -32,768 到 32,767 |
| 无符号 16 位 | 0 到 65,535 |
| 有符号 32 位 | -2,147,483,648 到 2,147,483,647 |
| 无符号 32 位 | 0 到 4,294,967,295 |
| 有符号 64 位 | -9,223,372,036,854,775,808 到 9,223,372,036,854,775,807 |
| 无符号 64 位 | 0 到 18,446,744,073,709,551,615 |

防止整数溢出的最简单方法是考虑算术表达式中的每一项，并尝试想象每个项可以假设的最大值。例如，如果在整数表达式 `m = j * k` 中，j 的最大预期值是 200，k 的最大预期值是 25，那么 m 的最大预期值是 200 * 25 = 5,000。这在 32 位机器上是可以的，因为最大整数是 2,147,483,647。另一方面，如果 j 的最大预期值是 200,000，k 的最大预期值是 100,000，那么 m 的最大预期值是 200,000 * 100,000 = 20,000,000,000。这是不行的，因为 20,000,000,000 大于 2,147,483,647。在这种情况下，你必须使用 64 位整数或浮点数来容纳 m 的最大预期值。

还要考虑程序的未来扩展。如果 m 永远不会大于 5,000，那就太好了。但是如果你期望 m 在几年内稳步增长，要考虑到这一点。

**检查中间结果中的溢出**。等式末尾的数字不是唯一需要担心的数字。假设你有以下代码：

示例 12-1. 中间结果溢出的 Java 示例

```java
int termA = 1000000;
int termB = 1000000;
int product = termA * termB / 1000000;
System.out.println( "( " + termA + " * " + termB + " ) / 1000000 = " + product );
```

如果你认为 product 赋值与 (100,000*100,000) / 100,000 相同，你可能期望得到答案 100,000。但是代码必须在除以最后的 100,000 之前计算 100,000*100,000 的中间结果，这意味着它需要一个高达 1,000,000,000,000 的数字。猜猜结果是什么？

```
( 1000000 * 1000000 ) / 1000000 = -727
```

如果你的整数只能到 2,147,483,647，中间结果对于整数数据类型来说太大了。在这种情况下，应该是 1,000,000,000,000 的中间结果是 727,379,968，所以当你除以 100,000 时，你得到 -727，而不是 100,000。

你可以用处理整数溢出的相同方法处理中间结果中的溢出，通过切换到长整数或浮点类型。

## 浮点数

使用浮点数时的主要考虑是，许多分数十进制数字无法在使用数字计算机上可用的 1 和 0 来精确表示。像 1/3 或 1/7 这样不终止的小数通常只能精确到 7 或 15 位。在我版本的 Microsoft Visual Basic 中，1/3 的 32 位浮点表示等于 0.33333330。它精确到 7 位。这对大多数目的来说足够精确，但有时会精确到足以愚弄你。

以下是使用浮点数的一些具体指导原则：

**避免对数量级差异很大的数字进行加法和减法**。使用 32 位浮点变量，1,000,000.00 + 0.1 可能产生 1,000,000.00 的答案，因为 32 位没有给你足够的有效数字来涵盖 1,000,000 和 0.1 之间的范围。同样，5,000,000.02 - 5,000,000.01 可能等于 0.0。

交叉参考

关于描述解决这些问题方法的算法书籍，请参见数据素养中的"数据类型附加资源"。

解决方案？如果你必须添加包含如此大差异的一系列数字，先对数字排序，然后从最小值开始相加。同样，如果你需要求无穷级数的和，从最小的项开始——本质上，倒着求和。这不能消除舍入问题，但可以最小化它们。许多算法书籍都有处理这种情况的建议。

**避免相等比较**。应该相等的浮点数并不总是相等的。主要问题是通过同一数字的两种不同路径并不总是导致相同的数字。例如，0.1 加 10 次很少等于 1.0。下面的例子显示了两个应该相等但不相等的变量 `nominal` 和 `sum`。

**预计舍入错误**。舍入误差问题与数量级差异很大的数字问题没有什么不同。涉及相同的Issue，许多相同的技术有助于解决舍入问题。此外，以下是舍入问题的常见具体解决方案：

- **切换到具有更高精度的变量类型**。如果你使用单精度浮点，切换到双精度浮点，等等。
- **切换到二进制编码的十进制（BCD）变量**。BCD 方案通常更慢，占用更多存储空间，但它可以防止许多舍入错误。如果你的变量代表美元和美分或其他必须精确平衡的数量，这尤其有价值。

交叉参考

通常转换为 BCD 的性能影响很小。如果你担心性能影响，请参见代码优化方法总结。

- **从浮点切换到整数变量**。这是 BCD 变量的自制方法。你可能需要使用 64 位整数来获得你想要的精度。这种技术要求你自己跟踪数字的小数部分。假设你最初使用浮点数跟踪美元， 美分表示为美元的小数部分。这是一种处理美元和美分的正常方式。当你切换到整数时，你必须用整数跟踪美分，用 100 美分的倍数跟踪美元。换句话说，你将美元乘以 100，并将美分保持在变量的 0 到 99 范围内。乍一看这似乎很荒谬，但在速度和准确性方面，这是一种有效的解决方案。你可以通过创建一个隐藏整数表示并支持必要数字操作的 DollarsAndCents 类来使这些操作更容易。

- **检查语言和库对特定数据类型的支持**。一些语言（包括 Visual Basic）有专门支持对舍入误差敏感的数据的数据类型，如 Currency。如果你的语言有提供此类功能的内置数据类型，使用它！

## 字符和字符串

本节提供了一些使用字符串的技巧。第一个适用于所有语言中的字符串。

交叉参考

使用幻字符和幻字符串的问题与前面"数字概述"中讨论的幻数字问题类似。

**避免幻字符和幻字符串**。幻字符是字面字符（如 'A'），幻字符串是字面字符串（如 "Gigamatic Accounting Program"），它们出现在整个程序中。如果你在支持使用命名常量的语言中编程，用它们代替。否则，使用全局变量。避免使用字面字符串有几个原因：

- 对于常见出现的字符串，如程序名、命令名、报告标题等，你可能需要在某些时候更改字符串的内容。例如，"Gigamatic Accounting Program" 可能更改为"New and Improved! Gigamatic Accounting Program"。
- 国际市场变得越来越重要，翻译集中在字符串资源文件中的字符串比在程序中就地翻译更容易。
- 字符串字面量往往会占用大量空间。它们用于菜单、消息、帮助屏幕、输入表单等。如果你有太多，它们会失控并导致内存问题。字符串空间在许多环境中不是问题，但在嵌入式系统编程和其他存储空间珍贵的应用程序中，如果字符串与源代码相对独立，字符串空间问题的解决方案更容易实现。
- 字符和字符串字面量是神秘的。注释或命名常量澄清你的意图。在下一个例子中，0x1B 的含义不清楚。使用 ESCAPE 常量使含义更明显。

```cpp
if ( input_char == ESCAPE ) ...       // 2
```

**了解你的语言和环境如何支持 Unicode**。在一些语言中（如 Java），所有字符串都是 Unicode。在其他语言中（如 C 和 C++），处理 Unicode 字符串需要自己的函数集。在与标准和第三方库通信时，通常需要在 Unicode 和其他字符集之间进行转换。如果某些字符串不会是 Unicode（例如，在 C 或 C++ 中），请尽早决定是否使用 Unicode 字符集。如果你决定使用 Unicode 字符串，决定在哪里以及何时使用它们。

如果你知道你只需要支持单一字母语言，考虑使用 ISO 8859 字符集。对于只需要支持单一字母语言（如英语）且不需要支持多种语言或表意语言（如中文）的应用程序，ISO 8859 扩展 ASCII 类型标准是 Unicode 的良好替代方案。

如果你需要支持多种语言，使用 Unicode。Unicode 比 ISO 8859 或其他标准更全面地支持国际字符集。

**在字符串类型之间确定一致的转换策略**。如果你使用多种字符串类型，一种有助于保持字符串类型不同的常见方法是，在程序中将所有字符串保持在单一格式中，并在尽可能接近输入和输出操作的地方将字符串转换为其他格式。

## C 中的字符串

C++ 的标准模板库字符串类已经消除了传统 C 字符串的大部分问题。对于直接使用 C 字符串的程序员，这里有一些避免常见陷阱的方法：

**注意字符串指针和字符数组之间的区别**。字符串指针和字符数组的问题源于 C 处理字符串的方式。用两种方式注意它们之间的区别：

- **对任何包含涉及等号的字符串的表达式保持警惕**。C 中的字符串操作几乎总是用 `strcmp()`、`strcpy()`、`strlen()` 和相关例程完成。等号通常意味着某种指针错误。在 C 中，赋值不会将字符串字面量复制到字符串变量中。假设你有这样的语句：

```cpp
StringPtr = "Some Text String";
```

在这种情况下，"Some Text String" 是指向字面文本字符串的指针，赋值只是将指针 StringPtr 设置为指向文本字符串。赋值不会将内容复制到 StringPtr。

- **使用命名约定来指示变量是字符数组还是字符串指针**。一个常见的约定是使用 `ps` 作为前缀表示指向字符串的指针，使用 `ach` 作为前缀表示字符数组。虽然它们并不总是错误的，但你应该对同时包含 `ps` 和 `ach` 前缀的表达式持怀疑态度。

**声明 C 风格字符串长度为 CONSTANT+1**。在 C 和 C++ 中，C 风格字符串的越界错误很常见，因为很容易忘记长度为 n 的字符串需要 n + 1 个字节的存储空间，并且忘记为空终止符（字符串末尾设置为 0 的字节）留出空间。避免此类问题的有效方法是使用命名常量声明所有字符串。这种方法的关键是你每次都以相同的方式使用命名常量。将字符串声明为长度 CONSTANT+1，然后在代码的其余部分使用 CONSTANT 来引用字符串的长度。这里有一个例子：

示例 12-5. 良好字符串声明的 C 示例

```c
/* Declare the string to have length of "constant+1".
   Every other place in the program, "constant" rather
   than "constant+1" is used. */
char name[ NAME_LENGTH + 1 ] = { 0 }; /* string of length NAME_LENGTH */       // 1

...
/* Example 1: Set the string to all 'A's using the constant,
   NAME_LENGTH, as the number of 'A's that can be copied.
   Note that NAME_LENGTH rather than NAME_LENGTH + 1 is used. */
for ( i = 0; i < NAME_LENGTH; i++ )       // 2
   name[ i ] = 'A';
...

/* Example 2: Copy another string into the first string using
   the constant as the maximum length that can be copied. */
strncpy( name, some_other_name, NAME_LENGTH );       // 3
```

(1) 字符串被声明为长度 NAME_LENGTH + 1。

(2) 这里使用 NAME_LENGTH 对字符串进行操作……

(3) ……这里也使用。

如果你没有约定来处理这个问题，你有时会声明字符串长度为 NAME_LENGTH，并让它用 NAME_LENGTH-1 操作；有时你会声明字符串长度为 NAME_LENGTH+1，并让它用长度 NAME_LENGTH 操作。每次你使用字符串时，你都必须记住你用哪种方式声明它。

当你每次以相同方式使用字符串时，你不必记住你如何单独处理每个字符串，也消除了因忘记单个字符串的细节而导致的错误。有了约定，最大限度地减少了精神负担和编程错误。

**将字符串初始化为 null 以避免无限字符串**。C 通过找到空终止符（字符串末尾设置为 0 的字节）来确定字符串的结尾。无论你认为字符串有多长，C 都不会找到字符串的结尾，直到找到 0 字节。如果你忘记在字符串末尾放一个空字符，你的字符串操作可能不会按预期工作。

交叉参考

关于初始化数据的更多细节，请参见"初始化变量的指导原则"。

你可以通过两种方式避免无限字符串。首先，在声明字符数组时将它们初始化为 0：

示例 12-6. 良好字符数组声明的 C 示例

```c
char EventName[ MAX_NAME_LENGTH + 1 ] = { 0 };
```

其次，当你动态分配字符串时，通过使用 `calloc()` 而不是 `malloc()` 将它们初始化为 0。`calloc()` 分配内存并将其初始化为 0。`malloc()` 分配内存而不初始化它，所以当你使用 `malloc()` 分配的内存时，你只能碰运气。

**在 C 中使用字符数组而不是指针**。如果内存不是约束——而且通常不是——将所有字符串变量声明为字符数组。这有助于避免指针问题，当你做错事时编译器会给出更多警告。

交叉参考

关于数组的更多讨论，请阅读本章后面的"数组"。

**使用 `strncpy()` 而不是 `strcpy()` 来避免无限字符串**。C 中的字符串例程有安全版本和危险版本。`strcpy()` 和 `strcmp()` 等更危险的例程会一直执行直到遇到空终止符。它们的更安全搭档 `strncpy()` 和 `strncmp()` 接受最大长度参数，这样即使字符串永远持续，你的函数调用也不会。

## 布尔变量

逻辑或布尔变量很难被滥用，周到地使用它们会使你的程序更干净。

**使用布尔变量来文档化你的程序**。与其仅仅测试布尔表达式，你可以将表达式赋值给使测试的含义无可置疑的变量。例如，在下一个片段中，不清楚 if 测试的目的是检查完成、错误条件还是其他什么：

示例 12-7. 目的不明确的布尔测试的 Java 示例

```java
if ( ( elementIndex < 0 ) || ( MAX_ELEMENTS < elementIndex ) ||
   ( elementIndex == lastElementIndex )
   ) {
   ...
}
```

交叉参考

关于使用注释来文档化程序的详细信息，请参见第32章。

交叉参考

关于使用布尔函数来文档化程序示例，请参见布尔表达式中的"简化复杂表达式"。

在下一个片段中，布尔变量的使用使 if 测试的目的更清晰：

示例 12-8. 目的明确的布尔测试的 Java 示例

```java
finished = ( ( elementIndex < 0 ) || ( MAX_ELEMENTS < elementIndex ) );
repeatedEntry = ( elementIndex == lastElementIndex );
if ( finished || repeatedEntry ) {
 ...
}
```

**使用布尔变量来简化复杂测试**。通常，当你必须编写一个复杂测试时，需要多次尝试才能正确。当稍后尝试修改测试时，可能很难理解测试最初在做什么。逻辑变量可以简化测试。在上一个例子中，程序实际上是在测试两个条件：例程是否完成，以及是否正在处理重复条目。通过创建布尔变量 `finished` 和 `repeatedEntry`，你使 if 测试更简单：更容易阅读、更不容易出错、更容易修改。

这里是另一个复杂测试的例子：

**复杂测试的 Visual Basic 示例**

示例 12-9.

```vb
If ( ( document.AtEndOfStream() ) And ( Not inputError ) ) And _
   ( ( MIN_LINES <= lineCount ) And ( lineCount <= MAX_LINES ) ) And _
   ( Not ErrorProcessing() ) Then
   ' do something or other
   ...
End If
```

例子中的测试相当复杂但并不罕见。它给读者造成了沉重的精神负担。我猜你甚至不会尝试理解 if 测试，而是看着它说，"如果我真的需要，以后再想清楚。"注意那个想法，因为当人们阅读你的包含这种测试的代码时，他们做的正是完全相同的事情。

下面是重写代码并添加布尔变量以简化测试的示例：

示例 12-10. 简化测试的 Visual Basic 示例

```vb
allDataRead = ( document.AtEndOfStream() ) And ( Not inputError )
legalLineCount = ( MIN_LINES <= lineCount ) And ( lineCount <= MAX_LINES )
If ( allDataRead ) And ( legalLineCount ) And ( Not ErrorProcessing() ) Then       // 1
   ' do something or other
   ...
End If
```

(1) 这里是简化后的测试。

第二个版本更简单。我猜你会毫不费力地阅读 if 测试中的布尔表达式。

**必要时创建你自己的布尔类型**。一些语言（如 C++、Java 和 Visual Basic）有预定义的布尔类型。其他语言（如 C）则没有。在 C 等语言中，你可以定义你自己的布尔类型。在 C 中，你会这样做：

示例 12-11. 使用简单 typedef 定义 BOOLEAN 类型的 C 示例

```c
typedef int BOOLEAN;
```

或者你可以这样做，这提供了同时定义 true 和 false 的额外好处：

示例 12-12. 使用枚举定义布尔类型的 C 示例

```c
enum Boolean {
   True=1,
   False=(!True)
};
```

将变量声明为 BOOLEAN 而不是 int 使其预期用途更加明显，并使你的程序更加自文档化。

## 枚举类型

枚举类型是一种数据类型，允许用英语描述一类对象的每个成员。枚举类型在 C++ 和 Visual Basic 中可用，通常在你知道变量的所有可能值并想用文字表达它们时使用。以下是 Visual Basic 中枚举类型的示例：

示例 12-13. Visual Basic 枚举类型示例

```vb
Public Enum Color
   Color_Red
   Color_Green
   Color_Blue
End Enum

Public Enum Country
   Country_China
   Country_England
   Country_France
   Country_Germany
   Country_India
   Country_Japan
   Country_Usa
End Enum

Public Enum Output
   Output_Screen
   Output_Printer
   Output_File
End Enum
```

枚举类型是替代你明确说"1 代表红色，2 代表绿色，3 代表蓝色……"这一老套方案的有力替代方案。这种能力暗示了使用枚举类型的几个指导原则：

**使用枚举类型来提高可读性**。与其写：

```vb
if chosenColor = 1
```

不如写更易读的表达式：

```vb
if chosenColor = Color_Red
```

任何时候你看到一个数字字面量，都要问用枚举类型替换它是否有意义。

枚举类型对于定义例程参数特别有用。谁知道这个函数调用的参数是什么？

**使用枚举类型的例程调用的 C++ 示例**

示例 12-14.

```cpp
int result = RetrievePayrollData( data, true, false, false, true );
```

相比之下，这个函数调用的参数更容易理解：

示例 12-15. 使用枚举类型提高可读性的 C++ 函数调用示例

```cpp
int result = RetrievePayrollData(
   data,
   EmploymentStatus_CurrentEmployee,
   PayrollType_Salaried,
   SavingsPlan_NoDeduction,
   MedicalCoverage_IncludeDependents
);
```

**使用枚举类型来提高可靠性**。在少数语言（尤其是 Ada）中，枚举类型让编译器执行比使用整数值和常量更彻底的类型检查。对于命名常量，编译器没有办法知道唯一合法的值是 `Color_Red`、`Color_Green` 和 `Color_Blue`。编译器不会反对诸如 `color = Country_England` 或 `country = Output_Printer` 这样的语句。如果你使用枚举类型，将变量声明为 Color，编译器将只允许变量被赋值为 `Color_Red`、`Color_Green` 和 `Color_Blue`。

**使用枚举类型来提高可修改性**。枚举类型使你的代码易于修改。如果你发现"1 代表红色，2 代表绿色，3 代表蓝色"方案有缺陷，你必须遍历你的代码并更改所有的 1、2、3 等。如果你使用枚举类型，你可以通过简单地将元素放入类型定义并重新编译来继续向列表添加元素。

**使用枚举类型作为布尔变量的替代**。通常，布尔变量不够丰富，无法表达它需要的含义。例如，假设你有一个例程，如果成功完成任务则返回 true，否则返回 false。后来你可能会发现你实际上有两种 false。第一种意味着任务失败了，效果仅限于例程本身；第二种意味着任务失败了，并导致了需要传播到程序其余部分的致命错误。在这种情况下，具有值 `Status_Success`、`Status_Warning` 和 `Status_FatalError` 的枚举类型比具有值 true 和 false 的布尔变量更有用。这个方案可以轻松扩展以处理成功或失败的额外区别。

**检查无效值**。当你在 if 或 case 语句中测试枚举类型时，检查无效值。在 case 语句中使用 else 子句来捕获无效值：

示例 12-16. 在枚举类型中检查无效值的好 Visual Basic 示例

```vb
Select Case screenColor
  Case Color_Red
     ...
  Case Color_Blue
     ...
  Case Color_Green
     ...
  Case Else       // 1
     DisplayInternalError( False, "Internal Error 752: Invalid color." )
End Select
```

(1) 这里测试无效值。

**定义枚举的第一个和最后一个条目作为循环限制**。将枚举中的第一个和最后一个元素定义为 `Color_First`、`Color_Last`、`Country_First`、`Country_Last` 等，允许你编写循环遍历枚举元素的循环。你通过使用显式值来设置枚举类型，如下所示：

示例 12-17. 在枚举类型中设置第一个和最后一个值的好 Visual Basic 示例

```vb
Public Enum Country
   Country_First = 0
   Country_China = 0
   Country_England = 1
   Country_France = 2
   Country_Germany = 3
   Country_India = 4
   Country_Japan = 5
   Country_Usa = 6
   Country_Last = 6
End Enum
```

现在 `Country_First` 和 `Country_Last` 值可以用作循环限制：

示例 12-18. 遍历枚举中元素的好 Visual Basic 示例

```vb
' compute currency conversions from US currency to target currency
Dim usaCurrencyConversionRate( Country_Last ) As Single
Dim iCountry As Country
For iCountry = Country_First To Country_Last
  usaCurrencyConversionRate( iCountry ) = ConversionRate( Country_Usa, iCountry )
Next
```

**将枚举类型的第一个条目保留为无效**。当你声明枚举类型时，将第一个值保留为无效值。许多编译器将枚举类型的第一个元素赋值为 0。将映射到 0 的元素声明为无效有助于捕获未正确初始化的变量，因为它们更可能是 0 而不是任何其他无效值。

下面是使用该方法的 Country 声明的样子：

示例 12-19. 将枚举类型的第一个值声明为无效的 Visual Basic 示例

```vb
Public Enum Country
   Country_InvalidFirst = 0
   Country_First = 1
   Country_China = 1
   Country_England = 2
   Country_France = 3
   Country_Germany = 4
   Country_India = 5
   Country_Japan = 6
   Country_Usa = 7
   Country_Last = 7
End Enum
```

**在项目编码标准中精确定义如何使用 First 和 Last 元素，并一致地使用它们**。在枚举中使用 InvalidFirst、First 和 Last 元素可以使数组声明和循环更易读。但它有可能造成关于枚举中的有效条目是从 0 还是 1 开始，以及枚举的第一个和最后一个元素是否有效方面的混淆。如果使用这种技术，项目编码标准应该要求 InvalidFirst、First 和 Last 元素在所有枚举中一致使用，以减少错误。

**注意为枚举元素分配显式值的陷阱**。某些语言允许你为枚举中的元素分配特定值，如以下 C++ 示例所示：

示例 12-20. 为枚举显式赋值值的 C++ 示例

```cpp
enum Color {
   Color_InvalidFirst = 0,
   Color_First = 1,
   Color_Red = 1,
   Color_Green = 2,
   Color_Blue = 4,
   Color_Black = 8,
   Color_Last = 8
};
```

在这个例子中，如果你声明一个 Color 类型的循环索引并尝试遍历 Colors，你不仅会遍历有效值 1、2、4 和 8，还会遍历无效值 3、5、6 和 7。

如果你的语言没有枚举类型，你可以用全局变量或类来模拟它们。例如，你可以在 Java 中使用这些声明：

示例 12-21. 模拟枚举类型的 Java 示例

```java
// set up Country enumerated type
class Country {
   private Country() {}
   public static final Country China = new Country();
   public static final Country England = new Country();
   public static final Country France = new Country();
   public static final Country Germany = new Country();
   public static final Country India = new Country();
   public static final Country Japan = new Country();
}

// set up Output enumerated type
class Output {
   private Output() {}
   public static final Output Screen = new Output();
   public static final Output Printer = new Output();
   public static final Output File = new Output();
}
```

交叉参考

## 命名常量

命名常量类似于变量，只是你一旦赋值就不能更改常量的值。命名常量使你能够通过名称而不是数字来引用固定数量——例如，`MAXIMUM_EMPLOYEES` 而不是 1000。

使用命名常量是"参数化"程序的一种方式——将程序中可能更改的某个方面放入参数中，你可以在一个地方更改，而不必在整个程序中进行更改。如果你曾经声明一个数组为你认为它永远需要的大小，然后因为它不够大而用完空间，你就会欣赏命名常量的价值。

当数组大小改变时，你只更改用于声明数组的常量的定义。这种"单点控制"在使软件真正"软"方面走得很远：易于使用和更改。

**在数据声明中使用命名常量**。使用命名常量有助于数据声明以及需要知道正在处理的数据大小的语句中的程序可读性和可维护性。在下面的例子中，你使用 `LOCAL_NUMBER_LENGTH` 来描述员工电话号码的长度，而不是字面量 7。

示例 12-22. 在数据声明中使用命名常量的好 Visual Basic 示例

```vb
Const AREA_CODE_LENGTH = 3
Const LOCAL_NUMBER_LENGTH = 7       // 1
...
Type PHONE_NUMBER
   areaCode( AREA_CODE_LENGTH ) As String
   localNumber( LOCAL_NUMBER_LENGTH ) As String       // 2
End Type
...

' make sure all characters in phone number are digits
For iDigit = 1 To LOCAL_NUMBER_LENGTH       // 3
  If ( phoneNumber.localNumber( iDigit ) < "0" ) Or _
     ( "9" < phoneNumber.localNumber( iDigit ) ) Then
     ' do some error processing
     ...
```

(1) `LOCAL_NUMBER_LENGTH` 在这里声明为常量。

(2) 它在这里使用。

(3) 它也在这里使用。

这是一个简单的例子，但你可能可以想象一个程序，其中电话号码长度的信息需要在很多地方使用。

在创建程序的时候，所有员工都住在一个国家，所以你只需要七个数字的电话号码。随着公司扩展和在不同国家建立分支机构，你需要更长的电话号码。如果你已经参数化了，你只需在一个地方进行更改：在命名常量 `LOCAL_NUMBER_LENGTH` 的定义中。

正如你可能期望的那样，使用命名常量已被证明对程序维护有很大帮助。一般来说，任何将可能更改的事物集中控制的技术都是减少维护工作的好技术（Glass 1991）。

延伸阅读

关于单点控制价值的更多细节，请参见《Software Conflict》第 57-60 页（Glass 1991）。

**避免字面量，即使是"安全"的字面量**。在下面的循环中，你认为 12 代表什么？

示例 12-23. 含义不明确的 Visual Basic 代码示例

```vb

```

由于代码的特殊性质，代码似乎可能在循环遍历一年中的 12 个月。但你确定吗？你会拿你的 Monty Python 收藏打赌吗？

在这种情况下，你不需要使用命名常量来支持未来的灵活性：一年内月数的变化不太可能在短期内发生。但如果代码的编写方式对其目的留下任何疑问，用一个命名良好的常量来澄清它，就像这样：

示例 12-24. 更清晰的 Visual Basic 代码示例

```vb
For i = 1 To NUM_MONTHS_IN_YEAR

```

这更好，但为了完成这个例子，循环索引也应该命名为更具信息量的名称：

示例 12-25. 更清晰的 Visual Basic 代码示例

```vb
For month = 1 To NUM_MONTHS_IN_YEAR

Next
```

这个例子看起来相当好，但我们可以通过使用枚举类型再推进一步：

示例 12-26. 非常清晰的 Visual Basic 代码示例

```vb
For month = Month_January To Month_December
```

**一致地使用命名常量**。在一个地方使用命名常量而在另一个地方使用字面量来表示同一实体是危险的。有些编程实践会招致错误；这就像是打一个 800 电话，让错误送到你家门口。如果命名常量的值需要更改，你会更改它，并认为你已经做了所有必要的更改。你会忽略硬编码的字面量，你的程序会出现神秘缺陷，修复它们比拿起电话大喊大叫要困难得多。

## 数组

数组是最简单和最常见的结构化数据类型。在某些语言中，数组是唯一的结构化数据类型。数组包含一组类型相同且可通过数组索引直接访问的项目。以下是使用数组的一些技巧。

**确保所有数组索引都在数组的边界内**。在某种程度上，所有数组问题都是由数组元素可以被随机访问这一事实引起的。最常见的问题出现在程序尝试访问越界数组元素时。在某些语言中，这会产生错误；在其他语言中，它只是产生奇怪和意想不到的结果。

**考虑使用容器而不是数组，或者将数组视为顺序结构**。一些计算机科学领域最聪明的人曾建议数组永远不应该被随机访问，而只能顺序访问（Mills and Linger 1986）。他们的论点是，数组中的随机访问类似于程序中的随机 goto：这种访问往往不受约束、容易出错、难以证明正确。他们建议使用集合、栈和队列，这些数据结构的元素是顺序访问的，而不是使用数组。

在 Mills 和 Linger 的一个小实验中，他们发现以这种方式创建的设计导致更少的变量和更少的变量引用。设计相对高效，并产生了高度可靠软件。

在自动选择数组之前，考虑使用可以顺序访问的容器类——集合、栈、队列等——作为替代。

**检查数组的端点**。正如考虑循环结构的端点有帮助一样，检查数组的端点可以捕获很多错误。问问自己代码是否正确访问数组的第一个元素，或者错误地访问了第一个元素之前或之后的元素。最后一个元素呢？代码会犯越界错误吗？最后，问问自己代码是否正确访问了数组的中间元素。

交叉参考

使用数组和循环的问题是相似的和相关的。关于循环的详细信息，请参见第16章。

**如果数组是多维的，确保其下标以正确的顺序使用**。很容易在想说 `Array[ j ][ i ]` 的时候说成 `Array[ i ][ j ]`，所以要花时间仔细检查索引顺序是否正确。在角色不能立即清楚的情况下，考虑使用比 `i` 和 `j` 更有意义的名称。

**注意索引串扰**。如果你使用嵌套循环，很容易在想说 `Array[ i ]` 的时候写 `Array[ j ]`。切换循环索引称为"索引串扰"。检查这个问题。更重要的是，使用比 `i` 和 `j` 更有意义的索引名称，使串扰错误从一开始就难以发生。

**在 C 中使用 ARRAY_LENGTH() 宏来处理数组**。你可以通过定义一个 `ARRAY_LENGTH()` 宏来为你的数组工作增加额外的灵活性：

示例 12-27. 定义 ARRAY_LENGTH() 宏的 C 示例

```c
#define ARRAY_LENGTH( x ) (sizeof(x)/sizeof(x[0]))
```

当你对数组使用操作时，不要使用命名常量作为数组大小的上限，而要使用 `ARRAY_LENGTH()` 宏。这里有一个例子：

示例 12-28. 使用 ARRAY_LENGTH() 宏进行数组操作的 C 示例

```c
ConsistencyRatios[] =
   { 0.0, 0.0, 0.58, 0.90, 1.12,
   1.24, 1.32, 1.41, 1.45, 1.49,
   1.51, 1.48, 1.56, 1.57, 1.59 };
   ...
for ( ratioIdx = 0; ratioIdx < ARRAY_LENGTH( ConsistencyRatios ); ratioIdx++ );       // 1
   ...
```

(1) 这里使用了宏。

这种技术对于像这个例子中的无维度数组特别有用。如果你添加或减去条目，你不必记住更改描述数组大小的命名常量。当然，这种技术也适用于有维度数组，但如果你使用这种方法，你并不总是需要为数组定义设置额外的命名常量。

## 创建你自己的类型（类型别名）

程序员定义的数据类型是语言可以给你的最强大功能之一，可以澄清你对程序的理解。它们保护你的程序免受不可预见的变化，使程序更易于阅读——所有这些都不需要你设计、构建或测试新类。如果你使用 C、C++ 或其他允许用户定义类型的语言，充分利用它们！

交叉参考

在许多情况下，创建类比创建简单数据类型更好。详细信息，请参见第6章。

为了理解类型创建的力量，假设你正在编写一个程序，将 x、y、z 系统中的坐标转换为纬度、经度和海拔。你认为可能需要双精度浮点数，但宁愿先用单精度浮点数编写程序，直到你绝对确定。你可以使用 typedef 语句在 C 或 C++ 中（或另一种语言中的等效语句）专门为坐标创建一个新类型。以下是你如何在 C++ 中设置类型定义：

示例 12-29. 在 C++ 中创建类型的示例

```cpp
typedef float Coordinate; // for coordinate variables
```

此类型定义声明了一个新类型 `Coordinate`，它在功能上与类型 `float` 相同。要使用新类型，就像使用预定义类型（如 float）一样声明变量。这里有一个例子：

示例 12-30. 在 C++ 中使用你创建的类型示例

```cpp
Routine1( ... ) {
   Coordinate latitude;      // latitude in degrees
   Coordinate longitude;     // longitude in degrees
   Coordinate elevation;     // elevation in meters from earth center
   ...
}
...
Routine2( ... ) {
   Coordinate x;   // x coordinate in meters
   Coordinate y;   // y coordinate in meters
   Coordinate z;   // z coordinate in meters
   ...
}
```

在这个代码中，变量 `latitude`、`longitude`、`elevation`、`x`、`y` 和 `z` 都声明为 `Coordinate` 类型。

现在假设程序改变了，你发现你最终需要对坐标使用双精度变量。因为你专门为坐标数据定义了一个类型，你只需更改类型定义。而且你只需在一个地方更改：在 typedef 语句中。以下是更改后的类型定义：

示例 12-31. 在 C++ 中更改类型定义示例

```cpp
typedef double Coordinate; // for coordinate variables       // 1
```

(1) 原来的 float 已更改为 double。

第二个例子是用 Pascal。假设你正在创建一个工资系统，其中员工姓名最多 30 个字符长。你的用户告诉你，永远不会有人名字超过 30 个字符。你会在整个程序中硬编码数字 30 吗？如果你这样做了，你对用户的信任比我对我的用户的信任多得多！更好的方法是为员工姓名定义一个类型：

示例 12-32. 在 Pascal 中为员工姓名创建类型的示例

```pascal
Type
   employeeName = array[ 1..30 ] of char;
```

当涉及字符串或数组时，通常明智的是定义一个表示字符串或数组长度的命名常量，然后在类型定义中使用命名常量。你会在程序中找到许多使用该常量的地方——这只是你将使用它的第一个地方。它看起来是这样的：

示例 12-33. 在 Pascal 中更好的类型创建示例

```pascal
Const
   NAME_LENGTH = 30;       // 1
   ...
Type
   employeeName = array[ 1..NAME_LENGTH ] of char;       // 2
```

(1) 这里声明了命名常量。

(2) 这里使用了命名常量。

一个更强大的例子会将创建你自己类型的想法与信息隐藏的想法结合起来。在某些情况下，你想隐藏的信息是关于数据类型的信息。

C++ 中的坐标示例大约一半实现了信息隐藏。如果你总是使用 `Coordinate` 而不是 `float` 或 `double`，你有效地隐藏了数据的类型。在 C++ 中，这就是语言为你做的全部信息隐藏。对于其余的，你或你的代码的后续用户必须自律，不去查找 `Coordinate` 的定义。C++ 给你比喻性的而不是字面的信息隐藏能力。

其他语言（如 Ada）走得更远，支持字面上的信息隐藏。以下是坐标代码片段在声明它的 Ada 包中的样子：

示例 12-34. 在包内隐藏类型细节的 Ada 示例

```ada
package Transformation is
   type Coordinate is private;       // 1
   ...

(1) 这个语句声明 Coordinate 对包是私有的。
```

以下是在另一个使用它的包中 `Coordinate` 的样子：

示例 12-35. 在 Ada 中从另一个包使用类型的示例

```ada
with Transformation;
...
procedure Routine1(...) ...
   latitude: Coordinate;
   longitude: Coordinate;
begin
   -- statements using latitude and longitude
   ...
end Routine1;
```

注意 `Coordinate` 类型在包规范中声明为 private。这意味着知道 `Coordinate` 类型定义的程序唯一部分是 Transformation 包的私有部分。在有多个程序员小组的开发环境中，你只能分发包规范，这使得在另一个包上工作的程序员更难查找 `Coordinate` 的底层类型。这些信息实际上是隐藏的。像 C++ 这样要求你在头文件中分发 `Coordinate` 定义的语言会破坏真正的信息隐藏。

这些例子说明了你创建自己类型的几个原因：

- **使修改更容易**。创建新类型工作量很小，而且给你很大的灵活性。
- **避免过多的信息分布**。硬类型将数据类型细节分散在程序中，而不是将它们集中在一个地方。这是"好的类接口"章节中讨论的信息隐藏原则的集中化的一个例子。
- **提高可靠性**。在 Ada 中，你可以定义类型如 `type Age is range 0..99`。然后编译器生成运行时检查，以验证任何 `Age` 类型的变量总是在 0..99 范围内。
- **弥补语言的弱点**。如果你的语言没有你想要的预定义类型，你可以自己创建。例如，C 没有布尔或逻辑类型。这种缺陷很容易通过自己创建类型来弥补：

```c
typedef int Boolean;
```

## 为什么创建自己类型的例子用 Pascal 和 Ada？

Pascal 和 Ada 已经像恐龙一样灭绝了总的来说取代它们的语言更可用。然而，在简单类型定义领域，我认为 C++、Java 和 Visual Basic 代表了三步前进一步后退的情况。像

```ada
currentTemperature: INTEGER range 0..212;
```

这样的 Ada 声明包含了像

```c
int temperature;
```

这样的语句所没有的重要语义信息。进一步地，像

```ada
type Temperature is range 0..212;
...
currentTemperature: Temperature;
```

这样的类型声明允许编译器确保 `currentTemperature` 只能赋给具有 `Temperature` 类型的其他变量，而且提供这种额外的安全边际不需要多少额外的编码。

当然，程序员可以创建一个 Temperature 类来强制执行 Ada 语言自动强制的相同语义，但从一行代码创建简单数据类型到创建类的步骤是一个很大的步骤。在许多情况下，程序员会创建简单类型，但不会迈出创建类的额外努力。

## 创建你自己类型的指导原则

交叉参考

在每种情况下，考虑创建类是否比创建简单数据类型更好。详细信息，请参见第6章。

创建你自己的"用户定义"类型时，记住这些指导原则：

**创建功能导向的类型名**。避免使用引用底层计算机数据类型的那种类型名。使用引用新类型代表的现实世界问题的部分的类型名。在前面的例子中，定义创建了坐标和姓名——现实世界实体——的良好命名的类型。同样，你可以为货币、支付代码、年龄等创建类型——现实世界问题的方面。

**警惕创建引用预定义类型的类型名**。像 `BigInteger` 或 `LongString` 这样的类型名指的是计算机数据而不是现实世界的问题。创建你自己类型的最大优势是它在程序和实现语言之间提供了一层绝缘。引用底层编程语言类型的类型名会在这种绝缘上戳洞。它们比使用预定义类型给你的优势不多。另一方面，面向问题的名称为你购买了轻松的修改性和自文档化的数据声明。

**避免使用预定义类型**。如果类型有任何更改的可能性，除了在 typedef 或类型定义中外，避免使用预定义类型。创建功能导向的新类型很容易，而更改使用硬编码类型的程序中的数据却很困难。此外，使用功能导向的类型声明部分地文档化了用它们声明的变量。像 `Coordinate x` 这样的声明比像 `float x` 这样的声明告诉你更多关于 `x` 的信息。尽可能多地使用你自己的类型。

**不要重新定义预定义类型**。更改标准类型的定义会造成混淆。例如，如果你的语言有一个预定义类型 Integer，不要创建你自己的名为 Integer 的类型。你代码的读者可能会忘记你已经重新定义了该类型，并假设他们看到的 Integer 是他们习惯看到的 Integer。

**为可移植性定义替代类型**。与不更改标准类型定义的建议相反，你可能想为标准类型定义替代类型，这样在不同的硬件平台上，你可以使变量代表完全相同的实体。例如，你可以定义一个类型 INT32 并用它代替 int，或定义一个类型 LONG64 代替 long。最初，两种类型之间的唯一区别是它们的大小写。但当你将程序移动到新的硬件平台时，你可以重新定义大写版本，以便它们可以匹配原始硬件上的数据类型。

**确保不要定义容易被误认为预定义类型的类型**。可以定义 INT 而不是 INT32，但最好在你定义的类型和语言提供的类型之间创建清晰的区别。

**考虑创建类而不是使用 typedef**。简单的 typedef 可以很好地隐藏变量底层类型的信息。然而，在某些情况下，你可能需要创建类来实现你将获得的额外灵活性和控制。详细信息，请参见第6章。

- [ ] 整数表达式是否避免整数溢出问题？

- [ ] 代码是否避免比较浮点数是否相等？

- [ ] 代码是否避免使用幻字符和幻字符串？

- 使用特定数据类型意味着记住每种类型的许多单独规则。使用本章的检查表确保你已经考虑了常见问题。
