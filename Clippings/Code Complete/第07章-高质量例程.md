在深入高质量例程的细节之前，厘清两个基本术语会很有用。什么是"例程"？例程是调用用于单一目的的单个方法或过程。例如C++中的函数、Java中的方法、Microsoft Visual Basic中的函数或子过程。在某些用途中，C和C++中的宏也可以被认为是例程。你可以将创建高质量例程的许多技术应用到这些变体上。

什么是高质量例程？这是一个更难回答的问题。也许最简单的答案是展示高质量例程不是什么。以下是一个低质量例程的示例：

**示例7-1。** C++ 低质量例程示例

```cpp
void HandleStuff( CORP_DATA & inputRec, int crntQtr, EMP_DATA empRec,
   double & estimRevenue, double ytdRevenue, int screenX, int screenY,
   COLOR_TYPE & newColor, COLOR_TYPE & prevColor, StatusType & status,
   int expenseType )
{
   int i;
   for ( i = 0; i < 100; i++ ) {
      inputRec.revenue[i] = 0;
      inputRec.expense[i] = corpExpense[ crntQtr ][ i ];
   }
   UpdateCorpDatabase( empRec );
   estimRevenue = ytdRevenue * 4.0 / (double) crntQtr;
   newColor = prevColor;
   status = SUCCESS;
   if ( expenseType == 1 ) {
      for ( i = 0; i < 12; i++ )
         profit[i] = revenue[i] - expense.type1[i];
   }
   else if ( expenseType == 2 ) {
      profit[i] = revenue[i] - expense.type2[i];
   }
   else if ( expenseType == 3 )
      profit[i] = revenue[i] - expense.type3[i];
}
```

这个例程有什么问题？这里有一个提示：你应该能找到至少10个不同的问题。一旦你想出了自己的列表，看看下面的列表：

1. 例程名称不好。HandleStuff()没有告诉你例程做什么。
2. 例程没有文档。（文档主题超出了单个例程的边界，在第32章讨论。）
3. 例程布局不好。代码在页面上的物理组织几乎没有给出其逻辑组织的提示。布局策略使用得很随意，在例程的不同部分使用不同的风格。比较expenseType == 2和expenseType == 3处的风格。（布局在第31章讨论。）
4. 例程的输入变量inputRec被更改了。如果它是输入变量，其值不应该被修改（在C++中它应该声明为const）。如果变量的值应该被修改，变量不应该被称为inputRec。
5. 例程读写全局变量——它从corpExpense读取并写入profit。它应该通过比读写全局变量更直接的方式与其他例程通信。
6. 例程没有单一目的。它初始化一些变量，写入数据库，进行一些计算——这些似乎没有任何方式相互关联。例程应该有单一、明确的目的。
7. 例程不防御坏数据。如果crntQtr等于0，表达式ytdRevenue * 4.0 / (double) crntQtr会导致除以零错误。
8. 例程使用几个幻数：100、4.0、12、2和3。幻数在"一般数字"中讨论。
9. 一些例程参数未使用：screenX和screenY在例程内没有被引用。
10. 一个例程参数传递不正确：prevColor被标记为引用参数（&），即使它没有在例程内被赋值。
11. 例程有太多参数。可理解的参数数量上限约为7个；这个例程有11个。参数以一种难以阅读的方式排列，大多数人不会尝试仔细检查甚至计算它们。
12. 例程的参数顺序不好且没有文档。（参数排序在本章讨论。文档在第32章讨论。）

## 创建例程的有效理由

以下是创建例程的有效理由列表。这些理由有些重叠，它们不是为了形成一个正交集合。

**降低复杂性。** 创建例程最重要的理由是降低程序的复杂性。创建一个例程来隐藏信息，这样你就不必考虑它。当然，你在编写例程时需要考虑它。但编写之后，你应该能够忘记细节，在不了解其内部工作原理的情况下使用例程。创建例程的其他理由——最小化代码大小、提高可维护性和提高正确性——也是好理由，但没有例程的抽象能力，复杂程序将不可能在智力上管理。

表明例程需要从另一个例程中分解出来的一个迹象是内部循环或条件语句的深层嵌套。通过将嵌套部分拉出并放入自己的例程来降低包含例程的复杂性。

**引入中间的可理解抽象。** 将一段代码放入命名良好的例程中是记录其目的的最好方式之一。与其阅读一系列语句如：

```cpp
if ( node <> NULL ) then
   while ( node.next <> NULL ) do
      node = node.next
      leafName = node.name
   end while
else
   leafName = ""
end if
```

不如阅读这样的语句：

```cpp
leafName = GetLeafName( node )
```

新例程很短，几乎所有需要的文档就是一个好名字。这个名字引入了比原始8行代码更高的抽象层次，使代码更易读、更容易理解，并降低了原本包含代码的例程内的复杂性。

**避免重复代码。** 创建例程最流行的理由无疑是为了避免重复代码。事实上，在两个例程中创建类似代码意味着分解存在错误。将重复代码从两个例程中拉出，将通用代码放入基类，然后将两个专用例程移动到派生类。或者，你可以将公共代码迁移到自己的例程中，然后让两者调用放入新例程的部分。将代码放在一个地方可以节省将被重复代码使用的空间。修改会更容易，因为你只需要在一个位置修改代码。代码会更可靠，因为你只需要检查一个地方以确保代码正确。修改会更可靠，因为你将避免在错误假设已做出相同修改的情况下进行连续且略有不同的修改。

**支持子类化。** 覆盖简短、良好分解的例程比覆盖长且分解不良的例程需要更少的新代码。如果你保持可覆盖例程简单，也会减少子类实现中的错误机会。

**隐藏顺序。** 隐藏事件发生的处理顺序是个好主意。例如，如果程序通常先从用户获取数据然后从文件获取辅助数据，获取用户数据的例程和获取文件数据的例程都不应该依赖于另一个例程先执行。顺序的另一个例子可能是当你有两行代码读取栈顶部并递减stackTop变量时。将那两行代码放入PopStack()例程以隐藏必须执行两个操作的顺序的假设。隐藏该假设比将其烘焙到系统一端到另一端的代码中更好。

**隐藏指针操作。** 指针操作往往难以阅读且容易出错。通过将它们隔离在例程中，你可以专注于操作的目的而不是指针操作的机制。同样，如果操作只在一个地方完成，你可以更确信代码是正确的。如果你找到了比指针更好的数据类型，你可以更改程序而不会让使用指针的代码受到创伤。

**提高可移植性。** 使用例程隔离不可移植的能力，明确识别和隔离未来的可移植性工作。不可移植的能力包括非标准语言特性、硬件依赖、操作系统依赖等。

**简化复杂的布尔测试。** 详细理解复杂布尔测试对于理解程序流很少是必要的。将这样的测试放入函数中会使代码更易读，因为（1）测试的细节被移开了，（2）描述性函数名总结了测试的目的。

给测试自己的函数强调了它的重要性。它鼓励额外努力使测试内部的细节在其函数内可读。结果是代码的主要流程和测试本身都变得更清晰。简化布尔测试是降低复杂性的一个例子，如前所述。

**提高性能。** 你可以优化一个地方的代码而不是在多个地方优化。将代码放在一个地方将更容易进行性能分析以发现低效。将代码集中到例程中意味着优化使使用该例程的所有代码受益，无论它直接或间接使用。将代码放在一个地方使得用更高效的算法或用更快、更高效的语言重新编码例程变得实用。

为了确保所有例程都很小？不。有这么多将代码放入例程的好理由，这个理由是不必要的。事实上，一些工作最好在单个大型例程中完成。（例程的最佳长度在"例程可以有多长？"中讨论。）

### 看起来太简单而不值得放入例程的操作

创建有效例程的最强心理障碍之一是不愿为简单目的创建简单例程。为两三行代码构建整个例程似乎是小题大做，但经验表明一个好的小例程可以多么有帮助。

小例程提供了几个优势。一个是它们提高可读性。我曾经在程序中约十几个地方有这一行代码：

**示例7-2。** 伪代码计算示例

```cpp
points = deviceUnits * ( POINTS_PER_INCH / DeviceUnitsPerInch() )
```

这不是你会读到的最复杂的代码。大多数人最终会弄清楚它将设备单位的测量值转换为点的测量值。他们会看到十几行都做同样的事情。然而，这可能更清楚，所以我创建了一个命名良好的例程在一个地方进行转换：

**示例7-3。** 伪代码转换为函数的计算示例

```cpp
Function DeviceUnitsToPoints ( deviceUnits Integer ): Integer
   DeviceUnitsToPoints = deviceUnits *
      ( POINTS_PER_INCH / DeviceUnitsPerInch() )
End Function
```

当例程被替换内联代码时，十几行代码都看起来或多或少像这一行：

**示例7-4。** 伪代码函数调用计算示例

```cpp
Function
points = DeviceUnitsToPoints( deviceUnits )
```

这一行更易读——甚至接近自文档化。

这个示例暗示了将小操作放入函数的另一个原因：小操作往往会变成更大的操作。当我编写例程时我不知道，但在某些条件下，当某些设备处于活动状态时，DeviceUnitsPerInch()返回0。这意味着我必须处理除以零，这需要三行额外的代码：

**伪代码扩展维护的计算示例**

```cpp
Function DeviceUnitsToPoints( deviceUnits: Integer ) Integer;
   if ( DeviceUnitsPerInch() <> 0 )
      DeviceUnitsToPoints = deviceUnits *
         ( POINTS_PER_INCH / DeviceUnitsPerInch() )
   else
      DeviceUnitsToPoints = 0
   end if
End Function
```

如果原始代码行仍然在十几个地方，测试将被重复12次，总共36行新代码。一个简单的例程将36行新代码减少到3行。

## 例程级别的设计

内聚性概念是在Wayne Stevens、Glenford Myers和Larry Constantine（1974）的论文中引入的。其他更现代的概念，包括抽象和封装，往往在类级别产生更多洞察（实际上，在类级别上已在很大程度上取代了内聚），但内聚作为个人例程级别的主力设计启发法仍然活跃且良好。

对于例程，内聚性指的是例程中操作的关联程度。一些程序员更喜欢"强度"这个术语：例程中的操作有多强关联？像Cosine()这样的函数是完美内聚的，因为整个例程专注于执行一个函数。像CosineAndTan()这样的函数内聚性较低，因为它试图做多件事。目标是让每个例程做好一件事并且不做其他任何事。

> **交叉参考**
>
> 有关内聚性的一般讨论，请参见"设计构建模块：启发法"中的"追求强内聚"。

收益是更高的可靠性。一项对450个例程的研究发现，50%的高内聚例程没有错误，而只有18%的低内聚例程没有错误（Card, Church, and Agresti 1986）。另一项对不同450个例程的研究（这只是巧合）发现，耦合-内聚比率最高的例程错误是耦合-内聚比率最低的例程的7倍，修复成本是其20倍（Selby and Basili 1991）。

关于内聚性的讨论通常指几种内聚层次。理解概念比记住特定术语更重要。将概念用作辅助来思考如何使例程尽可能内聚。

**功能内聚是最强和最好的内聚**，当例程执行一个且仅一个操作时发生。高度内聚例程的示例包括sin()、GetCustomerName()、EraseFile()、CalculateLoanPayment()和AgeFromBirthdate()。当然，它们内聚性的这种评估假设例程做了其名称所说的事情——如果它们做其他任何事情，它们的内聚性就会降低，命名也不好。

通常认为以下几种其他内聚性不如理想：

**顺序内聚**存在于例程包含必须以特定顺序执行的操作、步骤之间共享数据且组合在一起不构成完整功能时。

顺序内聚的一个例子是，给定出生日期，计算员工年龄和退休时间的例程。如果例程计算年龄然后使用该结果计算员工的退休时间，它具有顺序内聚性。如果例程计算年龄然后在完全独立的计算中使用恰好相同的出生日期数据计算退休时间，它只有通信内聚性。

如何使例程具有功能内聚？你将创建单独的例程来根据出生日期计算员工年龄，并根据出生日期计算退休时间。退休时间例程可以调用年龄例程。它们都将具有功能内聚性。其他例程可以调用任一或两个例程。

**通信内聚**发生在例程中的操作使用相同数据且以其他方式不相关时。如果一个例程打印汇总报告然后重新初始化传入的汇总数据，该例程具有通信内聚性：两个操作仅因为它们使用相同数据这一事实而相关。

为了给这个例程更好的内聚性，汇总数据应该在创建它的地方附近重新初始化，而不应该在报告打印例程中。将操作拆分为单独的例程。第一个打印报告。第二个在创建或修改数据的代码附近重新初始化数据。从原本调用通信内聚例程的更高级例程调用两个例程。

**时间内聚**发生在操作被组合到例程中是因为它们都在同一时间完成时。典型示例是Startup()、CompleteNewEmployee()和Shutdown()。一些程序员认为时间内聚是不可接受的，因为它有时与糟糕的编程实践相关，如在Startup()例程中有一堆代码。

为避免这个问题，将时间例程视为其他事件的组织者。例如，Startup()例程可能读取配置文件、初始化临时文件、设置内存管理器并显示初始屏幕。为使其最有效，让时间内聚例程调用其他例程来执行特定活动，而不是直接自己执行操作。这样，例程的目的是协调活动而不是直接执行它们就很清楚了。

这个示例提出了在正确层次上命名描述例程的问题。你可以决定将例程命名为ReadConfigFileInitScratchFileEtc()，这将暗示例程只有偶然内聚。然而，如果你将其命名为Startup()，将很清楚它有单一目的，并且很清楚它具有功能内聚性。

**其余内聚类型通常是不可接受的。** 它们导致代码组织不良、难以调试和难以修改。如果例程具有不良内聚性，最好投入精力重写以获得更好的内聚性，而不是投资于精确诊断问题。然而，知道要避免什么是有用的，所以这里是不可接受的几种内聚类型：

**过程内聚**发生在例程中的操作以指定顺序执行时。示例是获取员工姓名、然后获取地址、然后获取电话号码的例程。这些操作的顺序重要只是因为它与输入屏幕上用户被询问数据的顺序匹配。另一个例程获取其余的员工数据。该例程具有过程内聚性，因为它将一组操作以指定顺序放置，并且操作不需要出于任何其他原因组合在一起。

为获得更好的内聚性，将单独的操作放入自己的例程中。确保调用例程有单一、完整的工作：GetEmployee()而不是GetFirstPartOfEmployeeData()。你可能需要修改获取其余数据的例程。通常在任何一个例程实现功能内聚之前你需要修改两个或更多原始例程。

**逻辑内聚**发生在几个操作被塞进同一个例程，并且通过传入的控制标志选择其中一个操作时。它被称为逻辑内聚是因为例程的控制流或"逻辑"是唯一将操作联系在一起的东西——它们都在一个大if语句或case语句中。这不是因为操作在任何其他意义上逻辑相关。考虑到定义逻辑内聚的属性是操作不相关，更好的名称可能是"不合逻辑内聚"。

一个示例是InputAll()例程，它根据传递给例程的标志输入客户名称、员工考勤卡信息或库存数据。其他示例是ComputeAll()、EditAll()、PrintAll()和SaveAll()。这种例程的主要问题是你不应该传入一个标志来控制另一个例程的处理。与其让一个例程根据传入的标志执行三个不同操作之一，不如有三个例程，每个执行一个不同的操作。如果操作使用一些相同的代码或共享数据，代码应该移动到较低级别的例程中，例程应该打包到一个类中。

但是，通常可以创建一个逻辑内聚例程，如果其代码仅由一系列if或case语句和对其他例程的调用组成。在这种情况下，如果例程的唯一功能是分派命令而自己不执行任何处理，那通常是一个好的设计。这种例程的技术术语是"事件处理器"。事件处理器常用于交互式环境，如Apple Macintosh、Microsoft Windows和其他GUI环境。

> **交叉参考**
>
> 虽然例程可能有更好的内聚性，但更高级别的设计问题是系统是否应该使用case语句而不是多态。有关此问题的更多信息，请参见"特定重构"中的"用多态替换条件语句（特别是重复的case语句）"。

**偶然内聚**发生在例程中的操作彼此之间没有明显关系时。其他好名称是"无内聚"或"混乱内聚"。本章开头的高质量C++例程具有偶然内聚性。将偶然内聚转换为任何更好的内聚几乎总是很困难——你通常需要进行更深入的重新设计和重新实现。

这些术语都不是神奇或神圣的。学习思想而不是术语。几乎总是可以编写具有功能内聚的例程，所以将注意力集中在功能内聚上以获得最大收益。

## 良好的例程命名

例程的良好名称清楚地描述例程执行的所有操作。以下是创建有效例程名称的指导原则：

> **交叉参考**
>
> 有关命名变量的详细信息，请参见第11章。

**描述例程的所有操作。** 在例程名称中，描述所有输出和副作用。如果一个例程计算报告总计并打开输出文件，ComputeReportTotals()不是该例程的充分名称。ComputeReportTotalsAndOpenOutputFile()是充分名称但太长且愚蠢。如果你有带副作用的例程，你会有许多长且愚蠢的名称。治疗方法不是使用描述性较少的例程名称；治疗方法是以直接方式而不是通过副作用导致事情发生。

**避免无意义、模糊或摇摆不定的动词。** 有些动词是弹性的，延伸到涵盖几乎任何含义。像HandleCalculation()、PerformServices()、OutputUser()、ProcessInput()和DealWithOutput()这样的例程名称不会告诉你例程做什么。最多，这些名称告诉你例程与计算、服务、用户、输入和输出有关。例外是当"handle"用于处理事件的特定技术含义时。

有时例程的唯一问题是名称摇摆不定；例程本身实际上可能设计得很好。如果HandleOutput()被替换为FormatAndPrintOutput()，你就相当清楚例程做什么。

在其他情况下，动词模糊是因为例程执行的操作模糊。例程存在目的弱点，名称弱是一个症状。如果是这样，最好的解决方案是重组例程和任何相关例程，使它们都有更强的目的和更准确地描述它们的更强名称。

**不要仅用数字区分例程名称。** 一个开发人员将他所有的代码写在一个大函数中。然后他每15行创建一个名为Part1、Part2等的函数。之后，他创建了一个高级函数来调用每个部分。这种创建和命名例程的方法特别恶劣（我希望也很罕见）。但程序员有时使用数字来区分像OutputUser、OutputUser1和OutputUser2这样的例程名称。这些名称末尾的数字没有提供例程所代表的不同抽象的指示，因此这些例程命名不好。

**使例程名称根据需要尽可能长。** 研究表明，变量名的最佳平均长度是9到15个字符。例程往往比变量更复杂，它们的好名称往往更长。另一方面，例程名称通常附加到对象名称上，这本质上免费提供了部分名称。总的来说，创建例程名称时的重点是使其尽可能清晰，这意味着你应该根据需要使其名称尽可能长或短以使其易于理解。

**命名函数时，使用对返回值的描述。** 函数返回一个值，函数应该根据其返回值命名。例如，cos()、customerId.Next()、printer.IsReady()和pen.CurrentColor()都是好的函数名称，准确表明函数返回什么。

> **交叉参考**
>
> 有关过程和函数之间区别的信息，请参见本章后面的"函数使用的特殊考虑"。

**命名过程时，使用强动词后跟对象。** 具有功能内聚的过程通常对对象执行操作。名称应该反映过程做什么，对对象操作意味着动词+对象名称。PrintDocument()、CalcMonthlyRevenues()、CheckOrderInfo()和RepaginateDocument()都是好的过程名称示例。

在面向对象语言中，你不需要在过程名称中包含对象名称，因为对象本身包含在调用中。你可以用这样的语句调用例程：document.Print()、orderInfo.Check()和monthlyRevenues.Calc()。像document.PrintDocument()这样的名称是冗余的，并且在延续到派生类时可能变得不准确。如果Check是从Document派生的类，check.Print()看起来明显是在打印支票，而check.PrintDocument()听起来可能是在打印支票簿寄存器或月度报表，但听起来不像是在打印支票。

**精确使用反义词。** 使用反义词的命名约定有助于一致性，这有助于可读性。像first/last这样的反义词对是普遍理解的。像FileOpen()和FileClose()这样的反义词对不是对称的，会造成混淆。以下是一些常见的反义词：

| add/remove | create/destroy | get/put |
|-----------|---------------|--------|
| begin/end | first/last | increment/decrement |
| show/hide | lock/unlock | next/previous |
| insert/delete | min/max | open/close |
| source/target | start/stop | up/down |
| old/new | | |

**为常见操作建立约定。** 在某些系统中，区分不同种类的操作很重要。命名约定通常是表明这些区别的最简单和最可靠方式。

我某个项目的代码为每个对象分配了唯一标识符。我们忽略了建立命名例程返回对象标识符的约定，所以我们有这样的例程名称：

```cpp
employee.id.Get()
dependent.GetId()
supervisor()
candidate.id()
```

Employee类暴露了它的id对象，id对象又暴露了它的Get()例程。Dependent类暴露了一个GetId()例程。Supervisor类使id成为其默认值返回。Candidate类利用了id对象的默认返回值为id这一事实，并暴露了id对象。

到项目中期，没有人能记住这些例程中哪一个应该用于哪个对象，但由于已经编写了太多代码无法回头使一切保持一致。结果，团队中的每个人都必须花费不必要的大量脑力来记住在哪个类上使用哪种语法来检索id。检索id的命名约定本可以消除这种烦恼。

## 例程可以有多长？

在前往美洲的途中，清教徒们争论例程的最佳最大长度。在整个旅程中争论之后，他们到达了普利茅斯岩并开始起草《五月花号公约》。他们仍然没有解决最大长度问题，由于在签署公约之前无法上岸，他们放弃了，没有包括它。结果是关于例程可以有多长的无休止争论。

理论上的最佳最大长度通常被描述为一屏或一到两页程序清单，大约50到150行。本着这种精神，IBM曾经将例程限制在50行，TRW将其限制在两页（McCabe 1976）。现代程序往往有极短例程与一些较长例程混合。长例程远未灭绝，就在完成本书前不久，我一个月内访问了两个客户现场。一个现场的程序员正在处理一个约4000行代码的例程，另一个现场的程序员试图驯服一个超过12000行的例程！

多年来，关于例程长度积累了大量研究，其中一些适用于现代程序，一些不适用：

- Basili和Perricone的一项研究发现，例程大小与错误负相关：随着例程大小增加（最多200行代码），每行代码的错误数量减少（Basili and Perricone 1984）。
- 另一项研究发现，例程大小与错误不相关，尽管结构复杂性和数据量与错误相关（Shen et al. 1985）。
- 1986年的一项研究发现，小例程（32行或更少的代码）与更低的成本或错误率不相关（Card, Church, and Agresti 1986; Card and Glass 1990）。证据表明，较大的例程（65行或更多代码）每行代码的开发成本更便宜。
- 一项对450个例程的实证研究发现，小例程（少于143条源语句，包括注释）每行代码的错误比大例程多23%，但修复成本比大例程低2.4倍（Selby and Basili 1991）。
- 另一项研究发现，当例程平均为100到150行代码时，代码需要更改的次数最少（Lind and Vairavan 1989）。
- IBM的一项研究发现，错误率最高的例程是大于500行代码的例程。超过500行，错误率往往与例程大小成正比（Jones 1986a）。

所有这些如何解决面向对象程序中例程长度的问题？面向对象程序中的例程将有很大比例是访问例程，它们会非常短。时不时地，复杂算法会导致更长的例程，在这种情况下，应该允许例程有机地增长到100-200行。（一行是非注释、非空白的源代码行。）数十年的证据表明，这种长度的例程不比更短的例程更容易出错。让内聚性、嵌套深度、变量数量、决策点数量、解释例程所需的注释数量以及其他与复杂性相关的考虑因素决定例程的长度，而不是施加长度限制本身。

也就是说，如果你想编写超过约200行的例程，要小心。没有研究报告较大例程降低错误率或成本降低，或者两者兼而有之，同时区分大于200行的大小，当你超过200行代码时，你肯定会遇到可理解性的上限。

## 如何使用例程参数

例程之间的接口是程序中最容易出错的区域之一。Basili和Perricone（1984）进行的一项经常被引用的研究发现，39%的所有错误是内部接口错误——例程之间通信的错误。以下是最小化接口问题的一些指导原则：

**按输入-修改-输出顺序放置参数。** 不要随机或按字母顺序排列参数，而是首先列出仅输入的参数，然后是输入和输出参数，最后是仅输出参数。这种排序暗示了例程内发生的操作序列——输入数据、更改数据并返回结果。以下是Ada中参数列表示例：

> **交叉参考**
>
> 有关记录例程参数的详细信息，请参见"注释例程"。有关格式化参数的详细信息，请参见"布局例程"。

**示例7-5。** Ada 按输入-修改-输出顺序排列参数的示例

```ada
procedure InvertMatrix(
   originalMatrix: in Matrix;       -- 1
   resultMatrix: out Matrix
);
...

procedure ChangeSentenceCase(
   desiredCase: in StringCase;
   sentence: in out Sentence
);
...

procedure PrintPageNumber(
   pageNumber: in Integer;
   status: out StatusType
);
```

(1) Ada使用in和out关键字使输入和输出参数清晰。

这个排序约定与C库约定（将修改的参数放在第一位）冲突。输入-修改-输出约定对我来说更有意义，但如果你一致地以某种方式排列参数，你仍然会为阅读你代码的人服务。

**考虑创建你自己的in和out关键字** 现代语言不像Ada那样支持in和out关键字。在这些语言中，你仍然可以使用预处理器创建你自己的in和out关键字：

**示例7-6。** C++ 定义你自己的In和Out关键字示例

```cpp
#define IN
#define OUT
void InvertMatrix(
   IN Matrix originalMatrix,
   OUT Matrix *resultMatrix
);
...

void ChangeSentenceCase(
   IN StringCase desiredCase,
   IN OUT Sentence *sentenceToEdit
);
...

void PrintPageNumber(
   IN int pageNumber,
   OUT StatusType &status
);
```

在这种情况下，IN和OUT宏关键字用于文档目的。为了使参数的值可被调用的例程更改，参数仍然需要作为指针或引用参数传递。

在采用这种技术之前，一定要考虑两个显著的缺点。定义你自己的IN和OUT关键字以大多数阅读你代码的人不熟悉的方式扩展了C++语言。如果你以这种方式扩展语言，一定要一致地做，最好是项目范围的。第二个限制是IN和OUT关键字不会被编译器强制执行，这意味着你可能潜在地将参数标记为IN，然后在例程内修改它。这可能会让阅读你代码的人错误地假设代码是正确的而实际上不是。使用C++的const关键字通常将是识别仅输入参数的首选方式。

**如果几个例程使用类似参数，将类似参数放在一致顺序。** 例程参数的顺序可以是一个助记符，不一致的顺序会使参数难以记住。例如，在C中，fprintf()例程与printf()例程相同，只是添加了一个文件作为第一个参数。类似的例程fputs()与puts()相同，只是添加了一个文件作为最后一个参数。这是一个令人恼火的、无意义的区别，使这些例程的参数比必要更难记住。

另一方面，C中的strncpy()例程按顺序接受目标字符串、源字符串和最大字节数，memcpy()例程以相同顺序接受相同参数。两个例程之间的相似性有助于记住任一例程中的参数。

**使用所有参数。** 如果你将参数传递给例程，使用它。如果你没有使用它，从例程接口中删除参数。未使用的参数与错误率增加相关。在一项研究中，46%没有未使用变量的例程没有错误，而只有一个以上未引用变量的例程中只有17%到29%没有错误（Card, Church, and Agresti 1986）。

删除未使用参数这条规则有一个例外。如果你是条件编译程序的一部分，你可能条件性地编译掉使用某个参数的例程部分。对这种实践要谨慎，但如果你确信它有效，那也可以。一般来说，如果你有好理由不使用参数，继续保留它。如果你没有好理由，就要努力清理代码。

**将状态或错误变量放在最后。** 按照惯例，状态变量和表示发生错误的变量放在参数列表的最后。它们对例程的主要目的是附带的，它们是仅输出参数，所以这是一个合理的约定。

**不要将例程参数用作工作变量。** 将传递给例程的参数用作工作变量是危险的。改用局部变量。例如，在以下Java代码片段中，变量inputVal被不正确地用于存储计算的中间结果：

**示例7-7。** Java 输入参数不正确使用的示例

```java
int Sample( int inputVal ) {
   inputVal = inputVal * CurrentMultiplier( inputVal );
   inputVal = inputVal + CurrentAdder( inputVal );
   ...
   return inputVal;       // 1
}
```

(1) 在这一点上，inputVal不再包含输入的值。

在这个代码片段中，inputVal是误导性的，因为当执行到达最后一行时，inputVal不再包含输入值；它包含部分基于输入值的计算值，因此名称错误。如果你以后需要修改例程以在其他地方使用原始输入值，你可能会使用inputVal并假设它包含原始输入值，而实际上它不包含。

如何解决这个问题？可以通过重命名inputVal解决吗？可能不行。你可以将其命名为workingVal之类的东西，但这是一个不完整的解决方案，因为名称没有表明变量的原始值来自例程外部。你可以将其命名为像inputValThatBecomesWorkingVal这样荒谬的名称，或者完全放弃并命名为x或val，但所有这些方法都很薄弱。

更好的方法是通过明确使用工作变量来避免当前和未来的问题。以下代码片段演示了该技术：

**示例7-8。** Java 输入参数的良好使用示例

```java
int Sample( int inputVal ) {
   int workingVal = inputVal;
   workingVal = workingVal * CurrentMultiplier( workingVal );
   workingVal = workingVal + CurrentAdder( workingVal );
   ...
   ...
   return workingVal;
}
```

(1) 如果你在这里或某处需要使用inputVal的原始值，它仍然可用。

引入新变量workingVal澄清了inputVal的角色，消除了在错误时间错误使用inputVal的机会。（不要将这个推理作为字面上命名变量为inputVal或workingVal的理由。一般来说，inputVal和workingVal是变量的糟糕名称，这些名称仅在本例中用于使变量的角色清晰。）

将输入值赋给工作变量强调值来自哪里。它消除了参数列表中的变量被意外修改的可能性。在C++中，可以使用const关键字通过编译器强制执行此做法。如果你将参数指定为const，你就不允许在例程内修改其值。

**记录关于参数的接口假设。** 如果你假设传入例程的数据具有某些特征，在做出假设时记录这些假设。在例程本身和调用例程的地方都记录你的假设，这不是浪费精力。不要等到写完例程后再回去写注释——你不会记住所有假设。比记录你的假设更好的是，使用断言将它们放入代码中。

> **交叉参考**
>
> 有关接口假设的详细信息，请参见第8章的介绍。有关文档的详细信息，请参见第32章。

关于参数你应该记录什么样的接口假设？

- 参数是仅输入、修改还是仅输出
- 数字参数的单位（英寸、英尺、米等）
- 状态代码和错误值的含义（如果未使用枚举类型）
- 预期值的范围
- 不应出现的特定值

**将例程的参数数量限制在约七个。** 七个是人们理解的魔法数字。心理学研究发现，人们通常一次无法跟踪超过约七个信息块（Miller 1956）。这一发现已应用于大量学科，似乎可以安全地推测大多数人一也不能同时跟踪超过约七个例程参数。

在实践中，你可以限制参数数量的程度取决于你的语言如何处理复杂数据类型。如果你使用支持结构化数据的现代语言编程，你可以传递一个包含13个字段的复合数据类型，并将其视为一个心理"块"数据。如果你使用更原始的语言编程，你可能需要单独传递所有13个字段。

如果你发现自己持续传递超过几个参数，你的例程之间的耦合太紧了。设计例程或例程组以减少耦合。如果你将相同数据传递给许多不同的例程，将例程分组到一个类中，并将常用数据视为类数据。

> **交叉参考**
>
> 有关如何思考接口的详细信息，请参见"良好的类接口"中的"良好的抽象"。

**考虑为参数使用输入、修改和输出命名约定。** 如果你认为区分输入、修改和输出参数很重要，建立一个识别它们的命名约定。你可以用i_、m_和o_作为前缀。如果你想要冗长一些，可以用Input_、Modify_和Output_作为前缀。

**传递例程需要维护其接口抽象的变量或对象。** 关于如何将对象的成员传递给例程，有两种相互竞争的思想流派。假设你有一个通过10个访问例程暴露数据的对象，而被调用的例程需要三个数据元素来完成其工作。

第一种思想流派的支持者认为，只应该传递例程需要的三个特定元素。他们认为这样做会将例程之间的连接保持在最小；减少耦合；并使它们更容易理解、重用等。他们说将整个对象传递给例程通过可能向被调用的例程暴露所有10个访问例程来违反封装原则。

第二种思想流派的支持者认为应该传递整个对象。他们认为，如果被调用的例程有灵活性使用对象的额外成员而无需更改例程接口，接口可以保持更稳定。他们认为传递三个特定元素通过暴露例程正在使用的特定数据元素来违反封装。

我认为这两条规则都过于简单化，错过了最重要的考虑：例程接口呈现什么抽象？如果抽象是例程期望你有三个特定数据元素，而这三个元素恰好由同一对象提供只是一个巧合，那么你应该分别传递三个特定数据元素。然而，如果抽象是你将始终手头有该特定对象，而例程将对该对象进行某种操作，那么当你暴露三个特定数据元素时，你确实破坏了抽象。

如果你传递整个对象，然后发现自己创建对象、用被调用例程需要的三个元素填充它，然后在调用例程后将元素从对象中拉出，这是一个迹象，表明你应该传递三个特定元素而不是整个对象。（一般来说，为调用例程"设置"或在接受调用例程后"拆卸"的代码是一个迹象，表明例程设计得不好。）

如果你发现自己经常更改例程的参数列表，而参数每次都来自同一个对象，这是一个迹象，表明你应该传递整个对象而不是特定元素。

**使用命名参数。** 在某些语言中，你可以将形参与实参明确关联。这使参数使用更加自文档化，并有助于避免参数不匹配的错误。以下是Visual Basic中的一个示例：

**示例7-9。** Visual Basic 明确标识参数的示例

```vb
Private Function Distance3d( _
   ByVal xDistance As Coordinate, _       // 1
   ByVal yDistance As Coordinate, _         |
   ByVal zDistance As Coordinate _       // 1

)
   ...
End Function
...
Private Function Velocity( _
   ByVal latitude as Coordinate, _
   ByVal longitude as Coordinate, _
   ByVal elevation as Coordinate _
)
   ...
   Distance = Distance3d( xDistance := latitude, yDistance := longitude, _       // 2
      zDistance := elevation )
   ...
End Function
```

(1) 这里是声明形参的地方。

(2) 这里是实参映射到形参的地方。

当你有比平均更长的同类型参数列表时，这种技术特别有用，这增加了你可以在编译器未检测到的情况下插入参数不匹配的机会。明确关联参数在许多环境中可能过度，但在安全关键或其他高可靠性环境中，确保参数按你预期的方式匹配的额外保证可能是值得的。

**确保实参与形参匹配。** 形参（也称为"哑参数"）是在例程定义中声明的变量。实参是在实际例程调用中使用的变量、常量或表达式。

一个常见错误是在例程调用中放入错误类型的变量——例如，在需要浮点数时使用整数。（这只是在不完整类型语言如C中不使用完整编译器警告时的问题。强类型语言如C++和Java没有这个问题。）当参数仅是输入时，这很少是问题；通常编译器在将参数传递给例程之前将实参类型转换为形参类型。如果这是个问题，通常你的编译器会给你一个警告。但在某些情况下，特别是当参数用于输入和输出时，你可能会因为传递错误类型的参数而受苦。

养成检查参数列表中参数类型并注意编译器关于不匹配参数类型的警告的习惯。

## 函数使用的特殊考虑

C++、Java和Visual Basic等现代语言同时支持函数和过程。函数是返回值的例程；过程是不返回值的例程。在C++中，所有例程通常被称为"函数"；但是，带void返回类型的函数在语义上是一个过程。函数和过程之间的区别既是语义的也是句法的区分，语义应该是你的指导。

---

### 何时使用函数，何时使用过程

纯粹主义者认为函数应该只返回一个值，就像数学函数一样。这意味着函数只接受输入参数并通过函数本身返回其唯一值。函数应该始终根据其返回的值命名，如sin()、CustomerID()和ScreenHeight()。另一方面，过程可以接受输入、修改和输出参数——每种可以有任意数量。

一种常见的编程实践是让一个函数作为过程运行并返回状态值。逻辑上，它作为过程工作，但因为它返回一个值，它正式是一个函数。例如，你可能有一个名为FormatOutput()的例程，与报告对象一起使用，如下语句：

```cpp
if ( report.FormatOutput( formattedReport ) = Success ) then ...
```

在这个例子中，report.FormatOutput()作为过程运行，因为它有一个输出参数formattedReport，但从技术上讲它是一个函数，因为例程本身返回一个值。这是一种有效的函数使用方式吗？为这种方法辩护，你可以说函数返回值与例程的主要目的（格式化输出）或例程名称report.FormatOutput()无关。从这个意义上说，它更像过程一样运行，即使它在技术上是一个函数。如果一致地使用这种技术，使用返回值来指示过程的成功或失败不会造成混淆。

替代方法是创建一个将状态变量作为显式参数的过程，这样代码如下：

```cpp
report.FormatOutput( formattedReport, outputStatus )
if ( outputStatus = Success ) then ...
```

我更喜欢第二种编码风格，不是因为我对函数和过程之间的区别很固执，而是因为它在例程调用和状态值测试之间提供了明确的分离。将调用和测试组合成一行代码会增加语句的密度，进而增加其复杂性。以下函数的使用也很好：

```cpp
outputStatus = report.FormatOutput( formattedReport )
if ( outputStatus = Success ) then ...
```

简而言之，如果例程的主要目的是返回函数名称所指示的值，则使用函数。否则，使用过程。

---

### 设置函数的返回值

使用函数会产生函数返回不正确返回值的风险。这通常发生在函数有多个可能路径而其中一个路径没有设置返回值时。为降低此风险，请执行以下操作：

**检查所有可能的返回路径。** 创建函数时， mentally执行每个路径以确保函数在所有可能情况下都返回一个值。在函数开头将返回值初始化为默认值是个好做法——如果正确的返回值未设置，这将提供安全网。

**不要返回对局部数据的引用或指针。** 一旦例程结束且局部数据超出范围，对局部数据的引用或指针将无效。如果对象需要返回关于其内部数据的信息，它应该将信息保存为类成员数据。然后它应该提供返回成员数据项值的访问函数，而不是对局部数据的引用或指针。

## 宏例程和内联例程

用预处理器宏创建的例程需要一些独特的考虑。以下规则和示例适用于在C++中使用预处理器。如果你使用不同的语言或预处理器，请根据你的情况调整规则。

> **交叉参考**
>
> 即使你的语言没有宏预处理器，你也可以构建自己的。详细信息请参见"构建你自己的编程工具"。

**完全括号化宏表达式。** 因为宏及其参数被展开成代码，要小心它们按你想要的方式展开。一个常见问题在于创建这样的宏：

**示例7-10。** C++ 不能正确展开的宏示例

```cpp
#define Cube( a ) a*a*a
```

如果你传递非原子值a给这个宏，它将不能正确进行乘法。如果你使用表达式Cube(x+1)，它会展开为x+1 * x + 1 * x + 1，由于乘法和加法运算符的优先级，这不是你想要的。更好但仍不完美的宏版本如下：

**示例7-11。** C++ 仍然不能正确展开的宏示例

```cpp
#define Cube( a ) (a)*(a)*(a)
```

这很接近，但仍然不行。如果你将Cube()用在具有比乘法更高优先级的运算符的表达式中，(a)*(a)*(a)会被撕开。为防止这种情况，将整个表达式用括号括起来：

**示例7-12。** C++ 正常工作的宏示例

```cpp
#define Cube( a ) ((a)*(a)*(a))
```

**用花括号包围多语句宏。** 宏可以有多个语句，如果你将其视为单个语句就会有问题。以下是一个注定要出问题的宏示例：

**示例7-13。** C++ 不工作的多语句宏示例

```cpp
#define LookupEntry( key, index ) \
   index = (key - 10) / 5; \
   index = min( index, MAX_INDEX ); \
   index = max( index, MIN_INDEX );
...

for ( entryCount = 0; entryCount < numEntries; entryCount++ )
   LookupEntry( entryCount, tableIndex[ entryCount ] );
```

这个宏注定要出问题，因为它不像常规函数那样工作。正如所示，在for循环中执行的宏只有第一行：

```cpp
index = (key - 10) / 5;
```

为避免此问题，用花括号包围宏：

**示例7-14。** C++ 正常工作的多语句宏示例

```cpp
#define LookupEntry( key, index ) { \
   index = (key - 10) / 5; \
   index = min( index, MAX_INDEX ); \
   index = max( index, MIN_INDEX ); \
}
```

将宏用作函数调用的替代品通常被认为是冒险且难以理解的——糟糕的编程实践——所以只有在你有特定情况需要时才使用这种技术。

**命名可以用例程替换的代码宏，使其可以像例程一样被替换。** C++中命名宏的约定是使用全大写字母。但是，如果宏可以替换为例程，则使用例程的命名约定而不是它。这样你就可以在不影响任何内容的情况下将宏替换为例程，反之亦然，只需更改涉及的例程。

遵循这个建议会有一些风险。如果你通常将++和--用作副作用（作为其他语句的一部分），当你使用你认为的例程宏时会吃亏。考虑到副作用的其他问题，这是避免使用副作用的又一个原因。

---

### 使用宏例程的限制

C++等现代语言提供了许多宏的替代品：

- const用于声明常量值
- inline用于定义将作为内联代码编译的函数
- template用于以类型安全的方式定义min、max等标准操作
- enum用于定义枚举类型
- typedef用于定义简单的类型替换

正如C++的设计者Bjarne Stroustrup所指出的："几乎每个宏都展示了编程语言、程序或程序员中的缺陷……当你使用宏时，你应该期望调试器、交叉引用工具和性能分析器等工具提供较差的服务"（Stroustrup 1997）。宏对于支持条件编译很有用——参见调试辅助——但细心的程序员通常只在万不得已时才将宏作为例程的替代品。

---

### 内联例程

C++支持inline关键字。内联例程允许程序员在编码时将代码视为例程，但编译器通常会在编译时将每个例程实例转换为内联代码。理论上，内联可以帮助生成避免例程调用开销的高效代码。

**谨慎使用内联例程。** 内联例程违反封装，因为C++要求程序员将内联例程实现代码放入头文件中，这向使用头文件的每个程序员暴露了它。

内联例程要求每次调用例程时都生成完整的例程代码，对于任何大小的内联例程都会增加代码大小。这本身就会产生问题。

出于性能原因内联的根本原因与任何其他以性能为动机的编码技术的根本原因相同：分析代码并测量改进。如果预期的性能提升不能证明分析代码以验证改进的麻烦是合理的，它也不能证明代码质量的侵蚀是合理的。

## 高质量例程检查清单

### 大局问题

- 创建例程的理由充分吗？
- 例程中所有可能受益于放入自己例程的部分都被放入自己的例程了吗？

### 命名

- 例程的名称对于过程是强、清晰的动词+对象名称，还是对于函数是返回值的描述？
- 例程名称是否描述了例程执行的所有操作？
- 你是否为常见操作建立了命名约定？
- 例程是否有强功能内聚——只做一件事且做好它？
- 例程是否松散耦合——例程与其他例程的连接是否小、亲密、可见且灵活？
- 例程的长度是由其功能和逻辑自然决定的，还是由人工编码标准决定的？

### 参数传递问题

- 从整体上看，例程的参数列表是否呈现一致的接口抽象？
- 例程的参数顺序是否合理，包括与类似例程的参数顺序匹配？
- 接口假设是否已记录？
- 例程是否有七个或更少的参数？
- 每个输入参数都被使用了吗？
- 每个输出参数都被使用了吗？
- 例程是否避免将输入参数用作工作变量？
- 如果例程是函数，它在所有可能情况下都返回有效值吗？

---

## 关键点

- 创建例程最重要的理由是提高程序在智力上的可管理性，你还可以基于许多其他好理由创建例程。节省空间是一个次要理由；提高可读性、可靠性和可修改性是更好的理由。
- 最受益于放入自己例程的操作有时是一个简单的操作。
- 你可以将例程分类为各种内聚类型，但你可以使大多数例程具有功能内聚，这是最好的。
- 例程的名称是其质量的指示。如果名称不好且准确，例程可能设计得不好。如果名称不好且不准确，它不会告诉你程序做什么。无论哪种方式，糟糕的名称意味着程序需要更改。
- 函数应该只在函数的主要目的是返回函数名称所描述的特定值时使用。
- 细心的程序员谨慎使用宏例程，只在万不得已时使用。
