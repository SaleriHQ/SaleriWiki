在计算机编程的大部分历史中，代码调优一直是一个受欢迎的话题。因此，一旦你决定需要提高性能，并且你想在代码级别进行（记住第25章的警告），你有丰富的技术可供使用。

本章专注于提高速度，并包含一些使代码更小的技巧。性能通常指速度和大小两个方面，但大小缩减往往更多地来自重新设计类和数据，而不是调优代码。代码调优指的是小规模更改，而不是更大规模设计中的更改。

本章中的技术很少有如此普遍适用，以至于你能将示例代码直接复制到程序中。这里的主要目的是说明一些你可以适应你的情况的代码调优方法。

本章中描述的代码调优更改可能看起来与第24章中描述的重构在美容上相似，但重构是改善程序内部结构的更改（Fowler 1999）。本章中的更改最好称为"反重构"。这些更改远远不是"改善内部结构"，它们为了性能提升而降低内部结构。这是按定义的。如果更改没有降低内部结构，我们就不会认为它们是优化；我们会默认使用它们，并将它们视为标准编码实践。

有些书将代码调优技术作为"经验法则"呈现，或引用研究表明特定调优将产生预期效果。正如你很快会看到的，"经验法则"概念对代码调优不太适用。唯一可靠的经验法则是测量每次调优在你的环境中的效果。因此，本章呈现了一个"尝试事项"目录，其中许多在你环境中不起作用，但有些确实非常有效。

### 知道答案时停止测试

假设你有这样的语句：

```cpp
if ( 5 < x ) and ( x < 10 ) then ...
```

一旦你确定x大于5，你不需要执行测试的第二部分。

有些语言提供一种称为"短路求值"的表达式求值形式，这意味着编译器生成在知道答案时自动停止测试的代码。短路求值是C++标准运算符和Java"条件"运算符的一部分。

交叉引用

有关短路求值的更多信息，请参见布尔表达式中的"知道布尔表达式如何求值"。

如果你的语言本身不支持短路求值，你必须避免使用and和or，改为添加逻辑。使用短路求值，上面的代码变为：

```cpp
if ( 5 < x ) then
   if ( x < 10 ) then ...
```

在知道答案后不测试的原则对许多其他类型的情况也是好的。搜索循环是一种常见情况。如果你正在扫描输入数字数组中的负值，而你只需要知道是否存在负值，一种方法是检查每个值，在找到时设置negativeFound变量。以下是搜索循环的样子：

```cpp
// 找到答案后不停止的示例
negativeInputFound = false;
for ( i = 0; i < count; i++ ) {
   if ( input[ i ] < 0 ) {
      negativeInputFound = true;
   }
}
```

更好的方法是在找到负值后立即停止扫描。以下任何方法都可以解决问题：

- 在`negativeInputFound = true`行后添加break语句。
- 如果你的语言没有break，用goto模拟break，跳到循环后的第一个语句。
- 将for循环改为while循环，并在递增循环计数器超过count的同时检查negativeInputFound。
- 将for循环改为while循环，在第一个数组元素之后放入一个哨兵值，然后在while测试中简单地检查负值。循环终止后，查看第一个找到的值的位置是在数组中还是刚刚超过末尾。哨兵在本章后面有更详细的讨论。

以下是使用C++和Java中break关键字的结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 4.27 | 3.68 | 14% |
| Java | 4.85 | 3.46 | 29% |

这种更改的影响因你有多少值以及你期望多久找到负值而差异很大。此测试假设平均有100个值，并假设50%的时间会发现负值。

### 按频率排列测试

排列测试，使最快且最可能为真的测试首先执行。正常的case应该很容易通过，如果有低效，它们应该在处理不常见情况中。本原则适用于case语句和if-then-else链。

以下是一个在文字处理器中响应键盘输入的Select-Case语句：

```vb
' 排列不佳的逻辑测试示例
Select inputCharacter
   Case "+", "="
      ProcessMathSymbol( inputCharacter )
   Case "0" To "9"
      ProcessDigit( inputCharacter )
   Case ",", ".", ":", ";", "!", "?"
      ProcessPunctuation( inputCharacter )
   Case " "
      ProcessSpace( inputCharacter )
   Case "A" To "Z", "a" To "z"
      ProcessAlpha( inputCharacter )
   Case Else
      ProcessError( inputCharacter )
End Select
```

这个case语句中的case大致按ASCII排序顺序排列。然而，在case语句中，效果通常与你写了一个大的if-then-else集合一样，所以如果你得到一个"a"作为输入字符，程序会在确定它是字母字符之前测试它是否是数学符号、标点符号、数字或空格。如果你知道输入字符的可能频率，你可以把最常见的case放在前面。以下是重新排序的case语句：

```vb
' 排列良好的逻辑测试示例
Select inputCharacter
   Case "A" To "Z", "a" To "z"
      ProcessAlpha( inputCharacter )
   Case " "
      ProcessSpace( inputCharacter )
   Case ",", ".", ":", ";", "!", "?"
      ProcessPunctuation( inputCharacter )
   Case "0" To "9"
      ProcessDigit( inputCharacter )
   Case "+", "="
      ProcessMathSymbol( inputCharacter )
   Case Else
      ProcessError( inputCharacter )
End Select
```

因为最常见的case通常在优化代码中更早找到，净效果将是减少测试的性能。以下是使用典型字符混合进行此优化的结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C# | 0.220 | 0.260 | -18% |
| Java | 2.56 | 2.56 | 0% |
| Visual Basic | 0.280 | 0.260 | 7% |

注意：使用78%字母字符、17%空格和5%标点符号的输入混合进行基准测试。

Microsoft Visual Basic结果如预期，但Java和C#结果不如预期。显然这是因为C++和Java中switch-case语句的结构方式——因为每个值必须单独枚举而不是按范围，C++和Java代码没有像Visual Basic代码那样从优化中受益。这个结果强调了一定不要盲目遵循任何优化建议的重要性——具体的编译器实现将显著影响结果。

你可能认为Visual Basic编译器为与case语句执行相同测试的if-then-else集合生成的代码是相似的。看一下这些结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C# | 0.630 | 0.330 | 48% |
| Java | 0.922 | 0.460 | 50% |
| Visual Basic | 1.36 | 1.00 | 26% |

结果相当不同。对于相同数量的测试，Visual Basic编译器在未优化情况下花费的时间大约是五倍，在优化情况下是四倍。这表明编译器为case方法生成的代码与为if-then-else方法生成的代码不同。

if-then-else的改进比case语句更一致，但这是好坏参半。在C#和Visual Basic中，case语句方法的两个版本都比if-then-else方法的两个版本更快，而在Java中两个版本都更慢。结果的这种变化表明了第三种可能的优化，在下一节中描述。

### 比较相似逻辑结构的性能

上述测试可以使用case语句或if-then-else执行。在不同环境中，任一方法都可能效果更好。以下是前面两个表格中的数据重新格式化为呈现比较if-then-else和case性能的"代码调优"时间：

| 语言 | case | if-then-else | 时间节省 | 性能比 |
|------|------|-------------|----------|---------|
| C# | 0.260 | 0.330 | -27% | 1:1 |
| Java | 2.56 | 0.460 | 82% | 6:1 |
| Visual Basic | 0.260 | 1.00 | 258% | 1:4 |

这些结果违背了任何逻辑解释。在一种语言中，case明显优于if-then-else，在另一种语言中，if-then-else明显优于case。在第三种语言中，差异相对较小。你可能认为因为C#和Java有相似的case语句语法，它们的结果应该相似，但实际上它们的结果恰恰相反。

这个例子清楚地说明了执行任何形式的"经验法则"或"逻辑"代码调优的困难——没有可靠的替代方法来测量结果。

### 用查表替换复杂表达式

在某些情况下，查表可能比遍历复杂的逻辑链更快。复杂逻辑链的重点通常是对某物进行分类，然后根据其类别采取行动。作为一个抽象的例子，假设你想根据某物落入三组——A组、B组和C组中的哪一组来分配类别编号：

交叉引用

有关使用查表替换复杂逻辑的详细信息，请参见第18章。

这个复杂的逻辑链分配类别编号：

```cpp
// 复杂逻辑链示例
if ( ( a && !c ) || ( a && b && c ) ) {
   category = 1;
}
else if ( ( b && !a ) || ( a && c && !b ) ) {
   category = 2;
}
else if ( c && !a && !b ) {
   category = 3;
}
else {
   category = 0;
}
```

你可以用更可修改和更高性能的查找表替换这个测试：

```cpp
// 使用查表替换复杂逻辑的示例
// 定义 categoryTable
static int categoryTable[ 2 ][ 2 ][ 2 ] = {       // 1
   // !b!c !bc b!c bc
       0, 3, 2, 2, // !a
       1, 2, 1, 1 // a
};
...
category = categoryTable[ a ][ b ][ c ];
```

(1)这个表定义有点难理解。你可以做任何注释使表定义可读，这会有帮助。

虽然表的定义很难阅读，但如果记录良好，它不会比复杂逻辑链的代码更难阅读。如果定义更改，表将比早期逻辑更容易维护。以下是性能结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 | 性能比 |
|------|----------|-------------|----------|---------|
| C++ | 5.04 | 3.39 | 33% | 1.5:1 |
| Visual Basic | 5.21 | 2.60 | 50% | 2:1 |

### 使用惰性求值

我以前的一个室友是一个严重的拖延者。他用这样的说法为他的懒惰辩解：人们感到迫切要做的许多事情根本不需要做。他说，如果等得足够长，不重要的事情会被拖延到被遗忘，他就不会浪费时间做这些事情。

惰性求值基于我室友使用的原则。如果程序使用惰性求值，它会避免做任何不需要的工作直到工作需要时。惰性求值类似于在需要时做工作的即时策略。

例如，假设你的程序包含一个5000值的表，在启动时生成整个表，然后在程序执行时使用它。如果程序只使用表中一小部分条目，在需要时计算它们而不是一次全部计算可能更有意义。一旦计算了一个条目，它仍然可以存储以供将来参考（也称为"缓存"）。

### 反转循环

反转循环指的是在每次执行时在循环内做出决定。如果决定在循环执行时不改变，你可以通过在循环外做出决定来反转循环。通常这需要将循环翻转，将循环放在条件内部而不是将条件放在循环内。以下是反转前的循环示例：

```cpp
// 反转前的循环示例
for ( i = 0; i < count; i++ ) {
   if ( sumType == SUMTYPE_NET ) {
      netSum = netSum + amount[ i ];
   }
   else {
      grossSum = grossSum + amount[ i ];
   }
}
```

在这段代码中，`if ( sumType == SUMTYPE_NET )`测试在每次迭代中重复，尽管每次循环时它都是一样的。你可以像这样重写代码以获得速度提升：

```cpp
// 反转后的循环示例
if ( sumType == SUMTYPE_NET ) {
   for ( i = 0; i < count; i++ ) {
      netSum = netSum + amount[ i ];
   }
}
else {
   for ( i = 0; i < count; i++ ) {
      grossSum = grossSum + amount[ i ];
   }
}
```

注意

这个代码片段违反了几条良好编程规则。可读性和可维护性通常比执行速度或大小更重要，但在本章中主题是性能，这意味着与其他目标的权衡。就像上一章一样，你会在这里看到本书其他部分不推荐的编码实践示例。

这大约节省20%的时间：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 2.81 | 2.27 | 19% |
| Java | 3.97 | 3.12 | 21% |
| Visual Basic | 2.78 | 2.77 | <1% |
| Python | 8.14 | 5.87 | 28% |

这种特殊情况的一个独特危险是两个循环必须并行维护。如果count变为clientCount，你必须记住在两个地方都更改它，这对你是烦恼，对必须处理代码的其他任何人来说是维护麻烦。

这个例子也说明了代码调优中的一个关键挑战：任何特定代码调优的效果是不可预测的。代码调优在四种语言中的三种产生了显著的改进，但在Visual Basic中没有。对于这个特定版本的Visual Basic执行这个特定的优化将产生更难维护的代码而没有任何性能补偿作为 offset。一般的教训是，你必须测量每个特定优化的效果以确定其效果——没有例外。

### 合并循环

合并循环，或"融合"，是合并对同一组元素进行操作的两个循环的结果。收益在于将两个循环的循环开销削减为一个。以下是适合合并循环的候选：

```vb
' 可以合并的单独循环示例
For i = 0 to employeeCount - 1
   employeeName( i ) = ""
Next
...
For i = 0 to employeeCount - 1
   employeeEarnings( i ) = 0
Next
```

当你合并循环时，你在两个循环中找到可以合并为一个的代码。通常，这意味着循环计数器必须相同。在这个例子中，两个循环都从0运行到employeeCount - 1，所以你可以合并它们：

```vb
' 合并后的循环示例
For i = 0 to employeeCount - 1
   employeeName( i ) = ""
   employeeEarnings( i ) = 0
Next
```

以下是节省：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 3.68 | 2.65 | 28% |
| PHP | 3.97 | 2.42 | 32% |
| Visual Basic | 3.75 | 3.56 | 4% |

注意：employeeCount等于100的情况进行了基准测试。

和以前一样，结果因语言而显著差异。

循环合并有两个主要危险。首先，合并的两部分的索引可能会改变，以至于它们不再兼容。其次，你可能无法轻松合并循环。在合并循环之前，确保它们相对于代码的其余部分仍然处于正确的顺序。

### 展开循环

循环展开的目标是减少循环开销。在第25章中，一个循环被完全展开，显示了10行代码比3行更快。在这种情况下，从3行到10行的循环被展开，以便所有10个数组访问都单独完成。

虽然完全展开循环是一个快速的解决方案，在你处理少量元素时效果很好，但当你有大量元素或你事先不知道有多少元素时，这不实用。以下是一个通用循环的示例：

```java
// 可以展开的循环示例
i = 0;       // 1
while ( i < count ) {
   a[ i ] = i;
   i = i + 1;
}
```

(1)通常，对于这样的工作你可能使用for循环，但为了优化，你需要转换为while循环。为了清晰起见，这里显示了while循环。

要部分展开循环，你在每次循环中处理两个或更多情况，而不是一个。这种展开损害了可读性但不会损害循环的通用性。以下是循环展开一次的样子：

```java
// 展开一次的循环示例
i = 0;
while ( i < count - 1 ) {
   a[ i ] = i;
   a[ i + 1 ] = i + 1;
   i = i + 2;
}

if ( i == count ) {       // 1
   a[ count - 1 ] = count - 1;|
}       // 1
```

(1)这些行处理了如果循环按两个而不是一个经过时可能掉落的最后情况。

这项技术用两行替换了原始的`a[ i ] = i`行，i每次增加2而不是1。while循环后的额外代码在count为奇数且循环终止后剩下一次迭代时需要。

当五行直接代码扩展为九行棘手代码时，代码变得更难阅读和维护。除非速度上有收益，其质量很差。然而，任何设计纪律的一部分是做出必要的权衡。因此，即使特定技术通常代表糟糕的编码实践，特定情况可能会使其成为最佳选择。

以下是展开循环的结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 1.75 | 1.15 | 34% |
| Java | 1.01 | 0.581 | 43% |
| PHP | 5.33 | 4.49 | 16% |
| Python | 2.51 | 3.21 | -27% |

注意：count等于100的情况进行了基准测试。

16%到43%的收益是可观的，尽管你必须注意不要损害性能，Python基准测试显示了这一点。循环展开的主要危险是在循环后挑选最后情况的代码中出现边界差一错误。

如果你进一步展开循环，进行两次或更多次展开呢？如果你展开循环两次，你会得到更多好处吗？

```java
// 展开两次的循环示例
i = 0;
while ( i < count - 2 ) {
   a[ i ] = i;
   a[ i + 1 ] = i+1;
   a[ i + 2 ] = i+2;
   i = i + 3;
}
if ( i <= count - 1 ) {
   a[ count - 1 ] = count - 1;
}
if ( i == count - 2 ) {
   a[ count -2 ] = count - 2;
}
```

以下是第二次展开循环的结果：

| 语言 | 直接时间 | 双重展开时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 1.75 | 1.01 | 42% |
| Java | 1.01 | 0.581 | 43% |
| PHP | 5.33 | 3.70 | 31% |
| Python | 2.51 | 2.79 | -12% |

注意：count等于100的情况进行了基准测试。

结果表明，进一步循环展开可以带来进一步的时间节省，但不一定如此，正如Java测量所示。主要关注的是你的代码变得多么拜占庭式。当你查看前面的代码时，你可能不认为它看起来非常复杂，但当你意识到它在几页前开始时是一个五行循环时，你可以欣赏性能和可读性之间的权衡。

### 最小化循环内的工作

编写有效循环的关键是最小化循环内完成的工作。如果你能将语句或语句的一部分在循环外求值，这样只有结果在循环内使用，那就这样做。这是良好的编程实践，在某些情况下提高了可读性。

假设你在热点循环内有一个像这样的复杂指针表达式：

```cpp
// 循环内复杂指针表达式示例
for ( i = 0; i < rateCount; i++ ) {
   netRate[ i ] = baseRate[ i ] * rates->discounts->factors->net;
}
```

在这种情况下，将复杂指针表达式赋值给命名良好的变量既提高了可读性又通常提高了性能。

```cpp
// 简化复杂指针表达式示例
quantityDiscount = rates->discounts->factors->net;
for ( i = 0; i < rateCount; i++ ) {
   netRate[ i ] = baseRate[ i ] * quantityDiscount;
}
```

额外的变量quantityDiscount清楚地表明baseRate数组乘以数量折扣因子来计算净费率。从循环中的原始表达式来看，这根本不清楚。将复杂指针表达式放入循环外的变量也节省了指针在每次循环通过时被解引用三次，结果如下节省：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 3.69 | 2.97 | 19% |
| C# | 2.27 | 1.97 | 13% |
| Java | 4.13 | 2.35 | 43% |

注意：rateCount等于100的情况进行了基准测试。

除了Java编译器，节省都不值一提，这意味着在初始编码时你可以使用哪种技术更可读，而不必担心代码的速度直到以后。

### 哨兵值

当你有一个带有复合测试的循环时，你通常可以通过简化测试来节省时间。如果循环是搜索循环，简化测试的一种方法是使用哨兵值，一个你放在搜索范围末尾之外的值，保证终止搜索。

可以通过使用哨兵改进的复合测试的经典例子是既检查是否找到了你寻找的值又检查是否已用完值的搜索循环。以下是代码：

```csharp
// 搜索循环中复合测试的示例
found = FALSE;
i = 0;
while ( ( !found ) && ( i < count ) ) {       // 1
   if ( item[ i ] == testValue ) {
      found = TRUE;
   }
   else {
      i++;
   }
}
if ( found ) {
   ...

(1)这里是复合测试。
```

在这段代码中，循环的每次迭代都测试!found和i < count。!found测试的目的是确定何时找到所需元素。i < count测试的目的是避免运行超过数组末尾。在循环内，item[]的每个值都被单独测试，所以循环在每次迭代中实际上有三次测试。

在这种搜索循环中，你可以通过在搜索范围末尾放入一个"哨兵"来停止循环，从而将三次测试合并为每次迭代只测试一次。在这种情况下，你只需将你寻找的值赋给搜索范围末尾之后的元素。（记住在声明数组时为该元素留出空间。）然后你检查每个元素，如果你直到找到你放在末尾的那个才找到该元素，你就知道你寻找的值实际上不在那里。以下是代码：

```csharp
// 使用哨兵值加速循环的示例

// 设置哨兵值，保留原始值
initialValue = item[ count ];
item[ count ] = testValue;       // 1

i = 0;
while ( item[ i ] != testValue ) {
   i++;
}

// 检查是否找到值
if ( i < count ) {
   ...

(1)记住在数组末尾为哨兵值留出空间。
```

### 将最忙的循环放在内部

当你有嵌套循环时，想想你希望哪个循环在外面，哪个在里面。以下是一个可以改进的嵌套循环示例：

```java
// 可以改进的嵌套循环示例
for ( column = 0; column < 100; column++ ) {
   for ( row = 0; row < 5; row++ ) {
      sum = sum + table[ row ][ column ];
   }
}
```

改进循环的关键是外循环比内循环执行得多得多。每次循环执行时，它必须初始化循环索引，在每次通过循环时递增它，并在每次通过后检查它。外循环的总循环执行次数是100，内循环是100 * 5 = 500，总共600次迭代。通过简单地切换内循环和外循环，你可以将总迭代次数改变为外循环5次和内循环5 * 100 = 500，总共505次迭代。从分析上，你期望通过切换循环节省约(600 – 505) / 600 = 16%。以下是性能差异的测量：

| Java | 5.39 | 3.56 | 34% |
| PHP | 4.16 | 3.65 | 12% |
| Python | 3.48 | 3.33 | 4% |

### 强度削减

削减强度意味着用更便宜的操作替换更贵的操作，如用加法替换乘法。有时你会在循环内有一个依赖于将循环索引乘以一个因子的表达式。加法通常比乘法快，如果你能通过在每次循环迭代中加法而不是乘法计算相同的数字，代码通常会运行得更快。以下是使用乘法的代码示例：

```vb
' 乘以循环索引的示例
For i = 0 to saleCount - 1
   commission( i ) = (i + 1) * revenue * baseCommission * discount
Next
```

这段代码是直接的但很贵。你可以重写循环，这样你累积倍数而不是每次计算。这将操作强度从乘法削减到加法。

```vb
' 加法而不是乘法的示例
incrementalCommission = revenue * baseCommission * discount
cumulativeCommission = incrementalCommission
For i = 0 to saleCount - 1
   commission( i ) = cumulativeCommission
   cumulativeCommission = cumulativeCommission + incrementalCommission
Next
```

乘法很贵，这种更改就像制造商优惠券一样给你循环成本的折扣。原始代码每次递增i并将其乘以revenue * baseCommission * discount——第一次乘以1，第二次乘以2，第三次乘以3，以此类推。优化的代码将incrementalCommission设置为等于revenue * baseCommission * discount。然后它在每次通过循环时将incrementalCommission加到cumulativeCommission。在第一次通过时，它被加了一次；第二次通过时，被加了两次；第三次通过时，被加了三次；依此类推。效果与将incrementalCommission乘以1、然后2、然后3等相同，但更便宜。

关键在于原始乘法必须依赖于循环索引。在这种情况下，循环索引是表达式中唯一变化的部分，所以表达式可以更经济地重新编码。以下是重写在某些测试案例中帮助的程度：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 |
|------|----------|-------------|----------|
| C++ | 4.33 | 3.80 | 12% |
| Visual Basic | 3.54 | 1.80 | 49% |

注意：使用saleCount等于20进行基准测试。所有计算变量都是浮点。

### 最小化数组引用

除了最小化访问二维或三维数组外，最小化数组引用通常也是有利的。重复使用数组一个元素的循环是应用此技术的好候选。以下是不必要的数组引用的示例：

```cpp
// 不必要引用循环内数组的示例
for ( discountType = 0; discountType < typeCount; discountType++ ) {
   for ( discountLevel = 0; discountLevel < levelCount; discountLevel++ ) {
      rate[ discountLevel ] = rate[ discountLevel ] * discount[ discountType ];
   }
}
```

当discountLevel在内循环中变化时，discount[ discountType ]的引用不会改变。因此，你可以将其移出内循环，这样外循环的每次执行只有一次数组访问，而不是内循环的每次执行一次。以下是修改后的代码。

```cpp
// 将数组引用移出循环的示例
for ( discountType = 0; discountType < typeCount; discountType++ ) {
   thisDiscount = discount[ discountType ];
   for ( discountLevel = 0; discountLevel < levelCount; discountLevel++ ) {
      rate[ discountLevel ] = rate[ discountLevel ] * thisDiscount;
   }
}
```

以下是结果：

| C++ | 32.1 | 34.5 | -7% |
| C# | 18.3 | 17.0 | 7% |
| Visual Basic | 23.2 | 18.4 | 20% |

### 使用补充索引

使用补充索引意味着添加使访问数据类型更高效的相关数据。你可以将相关数据添加到主数据类型中，或者你可以将其存储在并行结构中。

**字符串长度索引**

使用补充索引的一个例子可以在不同的字符串存储策略中找到。在C中，字符串以设置为0的字节终止。在Visual Basic字符串格式中，每个字符串开头隐藏的长度字节指示字符串有多长。要确定C中字符串的长度，程序必须从字符串开头开始，计算每个字节直到找到设置为0的字节。要确定Visual Basic字符串的长度，程序只需查看长度字节。Visual Basic长度字节是使用索引增强数据类型以使某些操作（如计算字符串长度）更快的例子。

你可以将长度索引的想法应用于任何可变长度数据类型。跟踪结构的长度通常比每次需要时计算长度更有效。

**独立并行索引结构**

有时操作数据类型的索引比操作数据类型本身更有效。如果数据类型中的项很大或很难移动（在磁盘上），排序和搜索索引引用比直接处理数据更快。如果每个数据项很大，你可以创建一个由键值和指向详细信息指针组成的辅助结构。如果数据结构项和辅助结构项之间的大小差异足够大，有时即使数据项必须存储在外部，你也可以将键项存储在内存中。所有搜索和排序都在内存中进行，你只需要在知道想要的项目的确切位置时访问磁盘一次。

### 使用缓存

缓存意味着以这样的方式保存几个值，使你最常用的值比不太常用的值更容易检索。例如，如果程序随机从磁盘读取记录，例程可能使用缓存来保存读取最频繁的记录。当例程收到记录请求时，它检查缓存是否有该记录。如果有，记录直接从内存返回而不是从磁盘返回。

除了缓存磁盘上的记录，你还可以在其他区域应用缓存。在Microsoft Windows字体校对程序中，性能瓶颈是在显示每个字符时检索其宽度。使用最近使用的字符宽度进行缓存大致使显示速度翻倍。

你也可以缓存耗时计算的结果——特别是如果计算参数简单的话。例如，假设你需要计算直角三角形斜边的长度，给定其他两条边的长度。例程的直接实现如下：

```java
// 适合缓存的例程示例
double Hypotenuse(
   double sideA,
   double sideB
   ) {
   return Math.sqrt( ( sideA * sideA ) + ( sideB * sideB ) );
}
```

如果你知道相同的值往往被重复请求，你可以像这样缓存值：

```java
// 缓存以避免昂贵计算的示例
private double cachedHypotenuse = 0;
private double cachedSideA = 0;
private double cachedSideB = 0;

public double Hypotenuse(
   double sideA,
   double sideB
   ) {

   // 检查三角形是否已经在缓存中
   if ( ( sideA == cachedSideA ) && ( sideB == cachedSideB ) ) {
      return cachedHypotenuse;
   }

   // 计算新的斜边并缓存它
   cachedHypotenuse = Math.sqrt( ( sideA * sideA ) + ( sideB * sideB ) );
   cachedSideA = sideA;
   cachedSideB = sideB;

   return cachedHypotenuse;
}
```

例程的第二版比第一版更复杂，占用更多空间，所以速度必须非常宝贵才能证明它是合理的。许多缓存方案缓存不止一个元素，所以它们有更多开销。以下是这两个版本之间的速度差异：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 | 性能比 |
|------|----------|-------------|----------|---------|
| C++ | 4.06 | 1.05 | 74% | 4:1 |
| Java | 2.54 | 1.40 | 45% | 2:1 |
| Python | 8.16 | 4.17 | 49% | 2:1 |
| Visual Basic | 24.0 | 12.9 | 47% | 2:1 |

注意：结果显示假设缓存每次设置时命中两次。

缓存的成功取决于访问缓存元素的相对成本、创建非缓存元素的成本以及在缓存中保存新元素的成本。成功还取决于缓存信息被请求的频率。在某些情况下，成功可能还取决于硬件完成的缓存。一般来说，生成新元素成本越高、相同信息被请求次数越多，缓存就越有价值。访问缓存元素和保存新元素到缓存的成本越低，缓存就越有价值。与其他优化技术一样，缓存增加了复杂性，往往容易出错。

## 表达式

程序中的大部分工作是在数学或逻辑表达式中完成的。复杂表达式往往很贵，所以本节看看使它们更便宜的方法。

### 利用代数恒等式

你可以使用代数恒等式用更便宜的操作替换更贵的操作。例如，以下表达式在逻辑上是等价的：

```
not a and not b
not (a or b)
```

如果你选择第二个表达式而不是第一个，你可以节省一个not操作。

虽然避免单个not操作节省可能无关紧要，但总体原则是强大的。Jon Bentley描述了一个测试`sqrt(x) < sqrt(y)`的程序（1982）。由于当x小于y时sqrt(x)才小于sqrt(y)，你可以用x < y替换第一个测试。考虑到sqrt()例程的成本，你期望节省是戏剧性的，而且确实如此。以下是结果：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 | 性能比 |
|------|----------|-------------|----------|---------|
| C++ | 7.43 | 0.010 | 99.9% | 750:1 |
| Visual Basic | 4.59 | 0.220 | 95% | 20:1 |
| Python | 4.21 | 0.401 | 90% | 10:1 |

### 使用强度削减

如前所述，强度削减意味着用更便宜的操作替换更贵的操作。以下是一些可能的替换：

- 用加法替换乘法。
- 用乘法替换幂运算。
- 用三角恒等式替换三角例程。
- 用long或int替换long long整数（但要注意使用本机长度与非本机长度整数相关的性能问题）。
- 用定点数或整数替换浮点数。
- 用单精度数替换双精度浮点数。
- 用移位操作替换乘以2和除以2的整数乘法和除法。

假设你需要评估一个多项式。如果你对多项式生疏了，它们看起来像Ax² + Bx + C。字母A、B和C是系数，x是变量。评估n阶多项式的一般代码如下：

```vb
' 评估多项式的示例
value = coefficient( 0 )
For power = 1 To order
   value = value + coefficient( power ) * x^power
Next
```

如果你考虑强度削减，你会用怀疑的眼光看待幂运算符。其中一个解决方案是在每次通过循环时用乘法替换幂运算，这类似于前面几节中乘法被加法替换的强度削减案例。以下是降低强度的多项式评估：

```vb
' 降低强度的多项式评估方法示例
value = coefficient( 0 )
powerOfX = x
For power = 1 to order
   value = value + coefficient( power ) * powerOfX
   powerOfX = powerOfX * x
Next
```

如果你使用二阶多项式——即最高次项是平方的多项式——或更高阶多项式，这会产生明显优势：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 | 性能比 |
|------|----------|-------------|----------|---------|
| Python | 3.24 | 2.60 | 20% | 1:1 |
| Visual Basic | 6.26 | 0.160 | 97% | 40:1 |

如果你认真对待强度削减，你仍然不会喜欢那两次浮点乘法。强度削减原则表明你可以通过累积幂而不是每次相乘来进一步降低循环中操作的强度：

```vb
' 进一步降低多项式评估强度的方法示例
value = 0
For power = order to 1 Step -1
   value = ( value + coefficient( power ) ) * x
Next
value = value + coefficient( 0 )
```

这种方法消除了额外的powerOfX变量，并用一次乘法替换了每次循环通过中的两次乘法。结果：

|------|----------|---------|-----------|-----------------|
| Python | 3.24 | 2.60 | 2.53 | 3% |
| Visual Basic | 6.26 | 0.16 | 0.31 | -94% |

这是一个理论在实践中不太好成立的很好例子。降低强度的代码看起来应该更快，但实际上不是。一种可能性是在Visual Basic中循环递减1而不是递增1会损害性能，但你必须测量该假设才能确定。

### 在编译时初始化

如果你在例程调用中使用命名常量或魔术数字，它是唯一参数，这是一个线索，表明你可以预先计算该数字，将其放入常量中，并避免例程调用。同样的原则适用于乘法、除法、加法和其他操作。

我曾经需要计算一个整数的以2为底的对数，截断到最近的整数。系统没有以2为底的对数例程，所以我写了我自己的。快速简单的方法是利用这个事实：

```
log(x)base = log(x) / log(base)
```

根据这个恒等式，我可以写这样的例程：

```cpp
// 基于系统例程的以2为底对数例程
unsigned int Log2( unsigned int x ) {
   return (unsigned int) ( log( x ) / log( 2 ) );
}
```

### 小心系统例程

系统例程很贵，提供的精度往往是浪费的。例如，典型的系统数学例程被设计为将宇航员放在距离目标±2英尺的范围内。如果你不需要那种精度，你也不需要花费时间来计算它。

在前面的例子中，Log2()例程返回一个整数值，但使用浮点log()例程计算它。这对于整数结果来说是大材小用，所以在第一次尝试后，我写了一系列整数测试，这些测试对于计算整数log2是完美准确的。以下是代码：

```cpp
// 基于整数的以2为底对数例程
unsigned int Log2( unsigned int x ) {
   if ( x < 2 ) return 0 ;
   if ( x < 4 ) return 1 ;
   if ( x < 8 ) return 2 ;
   if ( x < 16 ) return 3 ;
   if ( x < 32 ) return 4 ;
   if ( x < 64 ) return 5 ;
   if ( x < 128 ) return 6 ;
   if ( x < 256 ) return 7 ;
   if ( x < 512 ) return 8 ;
   if ( x < 1024 ) return 9 ;
   ...
   if ( x < 2147483648 ) return 30;
   return 31 ;
}
```

这个例程使用整数操作，从不转换为浮点，而且彻底打败了两种浮点版本：

| 语言 | 直接时间 | 代码调优时间 | 时间节省 | 性能比 |
|------|----------|-------------|----------|---------|
| C++ | 9.66 | 0.662 | 93% | 15:1 |
| Java | 17.0 | 0.882 | 95% | 20:1 |
| PHP | 2.45 | 3.45 | -41% | 2:3 |

大多数所谓的"超越"函数是为最坏情况设计的——也就是说，即使你给它们一个整数参数，它们也会在内部转换为双精度浮点。如果你在代码的紧密部分发现一个而不需要那么多精度，你要立即关注它。

另一个选择是利用右移操作与除以2相同的事实。你可以多少次将一个数字除以2而仍然得到非零值，这与该数字的log2相同。以下是基于该观察的代码：

```cpp
// 基于右移操作员的替代以2为底对数例程
unsigned int Log2( unsigned int x ) {
   unsigned int i = 0;
   while ( ( x = ( x >> 1 ) ) != 0 ) {
      i++;
   }
   return i ;
}
```

对于非C++程序员，这段代码特别难读。while条件中的复杂表达式是一个你应该避免的编码实践，除非你有充分的理由使用它。

这个例程比上面更长的版本花费大约350%的时间，执行2.4秒而不是0.66秒。但它比第一种方法快，而且容易适应32位、64位和其他环境。

这个例子强调了在一次成功优化后不停下来的价值。第一次优化获得了可观的30-40%节省，但没有接近第二次或第三次优化的影响。

### 预计算结果

一个常见的低级设计决策是选择是动态计算结果还是计算一次、保存它们并在需要时查找。如果结果使用多次，通常计算一次然后查找其余时间更便宜。

这种选择在几个方面表现出来。在最简单的层面上，你可能将表达式的一部分在循环外计算而不是循环内。本章前面出现了这种例子。在更复杂的层面上，你可能在程序开始执行时计算一次查找表，然后每次使用它，或者你可能将结果存储在数据文件中或嵌入程序中。

例如，在太空大战电子游戏中，程序员最初计算太阳不同距离的重力系数。重力度系数的计算很贵，影响性能。然而，程序认识到从太阳到相对较少的不同距离，所以程序员能够预计算重力度系数并将它们存储在一个10元素数组中。数组查找比昂贵的计算快得多。

### 消除公共子表达式

如果你发现一个表达式重复了几次，将其赋给一个变量并引用该变量而不是在几个地方重新计算表达式。贷款计算示例有一个你可以消除的公共子表达式。这是原始代码：

```java
// 公共子表达式示例
payment = loanAmount / (
      ( 1.0 - Math.pow( 1.0 + ( interestRate / 12.0 ), -months ) ) /
      ( interestRate / 12.0 )
   );
```

在这个例子中，你可以将interestRate/12.0赋给一个变量，然后引用该变量两次而不是计算表达式两次。如果你选择了良好的变量名，这种优化可以在提高性能的同时提高代码的可读性。这是修改后的代码：

```java
// 消除公共子表达式的示例
monthlyInterest = interestRate / 12.0;
payment = loanAmount / (
      ( 1.0 - Math.pow( 1.0 + monthlyInterest, -months ) ) /
      monthlyInterest
   );
```

## 例程

代码调优中最强大的工具之一是良好的例程分解。小的、定义良好的例程节省空间，因为它们取代了在多个地方单独做工作的方式。它们使程序易于优化，因为你可以在一个例程中重构代码，从而改进调用它的每个例程。小的例程相对容易用低级语言重写。长而曲折的例程本身就足够难理解；在像汇编器这样的低级语言中，它们是不可能的。

### 内联重写例程

在计算机编程的早期，一些机器对调用例程施加了禁止的性能惩罚。调用例程意味着操作系统必须换出程序，换入例程目录，换入特定例程，执行例程，换出例程，然后将调用例程换回。所有这些交换消耗了资源，使程序变慢。

在某些情况下，你可能通过像C++的inline关键字这样的语言特性将例程中的代码直接放在需要的地方来节省几纳秒。如果你使用的语言不支持直接内联但确实有宏预处理器，你可以使用宏来放入代码，根据需要切换进去和出来。但现代机器——"现代"意味着你可能永远不会在上面工作的任何机器——对调用例程几乎不施加惩罚。正如例子所示，通过保持代码内联，你很可能降低性能而不是优化它。

## 用低级语言重编码

一个长期存在的传统智慧不应被遗漏，即当你遇到性能瓶颈时，你应该用低级语言重编码。如果你在C++中编码，低级语言可能是汇编器。如果你在Python中编码，低级语言可能是C。用低级语言重编码倾向于同时提高速度和代码大小。以下是使用低级语言优化的典型方法：

1. 用高级语言编写100%的应用程序。
2. 完全测试应用程序，并验证它是正确的。
3. 如果之后需要性能改进，分析应用程序以识别热点。由于约5%的程序通常占约50%的运行时间，你通常可以将程序的小片段识别为热点。

   交叉引用

   有关程序的一小部分占大部分运行时间的现象的详细信息，请参见代码调优简介中的"帕累托原则"。

4. 用低级语言重编码几个小片段以提高整体性能。

你是否遵循这个久经考验的方法取决于你对低级语言的舒适程度、问题对低级语言的适合程度，以及你的绝望程度。我在第25章提到的数据加密标准程序上第一次接触到这种技术。我尝试了我听说过的每一种优化，程序仍然比速度目标慢两倍。用汇编器重编码程序的部分是唯一剩下的选择。作为汇编器新手，大约我所做的只是从高级语言到汇编器的直接翻译，但我甚至在那个初级水平上也获得了50%的改进。

假设你有一个将二进制数据转换为大写ASCII字符的例程。下一个示例显示了执行此操作的Delphi代码：

```delphi
// 更适合汇编器的代码示例
procedure HexExpand(
   var source: ByteArray;
   var target: WordArray;
   byteCount: word
);
var
   index: integer;
   lowerByte: byte;
   upperByte: byte;
   targetIndex: integer;
begin
   targetIndex := 1;
   for index := 1 to byteCount do begin
      target[ targetIndex ] := ( (source[ index ] and $F0) shr 4 ) + $41;
      target[ targetIndex+1 ] := (source[ index ] and $0f) + $41;
      targetIndex := targetIndex + 2;
   end;
end;
```

虽然很难看出这段代码的脂肪在哪里，但它包含很多位操作，这不是Delphi的专长。然而，位操作是汇编器的专长，所以这段代码是重编码的好候选。以下是汇编器代码：

```asm
// 用汇编器重编码的例程示例
procedure HexExpand(
   var source;
   var target;
   byteCount : Integer
);
   label
   EXPAND;

   asm
         MOV   ECX,byteCount      // 加载要扩展的字节数
         MOV   ESI,source         // 源偏移量
         MOV   EDI,target         // 目标偏移量
         XOR   EAX,EAX            // 将数组偏移量清零

   EXPAND:
         MOV   EBX,EAX            // 数组偏移量
         MOV   DL,[ESI+EBX]       // 获取源字节
         MOV   DH,DL              // 复制源字节

         AND   DH,$F              // 获取高位
         ADD   DH,$41             // 加65转换为大写

         SHR   DL,4               // 将低位移动到位
         AND   DL,$F              // 获取低位
         ADD   DL,$41             // 加65转换为大写
         SHL   BX,1               // 加倍目标数组偏移量的偏移量
         MOV   [EDI+EBX],DX       // 放入目标字

         INC   EAX                // 增加数组偏移量
         LOOP  EXPAND             // 重复直到完成
   end;
```

在这种情况下用汇编器重写是有利可图的，导致时间节省41%。可以合理地假设，在更适合位操作的语言（如C++）中的代码收益会比Delphi代码少。以下是结果：

| 语言 | 高级语言时间 | 汇编器时间 | 时间节省 |
|------|-------------|------------|----------|
| C++ | 4.25 | 3.02 | 29% |
| Delphi | 5.18 | 3.04 | 41% |

"之前"图片反映了两种语言在位操作方面的优势。"之后"图片看起来几乎相同，而且汇编器代码似乎最小化了Delphi和C++之间的初始性能差异。

汇编器例程表明用汇编器重写不一定产生巨大而丑陋的例程。这样的例程通常相当适中，就像这个。有时汇编器代码几乎与高级语言等效代码一样紧凑。

用汇编器重编码的一个相对简单有效的策略是从编译器开始，该编译器将汇编器列表作为编译的副产品生成。提取你需要调优的例程的汇编器代码，并将其保存在单独源文件中。使用编译器的汇编器代码作为基础，手动优化代码，在每一步检查正确性并测量改进。有些编译器在汇编器代码中将高级语言语句作为注释插入。如果你的编译器这样做，你可能将它们保留在汇编器代码中作为文档。

## 变化越多，保持不变的越多

你可能期望自我在《代码大全》第一版中写下这些内容以来，系统的性能属性在某些方面有所改变，而且确实如此。计算机速度快得多，内存也更丰富。在第一版中，我运行了本章大部分测试10,000到50,000次以获得有意义的、可测量的结果。在本版中，我必须运行大部分测试100万到1亿次。当你必须运行测试1亿次才能获得可测量的结果时，你会问是否有人会在真实程序中注意到影响。计算机已经变得如此强大，以至于对于许多常见类型的程序，本章讨论的性能优化水平已经变得无关紧要。

在其他方面，性能问题几乎完全没有改变。编写桌面应用程序的人可能不需要这些信息，但为嵌入式系统、实时系统以及其他具有严格速度或空间限制的系统编写软件的人仍然可以从中受益。

自Donald Knuth于1971年发表他对Fortran程序的研究以来，每次代码调优尝试都必须测量其影响一直是常数。根据本章中的测量，任何特定优化的效果实际上比10年前更不可预测。每次代码调优的效果受到编程语言、编译器、编译器版本、代码库、库版本和编译器设置等因素的影响。

代码调优总是涉及复杂性、可读性、简单性和可维护性之间的权衡，以及提高性能的愿望。它引入了高度维护开销，因为所有需要的重新分析。

我发现坚持可测量的改进是抵抗过早优化的诱惑的好方法，也是强制对清晰、直接代码的偏好的好方法。如果优化足够重要以至于值得拿出分析器并测量优化的效果，那么只要它有效，它可能就足够重要以至于值得保留。但如果不是优化足够重要以至于值得拿出分析设备，那么它就不重要到足以降低可读性、可维护性和其他代码特性。未经测量的代码调优对性能的影响充其量是推测性的，而对可读性的影响是确定的和有害的。

## 关键要点

- 优化结果因不同语言、编译器、环境而差异很大。没有测量每个特定优化，你就不知道它会帮助还是损害你的程序。
- 第一次优化通常不是最好的。即使在找到一个好的之后，继续寻找更好的。
- 代码调优有点像核能。这是一个有争议的、情绪化的话题。有些人认为它对可靠性和可维护性非常不利，以至于他们根本不会做。其他人认为有了适当的保障，它是有益的。如果你决定使用本章中的技术，要谨慎应用。
