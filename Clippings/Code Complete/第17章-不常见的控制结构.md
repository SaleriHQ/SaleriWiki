## 子程序的多重返回

大多数语言支持某种从子程序中途退出的方式。return 和 exit 语句是控制结构，使程序能够随意从子程序退出。它们使子程序通过正常的退出通道终止，将控制权返回给调用子程序。这里 return 是作为 C++ 和 Java 中的 return、Microsoft Visual Basic 中的 Exit Sub 和 Exit Function、以及类似结构的通用术语。以下是使用 return 语句的指导原则：

**在能增强可读性时使用 return**。在某些子程序中，一旦你知道答案，就想立即将它返回给调用子程序。如果子程序的定义方式使得它在检测到错误后不需要任何进一步的清理，那么不立即返回意味着你必须编写更多的代码。

以下是子程序中多个返回位置有意义的良好示例：

示例 17-1. C++ 中良好的多重返回子程序示例

```cpp
Comparison Compare( int value1, int value2 ) {       <-- 1
   if ( value1 < value2 ) {
      return Comparison_LessThan;
   }
   else if ( value1 > value2 ) {
      return Comparison_GreaterThan;
   }
   return Comparison_Equal;
}
```

(1)此子程序返回一个 Comparison 枚举类型。

其他示例不太明确，如下一节所示。

**使用守卫子句（提前返回或退出）来简化复杂的错误处理**。需要在执行名义操作之前检查大量错误条件的代码可能导致深度嵌套的代码，并使名义情况变得模糊，如下所示：

示例 17-2. Visual Basic 中使名义情况变得模糊的代码

```vb
If file.validName() Then
   If file.Open() Then
      If encryptionKey.valid() Then
         If file.Decrypt( encryptionKey ) Then
            ' 大量代码       <-- 1
            ...
         End If
      End If
   End If
End If
```

(1)这是名义情况的代码。

将子程序的主体嵌套在四个 if 语句中在美学上很难看，特别是如果最内层 if 语句中有大量代码。在这种情况下，如果首先检查错误情况，为名义路径扫清道路，代码的流程有时会更清晰。以下是它的样子：

示例 17-3. 使用守卫子句简化名义情况的简单 Visual Basic 代码

```vb
' 设置，如果发现错误则退出
If Not file.validName() Then Exit Sub
If Not file.Open() Then Exit Sub
If Not encryptionKey.valid() Then Exit Sub
If Not file.Decrypt( encryptionKey ) Then Exit Sub

' 大量代码
...
```

这个简单的代码使这项技术看起来是一个整洁的解决方案，但生产代码通常在检测到错误条件时需要更广泛的清理工作。以下是一个更现实的例子：

示例 17-4. 使用守卫子句简化名义情况的更现实的 Visual Basic 代码

```vb
' 设置，如果发现错误则退出
If Not file.validName() Then
   errorStatus = FileError_InvalidFileName
   Exit Sub
End If

If Not file.Open() Then
   errorStatus = FileError_CantOpenFile
   Exit Sub
End If

If Not encryptionKey.valid() Then
   errorStatus = FileError_InvalidEncryptionKey
   Exit Sub
End If

If Not file.Decrypt( encryptionKey ) Then
   errorState = FileError_CantDecryptFile
   Exit Sub
End If

' 大量代码       <-- 1
...

(1)这是名义情况的代码。
```

对于生产规模的代码，Exit Sub 方法在处理名义情况之前会产生相当多的代码。但是，Exit Sub 方法确实避免了第一个示例的深度嵌套，而且，如果第一个示例被扩展为显示设置 errorStatus 变量，Exit Sub 方法在保持相关语句在一起方面会做得更好。当一切尘埃落定，Exit Sub 方法确实看起来更易读和可维护，只不过优势不大。

**在每个子程序中最小化返回的数量**。当你从底部阅读一个子程序却不知道它可能在上面的某个地方返回时，就更难理解它了。因此，谨慎使用返回——仅在能提高可读性时使用。

## 递归

在递归中，一个子程序自己解决一个小问题，然后将问题分成更小的块，再调用自身来解决每个更小的块。递归通常在问题的某小部分容易解决而大部分容易分解成更小块时派上用场。

递归并不经常有用，但谨慎使用它会产生优雅的解决方案，例如排序算法巧妙使用递归的这个例子：

示例 17-5. 使用递归的排序算法的 Java 示例

```java
void QuickSort( int firstIndex, int lastIndex, String [] names ) {
   if ( lastIndex > firstIndex ) {
      int midPoint = Partition( firstIndex, lastIndex, names );
      QuickSort( firstIndex, midPoint-1, names );       <-- 1
      QuickSort( midPoint+1, lastIndex, names )       <-- 1
   }
}
```

(1)这里是递归调用。

在这种情况下，排序算法将数组切成两半，然后调用自身对数组的每一半进行排序。当它用太小的子数组调用自身时——比如（lastIndex <= firstIndex）——它停止调用自身。

对于一小部分问题，递归可以产生简单、优雅的解决方案。对于稍大一些的问题，它可能产生简单、优雅但难以理解的解决方案。对于大多数问题，它会产生极其复杂的解决方案——在这些情况下，简单的迭代通常更容易理解。谨慎选择使用递归。

## 递归示例

假设你有一个表示迷宫的数据类型。迷宫基本上是一个网格，在网格的每个点上，你可能能够向左转、向右转、向上或向下。你通常能够朝不止一个方向移动。

你如何编写一个程序来找到穿过迷宫的路，如图 17-1 所示？如果你使用递归，答案相当直接。你从起点开始，然后尝试所有可能的路径直到找到迷宫出口。第一次访问一个点时，你尝试向左移动。如果不能向左，就尝试向上或向下，如果也不能向上或向下，就尝试向右。你不必担心迷路，因为你在访问每个点时会撒下一些面包屑，而且你不会两次访问同一个点。

> **图 17-1**。递归可以成为对抗复杂性的有价值的工具——当用于攻击合适的问题时

递归代码看起来像这样：

示例 17-6. 通过递归移动穿过迷宫的 C++ 示例

```cpp
bool FindPathThroughMaze( Maze maze, Point position ) {
   // 如果这个位置已经被尝试过，就不要再尝试
   if ( AlreadyTried( maze, position ) ) {
      return false;
   }

   // 如果这个位置是出口，宣布成功
   if ( ThisIsTheExit( maze, position ) ) {
      return true;
   }

   // 记住这个位置已经被尝试过
   RememberPosition( maze, position );

   // 检查向左、向上、向下和向右的路径；如果任何路径成功，停止寻找
   if ( MoveLeft( maze, position, &newPosition ) ) {
      if ( FindPathThroughMaze( maze, newPosition ) ) {
         return true;
      }
   }

   if ( MoveUp( maze, position, &newPosition ) ) {
      if ( FindPathThroughMaze( maze, newPosition ) ) {
         return true;
      }
   }

   if ( MoveDown( maze, position, &newPosition ) ) {
      if ( FindPathThroughMaze( maze, newPosition ) ) {
         return true;
      }
   }

   if ( MoveRight( maze, position, &newPosition ) ) {
      if ( FindPathThroughMaze( maze, newPosition ) ) {
         return true;
      }
   }
   return false;
}
```

第一行代码检查该位置是否已经被尝试过。编写递归子程序的一个关键目标是防止无限递归。在这种情况下，如果你不检查是否已经尝试过一个点，你可能会无限地尝试它。

第二个语句检查该位置是否是迷宫的出口。如果 ThisIsTheExit() 返回 true，子程序本身就返回 true。

第三个语句记住该位置已被访问。这防止了由循环路径导致的无限递归。

子程序中剩余的行尝试向左、向上、向下和向右找到路径。如果子程序曾经返回 true（即当子程序找到穿过迷宫的路时），代码停止递归。

这个子程序中使用的逻辑相当直接。大多数人在最初使用递归时会感到一些不适，因为它是自引用的。然而，在这种情况下，替代解决方案会更加复杂，而递归效果很好。

## 使用递归的技巧

使用递归时记住以下技巧：

**确保递归停止**。检查子程序以确保它包含一个非递归路径。这通常意味着子程序有一个测试，当不需要时停止进一步的递归。在迷宫示例中，对 AlreadyTried() 和 ThisIsTheExit() 的测试确保递归停止。

**使用安全计数器来防止无限递归**。如果你在不允许像刚才描述的简单测试的情况下使用递归，使用安全计数器来防止无限递归。安全计数器必须是一个每次调用子程序时不会被重新创建的变量。使用类成员变量或将安全计数器作为参数传递。以下是一个示例：

示例 17-7. 使用安全计数器防止无限递归的 Visual Basic 示例

```vb
Public Sub RecursiveProc( ByRef safetyCounter As Integer )       <-- 1
   If ( safetyCounter > SAFETY_LIMIT ) Then
      Exit Sub
   End If
   safetyCounter = safetyCounter + 1
   ...
   RecursiveProc( safetyCounter )
End Sub
```

(1)递归子程序必须能够更改 safetyCounter 的值，所以在 Visual Basic 中它是一个 ByRef 参数。

在这种情况下，如果子程序超过安全限制，它就停止递归。

如果你不想将安全计数器作为显式参数传递，你可以在 C++、Java 或 Visual Basic 中使用成员变量，或者在其他语言中使用等效方法。

**将递归限制在一个子程序中**。循环递归（A 调用 B 调用 C 调用 A）是危险的，因为很难检测。在一个子程序中管理递归已经够难了；理解跨子程序的递归就太过分了。如果你有循环递归，你通常可以重新设计子程序，使递归限制在单个子程序中。如果你做不到，而你仍然认为递归是最好的方法，使用安全计数器作为递归保险单。

**密切关注栈**。使用递归，你无法保证程序使用多少栈空间，而且很难提前预测程序在运行时的行为。但是，你可以采取一些步骤来控制其运行时行为。

首先，如果你使用安全计数器，在为其设置限制时应考虑你愿意为递归子程序分配多少栈。设置足够低的安全限制以防止栈溢出。

其次，注意递归函数中局部变量的分配，特别是内存密集型对象。换句话说，使用 new 在堆上创建对象，而不是让编译器在栈上创建自动对象。

**不要使用递归计算阶乘或斐波那契数列**。计算机科学教科书的一个问题是它们呈现了愚蠢的递归示例。典型的例子是计算阶乘或计算斐波那契数列。递归是一个强大的工具，用在这两个案例中的任何一个都是非常愚蠢的。如果为我工作的程序员使用递归来计算阶乘，我会雇别人。以下是阶乘子程序的递归版本：

示例 17-8. 不适当使用递归计算阶乘的 Java 示例

```java
int Factorial( int number ) {
   if ( number == 1 ) {
      return 1;
   }
   else {
      return number * Factorial( number - 1 );
   }
}
```

除了运行速度慢且使运行时内存的使用不可预测之外，这个子程序的递归版本比迭代版本更难理解，如下所示：

示例 17-9. 使用迭代计算阶乘的适当 Java 示例

```java
int Factorial( int number ) {
   int intermediateResult = 1;
   for ( int factor = 2; factor <= number; factor++ ) {
      intermediateResult = intermediateResult * factor;
   }
   return intermediateResult;
}
```

从这个例子中可以得出三个教训。首先，计算机科学教科书用它们的递归示例并没有给世界带来好处。其次，更重要的是，递归是一个比其在计算阶乘或斐波那契数列中的混乱使用所暗示的更强大的工具。第三，也是最重要的，你应该在使用递归之前考虑替代方案。你可以用栈和迭代做你用递归做的任何事情。有时一种方法效果更好；有时另一种更好。在选择之前两者都要考虑。

## 反对 goto 的论点

反对 goto 的一般论点是，没有 goto 的代码是更高质量的代码。引发这场争议的著名信件是 Edsger Dijkstra 1968 年 3 月发表在《ACM 通讯》上的"Go To Statement Considered Harmful"。Dijkstra 观察到代码质量与程序员使用的 goto 数量成反比。在随后的工作中，Dijkstra 认为不含 goto 的代码可以更容易地被证明是正确的。

包含 goto 的代码很难格式化。缩进应该用来显示逻辑结构，而 goto 对逻辑结构有影响。然而，用缩进来显示 goto 及其目标的逻辑结构是困难的或不可能的。

使用 goto 会阻止编译器优化。一些优化依赖于程序的控制流位于少数几条语句内。无条件 goto 使控制流更难分析，并降低编译器优化代码的能力。因此，即使引入 goto 在源代码层面产生了效率，它很可能通过阻止编译器优化来降低整体效率。

goto 的支持者有时争辩说它们使代码更快或更小。但包含 goto 的代码很少是最快或最小的。Donald Knuth 的精彩经典文章"Structured Programming with go to Statements"给出了几个例子，说明使用 goto 会导致更慢和更大的代码（Knuth 1974）。

实际上，使用 goto 会导致违反代码应该严格从上到下流动的原则。即使小心使用 goto 不会造成混淆，一旦引入 goto，它们就会像白蚁通过腐烂的房子一样在代码中蔓延。如果允许任何 goto，坏的会与好的一起渗入，所以最好不允许任何 goto。

总体而言，Dijkstra 的信发表后二十年的经验表明，产生充斥 goto 的代码是愚蠢的。在对文献的调查中，Ben Shneiderman 得出结论，证据支持 Dijkstra 的观点，即我们最好没有 goto（1980），许多现代语言，包括 Java，根本没有 goto。

## 支持 goto 的论点

支持 goto 的论点以在特定情况下谨慎使用而不是不加选择地使用为特征。大多数反对 goto 的论点都是反对不加选择地使用的。goto 争议爆发时，Fortran 是最流行的语言。Fortran 没有像样的循环结构，在没有关于用 goto 编程循环的良好建议的情况下，程序员写了很多意大利面代码。这样的代码无疑与生产低质量程序相关，但它与谨慎使用 goto 来弥补现代语言能力中的空白关系不大。

**一个位置得当的 goto 可以消除重复代码的需要**。重复代码如果在两组代码被不同修改时会带来问题。重复代码增加了源代码和可执行文件的大小。在这种情况下，goto 的坏处被重复代码的风险所抵消。

**goto 在分配资源、执行操作然后释放资源的子程序中很有用**。使用 goto，你可以在代码的一个部分进行清理。goto 减少了在检测到错误的每个地方忘记释放资源的可能性。

交叉参考

有关在分配资源的代码中使用 goto 的详细信息，请参见本节中的"错误处理和 goto"。另请参阅异常处理中的讨论。

**在某些情况下，goto 可以产生更快更小的代码**。Knuth 1974 年的文章引用了几个 goto 产生合法收益的例子。

**好的编程并不意味着消除 goto**。有条不紊的分解、细化和控制结构的选择在大多数情况下会自动导致无 goto 的程序。实现无 goto 的代码不是目标而是结果，把重点放在避免 goto 上并没有帮助。

数十年来对 goto 的研究未能证明它们是有害的。在对文献的调查中，B. A. Sheil 得出结论：

不切实际的测试条件、糟糕的数据分析和不确定的结果无法支持 Shneiderman 等人的说法，即代码中的错误数量与 goto 数量成正比（1981）。Sheil 没有走得太远以至于得出使用 goto 是一个好主意的结论——不如说，针对它们的实验证据并不具有结论性。

证据表明，只有故意混乱的控制结构才会降低 [程序员] 的表现。这些实验几乎没有为任何特定方法的控制流结构化的有益效果提供证据。
— B. A. Sheil

最后，goto 已被纳入许多现代语言，包括 Visual Basic、C++ 和 Ada 语言，这是历史上最精心设计的编程语言。Ada 是在 goto 辩论双方的观点完全展开之后很久才开发的，在考虑了问题的方方面面之后，Ada 的工程师决定包含 goto。

## 伪 goto 辩论

大多数 goto 讨论的一个主要特征是对问题采取了肤浅的方法。"goto 是邪恶的"一方的争论者呈现了一个使用 goto 的琐碎代码片段，然后展示重写该片段而不用 goto 是多么容易。这主要证明了不用 goto 编写琐碎代码很容易。

"我离不开 goto"一方的争论者通常呈现一个消除 goto 会导致额外比较或代码行重复的情况。这主要证明了在这种情况下使用 goto 会减少一次比较——在今天的计算机上不是一个显著的收益。

大多数教科书没有帮助。它们提供了一个不用 goto 重写某些代码的简单例子，就好像这涵盖了主题。以下是来自此类教科书的一个伪装的简单代码示例：

示例 17-10. C++ 中据说容易不用 goto 重写的代码示例

```cpp
do {
   GetData( inputFile, data );
   if ( eof( inputFile ) ) {
      goto LOOP_EXIT;
   }
   DoSomething( data );
} while ( data != -1 );
LOOP_EXIT:
```

该书很快用无 goto 的代码替换了这段代码：

示例 17-11. C++ 中据说等效的、不用 goto 重写的代码示例

```cpp
GetData( inputFile, data );
while ( ( !eof( inputFile ) ) && ( ( data != -1 ) ) ) {
   DoSomething( data );
   GetData( inputFile, data )
}
```

这个所谓的"简单"示例包含一个错误。在 data 等于 -1 进入循环的情况下，转换后的代码在执行 DoSomething() 之前检测 -1 并退出循环。原始代码在检测 -1 之前执行 DoSomething()。试图展示不用 goto 编程有多容易的编程书错误地翻译了它自己的例子。但这本书的作者不应该感到太糟糕；其他书也犯了类似的错误。即使专业人士在翻译使用 goto 的代码时也有困难。

以下是不使用 goto 的忠实翻译：

示例 17-12. C++ 中真正等效的、不用 goto 重写的代码示例

```cpp
do {
   GetData( inputFile, data );
   if ( !eof( inputFile )) {
      DoSomething( data );
   }
} while ( ( data != -1 ) && ( !eof( inputFile ) ) );
```

即使对代码进行了正确的翻译，这个例子仍然是假的，因为它展示了对 goto 的微不足道的使用。这样的情况不是深思熟虑的程序员选择 goto 作为他们首选控制形式的那些情况。

在这个晚期，很难在理论上的 goto 辩论中添加任何有价值的东西。然而，通常没有解决的问题是这样一个情况：程序员完全了解无 goto 的替代方案，选择使用 goto 来增强可读性和可维护性。

以下部分呈现了一些有经验的程序员主张使用 goto 的情况。这些讨论提供了使用 goto 的代码和不使用 goto 重写的代码的示例，并评估了版本之间的权衡。

## 错误处理和 goto

编写高度交互式代码需要大量关注错误处理和在错误发生时清理资源。以下代码示例清除一组文件。该子程序首先获取要清除的文件组，然后找到每个文件，打开它，覆盖它，然后擦除它。子程序在每一步检查错误。

示例 17-13. 使用 goto 处理错误和清理资源的 Visual Basic 代码

```vb
' 此子程序清除一组文件。
Sub PurgeFiles( ByRef errorState As Error_Code )
  Dim fileIndex As Integer
  Dim fileToPurge As Data_File
  Dim fileList As File_List
  Dim numFilesToPurge As Integer

  MakePurgeFileList( fileList, numFilesToPurge )

  errorState = FileStatus_Success
  fileIndex = 0
  While ( fileIndex < numFilesToPurge )
     fileIndex = fileIndex + 1
     If Not ( FindFile( fileList( fileIndex ), fileToPurge ) ) Then
        errorState = FileStatus_FileFindError
        GoTo END_PROC       <-- 1
     End If

    If Not OpenFile( fileToPurge ) Then
       errorState = FileStatus_FileOpenError
       GoTo END_PROC       <-- 2
    End If

    If Not OverwriteFile( fileToPurge ) Then
       errorState = FileStatus_FileOverwriteError
       GoTo END_PROC       <-- 3
    End If

    if Not Erase( fileToPurge ) Then
       errorState = FileStatus_FileEraseError
       GoTo END_PROC       <-- 4
    End If
 Wend

END_PROC:       <-- 5
   DeletePurgeFileList( fileList, numFilesToPurge )
End Sub
```

(1)这里是 GoTo。

(2)这里是 GoTo。

(3)这里是 GoTo。

(4)这里是 GoTo。

(5)这里是 GoTo 标签。

这个子程序是有经验的程序员决定使用 goto 的典型情况。类似的情况出现在子程序需要分配和清理数据库连接、内存或临时文件等资源时。在这些情况下，避免 goto 的替代方案通常是重复清理资源的代码。在这样的情况下，程序员可能会平衡 goto 的坏处和重复代码维护的麻烦，并决定 goto 是较小的坏事。

你可以用几种方式重写前面的子程序来避免 goto，两种方式都涉及权衡。以下是可能的重写策略：

**用嵌套 if 语句重写**。要使用嵌套 if 语句重写，嵌套 if 语句，使得每个仅在前一个测试成功时才执行。这是消除 goto 的标准教科书编程方法。以下是使用标准方法重写的子程序：

示例 17-14. 使用嵌套 if 避免 goto 的 Visual Basic 代码

```vb
' 此子程序清除一组文件。
Sub PurgeFiles( ByRef errorState As Error_Code )
   Dim fileIndex As Integer
   Dim fileToPurge As Data_File
   Dim fileList As File_List
   Dim numFilesToPurge As Integer

   MakePurgeFileList( fileList, numFilesToPurge )

   errorState = FileStatus_Success
   fileIndex = 0
   While ( fileIndex < numFilesToPurge And errorState = FileStatus_Success )       <-- 1
      fileIndex = fileIndex + 1

      If FindFile( fileList( fileIndex ), fileToPurge ) Then
         If OpenFile( fileToPurge ) Then
            If OverwriteFile( fileToPurge ) Then
               If Not Erase( fileToPurge ) Then
                  errorState = FileStatus_FileEraseError
               End If
            Else ' 无法覆盖文件
               errorState = FileStatus_FileOverwriteError
            End If
         Else ' 无法打开文件
            errorState = FileStatus_FileOpenError
         End If
      Else ' 无法找到文件

         errorState = FileStatus_FileFindError       <-- 2
      End If
   Wend
   DeletePurgeFileList( fileList, numFilesToPurge )
End Sub
```

(1)While 测试已更改，添加了对 errorState 的测试。

(2)这行距离调用它的 If 语句 13 行。

交叉参考

这个子程序也可以用 break 重写而不用 goto。关于该方法的详细信息，请参见控制循环中的"提前退出循环"。

对于习惯不用 goto 编程的人来说，这段代码可能比 goto 版本更容易阅读，如果你使用它，你就不必面对 goto 纠察队的质问。

这种嵌套 if 方法的主要缺点是嵌套深度很深，非常深。要理解代码，你必须同时将整套嵌套的 if 记在脑中。而且，错误处理代码与调用它的代码之间的距离太远：例如，将 errorState 设置为 FileStatus_FileFindError 的代码距离调用它的 if 语句 13 行。

交叉参考

有关缩进和其他编码布局问题的更多详细信息，请参见第31章。关于嵌套级别的详细信息，请参见驯服危险的深度嵌套。

使用 goto 版本，没有语句距离调用它的条件超过四行。而且你不必同时将整个结构记在脑中。你基本上可以忽略任何已成功的前置条件，专注于下一个操作。在这种情况下，goto 版本比嵌套 if 版本更易读和可维护。

**用状态变量重写**。要使用状态变量（也称为状态变量）重写，创建一个指示子程序是否处于错误状态的变量。在这种情况下，子程序已经使用了 errorState 状态变量，所以你可以使用它。

示例 17-15. 使用状态变量避免 goto 的 Visual Basic 代码

```vb
' 此子程序清除一组文件。
Sub PurgeFiles( ByRef errorState As Error_Code )
   Dim fileIndex As Integer
   Dim fileToPurge As Data_File
   Dim fileList As File_List
   Dim numFilesToPurge As Integer

   MakePurgeFileList( fileList, numFilesToPurge )

   errorState = FileStatus_Success
   fileIndex = 0

   While ( fileIndex < numFilesToPurge ) And ( errorState = FileStatus_Success )
      fileIndex = fileIndex + 1

      If Not FindFile( fileList( fileIndex ), fileToPurge ) Then
         errorState = FileStatus_FileFindError

      End If
      If ( errorState = FileStatus_Success ) Then       <-- 2
         If Not OpenFile( fileToPurge ) Then
            errorState = FileStatus_FileOpenError
       End If
      End If
      If ( errorState = FileStatus_Success ) Then       <-- 3
         If Not OverwriteFile( fileToPurge ) Then
            errorState = FileStatus_FileOverwriteError
         End If
      End If
      If ( errorState = FileStatus_Success ) Then       <-- 4
         If Not Erase( fileToPurge ) Then
            errorState = FileStatus_FileEraseError
         End If
      End If
   Wend
   DeletePurgeFileList( fileList, numFilesToPurge )
End Sub
```

(1)While 测试已更改，添加了对 errorState 的测试。

(2)测试状态变量。

(3)测试状态变量。

(4)测试状态变量。

状态变量方法的优势是它避免了第一种重写中深度嵌套的 if-then-else 结构，因此更容易理解。它还将 if-then-else 测试后的动作放在比嵌套 if 方法更靠近测试的位置，并且完全避免了 else 子句。

理解嵌套 if 版本需要一些脑筋急转弯。状态变量版本更容易理解，因为它非常接近人们思考问题的方式。你找到文件。如果一切正常，你打开文件。如果一切仍然正常，你覆盖文件。如果一切仍然正常……

这种方法的缺点是使用状态变量并不像它应该的那样普遍。充分记录它们的使用，否则一些程序员可能不理解你在做什么。在这个例子中，使用命名良好的枚举类型帮助很大。

**用 try-finally 重写**。一些语言，包括 Visual Basic 和 Java，提供 try-finally 语句，可用于在错误条件下清理资源。

要使用 try-finally 方法重写，将原本需要检查错误的代码包含在 try 块中，并将清理代码放在 finally 块中。try 块指定异常处理的范围，finally 块执行任何资源清理。finally 块将始终被调用，无论是否抛出异常，也无论 PurgeFiles() 子程序是否捕获任何抛出的异常。

示例 17-16. 使用 try-finally 避免 goto 的 Visual Basic 代码

```vb
' 此子程序清除一组文件。异常传递给调用者。
Sub PurgeFiles()
   Dim fileIndex As Integer
   Dim fileToPurge As Data_File
   Dim fileList As File_List
   Dim numFilesToPurge As Integer
   MakePurgeFileList( fileList, numFilesToPurge )
   Try
      fileIndex = 0
      While ( fileIndex < numFilesToPurge )
         fileIndex = fileIndex + 1
         FindFile( fileList( fileIndex ), fileToPurge )
         OpenFile( fileToPurge )
         OverwriteFile( fileToPurge )
         Erase( fileToPurge )
      Wend
   Finally
      DeletePurgeFileList( fileList, numFilesToPurge )
   End Try
End Sub
```

这种方法假设所有函数调用在失败时抛出异常而不是返回错误代码。

try-finally 方法的优势是它比 goto 方法更简单，不使用 goto。它还避免了深度嵌套的 if-then-else 结构。

try-finally 方法的局限性是它必须在整个代码库中一致实现。如果前面的代码是已经同时使用错误代码和异常之外的代码库的一部分，异常代码需要为每个可能的错误设置错误代码，这一要求会使代码与其他方法一样复杂。

## 方法比较

这四种方法各有其优点。goto 方法避免了深度嵌套和不必要的测试，但当然有 goto。嵌套 if 方法避免了 goto 但深度嵌套，并夸大了子程序的逻辑复杂性。状态变量方法避免了 goto 和深度嵌套，但引入了额外的测试。try-finally 方法避免了 goto 和深度嵌套，但在所有语言中都不可用。

交叉参考

有关可应用于这种情况的完整技术列表，请参见驯服危险的深度嵌套中的"减少深度嵌套的技术总结"。

try-finally 方法在提供 try-finally 的语言和尚未标准化其他方法的代码库中是最直接的。如果 try-finally 不是选项，状态变量方法比 goto 和嵌套 if 方法略胜一筹，因为它更易读，而且更好地模拟问题，但这并不能使其成为所有情况下的最佳方法。

任何这些技术在一致地应用于项目中所有代码时都效果很好。考虑所有权衡，然后对在项目中首选哪种方法做出项目范围的决策。

## goto 和 else 子句中的代码共享

一些程序员会使用 goto 的一个具有挑战性的情况是：你有两个条件测试和一个 else 子句，你想在一个条件中和 else 子句中执行代码。以下是一个可能驱使某人使用 goto 的情况的示例：

示例 17-17. 在 else 子句中使用 goto 共享代码的 C++ 示例

```cpp
if ( statusOk ) {
   if ( dataAvailable ) {
      importantVariable = x;
      goto MID_LOOP;
   }
}
else {
   importantVariable = GetValue();

   MID_LOOP:

   // 大量代码
   ...
}
```

这是一个好例子，因为它在逻辑上很曲折——按原样几乎不可能阅读，而且不用 goto 重写它是困难的。如果你认为可以不用 goto 轻松重写它，请找人审查你的代码！几位专业程序员错误地重写了它。

你可以用几种方式重写代码。你可以重复代码，将公共代码放入一个子程序并从两个地方调用它，或者重新测试条件。在大多数语言中，重写会比原来的稍大稍慢，但它会非常接近。除非代码在一个真正热的循环中，否则不要考虑效率就直接重写。

最好的重写是将 // 大量代码部分放入它自己的子程序。然后你可以从你本可以使用作为 goto 原点或目的地的位置调用该子程序，并保留条件语句的原始结构。以下是它的样子：

示例 17-18. 通过将公共代码放入子程序在 else 子句中共享代码的 C++ 示例

```cpp
if ( statusOk ) {
   if ( dataAvailable ) {
      importantVariable = x;
      DoLotsOfCode( importantVariable );
   }
}
else {
   importantVariable = GetValue();
   DoLotsOfCode( importantVariable );
}
```

通常，编写一个新的子程序是最好的方法。然而，有时候将重复的代码放入它自己的子程序是不实际的。在这种情况下，你可以通过重构条件来绕过不切实际的解决方案，这样你就可以将代码保留在同一子程序中而不是放入新子程序：

示例 17-19. 不用 goto 在 else 子句中共享代码的 C++ 示例

```cpp
if ( ( statusOk && dataAvailable ) || !statusOk ) {
   if ( statusOk && dataAvailable ) {
      importantVariable = x;
   }
   else {
      importantVariable = GetValue();
   }

   // 大量代码
   ...
}
```

这是对 goto 版本中逻辑的忠实和机械翻译。它额外测试 statusOK 两次和 dataAvailable 一次，但代码是等效的。如果重新测试条件让你困扰，请注意在第一个 if 测试中不需要测试 statusOK 两次。你也可以在第二个 if 测试中删除对 dataAvailable 的测试。

## 使用 goto 的指导原则总结

使用 goto 是一个宗教问题。我的教条是在现代语言中，你可以轻松地将十分之九的 goto 替换为等效的顺序结构。在这些简单的情况下，你应该出于习惯替换 goto。在困难的情况下，你仍然可以在十分之九的情况下消除 goto：你可以将代码分解成更小的子程序，使用 try-finally，使用嵌套 if，测试和重新测试状态变量，或重构条件。消除 goto 在这些情况下更难，但这是好的脑力锻炼，而且本节讨论的技术为你提供了这样做的工具。

在剩下的百分之一的情况下，goto 是问题的合法解决方案，用它。穿上雨靴，绕过泥坑是值得的。但对其他程序员提出的无 goto 方法保持开放的心态。他们可能看到你看不到的东西。

以下是使用 goto 的指导原则总结：

- 在没有直接支持的构造的语言中，使用 goto 来模拟结构化控制构造。当你这样做时，精确地模拟它们。不要滥用 goto 给你的额外灵活性。
- 当有等效的内置构造时不要使用 goto。
- 衡量任何用于提高效率的 goto 的性能。在大多数情况下，你可以不用 goto 重写以提高可读性而不会损失效率。如果你的情况是例外，记录效率改进，这样无 goto 布道者看到它时不会删除 goto。
- 将每个子程序中的 goto 标签限制为一个，除非你是在模拟结构化构造。
- 将 goto 限制为向前跳转，而不是向后跳转，除非你是在模拟结构化构造。
- 确保所有 goto 标签都被使用。未使用的标签可能表明缺少代码，即 goto 标签指向的代码。如果标签未使用，删除它们。
- 确保 goto 不会创建不可达代码。
- 如果你是一个经理，采取这样的观点：关于单个 goto 的争论不值得失去这场战争。如果程序员知道替代方案并愿意争论，goto 可能是可以的。

## 关于不常见控制结构的观点

在某个时候，有人认为以下每个控制结构都是一个好主意：

- 无限制使用 goto
- 动态计算 goto 目标并跳转到计算位置的能力
- 使用 goto 从一个子程序的中间跳转到另一个子程序的中间的能力
- 用行号或标签调用子程序的能力，允许执行从子程序的中间开始
- 程序能够即时生成代码然后执行它刚写的代码的能力

在某个时候，这些想法中的每一个都被认为是可以接受的甚至是可取的，即使现在它们看起来都无可救药地过时了。软件开发领域的发展很大程度上是通过限制程序员可以用代码做什么实现的。因此，我对非传统的控制结构持强烈怀疑态度。我怀疑这一章中的大多数结构最终会与计算 goto 标签、可变子程序入口点、自我修改代码以及其他有利于灵活性和便利性而非结构和处理复杂性的能力的结构一起进入程序员的废料堆。

## 关键点

- 多重返回可以增强子程序的可读性和可维护性，并有助于防止深度嵌套的逻辑。然而，它们应该谨慎使用。
- 递归为一小部分问题提供了优雅的解决方案。使用时也要谨慎。
- 在少数情况下，goto 是编写可读和可维护代码的最佳方式。这种情况很少见。goto 只应作为最后手段使用。
