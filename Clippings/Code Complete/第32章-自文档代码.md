## 外部文档

软件项目上的文档包括源代码清单内部和外部的信息——通常以外部文档或单元开发文件夹的形式存在。在大型、正式的项目上，大部分文档位于源代码之外（Jones 1998）。与代码相比，外部构建文档处于较高层次；与问题定义、需求和架构活动的文档相比，则处于较低层次。

交叉引用

更多关于外部文档的内容，请参见 IEEE 标准。

**单元开发文件夹**。单元开发文件夹（UDF）或软件开发文件夹（SDF）是一种非正式文档，包含开发人员在构建期间使用的笔记。"单元"的定义比较宽松，通常指一个类，尽管它也可以指一个包或组件。UDF 的主要目的是提供设计决策的踪迹，而这些决策没有在其他地方记录。许多项目有标准来指定 UDF 的最少内容，例如相关需求的副本、该单元实现的高级设计部分、构建标准的副本、当前代码清单以及该单元开发人员的设计笔记。有时客户要求软件开发人员交付项目的 UDF；通常它们仅供内部使用。

进一步阅读

详细描述请参见"单元开发文件夹（UDF）：软件开发的有效管理工具"（Ingrassia 1976）或"单元开发文件夹（UDF）：十年回顾"（Ingrassia 1987）。

**详细设计文档**。详细设计文档是低层设计文档。它描述类级别或例程级别的设计决策、考虑过的替代方案以及选择所选方法的原因。有时此信息包含在正式文档中。在这种情况下，详细设计通常被认为是与构建分离的。有时它主要由收集到 UDF 中的开发人员笔记组成。有时——通常是——它仅存在于代码本身。

## 编程风格作为文档

与外部文档相反，内部文档是在程序清单本身中发现的。它是最详细的文档类型，处于源代码语句级别。因为它与代码最密切相关，内部文档也是在代码被修改时最有可能保持正确的文档类型。

代码级文档的主要贡献者不是注释，而是良好的编程风格。风格包括良好的程序结构、使用直接且易于理解的方法、好的变量名、好的例程名、使用命名常量而非字面量、清晰的布局以及最小化控制流和数据结构的复杂性。

这里是一个风格不佳的代码片段：

**风格不佳导致文档不良的 Java 示例**

```java
// 示例 32-1
for ( i = 2; i <= num; i++ ) {
    meetsCriteria[ i ] = true;
}
for ( i = 2; i <= num / 2; i++ ) {
    j = i + i;
    while ( j <= num ) {
        meetsCriteria[ j ] = false;
        j = j + i;
    }
}
for ( i = 1; i <= num; i++ ) {
    if ( meetsCriteria[ i ] ) {
        System.out.println ( i + " meets criteria." );
    }
}
```

你觉得这个例程是做什么的？它不必要地晦涩。它的文档不良不是因为缺少注释，而是因为缺少良好的编程风格。变量名没有提供信息，布局粗糙。下面是改进后的相同代码——仅改进编程风格就使它的含义更加清晰：

**无注释但有良好风格的文档化 Java 示例**

```java
// 示例 32-2
for ( primeCandidate = 2; primeCandidate <= num; primeCandidate++ ) {
    isPrime[ primeCandidate ] = true;
}

for ( int factor = 2; factor < ( num / 2 ); factor++ ) {
    int factorableNumber = factor + factor;
    while ( factorableNumber <= num ) {
        isPrime[ factorableNumber ] = false;
        factorableNumber = factorableNumber + factor;
    }
}

for ( primeCandidate = 1; primeCandidate <= num; primeCandidate++ ) {
   if ( isPrime[ primeCandidate ] ) {
      System.out.println( primeCandidate + " is prime." );
   }
}
```

与第一段代码不同，这段代码让你一眼就知道它与质数有关。再仔细一看，你发现它找出 1 到 Num 之间的所有质数。对于第一段代码，仅确定循环在哪里结束就需要超过两眼的观察。

这两段代码之间的差异与注释无关——两段代码都没有注释。然而，第二段代码更易读，接近可读性的圣杯：自文档代码。这样的代码依靠良好的编程风格来承担大部分文档负担。在编写良好的代码中，注释是可读性蛋糕上的糖霜。

### 自文档代码检查清单

**类**

- 类的接口是否呈现一致的抽象？
- 类是否命名良好，其名称是否描述了其核心目的？
- 类的接口是否明显地说明了如何使用该类？
- 类的接口是否足够抽象，使你不必考虑其服务是如何实现的？你能否将该类当作黑盒？

**例程**

- 每个例程的名称是否准确描述了该例程做什么？
- 每个例程是否执行一个定义良好的任务？
- 是否已将每个例程中受益于独立例程的部分放入独立的例程中？
- 每个例程的接口是否明显且清晰？

**数据名称**

- 类型名称是否足够描述性以帮助文档化数据声明？
- 变量命名是否良好？
- 变量是否仅用于其名称所描述的目的？
- 循环计数器是否被赋予了比 i、j 和 k 更有信息的名称？
- 是否使用了良好命名的枚举类型而非临时标志或布尔变量？
- 是否使用了命名常量而非幻数或幻字符串？
- 命名约定是否区分了类型名称、枚举类型、命名常量、局部变量、类变量和全局变量？

**数据组织**

- 是否在需要时使用额外的变量来增强清晰度？
- 对变量的引用是否紧密相邻？
- 数据类型是否简单以最小化复杂性？
- 复杂数据是否通过抽象访问例程（抽象数据类型）访问？

**控制**

- 通过代码的规范路径是否清晰？
- 相关语句是否分组在一起？
- 相对独立的语句组是否被封装到它们自己的例程中？
- 正常情况是否遵循 if 而非 else？
- 控制结构是否简单以最小化复杂性？
- 每个循环是否像定义良好的例程一样只执行一个功能？
- 是否最小化了嵌套？
- 是否通过使用额外的布尔变量、布尔函数和决策表简化了布尔表达式？

**布局**

- 程序的布局是否显示其逻辑结构？

**设计**

- 代码是否直接，避免聪明？
- 实现细节是否尽可能隐藏？
- 程序是否尽可能用问题领域而非计算机科学或编程语言结构的术语编写？

## 是否需要注释

注释容易写糟而非写好，注释可能弊大于利。关于注释优点的激烈讨论听起来常常像关于道德美德的哲学辩论，这让我想到如果苏格拉底是一个程序员，他和他的学生可能会有以下讨论。

**Java 神秘例程一**

```java
// 示例 32-3
// write out the sums 1..n for all n from 1 to num
current = 1;
previous = 0;
sum = 1;
for ( int i = 0; i < num; i++ ) {
   System.out.println( "Sum = " + sum );
   sum = current + previous;
   previous = current;
   current = sum;
}
```

你的最佳猜测？

这个例程计算前 num 个斐波那契数。它的编码风格比本章开头的例程稍微好一点，但注释是错误的，如果你盲目相信注释，你会走向错误的方向。

**Java 神秘例程二**

```java
// 示例 32-4
// set product to "base"
product = base;

// loop from 2 to "num"
for ( int i = 2; i <= num; i++ ) {
   // multiply "base" by "product"
   product = product * base;
}
System.out.println( "Product = " + product );
```

这个例程将整数 base 的 num 次幂。注释是准确的，但它们没有给代码增加任何东西。它们只是代码本身的更冗长版本。

```java
// 示例 32-5
```

### 注释的种类

注释可以分为六个类别：

**代码的重复**

重复性注释用不同的词重述代码做什么。它只是给代码的读者更多要读的东西，没有提供额外的信息。

**代码的解释**

解释性注释通常用于解释复杂的、棘手的或敏感的代码片段。在这种情况下它们是有用的，但通常这只是因为代码混乱。如果代码太复杂需要解释，几乎总是改进代码比添加注释更好。使代码本身更清晰，然后使用总结或意图注释。

**代码中的标记**

标记注释不是要留在代码中的。它是给开发人员的工作未完成的提示。一些开发人员输入在语法上不正确的标记（如 ******），这样编译器会标记它，提醒他们还有更多工作要做。其他开发人员在注释中放置一组指定的字符，不干扰编译，这样他们可以搜索它们。

没有什么比客户报告代码中的问题、调试问题并追踪到一个部分发现这样的东西更糟糕的感觉了：

```java
return NULL; // ****** NOT DONE! FIX BEFORE RELEASE!!!
```

向客户发布有缺陷的代码已经够糟糕了；发布你知道有缺陷的代码更糟。

我发现标准化标记注释的样式是有帮助的。如果你不标准化，一些程序员会使用 ******，一些会用 !!!!!!，一些会用 TBD，还有一些会用各种其他约定。使用各种符号使机械搜索不完整的代码变得容易出错或不可能。标准化一个特定的标记样式允许你在发布清单的步骤中机械搜索不完整的代码部分，这避免了 FIX BEFORE RELEASE!!! 问题。一些编辑器支持"待办"标签，允许你轻松导航到它们。

**代码的总结**

总结代码的注释正是这样做的：将几行代码归纳为一两句话。这样的注释比仅仅重复代码的注释更有价值，因为读者可以比读代码更快地扫描它们。当代码的原作者以外的其他人试图修改代码时，总结注释特别有用。

**代码意图的描述**

意图级别的注释解释了一段代码的目的。意图注释在问题的层次而非解决方案的层次上运作。例如：

```python
-- get current employee information
```

是意图注释，而

```python
-- update employeeRecord object
```

是关于解决方案的总结注释。IBM 进行的一项为期六个月的研究发现，维护程序员"最常说的是，理解原始程序员的意图是最困难的问题"（Fjelstad and Hamlen 1979）。意图和总结注释之间的区别并不总是清晰的，而且通常也不重要。本章中到处都有意图注释的示例。

**代码本身无法表达的信息**

有些信息无法在代码中表达，但仍然必须放在源代码中。这类别注释包括版权声明、保密声明、版本号和其他管理细节；关于代码设计的笔记；指向相关需求或架构文档的参考；联机参考的指针；优化注释；编辑工具（如 Javadoc 和 Doxygen）所需的注释；等等。

对于完成代码，有三种可接受的注释：代码无法表达的信息、意图注释和总结注释。

## 高效注释

有效注释不会太耗时。太多注释和太少注释一样糟糕，你可以经济地达到中间立场。

注释可能因为两个常见原因耗费大量时间。首先，注释风格可能耗时或繁琐。如果是这样，找一个新的风格。一种需要大量繁琐工作的注释风格是维护的噩梦。如果注释难以更改，它们将不会被更改，它们将变得不准确和误导，这比根本没有注释更糟。

其次，注释可能很困难，因为描述程序正在做什么的词不容易出来。这通常是表明你不理解程序在做什么的信号。你花在"注释"上的时间实际上是花在更好地理解程序上的，这是无论你是否注释都需要花的时间。

以下是高效注释的指南：

**使用不易破坏或阻碍修改的样式**。任何过于花哨的样式都是令人烦恼的维护。例如，从下面的注释中找出不会被维护的部分：

**难以维护的注释风格 Java 示例**

```java
// 示例 32-6
//  Variable        Meaning
//  --------        -------
//  xPos .......... XCoordinate Position (in meters)
//  yPos .......... YCoordinate Position (in meters)
//  ndsCmptng...... Needs Computing (= 0 if no computation is needed,
//                                   = 1 if computation is needed)
//  ptGrdTtl....... Point Grand Total
//  ptValMax....... Point Value Maximum
//  psblScrMax..... Possible Score Maximum
```

如果你说前导点（.....）难以维护，你是对的！它们看起来不错，但列表没有它们也很好。它们为修改注释的工作增加了繁琐的工作，你宁愿有准确的注释而非好看的注释，如果有选择的话——通常是有选择的。

这是另一个难以维护的常见样式示例：

**难以维护的注释风格 C++ 示例**

```cpp
 *                                                                    *
```

这是一段好看的块注释。很明显整个块属于一起，块的开始和结束是明显的。关于这个块，不清楚的是它有多容易更改。如果你必须将文件名添加到注释底部，很可能你必须弄乱右边漂亮的星号列。如果你需要更改段落注释，你必须同时在左边和右边弄乱星号。实际上，这意味着块将不会被维护，因为太费劲了。如果你按一个键就能得到整齐的星号列，那很好。使用它。问题不是星号而是它们难以维护。以下注释看起来几乎一样，但维护起来很简单：

Java 和 C++ 的一个常见指南来自类似的动机，对单行注释使用 // 语法，对较长的注释使用 /* … */ 语法，如下所示：

要点是你应该注意你如何花费你的时间。如果你花大量时间输入和删除破折号以使加号对齐，你不是在编程；你在浪费时间。找到一种更有效的样式。在带加号的下划线的情况下，你可以选择只有注释而没有任何下划线。如果你需要使用下划线来强调，找到下划线带加号以外的其他方式来强调那些注释。一种方式是有一个标准下划线，无论注释长度如何总是相同的长度。这样的线不需要维护，你可以使用文本编辑器宏来首先输入它。

**使用伪代码编程过程来减少注释时间**。如果你在编写代码之前用注释勾勒代码大纲，你会以多种方式获胜。当你完成代码时，注释也完成了。你不需要专门花时间在注释上。你也可以在用高级伪代码设计之后填充低级编程语言代码中获得所有设计好处。

**将注释集成到你的开发风格中**。将注释集成到你的开发风格中的替代方案是将注释留到项目结束，这有太多缺点。它成为一项独立的任务，这让它看起来比一点一点做的工作更多。后来做的注释需要更多时间，因为你必须记住或弄清楚代码在做什么，而不是写下你已经想到的东西。它也不太准确，因为你倾向于忘记设计中的假设或微妙之处。

反对边写边注释的常见论点是"当你专注于代码时，你不应该中断你的思路来写注释。"适当的回应是，如果你在编写代码时必须如此专注以至于注释打断了你的思路，你需要先用伪代码澄清你的思路，然后再将伪代码转换为注释。需要那么多专注的代码是一个警告信号。

> 如果你的设计难以编码，先简化设计再担心注释或代码。如果你用伪代码来澄清你的思路，编码是直接的，注释是自动的。

**性能不是避免注释的好理由**。技术浪潮中反复出现的一个属性是解释环境，其中注释会带来可测量的性能损失。在 1980 年代，原始 IBM PC 上 Basic 程序中的注释会减慢程序。在 1990 年代，.asp 页面做了同样的事情。在 2000 年代，JavaScript 代码和其他需要通过网络连接发送的代码提出了类似的问题。

在每种情况下，最终的解决方案不是避免注释；而是创建与开发版本不同的代码发布版本。这通常通过运行代码通过一个工具来实现，该工具在构建过程中剥离注释。

### 最佳注释数量

Capers Jones 指出，IBM 的研究发现，大约每 10 条语句一个注释的注释密度是清晰度似乎达到峰值的密度。更少的注释使代码难以理解。更多的注释也降低了代码可理解性（Jones 2000）。

这种研究可能被滥用，项目有时采用诸如"程序必须至少有每五行一个注释"之类的标准。这个标准解决了程序员不写清晰代码的症状，但没有解决原因。

如果你有效地使用伪代码编程过程，你可能会最终每几行代码就有一个注释。然而，注释的数量将是过程本身的副作用。不要专注于注释的数量，而要专注于每个注释是否有效。如果注释描述了为什么写代码并满足本章建立的其他标准，你就会有足够的注释。

## 注释技术

注释可以根据注释适用的级别采用几种不同的技术：程序、文件、例程、段落或单独的行。

### 注释单独的行

在好的代码中，注释单独的行是罕见的。这里有两个代码行可能需要注释的可能原因：

- 单行足够复杂需要解释。
- 单行曾经有一个错误，你想要一个错误记录。

以下是注释一行代码的一些指南：

**避免自我放纵的注释**。很多年前，我听到一个故事，一个维护程序员被叫醒去修复一个故障程序。程序的作者已经离开公司，无法联系。维护程序员之前没有处理过这个程序，仔细检查文档后，他只找到一个注释。它看起来像这样：

```asm
MOV AX, 723h   ; R. I. P. L. V. B.
```

在通宵处理程序并对注释感到困惑之后，程序员成功修补并回家睡觉。几个月后，他在会议上遇到了程序的作者，发现注释代表"愿路德维希·范·贝多芬安息"。贝多芬于 1827 年（十进制）去世，即 723（十六进制）。723h 在那个位置被需要与注释无关。Aaarrrrghhhhh！

### 行尾注释及其问题

行尾注释是出现在代码行末尾的注释：

**行尾注释的 Visual Basic 示例**

```vb
' 示例 32-11
For employeeId = 1 To employeeCount
   GetBonus( employeeId, employeeType, bonusAmount )
   If employeeType = EmployeeType_Manager Then
      PayManagerBonus( employeeId, bonusAmount ) ' pay full amount
   Else
      If employeeType = EmployeeType_Programmer Then
         If bonusAmount >= MANAGER_APPROVAL_LEVEL Then
            PayProgrammerBonus( employeeId, StdAmt() ) ' pay std. amount
         Else
            PayProgrammerBonus( employeeId, bonusAmount ) ' pay full amount
         End If
      End If
   End If
Next
```

虽然在某些情况下有用，行尾注释有几个问题。注释必须与代码的右边对齐，这样它们才不会干扰代码的视觉结构。如果你没有整齐地对齐它们，它们会让你的列表看起来像是经过了洗衣机。行尾注释往往难以格式化。如果你使用很多注释，花时间对齐它们。这些时间不是用来学习更多关于代码的；它只用于按下空格键或 Tab 键的繁琐任务。

行尾注释也很难维护。如果包含行尾注释的任何代码行增长，它会将注释撞得更远，所有其他行尾注释都必须被撞出来以匹配。难以维护的样式在修改下不会得到维护，注释会退化而非改进。

行尾注释也往往是隐晦的。行的右边通常没有太多空间，保持注释在一行内的愿望意味着注释必须简短。然后工作变成了使行尽可能短而非尽可能清晰。

**避免在单独的行上使用行尾注释**。除了它们的实际问题，行尾注释还有几个概念问题。这里是一组行尾注释的示例：

**无用行尾注释的 C++ 示例**

```cpp
// 示例 32-12
memoryToInitialize = MemoryAvailable();    // get amount of memory available       (1)
pointer = GetMemory( memoryToInitialize ); // get a ptr to the available memory      |
ZeroMemory( pointer, memoryToInitialize ); // set memory to 0                        |
...                                                                                  |
FreeMemory( pointer );                     // free memory allocated       <-- 1
```

(1)注释只是重复代码。

行尾注释的一个系统问题是为一行代码写有意义的注释是困难的。大多数行尾注释只是重复代码行，这弊大于利。

**避免对多行代码使用行尾注释**。如果行尾注释旨在适用于多行代码，格式不能显示注释适用于哪些行：

**多行代码上令人困惑的行尾注释的 Visual Basic 示例**

```vb
' 示例 32-13
For rateIdx = 1 to rateCount                  ' Compute discounted rates
   LookupRegularRate( rateIdx, regularRate )
   rate( rateIdx ) = regularRate * discount( rateIdx )
Next
```

尽管这个特定注释的内容很好，但它的位置不是。你必须阅读注释和代码才能知道注释是适用于特定语句还是整个循环。

**数据声明的良好行尾注释的 Java 示例**

### 注释代码段落

在文档完善的程序中，大多数注释是一两句话的注释，描述代码段落：

**代码段落良好注释的 Java 示例**

```java
// 示例 32-15
// swap the roots
oldRoot = root[0];
root[0] = root[1];
root[1] = oldRoot;
```

注释不重复代码——它描述代码的意图。这样的注释相对容易维护。例如，即使你发现交换根的方式有错误，注释也不需要更改。不是在意图级别编写的注释更难维护。

**在代码意图级别编写注释**。描述注释后面代码块的目的。以下是一个无效注释的示例，因为它不在意图级别运作：

**无效注释的 Java 示例**

```java
// 示例 32-16
/* check each character in "inputString" until a dollar sign
is found or all characters have been checked
*/
done = false;
maxLen = inputString.length();
i = 0;
while ( !done && ( i < maxLen ) ) {
   if ( inputString[ i ] == '$' ) {
      done = true;
   }
   else {
      i++;
   }
}
```

交叉引用

这段执行简单字符串搜索的代码仅用于说明目的。对于实际代码，你会使用 Java 内置的字符串库函数。更多关于理解语言能力的重要性的内容，请参见"阅读！"在好奇心部分。

你可以从阅读代码中弄清楚循环寻找 $，将这一点总结在注释中是有帮助的。这个注释的问题在于它只是重复代码，没有给你任何关于代码应该做什么的洞察。这个注释会稍微好一点：

```java
// find '$' in inputString
```

这个注释更好，因为它表明循环的目的是在 inputString 中找到 $。但它仍然没有给你太多关于为什么循环需要找到 $ 的洞察——换句话说，关于循环的更深层意图。以下注释更好：

```java
// find the command-word terminator ($)
```

这个注释实际上包含代码清单没有的信息，即 $ 终止一个命令词。在任何情况下，你都不能仅仅通过阅读代码片段来推断这个事实，所以注释是真正有帮助的。

关于在意图级别注释的另一种思考方式是思考你会给做与你想注释的代码相同事情的例程起什么名字。如果你正在编写每个目的单一的代码段落，这并不困难。前面代码示例中的注释是一个很好的例子。FindCommandWordTerminator() 会是一个不错的例程名称。其他选项 Find$InInputString() 和 CheckEachCharacterInInputStrUntilADollarSignIsFoundOrAllCharactersHaveBeenChecked() 由于明显的原因是糟糕的名称（或无效）。不要缩短或缩写描述，就像你可能为例程名称做的那样。那个描述就是你的注释，它可能在意图级别。

**将文档工作重点放在代码本身上**。作为记录，代码本身始终是你应该检查的第一个文档。在前面的例子中，字面量 $ 应该被替换为命名常量，变量应该提供更多关于发生了什么事的线索。如果你想推动可读性的边缘，添加一个变量来包含搜索的结果。这样做可以清楚地区分循环索引和循环的结果。下面是用好的注释和好的风格重写的代码：

**好的注释和好的代码的 Java 示例**

```java
// 示例 32-17
// find the command-word terminator
foundTheTerminator = false;
commandStringLength = inputString.length();
testCharPosition = 0;
while ( !foundTheTerminator && ( testCharPosition < commandStringLength ) ) {
   if ( inputString[ testCharPosition ] == COMMAND_WORD_TERMINATOR ) {
      foundTheTerminator = true;
      terminatorPosition = testCharPosition;       (1)
   }
   else {
      testCharPosition = testCharPosition + 1;
   }
}
```

(1)这是包含搜索结果的变量。

如果代码足够好，它开始接近意图级别可读，侵入注释对代码意图的解释。在那一点上，注释和代码可能变得有些冗余，但这是一个很少程序有的问题。

这个代码的另一个好步骤是创建一个名为 FindCommandWordTerminator() 之类的例程，并将示例中的代码移入该例程。描述那个想法的注释是有用的，但比例程名称更容易随着软件演变变得不准确。

**将段落注释集中在"为什么"而非"怎么做"**。解释某事如何做的注释通常在编程语言级别而非问题级别运作。几乎不可能让专注于操作如何做的注释解释操作的意图，告诉你怎么做的注释通常是冗余的。以下注释告诉你代码没有告诉你的东西吗？

**关注"怎么做"的注释的 Java 示例**

```java
// 示例 32-18
// if account flag is zero
if ( accountFlag == 0 ) ...
```

注释没有告诉你代码本身没有告诉你的任何东西。这个注释怎么样？

**关注"为什么"的注释的 Java 示例**

```java
// 示例 32-19
// if establishing a new account
if ( accountFlag == 0 ) ...
```

这个注释要好得多，因为它告诉你无法从代码本身推断的东西。代码本身仍然可以通过使用有意义的枚举类型名称而非 0 和更好的变量名来改进。以下是这个注释和代码的最佳版本：

**除了"为什么"注释外还使用好风格的 Java 示例**

```java
// 示例 32-20
// if establishing a new account
if ( accountType == AccountType.NewAccount ) ...
```

当代码达到这种可读性水平时，质疑注释的价值是合适的。在这种情况下，注释已被改进的代码变得多余，应该可能删除它。或者，注释的目的可以巧妙地转变，像这样：

**使用"章节标题"注释的 Java 示例**

```java
// 示例 32-21
// establish a new account
if ( accountType == AccountType.NewAccount ) {
   ...
}
```

如果这个注释文档化了 if 测试后面的整个代码块，它作为它引用的代码段落的总结级注释，保留它作为章节标题是合适的。

**使用注释为读者准备接下来会发生什么**。好的注释告诉阅读代码的人应该期待什么。读者应该能够仅扫描注释就很好地了解代码在做什么以及在哪里寻找特定活动。这条规则的一个推论是注释应该总是在它描述的代码之前。这在编程课程中并不总是被教授，但它是商业实践中成熟的约定。

**让每个注释都有价值**。过度注释没有好处——太多注释会遮蔽它们本应澄清的代码。不要写更多注释，而是把额外的努力放在使代码本身更易读上。

**记录意外情况**。如果你发现代码本身不明显的任何东西，把它放进注释。如果你使用了一个棘手的技术而非直接的技术来提高性能，使用注释来指出直接的技术会是什么，并量化使用棘手技术实现的性能提升。例如：

**记录意外的 C++ 示例**

```cpp
// 示例 32-22
for ( element = 0; element < elementCount; element++ ) {
   // Use right shift to divide by two. Substituting the
   // right-shift operation cuts the loop time by 75%.
   elementList[ element ] = elementList[ element ] >> 1;
}
```

在这个例子中选择右移是有意的。在有经验的程序员中，众所周知对于整数，右移在功能上等同于除以二。

如果是常识，为什么要文档化？因为操作的目的不是执行右移；它是执行除以二。代码没有使用最适合其目的的技术这一事实是重要的。此外，大多数编译器无论如何都会将整数除以二优化为右移，这意味着减少的清晰度通常是不必要的。在这个特定情况下，编译器显然没有优化除以二，节省的时间将是显著的。有了文档，阅读代码的程序员会看到使用非显而易见技术的动机。没有注释，相同的程序员会倾向于抱怨代码"不必要地聪明"而没有有意义的性能提升。通常这种抱怨是有道理的，所以记录例外是重要的。

**避免缩写**。注释应该明确无误，不需要努力去弄清楚缩写。除了最常见的缩写外，避免在注释中使用所有缩写。除非你使用行尾注释，使用缩写通常不是诱惑。如果是的话，意识到缩写是针对一种已经几次三振的技术的又一出局。

**区分主要和次要注释**。在少数情况下，你可能想要区分不同级别的注释，表明详细注释是前一个更广泛注释的一部分。你可以用几种方式处理这个问题。你可以尝试给主要注释加下划线而不给次要注释加下划线：

**用下划线区分主要和次要注释的 C++ 示例——不推荐**

```cpp
// 示例 32-23
// copy the string portion of the table, along the way omitting       (1)
// strings that are to be deleted                                       |
//--------------------------------------------------------------------------       (1)
// determine number of strings in the table       (2)
...

// mark the strings to be deleted       (3)
...
```

(1)主要注释加了下划线。

(2)作为主要注释描述的动作的一部分的次要注释在这里没有加下划线……

(3)……或这里。

这种方法的问题是它迫使你比你真正想加的下划线加下划线更多。如果你给一个注释加下划线，假设它后面所有不加下划线的注释都从属于它。因此，当你写出第一个不从属于加下划线注释的注释时，它也必须加下划线，循环重新开始。结果是在某些地方有太多或不一致的下划线，在其他地方没有下划线。

这个主题有几种变体，都有相同的问题。如果你把主要注释全部大写而次要注释用小写，你用太多全大写注释的问题替代了太多加下划线注释的问题。一些程序员在主要语句上使用首字母大写而在次要语句上不使用首字母，但这是一个太容易被忽视的微妙视觉线索。

更好的方法是使用省略号在次要注释前面：

**用省略号区分主要和次要注释的 C++ 示例**

```cpp
// 示例 32-24
// copy the string portion of the table, along the way omitting       (1)
// strings that are to be deleted
// ... determine number of strings in the table       (2)
...

// ... mark the strings to be deleted       (3)
...
```

(1)主要注释格式正常。

(2)作为主要注释描述的动作的一部分的次要注释在这里用省略号前缀……

(3)……和这里。

另一种通常最好的方法是将近程注释操作放入它自己的例程。例程应该在逻辑上"扁平"，所有活动都在大约相同的逻辑级别。如果你的代码在例程中区分主要和次要活动，例程不是扁平的。将复杂的活动组放入它们自己的例程会产生两个逻辑扁平的例程，而非一个逻辑凹凸的例程。

这个关于主要和次要注释的讨论不适用于循环和条件内的缩进代码。在这种情况下，你通常会在循环顶部有一个广泛的注释，对缩进代码内的操作有更详细的注释。在这些情况下，缩进提供了注释逻辑组织的线索。这个讨论仅适用于顺序代码段落，其中几个段落组成一个完整的操作，一些段落从属于其他段落。

**注释任何绕过语言或环境中错误或未记录功能的代码**。如果是错误，它可能没有记录。即使在某处有记录，在你的代码中再次记录它也没有坏处。如果是未记录的功能，根据定义它没有在其他地方记录，应该在你的代码中记录。

假设你发现库例程 WriteData( data, numItems, blockSize ) 除了当 blockSize 等于 500 时外都正常工作。它对 499、501 和你试过的其他每个值都很好，但你发现当 blockSize 等于 500 时例程有一个缺陷，只在那个参数等于 500 时出现。在使用 WriteData() 的代码中，记录当你使 blockSize 为 500 时做一个特殊处理的原因。以下是它可能的样子：

**记录错误变通方法的 Java 示例**

```java
// 示例 32-25
blockSize = optimalBlockSize( numItems, sizePerItem );

/* The following code is necessary to work around an error in
WriteData() that appears only when the third parameter
equals 500. '500' has been replaced with a named constant
for clarity.
*/
if ( blockSize == WRITEDATA_BROKEN_SIZE ) {
   blockSize = WRITEDATA_WORKAROUND_SIZE;
}
WriteData ( file, data, blockSize );
```

**说明违反良好编程风格的原因**。如果你不得不违反良好的编程风格，解释原因。那将防止好心程序员将代码改为更好的风格，可能破坏你的代码。解释会让你清楚你知道你在做什么，而不是只是马虎——该得到表扬的地方就要表扬！

**不要注释棘手的代码；重写它**。这是我参与的一个项目的一条注释：

**注释聪明代码的 C++ 示例**

```cpp
// 示例 32-26
// VERY IMPORTANT NOTE:
// The constructor for this class takes a reference to a UiPublication.
// The UiPublication object MUST NOT BE DESTROYED before the DatabasePublication
// object. If it is, the DatabasePublication object will cause the program to
// die a horrible death.
```

这是最流行和危险的编程神话之一的一个好例子：注释应该用于文档化特别"棘手"或"敏感"的代码部分。推理是人们应该知道在某些区域工作时需要小心。

这是个可怕的想法。

注释棘手的代码恰恰是错误的方法。注释不能拯救困难的代码。正如 Kernighan 和 Plauger 强调的，"不要文档化糟糕的代码——重写它"（1978）。

一项研究发现，源代码中大量注释的区域也往往有最多的缺陷并消耗最多的开发工作（Lind and Vairavan 1989）。作者假设程序员倾向于大量注释困难的代码。

当有人说"这真是棘手的代码"时，我听到他们说"这真是糟糕的代码"。如果某事对你来说似乎棘手，它对另一个人来说是不可理解的。即使对你来说似乎不那么棘手的东西，对于没有见过这个技巧的另一个人来说可能看起来极其复杂。如果你不得不问自己"这是棘手的吗？"它就是。你总能找到不棘手的重写，所以重写代码。让你的代码好到你不需要注释，然后注释它让它更好。

这个建议主要适用于你第一次编写的代码。如果你正在维护一个程序，没有重写糟糕代码的余地，注释棘手的部分是一个好做法。

**注释数字数据的单位**。如果数字代表长度，指出长度是以英寸、英尺、米还是公里表示。如果是时间，指出是以自 1980-1-1 以来的经过秒数、自程序开始以来的毫秒数等表示。如果是坐标，指出它们代表纬度、经度和高度还是其他；是以弧度还是度为单位；它们代表以地球中心为原点的 X、Y、Z 坐标系；等等。不要假设单位是明显的。对于新程序员，它们不会是。对于一直在系统另一部分工作的人，它们不会是。在程序大幅修改之后，它们不会是。

或者，在许多情况下你应该将单位嵌入变量名而非注释中。像 distanceToSurface = marsLanderAltitude 这样的表达式看起来可能是正确的，但 distanceToSurfaceInMeters = marsLanderAltitudeInFeet 暴露了一个明显的错误。

**注释允许的数值范围**。如果变量有期望的数值范围，文档化期望的范围。Ada 编程语言的一个强大特性是能够将数值变量的允许值限制在一定范围内。如果你的语言不支持这种能力（大多数语言不支持），使用注释来文档化期望的数值范围。例如，如果变量代表美元金额，指出你期望它在 $1 到 $100 之间。如果变量表示电压，指出它应该在 105v 到 125v 之间。

交叉引用

**注释编码含义**。如果你的语言支持枚举类型——如 C++ 和 Visual Basic 所做的那样——使用它们来表达编码含义。如果没有，使用注释来指示每个值代表什么——并为每个值使用命名常量而非字面量。如果变量代表电流类型，注释 1 代表交流电，2 代表直流电，3 代表未定义。

### 注释控制结构

控制结构之前的空间通常是放置注释的自然位置。如果是 if 或 case 语句，你可以提供决策的原因和结果的总结。如果是循环，你可以指示循环的目的。

交叉引用

有关控制结构的其他详细信息，请参见布局样式、布局控制结构以及第14章到第19章。

**注释控制结构目的的 C++ 示例**

```cpp
// 示例 32-29
// copy input field up to comma       (1)
while ( ( *inputString != ',' ) && ( *inputString != END_OF_STRING ) ) {
   *field = *inputString;
   field++;
   inputString++;
} // while -- copy input field       (2)

*field = END_OF_STRING;

if ( *inputString != END_OF_STRING ) {
   // read past comma and subsequent blanks to get to the next input field       (3)
   inputString++;
   while ( ( *inputString == ' ' ) && ( *inputString != END_OF_STRING ) ) {
      inputString++;
   }
} // if -- at end of string
```

(1)以下循环的目的。

(2)循环的结束（对更长、更深的循环有用——尽管需要这种注释表明代码过于复杂）。

(3)循环的目的。注释的位置清楚表明 inputString 正在为循环做准备。

这个例子暗示了一些指南：

**在每个 if、case、loop 或语句块之前放置注释**。这样的位置是注释的自然位置，这些构造通常需要解释。使用注释来澄清控制结构的目的。

**注释每个控制结构的结束**。使用注释显示什么结束了——例如：

```cpp
} // for clientIndex - process record for each client
```

注释在长循环的结束和澄清循环嵌套时特别有帮助。以下是使用注释来澄清循环结构结束的 Java 示例：

**使用注释显示嵌套的 Java 示例**

```java
// 示例 32-30
for ( tableIndex = 0; tableIndex < tableCount; tableIndex++ ) {
   while ( recordIndex < recordCount ) {
      if ( !IllegalRecordNumber( recordIndex ) ) {
         ...
      } // if       (1)
   } // while         |
} // for       <-- 1
```

(1)这些注释指示哪个控制结构正在结束。

这种注释技术补充了代码缩进给出的关于逻辑结构的视觉线索。对于不嵌套的短循环，你不需要使用这种技术。然而，当嵌套深或循环长时，这种技术会得到回报。

**将循环结束注释视为警告，表明代码复杂**。如果循环复杂到需要循环结束注释，将注释视为警告信号：循环可能需要简化。相同的规则适用于复杂的 if 测试和 case 语句。

循环结束注释提供关于逻辑结构的有用线索，但最初编写它们然后维护它们可能会变得繁琐。避免这种繁琐工作的最好方法通常是重写任何复杂到需要繁琐文档的代码。

```vb
'**********************************************************************
```

这很荒谬。CopyString 可能是一个平凡的例程——可能少于五行代码。注释与例程的规模完全不成比例。关于例程目的和算法的部分很牵强，因为很难在"复制字符串"和代码本身之间的细节级别上描述像 CopyString 这样简单的东西。样板注释 Interface Assumptions 和 Modification History 也没有用——它们只是在清单中占用空间。要求每个例程都有所有这些成分是注释不准确和维护失败的秘诀。这是一大堆无意义的工作，永远不会有回报。

重型例程开头的另一个问题是它们阻止良好的代码分解——创建新例程的开销如此之高，以至于程序员倾向于犯创建更少例程而非更多的错误。编码约定应该鼓励良好实践；重型例程开头做的是相反的。

以下是注释例程的一些指南：

**将注释保持在接近它们描述的代码**。例程的开头不应该包含大量文档的原因之一是这种做法将注释放在离它们描述的例程部分很远的地方。在维护期间，远离代码的注释倾向于不随代码维护。注释和代码开始不一致，突然注释就一文不值了。相反，遵循邻近原则，将注释尽可能靠近它们描述的代码。它们更有可能被维护，它们将继续有价值。

以下描述了例程开头的几个组件，应该根据需要包含。为了你的方便，创建一个样板文档开头。只是不要觉得有义务在每种情况下都包含所有信息。填写重要的部分，删除其余的。

**在例程顶部用一两句话描述每个例程**。如果你不能用短的一两句话描述例程，你可能需要更仔细地思考它应该做什么。难以创建简短描述是设计不够好的迹象。回到设计绘图板再试一次。简短的总结语句应该存在于实际上所有例程中，除了简单的 Get 和 Set 访问例程。

交叉引用

好的例程名称是例程文档的关键。更多关于如何创建它们的信息，请参见好的例程名称。

**在声明参数的地方文档化参数**。文档化输入和输出变量的最简单方法是将注释放在参数声明旁边：

**在声明处文档化输入和输出数据的 Java 示例——好做法**

```java
// 示例 32-32
public void InsertionSort(
   int[] dataToSort, // elements to sort in locations firstElement..lastElement
   int firstElement, // index of first element to sort (>=0)
   int lastElement // index of last element to sort (<= MAX_ELEMENTS)
)
```

这是不使用行尾注释规则的良好例外；它们在文档化输入和输出参数时特别有用。这种注释场合也是展示使用标准缩进而非行尾缩进用于例程参数列表价值的好说明——如果你使用行尾缩进，你不会有关于有意义的行尾注释的空间。即使使用标准缩进，示例中的注释也因空间而显得勉强。这个示例还表明注释不是唯一形式的文档。如果你的变量名足够好，你可能可以跳过注释它们。最后，文档化输入和输出变量是避免全局数据的好理由。你在哪里文档化它？大概，你在怪物开头文档化全局数据。那会需要更多工作，不幸的是，在实践中通常意味着全局数据没有被文档化。这太糟糕了，因为全局数据至少和其他任何东西一样需要被文档化。

交叉引用

行尾注释的更多详细信息在本节前面的"行尾注释及其问题"中。

**利用 Javadoc 等代码文档工具**。如果前一个例子中的代码实际上是用 Java 编写的，你将有能力利用 Java 的文档提取工具 Javadoc 来设置代码。在这种情况下，"在声明处文档化参数"会变为看起来像这样：

**文档化输入和输出数据以利用 Javadoc 的 Java 示例**

```java
// 示例 32-33
/**
 * ... <description of the routine> ...
 *
 * @param dataToSort elements to sort in locations firstElement..lastElement
 * @param firstElement index of first element to sort (>=0)
 * @param lastElement index of last element to sort (<= MAX_ELEMENTS)
 */
public void InsertionSort(
   int[] dataToSort,
   int firstElement,
   int lastElement
)
```

使用像 Javadoc 这样的工具，设置代码提取文档的好处超过了与将参数描述与参数声明分离相关的风险。如果你不是在支持文档提取（如 Javadoc）的环境中工作，你通常最好将注释保持在更接近参数名称以避免不一致的编辑和名称本身的重复。

**区分输入数据和输出数据**。知道哪些数据用作输入哪些用作输出是有用的。Visual Basic 相对容易区分，因为输出数据前面有 ByRef 关键字，输入数据前面有 ByVal 关键字。如果你的语言不支持这样的自动区分，将其放入注释中。以下是 C++ 中的示例：

**区分输入和输出数据的 C++ 示例**

```cpp
// 示例 32-34
void StringCopy(
   char *target,          // out: string to copy to
   const char *source     // in: string to copy from
)
...
```

C++ 语言例程声明有点棘手，因为有些时候星号 (*) 表示参数是输出参数，很多时候它只是意味着变量作为指针类型比作为非指针类型更容易处理。你通常最好显式标识输入和输出参数。

如果你的例程足够短，你保持输入和输出数据之间的清晰区分，记录数据是输入还是输出可能是不必要的。然而，如果例程更长，对任何阅读例程的人来说是一个有用的服务。

**文档化接口假设**。文档化接口假设可能被视为其他注释建议的子集。如果你对你接收的变量状态做了任何假设——合法和非法值、数组排序、成员数据已初始化或只包含好数据等等——在例程开头或数据声明处文档化它们。这个文档应该存在于实际上每个例程中。

交叉引用

有关例程接口其他注意事项的详细信息，请参见如何使用例程参数。使用断言文档化假设的信息，请参见"使用断言来文档化和验证前置条件和后置条件"在断言中。

**确保使用的全局数据被文档化**。全局变量与例程的其他任何东西一样是接口，而且它更危险，因为它有时看起来不像一个接口。

当你写例程时，意识到你正在做一个接口假设，立即把它写下来。

**注释例程的限制**。如果例程提供数值结果，指出结果的准确性。如果计算在某些条件下未定义，文档化条件。如果例程在遇到麻烦时有默认行为，文档化该行为。如果例程预期只适用于某种大小的数组或表，指出这一点。如果你知道会破坏例程的程序修改，文档化它们。如果你在例程开发过程中遇到陷阱，也文档化那些。

**注释例程的全局效果**。如果例程修改全局数据，准确描述它对全局数据做什么。如在全局数据中提到的，修改全局数据至少比仅仅读取它危险一个数量级，所以修改应该小心进行，部分的关心是清晰的文档。像往常一样，如果文档化变得太繁琐，重写代码以减少全局数据。

**注释所用算法的来源**。如果你使用了来自书籍或杂志的算法，文档化你取自它的卷号和页码。如果你开发了自己的算法，指出读者可以在哪里找到你为它做的笔记。

**使用注释标记程序的部分**。一些程序员使用注释来标记程序的部分，以便他们可以轻松找到它们。在 C++ 和 Java 中，这样的一种技术是在每个例程顶部用一个以这些字符开头的注释标记：

```
/**
```

这允许你通过搜索 /** 的字符串搜索或使用你的编辑器如果它支持自动跳转来从一个例程跳到另一个例程。

类似的技术是不同地标记不同种类的注释，取决于它们描述什么。例如，在 C++ 中你可以使用 @keyword，其中 keyword 是你用来指示注释类型的代码。注释 @param 可能表示注释描述例程的参数，@version 可能表示文件版本信息，@throws 可能文档化例程抛出的异常，等等。这种技术允许你使用工具从源文件中提取不同种类的信息。例如，你可以搜索 @throws 来检索关于程序中所有例程抛出的所有异常的信息。

这个 C++ 约定基于 Javadoc 约定，它是 Java 程序的成熟接口文档约定（java.sun.com/j2se/javadoc/）。你可以在其他语言中定义自己的约定。

### 注释类、文件和程序

类、文件和程序都具有包含多个例程的特征。文件或类应该包含相关例程的集合。程序包含一个程序中的所有例程。在每种情况下，文档任务是提供关于文件、类或程序内容的有意义的顶层视图。

有关布局详细信息，请参见布局类。更多关于使用类的信息，请参见第6章。

#### 类文档的一般指南

对于每个类，使用块注释来描述类的一般属性：

**描述类的设计方法**。提供不能轻易从编码细节反向工程的概述信息的注释特别有用。描述类的设计哲学、整体设计方法、考虑和丢弃的设计替代方案等等。

**描述限制、使用假设等**。与例程类似，确保描述类设计施加的任何限制。还要描述关于输入和输出数据、错误处理责任、全局效果、算法来源等的假设。

**注释类的接口**。另一个程序员能否在不查看类的实现的情况下理解如何使用类？如果不能，类封装严重危险。类的接口应该包含任何使用类的人需要的所有信息。Javadoc 约定要求至少为每个参数和每个返回值提供文档（Sun Microsystems 2000）。这应该对每个类的所有公开例程完成（Bloch 2001）。

**不要在类接口中文档化实现细节**。封装的一个基本规则是你只在需要知道的基础上暴露信息：如果关于是否需要暴露信息有任何问题，默认是保持隐藏。因此，类接口文件应该包含使用类所需的信息，而不是实现或维护类内部工作所需的信息。

#### 文件文档的一般指南

在文件顶部，使用块注释来描述文件的内容：

**描述每个文件的目的和内容**。文件头注释应该描述文件中包含的类或例程。如果一个程序的所有例程在一个文件中，文件的目的相当明显——这是包含整个程序的文件。如果文件的目的包含一个特定的类，目的也是明显的——这是包含名称相似的类的文件。

如果文件包含多个类，解释为什么类需要组合成单个文件。

如果分成多个源文件是出于模块化以外的原因，对文件目的的良好描述对于修改程序的程序员会更有帮助。如果有人正在寻找做 x 的例程，文件头注释是否帮助那个人确定这个文件是否包含这样的例程？

**将你的姓名、电子邮件地址和电话号码放在块注释中**。 authorship 和对源代码特定区域的主要责任在大项目中很重要。小项目（少于 10 人）可以使用协作开发方法，如共享代码所有权，所有团队成员对所有代码部分同等负责。较大的系统需要程序员专攻代码的不同区域，这使得全团队共享代码所有权不切实际。

在这种情况下，authorship 是清单中需要的重要信息。它给在代码上工作的其他程序员关于编程风格的线索，如果他们需要帮助，它给他们可以联系的人。根据你是工作在单独例程、类还是程序上，你应该在例程、类或程序级别包含作者信息。

**包含版本控制标签**。许多版本控制工具会将版本信息插入文件。例如，在 CVS 中，字符

```cpp
// $Id$
```

会自动展开为

```cpp
// $Id: ClassName.java,v 1.1 2004/02/05 00:36:43 ismene Exp $
```

这允许你在文件中维护当前版本信息，不需要开发者付出任何努力，除了插入原始 $Id$ 注释。

**在块注释中包含法律声明**。许多公司喜欢在其程序中包含版权声明、保密声明和其他法律声明。如果你的公司是其中之一，在你的文件中包含类似于下面的一行。与你公司的法律顾问检查以确定要包含什么信息（如果有的话）。

**版权声明的 Java 示例**

```java
// 示例 32-35
// (c) Copyright 1993-2004 Steven C. McConnell. All Rights Reserved.
...
```

**给文件一个与其内容相关的名称**。通常，文件名应该与文件中包含的公共类的名称密切相关。例如，如果类名为 Employee，文件应该命名为 Employee.cpp。一些语言，特别是 Java，要求文件名匹配类名。

### 文档的"书籍范式"

大多数有经验的程序员都同意前面章节中描述的文档技术是有价值的。任何一个技术的价值的硬科学证据仍然薄弱。然而，当这些技术结合使用时，它们有效性的证据是强有力的。

进一步阅读

这个讨论改编自"改进维护的书籍范式"（Oman and Cook 1990a）和"排版风格不仅仅是美容"（Oman and Cook 1990b）。在"更易读程序的人类因素和排版"（Baecker and Marcus 1990）中提出了类似的详细分析。

1990 年，Paul Oman 和 Curtis Cook 发表了一对关于文档"书籍范式"的研究（1990a，1990b）。他们寻找支持几种不同代码阅读方式的编码风格。一个目标是支持自顶向下、自底向上和集中搜索。另一个是将代码分解成程序员比长列表同类代码更容易记住的块。Oman 和 Cook 希望这种风格为代码组织提供高层和低层线索。

他们发现，通过将代码视为一种特殊书籍并相应地格式化，可以实现他们的目标。在书籍范式中，代码及其文档被组织成类似于书籍组件的几个部分，以帮助程序员获得程序的高层视图。

"前言"是一组介绍性注释，如通常在文件开头找到的那些。它像书籍的前言一样发挥作用。它给程序员一个程序概览。

"目录"显示顶层文件、类和例程（章节）。它们可能显示在列表中，像传统书籍的章节一样，或者以结构图的形式图形化显示。

"章节"是例程内的划分——例如例程声明、数据声明和可执行语句。

"交叉引用"是代码的交叉引用映射，包括行号。

Oman 和 Cook 用来利用书籍和代码清单之间相似性的低层技术与第31章和本章描述的技术类似。

使用他们的技术组织代码的结果是，当 Oman 和 Cook 给一组有经验的专业程序员一个维护任务时，在 1000 行程序中执行维护任务的平均时间只有程序员在传统源清单上做相同任务所需时间的四分之三左右（1990b）。此外，使用书籍范式文档化的代码上的程序员维护分数平均比传统文档化代码高约 20%。Oman 和 Cook 得出结论，通过注意书籍设计的排版原则，你可以获得 10% 到 20% 的理解力改进。多伦多大学对程序员的研究产生了类似的结果（Baecker and Marcus 1990）。

书籍范式强调提供解释程序高层和低层组织的文档的重要性。

## 好注释技术检查清单

**一般**

- 有人能拿起代码立即开始理解它吗？
- 注释是否解释了代码的意图或总结了代码做了什么，而不仅仅是重复代码？
- 是否使用伪代码编程过程来减少注释时间？
- 棘手的代码是否被重写而非注释？
- 注释是最新的吗？
- 注释是否清晰和正确？
- 注释风格是否允许轻松修改注释？

**语句和段落**

- 代码是否避免行尾注释？
- 注释是否集中在为什么而非怎么做？
- 注释是否为读者准备接下来会发生什么？
- 每个注释都有价值吗？冗余、无关和自我放纵的注释是否已被删除或改进？
- 意外情况是否被记录？
- 是否避免了缩写？
- 主要和次要注释之间的区别是否清晰？
- 绕过错误或未记录功能的代码是否被注释？

**数据声明**

- 数据声明上的单位是否被注释？
- 数值数据的值范围是否被注释？
- 编码含义是否被注释？
- 输入数据的限制是否被注释？
- 标志是否被文档化到位级别？
- 是否在声明处注释了每个全局变量？
- 是否在每个使用处通过命名约定、注释或两者来识别每个全局变量？
- 幻数是否被替换为命名常量或变量而非仅仅被文档化？

**控制结构**

- 是否注释了每个控制语句？
- 是否注释了长或复杂控制结构的结束，或者在可能的情况下被简化而不需要注释？

**例程**

- 是否注释了每个例程的目的？
- 当相关时，是否在注释中给出了关于每个例程的其他事实，包括输入和输出数据、接口假设、限制、错误纠正、全局效果和算法来源？

**文件、类和程序**

- 程序是否有像书籍范式中描述的简短文档，给出了程序如何组织的高层视图？
- 是否描述了每个文件的目的？
- 作者的姓名、电子邮件地址和电话号码是否在清单中？

## 关键要点

- 是否需要注释是一个合理的问题。做不好，注释是浪费时间，有时是有害的。做好，注释是有价值的。
- 源代码应该包含关于程序的大部分关键信息。只要程序在运行，源代码比其他任何资源都更有可能保持最新，将重要信息与代码捆绑在一起是有用的。
- 好的代码是其自身最好的文档。如果代码糟糕到需要大量注释，先尝试改进代码，使其不需要大量注释。
- 注释应该说代码本身无法表达的事情——在总结级别或意图级别。
- 一些注释风格需要大量繁琐的文书工作。开发一种易于维护的风格。
