示例31-3。Java 布局示例 #3

```java
/* Use the insertion sort technique to sort the "data" array in ascending
order. This routine assumes that data[ firstElement ] is not the
first element in data and that data[ firstElement-1 ] can be accessed.
*/

public void InsertionSort( int[] data, int firstElement, int lastElement ) {
   // Replace element at lower boundary with an element guaranteed to be
   // first in a sorted list.
   int lowerBoundary = data[ firstElement-1 ];
   data[ firstElement-1 ] = SORT_MIN;

   /* The elements in positions firstElement through sortBoundary-1 are
   always sorted. In each pass through the loop, sortBoundary
   is increased, and the element at the position of the
   new sortBoundary probably isn't in its sorted place in the
   array, so it's inserted into the proper place somewhere
   between firstElement and sortBoundary.
   */
   for ( int sortBoundary = firstElement + 1; sortBoundary <= lastElement;
      sortBoundary++ ) {
      int insertVal = data[ sortBoundary ];
      int insertPos = sortBoundary;
      while ( insertVal < data[ insertPos - 1 ] ) {
         data[ insertPos ] = data[ insertPos - 1 ];
         insertPos = insertPos - 1;
      }

      data[ insertPos ] = insertVal;
   }

   // Replace original lower-boundary element
   data[ firstElement - 1 ] = lowerBoundary;
}
```

这个例程的布局在坏到好布局的数轴上是一个强烈的 positive。该例程现在根据本章通篇解释的原则布局。该例程变得更容易阅读，并且投入在文档和良好变量名上的努力现在 evident 了。变量名在前面的示例中一样好，但布局太差了以至于它们没有帮助。

这个示例与前两个示例之间的唯一区别是空格的使用——代码和注释完全相同。空格只对人类读者有用——你的计算机可以 equally 容易地解释这三个片段中的任何一个。如果你不能做得像你的计算机一样好，不要感觉糟糕！

### 格式化基本定理

格式化基本定理说，良好的视觉布局显示程序的逻辑结构。

让代码看起来漂亮是值得的，但它比显示代码结构 worth 更少。如果一种技术更好地显示结构，另一种看起来更好，使用显示结构更好的那种。本章提出了许多看起来好但 mispresent 代码逻辑组织的格式化风格示例。在实践中，优先考虑逻辑表示通常不会 create 丑陋的代码——除非代码的逻辑是丑陋的。使好代码看起来好、使坏代码看起来坏的技术比使所有代码看起来好的技术更有用。

### 程序的人类和计算机解释

布局是程序结构的有用线索。虽然计算机可能 exclusively 关心大括号或 begin 和 end，但人类读者可能会从代码的视觉展示中获取线索。考虑示例31-4中的代码片段，其中缩进方案使人类看起来好像每次执行循环时都执行三个语句。

任何傻瓜都能写出计算机能理解的代码。好的程序员写出人类能理解的代码。——Martin Fowler

示例31-4。Java 示例：布局对人类和计算机讲述不同的故事

```java
// swap left and right elements for whole array
for ( i = 0; i < MAX_ELEMENTS; i++ )
   leftElement = left[ i ];
   left[ i ]   = right[ i ];
   right[ i ]  = leftElement;
```

如果代码没有封闭大括号，编译器将执行第一条语句 MAX_ELEMENTS 次，第二条和第三条语句各一次。缩进对你我来说清楚地表明代码的作者希望所有三条语句一起执行，并打算在大括号周围放置它们。这对编译器不会 clear。示例31-5是另一个例子：

示例31-5。另一个 Java 示例：布局对人类和计算机讲述不同的故事

```java
x = 3+4 * 2+7;
```

这段代码的人类读者会倾向于解释该语句意味着 x 被赋值为 (3+4) * (2+7)，即63。计算机将忽略空格并遵守优先级规则，将表达式解释为 3 + (4*2) + 7，即18。重点是，良好的布局方案应该使程序的视觉结构匹配逻辑结构，或者对人类讲述与对计算机相同的 story。

### 好的布局值多少钱？

我们的研究支持这一说法，即对编程计划和编程话语规则的了解可以对程序理解产生重大影响。Kernighan 和 Plauger 在他们名为《编程风格的要素》的书中 also 确定了我们会称之为话语规则的东西。我们的经验结果为这些规则提供了依据：程序应该以特定风格编写不仅仅是一个美学问题。更确切地说，以传统方式编写程序有心理基础：程序员对其他程序员将遵循这些话语规则有强烈期望。如果规则被违反，那么程序员 over time 建立起来的期望所提供的效用被有效地 nullified 了。本文中描述的关于新手和高级学生程序员以及专业程序员的实验结果为这些说法提供了明确支持。——Elliot Soloway 和 Kate Ehrlich

在布局中，也许比编程的任何其他方面更 communicating with 计算机和 communicating with 人类读者之间的差异更大。编程工作的较小部分是编写程序以便计算机可以阅读它；较大部分是编写它以便其他人可以阅读它。

交叉引用

良好的布局是可读性的关键之一。有关可读性价值的详细信息，请参见首先为人编写程序，其次为计算机编写程序。

在他们经典的论文"象棋中的感知"中，Chase 和 Simon 报告了一项比较专家和新手记忆棋子在棋盘上位置的能力的研究（1973）。当棋子按照游戏中可能出现的方式排列时，专家的记忆远远 superior to 新手。当 piece 被随机排列时，专家和新手的记忆之间几乎没有差异。该结果的传统解释是，专家的记忆并不 inherently 比新手更好，而是专家有一个知识结构来帮助他或她记住特定种类的信息。当新信息对应于知识结构时——在本例中是棋子 sensible 放置——专家可以轻松记住它。当新信息不对应于知识结构时——棋子被随机放置——专家不能比新手更好地记住它。

几年后，Ben Shneiderman 在计算机编程领域复制了 Chase 和 Simon 的结果，并在题为"程序员行为的探索性实验"的论文中报告了他的结果（1976）。Shneiderman 发现，当程序语句以 sensible 顺序排列时，专家能够比新手更好地记住它们。当语句被 shuffle 时，专家的优势减少了。Shneiderman 的结果已在其他研究中得到确认（McKeithen et al. 1981, Soloway and Ehrlich 1984）。基本概念 also 在围棋和桥牌以及电子学、音乐和物理学中得到确认（McKeithen et al. 1981）。

### 布局作为宗教

通过熟悉的方式构建 one's 环境的对理解和记忆的重要性已导致一些研究人员假设，如果布局不同于专家使用的方案，布局可能会损害专家阅读程序的能力（Sheil 1981, Soloway and Ehrlich 1984）。这种可能性，加上布局既是美学又是逻辑练习的事实，意味着关于程序格式化的辩论往往听起来更像是宗教战争而不是哲学讨论。

在粗粒度级别上，很明显某些形式的布局比其他形式更好。本章开头相同代码的 successive 更好的布局清楚地表明了这一点。本书不会仅仅因为它们有争议就回避布局的更精细的点。好的程序员应该对他们的布局实践 open-minded，接受被证明比他们习惯使用的更好的实践，即使调整到新方法会导致一些初始不适。

### 良好布局的目标

关于布局细节的许多决定是主观美学的问题；通常，你可以通过多种方式实现相同的目标。如果你明确指定你的偏好标准，你可以使关于主观问题的辩论变得更少主观。明确地说，良好的布局方案应该做到以下几点：

结果指出了编程专业知识的 fragile 性质：高级程序员对程序应该是什么样子有强烈期望，当这些期望被违反时——以看似无害的方式——他们的表现急剧下降。——Elliot Soloway Kate Ehrlich

准确地表示代码的逻辑结构。这又是格式化基本定理：良好布局的主要目的是显示代码的逻辑结构。通常，程序员使用缩进和其他空格来显示逻辑结构。

一致地表示代码的逻辑结构。一些布局风格有太多异常的规则，以至于很难一致地 follow 规则。良好的风格适用于大多数情况。

提高可读性。一种合乎逻辑但使代码更难阅读的缩进策略是无用的。一种只要求在编译器需要的地方使用空格的布局方案是合乎逻辑的但不可读。良好的布局方案使代码更容易阅读。

承受修改。最好的布局方案在代码修改下表现良好。修改一行代码不应该要求修改其他几行。

除了这些标准之外，最小化实现简单语句或块所需的代码行数有时也是 considered。

### 如何使用布局目标

你可以使用良好布局方案的标准来 ground 布局讨论，以便preferring 一种风格 over 另一种风格的主观原因被公开。

以不同方式 weighting 标准可能会导致不同的结论。例如，如果你强烈认为最小化屏幕上使用的行数很重要——也许是因为你有一个小电脑屏幕——你可能会批评一种风格，因为它比另一种风格在例程参数列表上多使用两行。

## 布局技术

你可以通过几种不同的方式使用一些布局工具来实现良好的布局。本节描述其中每一种。

### 空格

使用空格增强可读性。空格，包括空格、制表符、换行符和空行，是你可用于显示程序结构的主要工具。

你不会想到写一本单词之间没有空格、没有段落分隔、没有章节划分的书。这样的书可能可以从头到尾 readable，但它几乎不可能被 skim 以寻找一行思想或找到一个重要段落。也许更重要的是，书的布局不会向读者显示作者 intend 如何组织信息。作者的组织是主题逻辑组织的重要线索。

交叉引用

一些研究人员探索了书的结构与程序结构之间的相似性。有关信息，请参见注释技术中的"程序文档的书本范式"。

将书分成章节、段落和句子向读者展示如何 mentally 组织一个主题。如果组织不明显，读者必须提供组织，这给读者增加了更大的负担，并增加了读者可能永远不会弄清楚主题是如何组织的可能性。

程序中包含的信息比大多数书中包含的信息更密集。虽然你可能在一两分钟内阅读和理解一页书，但大多数程序员无法以接近该速度的速度阅读和理解一个裸的程序列表。程序应该比书提供更多组织线索，而不是更少。

分组。从 looking glass 的另一面，空格就是分组，确保相关语句被分组在一起。

在写作中，思想被分组为段落。一个写得好的段落只包含与特定思想相关的句子。它不应该包含无关的句子。类似地，代码的一个段落应该包含完成单个任务且彼此相关的语句。

空行。正如将相关语句分组很重要一样，将不相关语句彼此分离也很重要。英语中新段落的开头用缩进或空行标识。代码新段落的开头应该用空行标识。

使用空行是一种表示程序如何组织的方式。你可以用它们将相关语句组划分为段落，将例程彼此分离，并突出显示注释。

虽然这个特定的统计数据可能很难付诸实践，但 Gorla、Benander 和 Benander 的一项研究发现，程序中空行的最佳数量约为8到16%。高于16%，调试时间急剧增加（1990）。

缩进。使用缩进来显示程序的逻辑结构。作为规则，你应该在你与之逻辑从属的语句下缩进语句。

缩进已被证明与程序员理解的提高相关。"程序缩进和可理解性"一文报告称，几项研究发现缩进与改善的理解之间存在相关性（Miaria et al. 1983）。当程序使用两个到四个空格的缩进方案时，受试者在理解测试中的得分比程序根本没有缩进时高20%到30%。

同一项研究发现，不低估或过度强调程序的逻辑结构是很重要的。最低的理解得分是在根本没有缩进的程序上获得的。第二低的得分是在使用六空格缩进的程序上获得的。研究得出结论，两到四个空格的缩进是最佳的。有趣的是，实验中的许多受试者认为六空格缩进比 smaller 缩进更易于使用，即使他们的分数较低。这可能是因为六空格缩进看起来更令人愉悦。但无论它看起来多么漂亮，六空格缩进原来是更不可读的。这是美学吸引力和可读性之间冲突的一个例子。

### 括号

比你认为需要的更多地使用括号。当表达式涉及超过两个项时，使用括号来澄清。它们可能不需要，但它们增加了清晰度，而且它们不会花费你任何东西。例如，以下表达式是如何计算的？

C++ 版本: `12 + 4 % 3 * 7 / 8`

Microsoft Visual Basic 版本: `12 + 4 mod 3 * 7 / 8`

关键问题是，你是否不得不考虑表达式是如何计算的？你能在不检查一些参考书的情况下对你的答案有信心吗？即使是有经验的程序员也不能自信地回答，这就是为什么你应该在你对表达式如何计算有任何疑问时使用括号。

## 布局风格

大多数布局问题涉及布局块，即控制语句下面的语句组。块被封闭在大括号或关键字之间：C++ 和 Java 中的 `{` 和 `}`，Visual Basic 中的 if-then-endif，以及其他语言中的其他类似结构。为简单起见，本讨论的很大一部分通用地使用 begin 和 end，假设你可以弄清楚讨论如何应用于 C++ 和 Java 中的大括号或其他语言中的其他阻塞机制。以下部分描述四种通用布局风格：

### 纯块

大部分布局争议源于更流行的编程语言的固有 awkwardness。设计良好的语言具有清晰的分块结构，适合自然的缩进风格。例如，在 Visual Basic 中，每个控制构造都有自己的终止符，你不能使用控制构造而不使用终止符。代码自然地被分块。Visual Basic 中的一些示例如示例31-6、示例31-7和示例31-8所示：

示例31-6。Visual Basic 纯 if 块示例

```vb
If pixelColor = Color_Red Then
   statement1
   statement2
   ...
End If
```

示例31-7。Visual Basic 纯 while 块示例

```vb
While pixelColor = Color_Red
   statement1
   statement2
   ...
Wend
```

示例31-8。Visual Basic 纯 case 块示例

```vb
Select Case pixelColor
   Case Color_Red
      statement1
      statement2
      ...
   Case Color_Green
      statement1
      statement2
      ...
   Case Else
      statement1
      statement2
      ...
End Select
```

Visual Basic 中的控制构造 always 有开始语句——示例中的 If-Then、While 和 Select-case——并且它 always 有相应的 End 语句。将结构内部缩进不是一个有争议的实践，并且对齐其他关键字的选项有些有限。示例31-9是该类型格式化的抽象表示：

示例31-9。纯块布局风格的抽象示例

在这个示例中，语句 A 开始控制构造，语句 D 结束控制构造。两者的对齐提供了 solid 视觉 closure。

关于格式化控制结构的争议部分源于某些语言不需要块结构这一事实。你可以有一个 if-then，后跟一个语句，没有正式的块。你必须添加 begin-end 对或开始和结束大括号来创建块，而不是在每个控制构造中自动获得一个。将 begin 和 end 与控制结构 uncouple——就像 C++ 和 Java 等语言用 `{` 和 `}` 做的那样——导致关于将 begin 和 end 放在哪里的问题。因此，许多缩进问题仅仅是因为你必须补偿设计糟糕的语言结构而成为问题。以下各节描述了各种补偿方式。

### Endline 布局

另一种布局策略是"endline 布局"，它指的是一大组布局策略，其中代码被缩进到行的中间或末尾。Endline 缩进用于将块与其开始的关键字对齐，使例程的后续参数在其第一个参数下对齐，使 case 语句中的 case 对齐，以及其他类似目的。示例31-20是一个抽象示例：

示例31-20。Endline 布局风格的抽象示例

在这个示例中，语句 A 开始控制构造，语句 D 结束它。语句 B、C 和 D 在语句 A 中开始块的 keyword 下对齐。

B、C 和 D 的一致缩进表明它们被分组在一起。示例31-21是使用此策略格式化的代码的较不抽象的示例：

示例31-21。Visual Basic while 块 endline 布局示例

```vb
While ( pixelColor = Color_Red )
        statement1;
        statement2;
        ...
        Wend
```

在示例中，begin 被放在行尾而不是相应关键字下。有些人喜欢将 begin 放在关键字下，但在这两个精细点之间进行选择是这种风格问题中最小的。

Endline 布局风格在少数情况下效果可以接受。示例31-22是一个它似乎有吸引力的示例：

示例31-22。Endline 布局似乎有吸引力的稀有 Visual Basic 示例

```vb

                             markdown = 0.10

                             profit = 0.05
                        Else       <-- 1
                             markdown = 0.05
                        End If
```

（1）else 关键字与其上方的 then 关键字对齐。

在这种情况下，Then、Else 和 End If 关键字是对齐的，它们后面的代码也是对齐的。视觉效果是清晰的逻辑结构。

如果你 critical 地看前面的 case 语句示例，你可能可以预测这种风格的 unraveling。随着条件表达式变得更加复杂，这种风格将给出无用或误导性的关于逻辑结构的线索。示例31-23是一个关于在更复杂的条件使用它时风格如何崩溃的示例：

更典型的 Visual Basic 示例，其中 endline 布局崩溃

示例31-23。

```vb
If ( soldCount > 10 And prevMonthSales > 10 ) Then
   If ( soldCount > 100 And prevMonthSales > 10 ) Then
      If ( soldCount > 1000 ) Then
                                 markdown = 0.1
                                 profit = 0.05
                              Else
                                 markdown = 0.05
                              End If
                                                  Else
                                                     markdown = 0.025
                                                  End If
                                               Else
                                                  markdown = 0.0
                                               End If
```

示例末尾 Else 子句的奇怪格式化是什么原因？它们在相应关键字下一致地缩进，但很难说它们的缩进澄清了逻辑结构。而且，如果代码被修改使得第一行的长度 changed，endline 风格将要求更改相应语句的缩进。这构成了一个维护问题，而纯块、纯块模拟和使用 begin-end designate 块边界没有这个问题。

你可能认为这些示例是编造的 just to make a point，但这种风格尽管有缺点却持续存在。众多教科书和编程参考资料都推荐这种风格。我看到的推荐这种风格的最早书籍出版于1970年代中期，最近的是2003年出版。

总的来说，endline 布局不准确、难以一致应用且难以维护。你将在本章通篇看到 endline 布局的其他问题。

### 哪种风格最好？

如果你在 Visual Basic 中工作，使用纯块缩进。（无论如何，Visual Basic IDE 使使用这种风格很难。）

在 Java 中，标准实践是使用纯块缩进。

在 C++ 中，你可能 simply 选择你喜欢的风格或你的团队中大多数人都喜欢的风格。纯块模拟或 begin-end 块边界都同样有效。唯一一项比较这两种风格的研究发现，就可理解性而言，两者之间没有统计学上的显著差异（Hansen and Yim 1987）。

这两种风格都不是万无一失的，每种都需要偶尔的"合理且明显"的妥协。你可能出于美学原因更喜欢一种或另一种。本书在其代码示例中使用纯块风格，所以只要你浏览示例，你就可以看到更多关于该风格如何工作的说明。一旦你选择了一种风格，当你一致应用它时，你会从良好布局中获得最大收益。

### 格式化控制结构块的精细点

使用控制结构块需要关注一些精细细节。以下是一些指南：

避免未缩进的 begin-end 对。在示例31-24所示的风格中，begin-end 对与控制结构对齐，begin 和 end enclose 的语句在 begin 下缩进。

示例31-24。Java 未缩进 begin-end 对示例

```java
for ( int i = 0; i < MAX_LINES; i++ )       <-- 1
{       <-- 1
   ReadLine( i );       <-- 2
   ProcessLine( i );       <-- 2
}       <-- 3
```

（1）begin 与 for 对齐。

（2）语句在 begin 下缩进。

（3）end 与 for 对齐。

虽然这种方法看起来不错，但它违反了格式化基本定理；它没有显示代码的逻辑结构。以这种方式使用时，begin 和 end 不是控制构造的一部分，但也不是其后语句的一部分。

示例31-25是这种方法抽象视图：

示例31-25。误导性缩进的抽象示例

在这个示例中，语句 B 从属于语句 A 吗？它看起来不是语句 A 的一部分，它也不看起来是从属于它的。如果你使用了这种方法，改为前文描述的两种布局风格之一，你的格式 will be more consistent。

避免使用 begin 和 end 进行双重缩进。反对非缩进 begin-end 对的规则的推论是反对双重缩进 begin-end 对的规则。在这种风格中，如示例31-26所示，begin 和 end 被缩进，它们 enclose 的语句再次缩进：

Java 不适当的双重缩进 begin-end 块的示例

示例31-26。

```java
for ( int i = 0; i < MAX_LINES; i++ )
  {
      ReadLine( i );       <-- 1
      ProcessLine( i );       <-- 1
  }
```

（1）begin 下的语句被缩进，就好像它们从属于它。

这是另一种看起来不错但违反格式化基本定理的风格。一项研究显示，单缩进和双缩进程序之间在理解上没有差异（Miaria et al. 1983），但这种风格没有准确地显示程序的逻辑结构。ReadLine() 和 ProcessLine() 显示为好像它们在逻辑上从属于 begin-end 对，但它们不是。

该方法 also 夸大了程序逻辑结构的复杂性。示例31-27和示例31-28中所示的结构，哪个看起来更复杂？

示例31-27。抽象结构 1

示例31-28。抽象结构 2

两者都是 for 循环结构的抽象表示。抽象结构 1 看起来更复杂，即使它表示与抽象结构 2 相同的代码。如果你将语句嵌套到两级或三级，双重缩进会给你四级或六级的缩进。结果的布局会比实际代码看起来更复杂。通过使用纯块模拟或将 begin 和 end 作为块边界并与它们 enclose 的语句对齐来避免问题。

### 其他注意事项

虽然块的缩进是格式化控制结构的主要问题，但你将遇到其他几种问题，所以这里有一些更多指南：

使用空行分隔段落。有些代码块没有用 begin-end 对 demarcated。逻辑块——属于一起的一组语句——应该像英语中的段落一样对待。将它们彼此分离。示例31-29显示应该分隔的段落示例：

示例31-29。C++ 应该分组和分隔的代码示例

```cpp
cursor.start = startingScanLine;
cursor.end   = endingScanLine;
window.title = editWindow.title;
window.dimensions      = editWindow.dimensions;
window.foregroundColor = userPreferences.foregroundColor;
cursor.blinkRate       = editMode.blinkRate;
window.backgroundColor = userPreferences.backgroundColor;
SaveCursor( cursor );
SetCursor( cursor );
```

这段代码看起来不错，但空行会在两个方面改进它。首先，当你有一组不必以任何特定顺序执行的语句时，很容易将它们 all 以这种方式 lump 在一起。你不需要进一步细化语句顺序 for the computer，但人类读者 appreciate 更多关于哪些语句需要以特定顺序执行、哪些语句只是 along for the ride 的线索。在程序中放置空行的 discipline 使你更仔细地思考哪些语句真的应该在一起。示例31-30中的修改后的片段显示了这个集合实际上应该如何组织。

交叉引用

如果你使用伪代码编程过程，你的代码块将自动分离。有关详细信息，请参见第9章。

示例31-30。C++ 适当分组和分隔的代码示例

```cpp
window.dimensions = editWindow.dimensions;       <-- 1

window.foregroundColor = userPreferences.foregroundColor;       <-- 1
```

（1）这些行设置文本窗口。

（2）这些行设置游标，应该与前面的行分隔。

重新组织的代码显示正在发生两件事。在第一个示例中，缺乏语句组织和空行，以及旧的 aligned–equals signs 技巧，使语句看起来比它们更相关。

使用空行改进代码的第二种方式是它为注释开辟了自然空间。在示例31-30中，每个块上方的注释将很好地补充改进的布局。

一致地格式化单语句块。单语句块是控制结构后面的单个语句，例如 if 测试后的单个语句。在这种情况下，不需要 begin 和 end 用于正确编译，你有示例31-31所示的三种风格选项：

示例31-31。Java 单语句块风格选项示例

```java
if ( expression )       <-- 1
   one-statement;

if ( expression ) {       <-- 2
   one-statement;
}

if ( expression )       <-- 3
   {
   one-statement;
   }

if ( expression ) one-statement;       <-- 4
```

（1）风格 1

（2）风格 2a

（3）风格 2b

（4）风格 3

这些方法各有论据支持。风格 1 follows 与块一起使用的缩进方案，所以它与其他方法一致。风格 2（2a 或 2b）也是一致的，begin-end 对减少了你在 if 测试后添加语句但忘记添加 begin 和 end 的机会。这将是一个特别微妙的错误，因为缩进会告诉你一切都是 OK 的，但编译器不会以相同方式解释缩进。风格 3相对于风格 2的主要优势是更容易输入。它相对于风格 1的优势是，如果它被复制到程序中的另一个地方，它更可能被正确复制。它的缺点是，在面向行的调试器中，调试器将行视为一行，调试器不会显示它是否执行 if 测试后的语句。

我使用过风格 1，并且多次成为错误修改的受害者。我不喜欢风格 3导致的缩进策略例外，所以我完全避免使用它。在 group 项目上，我倾向于使用风格 2的任何变体，因为它的一致性和安全可修改性。无论你选择哪种风格，一致地使用它，对 if 测试和所有循环使用相同的风格。

对于复杂表达式，将单独的条件放在单独的行上。将复杂表达式的每个部分放在自己的行上。示例31-32显示了一个没有注意可读性的格式化表达式：

```cpp
         goto END_PROC;       <-- 1

         goto END_PROC;       <-- 3

      if ( !Erase( fileToPurge ) ) {
         errorCode = FileError_CantErase;
         goto END_PROC;       <-- 4
      }
      fileIndex++;
   }

END_PROC:       <-- 5

   DeletePurgeFileList( fileList, numFilesToPurge );
}
```

- 80字符限制 discourage 深度嵌套。

### 使用空格增强清晰度

在语句中添加空格以提高可读性：

使用空格使逻辑表达式可读。表达式

```vb
while(pathName[startPath+position]<>';') and
   ((startPath+position)<length(pathName)) do
```

和读这个一样可读。

作为规则，你应该用空格分隔标识符与其他标识符。如果你使用这个规则，while 表达式如下所示：

```vb
while ( pathName[ startPath+position ] <> ';' ) and
   ( ( startPath + position ) < length( pathName )) do
```

一些软件艺术家可能建议用额外空格增强这个特定表达式以强调其逻辑结构，像这样：

```vb
while ( pathName[ startPath + position ] <> ';' ) and
   ( ( startPath + position ) < length( pathName ) ) do
```

这很好，尽管第一次使用空格足以确保可读性。然而，额外空格几乎没有伤害，所以要 generous with them。

使用空格使数组引用可读。表达式

```vb
grossRate[census[groupId].gender,census[groupId].ageGroup]
```

不如前面的密集 while 表达式可读。在数组中每个索引周围使用空格使索引可读。如果你使用这个规则，表达式如下所示：

```vb
grossRate[ census[ groupId ].gender, census[ groupId ].ageGroup ]
```

使用空格使例程参数可读。以下例程的第四个参数是什么？

```cpp
ReadEmployeeData(maxEmps,empData,inputFile,empCount,inputError);
```

现在，以下例程的第四个参数是什么？

```cpp
GetCensus( inputFile, empCount, empData, maxEmps, inputError );
```

哪个更容易找到？这是一个现实的、有价值的问题，因为在所有主要过程语言中，参数位置都是 significant。通常，在一个半屏幕上有一个例程规范，另一个半屏幕上是例程调用，并将每个形式参数与每个实际参数进行比较。

### 格式化续行

程序布局最棘手的问题之一是决定如何处理溢出到下一行的语句部分。你是否按正常缩进量缩进它？你是否在关键字下对齐？赋值呢？

以下是一个 sensible、一致的方法，在 Java、C、C++、Visual Basic 和其他鼓励长变量名的语言中特别有用：

使语句的不完整性 obvious。有时候语句必须跨行分解，要么是因为它比编程标准允许的更长，要么是因为它太长而不能放在一行上。使第一行上的语句部分仅为语句的一部分是明显的。最简单的方法是这样分解语句：如果单独站立，第一行在语法上明显不正确。一些示例如示例31-37所示：

示例31-37。Java 明显不完整语句示例

```java
while ( pathName[ startPath + position ] != ';' ) &&       <-- 1
   ( ( startPath + position ) <= pathName.length() )
...

totalBill = totalBill + customerPurchases[ customerID ] +       <-- 2
   SalesTax( customerPurchases[ customerID ] );
...

DrawLine( window.north, window.south, window.east, window.west,       <-- 3
   currentWidth, currentAttribute );
...
```

（1）&& 信号语句不完整。

（2）加号 (+) 信号语句不完整。

（3）逗号 (,) 信号语句不完整。

除了告诉读者第一行上的语句不完整之外，断行还有助于防止错误的修改。如果语句的续行被删除，第一行看起来不像你只是忘记了括号或分号——它明显需要更多的东西。

另一种也很好用的方法是在续行的开头放置续行字符，如示例31-38所示。

示例31-38。Java 明显不完整语句示例——备用风格

虽然这种风格不会因为 hanging && 或 + 而导致语法错误，但它确实使在列左边缘扫描运算符比在右边缘（在那里它是 ragged 的）更容易。它还有一个额外优势，即照亮操作的结构，如示例31-39所示。

### 每行只使用一条语句

C++ 和 Java 等现代语言允许每行多个语句。当涉及在行上放置多个语句时，自由格式化的力量是一把双刃剑。这行包含几个可以逻辑上分离到自己的行上的语句：

```cpp
i = 0; j = 0; k = 0; DestroyBadLoopNames( i, j, k );
```

支持每行放置多个语句的一个论据是它需要更少的屏幕空间或打印纸行，这允许一次查看更多代码。这也是对相关语句进行分组的一种方式，一些程序员认为它为编译器提供了优化线索。

这些是好理由，但限制自己每行一个语句的理由更有说服力：

- 将每个语句放在自己的行上提供了程序复杂性的准确视图。它不通过使复杂语句看起来平凡来隐藏复杂性。复杂的语句看起来复杂。简单的语句看起来简单。
- 在一行上放置多个语句不会为现代编译器提供优化线索。今天的优化编译器不依赖格式化线索来进行优化。这在本文后面 section 进行了说明。

交叉引用

代码级性能优化在第25章和第26章中讨论。

- 有了在自己的行上的语句，代码从上到下读取，而不是从上到下和从左到右。当你在寻找特定的代码行时，你的眼睛应该能够 follow 代码的左边缘。它不应该必须 dip 进入每一行 just because 一行可能包含两个语句。
- 使用编译器 only 提供错误行号的行时，很容易找到语法错误。如果你在一行上有多个语句，行号不会告诉你哪个语句有错误。
- 有了每行一条语句，用面向行的调试器逐步执行代码很容易。如果你在一行上有几个语句，调试器立即执行它们 all，你必须切换到汇编器来逐步执行单个语句。
- 有了每行一条，很容易编辑单个语句——删除一行或暂时将一行转换为注释。如果你在一行上有多个语句，你不得不在其他语句之间进行编辑。

在 C++ 中，避免每行使用多个操作（副作用）。副作用是语句除其主要结果外的后果。在 C++ 中，包含其他操作的一行上的 `++` 运算符是一种副作用。同样，给变量赋值并在条件中使用赋值的左侧是一种副作用。

副作用往往使代码难以阅读。例如，如果 n 等于4，示例31-48中语句的输出是什么？

示例31-48。C++ 不可预测副作用示例

```cpp
PrintMessage( ++n, n + 2 );
```

是4和6吗？是5和7吗？是5和6吗？答案是"以上都不是。"第一个参数 ++n 是5。但 C++ 语言没有定义表达式中或例程参数中项的评估顺序。所以编译器可以 before 或 after 第一个参数评估第二个参数 n + 2；结果可能是6或7，取决于编译器。示例31-49显示了如何重写语句以使意图清晰：

示例31-49。C++ 避免不可预测副作用示例

```cpp
++n;
PrintMessage( n, n + 2 );
```

如果你仍然不相信你应该把副作用放在自己的行上，试着弄清楚示例31-50中例程的作用：

示例31-50。C 每行太多操作的示例

```c
strcpy( char * t, char * s ) {

   while ( *++t = *++s )
      ;
}
```

一些有经验的 C 程序员在这个示例中看不到复杂性，因为它是一个熟悉的函数。他们看着它说，"那是 strcpy()。"然而，在这种情况下，它不完全是 strcpy()。它包含一个错误。如果你看到代码时说"That's strcpy()"，你是在 recognize 代码，而不是 reading it。这正是你在调试程序时所处的情况：你 overlook 的代码因为你"recognize"它而不是阅读它可能包含的错误比它需要的更难找到。

示例31-51中的片段在功能上与第一个相同，更可读：

示例31-51。C 每行可读操作数的示例

```c
strcpy( char * t, char * s ) {
   do {
      ++t;
      ++s;
      *t = *s;
   }
   while ( *t != '\0' );
}
```

在重新格式化的代码中，错误是 apparent。明显地，t 和 s 在 *s 被复制到 *t 之前递增。第一个字符被 missed。

第二个示例看起来比第一个更 elaborate，即使第二个示例中执行的操作 identical。它看起来更 elaborate 的原因是它不隐藏操作复杂性。

改进的性能也不能证明在同一行上放置多个操作是合理的。因为两个 strcpy() 例程在逻辑上等效，你期望编译器为它们生成 identical 代码。当两个版本的例程都被 profiling 时，第一个版本在复制5,000,000个字符串时花费4.81秒，第二个花费4.35秒。

在这种情况下，"聪明"版本带有11%的速度惩罚，这使它看起来不那么聪明。结果因编译器而异，但总的来说，它们表明在你 measurement 了性能提升之前，你最好首先争取清晰度和正确性，其次才是性能。

即使你可以轻松阅读带有副作用的语句，也要同情将阅读你的代码的其他大多数人。大多数优秀程序员需要思考两次才能理解带有副作用的表达式。让他们使用他们的脑细胞来理解你的代码如何工作的更大问题，而不是特定表达式的语法细节。

每行只使用一个数据声明。如前面的示例所示，你应该为每个数据声明提供自己的行。如果每个声明都在自己的行上，在每个声明旁边放置注释更容易。如果每个声明都是自包含的，修改声明更容易。如果你可以扫描单列而不是阅读每一行，找到特定变量更容易。如果编译器给你的行号只有一个声明，找到和修复语法错误更容易。

## 布局文件和程序

除了类的格式化技术之外，还有一个更大的格式化问题：你如何在文件中组织类和例程，你首先决定将哪些类放在文件中？

交叉引用

有关文档详细信息，请参见注释技术中的"注释类、文件和程序"。

一个类一个文件。文件不仅仅是 holding 一些代码的桶。如果你的语言允许，文件应该保存支持一个且仅一个目的的例程集合。文件强化了例程集合在同一类中的想法。

文件内的所有例程构成了类。该类可能是程序真正识别为这样的类，也可能只是你在设计中创建的逻辑实体。

交叉引用

有关类和例程之间的差异以及如何将例程集合变成类的详细信息，请参见第6章。

类是语义语言概念。文件是物理操作系统概念。类和文件之间的对应关系是偶然的，并且随着更多环境支持将代码放入数据库或以其他方式 obscure 例程、类和文件之间的关系而 continue to weaken。

给与类名相关的文件命名。大多数项目在类名和文件名之间具有一对一对应关系。例如，一个名为 CustomerAccount 的类将有名为 CustomerAccount.cpp 和 CustomerAccount.h 的文件。

在文件中清晰分隔例程。用至少两个空行将每个例程与其他例程分隔。空行与大的星号或破折号行一样有效，而且更容易输入和维护。使用两个或三个在例程内部的空行和分隔例程的空行之间产生视觉差异。示例31-69显示了一个示例：

示例31-69。Visual Basic 在例程之间使用空行的示例

```vb
'find the arithmetic maximum of arg1 and arg2
Function Max( arg1 As Integer, arg2 As Integer ) As Integer
   If ( arg1 > arg2 ) Then
      Max = arg1
   Else
      Max = arg2
   End If
End Function

         <-- 1

         <-- 1
'find the arithmetic minimum of arg1 and arg2
Function Min( arg1 As Integer, arg2 As Integer ) As Integer
   If ( arg1 < arg2 ) Then
      Min = arg1
   Else
      Min = arg2
   End If
end Function
```

（1）至少两个空行分隔两个例程。

空行比其他任何分隔符都更容易输入，而且看起来至少一样好。在本例中使用三个空行，使例程之间的分隔比每个例程内部的空行更 noticeable。

按字母顺序排列例程。文件中等效于将相关例程分组的替代方法是按字母顺序放置它们。如果你不能将程序分解为类，或者如果你的编辑器不容易找到函数，字母方法可以节省搜索时间。

在 C++ 中，仔细排序源文件。以下是 C++ 中典型的源文件内容顺序：

1. 文件描述注释

格式化主要是为了照亮代码的逻辑结构吗？

格式化方案可以一致使用吗？

格式化方案产生的代码易于维护吗？

格式化方案提高了代码可读性吗？

#### 控制结构

代码是否避免双重缩进的 begin-end 或 `{}` 对？

顺序块是否用空行彼此分隔？

复杂表达式是否格式化为可读？

单语句块是否格式化为一致？

Case 语句的格式是否与控制结构的其他格式一致？

Goto 是否以使其使用 obvious 的方式格式化？

#### 单个语句

是否使用空格使逻辑表达式、数组引用和例程参数可读？

注释风格是否易于维护？

#### 例程

#### 类、文件和程序

### 关键要点

- 视觉布局的第一个优先级是照亮代码的逻辑组织。用于评估是否实现该优先级的标准包括准确性、一致性、可读性和可维护性。
- 看起来好是次要的——远次要于其他标准。然而，如果满足其他标准且底层代码良好，布局 will look fine。
- Visual Basic 有纯块，Java 中的常规实践是使用纯块风格，所以如果你用这些语言编程，你可以使用纯块布局。在 C++ 中，纯块模拟或 begin-end 块边界都效果很好。
- 构建代码本身很重要。你遵循的特定约定不如你一致遵循某个约定重要。一个不一致遵循的布局约定实际上可能损害可读性。
- 布局的许多方面是宗教问题。尝试将客观偏好与主观偏好分开。使用明确的标准来帮助 ground 你关于风格偏好的讨论。
