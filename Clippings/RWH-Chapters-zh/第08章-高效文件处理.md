# 第08章-高效文件处理

## 高效文件处理

这个简单的基准测试读取一个充满数字的文本文件并打印它们的和：

```haskell
-- file: ch08/SumFile.hs
main = do
    contents <- getContents
    print (sumFile contents)
  where sumFile = sum . map read . words
```

尽管 String 类型是读写文件的默认类型，但它效率不高，所以这样的简单程序性能会很差。

String 被表示为一个 Char 值的列表；列表的每个元素都是单独分配的，并且有一些簿记开销。这些因素影响了必须读写文本或二进制数据的程序的内存消耗和性能。

在这个简单的基准测试中，即使是用 Python 等解释型语言编写的程序，其性能也可能超过使用 String 的 Haskell 代码一个数量级。

bytestring 库提供了 String 类型的快速、廉价的替代品。使用 bytestring 编写的代码通常可以匹配或超过 C 的性能和内存占用，同时保持 Haskell 的表达力和简洁性。

该库提供了两个模块——每个都定义了几乎是 String 对应物的直接替代品：

**Data.ByteString**
定义了一个名为 ByteString 的严格类型。这表示在单个数组中的二进制或文本数据字符串。

**Data.ByteString.Lazy**
提供了一个同样是 ByteString 的惰性类型。这表示数据字符串作为块列表，每个块最多 64KB 的数组。

每种 ByteString 类型在特定情况下表现更好。对于流式传输大量数据（数百兆字节到太字节），惰性 ByteString 类型通常是最佳选择。它的块大小被调整为对现代 CPU 的 L1 缓存友好，垃圾收集器可以快速丢弃不再使用的流数据块。

严格的 ByteString 类型对于不太关心内存占用或需要随机访问数据的应用程序表现最佳。

## 二进制 I/O 与限定导入

让我们开发一个小函数来说明 ByteString API 的一些用法。我们将确定一个文件是否是 ELF 对象文件——这是在几乎所有现代类 Unix 系统上用于可执行文件的格式。

这简单地说就是查看文件的前四个字节，看它们是否匹配特定的字节序列。识别文件类型的字节序列通常被称为魔数（magic number）：

```haskell
-- file: ch08/ElfMagic.hs
import qualified Data.ByteString.Lazy as L
hasElfMagic :: L.ByteString -> Bool
hasElfMagic content = L.take 4 content == elfMagic
    where elfMagic = L.pack [0x7f, 0x45, 0x4c, 0x46]
```

我们使用 Haskell 的限定导入语法 import qualified 来导入 ByteString 模块，这我们刚才看到的。这允许我们用自己选择的名称来引用模块。

例如，当我们想引用惰性 ByteString 模块的 take 函数时，必须写 L.take，因为我们用名称 L 导入了该模块。如果我们不明确指定，例如，想要哪个版本的 take，编译器将报告错误。

我们将始终对 ByteString 模块使用限定导入语法，因为它们提供了许多与 Prelude 函数同名的函数。

限定导入使得在 ByteString 类型之间切换变得容易。你需要做的就是在源文件顶部修改导入声明；你的其余代码可能不需要任何更改。

因此，你可以方便地对两种类型进行基准测试，看看哪种最适合你的应用程序的需求。

无论我们是否使用限定导入，我们总是可以使用模块的完整名称来明确识别某物。例如，Data.ByteString.Lazy.length 和 L.length 都标识同一个函数，Prelude.sum 和 sum 也是如此。

惰性和严格的 ByteString 模块用于二进制 I/O。表示字节的 Haskell 数据类型是 Word8；如果我们需要通过名称引用它，可以从 Data.Word 模块导入。

L.pack 函数获取一个 Word8 值列表，并将它们打包成惰性 ByteString。（L.unpack 函数执行反向转换。）我们的 hasElfMagic 函数简单地比较 ByteString 的前四个字节与魔数。

我们以经典 Haskell 风格编写，这里的 hasElfMagic 函数不执行 I/O。下面是使用它的函数：

```haskell
-- file: ch08/ElfMagic.hs
isElfFile :: FilePath -> IO Bool
isElfFile path = do
  content <- L.readFile path
  return (hasElfMagic content)
```

L.readFile 函数是 readFile 的惰性 ByteString 等价物。它惰性操作，在需要数据时读取文件。它也很高效，一次最多读取 64KB 的块。惰性 ByteString 是我们任务的不错选择：由于我们只需要读取文件的前四个字节，我们可以安全地对任何大小的文件使用此函数。

## 文本 I/O

为方便起见，bytestring 库提供了两个具有有限文本 I/O 功能的模块：Data.ByteString.Char8 和 Data.ByteString.Lazy.Char8。这些模块将单个字符串元素作为 Char 而不是 Word8 暴露。

这些模块中的函数只处理字节大小的 Char 值，所以它们只适合与 ASCII 和一些欧洲字符集一起使用。超过 255 的值会被截断。

面向字符的 bytestring 模块为文本处理提供了有用的函数。这是一个包含一家知名互联网公司 2008 年中期每月股票价格的文件：

```ghci
> putStr =<< readFile "prices.csv"
Date,Open,High,Low,Close,Volume,Adj Close
2008-08-01,20.09,20.12,19.53,19.80,19777000,19.80
2008-06-30,21.12,21.20,20.60,20.66,17173500,20.66
2008-05-30,27.07,27.10,26.63,26.76,17754100,26.76
2008-04-30,27.17,27.78,26.76,27.41,30597400,27.41
```

我们如何从这样的条目系列中找到最高的收盘价？收盘价在第四个逗号分隔的列中。这个函数从一行数据中获取收盘价：

```haskell
-- file: ch08/HighestClose.hs
import qualified Data.ByteString.Lazy.Char8 as L
closing = readPrice . (!!4) . L.split ','
```

由于这个函数以无点风格编写，我们从右向左阅读。L.split 函数每次找到匹配字符时就将惰性 ByteString 分割成一个列表。(!!) 运算符检索列表的第 k 个元素。我们的 readPrice 函数将表示小数价格的字符串转换为整数：

```haskell
-- file: ch08/HighestClose.hs
readPrice :: L.ByteString -> Maybe Int
readPrice str =
    case L.readInt str of
      Nothing             -> Nothing
      Just (dollars,rest) ->
        case L.readInt (L.tail rest) of
          Nothing           -> Nothing
          Just (cents,more) ->
            Just (dollars * 100 + cents)
```

我们使用 L.readInt 函数来解析整数。它返回整数以及消费完一串数字后字符串的剩余部分。我们的定义因为 L.readInt 在解析失败时返回 Nothing 而稍微复杂。

我们查找最高收盘价的函数很简单：

```haskell
-- file: ch08/HighestClose.hs
highestClose = maximum . (Nothing:) . map closing . L.lines
highestCloseFrom path = do
    contents <- L.readFile path
    print (highestClose contents)
```

我们使用一个技巧来克服不能向 maximum 函数提供空列表的事实：

```ghci
> maximum [3,6,2,9]
9
> maximum []
*** Exception: Prelude.maximum: empty list
```

由于我们不希望代码在没有股票数据时抛出异常，(Nothing:) 表达式确保我们提供给 maximum 的 Maybe Int 值列表永远不会为空：

```ghci
> maximum [Nothing, Just 1]
Just 1
> maximum [Nothing]
Nothing
```

我们的函数有效吗？

```ghci
> :load HighestClose
[1 of 1] Compiling Main             ( HighestClose.hs, interpreted )
Ok, modules loaded: Main.
> highestCloseFrom "prices.csv"
Loading package array-0.1.0.0 ... linking ... done.
Loading package bytestring-0.9.0.1.1 ... linking ... done.
Just 2741
```

由于我们将 I/O 与逻辑分离开了，我们可以在不创建空文件的情况下测试无数据的情况：

```ghci
> highestClose L.empty
Nothing
```

## 文件名匹配

许多面向系统的编程语言提供了允许我们匹配文件名与模式的库例程，或者会返回与模式匹配的文件的列表。在其他语言中，这个函数通常命名为 fnmatch。）尽管 Haskell 的标准库通常具有良好的系统编程功能，但它不提供这些类型的模式匹配函数。我们将此作为开发我们自己的机会。

我们将处理的模式类型通常称为 glob 模式（我们将使用的术语）、通配符模式或 shell 风格模式。它们只有几个简单的规则。你可能已经知道它们，但这里我们快速回顾一下：

- 模式匹配从字符串的开头开始，到字符串的结尾结束。
- 大多数字面字符匹配它们自己。例如，模式中的文本 foo 将匹配 foo，也只能匹配 foo。
- *（星号）字符表示"匹配任何东西"；它可以匹配任何文本，包括空字符串。例如，模式 foo* 将匹配任何以 foo 开头的字符串，如 foo 本身、foobar 或 foo.c。模式 quux*.c 将匹配任何以 quux 开头并以 .c 结尾的字符串，如 quuxbaz.c。
- ?（问号）字符匹配任何单个字符。模式 pic??.jpg 将匹配像 picaa.jpg 或 pic01.jpg 这样的名称。
- [（左方括号）字符开始一个字符类，由 ] 结束。它的意思是"匹配此类中的任何字符"。字符类可以通过在开头的 [ 后跟一个 ! 来否定，使其意思是"匹配不在此类中的任何字符"。

作为简写，由 -（破折号）分隔的字符表示一个范围："匹配此集合中的任何字符"。

字符类有一个额外的微妙之处；它们不能为空。打开 [ 或 [! 后的第一个字符是类的一部分，所以我们可以写一个包含 ] 字符的类，如 []aeiou]。模式 pic[0-9].[pP][nN][gG] 将匹配由字符串 pic、后跟单个数字、再后跟 .png 的任何大小写变体组成的名称。

虽然 Haskell 在其标准库中没有提供匹配 glob 模式的方法，但它提供了一个很好的正则表达式匹配库。Glob 模式不过是通过略微不同语法简化了的正则表达式。将 glob 模式转换为正则表达式很容易，但要做到这一点，我们必须首先了解如何在 Haskell 中使用正则表达式。

## Haskell 中的正则表达式

在本节中，我们假设你通过其他语言（如 Python、Perl 或 Java）对正则表达式已经很熟悉了。

为简洁起见，我们将从现在开始将"正则表达式"缩写为 regexp。

我们不会将 regexp 作为新东西介绍，而是关注与其他语言相比，Haskell 中 regexp 处理的不同之处。Haskell 的正则表达式匹配库比其他语言的表达力强得多，所以有很多可以讨论的内容。

为了开始探索 regexp 库，我们唯一需要使用的模块是 Text.Regex.Posix。像往常一样，通过 ghci 与它交互是探索这个模块最方便的方式：

```ghci
> :module +Text.Regex.Posix
```

我们可能需要用于正常使用的唯一函数是正则表达式匹配函数，一个名为 (=~) 的中缀运算符（从 Perl 借来的）。需要克服的第一个障碍是 Haskell 的 regexp 库大量使用多态。因此，(=~) 运算符的类型签名很难理解，所以我们不在这里解释它。=~ 运算符为其两个参数和返回类型都使用类型类。第一个参数（在 =~ 左边）是要匹配的文本；第二个（在右边）是要匹配的正则表达式。我们可以传入 String 或 ByteString 作为参数。

## 结果的多种类型

=~ 运算符在其返回类型上是多态的，所以 Haskell 编译器需要知道我们想要什么类型的结果。在实际代码中，由于我们随后使用结果的方式，它可能能够推断出正确的类型。但当我们在 ghci 中探索时，这样的线索往往缺失。如果我们省略结果的具体类型，我们会从解释器收到错误，因为它没有足够的信息来成功推断结果类型。

当 ghci 不能推断目标类型时，我们告诉它我们希望类型是什么。如果我们想要 Bool 类型的结果，我们会得到通过/失败的回答：

```ghci
> "my left foot" =~ "foo" :: Bool
Loading package array-0.1.0.0 ... linking ... done.
Loading package bytestring-0.9.0.1.1 ... linking ... done.
Loading package regex-base-0.72.0.1 ... linking ... done.
Loading package regex-posix-0.72.0.2 ... linking ... done.
True
> "your right hand" =~ "bar" :: Bool
False
> "your right hand" =~ "(hand|foot)" :: Bool
True
```

在 regexp 库的内部，有一个名为 RegexContext 的类型类，它描述了目标类型应该如何行为；基础库为我们定义了这个类型类的许多实例。Bool 类型是这个类型类的一个实例，所以我们得到了一个可用的结果。另一个这样的实例是 Int，它给我们匹配次数的计数：

```ghci
> "a star called henry" =~ "planet" :: Int
0
> "honorificabilitudinitatibus" =~ "[aeiou]" :: Int
13
```

如果我们请求 String 结果，我们将得到第一个匹配的子字符串，如果没有匹配则返回空字符串：

```ghci
> "I, B. Ionsonii, uurit a lift'd batch" =~ "(uu|ii)" :: String
"ii"
> "hi ludi, F. Baconis nati, tuiti orbi" =~ "Shakespeare" :: String
""
```

另一种有效的结果类型是 [String]，它返回所有匹配字符串的列表：

```ghci
> "I, B. Ionsonii, uurit a lift'd batch" =~ "(uu|ii)" :: [String]
["ii","uu"]
> "hi ludi, F. Baconis nati, tuiti orbi" =~ "Shakespeare" :: [String]
[]
```

小心 String 结果

如果你想要一个普通的 String 结果，要注意。由于 (=~) 返回空字符串来表示"无匹配"，如果空字符串也可能是该正则表达式的有效匹配，这就会造成明显的困难。如果出现这种情况，你应该使用不同的返回类型，如 [String]。

这就是"简单"结果类型的所有内容，但我们绝非完成了。在我们继续之前，让我们为剩余的示例使用一个单一的模式。我们可以将其定义为一个变量在 ghci 中，以节省一些打字：

```ghci
> let pat = "(foo[a-z]*bar|quux)"
```

我们可以获得关于匹配上下文的相当多的信息。如果我们请求 (String, String, String) 元组，我们将得到匹配前的文本、该匹配的文本以及跟随它的文本：

```ghci
> "before foodiebar after" =~ pat :: (String,String,String)
("before ","foodiebar"," after")
```

如果匹配失败，整个文本作为元组的"前"元素返回，其他两个元素为空：

```ghci
> "no match here" =~ pat :: (String,String,String)
("no match here","","")
```

请求四元素元组会给我们第四个元素，它是模式中所有匹配组的列表：

```ghci
> "before foodiebar after" =~ pat :: (String,String,String,[String])
("before ","foodiebar"," after",["foodiebar"])
```

我们也可以获取关于匹配的数字信息。一对 Int 给出第一次匹配的起始偏移量及其长度。如果我们请求这些对的列表，我们将获得所有匹配的此信息：

```ghci
> "before foodiebar after" =~ pat :: (Int,Int)
(7,9)
> "i foobarbar a quux" =~ pat :: [(Int,Int)]
[(2,9),(14,4)]
```

失败的匹配用值 -1 作为元组的第一个元素（匹配偏移量）表示（如果我们请求单个元组），或者如果请求元组列表则返回空列表：

```ghci
> "eleemosynary" =~ pat :: (Int,Int)
(-1,0)
> "mondegreen" =~ pat :: [(Int,Int)]
[]
```

这不是 RegexContext 类型类的内置实例的完整列表。有关完整列表，请参阅 Text.Regex.Base.Context 模块的文档。

使函数在其结果类型上具有多态性的能力是静态类型语言中的一个不寻常特性。

## 更多关于正则表达式

### 混合和匹配字符串类型

正如我们之前指出的，=~ 运算符将其参数类型和返回类型用于类型类。我们可以对正则表达式和要匹配的文本使用 String 或严格的 ByteString 值：

```ghci
> :module +Data.ByteString.Char8
> :type pack "foo"
pack "foo" :: ByteString
```

然后我们可以尝试使用 String 和 ByteString 的不同组合：

```ghci
> pack "foo" =~ "bar" :: Bool
False
> "foo" =~ pack "bar" :: Int
0
> pack "foo" =~ pack "o" :: [(Int, Int)]
[(1,1),(2,1)]
```

但是，我们需要注意的是，如果我们想要字符串值作为匹配的结果，我们匹配的目标文本必须是相同类型的字符串。让我们在实践中看看这意味着什么：

```ghci
> pack "good food" =~ ".ood" :: [ByteString]
["good","food"]
```

在上面的例子中，我们使用 pack 将 String 转换为 ByteString。类型检查器接受这个，因为 ByteString 出现在结果类型中。但是如果我们尝试获取 String，那就不行了：

```ghci
> "good food" =~ ".ood" :: [ByteString]
<interactive>:1:0:
    No instance for (Text.Regex.Base.RegexLike.RegexContext
                       Regex [Char] [ByteString])
      arising from a use of `=~' at <interactive>:1:0-20
    Possible fix:
      add a instance declaration for
      (Text.Regex.Base.RegexLike.RegexContext Regex [Char] [ByteString])
      In the expression: "good food" =~ ".ood" :: [ByteString]
      In the definition of `it': it = "good food" =~ ".ood" :: [ByteString]
```

我们可以通过再次使左手边和结果的字符串类型匹配来轻松解决这个问题：

```ghci
> "good food" =~ ".ood" :: [String]
["good","food"]
```

此限制不适用于我们匹配的正则表达式的类型。它可以是 String 或 ByteString，不受其他使用中的类型约束。

### 你应该知道的其他事情

当你浏览 Haskell 库文档时，你会看到几个 regexp 相关的模块。Text.Regex.Base 下的模块定义了所有其他 regexp 模块遵循的公共 API。可以一次安装多个 regexp API 实现。在撰写本文时，GHC 附带了其中一个实现 Text.Regex.Posix。顾名思义，这个包提供了 POSIX regexp 语义。

### Perl 和 POSIX 正则表达式

如果你从 Perl、Python 或 Java 等语言转向 Haskell，并且你曾在其中一种语言中使用过正则表达式，你应该注意 Text.Regex.Posix 模块处理的 POSIX regexp 在一些重要方面与 Perl 风格的 regexp 不同。以下是一些更显著的区别。

当匹配替代方案时，Perl regexp 引擎执行左偏匹配，而 POSIX 引擎选择最贪婪的匹配。这意味着给定一个 regexp (foo|fo*) 和文本字符串 foooooo，Perl 风格的引擎将给出 foo 的匹配（最左边的匹配），而 POSIX 引擎将匹配整个字符串（最贪婪的匹配）。

POSIX regexp 的语法不如 Perl 风格的 regexp 统一。它们也缺乏 Perl 风格 regexp 提供的许多功能，如零宽断言和对贪婪匹配的控制。

其他 Haskell regexp 包可从 Hackage 下载。一些提供了比当前 POSIX 引擎更好的性能（例如 regex-tdfa）；其他提供了大多数程序员现在熟悉的 Perl 风格匹配（例如 regex-pcre）。都遵循我们本节涵盖的标准 API。

## 将 glob 模式转换为正则表达式

既然我们已经看到了匹配文本对正则表达式的多种方法，让我们把注意力转回到 glob 模式。我们想写一个函数，它接受一个 glob 模式并将其表示为正则表达式。glob 模式和 regexp 都是文本字符串，所以我们函数的类型似乎是明确的：

```haskell
-- file: ch08/GlobRegex.hs
module GlobRegex
    (
      globToRegex
    , matchesGlob
    ) where
import Text.Regex.Posix ((=~))
globToRegex :: String -> String
```

我们生成的正则表达式必须被锚定，以便它从字符串的开头开始匹配，到字符串的结尾结束：

```haskell
-- file: ch08/GlobRegex.hs
globToRegex cs = '^' : globToRegex' cs ++ "$"
```

回想一下，String 只是 [Char] 的同义词，一个 Char 列表。: 运算符将一个值（这里是 ^ 字符）放到列表的前面，其中列表是由尚未见到的 globToRegex' 函数返回的值。

### 使用一个在定义之前使用的值

Haskell 不要求值或函数在使用之前在源文件中声明或定义。一个定义出现在它首次使用的地方之后是完全正常的。Haskell 编译器此时不关心顺序。这给了我们灵活性，以对我们最有意义的方式组织代码，而不是遵循让编译器作者最轻松顺序。

Haskell 模块作者经常使用这种灵活性，在源文件中把"更重要"的代码放在前面，把"管道"工作降级到后面。这正是我们在这里展示 globToRegex 函数及其辅助函数的方式。

有了正则表达式的锚定，globToRegex' 函数将完成大部分翻译工作。我们将使用 Haskell 模式匹配的便利性来列举我们需要覆盖的每种情况：

```haskell
-- file: ch08/GlobRegex.hs
globToRegex' :: String -> String
globToRegex' "" = ""
globToRegex' ('*':cs) = ".*" ++ globToRegex' cs
globToRegex' ('?':cs) = '.' : globToRegex' cs
globToRegex' ('[':'!':c:cs) = "[^" ++ c : charClass cs
globToRegex' ('[':c:cs)     = '['  :  c : charClass cs
globToRegex' ('[':_)        = error "unterminated character class"
globToRegex' (c:cs) = escape c ++ globToRegex' cs
```

我们的第一个子句规定，如果我们到达 glob 模式的末尾（此时我们看到空字符串），我们返回 $，这是正则表达式中"匹配行尾"的符号。接下来是一系列将模式从 glob 语法切换到 regexp 语法的子句。最后一个子句传递所有其他字符，可能先转义它。

escape 函数确保 regexp 引擎不会将某些字符解释为正则表达式语法的组成部分：

```haskell
-- file: ch08/GlobRegex.hs
escape :: Char -> String
escape c | c `elem` regexChars = '\\' : [c]
         | otherwise = [c]
    where regexChars = "\\+()^$.{}]|"
```

charClass 辅助函数只检查字符类是否正确终止。它原样传递其输入，直到遇到 ]，然后将控制权交回 globToRegex'：

```haskell
-- file: ch08/GlobRegex.hs
charClass :: String -> String
charClass (']':cs) = ']' : globToRegex' cs
charClass (c:cs)   = c : charClass cs
charClass []       = error "unterminated character class"
```


确实，这看起来像一个合理的 regexp。我们能用它来匹配字符串吗？

```ghci

> "test.c" =~ globToRegex "t[ea]s*" :: Bool

```

它有效！现在让我们在 ghci 中稍微玩一下。我们可以为 fnmatch 创建一个临时定义并尝试：

```ghci

> :type fnmatch

           String -> source1 -> Bool
> fnmatch "d*" "myname"

```

fnmatch 这个名字实际上并没有"Haskell 风格"。迄今为止最常见的 Haskell 风格是函数具有描述性的"驼峰命名"名称。驼峰命名法将单词连接起来，除第一个单词外都大写。例如，"filename matches"这个词会变成 fileNameMatches。"驼峰命名法"这个名字来自于大写字母带来的"驼峰"。在我们的库中，我们将给这个函数命名为 matchesGlob：

```haskell
-- file: ch08/GlobRegex.hs
matchesGlob :: FilePath -> String -> Bool
name `matchesGlob` pat = name =~ globToRegex pat
```

你可能已经注意到，到目前为止我们用于变量的大多数名称都很短。作为一个经验法则，描述性的变量名在更长的函数定义中更有用，因为它们有助于可读性。对于一个两行的函数，长的变量名价值较小。

## 重要的题外话：编写惰性函数

在命令式语言中，globToRegex' 函数通常我们会用循环来表达。例如，Python 的标准 fnmatch 模块包含一个名为 translate 的函数，它的工作与我们的 globToRegex 函数完全相同。它是用循环编写的。

如果你通过 Scheme 或 ML 等语言接触过函数式编程，你可能已经被灌输了"模拟循环的方式是通过尾递归"的概念。

看看 globToRegex' 函数，我们可以看到它不是尾递归的。要了解原因，让我们再次查看它的最后一个子句（它的其他几个子句结构类似）：

```haskell
-- file: ch08/GlobRegex.hs
globToRegex' (c:cs) = escape c ++ globToRegex' cs
```

它递归地调用自己，递归应用的结果被用作 (++) 函数的参数。由于递归应用不是函数做的最后一件事，globToRegex' 不是尾递归的。

为什么我们的函数定义不是尾递归的？答案在于 Haskell 的非严格求值策略。在我们开始讨论这个之前，让我们快速谈谈为什么在传统语言中我们会尝试避免这种递归定义。这里是 (++) 运算符的一个更简单的定义。它是递归的，但不是尾递归的：

```haskell
-- file: ch08/append.hs
(++) :: [a] -> [a] -> [a]
(x:xs) ++ ys = x : (xs ++ ys)
[]     ++ ys = ys
```

在严格语言中，如果我们求值 "foo" ++ "bar"，整个列表被构造，然后返回。非严格求值将大部分工作推迟到需要的时候。

如果我们要求表达式 "foo" ++ "bar" 的一个元素，函数定义的第一个模式匹配，我们返回表达式 x : (xs ++ ys)。因为 (:) 构造函数是非严格的，xs ++ ys 的求值可以被推迟：我们在需要时以任何速率生成结果的更多元素。当我们生成更多结果时，我们将不再使用 x，所以垃圾收集器可以回收它。因为我们按需生成结果，而不是保留我们用完的部分，编译器可以用恒定空间求值我们的代码。

## 利用我们的模式匹配器

拥有一个可以匹配 glob 模式的函数很好，但我们希望能够实际使用它。在类 Unix 系统上，glob 函数返回与给定 glob 模式匹配的所有文件和目录的名称。让我们在 Haskell 中构建一个类似的函数。遵循 Haskell 的描述性命名规范，我们称我们的函数为 namesMatching：


我们指定 namesMatching 是我们的 Glob 模块的用户唯一能看到的名字。

将 System.FilePath 与 System.Directory 模块一起使用，我们可以编写一个可移植的 namesMatching 函数，它可以在类 Unix 和 Windows 系统上运行：

```haskell
-- file: ch08/Glob.hs
import System.FilePath (dropTrailingPathSeparator, splitFileName, (</>))
```

在这个模块中，我们将模拟一个"for"循环；获得我们对异常处理的第一次体验；当然还有使用我们刚刚写的 matchesGlob 函数：

```haskell
-- file: ch08/Glob.hs
import Control.Exception (handle)
import Control.Monad (forM)
import GlobRegex (matchesGlob)
```

由于目录和文件存在于"现实世界"的有效果的活动中，我们的 globbing 函数必须在其结果类型中包含 IO。

如果我们传入的字符串不包含模式字符，我们只需检查给定的名称在文件系统中是否存在。（注意，我们在这里使用 Haskell 的函数 guard 语法来写一个简洁的定义。"if" 也可以做到，但不如 guard 美观。）

```haskell
-- file: ch08/Glob.hs
isPattern :: String -> Bool
isPattern = any (`elem` "[*?")
namesMatching pat
  | not (isPattern pat) = do
    exists <- doesNameExist pat
    return (if exists then [pat] else [])
```

doesNameExist 指的是我们稍后将定义的函数。

如果字符串是一个 glob 模式呢？我们的函数定义继续：

```haskell
-- file: ch08/Glob.hs
  | otherwise = do
    case splitFileName pat of
      ("", baseName) -> do
          curDir <- getCurrentDirectory
          listMatches curDir baseName
      (dirName, baseName) -> do
          dirs <- if isPattern dirName
                  then namesMatching (dropTrailingPathSeparator dirName)
                  else return [dirName]
          let listDir = if isPattern baseName
                        then listMatches
                        else listPlain
          pathNames <- forM dirs $ \dir -> do
                           baseNames <- listDir dir baseName

```

我们使用 splitFileName 将字符串分割成"除了最后一个名称之外的所有东西"和"最后一个名称"这对。如果第一个元素为空，我们正在当前目录中查找模式。否则，我们必须检查目录名并查看它是否包含模式。如果不包含，我们创建目录名的单例列表。如果包含模式，我们列出所有匹配的目录。

### 需要注意的事项

System.FilePath 模块可能有点棘手。刚刚显示的例子就是一个很好的例子；splitFileName 函数在它返回的目录名末尾保留一个尾随斜杠：

```ghci

("foo/","bar")
```

如果我们不记得（或不知道足够多）来删除那个斜杠，我们会在 namesMatching 中无限递归，因为 splitFileName 的以下行为：

```ghci
> splitFileName "foo/"
("foo/","")
```

（你可以猜测我们发生了什么，导致我们添加了这个注释！）

最后，我们在每个目录中收集所有匹配项，给我们一个列表的列表，并将它们连接成单个名称列表。

上面不熟悉的 forM 函数有点像"for"循环：它将第二个参数（一个动作）映射到第一个参数（一个列表），并返回结果列表。

我们有一些需要清理的松散结尾。第一个是上面使用的 doesNameExist 函数的定义。System.Directory 模块不允许我们检查文件系统中是否存在名称。它迫使我们决定要检查文件还是目录。这个 API 很笨拙，所以我们将两个检查合并到一个函数中。以性能的名义，我们首先检查文件，因为文件比目录常见得多：

```haskell
-- file: ch08/Glob.hs
doesNameExist :: FilePath -> IO Bool
doesNameExist name = do
    fileExists <- doesFileExist name
    if fileExists
      then return True
      else doesDirectoryExist name
```

我们还有两个其他函数要定义，每个都返回目录中的名称列表。listMatches 函数返回目录中与给定 glob 模式匹配的所有文件的列表：

```haskell
-- file: ch08/Glob.hs
listMatches :: FilePath -> String -> IO [String]
listMatches dirName pat = do
    dirName' <- if null dirName
                then getCurrentDirectory
                else return dirName
    handle (const (return [])) $ do
        names <- getDirectoryContents dirName'
        let names' = if isHidden pat
                     then filter isHidden names
                     else filter (not . isHidden) names
        return (filter (`matchesGlob` pat) names')
isHidden ('.':_) = True
isHidden _       = False
```

listPlain 函数根据传入的单 个名称是否存在返回空列表或单例列表：

```haskell
-- file: ch08/Glob.hs
listPlain :: FilePath -> String -> IO [String]
listPlain dirName baseName = do
    exists <- if null baseName
              then doesDirectoryExist dirName
              else doesNameExist (dirName </> baseName)
    return (if exists then [baseName] else [])
```

如果我们仔细看 listMatches 的定义，会看到一个名为 handle 的函数调用。早些时候，我们从 Control.Exception 模块导入了它；正如该导入所暗示的，这给了我们对 Haskell 中异常处理的第一次体验。让我们进入 ghci 看看我们能发现什么：

```ghci
> :module +Control.Exception
> :type handle
handle :: (Exception -> IO a) -> IO a -> IO a
```

这告诉我们 handle 接受两个参数。第一个是一个函数，它被传入一个异常值，并且可能有副作用（参见其返回类型中的 IO）；这是如果抛出异常则运行的处理器。第二个参数是可能抛出异常的代码。

至于异常处理器，handle 的类型约束它返回与抛出异常的代码体相同类型的值。所以它的选择是要么抛出异常，要么像我们的情况一样返回一个字符串列表。

const 函数接受两个参数——无论其第二个参数是什么，它总是返回其第一个参数：

```ghci
> :type const
const :: a -> b -> a
> :type return []
return [] :: (Monad m) => m [a]
> :type handle (const (return []))
handle (const (return [])) :: IO [a] -> IO [a]
```

我们使用 const 来编写一个忽略传入的异常的异常处理器。相反，它导致我们的代码在捕获异常时返回一个空列表。

我们不会再多说什么关于异常处理的内容。还有更多要讲的，所以我们将在第 19 章返回异常处理的主题。

## 通过 API 设计处理错误

如果我们的 globToRegex 被传递了一个格式错误的模式，不一定是灾难。也许用户输入模式时拼写错误，在这种情况下，我们希望能够报告一个有意义的错误消息。

在这种问题发生时调用 error 函数可能是一种激烈的反应（探索其后果是第 210 页"练习"的焦点）。error 抛出一个异常。纯 Haskell 代码无法处理异常，所以控制将从我们的纯代码中冲出，进入最近的位于 IO 中并安装了适当异常处理程序的调用者。如果没有安装这样的处理程序，Haskell 运行时将默认终止我们的程序（或者在 ghci 中打印一个讨厌的错误消息）。

所以调用 error 有点像拉动战斗机弹射座椅的手柄。我们正在从一个我们无法优雅处理的灾难性情况中跳出来，当我们落地时可能会有很多燃烧的残骸。

我们已经确定 error 是用于灾难的，但我们在 globToRegex 中仍在使用它。在这种情况下，格式错误的输入应该被拒绝，但不要变成大事。什么是更好的处理方式？

Haskell 的类型系统和库来拯救！我们可以使用预定义的 Either 类型对 globToRegex 的类型签名中的失败可能性进行编码：

```haskell
-- file: ch08/GlobRegexEither.hs
type GlobError = String
globToRegex :: String -> Either GlobError String
```

由 globToRegex 返回的值现在要么是 Left "错误消息"，要么是 Right "有效 regexp"。这个返回类型迫使我们的调用者处理错误的可能性。（你会发现 Either 类型的这种用法在 Haskell 代码中经常出现。）

## 将我们的代码投入使用

namesMatching 函数本身并不令人兴奋，但它是一个有用的构建块。将其与更多函数结合使用，我们可以开始做有趣的事情。

这是一个例子。让我们定义一个 renameWith 函数，它不是简单地重命名文件，而是将一个函数应用到文件名上，并将文件重命名 为该函数返回的任何内容：

```haskell
-- file: ch08/Useful.hs
import System.FilePath (replaceExtension)
import System.Directory (doesFileExist, renameDirectory, renameFile)
import Glob (namesMatching)
renameWith :: (FilePath -> FilePath)
           -> FilePath
           -> IO FilePath
renameWith f path = do
    let path' = f path
    rename path path'
    return path'
```

又一次，我们用一个辅助函数来克服 System.Directory 中文件/目录分割的笨拙：

```haskell
-- file: ch08/Useful.hs
rename :: FilePath -> FilePath -> IO ()
rename old new = do
    isFile <- doesFileExist old
    let f = if isFile then renameFile else renameDirectory
    f old new
```

System.FilePath 模块提供了许多用于操作文件名的有用函数。这些函数与我们的 renameWith 和 namesMatching 函数很好地结合在一起，这样我们就可以快速使用它们来创建具有复杂行为的函数。例如，这个简洁的函数改变了 C++ 源文件的文件名后缀约定：

```haskell
-- file: ch08/Useful.hs
cc2cpp =
  mapM (renameWith (flip replaceExtension ".cpp")) =<< namesMatching "*.cc"
```

cc2cpp 函数使用了一些我们会反复看到的函数。flip 函数接受另一个函数作为参数并交换其参数的顺序（在 ghci 中检查 replaceExtension 的类型以查看原因）。=<< 函数将其右侧动作的结果传递给其左侧的动作。

1. Glob 模式足够简单，可以直接用 Haskell 编写匹配器，而不是通过 regexp 机制。试试看。
