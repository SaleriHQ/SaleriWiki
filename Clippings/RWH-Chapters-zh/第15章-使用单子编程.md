# 第15章-使用单子编程

## 高尔夫练习：关联列表

Web 客户端和服务器经常以简单的键值对文本列表的形式传递信息：

```
name=Attila+%42The+Hun%42&occupation=Khan
```

这种编码名为 application/x-www-form-urlencoded，很容易理解。每个键值对由 & 字符分隔。在一对中，键是一系列字符，后跟 =，后跟值。

我们显然可以将键表示为 String，但 HTTP 规范并不清楚键是否必须后跟值。我们可以通过将值表示为 `Maybe String` 来捕捉这种歧义。如果我们用 Nothing 表示一个值，则没有值存在。如果我们将字符串包装在 Just 中，则有一个值。使用 Maybe 可以让我们区分"无值"和"空值"。

Haskell 程序员使用关联列表这个名称来表示类型 `[(a, b)]`，我们可以将每个元素视为键和值之间的关联。这个名称源自 Lisp 社区，通常缩写为 alist。因此，我们可以将前面的字符串表示为以下 Haskell 值：

```haskell
-- file: ch15/MovieReview.hs
    [("name",       Just "Attila \"The Hun\""),
     ("occupation", Just "Khan")]
```

在"解析 URL 编码的查询字符串"一节中，我们将解析 application/x-www-form-urlencoded 字符串，并将结果表示为 `[(String, Maybe String)]` 的 alist。假设我们想使用其中一个 alist 来填充数据结构：

```haskell
-- file: ch15/MovieReview.hs
data MovieReview = MovieReview {
      revTitle :: String
    , revUser :: String
    , revReview :: String
    }
```

我们将从用一个 naive 函数 belabor the obvious 开始：

```haskell
-- file: ch15/MovieReview.hs
simpleReview :: [(String, Maybe String)] -> Maybe MovieReview
simpleReview alist =
  case lookup "title" alist of
    Just (Just title@(_:_)) ->
      case lookup "user" alist of
        Just (Just user@(_:_)) ->
          case lookup "review" alist of
            Just (Just review@(_:_)) ->
                Just (MovieReview title user review)
            _ -> Nothing -- no review
        _ -> Nothing -- no user
    _ -> Nothing -- no title
```

只有当 alist 包含所有必要的值且它们都是非空字符串时，它才返回 MovieReview。然而，它验证其输入这一事实是它唯一的优点。它严重遭受我们已经学会警惕的"阶梯式"问题，而且它知道 alist 表示的详细信息。

由于我们现在非常熟悉 Maybe monad，我们可以整理阶梯式嵌套：

```haskell
-- file: ch15/MovieReview.hs
maybeReview alist = do
    title <- lookup1 "title" alist
    user <- lookup1 "user" alist
    review <- lookup1 "review" alist
    return (MovieReview title user review)
lookup1 key alist = case lookup key alist of
                      Just (Just s@(_:_)) -> Just s
                      _ -> Nothing
```

虽然这更整洁，但我们仍在重复自己。我们可以利用 MovieReview 构造函数充当普通纯函数这一事实，通过将其提升到 monad 中，就像我们在"混合纯代码和 Monadic 代码"一文中讨论的那样：

```haskell
-- file: ch15/MovieReview.hs
liftedReview alist =
    liftM3 MovieReview (lookup1 "title" alist)
                       (lookup1 "user" alist)
                       (lookup1 "review" alist)
```

我们这里仍然有一些重复，但它大大减少了，而且也更难去除。

## 广义提升

虽然使用 liftM3 整理了我们的代码，但我们通常不能使用 liftM 系列函数来解决这类问题，因为标准库只定义到 liftM5。我们可以写任意数量的变体，但这将是苦差事。

如果我们有一个 say 接受 10 个参数的构造函数或纯函数，并决定坚持使用标准库，你可能认为我们会运气不佳。

当然，我们的工具箱还没有空。在 Control.Monad 中，有一个名为 ap 的函数，其类型签名很有趣：

```ghci
ghci> :m +Control.Monad
ghci> :type ap
ap :: (Monad m) => m (a -> b) -> m a -> m b
```

你可能想知道谁会将单参数纯函数放入 monad 中，为什么。回想一下，然而，所有 Haskell 函数实际上只接受一个参数，你就会开始看到这个与 MovieReview 构造函数的关系：

```ghci
ghci> :type MovieReview
MovieReview :: String -> String -> String -> MovieReview
```

我们可以一样容易地将该类型写为：

```haskell
String -> (String -> (String -> MovieReview))
```

如果我们使用普通的 liftM 将 MovieReview 提升到 Maybe monad，我们将有一个类型为：

```haskell
Maybe (String -> (String -> (String -> MovieReview)))
```

我们现在可以看到这个类型适合作为 ap 的参数，在这种情况下，结果类型将是：

```haskell
Maybe (String -> (String -> MovieReview))
```

我们可以依次传递这个，再次调用 ap，并继续链接，直到我们结束这个定义：

```haskell
-- file: ch15/MovieReview.hs
apReview alist =
    MovieReview `liftM` lookup1 "title" alist
                   `ap` lookup1 "user" alist
                   `ap` lookup1 "review" alist
```

我们可以根据需要链接任意多次 ap 的应用，从而绕过 liftM 系列函数。

看待 ap 的另一种有用方式是，它是熟悉的 ($) 运算符的 monadic 等价物；想想将 ap 发音为 apply。当我们比较两个函数的类型签名时，我们可以清楚地看到这一点：

```ghci
ghci> :type ($)
($) :: (a -> b) -> a -> b
ghci> :type ap
ap :: (Monad m) => m (a -> b) -> m a -> m b
```

事实上，ap 通常被定义为 liftM2 id 或 liftM2 ($)。

## 寻找替代方案

这里有一个表示个人电话号码的简单表示：

```haskell
-- file: ch15/VCard.hs
data Context = Home | Mobile | Business
               deriving (Eq, Show)
type Phone = String
albulena = [(Home, "+355-652-55512")]
nils = [(Mobile, "+47-922-55-512"), (Business, "+47-922-12-121"),
        (Home, "+47-925-55-121"), (Business, "+47-922-25-551")]
twalumba = [(Business, "+260-02-55-5121")]
```

假设我们想联系某个人进行个人通话。我们不想要他的商务号码，而且我们更愿意使用他的家庭号码（如果他有的话）而不是他的手机号码：

```haskell
-- file: ch15/VCard.hs
onePersonalPhone :: [(Context, Phone)] -> Maybe Phone
onePersonalPhone ps = case lookup Home ps of
                        Nothing -> lookup Mobile ps
                        Just n -> Just n
```

当然，如果我们使用 Maybe 作为结果类型，我们就无法容纳某人可能有多个满足我们条件的号码的情况。为此，我们切换到列表：

```haskell
-- file: ch15/VCard.hs
allBusinessPhones :: [(Context, Phone)] -> [Phone]
allBusinessPhones ps = map snd numbers
    where numbers = case filter (contextIs Business) ps of
                      [] -> filter (contextIs Mobile) ps
                      ns -> ns
contextIs a (b, _) = a == b
```

注意这两个函数的 case 表达式结构相似——一个替代方案处理第一个 lookup 返回空结果的情况，而另一个处理非空情况：

```ghci
ghci> onePersonalPhone twalumba
Nothing
ghci> onePersonalPhone albulena
Just "+355-652-55512"
ghci> allBusinessPhones nils
["+47-922-12-121","+47-922-25-551"]
```

Haskell 的 Control.Monad 模块定义了一个类型类 MonadPlus，它让我们从 case 表达式中抽象出共同模式：

```haskell
-- file: ch15/VCard.hs
class Monad m => MonadPlus m where
   mzero :: m a
   mplus :: m a -> m a -> m a
```

值 mzero 代表空结果，而 mplus 将两个结果组合成一个。

以下是 Maybe 和列表的 mzero 和 mplus 的标准定义：

```haskell
-- file: ch15/VCard.hs
instance MonadPlus [] where
   mzero = []
   mplus = (++)
instance MonadPlus Maybe where
   mzero = Nothing
   Nothing `mplus` ys  = ys
   xs      `mplus` _ = xs
```

我们现在可以使用 mplus 完全摆脱 case 表达式。为了多样化，让我们获取一个商务电话和所有个人电话号码：

```haskell
-- file: ch15/VCard.hs
oneBusinessPhone :: [(Context, Phone)] -> Maybe Phone
oneBusinessPhone ps = lookup Business ps `mplus` lookup Mobile ps
allPersonalPhones :: [(Context, Phone)] -> [Phone]
allPersonalPhones ps = map snd $ filter (contextIs Home) ps `mplus`
                                 filter (contextIs Mobile) ps
```

在这些函数中，因为我们知道 lookup 返回类型为 Maybe 的值，而 filter 返回一个列表，所以在每种情况下使用哪个版本的 mplus 是明显的。

更有趣的是，我们可以使用 mzero 和 mplus 来编写对任何 MonadPlus 实例都有用的函数。例如，这里的标准 lookup 函数，它返回类型为 Maybe 的值：

```haskell
-- file: ch15/VCard.hs
lookup :: (Eq a) => a -> [(a, b)] -> Maybe b
lookup _ []                      = Nothing
lookup k ((x,y):xys) | x == k    = Just y
                     | otherwise = lookup k xys
```

我们可以轻松地将结果类型推广到任何 MonadPlus 实例，如下所示：

```haskell
-- file: ch15/VCard.hs
lookupM :: (MonadPlus m, Eq a) => a -> [(a, b)] -> m b
lookupM _ []    = mzero
lookupM k ((x,y):xys)
    | x == k    = return y `mplus` lookupM k xys
    | otherwise = lookupM k xys
```

这让我们获得无结果或一个结果（如果我们的结果类型是 Maybe）；所有结果（如果我们的结果类型是列表）；或者更适合其他一些奇特的 MonadPlus 实例的结果。

对于我们在这里提供的小函数，使用 mplus 没什么好处。优势在于更复杂的代码以及独立于其执行的 monad 的代码。即使你发现自己不需要为你自己的代码使用 MonadPlus，你也可能在其他人的项目中遇到它。

### mplus 这个名称并不意味着加法

即使 mplus 函数包含文本"plus"，你不应该认为它必然意味着我们正在尝试添加两个值。

取决于我们工作的 monad，mplus 可能会实现看起来像加法的操作。例如，列表 monad 中的 mplus 实现为 (++) 运算符：

```ghci
ghci> [1,2,3] `mplus` [4,5,6]
[1,2,3,4,5,6]
```

然而，如果我们切换到另一个 monad，与加法的明显相似性就消失了：

```ghci
ghci> Just 1 `mplus` Just 2
Just 1
```

### 使用 MonadPlus 的规则

MonadPlus 类型类的实例除了通常的 monad 规则外，还必须遵循一些简单的规则。

如果 mzero 出现在绑定表达式左侧，实例必须短路。换句话说，表达式 `mzero >>= f` 必须求值为与 mzero 相同的结果：

```haskell
-- file: ch15/MonadPlus.hs
    mzero >>= f == mzero
```

如果 mzero 出现在序列表达式的右侧，实例必须短路：

```haskell
-- file: ch15/MonadPlus.hs
    v >> == mzero
```

### 使用 MonadPlus 安全失败

当我们在"Monad 类型类"一文中介绍 fail 函数时，我们努力警告反对它的使用：在许多 monad 中，它被实现为对 error 的调用，这有令人不愉快的后果。

MonadPlus 类型类为我们提供了一种更温和的方式来使计算失败，而不会让 fail 或 error 在我们面前爆炸。我们刚刚介绍的规则允许我们在需要的地方在代码中引入 mzero，计算将在该点短路。

在 Control.Monad 模块中，标准函数 guard 以一种方便的形式包装了这个想法：

```haskell
-- file: ch15/MonadPlus.hs
guard        :: (MonadPlus m) => Bool -> m ()
guard True   =  return ()
guard False  =  mzero
```

作为一个简单的例子，这里有一个取数字 x 并计算其对某个其他数字 n 的模值的函数。如果结果为零，它返回 x；否则，返回当前 monad 的 mzero：

```haskell
-- file: ch15/MonadPlus.hs
x `zeroMod` n = guard ((x `mod` n) == 0) >> return x
```

## 隐藏管道的冒险

在"使用 State Monad：生成随机值"一文中，我们展示了如何使用 State monad 以一种易于使用的方式访问随机数。

我们开发的代码的一个缺点是它是泄漏的：用户知道他们正在 State monad 内部执行。这意味着他们可以像我们——作者——一样容易地检查和修改随机数生成器的状态。

人类的本性决定，如果我们暴露了我们的内部工作，某人肯定会来修改它们。对于一个足够小的程序，这可能很好，但在较大的软件项目中，当一个库的使用者以其他使用者没有准备好的方式修改其内部时， resulting bugs 可能是最难追踪的。这些 bug 发生在一个层面上，我们不太可能质疑我们对库的基本假设，直到我们穷尽了所有其他查询途径之后。

更糟糕的是，一旦我们让我们的实现暴露了一段时间，而且某个善意的人不可避免地绕过我们的 API 直接使用实现，那么如果我们需要修复 bug 或进行增强，我们就会陷入困境。我们要么可以修改我们的内部并破坏依赖于它们的代码；要么我们被现有的内部困住，必须尝试找到另一种我们需要做的改变方式。

我们如何修改我们的随机数 monad，以便隐藏我们正在使用 State monad 这一事实？我们需要以某种方式阻止我们的用户能够调用 get 或 put。这并不难做到，而且它引入了一些我们将在日常 Haskell 编程中经常重用的技巧。

为了扩大范围，我们将超越随机数，实现一个提供任何类型的唯一值的 monad。我们给 monad 的名称是 Supply。我们将提供执行函数 runSupply，带有一个值列表（我们有责任确保每个值都是唯一的）：

```haskell
-- file: ch15/Supply.hs
runSupply :: Supply s a -> [s] -> (a, [s])
```

monad 不关心这些值是什么。它们可能是随机数、临时文件的名称或 HTTP cookie 的标识符。

在 monad 内部，每次消费者请求一个值时，下一个 action 将从列表中取出下一个并将其交给消费者。每个值都包装在 Maybe 构造函数中，以防列表不够长以满足需求：

```haskell
-- file: ch15/Supply.hs
next :: Supply s (Maybe s)
```

为了隐藏我们的管道，在我们的模块声明中，我们只导出类型构造函数、执行函数和 next action：

```haskell
-- file: ch15/Supply.hs
module Supply
    (
      Supply
    , next
    , runSupply
    ) where
```

由于导入库的模块无法看到 monad 的内部，它无法操作它们。

我们的管道非常简单。我们使用 newtype 声明来包装现有的 State monad：

```haskell
-- file: ch15/Supply.hs
import Control.Monad.State
newtype Supply s a = S (State [s] a)
```

s 参数是我们将要提供的唯一值的类型，a 是通常的类型参数，我们必须提供它以使我们的类型成为一个 monad。

我们对 Supply 类型使用 newtype 以及我们的模块头 join forces 阻止我们的客户使用 State monad 的 get 和 set actions。因为我们的模块不导出 S 数据构造函数，客户端没有办法以编程方式看到我们正在包装 State monad 或访问它。

在这一点上，我们有了一个类型 Supply，需要使其成为 Monad 类型类的实例。我们可以遵循定义 `(>>=)` 和 return 的通常模式，但这将是纯样板代码。我们所做的只是用我们的 S 值构造函数包装和解包 State monad 的 `(>>=)` 和 return 版本。以下是此类代码的样子：

```haskell
-- file: ch15/AltSupply.hs
unwrapS :: Supply s a -> State [s] a
unwrapS (S s) = s
instance Monad (Supply s) where
    s >>= m = S (unwrapS s >>= unwrapS . m)
    return = S . return
```

Haskell 程序员不喜欢样板代码，确实 GHC 有一个可爱的语言扩展来消除这项工作。要使用它，我们在源文件顶部、模块头之前添加以下指令：

```haskell
-- file: ch15/Supply.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
```

通常，我们只能自动派生少数标准类型类的实例，如 Show 和 Eq。顾名思义，GeneralizedNewtypeDeriving 扩展扩展了我们派生类型类实例的能力，并且它是特定于 newtype 声明的。如果我们要包装的类型是任何类型类的实例，这些扩展可以自动使我们的新类型成为该类型类的实例，如下所示：

```haskell
-- file: ch15/Supply.hs
    deriving (Monad)
```

这获取底层类型的 `(>>=)` 和 return 实现，用我们的 S 数据构造函数添加必要的包装和拆包装，并使用这些函数的新版本来为我们派生一个 Monad 实例。

我们在这里获得的不仅仅在这个例子中有用。我们可以使用 newtype 来包装任何底层类型；我们有选择地只暴露那些我们想要公开的类型类实例；而且我们几乎不费力气来创建这些更窄、更专门的类型。

现在我们看到了 GeneralizedNewtypeDeriving 技术，剩下的就是提供 next 和 runSupply 的定义：

```haskell
-- file: ch15/Supply.hs
next = S $ do st <- get
              case st of
                [] -> return Nothing
                (x:xs) -> do put xs
                             return (Just x)
runSupply (S m) xs = runState m xs
```

如果我们将模块加载到 ghci，我们可以尝试一些简单的方法：

```ghci
ghci> :load Supply
[1 of 1] Compiling Supply           ( Supply.hs, interpreted )
Ok, modules loaded: Supply.
ghci> runSupply next [1,2,3]

(Just 1,[2,3])
ghci> runSupply (liftM2 (,) next next) [1,2,3]
((Just 1,Just 2),[3])
ghci> runSupply (liftM2 (,) next next) [1]
((Just 1,Nothing),[])
```

我们也可以验证 State monad 没有以某种方式泄漏：

```ghci
ghci> :browse Supply

runSupply :: Supply s a -> [s] -> (a, [s])
ghci> :info Supply

```

### 提供随机数

如果我们想将 Supply monad 用作随机数来源，我们面临一个小困难。理想情况下，我们希望能够为其提供随机数的无限流。我们可以在 IO monad 中获取 StdGen，但完成后我们必须"放回"一个不同的 StdGen。如果没有，我们将得到相同的 state 下一个获取 StdGen 的代码段。这意味着它将生成与我们相同的随机数，这可能是灾难性的。

从我们目前看到的 System.Random 模块的部分来看，很难调和这些需求。我们可以使用 getStdRandom，其类型确保当我们获取 StdGen 时，我们会放回一个：

```ghci
ghci> :type getStdRandom
getStdRandom :: (StdGen -> (a, StdGen)) -> IO a
```

我们可以使用 random 获取一个新的 StdGen 当他们给我们一个随机数时。我们可以使用 randoms 获取无限列表的随机数。但是我们如何同时获得随机数的无限列表和新的 StdGen？

答案在于 RandomGen 类型类的 split 函数，它接受一个随机数生成器并将其转换为两个生成器。像这样拆分随机生成器是一件很不寻常的事情：在一个纯函数设置中，它显然非常有用，但在非纯语言中，它本质上要么从不必要，要么语言不提供。

使用 split 函数，我们可以使用一个 StdGen 生成随机数的无限列表来提供给 runSupply，同时将另一个提供给 IO monad：

```haskell
-- file: ch15/RandomSupply.hs
import Supply
import System.Random hiding (next)
randomsIO :: Random a => IO [a]
randomsIO =
    getStdRandom $ \g ->
        let (a, b) = split g
        in (randoms a, b)
```

如果我们正确编写了这个函数，我们的例子应该在每次调用时打印不同的随机数：

```ghci
ghci> :load RandomSupply
[1 of 2] Compiling Supply           ( Supply.hs, interpreted )
[2 of 2] Compiling RandomSupply     ( RandomSupply.hs, interpreted )
Ok, modules loaded: RandomSupply, Supply.
ghci> (fst . runSupply next) `fmap` randomsIO
<interactive>:1:17:
    Ambiguous occurrence `next'
    It could refer to either `Supply.next', imported from Supply at RandomSupply.hs:4:
                                              (defined at Supply.hs:32:0)
                          or `System.Random.next', imported from System.Random
ghci> (fst . runSupply next) `fmap` randomsIO
<interactive>:1:17:
    Ambiguous occurrence `next'
    It could refer to either `Supply.next', imported from Supply at RandomSupply.hs:4:
                                              (defined at Supply.hs:32:0)
                          or `System.Random.next', imported from System.Random
```

回想一下我们的 runSupply 函数返回执行 monadic action 的结果和列表的未消耗 remainder。由于我们传递了随机数的无限列表，我们与 fst 组合以确保当 ghci 尝试打印结果时我们不会被随机数淹没。

### 又一轮高尔夫

在 Haskell 代码中，将函数应用于元组的一个元素并用另一个原始元素构造新元组而保持另一个原始元素不变的图案很常见，它已被转换为标准代码。

两个函数 first 和 second 在 Control.Arrow 模块中执行此操作：

```ghci
ghci> :m +Control.Arrow
ghci> first (+3) (1,2)
(4,2)
ghci> second odd ('a',1)
('a',True)
```

（事实上，我们已经在"无重叠实例的 JSON 类型类"一文中遇到过 second。）我们可以使用 first 来简化我们的 randomsIO 定义，将其转换为一行的形式：

```haskell
-- file: ch15/RandomGolf.hs
import Control.Arrow (first)
randomsIO_golfed :: Random a => IO [a]
randomsIO_golfed = getStdRandom (first randoms . split)
```

## 分离接口和实现

在上一节中，我们看到了如何隐藏我们正在使用 State monad 来保存 Supply monad 状态的事实。

另一种使代码更具模块化的重要方法涉及将其接口（代码可以做什么）与其实现（它如何做）分离。

System.Random 中的标准随机数生成器众所周知非常慢。如果我们使用我们的 randomsIO 函数为其提供随机数，那么我们的 next action 性能不会好。

我们可以处理这个问题的一个简单有效的方法是使用更好的随机数来源为 Supply 提供服务。不过，让我们先搁置这个想法，考虑另一种方法，这在许多情况下很有用。我们将使用类型类将我们可以对 monad 执行的操作与其工作方式分开：

```haskell
-- file: ch15/SupplyClass.hs
class (Monad m) => MonadSupply s m | m -> s where
    next :: m (Maybe s)
```

这个类型类定义了任何供应 monad 必须实现的接口。它值得仔细检查，因为它使用了几种不熟悉的 Haskell 语言扩展。我们将在以下各节中介绍每一个。

### 多参数类型类

我们应该如何阅读类型类中的片段 MonadSupply s m？如果我们添加括号，等价的表达式是 (MonadSupply s) m，这更清楚一些。换句话说，给定一些是 Monad 的类型变量 m，我们可以使其成为类型类 MonadSupply s 的实例。与常规类型类不同，这个有一个参数。

由于此语言扩展允许类型类有多个参数，其名称为 MultiParamTypeClasses。参数 s 与同名 Supply 类型的参数起到相同的作用：它表示 next 函数分发的值的类型。

注意我们不需要在 MonadSupply s 的定义中提及 `(>>=)` 或 return，因为类型类的上下文（超类）要求 MonadSupply s 已经是 Monad。

### 函数依赖

回顾一下我们之前忽略的片段，| m -> s 是一个函数依赖，通常称为 fundep。我们可以将竖线 | 读作"使得"，将箭头 -> 读作"唯一确定"。我们的函数依赖在 m 和 s 之间建立了一种关系。

FunctionalDependencies 语言 pragma 控制函数依赖的可用性。

我们声明这种关系的目的是帮助类型检查器。回想一下 Haskell 类型检查器本质上是一个定理证明器，而且它在操作上是保守的：它坚持其证明必须终止。非终止证明导致编译器放弃或陷入无限循环。

通过我们的函数依赖，我们告诉类型检查器，每次它在 MonadSupply s 的上下文中看到某个 monad m 使用时，类型 s 是与之一起使用的唯一可接受的类型。如果我们省略函数依赖，类型检查器将简单地放弃并显示错误消息。

很难想象 m 和 s 之间的关系真正意味着什么，所以让我们看一个这个类型类的实例：

```haskell
-- file: ch15/SupplyClass.hs
import qualified Supply as S
instance MonadSupply s (S.Supply s) where
    next = S.next
```

这里，类型变量 m 被类型 S.Supply s 替换。多亏了我们的函数依赖，类型检查器现在知道当它看到一个类型 S.Supply s 时，该类型可以用作类型类 MonadSupply s 的实例。

如果我们没有函数依赖，类型检查器将无法找出类型类 MonadSupply s 的类型参数与类型 Supply s 的类型参数之间的关系，它将中止编译并显示错误。定义本身可以编译；类型错误只会在我们第一次尝试使用它时出现。

为了剥离最后一层抽象，考虑类型 S.Supply Int。如果没有函数依赖，我们可以声明这是 MonadSupply s 的实例。然而，如果我们尝试编写使用此实例的代码，编译器将无法弄清楚类型的 Int 参数需要与类型类的 s 参数相同，并将报告错误。

函数依赖可能难以理解，一旦我们超越简单的用法，它们往往在实践中证明难以使用。幸运的是，函数依赖的最常见用途是在像我们这样简单的情况下，它们几乎不会造成麻烦。

### 完善我们的模块

如果我们将类型类和实例保存在名为 SupplyClass.hs 的源文件中，我们需要添加如下模块头：

```haskell
-- file: ch15/SupplyClass.hs
{-# LANGUAGE FlexibleInstances, FunctionalDependencies,
             MultiParamTypeClasses #-}
module SupplyClass
    (
      MonadSupply(..)
    , S.Supply
    , S.runSupply
    ) where
```

FlexibleInstances 扩展是必要的，以便编译器接受我们的实例声明。此扩展在某些情况下放宽了编写实例的正常规则，但仍让编译器的类型检查器保证它将终止。我们在这里需要 FlexibleInstances 是由于我们使用函数依赖，但细节不幸地超出了本书的范围。

### 如何知道何时需要语言扩展

如果 GHC 无法编译一段代码，因为它需要启用某种语言扩展，它会告诉我们应该使用哪个扩展。例如，如果它决定我们的代码需要灵活实例支持，它会建议我们尝试使用 -XFlexibleInstances 选项编译。-X 选项与 LANGUAGE 指令具有相同的效果：它启用特定扩展。

最后，请注意我们从此模块重新导出 runSupply 和 Supply 名称。从一个模块导出一个在其他模块中定义的名称是完全合法的。在我们的例子中，这意味着客户端代码只需导入 SupplyClass 模块，而不需要同时导入 Supply 模块。这减少了使用我们代码的用户需要记住的"活动部件"数量。

## 针对 Monad 接口编程

这里有一个简单的函数，它从我们的 Supply monad 获取两个值，将它们格式化为字符串，然后返回它们：

```haskell
-- file: ch15/Supply.hs
showTwo :: (Show s) => Supply s String
showTwo = do
  a <- next
  b <- next
  return (show "a: " ++ show a ++ ", b: " ++ show b)
```

此代码通过其结果类型绑定到我们的 Supply monad。通过修改函数的类型，我们可以轻松地推广到任何实现我们 MonadSupply 接口的 monad。请注意，函数体保持不变：

```haskell
-- file: ch15/SupplyClass.hs
showTwo_class :: (Show s, Monad m, MonadSupply s m) => m String
showTwo_class = do
  a <- next
  b <- next
  return (show "a: " ++ show a ++ ", b: " ++ show b)
```

## The Reader Monad

State monad 让我们通过代码传递一块可变状态。有时，我们希望能够传递一些不可变状态，比如程序的配置数据。我们可以使用 State monad 来实现此目的，但我们可能会发现自己意外修改了应该保持不变的数据。

让我们暂时忘记 monad，考虑一个具有我们所需特性的函数应该做什么。它应该接受某个类型 e 的值（对于环境）作为代表我们传入的数据，并返回某个其他类型 a 的值作为其结果。我们想要的整体类型是 `e -> a`。

要将此类型转换为方便的 Monad 实例，我们将其包装在 newtype 中：

```haskell
-- file: ch15/SupplyInstance.hs
newtype Reader e a = R { runReader :: e -> a }
```

将其制作为 Monad 实例不需要太多工作：

```haskell
-- file: ch15/SupplyInstance.hs
instance Monad (Reader e) where
    return a = R $ \_ -> a
    m >>= k = R $ \r -> runReader (k (runReader m r)) r
```

我们可以将类型 e 的值视为我们正在评估某个表达式的环境。return action 应该具有相同的效果无论环境如何，所以我们的版本忽略其环境。

我们 `(>>=)` 的定义稍微复杂一些，但只是因为我们必须使环境——这里的变量 r——在当前计算和我们链接到的计算中都可用。

在这个 monad 中执行的代码如何找出其环境中的内容？它只需要问一下：

```haskell
-- file: ch15/SupplyInstance.hs
ask :: Reader e e
ask = R id
```

在给定的 action 链中，每次调用 ask 将返回相同的值，因为存储在环境中的值不会改变。我们的代码在 ghci 中很容易测试：

```ghci
ghci> runReader (ask >>= \x -> return (x * 3)) 2
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package random-1.0.0.0 ... linking ... done.
6
```

Reader monad 包含在标准 mtl 库中，通常与 GHC 捆绑在一起。你可以在 Control.Monad.Reader 模块中找到它。这个 monad 的动机最初可能看起来有点单薄，因为它在复杂代码中最有用。我们经常需要在程序深处访问一块配置信息；将其作为正常参数传入将需要对我们代码进行痛苦的重组。通过将此信息隐藏在 monad 的管道中，不关心配置信息的中间函数不需要看到它。

Reader monad 最清晰的动机将在第十八章中出现，那时我们讨论组合多个 monad 来构建一个新的 monad。在那里，我们将看到如何获得对状态的更细粒度控制，使我们的代码可以通过 State monad 修改某些值，而其他值保持不变，这要归功于 Reader monad。

### 回到自动派生

现在我们了解了 Reader monad，让我们使用它来创建 MonadSupply 类型类的实例。为了保持示例简单，我们将违反 MonadSupply 的精神：我们的 next action 将始终返回相同的值，而不是始终返回不同的值。

直接使 Reader 类型成为 MonadSupply 类的实例将是一个坏主意，因为那样任何 Reader 都可以充当 MonadSupply。这通常没有意义。

相反，我们创建一个基于 Reader 的 newtype。newtype 隐藏了我们内部使用 Reader 的事实。我们现在必须使我们的类型成为我们关心的两个类型类的实例。启用了 GeneralizedNewtypeDeriving 扩展后，GHC 将为我们完成大部分艰苦工作：

```haskell
-- file: ch15/SupplyInstance.hs
newtype MySupply e a = MySupply { runMySupply :: Reader e a }
    deriving (Monad)
instance MonadSupply e (MySupply e) where
    next = MySupply $ do
             v <- ask
             return (Just v)
    -- more concise:
    -- next = MySupply (Just `liftM` ask)
```

请注意，我们必须使我们的类型成为 MonadSupply e 的实例，而不是 MonadSupply。如果我们省略类型变量，编译器会抱怨。

为了尝试我们的 MySupply 类型，我们首先创建一个应该适用于任何 MonadSupply 实例的简单函数：

```haskell
-- file: ch15/SupplyInstance.hs
xy :: (Num s, MonadSupply s m) => m s
xy = do
  Just x <- next
  Just y <- next
  return (x * y)
```

如果我们将其与我们的 Supply monad 和 randomsIO 函数一起使用，我们每次都会得到不同的答案，正如我们所期望的：

```ghci
ghci> (fst . runSupply xy) `fmap` randomsIO
3155268008533561605104245047686121354
ghci> (fst . runSupply xy) `fmap` randomsIO
1764220767702892260034822063450517650
```

因为我们的 MySupply monad 有两层 newtype 包装，我们可以为其编写一个自定义执行函数以使其更易于使用：

```haskell
-- file: ch15/SupplyInstance.hs
runMS :: MySupply i a -> i -> a
runMS = runReader . runMySupply
```

当我们使用此执行函数应用我们的 xy action 时，我们每次都会得到相同的答案。我们的代码保持不变，但因为我们在 MonadSupply 的不同实现中执行它，其行为发生了变化：

```ghci
ghci> runMS xy 2
4
ghci> runMS xy 2
4
```

像我们的 MonadSupply 类型类和 Supply monad 一样，几乎所有常见的 Haskell monads 都是通过接口和实现的分离来构建的。例如，我们介绍为"属于"State monad 的 get 和 put 函数实际上是 MonadState 类型类的方法；State 类型是此类的实例。同样，标准 Reader monad 是 MonadReader 类型类的实例，它指定了 ask 方法。

虽然我们讨论的接口和实现的分离因其架构整洁而吸引人，但它具有重要的实际应用，这些将在后面变得更清楚。当我们开始在第十八章中组合 monads 时，我们将通过使用 GeneralizedNewtypeDeriving 和类型类节省大量工作。

## 隐藏 IO Monad

IO monad 的祝福和诅咒是它极其强大。如果我们相信仔细使用类型有助于我们避免编程错误，那么 IO monad 应该是一个很大的不安来源。因为 IO monad 对我们可以做什么没有施加任何限制，它使我们容易受到各种事故的影响。

我们如何驯服它的力量？假设我们希望保证一段代码可以读取和写入本地文件系统，但它不会访问网络。我们不能使用普通的 IO monad，因为它不会限制我们。

### 使用 newtype

让我们创建一个提供一组小功能用于读取和写入文件的模块：

```haskell
-- file: ch15/HandleIO.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
module HandleIO
    (
      HandleIO
    , Handle
    , IOMode(..)
    , runHandleIO
    , openFile
    , hClose
    , hPutStrLn
    ) where

import System.IO (Handle, IOMode(..))
import qualified System.IO
```

我们创建受限版本 IO 的第一种方法是使用 newtype 包装它：

```haskell
-- file: ch15/HandleIO.hs
newtype HandleIO a = HandleIO { runHandleIO :: IO a }
    deriving (Monad)
```

我们用现在熟悉的技巧从模块中导出类型构造函数和 runHandleIO 执行函数，但不导出数据构造函数。这将阻止在 HandleIO monad 中运行的代码获取它包装的 IO monad。

剩下的就是包装我们想要 monad 允许的每个 action。这很简单，只需用 HandleIO 数据构造函数包装每个 IO：

```haskell
-- file: ch15/HandleIO.hs
openFile :: FilePath -> IOMode -> HandleIO Handle
openFile path mode = HandleIO (System.IO.openFile path mode)
hClose :: Handle -> HandleIO ()
hClose = HandleIO . System.IO.hClose
hPutStrLn :: Handle -> String -> HandleIO ()
hPutStrLn h s = HandleIO (System.IO.hPutStrLn h s)
```

我们现在可以使用受限的 HandleIO monad 来执行 I/O：

```haskell
-- file: ch15/HandleIO.hs
safeHello :: FilePath -> HandleIO ()
safeHello path = do
  h <- openFile path WriteMode
  hPutStrLn h "hello world"
  hClose h
```

要运行此 action，我们使用 runHandleIO：

```ghci
ghci> :load HandleIO
[1 of 1] Compiling HandleIO         ( HandleIO.hs, interpreted )
Ok, modules loaded: HandleIO.
ghci> runHandleIO (safeHello "hello_world_101.txt")
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package filepath-1.1.0.0 ... linking ... done.
Loading package directory-1.0.0.1 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
ghci> :m +System.Directory
ghci> removeFile "hello_world_101.txt"
```

如果我们尝试将 HandleIO monad 中运行的 action 与不允许的 action 链接，类型系统将禁止它：

```ghci
ghci> runHandleIO (safeHello "goodbye" >> removeFile "goodbye")
<interactive>:1:36:
    Couldn't match expected type `HandleIO a'
           against inferred type `IO ()'
    In the second argument of `(>>)', namely `removeFile "goodbye"'
    In the first argument of `runHandleIO', namely
        `(safeHello "goodbye" >> removeFile "goodbye")'
    In the expression:
        runHandleIO (safeHello "goodbye" >> removeFile "goodbye")
```

### 为意外用途设计

我们的 HandleIO monad 有一个小但重要的问题：它没有考虑到我们偶尔可能需要逃生舱的可能性。如果我们定义一个这样的 monad，很可能我们会偶尔需要执行我们的 monad 设计不允许的 I/O action。

我们定义这样一个 monad 的目的是在常见情况下更容易编写可靠的代码，而不是使角落情况不可能。让我们给自己一条出路。

Control.Monad.Trans 模块定义了一个"标准逃生舱"，即 MonadIO 类型类。这定义了一个函数 liftIO，它允许我们在另一个 monad 中嵌入 IO action：

```ghci
ghci> :m +Control.Monad.Trans
ghci> :info MonadIO
class (Monad m) => MonadIO m where liftIO :: IO a -> m a
      -- Defined in Control.Monad.Trans
instance MonadIO IO -- Defined in Control.Monad.Trans
```

我们对此类型类的实现很简单；我们只是用我们的数据构造函数包装 IO：

```haskell
-- file: ch15/HandleIO.hs
import Control.Monad.Trans (MonadIO(..))
instance MonadIO HandleIO where
    liftIO = HandleIO
```

通过 judicious 使用 liftIO，我们可以逃离束缚并在必要时调用 IO actions：

```haskell
-- file: ch15/HandleIO.hs
tidyHello :: FilePath -> HandleIO ()
tidyHello path = do
  safeHello path
  liftIO (removeFile path)
```

### 自动派生和 MonadIO

我们可以通过将类型类添加到 HandleIO 的派生子句中，让编译器自动为我们派生 MonadIO 的实例。实际上，在生产代码中，这将是我们的常规策略。

我们在这里避免这样做只是为了将早期材料的呈现与 MonadIO 的呈现分开。

## 使用类型类

在另一个 monad 中隐藏 IO 的缺点是我们仍然绑定到具体实现。如果我们想将 HandleIO 换成其他 monad，我们必须更改使用 HandleIO 的每个 action 的类型。

作为替代方案，我们可以创建一个指定我们想要的来自操作文件的 monad 接口的类型类：

```haskell
-- file: ch15/MonadHandle.hs
{-# LANGUAGE FunctionalDependencies, MultiParamTypeClasses #-}
module MonadHandle (MonadHandle(..)) where
import System.IO (IOMode(..))
class Monad m => MonadHandle h m | m -> h where
    openFile :: FilePath -> IOMode -> m h
    hPutStr :: h -> String -> m ()
    hClose :: h -> m ()
    hGetContents :: h -> m String
    hPutStrLn :: h -> String -> m ()
    hPutStrLn h s = hPutStr h s >> hPutStr h "\n"
```

在这里，我们选择抽象掉 monad 的类型和文件句子的类型。为了满足类型检查器，我们添加了一个函数依赖：对于 MonadHandle 的任何实例，只有一种我们可以使用的句柄类型。当我们使 IO monad 成为此类的实例时，我们使用常规 Handle：

```haskell
-- file: ch15/MonadHandleIO.hs
{-# LANGUAGE FunctionalDependencies, MultiParamTypeClasses #-}
import MonadHandle
import qualified System.IO
import System.IO (IOMode(..))
import Control.Monad.Trans (MonadIO(..), MonadTrans(..))
import System.Directory (removeFile)
import SafeHello
instance MonadHandle System.IO.Handle IO where
    openFile = System.IO.openFile
    hPutStr = System.IO.hPutStr
    hClose = System.IO.hClose
    hGetContents = System.IO.hGetContents
    hPutStrLn = System.IO.hPutStrLn
```

因为任何 MonadHandle 也必须是 Monad，我们可以编写使用正常 do 记法操作文件的代码，而不关心它最终将在哪个 monad 中执行：

```haskell
-- file: ch15/SafeHello.hs
safeHello :: MonadHandle h m => FilePath -> m ()
safeHello path = do
  h <- openFile path WriteMode
  hPutStrLn h "hello world"
  hClose h
```

因为我们使 IO 成为此类型类的实例，我们可以从 ghci 执行此 action：

```ghci
ghci> safeHello "hello to my fans in domestic surveillance"
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package filepath-1.1.0.0 ... linking ... done.
Loading package directory-1.0.0.1 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
ghci> removeFile "hello to my fans in domestic surveillance"
```

类型类方法的美妙之处在于我们可以交换一个底层 monad 为另一个，而无需触及太多代码，因为大部分代码不知道或不在乎实现。例如，我们可以用在写入文件时压缩文件的 monad 替换 IO。

通过类型类定义 monad 的接口有另一个好处。它允许另一个用户在 newtype 包装器中隐藏我们的实现，并自动派生她只想公开的类型类的实例。

### 隔离和测试

事实上，因为我们的 safeHello 函数不使用 IO 类型，我们甚至可以使用一个不能执行 I/O 的 monad。这允许我们在完全纯的、受控的环境中测试通常有副作用的代码。

为此，我们将创建一个不执行 I/O 而是记录每个文件相关事件以供后续处理的 monad：

```haskell
-- file: ch15/WriterIO.hs
data Event = Open FilePath IOMode
           | Put String String
           | Close String
           | GetContents String
             deriving (Show)
```

虽然我们已经在"使用新的 Monad：展示你的工作！"一文中开发了一个 Logger 类型，但这里我们将使用标准的、更通用的 Writer monad。与其他 mtl monads 一样，Writer 提供的 API 在一个类型类中定义——在这种情况下是 MonadWriter。它最有用的方法是 tell，它记录一个值：

```ghci
ghci> :m +Control.Monad.Writer
ghci> :type tell
tell :: (MonadWriter w m) => w -> m ()
```

我们记录的值可以是任何 Monoid 类型。因为列表类型是一个 Monoid，我们将记录到 Event 列表。

我们可以使 Writer [Event] 成为 MonadHandle 的实例，但制作一个专用 monad 更便宜、更容易、更安全：

```haskell
-- file: ch15/WriterIO.hs
newtype WriterIO a = W { runW :: Writer [Event] a }
    deriving (Monad, MonadWriter [Event])
```

我们的执行函数只是删除我们添加的 newtype 包装器，然后调用正常 Writer monad 的执行函数：

```haskell
-- file: ch15/WriterIO.hs
runWriterIO :: WriterIO a -> (a, [Event])
runWriterIO = runWriter . runW
```

### Writer Monad 和列表

每次我们使用 tell 时，Writer monad 使用 Monoid 的 mappend 函数。因为列表的 mappend 是 (++)，列表不是与 Writer 一起使用的好选择：重复追加是昂贵的。我们之前纯粹为了简单起见使用列表。

在生产代码中，如果你想使用 Writer monad 并且需要类似列表的行为，请使用具有更好追加特性的类型。一种这样的类型是差异列表，我们在"利用函数作为数据"一文中介绍过。你不需要滚动自己的差异列表实现：一个调优良好的库可从 Hackage（Haskell 包数据库）下载。或者，你可以使用 Data.Sequence 模块中的 Seq 类型，我们在"通用序列"一文中介绍过。

### 重新访问任意 I/O

如果我们使用类型类方法限制 IO，我们可能仍希望保留执行任意 I/O actions 的能力。我们可以尝试将 MonadIO 作为约束添加到我们的类型类中：

```haskell
-- file: ch15/MonadHandleIO.hs
class (MonadHandle h m, MonadIO m) => MonadHandleIO h m | m -> h
instance MonadHandleIO System.IO.Handle IO
tidierHello :: (MonadHandleIO h m) => FilePath -> m ()
tidierHello path = do
  safeHello path
  liftIO (removeFile path)
```

然而，这种方法有问题：添加的 MonadIO 约束剥夺了我们了在纯环境中测试代码的能力，因为我们无法再判断测试是否可能有破坏性副作用。另一种方法是将此约束从类型类——它"感染"所有函数——移至只有那些真正需要执行 I/O 的函数：

```haskell
-- file: ch15/MonadHandleIO.hs
tidyHello :: (MonadIO m, MonadHandle h m) => FilePath -> m ()
tidyHello path = do
  safeHello path
  liftIO (removeFile path)
```

我们可以对缺少 MonadIO 约束的函数使用纯属性测试，对其余人使用传统单元测试。

不幸的是，我们用一个问题替换了另一个：我们无法从只有 MonadHandle 约束的代码调用同时具有 MonadIO 和 MonadHandle 约束的代码。如果我们发现在我们仅 MonadHandle 代码的深处我们确实需要 MonadIO 约束，我们必须将其添加到通向此点的所有代码路径中。

允许任意 I/O 是有风险的，它对我们开发和测试代码的方式有深远的影响。当我们必须在允许和更容易推理和测试之间做出选择时，我们通常选择后者。
