# 第09章-案例研究-文件系统搜索库

"我知道我有这个文件，但我不知道它在哪里"这个问题自计算机拥有层次文件系统以来就一直存在。Unix 第五版在 1974 年引入了 find 命令；它至今仍然是不可或缺的。最新技术已经走了很长的路：现代操作系统附带高级文档索引和搜索功能。

在程序员的工具箱中，find 类的功能仍然有其宝贵的地位。在本章中，我们将开发一个库，它给我们许多 find 的功能，而不离开 Haskell。我们将探索编写这个库的几种不同方法，每种都有不同的优势。

## find 命令

如果你不使用类 Unix 操作系统，或者你不是重度 shell 用户，你很可能没有听说过 find。给定一个目录列表，它递归搜索每个目录并打印匹配表达式的每个条目的名称。

单个表达式可以采取以下形式："名称匹配此 glob 模式"、"条目是普通文件"、"最后修改日期在此之前"，以及更多。它们可以使用"和"和"或"运算符拼接成更复杂的表达式。

## 从简单开始：递归列出目录

在我们深入设计库之前，让我们解决几个较小的问题。我们的第一个问题是递归列出目录及其子目录的内容：

```haskell
-- file: ch09/RecursiveContents.hs
module RecursiveContents (getRecursiveContents) where
import Control.Monad (forM)
import System.Directory (doesDirectoryExist, getDirectoryContents)
import System.FilePath ((</>))
getRecursiveContents :: FilePath -> IO [FilePath]
getRecursiveContents topdir = do
  names <- getDirectoryContents topdir
  let properNames = filter (`notElem` [".", ".."]) names
  paths <- forM properNames $ \name -> do
    let path = topdir </> name
    isDirectory <- doesDirectoryExist path
    if isDirectory
      then getRecursiveContents path
      else return [path]
  return (concat paths)
```

filter 表达式确保单个目录的列表不包含特殊目录名 . 或 ..，它们分别指当前目录和父目录。如果我们忘记过滤掉这些，我们会无限递归。

我们在前一章遇到了 forM；它是参数翻转的 mapM：

```ghci
> :m +Control.Monad
> :type mapM
mapM :: (Monad m) => (a -> m b) -> [a] -> m [b]
> :type forM
forM :: (Monad m) => [a] -> (a -> m b) -> m [b]
```

循环体检查当前条目是否是目录。如果是，它递归调用 getRecursiveContents 来列出该目录。否则，它返回一个单元素列表，即当前条目的名称。（不要忘记 return 函数在 Haskell 中有独特的含义：它用 monad 的类型构造函数包装一个值。）

另一点值得注意的是 isDirectory 变量的使用。在 Python 等命令式语言中，我们通常会写 if os.path.isdir(path)。但是，doesDirectoryExist 函数是一个动作；它的返回类型是 IO Bool，而不是 Bool。由于 if 表达式需要 Bool 类型的表达式，我们必须使用 <- 从其 IO 包装器中获取 Bool 结果，以便我们可以在 if 中使用普通的、未包装的 Bool。

循环体的每次迭代都会产生一个名称列表，所以这里 forM 的结果类型是 IO [[FilePath]]。我们使用 concat 将其展平为单个列表。

## 重新审视匿名和命名函数

在第 99 页的"匿名（lambda）函数"中，我们列出了一些不使用匿名函数的原因，然而我们在这里使用了一个作为循环体。这是 Haskell 中匿名函数最常见的用途之一。

我们已经从它们的类型看到 forM 和 mapM 接受函数作为参数。大多数循环体是在程序中只出现一次的代码块。既然我们很可能只在一个地方使用循环体，为什么要给它起名字呢？

当然，有时我们需要在几个不同的循环中部署完全相同的代码。与其复制粘贴相同的匿名函数，在这种情况下给现有的匿名函数起一个名字是有意义的。

## 为什么同时提供 mapM 和 forM？

看起来有点奇怪，两个函数相同但接受参数的顺序不同。然而，mapM 和 forM 在不同情况下都很方便。

考虑我们之前的例子，使用匿名函数作为循环体。如果我们用 mapM 代替 forM，我们必须把变量 properNames 放在函数的函数体之后。为了使代码能够正确解析，我们必须将整个匿名函数用括号括起来，或者用一个不必要的命名函数替换它。你可以自己尝试：复制刚才显示的代码，用 mapM 替换 forM，看看这对代码的可读性有什么影响。

相比之下，如果循环体已经是一个命名函数，而我们循环的列表是由一个复杂的表达式计算的，我们就有了使用 mapM 的好理由。

这里要遵循的风格经验法则是使用 mapM 或 forM 中能让你写出最整洁代码的那个。如果循环体和计算数据的表达式都很短，用哪个都无所谓。如果循环短但数据长，用 mapM。如果循环长但数据短，用 forM。如果两者都长，用 let 或 where 子句使其中一个变短。只要稍加练习，在每种情况下哪种方法最好就会变得很明显。

## 一个简单的查找函数

我们可以使用 getRecursiveContents 函数作为简单文件查找器的基础：

```haskell
-- file: ch09/SimpleFinder.hs
import RecursiveContents (getRecursiveContents)
simpleFind :: (FilePath -> Bool) -> FilePath -> IO [FilePath]
simpleFind p path = do
  names <- getRecursiveContents path
  return (filter p names)
```

这个函数接受一个谓词，我们用它来过滤 getRecursiveContents 返回的名称。传递给谓词的每个名称都是一个完整路径，所以我们如何执行"查找所有以 .c 扩展名结尾的文件"这样的常见操作？

System.FilePath 模块包含大量有用的函数来帮助我们操作文件名。在这种情况下，我们想要 takeExtension：

```ghci
> :m +System.FilePath
> :type takeExtension
takeExtension :: FilePath -> String
> takeExtension "foo/bar.c"
Loading package filepath-1.1.0.0 ... linking ... done.
".c"
> takeExtension "quux"
""
```

这给了我们一个简单的方法来编写一个获取路径、提取其扩展名并与 .c 进行比较的函数：

```ghci
> :load SimpleFinder
[1 of 2] Compiling RecursiveContents ( RecursiveContents.hs, interpreted )
[2 of 2] Compiling Main             ( SimpleFinder.hs, interpreted )
Ok, modules loaded: RecursiveContents, Main.
> :type simpleFind (\p -> takeExtension p == ".c")
simpleFind (\p -> takeExtension p == ".c") :: FilePath -> IO [FilePath]
```

虽然 simpleFind 可以工作，但它有几个明显的问题。第一个是谓词表达力不强。它只能查看目录条目的名称；例如，它无法找出它是文件还是目录。这意味着我们使用 simpleFind 的尝试会将 .c 结尾的目录以及具有相同扩展名的文件都列出来。

第二个问题是 simpleFind 对如何遍历文件系统没有控制。要理解这为什么重要，请考虑在由 Subversion 版本控制系统管理的树中搜索源文件的问题。Subversion 在它管理的每个目录中维护一个私有的 .svn 目录；每个目录包含许多对我们没有兴趣的子目录和文件。虽然我们可以轻松过滤掉任何包含 .svn 的路径，但首先避免遍历这些目录更高效。例如，我们其中一人有一个包含 45,000 个文件的 Subversion 源树，其中 30,000 个存储在 1,200 个不同的 .svn 目录中。避免遍历这 1,200 个目录比过滤掉它们包含的 30,000 个文件更便宜。

最后，simpleFind 是严格的，因为它由 IO monad 中执行的一系列动作组成。如果我们有一百万个文件要遍历，我们会遇到很长的延迟，然后收到一个包含一百万个名称的巨大结果。这对资源使用和响应性都不利。我们可能更喜欢惰性流结果，在它们到达时交付。

在接下来的章节中，我们将克服这些问题。

## 谓词：从贫乏到丰富，同时保持纯函数

我们的谓词只能查看文件名。这排除了一系列有趣的行为——例如，如果我们想列出大于给定大小的文件怎么办？

一个简单的反应是转向 IO：为什么不让我们的谓词从 FilePath -> Bool 改为 FilePath -> IO Bool？这样我们就可以在谓词中执行任意 I/O。虽然这看起来很有吸引力，但也可能是一个问题：这样的谓词可能有任意的副作用，因为具有 IO a 返回类型的函数可以有任何它想要的副作用。

让我们在编写更可预测、更少 bug 的代码的追求中借助类型系统的帮助；我们将通过避免"IO"的污染来保持谓词纯。这将确保它们不会有任何讨厌的副作用。我们也会向它们提供更多信息，这样它们可以获得我们想要的表达力，而不会变得潜在危险。

Haskell 的可移植 System.Directory 模块提供了一套有用但有限的ファイル元数据：

```ghci
> :m +System.Directory
```

我们可以使用 doesFileExist 和 doesDirectoryExist 来确定目录条目是文件还是目录。还没有可移植的方式来查询近年来已广泛使用的其他文件类型，如命名管道、硬链接和符号链接：

```ghci
> :type doesFileExist
doesFileExist :: FilePath -> IO Bool
> doesFileExist "."
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package directory-1.0.0.1 ... linking ... done.
False
> :type doesDirectoryExist
doesDirectoryExist :: FilePath -> IO Bool
> doesDirectoryExist "."
True
```

getPermissions 函数让我们找出是否允许对文件或目录执行某些操作：

```ghci
> :type getPermissions
getPermissions :: FilePath -> IO Permissions
> :info Permissions
data Permissions
  = Permissions {readable :: Bool,
                 writable :: Bool,
                 executable :: Bool,
                 searchable :: Bool}
      -- Defined in System.Directory
instance Eq Permissions -- Defined in System.Directory
instance Ord Permissions -- Defined in System.Directory
instance Read Permissions -- Defined in System.Directory
instance Show Permissions -- Defined in System.Directory
> getPermissions "."
Permissions {readable = True, writable = True, executable = False, searchable = True}
> :type searchable

True
```

最后，getModificationTime 告诉我们条目最后修改的时间：

```ghci
> :type getModificationTime
getModificationTime :: FilePath -> IO System.Time.ClockTime
> getModificationTime "."
Sat Aug 23 22:28:16 PDT 2008
```

如果我们坚持使用可移植的标准 Haskell 代码，这些函数是我们拥有的全部。（我们也可以用一个小技巧找到文件的大小；见下文。）它们也足以让我们说明我们感兴趣的原则，而不会让我们的例子过于扩展。如果你需要编写更多要求的代码，System.Posix 和 System.Win32 模块系列为两个主要现代计算平台提供了更详细的文件元数据。还有一个 Hackage 上的 unix-compat 包，在 Windows 上提供类 Unix API。

我们的新的、更丰富的谓词需要查看多少数据？由于我们可以通过查看其 Permissions 找出条目是文件还是目录，我们不需要传入 doesFileExist 或 doesDirectoryExist 的结果。因此，我们有四个数据片段是更丰富的谓词需要查看的：

```haskell
-- file: ch09/BetterPredicate.hs
import Control.Monad (filterM)
import System.Directory (Permissions(..), getModificationTime, getPermissions)
import System.Time (ClockTime(..))
import System.FilePath (takeExtension)
import Control.Exception (bracket, handle)
import System.IO (IOMode(..), hClose, hFileSize, openFile)
-- the function we wrote earlier
import RecursiveContents (getRecursiveContents)
type Predicate =  FilePath      -- path to directory entry
               -> Permissions   -- permissions
               -> Maybe Integer -- file size (Nothing if not file)
               -> ClockTime     -- last modified
               -> Bool
```

我们的 Predicate 类型只是一个四参数函数的同义词。它会为我们节省一些键盘工作和屏幕空间。

注意，这个谓词的返回值是 Bool，而不是 IO Bool：谓词是纯的，不能执行 I/O。有了这个类型，我们更具表达力的查找器函数仍然很简洁：

```haskell
-- file: ch09/BetterPredicate.hs
-- soon to be defined
getFileSize :: FilePath -> IO (Maybe Integer)
betterFind :: Predicate -> FilePath -> IO [FilePath]
betterFind p path = getRecursiveContents path >>= filterM check
    where check name = do
            perms <- getPermissions name
            size <- getFileSize name
            modified <- getModificationTime name
            return (p name perms size modified)
```

让我们一步一步走过代码。我们稍后会详细讨论 getFileSize，所以现在先跳过。

我们不能使用 filter 来调用我们的谓词 p，因为 p 的纯度意味着它不能执行收集其所需的元数据所需的 I/O。

这引导我们使用不熟悉的函数 filterM。它的行为类似于正常的 filter 函数，但在这个情况下它在 IO monad 中求值其谓词，允许谓词执行 I/O：

```ghci
> :m +Control.Monad
> :type filterM
filterM :: (Monad m) => (a -> m Bool) -> [a] -> m [a]
```

我们的 check 谓词是我们纯谓词 p 的一个支持 I/O 的包装器。它代表 p 做所有"肮脏"的 I/O 工作，这样我们就可以保持 p 没有不必要的副作用。收集元数据后，check 调用 p，然后用 return 将 p 的结果用 IO 包装。

## 安全地获取文件大小

虽然 System.Directory 不让我们找出文件有多大，但我们可以使用同样可移植的 System.IO 模块来做到这一点。它包含一个名为 hFileSize 的函数，返回打开文件的字节大小。这是一个包装它的简单函数：

```haskell
-- file: ch09/BetterPredicate.hs
simpleFileSize :: FilePath -> IO Integer
simpleFileSize path = do
  h <- openFile path ReadMode
  size <- hFileSize h
  hClose h
  return size
```

虽然这个函数可以工作，但它还不适合我们使用。在 betterFind 中，我们无条件地对任何目录条目调用 getFileSize；如果条目不是普通文件，它应该返回 Nothing，否则返回包装在 Just 中的大小。相反，这个函数如果条目不是普通文件或无法打开（可能由于权限不足），会抛出异常，并返回未包装的大小。

这里是这个函数的一个更安全的版本：

```haskell
-- file: ch09/BetterPredicate.hs
saferFileSize :: FilePath -> IO (Maybe Integer)
saferFileSize path = handle (\_ -> return Nothing) $ do
  h <- openFile path ReadMode
  size <- hFileSize h
  hClose h
  return (Just size)
```

函数体几乎相同，只是多了 handle 子句。

我们的异常处理器忽略传入的异常并返回 Nothing。后面函数体的唯一变化是它用 Just 包装了文件大小。

saferFileSize 函数现在有了正确的类型签名，并且不会抛出任何异常。但它仍然不是完全行为良好的。对于某些目录条目，openFile 会成功，但 hFileSize 会抛出异常。例如命名管道就会发生这种情况。这样的异常会被 handle 捕获，但我们对 hClose 的调用永远不会发生。

Haskell 实现会在注意到句柄不再被使用时自动关闭文件句柄。这要等到垃圾收集器运行才会发生，而到下一次垃圾收集通过的时间是不可预测的。

文件句柄是稀缺资源，由底层操作系统强制执行。例如，在 Linux 上，进程默认只允许同时打开 1,024 个文件。

不难想象一个场景，使用 saferFileSize 版本的 betterFind 的程序可能会崩溃，因为 betterFind 在足够多的垃圾文件句柄被关闭之前就耗尽了打开文件句柄的供应。

这是一种特别有害的 bug：它有几个方面结合在一起，使其难以置信地难以追踪。它只会在 betterFind 访问足够多的非文件达到进程的打开文件句柄限制时触发，然后返回到一个试图在累积的垃圾文件句柄关闭之前打开另一个文件的调用者。

更糟糕的是，任何后续错误都是由程序中不再可访问的数据引起的，而这些数据尚未被垃圾收集。因此，这样的 bug 取决于程序的结构、文件系统的内容，以及程序当前运行离触发垃圾收集有多近。

这类问题在开发过程中很容易被忽视，当它后来在现场发生时（正如这些尴尬的问题总是似乎会发生的），诊断起来会更加困难。

幸运的是，我们可以非常容易地避免这种错误，同时也使我们的函数更短。

## 获取-使用-释放循环

如果 openFile 成功，我们需要始终调用 hClose。Control.Exception 模块提供了 bracket 函数来 exactly 这个目的：

```ghci
> :type bracket
bracket :: IO a -> (a -> IO b) -> (a -> IO c) -> IO c
```

bracket 函数接受三个动作作为参数。第一个动作获取资源。第二个释放资源。第三个在资源被获取之间运行；我们称之为"使用"动作。如果"获取"动作成功，"释放"动作总是会被调用。这保证了资源总是会被释放。

"使用"和"释放"动作各自接收由"获取"动作获取的资源。

如果在"使用"动作执行期间发生异常，bracket 调用"释放"动作并重新抛出异常。如果"使用"动作成功，bracket 调用"释放"动作并返回"使用"动作返回的值。

我们现在可以写一个完全安全的函数——它不会抛出异常，也不会累积可能导致程序其他地方出现虚假失败的垃圾文件句柄：

```haskell
-- file: ch09/BetterPredicate.hs
getFileSize path = handle (\_ -> return Nothing) $
  bracket (openFile path ReadMode) hClose $ \h -> do
    size <- hFileSize h
    return (Just size)
```

再次仔细查看 bracket 的参数。第一个打开文件并返回打开的文件句柄。第二个关闭句柄。第三个简单地调用 hFileSize 并将结果包装在 Just 中。

我们需要同时使用 bracket 和 handle 才能使这个函数正确操作。前者确保我们不会累积垃圾文件句柄，而后者则处理异常。

## 谓词的领域特定语言

让我们尝试编写一个检查大于 128 KB 的 C++ 源文件的谓词：

```haskell
-- file: ch09/BetterPredicate.hs
myTest path _ (Just size) _ =
    takeExtension path == ".cpp" && size > 131072
myTest _ _ _ _ = False
```

这并不是特别令人满意。谓词接受四个参数，总是忽略其中两个，需要两个方程来定义。我们肯定可以做得更好。让我们创建一些代码来帮助我们编写更简洁的谓词。

有时候，这类库被称为嵌入式领域特定语言：我们使用编程语言的原生设施（因此是嵌入式的）来编写代码，让我们能够特别优雅地解决一些狭窄的问题（因此是领域特定的）。

我们的第一步是写一个返回其参数之一的函数。这个从传递给 Predicate 的参数中提取路径：

```haskell
-- file: ch09/BetterPredicate.hs
pathP path _ _ _ = path
```

如果我们不提供类型签名，Haskell 实现会为这个函数推断一个非常通用的类型。这以后可能导致难以解释的错误消息，所以让我们给 pathP 一个类型：

```haskell
-- file: ch09/BetterPredicate.hs
type InfoP a =  FilePath        -- path to directory entry
             -> Permissions     -- permissions
             -> Maybe Integer   -- file size (Nothing if not file)
             -> ClockTime       -- last modified
             -> a
pathP :: InfoP FilePath
```

我们创建了一个类型同义词，可以用作编写其他类似结构函数的简写。我们的类型同义词接受一个类型参数，这样我们就可以指定不同的返回类型：

```haskell
-- file: ch09/BetterPredicate.hs
sizeP :: InfoP Integer
sizeP _ _ (Just size) _ = size
sizeP _ _ Nothing     _ = -1
```

（我们在这里有点狡猾，对于不是文件或我们无法打开的条目返回 -1 大小。）

事实上，扫一眼就知道我们在本章开头附近定义的 Predicate 类型与 InfoP Bool 类型相同。（因此我们可以合法地去掉 Predicate 类型。）

pathP 和 sizeP 有什么用？只要再加一点粘合剂，我们就可以在谓词中使用它们（每个名字后面的 P 后缀旨在暗示"谓词"）。这就是事情开始变得有趣的地方：

```haskell
-- file: ch09/BetterPredicate.hs
equalP :: (Eq a) => InfoP a -> a -> InfoP Bool
equalP f k = \w x y z -> f w x y z == k
```

equalP 的类型签名值得关注。它接受一个 InfoP a 类型，它与 pathP 和 sizeP 都兼容。接下来它接受一个 a 并返回一个 InfoP Bool，我们已经观察到这是 Predicate 的同义词。换句话说，equalP 构建了一个谓词。

equalP 函数通过返回一个匿名函数来工作。那个函数接受谓词接受的参数，将它们传递给 f，并将结果与 k 比较。

equalP 的这个方程强调了我们认为它接受两个参数的事实。由于 Haskell 对所有函数进行柯里化，这样写 equalP 实际上不是必要的。我们可以省略匿名函数，依赖柯里化来为我们工作，让我们写一个行为完全相同的函数：

```haskell
-- file: ch09/BetterPredicate.hs
equalP' :: (Eq a) => InfoP a -> a -> InfoP Bool
equalP' f k w x y z = f w x y z == k
```

## 用提升避免样板代码

除了 equalP，我们希望能编写其他二元函数。我们宁愿不为每一个写完整定义，因为这似乎不必要地冗长。

为此，让我们利用 Haskell 的抽象能力。我们将采用 equalP 的定义，而不是直接调用 (==)，而是传入我们想要调用的二元函数作为另一个参数：

```haskell
-- file: ch09/BetterPredicate.hs
liftP :: (a -> b -> c) -> InfoP a -> b -> InfoP c
liftP q f k w x y z = f w x y z `q` k
greaterP, lesserP :: (Ord a) => InfoP a -> a -> InfoP Bool
greaterP = liftP (>)
lesserP = liftP (<)
```

这个行为称为提升（lifting）。取一个函数，例如 (>)，并将其转换为在不同上下文中操作的另一个函数（这里是 greaterP），称为将其提升（lift）到该上下文中。（这解释了函数名中 lift 的存在。）提升让我们重用代码并减少样板代码。我们将在本书的其余部分以不同的形式大量使用它。

当我们提升一个函数时，我们通常将其原始版本和新版本分别称为未提升（unlifted）和提升后（lifted）。

顺便说一句，我们将 q（要提升的函数）作为 liftP 的第一个参数是完全故意的。这使得我们能够写出 greaterP 和 lesserP 如此简洁的定义。偏应用使得在 Haskell 中寻找参数的"最佳"顺序成为 API 设计中比在其他语言中更重要的部分。在没有偏应用的语言中，参数顺序是品味和约定的问题。然而，如果把参数放在 Haskell 中错误的位置，我们会失去偏应用带来的简洁性。

我们可以通过组合器恢复一些简洁性。例如，forM 直到 2007 年才被添加到 Control.Monad 模块。在此之前，人们写 flip mapM 来代替：

```ghci
> :m +Control.Monad
> :t mapM
mapM :: (Monad m) => (a -> m b) -> [a] -> m [b]
> :t forM
forM :: (Monad m) => [a] -> (a -> m b) -> m [b]
> :t flip mapM
flip mapM :: (Monad m) => [a] -> (a -> m b) -> m [b]
```

## 将谓词粘合在一起

如果我们想组合谓词，当然我们可以遵循明显的路径手工来做：

```haskell
-- file: ch09/BetterPredicate.hs
simpleAndP :: InfoP Bool -> InfoP Bool -> InfoP Bool
simpleAndP f g w x y z = f w x y z && g w x y z
```

现在我们了解了提升，用提升我们现有的布尔运算符来减少我们必须编写的代码量变得更自然：

```haskell
-- file: ch09/BetterPredicate.hs
liftP2 :: (a -> b -> c) -> InfoP a -> InfoP b -> InfoP c
liftP2 q f g w x y z = f w x y z `q` g w x y z
andP = liftP2 (&&)
orP = liftP2 (||)
```

注意 liftP2 与我们之前的 liftP 非常相似。事实上，它更通用，因为我们可以用 liftP2 来写 liftP：

```haskell
-- file: ch09/BetterPredicate.hs
constP :: a -> InfoP a
constP k _ _ _ _ = k
liftP' q f k w x y z = f w x y z `q` constP k w x y z
```

### 组合器

在 Haskell 中，我们将接受其他函数作为参数并返回新函数的函数称为组合器。

现在我们有了一些辅助函数，我们可以回到前面定义的 myTest 函数：

```haskell
-- file: ch09/BetterPredicate.hs
myTest path _ (Just size) _ =
    takeExtension path == ".cpp" && size > 131072
myTest _ _ _ _ = False
```

如果我们用新的组合器来写，这个函数会是什么样子？

```haskell
-- file: ch09/BetterPredicate.hs
liftPath :: (FilePath -> a) -> InfoP a
liftPath f w _ _ _ = f w
myTest2 = (liftPath takeExtension `equalP` ".cpp") `andP`
          (sizeP `greaterP` 131072)
```

我们添加了最后一个组合器 liftPath，因为操作文件名是如此常见的活动。

## 定义和使用新的运算符

我们可以通过定义新的中缀运算符来进一步扩展我们的领域特定语言：

```haskell
-- file: ch09/BetterPredicate.hs
(==?) = equalP
(&&?) = andP
(>?) = greaterP
myTest3 = (liftPath takeExtension ==? ".cpp") &&? (sizeP >? 131072)
```

我们特意为提升函数选择如 (==?) 这样的名字，是因为它们与未提升的对应物有视觉上的相似性。

我们定义中的括号是必要的，因为我们还没有告诉 Haskell 关于新运算符的优先级或结合性。该语言指定没有 fixity 声明的运算符应该被视为 infixl 9，即从左到右以最高优先级求值。如果我们省略括号，表达式会被解析为 (((liftPath takeExtension) ==? ".cpp") &&? sizeP) >? 131072，这是非常错误的。

我们可以通过为新运算符写 fixity 声明来回应。我们的第一步是找出未提升运算符的 fixity，这样我们就可以模仿它们：

```ghci
> :info ==
class Eq a where
  (==) :: a -> a -> Bool
  ...
      -- Defined in GHC.Base
infix 4 ==
> :info &&
(&&) :: Bool -> Bool -> Bool     -- Defined in GHC.Base
infixr 3 &&
> :info >
class (Eq a) => Ord a where
  ...
  (>) :: a -> a -> Bool
  ...
      -- Defined in GHC.Base
infix 4 >
```

有了这些，我们现在可以写一个无括号的表达式，它的解析与 myTest3 完全相同：

```haskell
-- file: ch09/BetterPredicate.hs
infix 4 ==?
infixr 3 &&?
infix 4 >?
myTest4 = liftPath takeExtension ==? ".cpp" &&? sizeP >? 131072
```

## 控制遍历

在遍历文件系统时，我们希望对自己有更多控制，决定进入哪些目录以及何时进入。一个简单的方法是传入一个函数，它接受给定目录的子目录列表并返回另一个列表。这个列表可以删除元素，或者可以与原始列表顺序不同，或两者兼有。最简单的控制函数是 id，它会原样返回其输入列表。

为了多样化，我们将在这里改变我们表示方式的几个方面。不是复杂的函数类型 InfoP a，我们将使用一个普通的代数数据类型来表示大致相同的信息：

```haskell
-- file: ch09/ControlledVisit.hs
data Info = Info {
      infoPath :: FilePath
    , infoPerms :: Maybe Permissions
    , infoSize :: Maybe Integer
    , infoModTime :: Maybe ClockTime
    } deriving (Eq, Ord, Show)
getInfo :: FilePath -> IO Info
```

我们使用记录语法来给自己"免费"的访问函数，如 infoPath。我们 traverse 函数的类型很简单，正如我们刚才提议的。为了获取关于文件或目录的 Info，我们调用 getInfo 动作：

```haskell
-- file: ch09/ControlledVisit.hs
traverse :: ([Info] -> [Info]) -> FilePath -> IO [Info]
```

traverse 的定义很短，但很密集：

```haskell
-- file: ch09/ControlledVisit.hs
traverse order path = do
    names <- getUsefulContents path
    contents <- mapM getInfo (path : map (path </>) names)
    liftM concat $ forM (order contents) $ \info -> do
      if isDirectory info && infoPath info /= path
        then traverse order (infoPath info)
        else return [info]
getUsefulContents :: FilePath -> IO [String]
getUsefulContents path = do
    names <- getDirectoryContents path
    return (filter (`notElem` [".", ".."]) names)
isDirectory :: Info -> Bool
isDirectory = maybe False searchable . infoPerms
```

虽然我们这里没有引入任何新技术，但这是我们迄今为止遇到的最密集的函数定义之一。让我们几乎逐行走过它，解释发生了什么。

前几行没有神秘感，因为它们几乎是我们已经见过的代码的逐字副本。当我们将变量 contents 赋值时，事情开始变得有趣。让我们从右向左读。我们已经知道 names 是目录条目的列表。我们确保当前目录被前置到列表的每个元素并包含在列表本身中。我们使用 mapM 将 getInfo 应用于生成的路径。

接下来的那一行更加密集。同样从右向左阅读，我们看到行的最后一个元素开始定义一个匿名函数，一直持续到段落结尾。给定一个 Info 值，这个函数要么递归地访问一个目录（有一个额外的检查确保我们不会再次访问路径），要么将该值作为单元素列表返回（以匹配 traverse 的结果类型）。

我们使用 forM 将这个函数应用于 order 返回的每个 Info 值，order 是用户提供的遍历控制函数。

在行的开头，我们使用在 new 上下文中提升的技术。liftM 函数取一个常规函数 concat，并将其提升到 IO monad 中。换句话说，它取出 forM 的结果（类型为 IO [[Info]]），对其应用 concat（产生类型为 [Info] 的结果，这是我们需要的），然后将结果放回 IO monad。

最后，我们不要忘记定义我们的 getInfo 函数：

```haskell
-- file: ch09/ControlledVisit.hs
maybeIO :: IO a -> IO (Maybe a)
maybeIO act = handle (\_ -> return Nothing) (Just `liftM` act)
getInfo path = do
  perms <- maybeIO (getPermissions path)
  size <- maybeIO (bracket (openFile path ReadMode) hClose hFileSize)
  modified <- maybeIO (getModificationTime path)
  return (Info path perms size modified)
```

这里唯一值得注意的是一个有用的组合器 maybeIO，它将可能抛出异常的 IO 动作转换为用 Maybe 包装其结果的动作。

## 密度、可读性和学习过程

像 traverse 这样密集的代码在 Haskell 中并不罕见。表达力的提升是显著的，只需要相对较少的练习就能够流利地阅读和编写这种风格的代码。

作为比较，这里是相同代码的不太密集的呈现（这可能是不太有经验的 Haskell 程序员的典型）：

```haskell
-- file: ch09/ControlledVisit.hs
traverseVerbose order path = do
    names <- getDirectoryContents path
    let usefulNames = filter (`notElem` [".", ".."]) names
    contents <- mapM getEntryName ("" : usefulNames)
    recursiveContents <- mapM recurse (order contents)
    return (concat recursiveContents)
  where getEntryName name = getInfo (path </> name)
        isDirectory info = case infoPerms info of
                             Nothing -> False
                             Just perms -> searchable perms
        recurse info = do
            if isDirectory info && infoPath info /= path
                then traverseVerbose order (infoPath info)
                else return [info]
```

我们在这里所做的只是进行了一些替换。我们没有广泛使用偏应用和函数组合，而是在 where 块中定义了一些局部函数。代替 maybe 组合器，我们使用 case 表达式。而不是使用 liftM，我们手动提升 concat。

这并不是说密度是一个普遍好的属性。原始 traverse 函数的每一行都很短。我们专门引入了一个局部变量（usefulNames）和一个局部函数（isDirectory）来保持行短和代码清晰。我们的名字是描述性的。虽然我们使用函数组合和管道，但最长的管道只包含三个元素。

编写可维护的 Haskell 代码的关键是在密度和可读性之间找到平衡。你的代码在这个连续体上的位置可能受到你的经验水平的影响，详见下文：

- 作为 Haskell 编程的初学者，Andrew 不太熟悉标准库。因此，他无意中重复了很多现有代码。
- Zack 编程几个月了，已经掌握了使用 (.) 来组合长管道。每次他的程序的需求略有变化，他都必须从头开始构建新管道；他不能再理解现有管道了，而且它无论如何都太脆弱无法更改。
- Monica 编程已经有一段时间了。她对 Haskell 库和习惯用法足够熟悉，可以写出紧凑的代码，但她避免过度密集的风格。她的代码是可维护的，当面对不断变化的需求时，她发现重构很容易。

## 另一种遍历视角

虽然 traverse 函数比我们原来的 betterFind 函数给了我们更多控制，但它仍然有一个显著的缺陷：我们可以避免递归进入目录，但我们不能在其他名称被生成之前过滤它们，直到我们生成了树中整个名称列表。如果我们遍历一个包含 100,000 个文件的目录，而我们只关心其中的 3 个，我们将分配一个 100,000 个元素的列表，然后才能将其修剪为我们真正想要的 3 个。

一种方法是将 filter 函数作为新参数提供给 traverse，我们会在生成名称时将其应用到名称列表上。这将允许我们只分配我们需要的元素数量的列表。

然而，这种方法也有弱点。假设我们知道我们只需要从列表中最多 3 个条目，而这 3 个条目恰好是我们遍历的前 3 个。在这种情况下，我们会不必要地访问 99,997 个其他条目。这绝不是一个人为的例子：例如，Maildir 邮箱格式将文件夹的电子邮件消息存储为单个文件的目录。表示邮箱的单个目录包含数万文件是很常见的。

我们可以通过采取不同的视角来克服我们之前两个遍历函数的弱点：如果我们将文件系统遍历视为对目录层次结构的 fold 会怎样？

熟悉的 folds，foldr 和 foldl'，简洁地概括了遍历列表同时累积结果的思想。将折叠的思想从列表扩展到目录树并不困难，但我们想为我们的 fold 添加一个控制元素。我们将把这个控制表示为一个代数数据类型：

```haskell
-- file: ch09/FoldDir.hs
data Iterate seed = Done     { unwrap :: seed }
                  | Skip     { unwrap :: seed }
                  | Continue { unwrap :: seed }
                    deriving (Show)
type Iterator seed = seed -> Info -> Iterate seed
```

Iterator 类型给我们提供了一个方便的别名，用于我们折叠的函数。它接受一个 seed 和一个表示目录条目的 Info 值，并返回一个新的 seed 和我们 fold 函数的指令，其中指令表示为 Iterate 类型的构造函数：

- 如果指令是 Done，遍历应立即停止。Done 包装的值应作为结果返回。
- 如果指令是 Skip 且当前 Info 类型表示一个目录，遍历将不会递归进入该目录。
- 否则，遍历应继续，使用包装的值作为下一次 fold 调用的输入。

我们的 fold 在逻辑上是一种左 fold，因为我们从遇到的第一个条目开始折叠。每一步的 seed 是前一步的结果：

```haskell
-- file: ch09/FoldDir.hs
foldTree :: Iterator a -> a -> FilePath -> IO a
foldTree iter initSeed path = do
    endSeed <- fold initSeed path
    return (unwrap endSeed)
  where
    fold seed subpath = getUsefulContents subpath >>= walk seed
    walk seed (name:names) = do
      let path' = path </> name
      info <- getInfo path'
      case iter seed info of
        done@(Done _) -> return done
        Skip seed'    -> walk seed' names
        Continue seed'
          | isDirectory info -> do
              next <- fold seed' path'
              case next of
                done@(Done _) -> return done
                seed''        -> walk (unwrap seed'') names
          | otherwise -> walk seed' names
    walk seed _ = return (Continue seed)
```

这段代码的写法有几个有趣的地方。第一个是使用作用域来避免传递额外的参数。顶级 foldTree 函数只是 fold 的包装器，用于剥离 fold 最终结果的构造函数。因为 fold 是一个局部函数，我们不必将 foldTree 的 iter 变量传递给它；它已经可以在外部作用域中访问它了。同样，walk 可以在其外部作用域中看到 path。

另一点值得注意的是，walk 是一个尾递归循环，而不是像我们早期函数中那样由 forM 调用的匿名函数。通过自己掌控缰绳，我们可以根据需要提前停止，这让我们可以在迭代器返回 Done 时退出。

虽然 fold 调用 walk，但 walk 递归调用 fold 来遍历子目录。每个函数返回一个包装在 Iterate 中的 seed：当 fold 被 walk 调用并返回时，walk 检查其结果以确定是否应该继续，或者因为返回了 Done 而退出。这样，从调用者提供的迭代器返回 Done 会立即终止两个函数之间的所有相互递归调用。

迭代器在实践中是什么样的？这里有一个有点复杂的例子，它最多查找三个位图图像，并且不会递归进入 Subversion 元数据目录：

```haskell
-- file: ch09/FoldDir.hs
atMostThreePictures :: Iterator [FilePath]
atMostThreePictures paths info
    | length paths == 3
      = Done paths
    | isDirectory info && takeFileName path == ".svn"
      = Skip paths
    | extension `elem` [".jpg", ".png"]
      = Continue (path : paths)
    | otherwise
      = Continue paths
  where extension = map toLower (takeExtension path)
        path = infoPath info
```

要使用它，我们会调用 foldTree atMostThreePictures []，得到一个 IO [FilePath] 类型的返回值。

当然，迭代器不必这么复杂。这里有一个计算它遇到的目录数量的迭代器：

```haskell
-- file: ch09/FoldDir.hs
countDirectories count info =
    Continue (if isDirectory info
              then count + 1
              else count)
```

这里，我们传递给 foldTree 的初始 seed 应该是数字零。

## 有用的编码指南

虽然许多好的 Haskell 编程习惯来自经验，但我们有一些一般性指南要提供，这样你就可以更快地写出可读的代码。

正如我们在第 66 页的"关于制表符与空格的一个注记"中提到的，永远不要在 Haskell 源文件中使用制表字符。使用空格。

如果你发现自己得意地认为某段代码极其聪明，停下来考虑一下，一个月后你是否还能理解它。

命名类型和变量的复合名称的常规方式是使用驼峰命名法，即 myVariableName。这种风格在 Haskell 代码中几乎普遍存在。不管你对其他命名实践的看法如何，如果你遵循非标准约定，你的 Haskell 代码对其他读者来说会有些刺眼。

在你用 Haskell 工作了相当长的时间之前，在编写小函数之前花几分钟搜索库函数。这特别适用于列表、Maybe 和 Either 等无处不在的类型。如果标准库已经提供了你需要的确切内容，你也许能够组合几个函数来获得你想要的结果。

组合函数的长管道很难阅读，长意味着由三个或四个以上元素组成的系列。如果你有这样的管道，使用 let 或 where 块将其分解为更小的部分。给这些管道元素中的每一个一个有意义的名称，然后将它们粘合在一起。如果你不能为一个元素想一个有意义的名称，问问自己你是否甚至能描述它的作用。如果答案是"否"，简化你的代码。

尽管在文本编辑器窗口中轻松调整大小超过 80 列很容易，但这个宽度仍然非常普遍。更宽的行在 80 列文本编辑器窗口中被换行或截断，这严重损害可读性。把行视为不超过 80 个字符长限制了你可以塞进单行的代码量。这有助于保持单行不那么复杂，因此更容易理解。

## 常见的布局风格

只要你的代码遵循布局规则因此可以被无歧义地解析，Haskell 实现就不会对缩进大惊小怪。也就是说，一些布局模式被广泛使用。

in 关键字通常直接对齐在 let 关键字下方，表达式紧跟其后：

```haskell
-- file: ch09/Style.hs
tidyLet = let foo = undefined
              bar = foo * 2
          in undefined
```

虽然以不同方式缩进 in 是合法的，或者让它"悬"在一系列方程的末尾，但以下通常被认为是奇怪的：

```haskell
-- file: ch09/Style.hs
weirdLet = let foo = undefined
               bar = foo * 2
    in undefined
strangeLet = let foo = undefined
                 bar = foo * 2 in
    undefined
```

相比之下，让 do 悬在行尾而不是坐在一行的开头是常见的：

```haskell
-- file: ch09/Style.hs
commonDo = do
  something <- undefined
  return ()
-- not seen very often
rareDo =
  do something <- undefined
     return ()
```

花括号和分号虽然合法，但几乎从不使用。它们没有错；只是因为稀少让代码看起来奇怪。它们真的只是为了让程序生成 Haskell 代码而不必实现布局规则，并不是为人类使用设计的。

```haskell
-- file: ch09/Style.hs
unusualPunctuation =
    [ (x,y) | x <- [1..a], y <- [1..b] ] where {
                                           b = 7;
 a = 6 }
preferredLayout = [ (x,y) | x <- [1..a], y <- [1..b] ]
    where b = 7
          a = 6
```

如果方程的右侧在新行开始，它通常相对于它定义的变量或函数的名称缩进少量空格：

```haskell
-- file: ch09/Style.hs
normalIndent =
    undefined
strangeIndent =
                           undefined
```

实际使用的缩进空格数各不相同，有时在同一文件中也会变化。两、三和四个空格的缩进深度大约同样常见。单个空格是合法的但视觉上不太突出，所以很容易误读。

缩进 where 子句时，最好让它醒目：

```haskell
-- file: ch09/Style.hs
goodWhere = take 5 lambdas
    where lambdas = []
alsoGood =
    take 5 lambdas
  where
    lambdas = []
badWhere =           -- legal, but ugly and hard to read
    take 5 lambdas
    where
    lambdas = []
```
