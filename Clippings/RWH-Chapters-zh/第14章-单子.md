# 第14章-单子

在第七章中，我们讨论了 IO monad，但我们有意将讨论范围缩小到如何与外部世界通信。我们没有讨论什么是 monad。

我们已经在第七章中看到，IO monad 很容易使用。除了记号上的差异之外，在 IO monad 中编写代码与在任何其他命令式语言中编写代码并没有太大区别。

当我们在前面的章节中遇到需要解决的实际问题时，我们引入了一些结构，这些结构稍后会被发现实际上是 monad。我们的目标是向您展示 monad 通常是一种明显而有用的工具，可以帮助解决问题。在本章中，我们将定义一些 monad，以展示它有多么容易。

## 重新审视早期的代码示例

### Maybe 链

让我们再看一下在第十章中编写的 parseP5 函数：

```haskell
-- file: ch10/PNM.hs
matchHeader :: L.ByteString -> L.ByteString -> Maybe L.ByteString
-- "nat" 在这里代表 "natural number"（自然数）
getNat :: L.ByteString -> Maybe (Int, L.ByteString)
getBytes :: Int -> L.ByteString
         -> Maybe (L.ByteString, L.ByteString)
parseP5 s =
  case matchHeader (L8.pack "P5") s of
    Nothing -> Nothing
    Just s1 ->
      case getNat s1 of
        Nothing -> Nothing
        Just (width, s2) ->
          case getNat (L8.dropWhile isSpace s2) of
            Nothing -> Nothing
            Just (height, s3) ->
              case getNat (L8.dropWhile isSpace s3) of
                Nothing -> Nothing
                Just (maxGrey, s4)
                  | maxGrey > 255 -> Nothing
                  | otherwise ->
                      case getBytes 1 s4 of
                        Nothing -> Nothing
                        Just (_, s5) ->
                          case getBytes (width * height) s5 of
                            Nothing -> Nothing
                            Just (bitmap, s6) ->
                              Just (Greymap width height maxGrey bitmap, s6)
```

当我们介绍这个函数时，它面临着如果变得更复杂就会延伸到页面右边的风险。我们使用 `(>>?)` 函数来控制这种阶梯式的嵌套：

```haskell
-- file: ch10/PNM.hs
(>>?) :: Maybe a -> (a -> Maybe b) -> Maybe b
Nothing >>? _ = Nothing
Just v  >>? f = f v
```

我们仔细选择了 `(>>?)` 的类型，使其能够链接返回 Maybe 值的函数。只要一个函数的结果类型与下一个函数的参数类型匹配，我们就可以无限地将返回 Maybe 的函数链接在一起。`(>>?)` 的主体隐藏了关于我们构建的函数链是在某处短路（因为返回 Nothing）还是完全被求值的细节。

### 隐式状态

虽然 `(>>?)` 对于清理 parseP5 的结构很有用，但我们必须在解析它时逐步消费字符串的各个部分。这迫使我们将字符串的当前值作为元组的一部分向下传递到我们的 Maybe 链中。链中的每个函数将结果放入元组的一个元素中，将未消费 remainder 放入另一个元素中：

```haskell
-- file: ch10/PNM.hs
parseP5_take2 :: L.ByteString -> Maybe (Greymap, L.ByteString)
parseP5_take2 s =
    matchHeader (L8.pack "P5") s      >>?
    \s -> skipSpace ((), s)           >>?
    (getNat . snd)                    >>?
    skipSpace                         >>?
    \(width, s) ->   getNat s         >>?
    skipSpace                         >>?
    \(height, s) ->  getNat s        >>?
    \(maxGrey, s) -> getBytes 1 s    >>?
    (getBytes (width * height) . snd) >>?
    \(bitmap, s) -> Just (Greymap width height maxGrey bitmap, s)
```

```haskell
skipSpace :: (a, L.ByteString) -> Maybe (a, L.ByteString)
skipSpace (a, s) = Just (a, L8.dropWhile isSpace s)
```

又一次，我们面临着重复行为的模式：消费一些字符串，返回一个结果，并为下一个函数返回剩余的字符串。然而，这种模式更加隐蔽。如果我们想把另一条信息向下传递，我们就必须修改链中几乎每个元素，将每个二元组变成三元组！

我们通过将管理当前字符串片段的责任从链中的各个函数转移到我们用来链接它们的函数上来解决这个问题：

```haskell
-- file: ch10/Parse.hs
(==>) :: Parse a -> (a -> Parse b) -> Parse b
firstParser ==> secondParser  =  Parse chainedParser
  where chainedParser initState   =
          case runParse firstParser initState of
            Left errMessage ->
                Left errMessage
            Right (firstResult, newState) ->
                runParse (secondParser firstResult) newState
```

我们还将解析状态的细节隐藏在 ParseState 类型中。甚至 getState 和 putState 函数也不检查解析状态，因此对 ParseState 的任何修改都不会影响任何现有代码。

### 寻找共享模式

当我们详细查看前面的示例时，它们似乎没有太多共同点。显然，它们都关心将函数链接在一起以及隐藏细节以使我们编写更整洁的代码。然而，让我们退后一步，更少关注细节地考虑它们。

首先，让我们看看类型定义：

```haskell
-- file: ch14/Maybe.hs
data Maybe a = Nothing
             | Just a
-- file: ch10/Parse.hs
newtype Parse a = Parse {
      runParse :: ParseState -> Either String (a, ParseState)
    }
```

这两个类型的一个共同特征是，每个都在定义的左侧有一个单一的类型参数，该参数出现在右侧的某个位置。因此这些是泛型类型，它们对有效载荷一无所知。

接下来，我们将检查为这两种类型编写的链接函数：

```ghci
ghci> :type (>>?)
(>>?) :: Maybe a -> (a -> Maybe b) -> Maybe b
ghci> :type (==>)
(==>) :: Parse a -> (a -> Parse b) -> Parse b
```

这些函数的类型非常相似。如果我们将这些类型构造函数转换为类型变量，我们将得到一个更抽象的单一类型：

```haskell
-- file: ch14/Maybe.hs
chain :: m a -> (a -> m b) -> m b
```

最后，在每种情况下，我们都有一个接受"普通"值并将其"注入"到目标类型的函数。对于 Maybe，这个函数就是值构造函数 Just，但对于 Parse，注入器更为复杂：

```haskell
-- file: ch10/Parse.hs
identity :: a -> Parse a
identity a = Parse (\s -> Right (a, s))
```

同样，我们感兴趣的不是细节或复杂性，而是这些类型都有一个"注入器"函数这一事实，如下所示：

```haskell
-- file: ch14/Maybe.hs
inject :: a -> m a
```

正是这三个属性，以及一些关于如何将它们结合使用的规则，定义了 Haskell 中的 monad。让我们以前面的列表为蓝本，以精简的形式重新回顾一下：

- 一个类型构造函数 m。
- 一个类型为 `m a -> (a -> m b) -> m b` 的函数，用于将一个函数的输出链接到另一个函数的输入。
- 一个类型为 `a -> m a` 的函数，用于将普通值注入到链中，也就是说，它用一个类型构造函数 m 包装类型 a。

使 Maybe 类型成为 monad 的属性是它的类型构造函数 `Maybe a`、我们的链接函数 `(>>?)` 和注入器函数 `Just`。

对于 Parse，相应的属性是类型构造函数 `Parse a`、链接函数 `(==>)` 和注入器函数 `identity`。

我们有意没有讨论 monad 的链接和注入函数应该如何表现，因为这几乎不重要。事实上，monad 在 Haskell 代码中无处不在，正是因为它们非常简单。许多常见的编程模式都有 monadic 结构：传递隐式数据或在某个失败时短路一系列求值，仅举两个例子。

## Monad 类型类

我们可以在 Haskell 类型类中捕获链接和注入的概念，以及我们希望它们具有的类型。标准 Prelude 已经定义了这样一个类型类，名为 Monad：

```haskell
-- file: ch14/Maybe.hs
class Monad m where
    -- chain
    (>>=)  :: m a -> (a -> m b) -> m b
    -- inject
    return :: a -> m a
```

这里，`(>>=)` 是我们的链接函数。我们已经在"序列化和"一文中介绍过它。它通常被称为 bind，因为它将左侧计算的结果绑定到右侧的参数上。

我们的注入函数是 return。正如我们在"Return 的真实本质"一文中所指出的，选择 return 这个名称有点不幸。这个名称在命令式语言中被广泛使用，它有着相当好理解的含义。在 Haskell 中，它的行为受到的约束要少得多。特别是在函数链中间调用 return 不会导致链提前退出。将它的行为与其名称联系起来的一种有用方式是，它返回一个纯值（类型 a）到 monad（类型 m a）中。但实际上，"inject"会是一个更好的名称。

虽然 `(>>=)` 和 return 是 Monad 类型类的核心函数，但它还定义了两个其他函数。第一个是 `(>>)`。与 `(>>=)` 一样，它执行链接，但忽略左侧的值：

```haskell
-- file: ch14/Maybe.hs
    (>>) :: m a -> m b -> m b
    a >> f = a >>= \_ -> f
```

当我们要按特定顺序执行操作但不关心其中一个的结果时，我们使用这个函数。这看起来可能毫无意义：我们为什么不关心一个函数的返回值呢？不过，回想一下，我们之前定义了一个 `(==>&)` 组合器来表达这一点。或者，考虑一个像 print 这样的函数，它提供了一个我们不需要检查的占位符结果：

```ghci
ghci> :type print "foo"
print "foo" :: IO ()
```

如果我们使用普通的 `(>>=)`，我们必须提供作为其右侧的一个忽略其参数的函数：

```ghci
ghci> print "foo" >>= \_ -> print "bar"
"foo"
"bar"
```

但如果我们使用 `(>>)`，我们可以省略这个多余的函数：

```ghci
ghci> print "baz" >> print "quux"
"baz"
"quux"
```

正如我们刚才所示，`(>>)` 的默认实现是用 `(>>=)` 来定义的。

第二个非核心 Monad 函数是 fail，它接受一个错误消息并做一些使函数链失败的事情：

```haskell
-- file: ch14/Maybe.hs
    fail :: String -> m a
    fail = error
```

### 警惕 fail

许多 Monad 实例不会覆盖我们在这里展示的 fail 的默认实现，因此在那些 monad 中，fail 使用 error。调用 error 通常是非常不可取的，因为它抛出一个调用者要么无法捕获要么不会期望的异常。

即使你知道你当前正在执行的 monad 中 fail 会做更合理的事情，我们仍然建议避免使用它。当你在以后重构代码并忘记以前安全使用 fail 在新上下文中可能是危险的时，很容易给自己带来问题。

为了重新查看我们在第十章开发的解析器，下面是它的 Monad 实例：

```haskell
-- file: ch10/Parse.hs
instance Monad Parse where
    return = identity
    (>>=) = (==>)
    fail = bail
```

## 现在是术语时刻

关于 monad 有一些你可能不熟悉的术语。这些不是正式的，但它们是常用的，所以了解它们是有帮助的：

- Monadic 只是意味着"与 monad 相关的"。monadic 类型是 Monad 类型类的实例；monadic 值具有 monadic 类型。
- 当我们说一个类型"是一个 monad"时，这实际上是一种简写，表示它是 Monad 类型类的实例。作为 Monad 的实例给了我们必要的 monad 三元组：类型构造函数、注入函数和链接函数。
- 同样，提到"Foo monad"意味着我们正在谈论名为 Foo 的类型，它是 Monad 的实例。
- Action 是 monadic 值的另一个名称。这个词的使用可能源于为 I/O 引入 monad，在那里，像 `print "foo"` 这样的 monadic 值可以产生可观察的副作用。返回类型为 monadic 的函数也可能被称为 action，尽管这不太常见。

## 使用一个新的 Monad：展示你的工作！

在我们对 monad 的介绍中，我们展示了一些预先存在的代码已经是 monadic 形式。现在我们开始理解什么是 monad 并看到了 Monad 类型类，让我们以前瞻性的方式构建一个 monad。我们将从定义它的接口开始，然后我们将它投入使用。一旦我们把那些整理好，我们最终会构建它。

纯 Haskell 代码编写起来非常干净，但当然，它不能执行 I/O。有时候，我们想要记录我们做出的决定，而不是将日志信息写入文件。让我们开发一个小库来帮助实现这一点。

回想一下我们在"将 glob 模式翻译成正则表达式"一节中开发的 globToRegex 函数。我们将修改它，以便它记录它翻译的每个特殊模式序列。我们重访熟悉的领域是有原因的：它允许我们比较同一代码的非 monadic 和 monadic 版本。

首先，我们将用 Logger 类型构造函数包装我们的结果类型：

```haskell
-- file: ch14/Logger.hs
globToRegex :: String -> Logger String
```

### 信息隐藏

我们将故意保持 Logger 模块的内部细节抽象：

```haskell
-- file: ch14/Logger.hs
module Logger
    (
      Logger
    , Log
    , runLogger
    , record
    ) where
```

这样隐藏细节有两个好处：它为我们提供了在如何实现我们的 monad 方面相当大的灵活性，更重要的是，它为用户提供了一个简单的接口。

我们的 Logger 类型只是一个类型构造函数。我们不导出用户创建这种类型的值所需的值构造函数。用户只能用 Logger 来编写类型签名。

Log 类型只是字符串列表的同义词，以使一些签名更易读。我们使用字符串列表来保持实现简单：

```haskell
-- file: ch14/Logger.hs
type Log = [String]
```

我们不给用户值构造函数，而是提供给他们一个函数 runLogger 来求值一个已记录的动作。这返回动作的结果以及计算结果时记录的所有内容：

```haskell
-- file: ch14/Logger.hs
runLogger :: Logger a -> (a, Log)
```

### 受控逃逸

Monad 类型类不提供任何值逃离其 monadic 束缚的方法。我们可以使用 return 将值注入 monad。我们可以使用 `(>>=)` 从 monad 中提取值，但右侧的函数可以看到解包的值，必须将其自己的结果重新包装起来。

大多数 monad 都有一个或多个像我们的 runLogger 这样的执行函数。值得注意的是 IO 当然除外，我们通常通过退出程序来逃离它。

Monadic 执行函数在 monad 内部运行代码并解包其结果。此类函数通常是值从其 monadic 包装器中逃逸的唯一提供的手段。因此，monad 的作者可以完全控制 monad 内部发生的任何事情如何出来。

一些 monad 有几个执行函数。在我们的例子中，我们可以想象 runLogger 的几个替代方案：一个可能只返回日志消息，而另一个可能只返回结果并丢弃日志消息。

### 留下痕迹

当在 Logger action 内部执行时，用户代码调用 record 来记录一些内容：

```haskell
-- file: ch14/Logger.hs
record :: String -> Logger ()
```

由于记录发生在我们 monad 的管道中，我们 action 的结果不提供信息。

通常，一个 monad 会提供一个或多个像我们的 record 这样的辅助函数。这些是我们访问该 monad 特殊行为的手段。

我们的模块还为 Logger 类型定义了 Monad 实例。这些定义是客户端模块能够使用此 monad 所需的全部。

以下是 ghci 中我们 monad 行为的预览：

```ghci
ghci> let simple = return True :: Logger Bool
ghci> runLogger simple
(True,[])
```

当我们使用 runLogger 运行已记录的动作时，我们得到一个对。第一个元素是我们代码的结果；第二个是动作执行期间记录的条目列表。我们没有记录任何内容，所以列表是空的。让我们修复这个问题：

```ghci
ghci> runLogger (record "hi mom!" >> return 3.1337)
(3.1337,["hi mom!"])
```

### 使用 Logger Monad

以下是我们如何在 Logger monad 内部启动 glob-to-regexp 转换的方法：

```haskell
-- file: ch14/Logger.hs
globToRegex cs =
    globToRegex' cs >>= \ds ->
    return ('^':ds)
```

这里有一些值得提及的编码风格问题。函数体从其名称的下一行开始。这给了我们一些水平空白。我们还将匿名函数的参数"挂"在行尾。这在 monadic 代码中是常见的做法。

记住 `(>>=)` 的类型：它从其 Logger 包装器中提取左侧的值，并将解包的值传递给右侧的函数。右侧的函数必须依次将其结果用 Logger 包装器包装。这正是 return 所做的。它接受一个纯值，并用 monad 的类型构造函数包装它：

```ghci
ghci> :type (>>=)
(>>=) :: (Monad m) => m a -> (a -> m b) -> m b
ghci> :type (globToRegex "" >>=)
(globToRegex "" >>=) :: (String -> Logger b) -> Logger b
```

即使我们编写一个几乎什么都不做的函数，我们也必须调用 return 来用正确的类型包装结果：

```haskell
-- file: ch14/Logger.hs
globToRegex' :: String -> Logger String
globToRegex' "" = return "$"
```

当我们调用 record 来保存日志条目时，我们使用 `(>>)` 而不是 `(>>=)` 来将其与以下动作链接：

```haskell
-- file: ch14/Logger.hs
globToRegex' ('?':cs) =
    record "any" >>
    globToRegex' cs >>= \ds ->
    return ('.':ds)
```

回想一下，这是 `(>>=)` 的一个变体，它忽略左侧的结果。我们知道 record 的结果总是 ()，所以捕获它没有意义。

我们可以使用 do 记法（我们在"序列化"一文中首次遇到）来稍微整理我们的代码：

```haskell
-- file: ch14/Logger.hs
globToRegex' ('*':cs) = do
    record "kleene star"
    ds <- globToRegex' cs
    return (".*" ++ ds)
```

关于 do 记法与带有匿名函数的显式 `(>>=)` 之间的选择，这主要是品味问题，尽管对于任何超过两行的东西，几乎所有人的品味都是使用 do 记法。这两种风格之间有一个重要区别，我们将在"do 块的脱糖"中回来讨论。

解析字符类大多遵循我们已经看到的相同模式：

```haskell
-- file: ch14/Logger.hs
globToRegex' ('[':'!':c:cs) =
    record "character class, negative" >>
    charClass cs >>= \ds ->
    return ("[^" ++ c : ds)
globToRegex' ('[':c:cs) =
    record "character class" >>
    charClass cs >>= \ds ->
    return ("[" ++ c : ds)
globToRegex' ('[':_) =
    fail "unterminated character class"
```

### 混合纯代码和 Monadic 代码

基于我们目前看到的代码，monad 似乎有一个严重的缺点：包装 monadic 值的类型构造函数使得在 trapped 在 monadic 包装器内的值上使用普通纯函数变得棘手。这里有一个简单说明明显问题的方法。假设我们有一段在 Logger monad 中运行的代码并返回一个字符串：

```ghci
ghci> let m = return "foo" :: Logger String
```

如果我们想知道那个字符串的长度，我们不能简单地调用 length。字符串被包装了，所以类型不匹配：

```ghci
ghci> length m
<interactive>:1:7:
    Couldn't match expected type `[a]'
           against inferred type `Logger String'
    In the first argument of `length', namely `m`
    In the expression: length m
    In the definition of `it': it = length m
```

到目前为止，为了解决这个问题，我们做这样的事情：

```ghci
ghci> :type   m >>= \s -> return (length s)
m >>= \s -> return (length s) :: Logger Int
```

我们使用 `(>>=)` 来解包字符串，然后写一个小匿名函数调用 length 并使用 return 重新包装结果。

这种需要在 Haskell 代码中经常出现。你不会惊讶于已经存在一个简写：我们使用提升技术，我们最初在"引入 Functors"一文中介绍过。将纯函数提升到 functor 中通常涉及解包 functor 内的值，对其调用函数，然后用相同的构造函数重新包装结果。

我们对 monad 做完全相同的事情。因为 Monad 类型类已经提供了知道如何解包和包装值的 `(>>=)` 和 return 函数，liftM 函数不需要知道 monad 实现的任何细节：

```haskell
-- file: ch14/Logger.hs
liftM :: (Monad m) => (a -> b) -> m a -> m b
liftM f m = m >>= \i ->
            return (f i)
```

当我们声明一个类型为 Functor 类型类的实例时，我们必须为该类型写一个专门定制的 fmap 版本。相比之下，liftM 不需要知道 monad 的内部细节，因为它们被 `(>>=)` 和 return 抽象了。我们只需要写一次，带着适当的类型约束。

liftM 函数在标准 Control.Monad 模块中为我们预定义了。

为了了解 liftM 如何帮助提高可读性，我们将比较两个在其他方面相同的代码片段。首先，我们将看看不使用 liftM 的熟悉类型：

```haskell
-- file: ch14/Logger.hs
charClass_wordy (']':cs) =
    globToRegex' cs >>= \ds ->
    return (']':ds)
charClass_wordy (c:cs) =
    charClass_wordy cs >>= \ds ->
    return (c:ds)
```

现在我们可以使用 liftM 消除 `(>>=)` 和匿名函数的混乱：

```haskell
-- file: ch14/Logger.hs
charClass (']':cs) = (']':) `liftM` globToRegex' cs
charClass (c:cs) = (c:) `liftM` charClass cs
```

与 fmap 一样，我们经常以中缀形式使用 liftM。阅读这样的表达式的一个简单方法是"将左侧的纯函数应用于右侧 monadic action 的结果"。

liftM 函数非常有用，以至于 Control.Monad 定义了几个变体，它们组合更长的 action 链。我们可以在 globToRegex' 函数的最后一个子句中看到一个：

```haskell
-- file: ch14/Logger.hs
globToRegex' (c:cs) = liftM2 (++) (escape c) (globToRegex' cs)
escape :: Char -> Logger String
escape c
    | c `elem` regexChars = record "escape" >> return ['\\',c]
    | otherwise           = return [c]
  where regexChars = "\\+()^$.{}]|"
```

我们在这里使用的 liftM2 函数定义如下：

```haskell
-- file: ch14/Logger.hs
liftM2 :: (Monad m) => (a -> b -> c) -> m a -> m b -> m c
liftM2 f m1 m2 =
    m1 >>= \a ->
    m2 >>= \b ->
    return (f a b)
```

它执行第一个 action，然后是第二个，然后用纯函数 f 组合它们的结果，并将该结果包装起来。除了 liftM2，Control.Monad 中的变体还有 liftM5。

## 消除一些误解

我们现在已经在实践中看到了足够多的 monad 示例，对正在发生的事情有了一些感觉。在我们继续之前，有几个关于 monad 的常见误解我们要指出。你肯定会在"野外"遇到这些断言，所以你最好准备好一些好的反驳：

### Monads 可能很难理解

我们已经表明 monad 从几个问题中"自然涌现"。我们发现理解它们的最好关键是解释几个具体的例子，然后讨论它们的共同点。

### Monads 只对 I/O 和命令式编码有用

虽然我们在 Haskell 中使用 monad 进行 I/O，但它们对许多其他目的也很有价值。我们已经用它们来短路计算链、隐藏复杂状态和记录日志。即便如此，我们只是触及了表面。

### Monads 是 Haskell 独有的

Haskell 可能是使 monad 使用最明确的语言，但人们在其他语言中也编写它们，范围从 C++ 到 OCaml。由于 do 记法、类型系统的能力和推理以及语言语法，它们在 Haskell 中特别易于处理。

### Monads 用于控制求值的顺序

## 构建 Logger Monad

我们 Logger 类型的定义非常简单：

```haskell
-- file: ch14/Logger.hs
newtype Logger a = Logger { execLogger :: (a, Log) }
```

它是一对，其中第一个元素是动作的结果，第二个是运行该动作时记录的消息列表。

我们用 newtype 包装元组以使其成为一个独特的类型。runLogger 函数从其包装器中提取元组。我们导出执行已记录动作的函数 runLogger，只是 execLogger 的同义词：

```haskell
-- file: ch14/Logger.hs
runLogger = execLogger
```

我们的 record 辅助函数创建我们传递给它的消息的单例列表：

```haskell
-- file: ch14/Logger.hs
record s = Logger ((), [s])
```

这个动作的结果是 ()，所以那是我们在结果槽中放置的值。

让我们从 return 开始我们的 Monad 实例，这很简单。它不记录任何内容并将输入存储在结果槽中：

```haskell
-- file: ch14/Logger.hs
instance Monad Logger where
    return a = Logger (a, [])
```

稍微复杂一点的是 `(>>=)`，这是 monad 的核心。它组合一个动作和一个 monadic 函数以给出新结果和新日志：

```haskell
-- file: ch14/Logger.hs
    -- (>>=) :: Logger a -> (a -> Logger b) -> Logger b
    m >>= k = let (a, w) = execLogger m
                  n      = k a
                  (b, x) = execLogger n
              in Logger (b, w ++ x)
```

让我们明确地写出正在发生的事情。我们使用 runLogger 从动作 m 中提取结果 a，并将其传递给 monadic 函数 k。然后我们依次从中提取结果 b，并将其放入最终动作的结果槽中。我们连接日志 w 和 x 以给出新日志。

### 顺序日志，不是顺序求值

我们 `(>>=)` 的定义确保左侧记录的消息将出现在新日志中右侧之前。然而，它没有说明值 a 和 b 何时被求值：`(>>=)` 是惰性的。

像 monad 的大部分行为一样，严格性由其实现者控制。它不是所有 monad 共享的常量。事实上，一些 monad 有多种风格，每种都有不同的严格程度。

## The Writer Monad

我们的 Logger monad 是标准 Writer monad 的专用版本，可以在 mtl 包的 Control.Monad.Writer 模块中找到。我们将在"使用类型类"一节中展示一个 Writer 示例。

## The Maybe Monad

Maybe 类型几乎是最简单的 Monad 实例。它表示可能不产生结果的计算：

```haskell
-- file: ch14/Maybe.hs
instance Monad Maybe where
    Just x >>= k  =  k x
    Nothing >>= _ =  Nothing
    Just _ >> k   =  k
    Nothing >> _  =  Nothing
    return x      =  Just x
    fail _        =  Nothing
```

如果当我们使用 `(>>=)` 或 `(>>)` 将多个 Maybe 计算链接在一起时，其中任何一个返回 Nothing，我们就不会求值任何剩余的计算。

但请注意，链并没有完全短路。链中的每个 `(>>=)` 或 `(>>)` 将仍然匹配其左侧的 Nothing 并在其右侧产生 Nothing，一直延伸到结束。很容易忘记这一点：当链中的计算失败时，Nothing 值的后续产生、链接和消费在运行时很便宜，但它们不是免费的。

### 执行 Maybe Monad

适合执行 Maybe monad 的函数是 maybe。（请记住，"执行"一个 monad 涉及对它求值并返回已移除 monad 类型包装的结果。）

```haskell
-- file: ch14/Maybe.hs
maybe :: b -> (a -> b) -> Maybe a -> b
maybe n _ Nothing  = n
maybe _ f (Just x) = f x
```

它的第一个参数是如果结果是 Nothing 时返回的值。第二个是应用于包装在 Just 构造函数中的结果的函数；然后返回该应用的结果。

因为 Maybe 类型非常简单，所以通常更常见的是直接对 Maybe 值进行模式匹配，而不是调用 maybe。每个在不同情况下都更易读。

### Maybe 的应用，以及好的 API 设计

这里有一个 Maybe 作为 monad 使用的例子。给定一个客户的姓名，我们想找到她的手机运营商的账单地址：

```haskell
-- file: ch14/Carrier.hs
import qualified Data.Map as M
type PersonName = String
type PhoneNumber = String
type BillingAddress = String
data MobileCarrier = Honest_Bobs_Phone_Network
                   | Morrisas_Marvelous_Mobiles
                   | Petes_Plutocratic_Phones
                     deriving (Eq, Ord)
findCarrierBillingAddress :: PersonName
                          -> M.Map PersonName PhoneNumber
                          -> M.Map PhoneNumber MobileCarrier
                          -> M.Map MobileCarrier BillingAddress
                          -> Maybe BillingAddress
```

我们的第一个版本是可怕的阶梯式代码，延伸到屏幕右侧，有许多样板 case 表达式：

```haskell
-- file: ch14/Carrier.hs
variation1 person phoneMap carrierMap addressMap =
    case M.lookup person phoneMap of
      Nothing -> Nothing
      Just number ->
          case M.lookup number carrierMap of
            Nothing -> Nothing
            Just carrier -> M.lookup carrier addressMap
```

Data.Map 模块的 lookup 函数具有 monadic 返回类型：

```ghci
ghci> :module +Data.Map
ghci> :type Data.Map.lookup
Data.Map.lookup :: (Ord k, Monad m) => k -> Map k a -> m a
```

换句话说，如果给定的键存在于映射中，lookup 使用 return 将其注入 monad。否则，它调用 fail。这是一段有趣的 API 设计，尽管我们认为这是一个糟糕的选择：

- 从积极的一面看，成功和失败的行为会自动根据我们从其中调用 lookup 的 monad 来定制。更妙的是，lookup 本身不知道或不在乎这些行为是什么。
- 这里显示的 case 表达式类型检查，因为我们正在将 lookup 的结果与 Maybe 类型的值进行比较。
- 问题是，当然，在错误的 monad 中使用 fail 会抛出烦人的异常。我们已经警告过使用 fail，所以这里不再重复。

在实践中，每个人都使用 Maybe 作为 lookup 的结果类型。这种概念上简单函数的结果类型提供了不需要的泛化：lookup 应该返回 Maybe。

让我们搁置 API 问题，处理我们代码的丑陋。我们可以更合理地利用 Maybe 作为 monad 的身份：

```haskell
-- file: ch14/Carrier.hs
variation2 person phoneMap carrierMap addressMap = do
  number <- M.lookup person phoneMap
  carrier <- M.lookup number carrierMap
  address <- M.lookup carrier addressMap
  return address
```

如果这些查找中有任何一个失败，`(>>=)` 和 `(>>)` 的定义意味着整个函数的结果将是 Nothing，就像我们第一个使用显式 case 的尝试一样。

这个版本更整洁，但 return 不是必需的。在风格上，它使代码看起来更规律，也许对命令式程序员的眼睛更熟悉，但在行为上是多余的。这里有一个等效的代码片段：

```haskell
-- file: ch14/Carrier.hs
variation2a person phoneMap carrierMap addressMap = do
  number <- M.lookup person phoneMap
  carrier <- M.lookup number carrierMap
  M.lookup carrier addressMap
```

当我们介绍 maps 时，我们在"部分应用的尴尬"一节中提到，Data.Map 模块中函数的类型签名通常使它们难以部分应用。lookup 函数就是一个很好的例子。如果我们翻转它的参数，我们可以将函数体写成一行的形式：

```haskell
-- file: ch14/Carrier.hs
variation3 person phoneMap carrierMap addressMap =
    lookup phoneMap person >>= lookup carrierMap >>= lookup addressMap
  where lookup = flip M.lookup
```

## The List Monad

虽然 Maybe 类型可以表示无值或有一个值，但在许多情况下，我们可能希望返回一些我们事先不知道数量的结果。显然，列表非常适合这个目的。列表的类型表明我们可能能够将它用作 monad，因为它的类型构造函数有一个自由变量。确实，我们可以将列表用作 monad。

我们不是简单地展示 Prelude 的列表类型的 Monad 实例，而是尝试找出实例应该是什么样子。这很容易做到：我们将查看 `(>>=)` 和 return 的类型，执行一些替换，看看我们是否可以使用一些熟悉的列表函数。

两个函数中更明显的是 return。我们知道它接受一个类型 a，并用一个类型构造函数 m 包装它以给出类型 m a。我们也知道这里的类型构造函数是 `[]`。将这个类型构造函数代入类型变量 m 得到类型 `[] a`（是的，这真的是有效的表示法！），我们可以用更熟悉的形式重写为 `[a]`。

我们现在知道 return 对列表应该有类型 `a -> [a]`。对于这个函数的实现，只有少数几个合理的选择。它可能返回空列表、单一元素列表或无限列表。基于目前我们对 monad 的了解，最吸引人的行为是单一元素列表——它不会丢弃信息，也不会无限重复它：

```haskell
-- file: ch14/ListMonad.hs
returnSingleton :: a -> [a]
returnSingleton x = [x]
```

如果我们对 return 的类型做同样的替换技巧，我们发现它应该有类型 `[a] -> (a -> [b]) -> [b]`。这似乎接近 map 的类型：

```ghci
ghci> :type (>>=)
(>>=) :: (Monad m) => m a -> (a -> m b) -> m b
ghci> :type map
map :: (a -> b) -> [a] -> [b]
```

map 参数类型的顺序不匹配，但这很容易修复：

```ghci
ghci> :type (>>=)
(>>=) :: (Monad m) => m a -> (a -> m b) -> m b
ghci> :type flip map
flip map :: [a] -> (a -> b) -> [b]
```

我们仍然有问题：flip map 的第二个参数类型是 `a -> b`，而 `(>>=)` 对列表的第二个参数类型是 `a -> [b]`。我们该怎么做呢？

让我们做更多的替换，看看类型的会发生什么。函数 flip map 可以返回任何类型 b 作为其结果。如果我们在 flip map 类型签名中出现 b 的两个地方都代入 `[b]`，其类型签名读作 `a -> (a -> [b]) -> [[b]]`。换句话说，如果我们将返回列表的函数映射到列表上，我们会得到一个列表的列表回来：

```ghci
ghci> flip map [1,2,3] (\a -> [a,a+100])
[[1,101],[2,102],[3,103]]
```

有趣的是，我们并没有真正改变我们的类型签名匹配程度。`(>>=)` 的类型是 `[a] -> (a -> [b]) -> [b]`，而 flip map 在映射函数返回列表时的类型是 `[a] -> (a -> [b]) -> [[b]]`。在一个类型术语中仍然存在不匹配——我们只是将该术语从类型签名的中间移到了末尾。然而，我们的变换并不是徒劳的——我们现在需要一个取 `[[b]]` 并返回 `[b]` 的函数，而 concat 以一种现成的形式提供了这个函数：

```ghci
ghci> :type concat
concat :: [[a]] -> [a]
```

类型表明我们应该翻转 map 的参数，然后连接结果以给出单个列表：

```ghci
ghci> :type \xs f -> concat (map f xs)
\xs f -> concat (map f xs) :: [a] -> (a -> [a1]) -> [a1]
```

这正是列表的 `(>>=)` 的定义：

```haskell
-- file: ch14/ListMonad.hs
instance Monad [] where
    return x = [x]
    xs >>= f = concat (map f xs)
```

它将 f 应用于 xs 中的每个元素，并连接结果以返回单个列表。

有了我们的两个核心 Monad 定义，剩下的非核心定义 `(>>)` 和 fail 的实现应该是显而易见的：

```haskell
-- file: ch14/ListMonad.hs
    xs >> f = concat (map (\_ -> f) xs)
    fail _ = []
```

### 理解 List Monad

list monad 类似于一个熟悉的 Haskell 工具，即列表推导式。我们可以通过计算两个列表的笛卡尔积来说明这种相似性。首先，我们将写一个列表推导式：

```haskell
-- file: ch14/CartesianProduct.hs
comprehensive xs ys = [(x,y) | x <- xs, y <- ys]
```

这一次，我们将使用括号表示法而不是布局表示法来表示 monadic 代码。这将突出 monadic 代码与列表推导式的结构有多么相似：

```haskell
-- file: ch14/CartesianProduct.hs
monadic xs ys = do { x <- xs; y <- ys; return (x,y) }
```

唯一的真正区别是我们正在构建的值出现在表达式序列的末尾，而不是像列表推导式那样在开头。此外，两个函数的结果是相同的：

```ghci
ghci> comprehensive [1,2] "bar"
[(1,'b'),(1,'a'),(1,'r'),(2,'b'),(2,'a'),(2,'r')]
ghci> comprehensive [1,2] "bar" == monadic [1,2] "bar"
True
```

list monad 早期很容易让人困惑，所以让我们更详细地再次走过我们的 monadic 笛卡尔乘积代码。这一次，我们将重新排列函数以使用布局而不是括号：

```haskell
-- file: ch14/CartesianProduct.hs
blockyDo xs ys = do
    x <- xs
    y <- ys
    return (x, y)
```

对于 xs 列表中的每个元素，函数的其余部分被求值一次，x 每次绑定到列表中的不同值。然后对于 ys 列表中的每个元素，函数的剩余部分被求值一次，y 每次绑定到列表中的不同值。

我们这里真正拥有的是一个双嵌套循环！这突出了关于 monad 的一个重要事实：除非你知道一个 monadic 代码块将如何执行，否则你无法预测它的行为。

现在我们将更明确地逐步讲解代码，但首先让我们去掉 do 记法，使底层结构更清晰。我们稍微不寻常地缩进了代码，使循环嵌套更明显：

```haskell
-- file: ch14/CartesianProduct.hs
blockyPlain xs ys =
    xs >>=
    \x -> ys >>=
    \y -> return (x, y)

blockyPlain_reloaded xs ys =
    concat (map (\x ->
                 concat (map (\y ->
                              return (x, y))
                         ys))
            xs)
```

如果 xs 的值是 `[1,2,3]`，接下来的两行会求值，x 依次绑定到 1、2 和 3。如果 ys 的值是 `[True, False]`，最后一行会求值六次：一次是 x 为 1 且 y 为 True；再次是 x 为 1 且 y 为 False；依此类推。return 表达式将每个元组包装在单一元素列表中。

### 使用 List Monad

这里有一个简单的暴力约束求解器。给定一个整数，它找到所有相乘得到该值的正整数对（这就是正在解决的约束）：

```haskell
-- file: ch14/MultiplyTo.hs
guarded :: Bool -> [a] -> [a]
guarded True  xs = xs
guarded False _  = []

multiplyTo :: Int -> [(Int, Int)]
multiplyTo n = do
  x <- [1..n]
  y <- [x..n]
  guarded (x * y == n) $
    return (x, y)
```

让我们在 ghci 中试试这个：

```ghci
ghci> multiplyTo 8
[(1,8),(2,4)]
ghci> multiplyTo 100
[(1,100),(2,50),(4,25),(5,20),(10,10)]
ghci> multiplyTo 891
[(1,891),(3,297),(9,99),(11,81),(27,33)]
```

## do 块的脱糖

Haskell 的 do 语法是语法糖的一个例子：它提供了一种替代方式来编写 monadic 代码，而不使用 `(>>=)` 和匿名函数。脱糖是将语法糖翻译回核心语言。

脱糖 do 块的规则很容易遵循。我们可以认为编译器将这些规则机械地重复应用于 do 块，直到没有 do 关键字剩余。

后跟单个 action 的 do 关键字被翻译为该 action 本身：

```haskell
-- file: ch14/Do.hs            -- file: ch14/Do.hs
doNotation1 =                  translated1 =
    do act                         act
```

后跟多个 action 的 do 关键字被翻译为第一个 action，然后是 `(>>)`，然后是 do 关键字和剩余的 action。当我们重复应用此规则时，整个 do 块最终通过 `(>>)` 的应用链接在一起：

```haskell
-- file: ch14/Do.hs            -- file: ch14/Do.hs
doNotation2 =                  translated2 =
    do act1                        act1 >>
       act2                        do act2
       {- ... etc. -}                 {- ... etc. -}
       actN                           actN

                              finalTranslation2 =
                                  act1 >>
                                  act2 >>
                                  {- ... etc. -}
                                  actN
```

`<-` 表示法的翻译值得密切关注。在 `<-` 左侧的是一个正常的 Haskell 模式。这可以是一个变量或更复杂的东西，但不允许 guard 表达式：

```haskell
-- file: ch14/Do.hs            -- file: ch14/Do.hs
doNotation3 =                  translated3 =
    do pattern <- act1             let f pattern = do act2
       act2                                           let f pattern = do act2
       {- ... etc. -}                                 actN
       actN                            f _     = fail "..."
                                   in act1 >>= f
```

这个模式被翻译成一个 let 绑定，声明一个具有唯一名称的局部函数（我们只是用 f 作为示例）。`<-` 右侧的 action 然后使用 `(>>=)` 与此函数链接。

关于这个翻译值得注意的是，如果模式匹配失败，局部函数会调用 monad 的 fail 实现。这里有一个使用 Maybe monad 的例子：

```haskell
-- file: ch14/Do.hs
robust :: [a] -> Maybe a
robust xs = do (_:x:_) <- Just xs
               return x
```

Maybe monad 中的 fail 实现只是返回 Nothing。如果前面函数中的模式匹配失败，那么我们得到的将是 Nothing：

```ghci
ghci> robust [1,2,3]
Just 2
ghci> robust [1]
Nothing
```

最后，当我们在 do 块中写一个 let 表达式时，我们可以省略通常的 in 关键字。块中的后续 action 必须与 let 关键字对齐：

```haskell
-- file: ch14/Do.hs            -- file: ch14/Do.hs
doNotation4 =                  translated4 =
    do let val1 = expr1            let val1 = expr1
           val2 = expr2                val2 = expr2
           {- ... etc. -}              valN = exprN
           valN = exprN            in do act1
       act1                              act2
       act2                              {- ... etc. -}
       {- ... etc. -}                    actN
       actN
```

## Monads 作为一个可编程的分号

早些时候在"边距规则不是强制的"一文中，我们提到布局是 Haskell 中的常态，但不是必需的。我们可以使用显式结构而不是布局来编写 do 块：

```haskell
-- file: ch14/Do.hs            -- file: ch14/Do.hs
semicolon = do                 semicolonTranslated =
  {                                act1 >>
    act1;                          let f val1 = let val2 = expr1
    val1 <- act2;                               in actN
    let { val2 = expr1 };              f _ = fail "..."
    actN;                          in act2 >>= f
  }
```

尽管这种显式结构的使用很少见，但使用分号分隔表达式的事实产生了一个恰当的口号：monads 是一种"可编程的分号"，因为 `(>>)` 和 `(>>=)` 在每个 monad 中的行为是不同的。

## 为什么要无糖？

当我们明确地在代码中编写 `(>>=)` 时，它提醒我们，我们正在使用组合器将函数缝合在一起，而不仅仅是序列化 actions。

只要你觉得自己是 monads 的新手，我们认为你应该更喜欢显式地编写 `(>>=)` 而不是 do 语法的语法糖。反复强化真正发生的事情似乎对许多程序员有帮助保持清晰。（对于命令式程序员来说，过多接触 IO monad 然后假设 do 块仅仅意味着 actions 的简单序列，可能很容易放松警惕。）

一旦你对 monads 更加熟悉，你可以选择看起来更适合编写特定函数的风格。确实，当你看其他人的 monadic 代码时，你会看到混合 do 表示法和 `(>>=)` 在单个函数中并不罕见，但绝非不可能。

`(=<<)` 函数无论我们是否使用 do 记法都很常见。它是 `(>>=)` 的翻转版本：

```ghci
ghci> :type (>>=)
(>>=) :: (Monad m) => m a -> (a -> m b) -> m b
ghci> :type (=<<)
(=<<) :: (Monad m) => (a -> m b) -> m a -> m b
```

如果我们想以通常的 Haskell 从右到左风格组合 monadic 函数，它会派上用场：

```haskell
-- file: ch14/CartesianProduct.hs
wordCount = print . length . words =<< getContents
```

## The State Monad

我们早些时候在本章中发现，第十章的 Parse 是一个 monad。它有两个逻辑上不同的方面。一个是解析失败并提供详细信息的消息（我们使用 Either 类型表示）；另一个是携带一块隐式状态，在我们的情况下是部分消费的 ByteString。

这种读写状态的需要在 Haskell 程序中足够常见，以至于标准库提供了一个名为 State 的 monad，专门用于此目的。这个 monad 在 Control.Monad.State 模块中。

在我们的 Parse 类型中，作为其状态携带 ByteString，而 State monad 可以携带任何类型的状态。我们将状态的未知类型称为 s。

对于状态，我们能做的显而易见且通用的事情是什么？给定一个状态值，我们检查它，然后产生一个结果和一个新状态值。让我们说结果可以是任何类型 a。捕获这个想法的类型签名是 `s -> (a, s)`。取一个状态 s，对它做些什么，并返回一个结果 a 和可能一个新的状态 s。

### 几乎是一个 State Monad

让我们开发一些简单的代码，它几乎是 State monad，然后看看真正的东西。我们将从我们的类型定义开始，它具有我们刚刚描述的明显类型：

```haskell
-- file: ch14/SimpleState.hs
type SimpleState s a = s -> (a, s)
```

我们的 monad 是一个转换状态为另一个状态的函数，在这样做时产生一个结果。正因为如此，State monad 有时被称为状态转换器 monad。

是的，这是一个类型同义词，而不是一个新类型，所以我们作弊了一点。请暂时忍受我们；这简化了下面的描述。

早些时候在本章中，我们说一个 monad 有一个单一类型变量的类型构造函数，然而这里我们有一个带有两个参数的类型。关键是理解我们可以像部分应用普通函数一样部分应用一个类型。这用一个例子最容易理解：

```haskell
-- file: ch14/SimpleState.hs
type StringState a = SimpleState String a
```

这里，我们将类型变量 s 绑定到 String。类型 StringState 仍然有一个类型参数 a。尽管如此，我们现在更明显地拥有了一个适合 monad 的类型构造函数。换句话说，我们的 monad 的类型构造函数是 `SimpleState s`，而不是 `SimpleState` 本身。

我们需要做的下一个成分是 return 函数的定义：

```haskell
-- file: ch14/SimpleState.hs
returnSt :: a -> SimpleState s a
returnSt a = \s -> (a, s)
```

所有这些做的就是获取结果和当前状态并将它们"组合"在一起。你现在可能已经习惯于理解具有多个参数的 Haskell 函数只是单一参数函数的链，但以防万一你还不习惯，这里有一个更熟悉的方式编写 returnSt，使它更明显这个函数有多简单：

```haskell
-- file: ch14/SimpleState.hs
returnAlt :: a -> SimpleState s a
returnAlt a s = (a, s)
```

我们 monadic 谜题的最后一块是 `(>>=)` 的定义。这里是，使用标准库中 `(>>=)` 对 State 的定义的实际变量名：

```haskell
-- file: ch14/SimpleState.hs
bindSt :: (SimpleState s a) -> (a -> SimpleState s b) -> SimpleState s b
```

那些单字母变量名对可读性没有什么帮助，让我们看看是否能用一些更有意义的名称替换它们：

```haskell
-- file: ch14/SimpleState.hs
-- m == step
-- k == makeStep
-- s == oldState
bindAlt step makeStep oldState =
    let (result, newState) = step oldState
    in  (makeStep result) newState
```

要理解这个定义，请记住 step 是类型 `s -> (a, s)` 的函数。当我们求值这个时，我们得到一个元组，我们必须用它来返回类型 `s -> (a, s)` 的新函数。如果我们从 bindAlt 的类型签名中去除 SimpleState 类型同义词，然后检查其参数和结果的类型，可能更容易理解：

```haskell
-- file: ch14/SimpleState.hs
bindAlt :: (s -> (a, s))        -- step
        -> (a -> s -> (b, s))   -- makeStep
        -> (s -> (b, s))        -- (makeStep result) newState
```

### 读写状态

对于 State monad，`(>>=)` 和 return 的定义只是充当管道：它们移动一块状态，但以任何方式都不 touch 它。我们需要其他几个简单的函数来实际对状态做有用的工作：

```haskell
-- file: ch14/SimpleState.hs
getSt :: SimpleState s s
getSt = \s -> (s, s)

putSt :: s -> SimpleState s ()
putSt s = \_ -> ((), s)
```

getSt 函数简单地获取当前状态并将其作为结果返回，而 putSt 忽略当前状态并用一个新的替换它。

### 真正的 State Monad 请起立

我们在上一节中玩的唯一简化技巧是使用类型同义词而不是 SimpleState 的类型定义。如果当时我们引入一个 newtype 包装器，额外的包装和拆包装会使我们的代码更难理解。

为了定义 Monad 实例，我们必须提供一个适当的类型构造函数以及 `(>>=)` 和 return 的定义。这引导我们到 State 的真正定义：

```haskell
-- file: ch14/State.hs
newtype State s a = State {
      runState :: s -> (a, s)
    }
```

我们所做的只是用 State 构造函数包装我们的 `s -> (a, s)` 类型。当我们使用 Haskell 的记录语法定义类型时，我们自动获得一个 runState 函数，它会从其构造函数中拆开一个 State 值。runState 的类型是 `State s a -> s -> (a, s)`。

return 的定义与 SimpleState 的几乎相同，只是我们用 State 构造函数包装我们的函数：

```haskell
-- file: ch14/State.hs
returnState :: a -> State s a
returnState a = State $ \s -> (a, s)
```

`(>>=)` 的定义稍微复杂一些，因为它必须使用 runState 来移除 State 包装器：

```haskell
-- file: ch14/State.hs
bindState :: State s a -> (a -> State s b) -> State s b
bindState m k = State $ \s -> let (a, s') = runState m s
                              in runState (k a) s'
```

这个函数与我们早期的 bindSt 只是在添加一些值的包装和拆包装方面有所不同。通过将"真正的工作"与"簿记"分开，我们希望能更清楚地了解真正发生的事情。

我们通过添加一点包装，以相同的方式修改用于读取和修改状态的函数：

```haskell
-- file: ch14/State.hs
get :: State s s
get = State $ \s -> (s, s)

put :: s -> State s ()
put s = State $ \_ -> ((), s)
```

## 使用 State Monad：生成随机值

我们已经使用了解析二进制数据的 Parse（我们的 State monad 前身）。在那种情况下，我们将正在操作的状态类型直接绑定到 Parse 类型中。

相比之下，State monad 接受任何类型的状态作为参数。我们提供状态的类型，例如 `State ByteString`。

如果你有命令式语言的背景，State monad 可能比其他许多 monad 让你感觉更熟悉。毕竟，命令式语言都是关于携带一些隐式状态、读取某些部分以及通过赋值修改其他部分，这正是 State monad 的用途。

所以，我们不再不必要地为使用 State monad 的想法加油，而是从演示如何将它用于简单的事情开始：伪随机值生成。在命令式语言中，通常有一个容易获得的均匀分布伪随机数来源。例如，在 C 中，有一个标准 rand 函数，它使用一个全局状态生成伪随机数并更新它。

Haskell 的标准随机值生成模块名为 System.Random。它允许生成任何类型的随机值，而不仅仅是数字。该模块包含几个驻留在 IO monad 中的便捷函数。例如，C 的 rand 函数的一个粗略等效物如下：

```haskell
-- file: ch14/Random.hs
import System.Random
rand :: IO Int
rand = getStdRandom (randomR (0, maxBound))
```

（randomR 函数接受生成的随机值应位于的包含范围。）

System.Random 模块提供了一个类型类 RandomGen，它让我们定义新的随机 Int 值来源。类型 StdGen 是标准的 RandomGen 实例。它生成伪随机值。如果我们有一个真正的随机数据的外部来源，我们可以让它成为 RandomGen 的实例，并获得真正的随机值，而不是仅仅是伪随机的。

另一个类型类 Random 表示如何生成特定类型的随机值。该模块为所有通常的简单类型定义了 Random 实例。

顺便说一句，这里的 rand 定义读取并修改了一个驻留在 IO monad 中的内置全局随机生成器。

### 第一次纯度尝试

在我们迄今为止强调避免 IO monad 的地方，仅仅为了生成一些随机值就被拖回 IO monad，那将是一种耻辱。事实上，System.Random 包含纯随机数生成函数。

传统的缺点是，我们必须获取或创建一个随机数生成器，然后将它从创建的点运送到需要的地方。当我们最终调用它时，它返回一个新的随机数生成器——我们记得这是在纯代码中，所以我们不能修改现有生成器的状态。

如果我们忘记不可变性并在函数内重用同一个生成器，我们每次都会得到完全相同的"随机"数：

```haskell
-- file: ch14/Random.hs
twoBadRandoms :: RandomGen g => g -> (Int, Int)
twoBadRandoms gen = (fst $ random gen, fst $ random gen)
```

不用说，这有令人不愉快的后果：

```ghci
ghci> twoBadRandoms `fmap` getStdGen
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package random-1.0.0.0 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
(639600350314210417,639600350314210417)
```

random 函数使用隐式范围而不是 randomR 采用的用户提供的范围。getStdGen 函数从 IO monad 检索全局标准数字生成器的当前值。

不幸的是，正确地传递和使用生成器的连续版本并不能产生可读的代码。这里有一个简单的例子：

```haskell
-- file: ch14/Random.hs
twoGoodRandoms :: RandomGen g => g -> ((Int, Int), g)
twoGoodRandoms gen = let (a, gen') = random gen
                        (b, gen'') = random gen'
                    in ((a, b), gen'')
```

然而，既然我们现在了解了 State monad，它看起来是一个很好的候选者来隐藏生成器。State monad 让我们整洁地管理我们的可变状态，同时保证我们的代码将没有其他意想不到的副作用，如修改文件或建立网络连接。这使得对我们代码行为的推理更容易。

### 在 State Monad 中生成随机值

这里有一个将 StdGen 作为其状态一部分的 State monad：

```haskell
-- file: ch14/Random.hs
type RandomState a = State StdGen a
```

类型同义词当然不是必需的，但它很方便。它节省了一点键盘敲击，如果我们想用另一个随机生成器替换 StdGen，它会减少我们需要更改的类型签名的数量。

生成随机值现在只需要获取当前生成器，使用它，然后修改状态以用新生成器替换它：

```haskell
-- file: ch14/Random.hs
getRandom :: Random a => RandomState a
getRandom =
  get >>= \gen ->
  let (val, gen') = random gen in
  put gen' >>
  return val
```

现在我们可以使用我们之前看到的一些 monadic 工具来编写一个更简洁的函数来获取一对随机数：

```haskell
-- file: ch14/Random.hs
getTwoRandoms :: Random a => RandomState (a, a)
getTwoRandoms = liftM2 (,) getRandom getRandom
```

### 运行 State Monad

正如我们已经提到的，每个 monad 都有其自己的专门求值函数。就 State monad 而言，我们有多个选择：

runState
返回结果和最终状态
evalState
只返回结果，丢弃最终状态
execState
丢弃结果，只返回最终状态

evalState 和 execState 函数分别只是 fst 和 snd 与 runState 的组合。因此，三者中最值得记住的是 runState。

这里有一个完整的例子，演示如何实现我们的 getTwoRandoms 函数：

```haskell
-- file: ch14/Random.hs
runTwoRandoms :: IO (Int, Int)
runTwoRandoms = do
  oldState <- getStdGen
  let (result, newState) = runState getTwoRandoms oldState
  setStdGen newState
  return result
```

对 runState 的调用遵循标准模式：我们将 State monad 中的函数和初始状态传递给它。它返回函数的结果和最终状态。

围绕 runState 调用的代码只是获取当前全局 StdGen 值，然后在之后替换它，以便对 runTwoRandoms 或其他随机生成函数的后续调用将拾取更新后的状态。

### 多一点状态会怎样？

很难想象编写多少有趣的代码，其中只有一个状态值要传递。当我们想同时跟踪多个状态片段时，通常的技巧是在一个数据类型中维护它们。以下是跟踪我们分发了多少随机数的示例：

```haskell
-- file: ch14/Random.hs
data CountedRandom = CountedRandom {
      crGen :: StdGen
    , crCount :: Int
    }

type CRState = State CountedRandom

getCountedRandom :: Random a => CRState a
getCountedRandom = do
  st <- get
  let (val, gen') = random (crGen st)
  put CountedRandom { crGen = gen', crCount = crCount st + 1 }
  return val
```

这个例子恰好每次调用都消费状态的这两个元素，并且它构造一个全新的状态。更常见的是，我们可能只想读取或修改状态的一部分。这个函数获取迄今为止生成的随机值的数量：

```haskell
-- file: ch14/Random.hs
getCount :: CRState Int
getCount = crCount `liftM` get
```

这个例子说明了我们为什么使用记录语法定义我们的 CountedRandom 状态。它给了我们访问函数，我们可以将它们与 get 粘合在一起以读取状态的特定部分。

如果我们想部分更新状态，代码不会显得那么吸引人：

```haskell
-- file: ch14/Random.hs
putCount :: Int -> CRState ()
putCount a = do
  st <- get
  put st { crCount = a }
```

这里，我们使用记录更新语法而不是函数。表达式 `st { crCount = a }` 创建一个新值，它是 st 的完全相同的副本，只是其 crCount 字段被赋予了值 a。因为这是一个语法 hack，我们不能像使用函数那样获得相同的灵活性。记录语法可能没有展示 Haskell 通常的优雅，但至少它完成了工作。

有一个名为 modify 的函数，它结合了 get 和 put 步骤。它接受一个状态转换函数作为参数，但它仍然不太令人满意——我们仍然无法逃避记录更新语法的笨拙：

```haskell
-- file: ch14/Random.hs
putCountModify :: Int -> CRState ()
putCountModify a = modify $ \st -> st { crCount = a }
```

## Monads 和 Functors

Functors 和 monads 密切相关。这些术语来自称为范畴论的数学分支，但它们在转移到 Haskell 时并没有完全毫发无损。

在范畴论中，monad 是从 functor 构建的。你可能期望在 Haskell 中 Monad 类型类因此成为 Functor 的子类，但在标准 Prelude 中它没有这样定义——这是一个不幸的疏忽。

然而，Haskell 库作者使用了一个变通方法：当程序员为类型定义 Monad 实例时，他们几乎总是也为它写一个 Functor 实例。你可以期望能够将 Functor 类型类的 fmap 函数与任何 monad 一起使用。

如果我们将 fmap 的类型签名与我们已经看到的一些标准 monad 函数的类型签名进行比较，我们会得到关于 fmap 在 monad 上做什么的提示：

```ghci
ghci> :type fmap
fmap :: (Functor f) => (a -> b) -> f a -> f b
ghci> :module +Control.Monad
ghci> :type liftM
liftM :: (Monad m) => (a1 -> r) -> m a1 -> m r
```

确实，fmap 就像 liftM 一样将纯函数提升到 monad 中。

## 另一种看待 Monads 的方式

既然我们知道了 functors 和 monads 之间的关系，如果我们回头看 list monad，我们可以看到一些有趣的东西。具体来说，看看 `(>>=)` 对列表的定义：

```haskell
-- file: ch14/ListMonad.hs
instance Monad [] where
    return x = [x]
    xs >>= f = concat (map f xs)
```

回想一下 f 的类型是 `a -> [a]`。当我们调用 `map f xs` 时，我们得到类型 `[[a]]` 的值，我们必须使用 concat 将其"展平"。

考虑一下如果 Monad 是 Functor 的子类我们能做什么。由于列表的 fmap 定义为 map，我们可以用 fmap 替换 `(>>=)` 定义中的 map。这本身并没有什么意思，但假设我们走得更远。

concat 函数的类型是 `[[a]] -> [a]`。正如我们提到的，我们可以将此类型签名从列表推广到 monads，给我们"移除一层嵌套"的类型 `m (m a) -> m a`。具有此类型的函数通常命名为 join。

如果我们有了 join 和 fmap 的定义，我们就不需要为每个 monad 写 `(>>=)` 的定义，因为它将是完全通用的。这里是 Monad 类型类的另一种定义可能是什么样子的，以及 `(>>=)` 的定义：

```haskell
-- file: ch14/AltMonad.hs
import Prelude hiding ((>>=), return)

class Functor m => AltMonad m where
    join :: m (m a) -> m a
    return :: a -> m a

(>>=) :: AltMonad m => m a -> (a -> m b) -> m b
xs >>= f = join (fmap f xs)
```

两种 monad 定义都没有"更好"，因为如果我们有 join 我们可以写 `(>>=)`，反之亦然，但不同的视角可以令人耳目一新。

事实上，在实际情况下，移除一层 monadic 包装可能真的有用。我们可以在 Control.Monad 模块中找到 join 的通用定义：

```haskell
-- file: ch14/MonadJoin.hs
join :: Monad m => m (m a) -> m a
join x = x >>= id
```

这里有一些它做什么的例子：

```ghci
ghci> join (Just (Just 1))
Just 1
ghci> join Nothing
Nothing
ghci> join [[1],[2,3]]
[1,2,3]
```

## Monad 定律和良好的编码风格

在"更多关于 Functors 的思考"一文中，我们介绍了 functor 应该如何始终行为的两个规则：

```haskell
-- file: ch14/MonadLaws.hs
fmap id        ==   id
fmap (f . g)   ==   fmap f . fmap g
```

也有 monad 应该如何行为的规则。以下段落中描述的三个规则被称为 monad 定律。Haskell 实现不强制这些定律——遵循它们是 Monad 实例作者的责任。

Monad 定律只是"monad 不应该让我惊讶"的正式表达方式。原则上，我们可能完全可以跳过它们。但如果我们这样做了，那就太可惜了，因为这些定律包含了我们可能会忽视的智慧宝石。

### 阅读定律

你可以将以下每个定律读为"== 左边的表达式等于右边的表达式"。

第一定律表明 return 是 `(>>=)` 的左 identity：

```haskell
-- file: ch14/MonadLaws.hs
return x >>= f            ===   f x
```

另一种表述方式是，如果你要做的一切就是用 `(>>=)` 解开它，就没有理由用 return 包装一个纯值。这实际上是新手程序员在 monads 中常犯的一种风格错误：用 return 包装一个值，然后在同一函数中几行之后用 `(>>=)` 解开它。以下是用 do 记法写的相同定律：

```haskell
-- file: ch14/MonadLaws.hs
do y <- return x
   f y                    ===   f x
```

这个定律对我们的编码风格有实际影响：我们不想编写不必要的代码，而该定律让我们相信简洁的代码将与其更冗长的版本具有相同的效果。

第二个 monad 定律表明 return 是 `(>>=)` 的右 identity：

```haskell
-- file: ch14/MonadLaws.hs
m >>= return              ===   m
```

这个定律在真实程序中也有风格影响，特别是如果你来自命令式语言：如果块中的最后一个 action 本来就会返回正确的结果，就不需要使用 return。让我们用 do 记法看看这个定律：

```haskell
-- file: ch14/MonadLaws.hs
do y <- m
   return y               ===   m
```

再一次，假设一个 monad 遵守这个定律，我们可以编写更短的代码，并确信它将与更长的代码具有相同的效果。

最后一个定律与结合性有关：

```haskell
-- file: ch14/MonadLaws.hs
m >>= (\x -> f x >>= g)   ===   (m >>= f) >>= g
```

这个定律可能有点难理解，所以让我们看看方程两边括号的内容。我们可以像这样重写左边的表达式：

```haskell
-- file: ch14/MonadLaws.hs
m >>= s
  where s x = f x >>= g
```

在右边，我们也可以重新安排事情：

```haskell
-- file: ch14/MonadLaws.hs
t >>= g
  where t = m >>= f
```

我们现在声称以下两个表达式是等价的：

```haskell
-- file: ch14/MonadLaws.hs
m >>= s                   ===   t >>= g
```

这意味着，如果我们想将一个 action 分解成更小的片段，我们提取哪些 subactions 来制作新的 actions 并不重要，只要我们保持它们的顺序。如果我们将三个 actions 链接在一起，我们可以替换前两个并保留第三个，或者我们可以替换后两个并保留第一个。

即使这个更复杂的定律也有实际后果。在软件重构的术语中，提取方法技术是一种花哨的说法，即剪出一段内联代码，将其转换为一个函数，并从剪出代码的站点调用该函数。这个定律本质上表明，这种技术可以应用于 monadic Haskell 代码。

我们现在看到了每个 monad 定律如何为我们编写更好的 monadic 代码提供洞察。前两个定律表明如何避免任何不必要地使用 return。第三个表明我们可以安全地将一个复杂的 action 重构为几个更简单的。那些"做我想做的"的直觉在我们使用正确编写的 monads 时不会被违反，现在我们可以放心地让细节淡去了。

顺便说一句，Haskell 编译器不能保证一个 monad 实际上遵循 monad 定律。满足——或者最好证明——他的代码遵循定律是 monad 作者的责任。
