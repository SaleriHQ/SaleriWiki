## 第二十四章：并发与多核编程

在我们撰写本书时，CPU 架构的变化比过去几十年更加迅速。

## 定义并发与并行

并发程序需要同时执行几个可能不相关的任务。考虑游戏服务器的示例：它通常由数十个组件组成，每个组件都与外部世界有复杂的交互。一个组件可能处理多用户聊天；更多组件处理玩家输入并向它们反馈状态更新；而另一个则执行物理计算。

并发程序的正确操作不需要多个核心，尽管它们可以提高性能和响应能力。

相比之下，并行程序解决单一问题。考虑一个试图预测单一股票价格下一分钟波动的金融模型。如果我们想将其应用于交易所上市的每只股票——例如，估计我们应该买卖哪些——我们希望如果在 500 个核心上运行模型比仅使用 1 个能更快得到答案。正如所暗示的，并行程序通常不依赖于多个核心的存在来正确工作。

并发程序和并行程序之间的另一个有用区别在于它们与外部世界的交互。根据定义，并发程序持续处理网络协议、数据库等。典型的并行程序可能更专注于：它流入数据，短暂处理（几乎没有进一步的 I/O），然后流回数据。

许多传统语言进一步模糊了本已模糊的并发和并行编程之间的界限，因为它们强迫程序员使用相同的原语来构建两种程序。

在本章中，我们将关注在单个操作系统进程边界内运行的并发和并行程序。

## 使用线程进行并发编程

作为并发程序的构建块，大多数编程语言提供了一种创建多个独立控制线程的方法。Haskell 也不例外，尽管在 Haskell 中使用线程编程看起来与其他语言有些不同。

在 Haskell 中，线程是一个 IO 操作，独立于其他线程执行。要创建线程，我们导入 Control.Concurrent 模块并使用 forkIO 函数：

```ghci
ghci> :m +Control.Concurrent
ghci> :t forkIO
forkIO :: IO () -> IO ThreadId
ghci> :m +System.Directory
ghci> forkIO (writeFile "xyzzy" "seo craic nua!") >> doesFileExist "xyzzy"
False
```

新线程几乎立即开始执行，创建它的线程继续并发执行。线程将在其 IO 操作结束处停止执行。

## 线程是不确定的

GHC 的运行时组件没有指定执行线程的顺序。因此，在前面的示例中，新线程创建的 xyzzy 文件在原始线程检查其是否存在时可能已被创建，也可能尚未创建。如果我们尝试一次这个示例，然后删除 xyzzy 再试一次，第二次可能会得到不同的结果。

## 隐藏延迟

假设我们有一个要压缩并写入磁盘的大文件，但我们希望足够快地处理用户输入，以便她将我们的程序视为立即响应。如果我们使用 forkIO 在单独线程中写入文件，我们可以同时做这两件事：

```haskell
-- file: ch24/Compressor.hs
import Control.Concurrent (forkIO)
import Control.Exception (handle)
import Control.Monad (forever)
import qualified Data.ByteString.Lazy as L
import System.Console.Readline (readline)
-- Provided by the 'zlib' package on http://hackage.haskell.org/
import Codec.Compression.GZip (compress)
main = do
    maybeLine <- readline "Enter a file to compress> "
    case maybeLine of
      Nothing -> return ()      -- user entered EOF
      Just "" -> return ()      -- treat no name as "want to quit"
      Just name -> do
           handle print $ do
             content <- L.readFile name
             forkIO (compressFile name content)
             return ()
           main
  where compressFile path = L.writeFile (path ++ ".gz") . compress
```

因为我们在这里使用惰性 ByteString I/O，我们在主线程中真正做的只是打开文件。实际读取在另一个线程中按需发生。

使用 handle print 可以在我们输入不存在的文件名时提供一种打印错误消息的廉价方式。

## 线程间的简单通信

在两个线程之间共享信息的最简单方法是让它们都使用一个变量。在我们的文件压缩示例中，主线程与另一个线程共享文件名及其内容。因为 Haskell 数据默认是不可变的，这不会带来风险：两个线程都无法修改另一方对文件名或内容的视图。

我们经常需要让线程主动相互通信。例如，GHC 没有提供让一个线程找出另一个线程是否仍在执行、已完u200b成或已崩溃的方法。* 然而，它提供了一种同步变量类型 MVar，我们可以使用它来为自己创建此功能。

MVar 就像一个单元素盒子：它可以是满的或空的。我们可以将一些东西放入盒子（使其变满），或者取出一些东西（使其变空）：

```ghci
ghci> :t putMVar
putMVar :: MVar a -> a -> IO ()
ghci> :t takeMVar
takeMVar :: MVar a -> IO a
```

* 正如我们稍后将展示的，GHC 线程非常轻量级。如果运行时为每个线程提供一种检查状态的方法，即使从未使用此信息，每个线程的开销也会增加。

如果我们尝试将值放入已满的 MVar，我们的线程将进入睡眠状态，直到另一个线程取出值。同样，如果我们尝试从空的 MVar 中取出值，我们的线程将进入睡眠状态，直到其他线程放入值：

```haskell
-- file: ch24/MVarExample.hs
import Control.Concurrent
communicate = do
  m <- newEmptyMVar
  forkIO $ do
    v <- takeMVar m
    putStrLn ("received " ++ show v)
  putStrLn "sending"
  putMVar m "wake up!"
```

newEmptyMVar 函数有一个描述性的名称。要创建起始为非空的 MVar，我们使用 newMVar：

```ghci
ghci> :t newEmptyMVar
newEmptyMVar :: IO (MVar a)
ghci> :t newMVar
newMVar :: a -> IO (MVar a)
```

让我们在 ghci 中运行示例：

```ghci
ghci> :load MVarExample
[1 of 1] Compiling Main             ( MVarExample.hs, interpreted )
Ok, modules loaded: Main.
ghci> communicate
sending
received "wake up!"
```

如果你来自传统语言并发编程的背景，你可以认为 MVar 对两个熟悉的目的是有用的：

- 从一个线程向另一个线程发送消息，例如通知。
- 为在线程之间共享的一块可变数据提供互斥。当任何线程不使用时，我们将数据放入 MVar。然后一个线程暂时取出它来读取或修改。

## 主线程和等待其他线程

GHC 的运行时系统将程序的原始控制线程与其他线程区别对待。当此线程完成执行时，运行时系统认为整个程序已完成。如果当时有任何其他线程在执行，它们将被终止。

因此，当我们有不能被杀死的长运行线程时，我们需要做出特殊安排以确保主线程在其他线程完成之前不会完成。让我们开发一个使这易于实现的小型库：

```haskell
-- file: ch24/NiceFork.hs
import Control.Concurrent
import Control.Exception (Exception, try)
import qualified Data.Map as M
data ThreadStatus = Running
                  | Finished         -- terminated normally
                  | Threw Exception  -- killed by uncaught exception
                    deriving (Eq, Show)
-- | Create a new thread manager.
newManager :: IO ThreadManager
-- | Create a new managed thread.
forkManaged :: ThreadManager -> IO () -> IO ThreadId
-- | Immediately return the status of a managed thread.
getStatus :: ThreadManager -> ThreadId -> IO (Maybe ThreadStatus)
-- | Block until a specific managed thread terminates.
waitFor :: ThreadManager -> ThreadId -> IO (Maybe ThreadStatus)
-- | Block until all managed threads terminate.
waitAll :: ThreadManager -> IO ()
```

我们使用通常的方法将 ThreadManager 类型保持为抽象的：我们将其包装在 newtype 中，并阻止客户端创建此类型的值。在我们模块的导出中，我们列出类型构造器和构造 manager 的 IO 操作，但不导出数据构造器：

```haskell
-- file: ch24/NiceFork.hs
module NiceFork
    (
      ThreadManager
    , newManager
    , forkManaged
    , getStatus
    , waitFor
    , waitAll
    ) where
```

对于 ThreadManager 的实现，我们维护从线程 ID 到线程状态的映射。我们将其称为线程映射：

```haskell
-- file: ch24/NiceFork.hs
newtype ThreadManager =
    Mgr (MVar (M.Map ThreadId (MVar ThreadStatus)))
    deriving (Eq)
newManager = Mgr `fmap` newMVar M.empty
```

这里我们使用了两层 MVar。我们在 MVar 中保留 Map。这让我们可以通过用新版本替换它来"修改"Map。我们还确保任何使用 Map 的线程都会看到一致的视图。

对于我们管理的每个线程，我们维护一个 MVar。每个线程的 MVar 起始为空，表示线程正在执行。当线程因正常终止或因未捕获的异常而被杀死时，我们将此信息放入 MVar。

要创建线程并监视其状态，我们必须执行一些簿记工作：

```haskell
-- file: ch24/NiceFork.hs
forkManaged (Mgr mgr) body =
    modifyMVar mgr $ \m -> do
      state <- newEmptyMVar
      tid <- forkIO $ do
        result <- try body
        putMVar state (either Threw (const Finished) result)
      return (M.insert tid state m, tid)
```

## 安全地修改 MVar

我们在前面代码的 forkManaged 中使用的 modifyMVar 函数非常有用。它是 takeMVar 和 putMVar 的安全组合：

```ghci
ghci> :t modifyMVar
modifyMVar :: MVar a -> (a -> IO (a, b)) -> IO b
```

它从 MVar 中取出值并将其传递给函数。这个函数可以生成新值并返回结果。如果函数抛出异常，modifyMVar 将原始值放回 MVar；否则，它放入新值。它将函数的另一个元素作为自己的结果返回。

当我们使用 modifyMVar 而不是手动使用 takeMVar 和 putMVar 管理 MVar 时，我们避免了两类常见的并发 bug：

- 忘记将值放回 MVar。这可能导致死锁，其中一些线程永远等待一个永远不会放入值的 MVar。
- 未能考虑可能抛出异常的可能性，破坏代码流程。这可能导致应该发生的 putMVar 调用实际上没有发生，再次导致死锁。

由于这些良好的安全特性，尽可能使用 modifyMVar 是明智的。

## 安全的资源管理：一个好主意，而且容易实现

我们可以采用 modifyMVar 遵循的模式并将其应用于许多其他资源管理情况。以下是模式的步骤：

1. 获取资源。
2. 将资源传递给将使用它的函数。
3. 始终释放资源，即使函数抛出异常。如果发生这种情况，重新抛出异常，以便应用代码可以捕获它。

除了安全性，这种方法还有另一个好处：它可以使我们的代码更短、更易于理解。正如我们从前面代码列表中的 forkManaged 所看到的，Haskell 用于匿名函数的轻量级语法使这种编码风格在视觉上不显眼。

以下是 modifyMVar 的定义，以便你可以看到这个模式的具体形式：

```haskell
-- file: ch24/ModifyMVar.hs
import Control.Concurrent (MVar, putMVar, takeMVar)
import Control.Exception (block, catch, throw, unblock)
import Prelude hiding (catch) -- use Control.Exception's version
modifyMVar :: MVar a -> (a -> IO (a,b)) -> IO b
modifyMVar m io =
  block $ do
    a <- takeMVar m
    (b,r) <- unblock (io a) `catch` \e ->
             putMVar m a >> throw e
    putMVar m b
    return r
```

你应该能够轻松地根据自己的需要进行调整，无论你使用的是网络连接、数据库句柄还是由 C 库管理的数据。

## 查找线程状态

我们的 getStatus 函数告诉我们线程的当前状态。如果线程不再被管理（或最初从未被管理），它返回 Nothing：

```haskell
-- file: ch24/NiceFork.hs
getStatus (Mgr mgr) tid =
  modifyMVar mgr $ \m ->
    case M.lookup tid m of
      Nothing -> return (m, Nothing)
      Just st -> tryTakeMVar st >>= \mst -> case mst of
                   Nothing -> return (m, Just Running)
                   Just sth -> return (M.delete tid m, Just sth)
```

如果线程仍在运行，它返回 Just Running。否则，它指示线程终止的原因并停止管理线程。

如果 tryTakeMVar 函数发现 MVar 为空，它立即返回 Nothing 而不是阻塞：

```ghci
ghci> :t tryTakeMVar
tryTakeMVar :: MVar a -> IO (Maybe a)
```

否则，它像平常一样从 MVar 中提取值。

waitFor 函数的行为类似，但它不是立即返回，而是在返回之前阻塞直到给定线程终止：

```haskell
-- file: ch24/NiceFork.hs
waitFor (Mgr mgr) tid = do
  maybeDone <- modifyMVar mgr $ \m ->
    return $ case M.updateLookupWithKey (\_ _ -> Nothing) tid m of
      (Nothing, _) -> (m, Nothing)
      (done, m') -> (m', done)
  case maybeDone of
    Nothing -> return Nothing
    Just st -> Just `fmap` takeMVar st
```

它首先提取保存线程状态的 MVar（如果存在）。Map 类型的 updateLookupWithKey 函数很有用——它将查找键与修改或删除值结合起来：

```ghci
ghci> :m +Data.Map
ghci> :t updateLookupWithKey
updateLookupWithKey :: (Ord k) =>
                       (k -> a -> Maybe a) -> k -> Map k a -> (Maybe a, Map k a)
```

在这种情况下，我们希望如果存在的话，始终删除保存线程状态的 MVar，这样我们的线程管理器就不再管理该线程。如果有要提取的值，我们从 MVar 中取出线程的退出状态并返回它。

我们最后一个有用的函数只是等待所有当前管理的线程完成并忽略它们的退出状态：

```haskell
-- file: ch24/NiceFork.hs
waitAll (Mgr mgr) = modifyMVar mgr elems >>= mapM_ takeMVar
    where elems m = return (M.empty, M.elems m)
```

## 编写更紧凑的代码

我们对 waitFor 的定义有点不令人满意，因为我们在两个地方执行或多或少相同的 case 分析：在 modifyMVar 调用的函数内部，以及在其返回值上再次分析。

确实，我们可以应用一个 earlier 遇到的函数来消除这种重复。该函数是 join，来自 Control.Monad 模块：

```ghci
ghci> :m +Control.Monad
ghci> :t join
join :: (Monad m) => m (m a) -> m a
```

这里的技巧是，我们可以摆脱第二个 case 表达式，方法是在 modifyMVar 返回之前让第一个 case 返回我们应在 modifyMVar 返回后执行的 IO 操作。我们将使用 join 来执行该操作：

```haskell
-- file: ch24/NiceFork.hs
waitFor2 (Mgr mgr) tid =
  join . modifyMVar mgr $ \m ->
    return $ case M.updateLookupWithKey (\_ _ -> Nothing) tid m of
      (Nothing, _) -> (m, return Nothing)
      (Just st, m') -> (m', Just `fmap` takeMVar st)
```

这是一个有趣的想法：我们可以在纯代码中创建一个单子函数或动作，然后将其传递，直到我们进入可以使用它的单子中。一旦你培养了对何时有意义的眼光，这可以是一种灵活编写代码的方式。

## 通过通道通信

对于线程之间的一次性通信，MVar 是完全合适的。另一种类型 Chan 提供单向通信通道。以下是其使用的一个简单示例：

```haskell
-- file: ch24/Chan.hs
import Control.Concurrent
import Control.Concurrent.Chan
chanExample = do
  ch <- newChan
  forkIO $ do
    writeChan ch "hello world"
    writeChan ch "now i quit"
  readChan ch >>= print
  readChan ch >>= print
```

如果 Chan 为空，readChan 会阻塞，直到有值可读。writeChan 函数从不阻塞；它立即将新值写入 Chan。

## 需要了解的有用知识

### MVar 和 Chan 是非严格的

像大多数 Haskell 容器类型一样，MVar 和 Chan 都是非严格的：两者都不会对其内容求值。我们提到这一点不是因为它是个问题，而是因为它是一个常见的盲点。人们倾向于假设这些类型是严格的，也许是因为它们在 IO monad 中使用。

对于其他容器类型，对 MVar 或 Chan 类型严格性的错误猜测的结果通常是空间或性能泄漏。这里有一个值得考虑的合理场景。

我们 fork 一个线程在另一个核心上执行一些昂贵的计算：

```haskell
-- file: ch24/Expensive.hs
import Control.Concurrent
notQuiteRight = do
  mv <- newEmptyMVar
  forkIO $ expensiveComputation_stricter mv
  someOtherActivity
  result <- takeMVar mv
  print result
```

它似乎做了一些事情并将结果放回 MVar：

```haskell
-- file: ch24/Expensive.hs
expensiveComputation mv = do
  let a = "this is "
      b = "not really "
      c = "all that expensive"
  putMVar mv (a ++ b ++ c)
```

当我们从父线程的 MVar 中取出结果并尝试对其进行处理时，我们的线程开始疯狂计算，因为我们从未强制计算实际发生在另一个线程中！

一如既往，一旦知道可能出现问题，解决方案就很简单了：我们将严格性添加到 forked 线程，以确保计算在那里发生。这种严格性最好在一个地方添加，以避免我们可能忘记添加它的可能性：

```haskell
-- file: ch24/ModifyMVarStrict.hs
{-# LANGUAGE BangPatterns #-}
import Control.Concurrent (MVar, putMVar, takeMVar)
import Control.Exception (block, catch, throw, unblock)
import Prelude hiding (catch) -- use Control.Exception's version
modifyMVar_strict :: MVar a -> (a -> IO a) -> IO ()
modifyMVar_strict m io = block $ do
  a <- takeMVar m
  !b <- unblock (io a) `catch` \e ->
        putMVar m a >> throw e
  putMVar m b
```

值得随时检查 Hackage

在 Hackage 包数据库中，你会找到一个库 strict-concurrency，它提供 MVar 和 Chan 类型的严格版本。

前面代码中的 ! 模式使用简单，但并不总是足以确保我们的数据被求值。对于更完整的方法，请参阅第 552 页的"将算法与求值分离"。

### Chan 是无界的

因为 writeChan 总是立即成功，使用 Chan 存在潜在风险。如果一个线程写入 Chan 的频率高于另一个线程读取的频率，Chan 将以不受控制的方式增长：未读消息会随着读者越来越落后而堆积。

### 共享状态并发仍然很困难

虽然 Haskell 用于在线程之间共享数据的原语与其他语言不同，但它仍然遭受相同的基本问题：编写正确的并发程序是极其困难的。实际上，其他语言中并发编程的几个陷阱同样适用于 Haskell。其中两个更知名的问题是死锁和饥饿。

### 死锁

在死锁情况下，两个或多个线程在争夺对共享资源的访问时陷入僵局。一种使多线程程序死锁的经典方式是忘记我们必须获取锁的顺序。这种 bug 非常常见，它有一个名字：锁顺序反转。虽然 Haskell 不提供锁，但 MVar 类型容易出现顺序反转问题。这是一个简单的示例：

```haskell
-- file: ch24/LockHierarchy.hs
import Control.Concurrent
nestedModification outer inner = do
  modifyMVar_ outer $ \x -> do
    yield  -- force this thread to temporarily yield the CPU
    modifyMVar_ inner $ \y -> return (y + 1)
    return (x + 1)
  putStrLn "done"
main = do
  a <- newMVar 1
  b <- newMVar 2
  forkIO $ nestedModification a b
  forkIO $ nestedModification b a
```

如果我们在 ghci 中运行它，通常——但并非总是——不会打印任何内容，表明两个线程都卡住了。

nestedModification 函数的问题很容易发现。在第一个线程中，我们取 MVar a，然后取 b。在第二个线程中，我们取 b，然后取 a。如果第一个线程成功取了 a，第二个线程取了 b，两个线程都会阻塞；每个都试图取另一个已经清空的 MVar，所以两者都无法取得进展。

跨语言，解决顺序反转问题的通常方法是始终遵循一致顺序获取资源。由于这种方法需要手动遵守编码约定，因此在实践中很容易被忽视。

更复杂的是，这些类型的反转问题在真实代码中可能很难发现。获取 MVar 通常分布在不同文件中的几个函数中，使视觉检查更加棘手。更糟糕的是，这些问题通常是间歇性的，这使得它们很难重现，更不用说隔离和修复了。

### 饥饿

并发软件也容易受到饥饿的影响，其中一个线程"占用"共享资源，阻止另一个使用它。很容易想象这可能如何发生：一个线程使用执行时间为 100 毫秒的 body 调用 modifyMVar，而另一个使用执行时间为 1 毫秒的 body 调用相同 MVar 上的 modifyMVar。第二个线程在第一个将值放回 MVar 之前无法取得进展。

MVar 类型的非严格性可能导致或加剧饥饿问题。如果我们把一个求值代价高昂的 thunk 放入 MVar，然后在一个看起来应该很便宜的线程中取出它，如果该线程必须求值该 thunk，它可能会突然变得计算成本高昂。这使得我们在第 539 页"MVar 和 Chan 是非严格的"中给出的建议特别相关。

### 还有希望吗？

幸运的是，我们在这里介绍的并发 API 绝不是故事的全部。Haskell 的一个更近期的补充——软件事务内存（STM）——使用起来既更容易也更安全。我们将在第 28 章中讨论它。

## 练习

1. Chan 类型使用 MVar 实现。使用 MVar 开发一个 BoundedChan 库。你的 newBoundedChan 函数应接受一个 Int 参数，限制一次可以在 BoundedChan 中存在的未读项目数。如果达到此限制，你的 writeBoundedChan 调用必须阻塞，直到读取器使用 readBoundedChan 消费一个值。
2. 虽然我们已经提到了 Hackage 仓库中 strict-concurrency 包的存在，但尝试开发你自己的，作为一个围绕内置 MVar 类型的包装器。遵循经典的 Haskell 实践，使你的库类型安全，这样用户就不会意外混合使用严格和非严格的 MVar。

## 使用 GHC 的多个核心

默认情况下，GHC 生成的程序只使用一个核心，即使我们编写了显式并发代码。要使用多个核心，我们必须明确选择这样做。我们在生成可执行程序时在链接时做出此选择：

- 非线程化运行时库在单个操作系统线程中运行所有 Haskell 线程。此运行时对于创建线程和在 MVar 中传递数据非常高效。
- 线程化运行时库使用多个操作系统线程来运行 Haskell 线程。它在创建线程和使用 MVar 方面有稍多的开销。

如果我们向编译器传递 -threaded 选项，它将链接到线程化运行时库。我们不需要在编译库或源文件时使用 -threaded——只有在最终生成可执行文件时才需要。

即使我们为程序选择了线程化运行时，它在运行仍将默认只使用一个核心。我们必须明确告诉运行时使用多少个核心。

## 运行时选项

我们可以在程序命令行上向 GHC 的运行时系统传递选项。在将控制权交给我们的代码之前，运行时扫描程序的参数，查找特殊命令行选项 +RTS。它将跟随的所有内容（直到特殊选项 -RTS）解释为运行时系统的选项，而不是我们程序的选项。它向我们隐藏所有这些选项。当我们使用 System.Environment 模块的 getArgs 函数获取命令行参数时，我们不会在列表中找到任何运行时选项。

线程化运行时接受选项 -N。† 这需要一个参数，指定 GHC 运行时系统应使用的核心数。选项解析器很挑剔：-N 和其后跟随的数字之间不能有任何空格。选项 -N4 是可接受的，但 -N 4 不是。

## 从 Haskell 查找可用核心数

模块 GHC.Conc 导出一个变量 numCapabilities，它告诉我们运行时系统已通过 -N RTS 选项给予了多少个核心：

```haskell
-- file: ch24/NumCapabilities.hs
import GHC.Conc (numCapabilities)
import System.Environment (getArgs)
main = do
  args <- getArgs
  putStrLn $ "command line arguments: " ++ show args
  putStrLn $ "number of cores: " ++ show numCapabilities
```

如果我们编译并运行此程序，我们可以看到运行时系统的选项对程序不可见，但我们可以看到它可以在多少个核心上运行：

```bash
$ ghc -c NumCapabilities.hs
$ ghc -threaded -o NumCapabilities NumCapabilities.o
$ ./NumCapabilities +RTS -N4 -RTS foo

number of cores: 4
```

† 非线程化运行时不理解此选项，并将用错误消息拒绝它。

## 选择正确的运行时

选择使用哪个运行时并不完全明确。虽然线程化运行时可以使用多个核心，但它有代价：线程和在它们之间共享数据比非线程化运行时更昂贵。

此外，GHC 6.8.3 版使用的垃圾收集器是单线程的：它在运行时暂停所有其他线程并在一个核心上执行。这限制了我们希望从使用多个核心中看到的性能改进。在许多现实世界的并发程序中，单个线程大部分时间都在等待网络请求或响应。在这些情况下，如果单个 Haskell 程序服务数万个并发客户端，非线程化运行时的较低开销可能会有所帮助。例如，如果我们设计服务器以便可以同时运行四个副本并使用非线程化运行时，而不是让单个服务器程序在四个核心上使用线程化运行时，我们可能会看到更好的性能。

我们这里的目的不是劝阻你使用线程化运行时。它并不比非线程化运行时贵多少——与其他大多数编程语言的运行时相比，线程仍然非常便宜。我们只是想明确指出，切换到线程化运行时不一定会导致自动获胜。

## Haskell 中的并行编程

现在我们将把重点转向并行编程。对于许多计算密集型问题，如果我们能够将解决方案划分并同时在多个核心上求值，我们可以更快地计算结果。配备多个核心的计算机已经无处不在，但很少有程序能够利用甚至现代笔记本电脑的计算能力。

在很大程度上，这是因为传统上认为并行编程非常困难。在典型的编程语言中，我们会使用与应用于并发程序相同的库和构造来开发并行程序。这迫使我们与熟悉的死锁、竞争条件、饥饿和纯粹复杂性等问题作斗争。

虽然我们当然可以使用 Haskell 的并发特性来开发并行代码，但我们可以使用一种更简单的方法。我们可以采用一个普通的 Haskell 函数，对其应用一些简单的转换，并让它并行求值。

## 正规形式和头正规形式

熟悉的 seq 函数将表达式求值到我们所说的头正规形式（HNF）。它一旦到达最外层的构造器（头）就停止。这与正规形式（NF）不同，在正规形式中，表达式被完全求值。

你还会听到 Haskell 程序员提到弱头正规形式（WHNF）。对于正常数据，弱头正规形式与头正规形式相同。差异仅在函数中出现，这里太深奥，不关我们的事。

## 顺序排序

这是一个使用分而治之方法对列表排序的普通 Haskell 函数：

```haskell
-- file: ch24/Sorting.hs
sort :: (Ord a) => [a] -> [a]
sort (x:xs) = lesser ++ x:greater
    where lesser  = sort [y | y <- xs, y <  x]
          greater = sort [y | y <- xs, y >= x]
sort _ = []
```

这个函数灵感来自著名的 Quicksort 算法，它是 Haskell 程序员中的经典。它经常在 Haskell 教程的早期作为一行代码展示，以用一个 Haskell 表达性示例来吸引读者。这里，我们将代码分成几行，以便更容易比较串行和并行版本。

以下是对 sort 操作方式的非常简要描述：

1. 它从列表中选择一个元素。这被称为 pivot。任何元素都可以作为 pivot；第一个只是最容易模式匹配的。
2. 它创建一个包含所有小于 pivot 的元素的子列表，并递归排序它们。
3. 它创建一个包含所有大于或等于 pivot 的元素的子列表，并递归排序它们。
4. 它附加两个排序后的子列表。

## 将代码转换为并行代码

函数的并行版本只比初始版本复杂一点：

```haskell
-- file: ch24/Sorting.hs
module Sorting where
import Control.Parallel (par, pseq)
parSort :: (Ord a) => [a] -> [a]
parSort (x:xs)    = force greater `par` (force lesser `pseq`
                                         (lesser ++ x:greater))
    where lesser  = parSort [y | y <- xs, y <  x]
          greater = parSort [y | y <- xs, y >= x]
parSort _         = []
```

我们几乎没有扰乱代码——我们只添加了三个函数：par、pseq 和 force。

par 函数由 Control.Parallel 模块提供。它的作用类似于 seq。它求值其左参数到 WHNF 并返回其右参数。正如其名称所暗示的，par 可以与其正在进行的任何其他求值并行求值其左参数。

至于 pseq，它类似于 seq：它在返回正确的参数之前求值左侧的表达式到 WHNF。两者之间的差异对并行程序很微妙但重要：如果编译器看到先求值正确的参数会提高性能，它不会承诺求值 seq 的左参数。这种灵活性对于在单个核心上执行的程序是好的，但对于在多个核心上运行的代码不够强。相比之下，编译器保证 pseq 会在其右侧之前求值其左侧参数。

我们对代码的更改在为所有我们不需要说的事情而 notable：

- 使用多少个核心
- 线程如何相互通信
- 如何在可用核心之间分配工作
- 哪些数据在线程之间共享，哪些是私有的
- 如何确定所有参与者何时完成

## 知道要并行求值什么

从并行 Haskell 代码获得良好性能的关键是找到有意义的chunk工作来做。惰性求值可能会阻碍这一点，这就是我们在并行排序中使用 force 函数的原因。为了最好地解释 force 函数的用途，我们首先看一个错误的示例：

```haskell
-- file: ch24/Sorting.hs
sillySort (x:xs) = greater `par` (lesser `pseq`
                                  (lesser ++ x:greater))
    where lesser   = sillySort [y | y <- xs, y <  x]
          greater  = sillySort [y | y <- xs, y >= x]
sillySort _        = []
```

看看在 par 的每次使用中的小变化。与其求值 lesser 和 greater，这里我们求值 lesser 和 greater。

记住，求值到 WHNF 只会计算表达式到足以看到其最外层构造器的程度。在这个错误的示例中，我们对每个排序子列表求值到 WHNF。因为在每种情况下最外层的构造器只是一个列表构造器，我们实际上只强制求值每个排序子列表的第一个元素！

列表的每个其他元素保持未求值。换句话说，我们几乎不做并行工作：我们的 sillySort 几乎是完全顺序的。

我们通过 force 函数避免这个问题，方法是在我们给回构造器之前强制整个列表 spine 被求值：

```haskell
-- file: ch24/Sorting.hs
force :: [a] -> ()
force xs = go xs `pseq` ()
    where go (_:xs) = go xs
          go [] = 1
```

注意，我们不关心列表中有设么；我们沿着它的 spine 走到最后，然后使用 pseq。显然这里没有涉及魔法——我们只是在使用对 Haskell 求值模型的 usual 理解。而且因为我们将在 par 或 pseq 的左侧使用 force，我们不需要返回一个有意义的值。

当然，在许多情况下，我们也需要强制求值列表的各个元素。下面，我们将讨论基于类型类的解决方案。

## par 做出什么承诺？

par 函数实际上并不承诺与另一个并行求值表达式。相反，它承诺如果"有意义"就这样做。这种模糊的承诺实际上比总是保证并行求值表达式更有用。它给运行时系统自由地在遇到 par 时智能行事。

例如，运行时可以决定表达式太便宜，不值得并行求值。或者它可能注意到所有核心当前都忙，以至于"激发"新的并行求值会导致比可用核心更多的可运行线程。

这种宽松的规范反过来影响我们编写并行代码的方式。因为 par 在运行时可能有点智能，我们几乎可以在任何地方使用它，假设性能不会因线程争夺繁忙核心而停滞。

## 运行我们的代码并测量性能

为了尝试我们的代码，让我们将 sort、parSort 和 parSort2 保存到一个名为 Sorting.hs 的模块中。我们创建一个小驱动程序程序，可以用来计时其中一个排序函数的性能：

```haskell
-- file: ch24/SortMain.hs
module Main where
import Data.Time.Clock (diffUTCTime, getCurrentTime)
import System.Environment (getArgs)
import System.Random (StdGen, getStdGen, randoms)
import Sorting
-- testFunction = sort
-- testFunction = seqSort
testFunction = parSort
-- testFunction = parSort2 2
randomInts :: Int -> StdGen -> [Int]
randomInts k g = let result = take k (randoms g)
                 in force result `seq` result
main = do
  args <- getArgs
  let count | null args = 500000
            | otherwise = read (head args)
  input <- randomInts count `fmap` getStdGen
  putStrLn $ "We have " ++ show (length input) ++ " elements to sort."
  start <- getCurrentTime
  let sorted = testFunction input
  putStrLn $ "Sorted all " ++ show (length sorted) ++ " elements."
  end <- getCurrentTime
  putStrLn $ show (end `diffUTCTime` start) ++ " elapsed."
```

为简单起见，我们通过 testFunction 变量在编译时选择要基准测试的排序函数。

我们的程序接受一个可选的命令行参数，即要生成的随机列表的长度。

惰性求值会将性能测量和分析变成一片雷区。这里有一些我们专门在驱动程序中努力避免的潜在问题：

测量我们认为只看一个时的多件事
Haskell 默认的伪随机数生成器（PRNG）很慢，randoms 函数按需生成随机数。在我们记录开始时间之前，我们强制输入列表的每个元素被求值，并打印列表的长度。这确保了我们提前创建我们将需要的所有随机数。如果我们省略此步骤，我们将在并行尝试处理它们时交错生成随机数。因此，我们既要衡量排序数字的成本，也要——不太明显地——生成它们的成本。

隐形数据依赖
当我们生成随机数列表时，仅打印列表的长度不会执行足够的求值。这会对列表的 spine 求值，但不会对其元素求值。实际的随机数直到排序比较它们时才会被求值。

这可能对性能产生严重影响。随机数的值取决于列表中前一个随机数的值，但我们将列表元素随机分散在处理器核心之间。如果我们不在排序之前对列表元素求值，我们会遭受可怕的"乒乓"效应：求值不仅会从一个核心弹到另一个核心，性能也会受到影响。

尝试从 main 的主体中删除 force 的应用。你应该发现并行代码最终容易慢三倍。

当我们相信代码正在执行有意义的工作时，对 thunk 进行基准测试
为了强制排序发生，我们在记录结束时间之前打印结果列表的长度。如果没有 putStrLn 要求列表的长度以便打印它，排序根本不会发生。

当我们构建程序时，我们启用优化和 ghc 的线程化运行时：

```bash

[2 of 2] Compiling Main             ( SortMain.hs, SortMain.o )
Linking SortMain ...
```

当我们运行程序时，我们必须告诉 ghc 的运行时使用多少个核心。首先，我们尝试原始排序，以建立性能基准：

```bash

```

启用第二个核心应该对性能没有影响：

-- file: ch24/Sorting.hs
seqSort :: (Ord a) => [a] -> [a]
seqSort (x:xs) = lesser `pseq` (greater `pseq`
                                (lesser ++ x:greater))
    where lesser  = seqSort [y | y <- xs, y <  x]
          greater = seqSort [y | y <- xs, y >= x]
seqSort _ = []
```

我们还放弃了 force 的使用，因此与原始排序相比，我们只应测量使用 pseq 的成本。单独使用 pseq 对性能有什么影响？

```

这表明 par 和 pseq 有类似的成本。我们怎样才能提高性能？

## 性能调优

在我们的 parSort 中，我们执行的 par 应用次数是要排序元素数量的两倍。虽然我们看到 par 很便宜，但它不是免费的。当我们递归应用 parSort 时，我们最终会对单个列表元素应用 par。在这个细粒度级别，使用 par 的成本超过了任何可能的有用性。为了减少这种影响，我们在经过某个阈值后切换到非并行排序：

```haskell
-- file: ch24/Sorting.hs
parSort2 :: (Ord a) => Int -> [a] -> [a]
parSort2 d list@(x:xs)
  | d <= 0     = sort list
  | otherwise = force greater `par` (force lesser `pseq`
                                     (lesser ++ x:greater))
      where lesser      = parSort2 d' [y | y <- xs, y <  x]
            greater     = parSort2 d' [y | y <- xs, y >= x]
            d' = d - 1
parSort2 _ _              = []
```

在这里，我们在可控深度后停止递归和激发新的并行求值。如果我们知道我们正在处理的数据大小，我们可以停止细分，一旦我们有足够少的剩余工作，就切换到非并行代码：

```bash
$ ./Sorting +RTS -N2 -RTS 700000
We have 700000 elements to sort.
Sorted all 700000 elements.
2.947872s elapsed.
```

在双核系统上，这给我们大约 25% 的加速。这不是一个很大的数字，但考虑到我们只需更改几个注释就能获得这种性能改进。

这个排序函数特别抗拒良好的并行性能。它执行的内存分配量迫使垃圾收集器频繁运行。我们可以通过使用 -sstderr RTS 选项运行程序来看到效果，它将垃圾收集统计信息打印到屏幕。这表明我们的程序大约 40% 的时间花在收集垃圾上。因为 GHC 6.8 中的垃圾收集器停止所有线程并在一个核心上运行，它成为了一个瓶颈。

当你使用 par 注释时，对于分配较少的代码，你可以期待更令人印象深刻的性能改进。我们已经看到一些简单的数值基准在双核系统上比单核运行快 1.8 倍。在撰写本文时，GHC 的并行垃圾收集器正在开发中，应该会大大有助于在多核系统上处理分配密集型代码的性能。

虽然我们可以毫不费力地创建一个直接的实现，但我们会抵制深入研究的诱惑。如果我们考虑解决一类问题而不是单个问题，我们最终可能会得到更广泛适用的代码。

当我们开发并行程序时，我们总是面临几个"坏硬币"问题，无论底层编程语言如何，这些问题都会出现。其中一些在这里描述：

- 我们的算法很快被分区和通信的细节所掩盖。这使得理解代码变得困难，反过来又使修改它变得有风险。
- 选择粒度大小——分配给核心的最小工作单元——可能很困难。如果粒度太小，核心花费太多时间在簿记上，并行程序很容易变得比串行对应物慢。如果粒度太大，一些核心可能由于负载平衡不良而空闲。

## 将算法与求值分离

在并行 Haskell 代码中，传统语言中通信代码带来的混乱被 par 和 pseq 注释的混乱所取代。例如，此函数操作类似于 map，但随着它的进行并行对每个元素求值到 WHNF：

```haskell
-- file: ch24/ParMap.hs
import Control.Parallel (par)
parallelMap :: (a -> b) -> [a] -> [b]
parallelMap f (x:xs) = let r = f x
                      in r `par` r : parallelMap f xs
parallelMap _ _      = []
```

类型 b 可能是一个列表或其他类型，对于这些类型，求值到 WHNF 不会做有用量的工作。我们宁愿不想为列表和每个其他需要特殊处理的类型编写特殊的 parallelMap。

为了解决这个问题，我们将首先考虑一个更简单的问题：如何强制值被求值。这是一个强制列表每个元素求值到 WHNF 的函数：

```haskell
-- file: ch24/ParMap.hs
forceList :: [a] -> ()
forceList (x:xs) = x `pseq` forceList xs
forceList _      = ()
```

我们的函数不对列表执行任何计算。（实际上，从检查其类型签名，我们可以判断它不能对元素执行任何计算，因为它对列表的元素一无所知。）它的唯一目的是确保列表的 spine 被求值到头正规形式。应用此函数的唯一合理地方是在 seq 或 par 的第一个参数中，如下所示：

```haskell
-- file: ch24/ParMap.hs
stricterMap :: (a -> b) -> [a] -> [b]
stricterMap f xs = forceList xs `seq` map f xs
```

这仍然将列表的元素仅求值到 WHNF。我们通过添加一个作为参数的函数来更深入地求值元素：

```haskell
-- file: ch24/ParMap.hs
forceListAndElts :: (a -> ()) -> [a] -> ()
forceListAndElts forceElt (x:xs) =
    forceElt x `seq` forceListAndElts forceElt xs
forceListAndElts _        _      = ()
```

Control.Parallel.Strategies 模块将此想法概括为我们可以作为库使用的东西。它引入了评估策略的概念：

```haskell
-- file: ch24/Strat.hs
type Done = ()
type Strategy a = a -> Done
```

评估策略不执行任何计算；它只是确保值被求值到某种程度。最简单的策略名为 r0，什么都不做：

```haskell
-- file: ch24/Strat.hs
r0 :: Strategy a
r0 _ = ()
```

接下来是 rwhnf，它将值求值到 WHNF：

```haskell
-- file: ch24/Strat.hs
rwhnf :: Strategy a
rwhnf x = x `seq` ()
```

要要将值求值到正规形式，模块提供了一个类型类，名为 rnf：

```haskell
-- file: ch24/Strat.hs
class NFData a where
  rnf :: Strategy a
  rnf = rwhnf
```

如果这些函数和类型的名称记不住，把它们当作首字母缩略词。rwhnf 扩展为 reduce to weak head normal form；NFData 变成 normal form data；等等。

对于基本类型（如 Int），弱头正规形式和正规形式是一样的，这就是为什么 NFData 类型类使用 rwhnf 作为 rnf 的默认实现。对于许多常见类型，Control.Parallel.Strategies 模块提供了 NFData 实例：

```haskell
-- file: ch24/Strat.hs
instance NFData Char
instance NFData Int
instance NFData a => NFData (Maybe a) where
    rnf Nothing  = ()
    rnf (Just x) = rnf x
{- ... and so on ... -}
```

从这些示例中，你应该清楚如何为你自己的类型编写 NFData 实例。你的 rnf 实现必须处理每个构造器并对构造器的每个字段应用 rnf。

## 将算法与策略分离

从这些策略构建块，我们可以构建更复杂的策略。许多已经由 Control.Parallel.Strategies 提供。例如，parList 对列表的每个元素并行应用评估策略：

```haskell
-- file: ch24/Strat.hs
parList :: Strategy a -> Strategy [a]
parList strat []     = ()
parList strat (x:xs) = strat x `par` (parList strat xs)
```

该模块使用它来定义并行映射函数：

```haskell
-- file: ch24/Strat.hs
parMap :: Strategy b -> (a -> b) -> [a] -> [b]
parMap strat f xs = map f xs `using` parList strat
```

这就是代码变得有趣的地方。在 using 的左侧，我们有 map 的正常应用。在右侧，我们有一个评估策略。using 组合符告诉我们如何将策略应用于值，允许我们将代码与如何求值它的计划分开：

```haskell
-- file: ch24/Strat.hs
using :: a -> Strategy a -> a
using x s = s x `seq` x
```

Control.Parallel.Strategies 模块提供了许多其他函数，可以对求值进行细粒度控制。例如，parZipWith 使用评估策略并行应用 zipWith：

```haskell
-- file: ch24/Strat.hs
vectorSum' :: (NFData a, Num a) => [a] -> [a] -> [a]
vectorSum' = parZipWith rnf (+)
```

## 编写一个简单的 MapReduce 定义

通过考虑必须做什么，我们可以快速为 mapReduce 函数建议一个类型。我们需要一个 map 组件，我们给它通常的类型 a -> b。我们需要一个 reduce；这个术语是 fold 的同义词。我们不是 commit 使用特定类型的 fold，而是使用更通用的类型 [b] -> c。这个类型让我们可以使用左 fold 或右 fold，所以我们可以選擇适合我们的数据和处理需求的。

如果我们将这些类型插入，完整的类型看起来像这样：

```haskell
-- file: ch24/MapReduce.hs
simpleMapReduce
    :: (a -> b)      -- map function
    -> ([b] -> c)    -- reduce function
    -> [a]           -- list to map over
    -> c
```

与类型配套的代码非常简单：

```haskell
-- file: ch24/MapReduce.hs
simpleMapReduce mapFunc reduceFunc = reduceFunc . map mapFunc
```

## MapReduce 和策略

我们对 simpleMapReduce 的定义太简单，不够有趣。为了使其有用，我们希望能够指定某些工作应该并行发生。我们将使用策略来实现，传入映射阶段的策略和归约阶段的策略：

```haskell
-- file: ch24/MapReduce.hs
mapReduce
    :: Strategy b    -- evaluation strategy for mapping
    -> (a -> b)      -- map function
    -> Strategy c    -- evaluation strategy for reduction
    -> ([b] -> c)    -- reduce function
    -> [a]           -- list to map over
    -> c
```

类型和函数体都必须稍微增大以适应策略参数。

```haskell
-- file: ch24/MapReduce.hs
mapReduce mapStrat mapFunc reduceStrat reduceFunc input =
    mapResult `pseq` reduceResult
  where mapResult    = parMap mapStrat mapFunc input
        reduceResult = reduceFunc mapResult `using` reduceStrat
```

## 适当调整工作大小

为了获得不错的性能，我们必须确保每次应用 par 的工作大大超过其簿记成本。如果我们处理一个巨大的文件，按行分割给我们的工作远远超过开销。我们将在后面的部分开发一种以更大块处理文件的方法。那些块应该由什么组成？因为 Web 服务器日志文件应该只包含 ASCII 文本，我们将使用惰性 ByteString 获得出色的性能。这种类型在我们从文件流式传输时非常高效且消耗很少内存：

```haskell
-- file: ch24/LineChunks.hs
module LineChunks
    (
      chunkedReadWith
    ) where
import Control.Exception (bracket, finally)
import Control.Monad (forM, liftM)
import Control.Parallel.Strategies (NFData, rnf)
import Data.Int (Int64)
import qualified Data.ByteString.Lazy.Char8 as LB
import GHC.Conc (numCapabilities)
import System.IO
data ChunkSpec = CS {
      chunkOffset :: !Int64
    , chunkLength :: !Int64
    } deriving (Eq, Show)
withChunks :: (NFData a) =>
              (FilePath -> IO [ChunkSpec])
           -> ([LB.ByteString] -> a)
           -> FilePath
           -> IO a
withChunks chunkFunc process path = do
  (chunks, handles) <- chunkedRead chunkFunc path
  let r = process chunks
  (rnf r `seq` return r) `finally` mapM_ hClose handles

chunkedReadWith :: (NFData a) =>
                   ([LB.ByteString] -> a) -> FilePath -> IO a
chunkedReadWith func path =
    withChunks (lineChunks (numCapabilities * 4)) func path
```

我们并行消费每个块，充分利用惰性 I/O 来确保我们可以安全地流式传输这些块。

## 减轻惰性 I/O 的风险

惰性 I/O 有一些众所周知的危害，我们希望避免：

- 通过不强制求值从文件提取数据的计算，我们可能将文件句柄保持打开的时间超过必要的时间。由于操作系统通常对我们一次可以打开的文件数量设置了一个小而固定的限制，如果我们不解决这个风险，我们可能会意外地耗尽程序其他部分的文件句柄。
- 如果我们不显式关闭文件句柄，垃圾收集器会自动为我们关闭它，但可能需要很长时间才能注意到应该关闭文件句柄。这会带来与前面提到的相同的饥饿风险。
- 我们可以通过显式关闭文件句柄来避免饥饿。然而，如果过早关闭，可能会导致惰性计算失败，如果它期望能够从关闭的文件句柄中提取更多数据的话。

除了这些众所周知的风险之外，我们不能使用单个文件句柄向多个线程提供数据。文件句柄有一个单一的寻址指针跟踪它应该从中读取的位置，但当我们想要读取多个块时，每个都需要从文件中的不同位置消耗数据。

考虑到这些想法，让我们充实惰性 I/O 的细节：

```haskell
-- file: ch24/LineChunks.hs
chunkedRead :: (FilePath -> IO [ChunkSpec])
            -> FilePath
            -> IO ([LB.ByteString], [Handle])
chunkedRead chunkFunc path = do
  chunks <- chunkFunc path
  liftM unzip . forM chunks $ \spec -> do
    h <- openFile path ReadMode
    hSeek h AbsoluteSeek (fromIntegral (chunkOffset spec))
    chunk <- LB.take (chunkLength spec) `liftM` LB.hGetContents h
    return (chunk, h)
```

我们通过显式关闭文件句柄来避免饥饿问题。我们通过为每个线程提供一个不同的文件句柄（都读取同一个文件）来允许多个线程同时读取不同的块。

我们试图减轻的最后一个问题是惰性计算在背后文件句柄被关闭的问题。我们使用 rnf 在从 withChunks 返回之前强制所有处理完成。然后我们可以显式关闭我们的文件句柄，因为它们应该不再被读取。如果你必须在程序中使用惰性 I/O，通常最好像这样"防火墙化"它，这样它就不会在代码的意外部分造成问题。

## 高效查找对齐行的块

由于服务器日志文件是面向行的，我们需要一种有效的方法将文件分割成大块，同时确保每个块在一行边界处结束。由于一个块可能有数十兆字节大小，我们不想扫描块中的所有数据来确定其最后一个边界应该在哪里。

我们的方法适用于选择固定块大小或固定数量的块。这里，我们选择后者。我们首先寻找接近块末尾位置的偏移量，然后向前扫描直到到达换行符字符。我们在下一次换行后开始以下块，并重复该过程：

```haskell
-- file: ch24/LineChunks.hs
lineChunks :: Int -> FilePath -> IO [ChunkSpec]
lineChunks numChunks path = do
  bracket (openFile path ReadMode) hClose $ \h -> do
    totalSize <- fromIntegral `liftM` hFileSize h
    let chunkSize = totalSize `div` fromIntegral numChunks
        findChunks offset = do
          let newOffset = offset + chunkSize
          hSeek h AbsoluteSeek (fromIntegral newOffset)
          let findNewline off = do
                eof <- hIsEOF h
                if eof
                  then return [CS offset (totalSize - offset)]
                  else do
                    bytes <- LB.hGet h 4096
                    case LB.elemIndex '\n' bytes of
                      Just n -> do
                        chunks@(c:_) <- findChunks (off + n + 1)
                        let coff = chunkOffset c
                        return (CS offset (coff - offset):chunks)
                      Nothing -> findNewline (off + LB.length bytes)
          findNewline newOffset
    findChunks 0
```

最后一个块最终比前面的短一点，但这种差异在实践中可以忽略不计。

## 计算行数

这个简单的示例说明瞭如何使用我们构建的脚手架：

```haskell
-- file: ch24/LineCount.hs
module Main where
import Control.Monad (forM_)
import Data.Int (Int64)
import qualified Data.ByteString.Lazy.Char8 as LB
import System.Environment (getArgs)
import LineChunks (chunkedReadWith)
import MapReduce (mapReduce, rnf)
lineCount :: [LB.ByteString] -> Int64
lineCount = mapReduce rnf (LB.count '\n')
                      rnf sum
main :: IO ()
main = do
  args <- getArgs
  forM_ args $ \path -> do
    numLines <- chunkedReadWith lineCount path
    putStrLn $ path ++ ": " ++ show numLines
```

如果我们使用 ghc -O2 --make -threaded 编译这个程序，在初始运行"热身"文件系统缓存后，它应该表现良好。在双核笔记本电脑上处理 248 兆字节（110 万行）的日志文件，此程序在使用单个核心时运行 0.576 秒，在使用两个核心时运行 0.361 秒（使用 +RTS -N2）。

## 查找最受欢迎的 URL

在这个示例中，我们计算每个 URL 被访问的次数。这个示例来自 Jeffrey Dean 和 Sanjay Ghemawat 的"MapReduce：简化大型集群上的数据处理"（http://labs.google.com/papers/mapreduce.html），Google 最初讨论 MapReduce 的论文。在 map 阶段，对于每个块，我们使用 URL 被访问的次数创建从 URL 到数字的 Map。在 reduce 阶段，我们将这些 Map union-merge 为一个：

```haskell
-- file: ch24/CommonURLs.hs
module Main where
import Control.Parallel.Strategies (NFData(..), rwhnf)
import Control.Monad (forM_)
import Data.List (foldl', sortBy)
import qualified Data.ByteString.Lazy.Char8 as L
import qualified Data.ByteString.Char8 as S
import qualified Data.Map as M
import Text.Regex.PCRE.Light (compile, match)
import System.Environment (getArgs)
import LineChunks (chunkedReadWith)
import MapReduce (mapReduce)
countURLs :: [L.ByteString] -> M.Map S.ByteString Int
countURLs = mapReduce rwhnf (foldl' augment M.empty . L.lines)
                      rwhnf (M.unionsWith (+))
  where augment map line =
            case match (compile pattern []) (strict line) [] of
              Just (_:url:_) -> M.insertWith' (+) url 1 map
              _ -> map
        strict  = S.concat . L.toChunks
        pattern = S.pack "\"(?:GET|POST|HEAD) ([^ ]+) HTTP/"
```

为了从日志行中挑选出 URL，我们使用在第 17 章开发的 PCRE 正则表达式库的绑定。

我们的驱动程序函数打印 10 个最受欢迎的 URL。与行计数示例一样，这个程序在两个核心上运行比在一个核心上运行大约快 1.8 倍，处理包含 110 万条目的日志文件需要 1.7 秒。

## 结论

给定一个非常适合其模型的问题，MapReduce 编程模型让我们用良好的性能和最小的额外工作在 Haskell 中编写"随意"并行程序。我们可以轻松扩展这个想法以使用其他数据源，例如文件集合或通过网络获取的数据。

在许多情况下，性能瓶颈将是数据流速度足以跟上核心处理能力的速率。例如，如果我们尝试在未缓存在内存中或从高带宽存储阵列流式传输的文件上使用刚才显示的示例程序，我们将大部分时间等待磁盘 I/O，从多核中获益。
