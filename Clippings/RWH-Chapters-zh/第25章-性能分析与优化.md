## 第二十五章：性能分析与优化

Haskell 是一种高级语言。一种真正的高级语言。我们可以整天完全在抽象中编程，在单子、函子和高态射中，远离任何特定硬件计算模型。语言规范极力避免规定任何特定求值模型。这些抽象层让我们将 Haskell 视为计算本身的符号，让我们专注于问题的本质，而不会陷入低层实现决策。我们用纯粹的思想编程。

然而，这是一本关于真实世界编程的书，在真实世界中，代码在资源有限的stock硬件上运行。我们的程序会有我们可能需要强制执行的时间和空间要求。因此，我们需要很好地了解程序数据是如何表示的，使用惰性或严格求值策略的精确后果，以及分析和控制空间和时间行为的技术。

在本章中，我们将看看 Haskell 程序员可能遇到的典型空间和时间问题，以及如何系统地分析、理解和解决它们。为此，我们将使用一系列技术：时间和空间分析、运行时统计以及关于严格和惰性求值的推理。我们还将看看编译器优化对性能的影响以及在纯粹函数式语言中变得可行的高级优化技术。让我们从一个挑战开始：在一些看似无害的代码中消除意外的内存使用。

## 分析 Haskell 程序

让我们考虑以下列表操作程序，它天计算一些大列表值的平均值。虽然只是一个程序片段（我们会强调我们正在实现的特定算法与此处的讨论无关），但它代表了我们可能在任何 Haskell 程序中找到的真实代码：通常是简洁的列表操作代码和对标准库函数的严重使用。它还展示了一些可能吸引毫无防备的程序员的常见性能麻烦点：

```haskell
-- file: ch25/A.hs
import System.Environment
import Text.Printf
main = do
    [d] <- map read `fmap` getArgs
    printf "%f\n" (mean [1..d])
mean :: [Double] -> Double
mean xs = sum xs / fromIntegral (length xs)
```

这个程序非常简单。我们导入用于访问系统环境（特别是 getArgs）的函数，以及用于格式化文本输出的 Haskell 版 printf。然后程序从命令行读取一个数字文字，用它来构建一个浮点值列表，我们通过将列表和除以长度来计算其平均值。结果打印为字符串。让我们将此源编译成本机代码（启用优化）并用 time 命令运行它来看看它的表现：

```bash
$ ghc --make -O2 A.hs
[1 of 1] Compiling Main             ( A.hs, A.o )
Linking A ...
$ time ./A 1e5
50000.5
./A 1e5  0.05s user 0.01s system 102% cpu 0.059 total
$ time ./A 1e6
500000.5
./A 1e6  0.26s user 0.04s system 99% cpu 0.298 total
$ time ./A 1e7
5000000.5
./A 1e7  63.80s user 0.62s system 99% cpu 1:04.53 total
```

它对小数字效果很好，但程序在 1000 万列表大小时确实开始吃力。仅凭这一点，我们知道有些不对劲，但不清楚正在使用什么资源。让我们来调查一下。

## 收集运行时统计

为了获取这类信息，GHC 让我们使用特殊的 +RTS 和 -RTS 标志将标志直接传递给 Haskell 运行时，用以分隔保留给运行时系统的参数。应用程序本身不会看到这些标志，因为它们会被 Haskell 运行时系统立即消耗。

特别是，我们可以要求运行时系统使用 -s 标志收集内存和垃圾收集器性能数据（以及使用 -N 控制 OS 线程数量或调整堆和栈大小）。我们还将使用运行时标志来启用不同种类的分析。Haskell 运行时接受的完整标志集记录在 GHC 用户指南中（http://www.haskell.org/ghc/docs/latest/html/users_guide/）。

所以让我们通过 +RTS -sstderr 运行启用统计报告的程序，得到这个结果：

当使用 -sstderr 时，程序的性能数字被打印到标准错误流，提供了关于我们程序在做什么的大量信息。特别是，它告诉我们花费在垃圾收集上的时间和最大实时内存使用量。结果是，为了计算 1000 万元素列表的平均值，我们的程序在堆上使用了最多 742 兆字节，并且 96.9% 的时间花在垃圾收集上！总体而言，程序运行时间中只有 3.1% 花在生产性工作上。

那么为什么我们的程序表现如此糟糕，我们能做些什么来改进它？毕竟，Haskell 是一种惰性语言——它不应该能够用常量空间处理列表吗？

## 时间分析

值得庆幸的是，GHC 附带了几个分析程序时间和空间使用的工具。特别是，我们可以使用分析支持编译程序，这会在运行时产生关于每个函数使用哪些资源的有用信息。分析分三步进行：编译支持分析的程序，启用特定分析模式运行它，并检查生成的统计信息。

要编译支持基本时间和分配分析的程序，我们使用 -prof 标志。我们还需要通过添加成本中心来告诉分析代码我们想要分析哪些函数。成本中心是程序中我们要收集其统计信息的位置。GHC 将生成代码来计算每个位置求值表达式的成本。成本中心可以使用 SCC 编译指示手动添加到任何表达式中：

```haskell
-- file: ch25/SCC.hs
mean :: [Double] -> Double
mean xs = {-# SCC "mean" #-} sum xs / fromIntegral (length xs)
```

或者，我们可以使用 -auto-all 标志编译，让编译器为我们插入所有顶层函数的成本中心。手动成本中心是对自动成本中心分析的有用补充，因为一旦确定了热点，我们就可以精确定位函数中昂贵的子表达式。

需要注意的是，在像 Haskell 这样的惰性纯语言中，无需参数的值只需要计算一次（例如，我们示例程序中的大列表），结果会被共享以供以后使用。这样的值不是程序调用图的一部分，因为它们不是在每次调用时求值，但当然我们仍然希望知道它们一次性的求值成本是多少。为了获得这些值的准确数字（称为常量应用形式 CAF），我们使用 -caf-all 标志。

编译我们的示例程序以进行分析（使用 -fforce-recomp 标志强制完整重新编译）：

```bash
$ ghc -O2 --make A.hs -prof -auto-all -caf-all -fforce-recomp
[1 of 1] Compiling Main             ( A.hs, A.o )
Linking A ...
```

我们现在可以运行这个带注释的程序，启用时间分析（我们会使用较小的输入大小，因为程序现在有额外的分析开销）：

```bash
$ time ./A  1e6 +RTS -p
Stack space overflow: current size 8388608 bytes.
Use `+RTS -Ksize' to increase it.
./A 1e6 +RTS -p  1.11s user 0.15s system 95% cpu 1.319 total
```

程序用完了栈空间！这是使用分析时需要注意的主要复杂情况：向程序添加成本中心会修改其优化方式，可能会改变其运行时行为，因为每个表达式现在都有额外的代码来跟踪求值步骤。从某种意义上说，观察正在执行的程序会修改它的执行方式。在这种情况下，我们可以简单地继续——我们使用 GHC 运行时标志 -K 为程序设置更大的栈限制（通常带有表示大小的后缀）：

```bash
$ time ./A 1e6 +RTS -p -K100M
500000.5
./A 1e6 +RTS -p -K100M  4.27s user 0.20s system 99% cpu 4.489 total
```

运行时将其分析信息转储到一个文件 A.prof（以所执行二进制文件命名），其中包含以下信息：

对于简单的性能热点识别，特别是在我们可能不太清楚时间花在哪里的较大程序中，初始时间分析可以突出显示特定有问题的模块和顶层函数，这通常足以揭示麻烦所在。一旦我们将代码缩小到有问题的部分（比如这里的示例），我们可以使用更复杂的分析工具来提取更多信息。

## 空间分析

除了基本的时间和分配统计外，GHC 还能够生成程序生命周期内堆内存使用的图形。这非常适合揭示空间泄漏，即内存被不必要地保留，导致我们在示例中看到的那种繁重的垃圾收集器活动。

构建堆分析遵循与构建正常时间分析相同的过程——即使用 -prof -auto-all -caf-all 编译。但是，当我们执行程序时，我们会要求运行时系统收集更详细的堆使用统计。我们可以通过多种方式细分堆使用信息：通过成本中心、通过模块、通过构造函数或通过数据类型。每种都有其自己的见解。堆分析 A.hs 日志到文件 A.hp，其原始数据由工具 hp2ps 处理，hp2ps 生成基于 PostScript 的图形可视化堆随时间的变化。

要从我们的程序中提取标准堆分析，我们使用 -hc 运行时标志运行它：

```bash
$ time ./A 1e6 +RTS -hc -p -K100M
500000.5
./A 1e6 +RTS -hc -p -K100M  4.15s user 0.27s system 99% cpu 4.432 total
```

堆分析日志 A.hp 被创建，内容如下：


样本在程序运行期间以固定间隔采集。我们可以使用 -iN 增加堆采样频率，其中 N 是堆大小样本之间的秒数（例如 0.01）。显然，采样越多，结果越准确，但程序运行速度越慢。我们现在可以使用 hp2ps 工具将堆分析呈现为图形：

大量工作正在分配包含双精度浮点值的列表节点。Haskell 列表是惰性的，所以完整的百万元素列表随时间构建。然而，关键的是，它在遍历时没有被释放，导致越来越大的常驻内存使用。最后，在程序运行过了一半多一点的时候，程序终于完成对列表求和并开始计算长度。如果我们看 mean 的原始片段，我们可以准确看到为什么内存被保留：

```haskell
-- file: ch25/Fragment.hs
mean :: [Double] -> Double
mean xs = sum xs / fromIntegral (length xs)
```

首先我们对列表求和，这会触发列表节点的分配，但我们无法在完成后释放列表节点，因为整个列表仍然被 length 需要。然而，一旦 sum 完成，length 开始消耗列表，垃圾收集器就可以追踪它，释放列表节点，直到完成。这两个求值阶段给出了两个显著不同的分配和释放阶段，并指出我们需要做的事情：仅遍历列表一次，一边求和一边平均。

## 控制求值

如果我们想编写只遍历列表一次的循环，我们有一些选项。例如，我们可以将循环写为对列表的 fold 或通过显式递归在列表结构上。我们坚持高级方法，先尝试 fold：

```haskell
-- file: ch25/B.hs
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    (n, s)     = foldl k (0, 0) xs
    k (n, s) x = (n+1, s+x)
```

现在，不是对列表求和并保留列表直到我们可以用 length 取其长度，而是左 fold 遍历列表，在一对中累积中间和与长度值（我们必须左 fold，因为右 fold 会使我们到达列表末尾并向后工作，这正是我们试图避免的）。

循环体是我们的 k 函数，它接受中间循环状态和当前元素，并返回长度增加 1、和增加当前元素的新状态。然而，当我们运行它时，遇到栈溢出：

```bash
$ ghc -O2 --make B.hs -fforce-recomp
$ time ./B 1e6
Stack space overflow: current size 8388608 bytes.
Use `+RTS -Ksize' to increase it.
./B 1e6  0.44s user 0.10s system 96% cpu 0.565 total
```

我们用浪费的堆换取了浪费的栈！实际上，如果我们使用 -K 运行时标志将栈大小增加到我们之前实现中的堆大小，程序会运行完成并有类似的分配数字：

```bash
$ ghc -O2 --make B.hs -prof -auto-all -caf-all -fforce-recomp
[1 of 1] Compiling Main             ( B.hs, B.o )
Linking B ...
$ time ./B 1e6 +RTS -i0.001 -hc -p -K100M
5000000.5
./B 1e6 +RTS -i0.001 -hc -p -K100M  38.70s user 0.27s system 99% cpu 39.241 total
```

生成堆分析，我们看到现在所有分配都在 mean 中。见图片 25-4。

问题是：为什么我们在遍历列表进行 fold 时不断累积越来越多的分配状态？原来，这是一个由于过度惰性而导致的经典空间泄漏。

## 严格性与尾递归

问题在于我们的左 fold，foldl，太惰性了。我们想要的是一个尾递归循环，它可以有效地实现为 goto，栈上不留任何状态。然而，在这种情况下，不是完全化简每一步的元组状态，而是创建了一个很长的 thunk 链，只在程序结束时才求值。我们在任何时候都没有要求化简循环状态，所以编译器无法推断任何严格性，必须纯粹惰性地化简值。

我们需要做的是稍微调整求值策略——惰性地展开列表，但严格地累积 fold 状态。这里的标准方法是使用 Data.List 模块中的 foldl' 替换 foldl：

```haskell
-- file: ch25/C.hs
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    (n, s)     = foldl' k (0, 0) xs
    k (n, s) x = (n+1, s+x)
```

然而，如果我们运行这个实现，我们会看到我们仍然没有完全正确：

```bash
$ ghc -O2 --make C.hs
[1 of 1] Compiling Main             ( C.hs, C.o )
Linking C ...
$ time ./C 1e6
Stack space overflow: current size 8388608 bytes.
Use `+RTS -Ksize' to increase it.
./C 1e6  0.44s user 0.13s system 94% cpu 0.601 total
```

还是不够严格！我们的循环继续在栈上累积未求值状态。这里的问题是 foldl' 只是最外层严格的：

```haskell
-- file: ch25/Foldl.hs
foldl' :: (a -> b -> a) -> a -> [b] -> a
foldl' f z xs = lgo z xs
    where lgo z []     = z
          lgo z (x:xs) = let z' = f z x in z' `seq` lgo z' xs
```

这个循环使用 `seq` 在每一步化简累积状态，但只是到循环状态的最外层构造器。也就是说，seq 将表达式化简到弱头正规形式（WHNF）。一旦到达第一个构造器，求值就在循环状态上停止。在这种情况下，最外层构造器是元组包装器 (,)，它不够深。问题仍然是元组内未求值的数值状态。

## 添加严格性

使这个函数完全严格有多种方法。例如，我们可以对自己的内部状态添加严格性提示，产生一个真正的尾递归循环：

```haskell
-- file: ch25/D.hs
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    (n, s)     = foldl' k (0, 0) xs
    k (n, s) x = n `seq` s `seq` (n+1, s+x)
```

在这个变体中，我们进入元组状态并明确告诉编译器每个状态组件应在每一步被化简。这给了我们一个最终在常量空间运行的版本：

```bash
$ ghc -O2 D.hs --make
[1 of 1] Compiling Main             ( D.hs, D.o )
Linking D ...
```

如果我们运行这个，启用分配统计，得到令人满意的结果：

### 正规形式化简

这里我们可以用其他一些方法解决严格性问题。对于深度严格性，我们可以使用 rnf 函数，它是并行策略库的一部分（以及 using），它不像 seq 那样化简到完全求值的"正规形式"（因此得名）。我们可以写一个深度 seq fold：

```haskell
-- file: ch25/E.hs
import System.Environment
import Text.Printf
import Control.Parallel.Strategies
main = do
    [d] <- map read `fmap` getArgs
    printf "%f\n" (mean [1..d])
foldl'rnf :: NFData a => (a -> b -> a) -> a -> [b] -> a
foldl'rnf f z xs = lgo z xs
    where
        lgo z []     = z
        lgo z (x:xs) = lgo z' xs
            where
                z' = f z x `using` rnf
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    (n, s)     = foldl'rnf k (0, 0) xs
    k (n, s) x = (n+1, s+x) :: (Int, Double)
```

我们改变 foldl' 的实现以使用 rnf 策略将状态化简到正规形式。这也提出了我们之前避免的一个问题：循环累加器状态的类型推断。之前，我们依赖类型默认来推断累加器中列表长度的数字、整型类型，但切换到 rnf 引入了 NFData 类约束，我们不能再依赖默认设置来设置长度类型。

### Bang Patterns

从语法上讲，添加必需严格性到过度惰性代码的最便宜方法可能是通过 bang patterns（这个名字来自"！"字符的发音"bang"），一种带有以下编译指示的语言扩展：

```haskell
-- file: ch25/F.hs
{-# LANGUAGE BangPatterns #-}
```

使用 bang patterns，我们可以对任何绑定形式提示严格性，使函数对该变量严格。就像显式类型注释可以指导类型推断一样，bang patterns 可以帮助指导严格性推断。Bangs patterns 是一种语言扩展，使用 BangPatterns 语言编译指示启用。我们现在可以简单地将循环状态重写为：

```haskell
-- file: ch25/F.hs
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    (n, s)       = foldl' k (0, 0) xs
    k (!n, !s) x = (n+1, s+x)
```

### 严格数据类型

严格数据类型是向编译器提供严格性信息的另一种有效方法。默认情况下，Haskell 数据类型是惰性的，但很容易将严格性信息添加到数据类型的字段中，然后传播到程序。例如，我们可以声明一个新的严格对类型：

```haskell
-- file: ch25/G.hs
data Pair a b = Pair !a !b
```

这创建了一个字段始终保持在 WHNF 的对类型。我们现在可以将循环重写为：

```haskell
-- file: ch25/G.hs
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    Pair n s       = foldl' k (Pair 0 0) xs
    k (Pair n s) x = Pair (n+1) (s+x)
```

这个实现再次具有相同的高效常量空间行为。此时，要从这段代码中挤出最后一点性能，我们必须更深入地研究。

## 理解 Core

除了查看运行时分析数据，确定程序确切在做什么的一个确切方法是查看编译器完成优化后的最终程序源代码，特别是在 Haskell 编译器的情况下，它们可以对代码执行非常积极的转换。GHC 使用被称为"简单函数式语言"（幽默地称为 Core）作为编译器中间表示。它本质上是 Haskell 的一个子集，增加了非装箱数据类型（原始机器类型，直接对应于 C 等语言中的原始数据类型），适合代码生成。GHC 通过转换进行优化，反复将源代码重写为越来越高效的形式。Core 表示是你的程序在翻译成低级命令式代码之前的最终函数版本。换句话说，Core 有最终决定权，如果你的目标是极致性能，值得理解。

要查看 Haskell 程序的 Core 版本，我们使用 -ddump-simpl 标志编译，或使用 ghc-core 工具，这是一个第三方实用程序，让我们可以在寻呼机中查看 Core。让我们看看我们最终使用严格数据类型的 fold 在 Core 形式中的表示：

```bash
$ ghc -O2 -ddump-simpl G.hs
```

生成了一屏文本。如果我们仔细看，会看到一个循环（这里为清晰起见稍作清理）：

```haskell
lgo :: Integer -> [Double] -> Double# -> (# Integer, Double #)
lgo = \ n xs s ->
    case xs of
      []       -> (# n, D# s #);
      (:) x ys ->
        case plusInteger n 1 of
            n' -> case x of
                D# y -> lgo n' ys (+## s y)
```

这是我们 foldl' 的最终版本，它告诉我们下一步优化的很多东西。fold 本身已被完全内联，在列表上产生一个显式递归循环。我们的严格对的循环状态完全消失了，函数现在将长度和sum累加器与列表一起作为直接参数传递。

列表元素的和用未装箱 Double# 值表示，一个保存在浮点寄存器中的原始机器 double。这是理想的，因为将和保存在堆上不会有内存流量。然而，列表的长度——因为我们没有给出显式类型注释——被推断为堆分配 Integer，它需要一个非原始的 plusInteger 来执行加法。如果算法上可以用 Int 代替，我们可以通过类型注释用 Int 替换 Integer，然后 GHC 将能够使用原始机器 Int# 作为长度。我们希望通过确保两个循环组件都是未装箱并保存在寄存器中来在时间和空间上有所改进。

循环的基本情况，其结束，产生一个未装箱对（仅在寄存器中分配的对），存储列表的最终长度和累积的和。注意返回类型是一个堆分配的 Double 值，由 D# 构造器指示，它将原始 double 值提升到堆上。同样，这对性能有影响，因为 GHC 需要检查是否有足够的堆空间在分配和从循环返回之前。

我们可以使用自定义对类型在循环中使 ghc 返回未装箱 Double# 值，这避免了最终的堆检查。此外，ghc 提供了一个优化，可以对数据类型的严格字段进行解箱，确保新对类型的字段将存储在寄存器中。使用 -funbox-strict-fields 启用此优化。

我们可以通过将对类型字段固定为 Int 和 Double 来进行这两种表示更改：

```haskell
-- file: ch25/H.hs
data Pair = Pair !Int !Double
mean :: [Double] -> Double
mean xs = s / fromIntegral n
  where
    Pair n s       = foldl' k (Pair 0 0) xs
    k (Pair n s) x = Pair (n+1) (s+x)
```

使用优化和 -funbox-strict-fields -ddump-simpl 编译，我们会在 Core 中得到一个更紧密的内部循环：

```haskell
lgo :: Int# -> Double# -> [Double] -> (# Int#, Double# #)
lgo = \ n s xs ->
    case xs of
      []       -> (# n, s #)
      (:) x ys ->
        case x of
            D# y -> lgo (+# n 1) (+## s y) ys
```

现在我们用来表示循环状态的对被表示为未装箱原始类型并将保存在寄存器中。最终版本现在只对列表节点分配堆内存，因为列表是惰性需要的。如果我们编译并运行这个调优版本，我们可以将分配和时间性能与我们原始程序进行比较：

我们可以从分析和优化过程中学到的一般规则是：

- 编译成本机代码，启用优化。
- 如有疑问，使用运行时统计和时间分析。
- 如果怀疑有分配问题，使用堆分析。
- 严格和惰性求值的仔细混合可以产生最佳结果。
- 优先对原子数据类型（Int、Double 和类似类型）使用严格字段。
- 使用更简单机器表示的数据类型（Int 优于 Integer）。

这些简单的策略足以识别和消除不正当的内存使用问题，并且在明智地使用时，可以首先阻止它们发生。

## 高级技术：融合

我们程序的最终瓶颈是惰性列表本身。虽然我们可以避免一次分配它，但循环中每次仍然有内存流量，因为我们需要下一个 cons 单元，分配到堆，对其操作，然后继续。列表类型也是多态的，所以列表的元素将表示为堆分配的 Double 值。

我们想做的是完全消除列表，只在寄存器中保存我们需要的下一个元素。也许令人惊讶的是，GHC 能够使用一种称为去森林化（deforestation）的优化将列表程序转换为一个无列表版本，这指的是涉及消除中间数据结构的一类通用优化。由于没有副作用，Haskell 编译器在重新排列代码时可以非常积极，有时会大规模重写和转换。我们将使用的特定去森林化优化是流融合（stream fusion）。

这种优化将递归列表生成和转换函数转换为非递归展开。当展开出现在折叠旁边时，它们之间的结构就被完全消除，产生一个单一的紧密循环，没有堆分配。这个优化不是默认启用的，它可能会极大地改变一段代码的复杂性，但它由多个数据结构库启用，这些库提供重写规则——自定义优化——编译器应用于库导出的函数。

我们将使用 uvector 库，它提供一套类似列表的操作，使用流融合来移除中间数据结构。重写我们的程序使用流是直接的：

```haskell
-- file: ch25/I.hs
import System.Environment
import Text.Printf
import Data.Array.Vector

main = do
    [d] <- map read `fmap` getArgs
    printf "%f\n" (mean (enumFromToFracU 1 d))
data Pair = Pair !Int !Double
mean :: UArr Double -> Double
mean xs = s / fromIntegral n
  where
    Pair n s       = foldlU k (Pair 0 0) xs
    k (Pair n s) x = Pair (n+1) (s+x)
```

从 Hackage 安装 uvector 库后，我们可以使用 -O2 -funbox-strict-fields 构建我们的程序，然后检查生成的 Core：

```haskell
fold :: Int# -> Double# -> Double# -> (# Int#, Double# #)
fold = \ n s t ->
    case >## t limit of {
      False -> fold (+# n 1) (+## s t) (+## t 1.0)
      True  -> (# n, s #)
```

这真的是最优结果！我们的列表已被完全融合，生成一个紧密循环，列表生成与累积交错，所有输入和输出变量都保存在寄存器中。运行它，我们看到性能又提升了一个数量级，运行时下降了：

```bash
$ time ./I 1e7
5000000.5
./I 1e7  0.06s user 0.00s system 72% cpu 0.083 total
```

## 调优生成的汇编

鉴于我们的 Core 现在是最优的，进一步提升这个程序的最后一步是直接看汇编。当然，在这一点上只能进行少量改进。我们可以使用 ghc-core 等工具查看生成的汇编，或使用 GHC 的 -ddump-asm 标志生成到标准输出的汇编。我们在生成汇编时可以调整的杠杆很少，但我们可以在 GHC 的 C 后端和本机代码后端之间选择。而且，如果我们选择 C 后端，则选择传递给 GCC 的优化标志。特别是对于浮点代码，有时通过 C 编译并启用特定的高性能 C 编译器优化很有用。

例如，我们可以使用 -funbox-strict-fields -fvia-C -optc-O2 从最终融合循环代码中挤出最后一点性能，这会将运行时间再减半（因为 C 编译器能够优化掉程序内部循环中一些冗余的移动指令）：

```bash
$ ghc -fforce-recomp --make -O2 -funbox-strict-fields -fvia-C -optc-O2 I.hs
[1 of 1] Compiling Main             ( I.hs, I.o )
Linking I ...
$ time ./I 1e7
5000000.5
./I 1e7  0.04s user 0.00s system 98% cpu 0.047 total
```

检查最终的 x86_64 汇编（通过 -keep-tmp-files），我们看到生成的循环只包含六条指令：

```asm
go:
  ucomisd     5(%rbx), %xmm6
  ja  .L31
  addsd       %xmm6, %xmm5
  addq        $1, %rsi
  addsd       .LC0(%rip), %xmm6
  jmp go
```

我们实际上已经将程序通过多个源级优化，一直到最终汇编。从这里没有其他地方可去了。当然，优化代码到这个级别很少有必要，而且通常只在编写低级库或优化特别重要的代码时才有意义，在已经确定所有算法选择之后。对于日常代码，选择更好的算法始终是更有效的策略，但有必要时知道我们可以优化到金属层面是很有用的。

## 结论

在本章中，我们看了一套工具和技术，你可以用它们来追踪和识别代码的有问题区域，以及各种可以大大有助于保持代码精简高效的约定。目标实际上是编程方式让你对代码在从源代码到编译器再到金属的所有层面的运作有很好的了解，并且能够在需要时专注于特定层面。

通过坚持简单规则、选择正确的数据结构并避免毫无防备的陷阱，完全可以可靠地从 Haskell 代码中获得高性能，同时能够以非常高的水平开发。结果是生产力和无情效率的完美平衡。
