# 第11章-测试与质量保证

构建真实系统意味着关心质量控制、健壮性和正确性。有了正确的质量保证机制，写得好的代码可以像精密机器一样，所有函数都按照规范精确执行任务。

边缘没有草率，最终结果可以是自解释的代码——明显正确的代码——这种代码能激发信心。

在 Haskell 中，我们有几种工具可用于构建这种精确系统。最明显的工具，也是内置于语言本身的，是表达力强的类型系统，它允许静态地强制执行复杂的不变量——使编写违反所选约束的代码成为不可能。此外，纯度和多态性鼓励了一种模块化、可重构和可测试的代码风格。这种代码就是不会出错。

测试在保持代码走在正道上起着关键作用。Haskell 中的主要测试机制是传统的单元测试（通过 HUnit 库）和更强大的后代，基于类型的"属性"测试，即 QuickCheck，一个用于 Haskell 的开源测试框架。属性测试以高级方法进行测试，鼓励用抽象不变量函数应该普遍满足的形式来处理测试规范，实际测试数据由测试库为程序员生成。这样，代码可以用数千个测试来敲打，而这些测试手工编写是不可行的，通常会发现否则找不到的微妙角落案例。

在本章中，我们将看看如何使用 QuickCheck 在代码中建立不变量，然后重新审视前几章中开发的简洁打印机，用框架测试它。我们还将看到如何通过 GHC 的代码覆盖工具 HPC 指导测试过程。

## QuickCheck：基于类型的测试

为了概述基于属性的测试是如何工作的，我们从一个简单的场景开始：你写了一个专门的排序函数并想测试它的行为。

首先，我们导入 QuickCheck 库*，以及我们需要的所有其他模块：

```haskell
-- file: ch11/QC-basics.hs
import Test.QuickCheck
import Data.List
```

以及我们想要测试的函数——一个自定义排序例程：

```haskell
-- file: ch11/QC-basics.hs
qsort :: Ord a => [a] -> [a]
qsort []     = []
qsort (x:xs) = qsort lhs ++ [x] ++ qsort rhs
    where lhs = filter  (< x) xs
          rhs = filter (>= x) xs
```

这是经典的 Haskell 排序实现：如果不讲效率的话（这不是原地排序），这是函数式编程优雅的研究。现在，我们想检查这个函数是否遵守一个好的排序应该遵循的基本规则。一个有用的开始不变量，在很多纯函数代码中出现的，是幂等性——应用函数两次与应用一次有相同的结果。对于我们的排序例程——一个稳定的排序算法——这当然应该是真的，否则事情就大错特错了！这个不变量可以简单地编码为一个属性，如下所示：

```haskell
-- file: ch11/QC-basics.hs
prop_idempotent xs = qsort (qsort xs) == qsort xs
```

我们将使用 QuickCheck 约定，用 prop_ 前缀测试属性，以便与正常代码区分开来。这个幂等性属性简单地写为一个 Haskell 函数，陈述一个对任何输入数据都必须成立的相等性。

我们可以手动检查一些简单情况以确认这是合理的：

```ghci
> prop_idempotent []
True
> prop_idempotent [1,1,1,1]
True
> prop_idempotent [1..100]
True
> prop_idempotent [1,5,2,1,2,0,9]
True
```

看起来不错。然而，手动写出输入数据是乏味的，违反了高效函数式程序员的道德准则：让机器做工作！为了自动化，QuickCheck 库带有一套所有基本 Haskell 数据类型的数据生成器。QuickCheck 使用 Arbitrary 类型类来呈现统一接口的（伪）随机数据生成，使用类型系统解析使用哪个生成器的问题。QuickCheck 通常隐藏数据生成细节；然而，我们也可以手动运行生成器来感受 QuickCheck 产生的数据分布。例如，生成一个随机布尔值列表：

```ghci
> generate 10 (System.Random.mkStdGen 2) arbitrary :: [Bool]
[False,False,False,False,False,True]
```

QuickCheck 像这样生成测试数据并将其传递给我们选择的属性，通过 quickCheck 函数。属性本身的类型决定了使用哪个数据生成器。quickCheck 然后检查对于生成的所有测试数据，属性是否成立。现在，由于我们的幂等性测试在列表元素类型上是多态的，我们需要选择一个特定类型来生成测试数据，我们将其写为属性的类型约束。要运行测试，我们只需用我们的属性函数调用 quickCheck，将其设置为所需的数据类型（否则，列表元素类型将默认为无趣的 () 类型）：

```ghci
> :type quickCheck
quickCheck :: (Testable a) => a -> IO ()
> quickCheck (prop_idempotent :: [Integer] -> Bool)
 passed 100 tests.
```

对于生成的 100 个不同的列表，我们的属性保持——很好！在开发测试时，看到每个测试生成的实际数据通常很有用。为此，我们可以用它的兄弟 verboseCheck 替换 quickCheck 来查看每个测试的（详细）输出。

现在，让我们看看我们的函数可能满足的更复杂的属性。

## 测试属性

好的库由一组正交的基元组成，它们彼此有合理的关系。我们可以使用 QuickCheck 来指定代码中函数之间的关系，帮助我们通过开发通过有用属性相互关联的函数来找到一个好的库接口。QuickCheck 这种方式充当 API"lint"工具——它提供机器支持来确保我们的库 API 有意义。

列表排序函数当然应该有许多将其与其他列表操作联系在一起的有趣属性。例如，排序列表中的第一个元素应该始终是输入列表中最小的元素。我们可能想用 Haskell 指定这个直觉，使用列表库的 minimum 函数：

```haskell
-- file: ch11/QC-basics.hs
prop_minimum xs         = head (qsort xs) == minimum xs
```

然而，测试时暴露了一个错误：

```ghci
> quickCheck (prop_minimum :: [Integer] -> Bool)
** Exception: Prelude.head: empty list
```

当排序空列表时属性失败了，因为 head 和 minimum 对空列表未定义，正如我们从它们的定义中看到的：

```haskell
-- file: ch11/minimum.hs
head       :: [a] -> a
head (x:_) = x
head []    = error "Prelude.head: empty list"
minimum    :: (Ord a) => [a] -> a
minimum [] =  error "Prelude.minimum: empty list"
minimum xs =  foldl1 min xs
```

所以这个属性只对非空列表成立。谢天谢地，QuickCheck 附带了一个完整的属性编写嵌入语言，所以我们可以用更精确的方式指定不变量，过滤掉我们不想考虑的值。对于空列表情况，我们真的想说如果列表非空，则排序结果的第一个元素是最小的。这是使用 (==>) 蕴含函数完成的，它在实际运行属性之前过滤掉无效数据：

```haskell
-- file: ch11/QC-basics.hs
prop_minimum' xs         = not (null xs) ==> head (qsort xs) == minimum xs
```

结果相当干净。通过分离空列表情况，我们现在可以确认该属性实际上成立：

```ghci
> quickCheck (prop_minimum' :: [Integer] -> Property)
 passed 100 tests.
```

注意我们必须将属性的类型从简单的 Bool 结果更改为更通用的 Property 类型（该属性现在是一个过滤非空列表的函数，在测试它们之前，而不是一个简单的布尔常量）。

现在，我们可以用其他它应该满足的不变量来完成排序函数的基本属性集：输出应该是排序的（每个元素应该小于或等于其后继）；输出应该是输入的排列（我们通过列表差函数 (\\) 实现）；排序后的最后一个元素应该是最大的元素；以及如果我们找到两个不同列表的最小元素，那应该是我们连接和排序那些列表后的第一个元素。这些属性可以表述为：

```haskell
-- file: ch11/QC-basics.hs
prop_ordered xs = ordered (qsort xs)
    where ordered []       = True
          ordered [x]      = True
          ordered (x:y:xs) = x <= y && ordered (y:xs)
prop_permutation xs = permutation xs (qsort xs)
    where permutation xs ys = null (xs \\ ys) && null (ys \\ xs)
prop_maximum xs         =
    not (null xs) ==>
        last (qsort xs) == maximum xs
prop_append xs ys       =
    not (null xs) ==>
    not (null ys) ==>
        head (qsort (xs ++ ys)) == min (minimum xs) (minimum ys)
```

## 针对模型进行测试

对某些代码获得信心的另一种技术是针对模型实现进行测试。我们可以将我们的列表排序实现与标准列表库中的参考排序函数联系起来，如果它们行为相同，我们就更有信心我们的排序做对了：

```haskell
-- file: ch11/QC-basics.hs
prop_sort_model xs      = sort xs == qsort xs
```

这种基于模型的测试非常强大。开发者通常会有一个参考实现或原型，虽然低效但是正确的。然后可以保留它并用它来确保优化的生产代码符合参考。通过构建大量这些基于模型的测试套件并定期运行它们（例如，每次提交时），我们可以廉价地确保代码的精确性。

大型 Haskell 项目通常附带的属性套件在规模上可与项目本身相媲美，在每次更改时测试数千个不变量，将代码与规范联系起来，并确保它按要求运行。

## 测试案例研究：为简洁打印机指定规格

为具有自然属性的单个函数进行测试是在 Haskell 中指导大型系统开发的基本构建块之一。我们现在来看一个更复杂的场景：采用前几章开发的简洁打印库并为其构建测试套件。

### 生成测试数据

回想一下，简洁打印机建立在 Doc 之上，这是一个代数数据类型，表示格式良好的文档：

```haskell
-- file: ch11/Prettify2.hs
data Doc = Empty
         | Char Char
         | Text String
         | Line
         | Concat Doc Doc
         | Union Doc Doc
         deriving (Show,Eq)
```

该库本身实现为一组在最终将完成的文档渲染为字符串之前构建和转换此文档类型值的函数。

QuickCheck 鼓励一种测试方法，开发者指定应该对我们可以扔给代码的任何数据成立的不变量。为了测试简洁打印库，那么，我们需要一个数据来源。为此，我们利用 QuickCheck 通过 Arbitrary 类提供的用于构建随机数据的小型组合器套件。该类提供了一个函数 arbitrary 来生成每种类型的数据。

要注意的是生成器在 Gen 环境中运行，由类型指示。这是一个简单的状态传递 monad，用于隐藏在代码中线程化的随机数生成器状态。我们将在后面的章节中彻底研究 monad，但现在只要知道 Gen 被定义为 monad，我们就可以使用 do 语法来编写访问隐式随机数源的新生成器。为了实际为我们的自定义类型编写生成器，我们使用库中定义的一组函数来引入新的随机值并将它们粘合在一起，以构建我们感兴趣的数据结构的类型。关键函数的类型是：

```haskell
-- file: ch11/Arbitrary.hs
  elements :: [a] -> Gen a
  choose   :: Random a => (a, a) -> Gen a
  oneof    :: [Gen a] -> Gen a
```

例如，elements 函数接受一个值列表并从该列表返回随机值的生成器。（我们稍后会使用 choose 和 oneof。）用它，我们可以开始为简单数据类型编写生成器。例如，如果我们为三值逻辑定义一个新的数据类型：

```haskell
-- file: ch11/Arbitrary.hs
data Ternary
    = Yes
    | No
    | Unknown
    deriving (Eq,Show)
```

我们可以通过定义一个从 Ternary 类型可能值的列表中挑选元素的函数来为 Ternary 类型编写一个 Arbitrary 实例：

```haskell
-- file: ch11/Arbitrary.hs
instance Arbitrary Ternary where
  arbitrary     = elements [Yes, No, Unknown]
```

另一种数据生成方法是为基础 Haskell 类型之一生成值，然后将那些值转换为我们实际感兴趣的类型。我们可以通过使用 choose 从 0 到 2 生成整数值，然后将其映射到三值上来编写 Ternary 实例：

```haskell
-- file: ch11/Arbitrary2.hs
instance Arbitrary Ternary where
  arbitrary     = do
      n <- choose (0, 2) :: Gen Int
      return $ case n of
                    0 -> Yes
                    1 -> No
                    _ -> Unknown
```

对于简单的和类型，这种方法效果很好，因为整数很好地映射到数据类型的构造函数上。对于产品类型（如结构和元组），我们需要分别生成产品的每个组件（对于嵌套类型递归），然后组合组件。例如，生成随机值对的随机值：

```haskell
-- file: ch11/Arbitrary.hs
instance (Arbitrary a, Arbitrary b) => Arbitrary (a, b) where
  arbitrary = do
      x <- arbitrary
      y <- arbitrary
      return (x, y)
```

所以现在让我们为 Doc 类型的所有不同变体编写一个生成器。我们将从分解问题开始，首先为每种类型生成随机构造函数，然后根据结果生成每个字段的组件。最复杂的情况是 union 和 concatenation 变体。

不过，首先我们需要为生成随机字符编写一个实例——QuickCheck 没有字符的默认实例，因为对于字符测试我们可能想使用的文本编码很丰富。我们将编写我们自己的，而且因为我们不关心文档的实际文本内容，一个简单的字母和标点符号生成器就足够了（更丰富的生成器是这个基本方法的简单扩展）：

```haskell
-- file: ch11/QC.hs
instance Arbitrary Char where
    arbitrary = elements (['A'..'Z'] ++ ['a' .. 'z'] ++ " ~!@#$%^&*()")
```

有了这个，我们现在可以通过列举构造函数并填充字段来为文档编写实例。我们选择一个随机整数来表示生成哪个文档变体，然后根据结果分派。要生成 concat 或 union 文档节点，我们只需对 arbitrary 进行递归，让类型推断确定我们指的是哪个 Arbitrary 实例：

```haskell
-- file: ch11/QC.hs
instance Arbitrary Doc where
    arbitrary = do
        n <- choose (1,6) :: Gen Int
        case n of
             1 -> return Empty
             2 -> do x <- arbitrary
                     return (Char x)
             3 -> do x <- arbitrary
                     return (Text x)
             4 -> return Line
             5 -> do x <- arbitrary
                     y <- arbitrary
                     return (Concat x y)
             6 -> do x <- arbitrary
                     y <- arbitrary
                     return (Union x y)
```

这相当简单，我们可以使用 oneof 函数进一步清理它，我们之前看到的它的类型是在列表中的不同生成器之间选择（我们也可以使用 monad 组合器 liftM，以避免命名每个生成器的中间结果）：

```haskell
-- file: ch11/QC.hs
instance Arbitrary Doc where
    arbitrary =
        oneof [ return Empty
              , liftM  Char   arbitrary
              , liftM  Text   arbitrary
              , return Line
              , liftM2 Concat arbitrary arbitrary
              , liftM2 Union  arbitrary arbitrary ]
```

后者更简洁——只是在一列生成器中选择——但它们以任何方式描述相同的数据。我们可以通过生成一个随机文档列表来检查输出是否有意义（用初始种子 2 种子伪随机生成器）：

```ghci
> generate 10 (System.Random.mkStdGen 2) arbitrary :: [Doc]
[Line,Empty,Union Empty Line,Union (Char 'R') (Concat (Union Line (Concat
(Text "i@BmSu") (Char ')'))) (Union (Concat (Concat (Concat (Text "kqV!iN")
Line) Line) Line) Line)),Char 'M',Text "YdwVLrQOQh"]
```

查看输出，我们看到了好的混合的基础案例和一些更复杂的嵌套文档。我们将在每次测试运行中生成数百个这些，所以应该做得很好。我们现在可以为我们的文档函数编写一些通用属性。

## 测试文档构建

文档上的基本函数之一是空文档常量（一个零元函数）empty，和 append 函数。它们的类型是：

```haskell
-- file: ch11/Prettify2.hs
empty :: Doc
(<>)  :: Doc -> Doc -> Doc
```

这些应该有一个好的属性：追加或前置空列表到第二个列表应该保持第二个列表不变。我们可以将这个不变量表述为一个属性：

```haskell
-- file: ch11/QC.hs
prop_empty_id x =
    empty <> x == x
  &&
    x <> empty == x
```

确认这确实是真的，我们就开始测试了：

```ghci
> quickCheck prop_empty_id
 passed 100 tests.
```

使用这个来查看生成了什么实际的测试文档（用 verboseCheck 替换 quickCheck）。如果我们看简单和复杂情况的良好混合，我们看到生成了良好的混合。如果需要，我们可以进一步细化数据生成，对生成数据的比例进行约束。

API 中的其他函数也足够简单，可以通过属性完全描述它们的行为。通过这样做，我们可以维护函数行为的外部可检查描述，以便以后的更改不会破坏这些基本不变量。

```haskell
-- file: ch11/QC.hs
prop_char c   = char c   == Char c
prop_text s   = text s   == if null s then Empty else Text s
prop_line     = line     == Line
prop_double d = double d == text (show d)
```

这些属性足以完全测试基本文档运算符返回的结构。测试库的其余部分需要更多工作。

## 使用列表作为模型

高阶函数是可重用编程的基本粘合剂，我们的简洁打印机库也不例外——一个自定义 fold 函数在内部用于实现文档连接和在文档块之间交错分隔符。

为文档定义的 fold 获取一个文档片段列表并用提供的组合函数将它们粘合在一起：

```haskell
-- file: ch11/Prettify2.hs
fold :: (Doc -> Doc -> Doc) -> [Doc] -> Doc
fold f = foldr f empty
```

我们可以轻松地为特定的 fold 实例隔离编写测试。例如，文档的水平连接可以通过在列表上编写参考实现来轻松指定：

```haskell
-- file: ch11/QC.hs
prop_hcat xs = hcat xs == glue xs
    where
        glue []     = empty
        glue (d:ds) = d <> glue ds
```

对于 punctuate 也是类似的故事，我们可以通过列表交错（来自 Data.List，intersperse 是一个取一个元素并将其交错到列表其他元素之间的函数）来模拟插入标点符号：

```haskell
-- file: ch11/QC.hs
prop_punctuate s xs = punctuate s xs == intersperse s xs
```

虽然这看起来不错，但运行它揭示了我们推理中的一个缺陷：

```ghci
>  quickCheck prop_punctuate
Falsifiable, after 6 tests:
Empty
[Line,Text "",Line]
```

简洁打印库优化掉了冗余的空文档，这是模型实现没有做的，所以我们需要增强我们的模型以匹配现实。首先，我们可以在文档列表中交错插入标点符号，然后用一个循环清理散布在其中的空文档，像这样：

```haskell
-- file: ch11/QC.hs
prop_punctuate' s xs = punctuate s xs == combine (intersperse s xs)
    where
        combine []           = []
        combine [x]          = [x]
        combine (x:Empty:ys) = x : combine ys
        combine (Empty:y:ys) = y : combine ys
        combine (x:y:ys)     = x `Concat` y : combine ys
```

在 GHCi 中运行这个，我们可以确认结果。测试框架发现我们推理中的缺陷是令人欣慰的——这正是我们想要的：

```ghci
> quickCheck prop_punctuate'
passed 100 tests.
```

## 将所有内容整合在一起

我们可以将所有这些测试放在一个文件中，并使用 QuickCheck 的驱动函数之一简单地运行它们。有几个存在，包括复杂的并行测试。然而，基本的批量驱动程序通常就足够好了。我们需要做的就是设置一些默认测试参数，然后列出我们想要测试的函数：

```haskell
-- file: ch11/Run.hs
import Prettify2
import Test.QuickCheck.Batch
options = TestOptions
      { no_of_tests         = 200
      , length_of_tests     = 1
      , debug_tests         = False }
main = do
    runTests "simple" options
        [ run prop_empty_id
        , run prop_char
        , run prop_text
        , run prop_line
        , run prop_double
        ]
    runTests "complex" options
        [ run prop_hcat
        , run prop_puncutate'
        ]
```

我们将代码构造为一个独立的独立测试脚本，实例和属性在自己的文件中，与库源分离。这是库项目的典型做法，测试与库本身分开保存，它们通过模块系统导入库。然后可以编译和执行测试脚本：

```bash
$ ghc --make Run.hs
$ ./Run
                 simple : .....                            (1000)
                complex : ..                               (400)
```

总共创建了 1,400 个单独的测试，这是令人欣慰的。我们可以轻松地增加深度，但要确切地了解代码被测试的程度，我们应该转向内置的代码覆盖工具 HPC，它可以精确地说明发生了什么。

## 使用 HPC 测量测试覆盖率

HPC（Haskell 程序覆盖率）是编译器的扩展，用于观察给定程序运行期间实际执行了哪些代码部分。这在测试上下文中很有用，因为它让我们准确观察哪些函数、分支和表达式被求值。结果是关于测试覆盖率百分比的精确知识，易于获得。HPC 附带了一个简单的实用程序来生成有用的程序覆盖率图表，使我们能够轻松放大测试套件中的薄弱环节。

要获得测试覆盖率数据，我们只需要在编译测试时在命令行添加 -fhpc 标志：

```bash
$ ghc -fhpc Run.hs --make
```

然后正常运行测试：

```bash
$ ./Run
                 simple : .....                            (1000)
                complex : ..                               (400)
```

在测试运行期间，程序的跟踪被写入当前目录中的 .tix 和 .mix 文件。之后，这些文件被命令行工具 hpc 用来显示关于发生的事情的各种统计信息。基本界面是文本的。首先，我们可以使用 hpc 的报告标志获取运行期间测试的代码摘要。我们将排除测试程序本身（使用 --exclude 标志），以便只关注简洁打印机库中的代码。在控制台中输入以下内容：

```bash
$ hpc report Run --exclude=Main --exclude=QC
  18% expressions used (30/158)
   0% boolean coverage (0/3)
        0% guards (0/3), 3 unevaluated
        100% 'if' conditions (0/0)
        100% qualifiers (0/0)
   23% alternatives used (8/34)
    0% local declarations used (0/4)
   42% top-level declarations used (9/21)
```

我们看到，在最后一行，42% 的顶级定义在测试运行期间被求值。第一次尝试还算不错。随着我们测试库中越来越多的函数，这个数字会上升。文本版本对快速摘要很有用，但要真正看到发生了什么，最好查看标记输出。要生成这个，使用标记标志代替：

```bash
$ hpc markup Run --exclude=Main --exclude=QC
```

这将为每个 Haskell 源文件生成一个 HTML 文件和一些索引文件。将 hpc_index.html 加载到浏览器中，我们可以看到代码覆盖率的一些不错的图表。

不算太差。点击进入 prettify 模块本身，我们看到程序的实际源代码。它用粗体黄色标记了未测试的代码，简单地用粗体标记了被测试的代码。

例如，我们忘记了测试 Monoid 实例，以及一些更复杂的函数。HPC 有助于保持我们的测试套件诚实。让我们为 Monoid 类型类的实例添加一个测试，这是支持追加和空元素的类型类：

```haskell
-- file: ch11/QC.hs
prop_mempty_id x =
    mempty `mappend` x == x
  &&
    x `mappend` mempty == (x :: Doc)
```

在 ghci 中运行这个属性，检查它是否正确：

```ghci
> quickCheck prop_mempty_id
 passed 100 tests.
```

我们现在可以重新编译并运行测试驱动。然而，首先删除旧的 .tix 文件很重要，否则会发生错误，因为 HPC 尝试合并来自单独运行的统计信息：

```bash
$ ghc -fhpc Run.hs --make -no-recomp
$ ./Run
 Hpc failure: inconsistent number of tick boxes
 (perhaps remove Run.tix file?)
$ rm *.tix
$ ./Run
                 simple : .....                            (1000)
                complex : ...                              (600)
```

又增加了 200 个测试到套件中，我们的覆盖率统计提高到代码库的 52%。

HPC 确保我们在测试中保持诚实，因为任何低于 100% 的覆盖率都会在刺眼的颜色中被指出。特别是，它确保程序员必须考虑错误情况、具有模糊条件的复杂分支和各种代码气味。当与饱和测试生成系统（如 QuickCheck 的）结合时，测试成为一项有益的活动和 Haskell 开发的核心部分。
