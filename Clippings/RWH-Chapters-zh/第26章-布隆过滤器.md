## 第二十六章：高级库设计：构建布隆过滤器

## 引入布隆过滤器

布隆过滤器是一种高效的集合数据结构。它只支持两个操作：插入和成员查询。与普通集合数据结构不同，布隆过滤器可能会给出错误的答案。如果我们查询一个已插入的元素是否存在，它会给出肯定的回答。如果我们查询一个未插入的元素是否存在，它可能会错误地声称该元素存在。

对于许多应用来说，低概率的假阳性是可以容忍的。例如，网络流量整形器的工作是限制批量传输（如 BitTorrent），以便交互式会话（如 ssh 会话或游戏）能够获得良好的响应时间。流量整形器可以使用布隆过滤器来确定属于特定会话的数据包是批量还是交互式的。如果它将万分之一批量数据包误认为交互式而未能限制，没有人会注意到。

布隆过滤器的吸引力在于其空间效率。如果我们想构建一个拼写检查器并拥有 50 万词的字典，集合数据结构可能占用 20 兆字节的空间。相比之下，布隆过滤器只需要大约半兆字节，代价是可能会漏掉约 1% 的拼写错误的单词。

布隆过滤器的内部结构非常简单。它由一个位数组和几个哈希函数组成。我们用 k 表示哈希函数的数量。如果我们要将一个值插入布隆过滤器，我们计算该值的 k 个哈希值，并打开位数组中对应的位。如果我们要查看一个值是否存在，我们计算 k 个哈希值并检查数组中所有对应的位是否都被打开。

要了解它是如何工作的，假设我们想将字符串 "foo" 和 "bar" 插入到一个 8 位的布隆过滤器中，并且我们有两个哈希函数：
1. 计算 "foo" 的两个哈希值，得到值 1 和 6。
2. 在位数组中设置位 1 和 6。
3. 计算 "bar" 的两个哈希值，得到值 6 和 3。
4. 在位数组中设置位 6 和 3。

这个例子应该清楚地说明为什么我们不能从布隆过滤器中删除元素："foo" 和 "bar" 都导致位 6 被设置。

假设我们现在要查询布隆过滤器以查看值 "quux" 和 "baz" 是否存在：
1. 计算 "quux" 的两个哈希值，得到值 4 和 0。
2. 检查位数组中的位 4。它没有设置，所以 "quux" 不可能存在。我们不需要检查位 0。
3. 计算 "baz" 的两个哈希值，得到值 1 和 3。
4. 检查位数组中的位 1。它被设置了，位 3 也是，所以我们就说 "baz" 存在，尽管它不存在。我们报告了一个假阳性。

有关布隆过滤器在网络中应用的研究，请参阅 Andrei Broder 和 Michael Mitzenmacher 的论文 "Network Applications of Bloom Filters: A Survey"（参见 http://www.eecs.harvard.edu/~michaelm/postscripts/im2005b.pdf）。

## 用例和包布局

并非所有布隆过滤器的用户都有相同的需求。在某些情况下，一次创建布隆过滤器并在之后仅查询它就足够了。对于其他应用，我们可能需要在创建布隆过滤器后继续更新它。为了适应这些需求，我们将用可变和不可变 API 来设计我们的库。

我们将通过将可变和不可变 API 放在不同的模块中来隔离它们：BloomFilter 用于不可变代码，BloomFilter.Mutable 用于可变代码。

此外，我们将创建几个"辅助"模块，它们不会提供公共 API 的部分，但会使内部代码更简洁。

最后，我们将要求 API 的用户提供一个能够生成元素多个哈希值的函数。这个函数的类型将是 `a -> [Word32]`。我们将使用该函数返回的所有哈希值，所以列表不能是无限的！

## 基本设计

我们用于 Haskell 布隆过滤器的数据结构是我们之前给出的简单描述的直接翻译——一个位数组和一个计算哈希的函数：

```haskell
-- file: BloomFilter/Internal.hs
module BloomFilter.Internal
    (
      Bloom(..)
    , MutBloom(..)
    ) where
import Data.Array.ST (STUArray)
import Data.Array.Unboxed (UArray)
import Data.Word (Word32)
data Bloom a = B {
      blmHash  :: (a -> [Word32])
    , blmArray :: UArray Word32 Bool
    }
```

当我们创建 Cabal 包时，不会导出这个 BloomFilter.Internal 模块。它存在只是为了让我们控制名称的可见性。我们会将 BloomFilter.Internal 导入到可变和不可变模块中，但从每个模块只重导出与该模块 API 相关的类型。

## 非装箱、提升和底值

与普通 Haskell 数组不同，UArray 包含非装箱值。

对于普通的 Haskell 类型，一个值可以是完全求值的、一个未求值的 thunk，或者是特殊值 ⊥（读作"底"）。⊥ 值是不成功计算的占位符。这样的计算可以有多种形式。它可能是一个无限循环、error 的应用，或者是特殊的 undefined 值。

能够包含 ⊥ 的类型称为被提升的。所有普通 Haskell 类型都是被提升的。实际上，这意味着我们总是可以用 `error "eek!"` 或 undefined 来代替正常表达式。

这种存储 thunk 或 ⊥ 的能力会带来性能开销：它增加了一层间接寻址。要理解为什么需要这种间接寻址，请考虑 Word32 类型。在 32 位系统上，这个类型的值是完整的 32 位宽，所以没有办法直接在小值中编码 ⊥。运行时系统必须维护和检查一些额外的数据来跟踪值是 ⊥ 还是不是。

非装箱值消除了这种间接寻址。这样做可以获得性能，但牺牲了表示 thunk 或 ⊥ 的能力。由于它比普通 Haskell 数组更紧凑，非装箱值数组是数值和位的绝佳选择。

GHC 通过将八个数组元素打包成每个字节来实现 Bool 值的 UArray，所以这种类型非常适合我们的需求。

## ST 单子

回到"修改数组元素"部分，我们提到修改不可变数组代价很高，因为它需要复制整个数组。使用 UArray 不会改变这一点，那么我们怎样才能将成本降低到可接受的水平？

在命令式语言中，我们只需就地修改数组的元素——这也将是我们在 Haskell 中的方法。

Haskell 提供了一个名为 ST 的特殊单子，它让我们能够安全地使用可变状态。与 State 单子相比，它有一些强大的附加能力：

- 我们可以将不可变数组融化成可变数组；就地修改可变数组；完成后冻结成新的不可变数组。
- 我们有能力使用可变引用。这让我们能够实现可以在构造后修改的数据结构，就像在命令式语言中一样。这种能力对于某些命令式数据结构和算法至关重要，对于这些结构和算法，目前还没有发现同样高效的纯函数式替代方案。

IO 单子也提供了这些能力。两者之间的主要区别是 ST 单子被故意设计成我们可以从中逃离回纯 Haskell 代码。我们通过 `runST` 执行函数进入 ST 单子（与大多数其他 Haskell 单子的方式相同——当然除了 IO），我们通过从 runST 返回来逃离它。

当我们应用单子的执行函数时，我们期望它能重复地行为：给定相同的函数体和参数，我们必须每次都获得相同的结果。这也适用于 runST。为了实现这种可重复性，ST 单子比 IO 单子有更多限制。我们不能读取或写入文件、创建全局变量或派生线程。事实上，虽然我们可以创建和使用可变引用和数组，但类型系统阻止它们逃逸到 runST 的调用者。可变数组必须在我们可以返回它之前被冻结成不可变数组，可变引用根本不能逃逸。

## 为受限导入设计 API

我们为使用布隆过滤器提供的公共接口值得稍加讨论：

```haskell
-- file: BloomFilter/Mutable.hs
module BloomFilter.Mutable
    (
      MutBloom
    , elem
    , notElem
    , insert
    , length
    , new
    ) where
import Control.Monad (liftM)
import Control.Monad.ST (ST)
import Data.Array.MArray (getBounds, newArray, readArray, writeArray)
import Data.Word (Word32)
import Prelude hiding (elem, length, notElem)
import BloomFilter.Internal (MutBloom(..))
```

我们导出了几个与 Prelude 导出的名称冲突的名称。这是故意的：我们期望我们模块的用户使用受限名称导入它们。这减轻了用户记忆的负担，因为他们应该已经熟悉 Prelude 的 elem、notElem 和 length 函数。

当我们使用以这种风格编写的模块时，我们通常会用单个字母前缀导入它——例如，`import qualified BloomFilter.Mutable as M`。这允许我们写 `M.length`，它保持紧凑和可读。

或者，我们可以无条件导入该模块，并在导入 Prelude 时用 `import Prelude hiding (length)` 隐藏冲突的名称。这用处不大，因为它给浏览代码的读者没有本地提示她实际上看到的不是 Prelude 的 length。

当然，在我们自己模块的头部似乎违反了这个规定：我们导入 Prelude 并隐藏它导出的一些名称。这有一个实际的原因。我们定义了一个名为 length 的函数。如果我们从模块中导出它而不先隐藏 Prelude 的 length，编译器会抱怨它无法判断是导出我们的 length 版本还是 Prelude 的。

## 创建可变布隆过滤器

我们将可变布隆过滤器的类型声明放在 BloomFilter.Internal 模块中，与不可变 Bloom 类型一起：

```haskell
-- file: BloomFilter/Internal.hs
data MutBloom s a = MB {
      mutHash :: (a -> [Word32])
    , mutArray :: STUArray s Word32 Bool
    }
```

STUArray 类型给我们提供了一个可在 ST 单子中工作的可变非装箱数组。要创建 STUArray，我们使用 newArray 函数。new 函数属于 BloomFilter.Mutable 模块：

```haskell
-- file: BloomFilter/Mutable.hs
new :: (a -> [Word32]) -> Word32 -> ST s (MutBloom s a)
new hash numBits = MB hash `liftM` newArray (0,numBits-1) False
```

STUArray 的大多数方法实际上是 MArray 类型类的实现，它定义在 Data.Array.MArray 模块中。

我们的 length 函数因两个因素而略微复杂。我们依赖于位数组对其自身边界的记录，而 MArray 实例的 getBounds 函数具有单子类型。我们还必须在答案上加一，因为数组的上界比其实际长度小一：

```haskell
-- file: BloomFilter/Mutable.hs
length :: MutBloom s a -> ST s Word32
length filt = (succ . snd) `liftM` getBounds (mutArray filt)
```

要将元素添加到布隆过滤器，我们设置哈希函数指示的所有位。我们使用 mod 函数确保所有哈希值保持在数组边界内，并将计算位数组偏移量的代码隔离在一个函数中：

```haskell
-- file: BloomFilter/Mutable.hs
insert :: MutBloom s a -> a -> ST s ()
insert filt elt = indices filt elt >>=
                  mapM_ (\bit -> writeArray (mutArray filt) bit True)
indices :: MutBloom s a -> a -> ST s [Word32]
indices filt elt = do
  modulus <- length filt
  return $ map (`mod` modulus) (mutHash filt elt)
```

成员测试并不困难。如果哈希函数指示的每个位都被设置，我们就认为元素存在于布隆过滤器中：

```haskell
-- file: BloomFilter/Mutable.hs
elem, notElem :: a -> MutBloom s a -> ST s Bool
elem elt filt = indices filt elt >>=
                allM (readArray (mutArray filt))
notElem elt filt = not `liftM` elem elt filt
```

我们需要编写一个小型的辅助函数——all 的单子版本，我们称之为 allM：

```haskell
-- file: BloomFilter/Mutable.hs
allM :: Monad m => (a -> m Bool) -> [a] -> m Bool
allM p (x:xs) = do
  ok <- p x
  if ok
    then allM p xs
    else return False
allM _ [] = return True
```

## 不可变 API

我们到不可变布隆过滤器的接口与可变 API 有相同的结构：

```haskell
-- file: ch26/BloomFilter.hs
module BloomFilter
    (
      Bloom
    , length
    , elem
    , notElem
    , fromList
    ) where
import BloomFilter.Internal
import BloomFilter.Mutable (insert, new)
import Data.Array.ST (runSTUArray)
import Data.Array.IArray ((!), bounds)
import Data.Word (Word32)
import Prelude hiding (elem, length, notElem)
length :: Bloom a -> Int
length = fromIntegral . len
len :: Bloom a -> Word32
len = succ . snd . bounds . blmArray
elem :: a -> Bloom a -> Bool
elt `elem` filt   = all test (blmHash filt elt)
  where test hash = blmArray filt ! (hash `mod` len filt)
notElem :: a -> Bloom a -> Bool
elt `notElem` filt = not (elt `elem` filt)
```

我们提供了一个通过 fromList 函数创建不可变布隆过滤器的简便方法。这对我们的用户隐藏了 ST 单子，这样他们看到的只是不可变类型：

```haskell
-- file: ch26/BloomFilter.hs
fromList :: (a -> [Word32])    -- family of hash functions to use
         -> Word32             -- number of bits in filter
         -> [a]                -- values to populate with
         -> Bloom a
fromList hash numBits values =
    B hash . runSTUArray $
      do mb <- new hash numBits
         mapM_ (insert mb) values
         return (mutArray mb)
```

这个函数的关键是 runSTUArray。我们之前提到过，为了从 ST 单子返回不可变数组，我们必须冻结可变数组。runSTUArray 函数将执行与冻结结合在一起。给定一个返回 STUArray 的动作，它使用 runST 执行该动作；冻结它返回的 STUArray；并将其作为 UArray 返回。

MArray 类型类提供了一个可以使用的 freeze 函数，但 runSTUArray 更方便且更高效。效率在于 freeze 必须将底层数据从 STUArray 复制到新的 UArray，以确保后续对 STUArray 的修改不会影响 UArray 的内容。多亏了类型系统，runSTUArray 可以保证当它使用 STUArray 创建 UArray 时，STUArray 不再可访问。因此它可以在两个数组之间共享底层内容，避免了复制。

## 创建一个友好的接口

虽然我们的不可变布隆过滤器 API 在我们创建了一个 Bloom 值之后使用起来很简单，但 fromList 函数留下了一些重要的决策未解决。我们仍然必须选择一个能够生成多个哈希值的函数，并确定布隆过滤器的容量应该有多大：

```haskell
-- file: BloomFilter/Easy.hs
easyList :: (Hashable a)
         => Double        -- false positive rate (between 0 and 1)
         -> [a]           -- values to populate the filter with
         -> Either String (B.Bloom a)
```

这是一个创建布隆过滤器的可能的"更友好"方式。它将哈希值的责任留给类型类 Hashable。它让我们根据更容易理解的参数配置布隆过滤器——即我们愿意容忍的假阳性率。并且它根据所需的假阳性率和输入列表中元素的数量为我们选择过滤器的大小。

当然，这个函数并非总是可用——例如，如果输入列表太长，它将失败。然而，它的简单性完善了我们提供的其他接口。它让我们为用户提供从完全命令式到完全声明式的创建控制范围。

## 为方便起见重导出名称

在我们模块的导出列表中，我们重导出了来自基础 BloomFilter 模块的一些名称。这允许临时用户只导入 BloomFilter.Easy 模块，并访问他们可能需要的所有类型和函数。

如果我们同时导入 BloomFilter.Easy 和 BloomFilter，你可能想知道如果尝试使用两者都导出的名称会发生什么。我们已经知道，如果无条件导入 BloomFilter 并尝试使用 length，GHC 会发出关于歧义的错误，因为 Prelude 也使名称 length 可用。

Haskell 标准要求实现能够判断几个名称是否引用相同的"东西"。例如，Bloom 类型由 BloomFilter 和 BloomFilter.Easy 导出。如果我们导入两个模块并尝试使用 Bloom，GHC 将能够看到从 BloomFilter.Easy 重导出的 Bloom 与从 BloomFilter 导出的相同，并且不会报告歧义。

## 哈希值

布隆过滤器依赖于快速、高质量的哈希函数来实现良好性能和低假阳性率。编写一个同时具有这两个属性的通用哈希函数是困难的。

对我们来说幸运的是，一个叫 Bob Jenkins 的人开发了一些恰好具有这些属性的哈希函数，他将代码放在了公共领域 http://burtleburtle.net/bob/hash/doobs.html。他的哈希函数是用 C 编写的，因此我们可以轻松使用 FFI 创建绑定。我们需要从该站点获取的特定源文件名为 lookup3.c (http://burtleburtle.net/bob/c/lookup3.c)。我们创建一个 cbits 目录并将文件下载到那里。

## 一点编辑工作

在你刚刚下载的 lookup3.c 副本的第 36 行，有一个名为 SELF_TEST 的宏。要将此源文件用作库，你必须删除这一行或将其注释掉。如果你忘记这样做，文件底部附近定义的 main 函数将覆盖你链接此库的任何 Haskell 程序的主函数。

还有一个问题：我们经常需要 7 个甚至 10 个哈希函数。我们真的不想凑齐那么多不同的函数，幸运的是我们也不需要。在大多数情况下，我们只需要两个就能凑合。Jenkins 哈希库包含两个函数 hashword2 和 hashlittle2，它们计算两个哈希值。下面是一个描述这两个函数 API 的 C 头文件。我们将其保存到 cbits/lookup3.h：

```c
/* save this file as lookup3.h */
#ifndef _lookup3_h
#define _lookup3_h
#include <stdint.h>
#include <sys/types.h>
/* only accepts uint32_t aligned arrays of uint32_t */
void hashword2(const uint32_t *key,  /* array of uint32_t */
           size_t length,         /* number of uint32_t values */
           uint32_t *pc,         /* in: seed1, out: hash1 */
           uint32_t *pb);         /* in: seed2, out: hash2 */
/* handles arbitrarily aligned arrays of bytes */
void hashlittle2(const void *key,   /* array of bytes */
         size_t length,     /* number of bytes */
         uint32_t *pc,      /* in: seed1, out: hash1 */
         uint32_t *pb);     /* in: seed2, out: hash2 */
#endif /* _lookup3_h */
```

salt 是一个扰动函数计算的哈希值的值。如果我们用两个不同的 salt 对同一值进行哈希，我们会得到两个不同的哈希。由于这些函数计算两个哈希，它们接受两个 salt。

以下是这些函数的 Haskell 绑定：

```haskell
-- file: BloomFilter/Hash.hs
{-# LANGUAGE BangPatterns, ForeignFunctionInterface #-}
module BloomFilter.Hash
    (
      Hashable(..)
    , hash
    , doubleHash
    ) where
import Data.Bits ((.&.), shiftR)
import Foreign.Marshal.Array (withArrayLen)
import Control.Monad (foldM)
import Data.Word (Word32, Word64)
import Foreign.C.Types (CSize)
import Foreign.Marshal.Utils (with)
import Foreign.Ptr (Ptr, castPtr, plusPtr)
import Foreign.Storable (Storable, peek, sizeOf)
import qualified Data.ByteString as Strict
import qualified Data.ByteString.Lazy as Lazy
import System.IO.Unsafe (unsafePerformIO)
foreign import ccall unsafe "lookup3.h hashword2" hashWord2
    :: Ptr Word32 -> CSize -> Ptr Word32 -> Ptr Word32 -> IO ()
foreign import ccall unsafe "lookup3.h hashlittle2" hashLittle2
    :: Ptr a -> CSize -> Ptr Word32 -> Ptr Word32 -> IO ()
```

我们指定可以在我们刚刚创建的 lookup3.h 头文件中找到这些函数的定义。

为了方便和高效，我们将 Jenkins 哈希函数消费的 32 位 salt 和计算的哈希值合并为一个 64 位值：

```haskell
-- file: BloomFilter/Hash.hs
hashIO :: Ptr a    -- value to hash
       -> CSize    -- number of bytes
       -> Word64   -- salt
       -> IO Word64
hashIO ptr bytes salt =
    with (fromIntegral salt) $ \sp -> do
      let p1 = castPtr sp
          p2 = castPtr sp `plusPtr` 4
      go p1 p2
      peek sp
  where go p1 p2
          | bytes .&. 3 == 0 = hashWord2 (castPtr ptr) words p1 p2
          | otherwise        = hashLittle2 ptr bytes p1 p2
        words = bytes `div` 4
```

没有显式类型来描述发生了什么，这段代码不是很明显。with 函数在 C 栈上为 salt 分配空间并将当前 salt 值存储在那里，所以 sp 是一个 Ptr Word64。指针 p1 和 p2 是 Ptr Word32；p1 指向 sp 的低位字，p2 指向高位字。这就是我们将单个 Word64 salt 分割成两个 Ptr Word32 参数的方式。

因为我们所有的数据指针都来自 Haskell 堆，我们知道它们将对齐到可以安全传递给 hashWord2（只接受 32 位对齐地址）或 hashLittle2 的地址。由于 hashWord2 是两者中更快的哈希函数，如果我们的数据是 4 字节的倍数，我们就调用它；否则调用 hashLittle2。

由于 C 哈希函数会将计算的哈希写入 p1 和 p2，我们只需要 peek 指针 sp 就能检索计算的哈希。

我们不希望这个模块的客户端被底层细节困扰，所以我们使用一个类型类来提供干净、高级的接口：

```haskell
-- file: BloomFilter/Hash.hs
class Hashable a where
    hashSalt :: Word64        -- ^ salt
             -> a             -- ^ value to hash
             -> Word64
hash :: Hashable a => a -> Word64
hash = hashSalt 0x106fc397cf62f64d3
```

我们还为这个类型类提供了许多有用的实现。要对基本类型进行哈希，我们必须编写一些样板代码：

```haskell
-- file: BloomFilter/Hash.hs
hashStorable :: Storable a => Word64 -> a -> Word64
hashStorable salt k = unsafePerformIO . with k $ \ptr ->
                      hashIO ptr (fromIntegral (sizeOf k)) salt
instance Hashable Char   where hashSalt = hashStorable
instance Hashable Int    where hashSalt = hashStorable
instance Hashable Double where hashSalt = hashStorable
```

我们可能想使用 Storable 类型类只写一条声明，如下所示：

```haskell
-- file: BloomFilter/Hash.hs
instance Storable a => Hashable a where
    hashSalt = hashStorable
```

不幸的是，Haskell 不允许我们写这种形式的实例，因为允许它们会使类型系统不可判定：它们可能导致编译器的类型检查器无限循环。这种关于不可决定类型的限制迫使我们写出单独的声明。然而，对于这样的定义不构成问题：

```haskell
-- file: BloomFilter/Hash.hs
hashList :: (Storable a) => Word64 -> [a] -> IO Word64
hashList salt xs =
    withArrayLen xs $ \len ptr ->
      hashIO ptr (fromIntegral (len * sizeOf x)) salt
  where x = head xs
instance (Storable a) => Hashable [a] where
    hashSalt salt xs = unsafePerformIO $ hashList salt xs
```

编译器将接受这个实例，这样我们就获得了哈希许多列表类型值的能力。最重要的是，由于 Char 是 Storable 的实例，我们现在可以哈希 String 值。

对于元组类型，我们利用函数组合。我们在一个组合管道的末端获取一个 salt，并使用哈希每个元组元素的结果作为下一个元素的 salt：

```haskell
-- file: BloomFilter/Hash.hs
hash2 :: (Hashable a) => a -> Word64 -> Word64
hash2 k salt = hashSalt salt k
instance (Hashable a, Hashable b) => Hashable (a,b) where
    hashSalt salt (a,b) = hash2 b . hash2 a $ salt
instance (Hashable a, Hashable b, Hashable c) => Hashable (a,b,c) where
    hashSalt salt (a,b,c) = hash2 c . hash2 b . hash2 a $ salt
```

对于 ByteString 类型，我们编写直接插入 ByteString 类型内部的特殊实例（这给了我们出色的哈希性能）：

```haskell
-- file: BloomFilter/Hash.hs
hashByteString :: Word64 -> Strict.ByteString -> IO Word64
hashByteString salt bs = Strict.useAsCStringLen bs $ \(ptr, len) ->
                         hashIO ptr (fromIntegral len) salt
instance Hashable Strict.ByteString where
    hashSalt salt bs = unsafePerformIO $ hashByteString salt bs
rechunk :: Lazy.ByteString -> [Strict.ByteString]
rechunk s
    | Lazy.null s = []
    | otherwise   = let (pre,suf) = Lazy.splitAt chunkSize s
                    in  repack pre : rechunk suf
    where repack    = Strict.concat . Lazy.toChunks
          chunkSize = 64 * 1024
instance Hashable Lazy.ByteString where
    hashSalt salt bs = unsafePerformIO $
                       foldM hashByteString salt (rechunk bs)
```

由于惰性 ByteString 表示为一连串的块，我们必须小心这些块之间的边界。字符串 "foobar" 可以有五种不同的表示方式——例如，["fo","obar"] 或 ["foob","ar"]。这对该类型的大多数用户是不可见的，但对我们来说不是，因为我们直接使用底层块。我们的 rechunk 函数确保我们传递给 C 哈希代码的块大小统一为 64 KB，这样无论原始块边界在哪里，我们都会给出一致的哈希值。

## 将两个哈希变成多个

正如我们之前提到的，我们需要远多于两个的哈希函数才能有效地使用布隆过滤器。我们可以使用一种叫做双重哈希的技术来组合 Jenkins 哈希函数计算的两个值，产生更多哈希值。结果哈希的质量足够满足我们的需求，而且比计算许多不同的哈希要便宜得多：

```haskell
-- file: BloomFilter/Hash.hs
doubleHash :: Hashable a => Int -> a -> [Word32]
doubleHash numHashes value = [h1 + h2 * i | i <- [0..num]]
    where h   = hashSalt 0x9150a946c4a8966e value
          h1  = fromIntegral (h `shiftR` 32) .&. maxBound
          h2  = fromIntegral h
          num = fromIntegral numHashes
```

## 实现简单的创建函数

在 BloomFilter.Easy 模块中，我们使用新的 doubleHash 函数来定义 easyList 函数，其类型我们之前定义过：

```haskell
-- file: BloomFilter/Easy.hs
module BloomFilter.Easy
    (
      suggestSizing
    , sizings
    , easyList
    -- re-export useful names from BloomFilter
    , B.Bloom
    , B.length
    , B.elem
    , B.notElem
    ) where
import BloomFilter.Hash (Hashable, doubleHash)
import Data.List (genericLength)
import Data.Maybe (catMaybes)
import Data.Word (Word32)
import qualified BloomFilter as B
easyList errRate values =
    case suggestSizing (genericLength values) errRate of
      Left err            -> Left err
      Right (bits,hashes) -> Right filt
        where filt = B.fromList (doubleHash hashes) bits values
```

这依赖于一个 suggestSizing 函数，该函数根据我们期望的假阳性率和我们希望过滤器包含的最大元素数量来估计过滤器大小和要计算的哈希数量的最佳组合：

```haskell
-- file: BloomFilter/Easy.hs
suggestSizing
    :: Integer       -- expected maximum capacity
    -> Double        -- desired false positive rate
    -> Either String (Word32,Int) -- (filter size, number of hashes)
suggestSizing capacity errRate
    | capacity <= 0                = Left "capacity too small"
    | errRate <= 0 || errRate >= 1 = Left "invalid error rate"
    | null saneSizes               = Left "capacity too large"
    | otherwise                    = Right (minimum saneSizes)
  where saneSizes = catMaybes . map sanitize $ sizings capacity errRate
        sanitize (bits,hashes)
          | bits > maxWord32 - 1 = Nothing
          | otherwise            = Just (ceiling bits, truncate hashes)
          where maxWord32 = fromIntegral (maxBound :: Word32)
sizings :: Integer -> Double -> [(Double, Double)]
sizings capacity errRate =
    [(((-k) * cap / log (1 - (errRate ** (1 / k)))), k) | k <- [1..50]]
  where cap = fromIntegral capacity
```

我们进行了一些相当偏执的检查。例如，sizings 函数建议数组大小和哈希计数的配对，但不验证其建议。由于我们使用 32 位哈希，我们必须过滤掉太大的建议数组大小。

在我们的 suggestSizing 函数中，我们尝试最小化位数组的大小，而不考虑哈希的数量。要了解为什么，让我们交互式地探索数组大小和哈希数量之间的关系。

我们创建了一个相当复杂的库，有四个公共模块和一个内部模块。为了将其变成我们可以轻松重新分配的包，我们创建了一个 rwh-bloomfilter.cabal 文件。

Cabal 允许我们在单个包中描述多个库。.cabal 文件以所有库通用的信息开始，后面是每个库的独立部分：

```cabal
Name:               rwh-bloomfilter
Version:            0.1
License:            BSD3
License-File:       License.txt
Category:           Data
Stability:          experimental

```

extra-source-files 指令对构建没有影响：它指示 Cabal 如果我们运行 `runhaskell Setup sdist` 创建用于重新分配的源 tarball 时捆绑一些额外的文件。

## 处理不同的构建设置

在 2007 年之前，标准 Haskell 库被组织在少数大型包中，其中最大的是 base。这个组织将许多不相关的库绑定在一起，所以 Haskell 社区将 base 包拆分成了更多模块化的库。例如，数组类型从 base 迁移到了一个名为 array 的包中。

Cabal 包需要指定它需要存在的其他包才能构建。这使得 Cabal 的命令行界面能够在需要时自动下载和构建包的依赖项。我们希望我们的代码能够与尽可能多的 GHC 版本一起工作，无论它们是否有现代的 base 和许多其他包的布局。因此，我们需要能够指定它依赖于 array 包（如果存在），否则只依赖于 base。

Cabal 提供了一个通用配置特性，我们可以使用它有选择地启用 .cabal 文件的部分。构建配置由一个布尔值标志控制。如果为 True，则使用 if flag 指令后面的文本；否则，使用关联的 else 后面的文本：

```cabal
Cabal-Version:      >= 1.2
Flag split-base
  Description: Has the base package been split up?
  Default: True
Flag bytestring-in-base
  Description: Is ByteString in the base or bytestring package?
  Default: False
```

- 配置特性是在 Cabal 1.2 版本中引入的，所以我们指定我们的包不能用旧版本构建。
- split-base 标志的含义应该不言自明。
- bytestring-in-base 标志处理更曲折的历史。当 byte string 包首次创建时，它与 GHC 6.4 捆绑在一起，与 base 包分开。在 GHC 6.6 中，它被合并到 base 包中，但在 base 包被拆分之前（在 GHC 6.8.1 发布之前）它又独立了。

这些标志通常对构建包的人是不可见的，因为 Cabal 自动处理它们。在我们解释会发生什么之前，看看我们 .cabal 文件 Library 部分的开头会有所帮助：

```cabal
Library
  if flag(bytestring-in-base)
    -- bytestring was in base-2.0 and 2.1.1
    Build-Depends: base >= 2.0 && < 2.2
  else
    -- in base 1.0 and 3.0, bytestring is a separate package
    Build-Depends: base < 2.0 || >= 3, bytestring >= 0.9
  if flag(split-base)
    Build-Depends: base >= 3.0, array
  else
    Build-Depends: base < 3.0
```

Cabal 使用标志的默认值创建一个包描述（缺少默认值假定为 True）。如果可以构建该配置（例如，因为所有需要的包版本都可用），将使用它。否则，Cabal 会尝试不同的标志组合，直到找到可以构建的配置或耗尽替代方案。

例如，如果我们从 split-base 和 bytestring-in-base 都设置为 True 开始，Cabal 将选择以下包依赖项：

```cabal
Build-Depends: base >= 2.0 && < 2.2
Build-Depends: base >= 3.0, array
```

base 包不能同时比 3.0 新且比 2.2 旧，所以 Cabal 会拒绝这个不一致的配置。对于现代版本的 GHC，在几次尝试之后，它会发现这个确实可以构建的配置：

```cabal
-- in base 1.0 and 3.0, bytestring is a separate package
Build-Depends: base < 2.0 || >= 3, bytestring >= 0.9
Build-Depends: base >= 3.0, array
```

当我们运行 `runhaskell Setup configure` 时，我们可以通过 --flag 选项手动指定标志的值，尽管在实践中我们很少需要这样做。

## 编译选项和 C 接口

继续我们的 .cabal 文件，我们填写我们库的 Haskell 端的剩余细节。如果我们启用分析构建，我们希望所有顶层函数都显示在任何分析输出中：

```cabal
GHC-Prof-Options: -auto-all
```

Other-Modules 属性列出了对库私有的 Haskell 模块。这些模块对使用此包的代码是不可见的。

当我们用 GHC 构建这个包时，Cabal 会将 GHC-Options 属性中的选项传递给编译器。

-O2 选项使 GHC 激进地优化我们的代码。没有优化编译的代码非常慢，所以我们应该始终对生产代码使用 -O2。

为了帮助自己编写更干净的代码，我们通常添加 -Wall 选项，它启用 GHC 的所有警告。如果遇到潜在问题（如重叠模式；未使用的函数参数；以及其他无数潜在的绊脚石），这将导致 GHC 发出抱怨。虽然忽略这些警告通常是安全的，但我们通常更愿意修复我们的代码以消除它们。少量额外的工作通常会产生更容易阅读和维护的代码。

当我们用 -fvia-C 编译时，GHC 会生成 C 代码并使用系统的 C 编译器来编译它，而不是像通常那样直接生成汇编语言。这会减慢编译速度，但有时 C 编译器可以进一步改进 GHC 优化的代码，所以这可能是值得的。

我们在这里包含 -fvia-C 主要是为了展示如何使用此选项编译：

```cabal
C-Sources:        cbits/lookup3.c
CC-Options:       -O3
Include-Dirs:     cbits
Includes:         lookup3.h
Install-Includes: lookup3.h
```

对于 C-Sources 属性，我们只需要列出必须编译到我们库中的文件。CC-Options 属性包含 C 编译器的选项（-O3 指定高级别的优化）。因为我们的 FFI 绑定引用了 lookup3.h 头文件，我们需要告诉 Cabal 在哪里可以找到头文件。我们还必须告诉它安装头文件（Install-Includes）；否则，当我们尝试构建客户端代码时，客户端代码将无法找到头文件。

## 使用 QuickCheck 测试

在我们关注性能之前，我们想建立我们的布隆过滤器行为正确的基准。我们可以轻松地使用 QuickCheck 来测试一些基本属性：

```haskell
-- file: examples/BloomCheck.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
module Main where
import BloomFilter.Hash (Hashable)
import Data.Word (Word8, Word32)
import System.Random (Random(..), RandomGen)
import Test.QuickCheck
import qualified BloomFilter.Easy as B
import qualified Data.ByteString as Strict
import qualified Data.ByteString.Lazy as Lazy
```

我们不会使用正常的 quickCheck 函数来测试我们的属性，因为它生成的 100 个测试输入没有提供太多覆盖率：

```haskell
-- file: examples/BloomCheck.hs
handyCheck :: Testable a => Int -> a -> IO ()
handyCheck limit = check defaultConfig {
                     configMaxTest = limit
                   , configEvery   = \_ _ -> ""
                   }
```

我们的第一个任务是确保如果我们将一个值添加到布隆过滤器，随后的成员测试将始终报告它存在，无论选择的假阳性率或输入值如何。

我们将使用 easyList 函数来创建布隆过滤器。Double 的 Random 实例生成零到一范围内的数字，所以 QuickCheck 几乎可以为我们提供任意的假阳性率。

但是，我们需要确保我们测试的假阳性率中都排除零和一。QuickCheck 给了我们两种方法：

**构造**
我们指定要生成的有效值范围。QuickCheck 为此目的提供了一个 forAll 组合器。

**消除**
当 QuickCheck 为我们生成任意值时，我们使用 (==>) 操作符过滤掉不符合我们条件的值。如果我们这样拒绝一个值，测试将显示为成功。

如果我们可以选择任一方法，优先采用构造方法。要了解为什么，假设 QuickCheck 为我们生成了 1000 个任意值，并且我们因为某种原因过滤掉了 800 个。我们看起来运行了 1000 个测试，但实际上只有 200 个做了有用的事情。

遵循这个想法，当我们生成期望的假阳性率时，我们可以消除 QuickCheck 给我们的任何零和一，但我们改为在总是有效的区间中构造值：

```haskell
-- file: examples/BloomCheck.hs
falsePositive :: Gen Double
falsePositive = choose (epsilon, 1 - epsilon)
    where epsilon = 1e-6
(=~>) :: Either a b -> (b -> Bool) -> Bool
k =~> f = either (const True) f k
prop_one_present _ elt =
    forAll falsePositive $ \errRate ->
      B.easyList errRate [elt] =~> \filt ->
        elt `B.elem` filt
```

我们的小组合符 (=~>) 让我们过滤掉 easyList 的失败。如果它失败了，测试自动通过。

## 为 ByteString 编写 Arbitrary 实例

QuickCheck 库没有为 ByteString 类型提供 Arbitrary 实例，所以我们必须编写自己的。我们不是直接创建 ByteString，而是使用 pack 函数从 [Word8] 创建一个：

```haskell
-- file: examples/BloomCheck.hs
instance Arbitrary Lazy.ByteString where
    arbitrary = Lazy.pack `fmap` arbitrary
    coarbitrary = coarbitrary . Lazy.unpack
instance Arbitrary Strict.ByteString where
    arbitrary = Strict.pack `fmap` arbitrary
    coarbitrary = coarbitrary . Strict.unpack
```

QuickCheck 同样缺少为 Data.Word 和 Data.Int 中定义的一些实例。我们至少需要为 Word8 创建一个 Arbitrary 实例：

```haskell
-- file: examples/BloomCheck.hs
instance Random Word8 where
  randomR = integralRandomR
  random = randomR (minBound, maxBound)
instance Arbitrary Word8 where
    arbitrary = choose (minBound, maxBound)
    coarbitrary = integralCoarbitrary
```

我们用几个常用函数支持这些实例，这样我们可以在为其他整数类型编写实例时重用它们：

```haskell
-- file: examples/BloomCheck.hs
integralCoarbitrary n =
    variant $ if m >= 0 then 2*m else 2*(-m) + 1
  where m = fromIntegral n
integralRandomR (a,b) g = case randomR (c,d) g of
                            (x,h) -> (fromIntegral x, h)
    where (c,d) = (fromIntegral a :: Integer,
                   fromIntegral b :: Integer)
instance Random Word32 where
  randomR = integralRandomR
  random = randomR (minBound, maxBound)
instance Arbitrary Word32 where
    arbitrary = choose (minBound, maxBound)
    coarbitrary = integralCoarbitrary
```

创建了这些 Arbitrary 实例后，我们可以尝试在 ByteString 类型上使用我们现有的属性：

```haskell

OK, passed 1000 tests.
ghci> handyCheck 1000 $ prop_all_present (undefined :: Strict.ByteString)
OK, passed 1000 tests.
```

## 建议的大小是否正确？

测试 easyList 属性的成本随着我们增加测试数量而迅速增加。我们仍然希望有一些保证 easyList 在巨大输入上表现良好。由于直接测试不现实，我们可以使用代理：suggestSizing 是否即使在极端输入下也能给出合理的数组大小和哈希数量？

这是一个稍微棘手的属性需要检查。我们需要改变所需的假阳性率和期望的容量。当我们查看 sizings 函数的某些结果时，我们看到这些值之间的关系并不容易预测。

我们可以尝试忽略复杂性：

```haskell
-- file: examples/BloomCheck.hs
prop_suggest_try1 =
  forAll falsePositive $ \errRate ->
    forAll (choose (1,maxBound :: Word32)) $ \cap ->
      case B.suggestSizing (fromIntegral cap) errRate of
        Left err -> False
        Right (bits,hashes) -> bits > 0 && bits < maxBound && hashes > 0
```

毫不奇怪，这给我们提供了一个实际上没用的测试：

```haskell
ghci> handyCheck 1000 $ prop_suggest_try1
Falsifiable, after 1 tests:
0.2723862775515961
2484762599
ghci> handyCheck 1000 $ prop_suggest_try1
Falsifiable, after 3 tests:
2.390547635799778e-2
2315209155
```

当我们将 QuickCheck 打印的反例插入 suggestSizings 时，我们可以看到这些输入被拒绝，因为它们产生的位数组太大：

```haskell
ghci> B.suggestSizing 1678125842 8.501133057303545e-3
Left "capacity too large"
```

由于我们不能轻易预测哪些组合会导致这个问题，我们必须设法消除Sizes和假阳性率然后再咬我们：

```haskell
-- file: examples/BloomCheck.hs
prop_suggest_try2 =
    forAll falsePositive $ \errRate ->
      forAll (choose (1,fromIntegral maxWord32)) $ \cap ->
        let bestSize = fst . minimum $ B.sizings cap errRate
        in bestSize < fromIntegral maxWord32 ==>
           either (const False) sane $ B.suggestSizing cap errRate
  where sane (bits,hashes) = bits > 0 && bits < maxBound && hashes > 0
        maxWord32 = maxBound :: Word32
```

如果我们用少量测试尝试，这似乎效果很好：

```haskell
ghci> handyCheck 1000 $ prop_suggest_try2
OK, passed 1000 tests.
```

在更大体量的测试上，我们过滤掉了太多组合：

```haskell
ghci> handyCheck 10000 $ prop_suggest_try2
Arguments exhausted after 2074 tests.
```

为了解决这个问题，我们尝试减少生成我们随后会拒绝的输入的可能性：

```haskell
-- file: examples/BloomCheck.hs
prop_suggestions_sane =
    forAll falsePositive $ \errRate ->
      forAll (choose (1,fromIntegral maxWord32 `div` 8)) $ \cap ->
        let size = fst . minimum $ B.sizings cap errRate
        in size < fromIntegral maxWord32 ==>
           either (const False) sane $ B.suggestSizing cap errRate
  where sane (bits,hashes) = bits > 0 && bits < maxBound && hashes > 0
        maxWord32 = maxBound :: Word32
```

最后，我们有一个看起来稳健的属性：

```haskell
ghci> handyCheck 40000 $ prop_suggestions_sane
OK, passed 40000 tests.
```

## 性能分析和调优

我们现在有了正确的基准：我们的 QuickCheck 测试通过了。当我们开始调整性能时，我们可以随时重新运行测试以确保我们没有无意中破坏任何东西。

我们的第一步是编写一个小测试应用程序，我们可以用来计时：

```haskell
-- file: examples/WordTest.hs
module Main where
import Control.Parallel.Strategies (NFData(..))
import Control.Monad (forM_, mapM_)
import qualified BloomFilter.Easy as B
import qualified Data.ByteString.Char8 as BS
import Data.Time.Clock (diffUTCTime, getCurrentTime)
import System.Environment (getArgs)
import System.Exit (exitFailure)
timed :: (NFData a) => String -> IO a -> IO a
timed desc act = do
    start <- getCurrentTime
    ret <- act
    end <- rnf ret `seq` getCurrentTime
    putStrLn $ show (diffUTCTime end start) ++ " to " ++ desc
    return ret
instance NFData BS.ByteString where
    rnf _ = ()
instance NFData (B.Bloom a) where
    rnf filt = B.length filt `seq` ()
```

我们借用了在"将算法与求值分离"部分介绍的 rnf 函数来开发一个简单的计时工具。我们的计时操作确保一个值被求值到规范形式，以便准确捕获求值的成本。

应用程序从文件内容创建布隆过滤器，将每一行视为要添加到过滤器的元素：

```haskell
-- file: examples/WordTest.hs
main = do
  args <- getArgs
  let files | null args = ["/usr/share/dict/words"]
            | otherwise = args
  forM_ files $ \file -> do
    words <- timed "read words" $
      BS.lines `fmap` BS.readFile file
    let len = length words
        errRate = 0.01
    putStrLn $ show len ++ " words"
    putStrLn $ "suggested sizings: " ++
               show (B.suggestSizing (fromIntegral len) errRate)
    filt <- timed "construct filter" $
      case B.easyList errRate words of
        Left errmsg -> do
          putStrLn $ "Error: " ++ errmsg
          exitFailure
        Right filt -> return filt
    timed "query every element" $
      mapM_ print $ filter (not . (`B.elem` filt)) words
```

我们使用 timed 来计算三个不同阶段的成本：将数据读入行并分割；填充布隆过滤器；以及查询其中的每个元素。

如果我们编译这个并运行几次，我们可以看到执行时间刚好足够有趣，而运行之间的计时变化很小。我们已经创建了一个看起来合理的微基准：

总是在调优之前——以及调优期间——进行分析！

在第一次分析运行之前，我们没有预料到 doubleHash 甚至会出现在"热"函数的前 10 名中，更不用说主导它了。如果没有这个知识，我们可能会开始调优完全无关的东西。

回想一下，doubleHash 的主体是一个简单的列表推导式：

```haskell
-- file: BloomFilter/Hash.hs
doubleHash :: Hashable a => Int -> a -> [Word32]
doubleHash numHashes value = [h1 + h2 * i | i <- [0..num]]
    where h   = hashSalt 0x9150a946c4a8966e value
          h1  = fromIntegral (h `shiftR` 32) .&. maxBound
          h2  = fromIntegral h
          num = fromIntegral numHashes
```

由于函数返回一个列表，它分配这么多内存是有道理的，但当这么简单的代码表现如此糟糕时，我们应该怀疑。

面对性能谜题，怀疑的心自然会想要检查编译器的输出。我们不需要开始摸索汇编语言转储：最好从更高层次开始。

GHC 的 -ddump-simpl 选项打印出它在执行所有高级优化后产生的代码：

```bash
$ ghc -O2 -c -ddump-simpl --make BloomFilter/Hash.hs > dump.txt
[1 of 1] Compiling BloomFilter.Hash ( BloomFilter/Hash.hs )
```

由此生成的文件大约有 1000 行长。其中大多数名称与原始 Haskell 表示略有不同。即使如此，搜索 doubleHash 会立即让我们到达函数的定义。例如，以下是我们如何从 Unix shell 正好从正确的位置开始：

```bash
$ less +/doubleHash dump.txt
```

阅读 GHC 简化器的输出可能很困难。其中有许多自动生成的名称，代码有许多模糊的注释。通过忽略我们不理解的东西，专注于看起来熟悉的部分，我们可以取得实质性进展。Core 语言与常规 Haskell 共享一些特性，特别是类型签名、let 绑定变量和 case 模式匹配。

如果我们浏览 doubleHash 的定义，我们会到达一个看起来像这样的部分：

```haskell

```

这是列表推导式主体的编译结果。看起来可能令人生畏，但我们可以逐段浏览它，发现它毕竟不那么复杂：

- `__letrec` 等同于普通的 Haskell let。
- GHC 将我们的列表推导式主体编译成一个名为 go_s1YC 的循环。
- 如果我们的 case 表达式匹配空列表，我们返回空列表。这是令人欣慰的熟悉。
- 这个模式在 Haskell 中会读作 `(y_a1DW:ys_a1DX)`。`(:)` 构造函数出现在其操作数之前，因为 Core 语言专门使用前缀表示法以简化。
- 这是 `(:)` 构造函数的应用。`@` 符号表示第一个操作数将具有 Word32 类型。
- 三个 case 表达式中的每一个都 unbox 一个 Word32 值，以获取其中的原始值。首先 unbox 的是 h1（在这里命名为 h1_s1YA），然后是 h2，然后是当前列表元素 y。
- unboxing 通过模式匹配发生：W32# 是一个构造函数，它 box 一个原始值。按照惯例，原始类型和值以及使用它们的函数总是在其名称中的某个地方包含 #。
- 在这里，我们将 W32# 构造函数应用于 Word32# 类型的值，以给出类型 Word32 的正常值。
- plusWord# 和 timesWord# 函数加法和乘法原始无符号整数。
- 这是 `(:)` 构造函数的第二个参数，其中 go_s1YC 函数递归应用自身。
- 在这里，我们应用我们的列表推导式循环函数。它的参数是表达式 `[0..n]` 的 Core 翻译。

通过阅读这段代码的 Core，我们可以看到两个有趣的行为：

- 我们正在创建一个列表，然后立即在 go_s1YC 循环中解构它。GHC 通常可以发现这种生产后立即消费的模式，并将其转换为不发生分配的循环。这种转换类被称为融合，因为生产者和消费者融合在一起。不幸的是，这里没有发生。
- 在循环体中重复 unboxing h1 和 h2 是浪费的。

为了解决这些问题，我们对 doubleHash 函数做了几个小改动：

```haskell
-- file: BloomFilter/Hash.hs
doubleHash :: Hashable a => Int -> a -> [Word32]
doubleHash numHashes value = go 0
    where go n | n == num  = []
               | otherwise = h1 + h2 * n : go (n + 1)
          !h1 = fromIntegral (h `shiftR` 32) .&. maxBound
          !h2 = fromIntegral h
          h   = hashSalt 0x9150a946c4a8966e value
          num = fromIntegral numHashes
```

我们手动将 `[0..num]` 表达式和消费它的代码融合成一个循环。我们为 h1 和 h2 添加了严格性注解。就这样。这个六行函数变成了八行。我们的更改对 Core 输出有什么影响？
