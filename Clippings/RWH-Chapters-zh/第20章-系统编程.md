## 第二十章：Haskell 系统编程

到目前为止，我们主要讨论的是高层概念。Haskell 也可以用于较低级的系统编程。使用 Haskell 在低级层面与操作系统交互是相当可能的。

在本章中，我们要尝试一些雄心勃勃的事情：一个类似 Perl 的"语言"，它是有效的 Haskell，用纯 Haskell 实现，使 shell 脚本变得容易。我们将实现管道、简单的命令调用，以及一些处理可能通常用 grep 或 sed 执行的任务的简单工具。

针对不同操作系统存在专门的模块。在本章中，我们将尽可能使用通用的操作系统无关模块。然而，我们将在大部分章节中关注 POSIX 环境。POSIX 是类 Unix 操作系统的标准，如 Linux、FreeBSD、MacOS X 或 Solaris。Windows 默认不支持 POSIX，但 Cygwin 环境为 Windows 提供了 POSIX 兼容性层。

## 目录和文件信息

System.Directory 模块包含相当多的函数，可用于从文件系统获取信息。你可以获取目录中的文件列表、重命名或删除文件、复制文件、更改当前工作目录或创建新目录。System.Directory 是可移植的，可以在 GHC 本身工作的任何平台上工作。

System.Directory 的库参考（http://www.haskell.org/ghc/docs/latest/html/libraries/base/System-Directory.html）提供了可用函数的完整列表。让我们用 ghci 演示其中的一些。这些函数大多数与 C 库调用或 shell 命令有直接的对应关系：

```ghci
ghci> :module System.Directory
ghci> setCurrentDirectory "/etc"
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package filepath-1.1.0.0 ... linking ... done.
Loading package directory-1.0.0.1 ... linking ... done.
ghci> getCurrentDirectory
"/etc"
ghci> setCurrentDirectory ".."
ghci> getCurrentDirectory
"/"
```

* 也有一个函数 system，它只接受一个字符串并通过 shell 传递它来解析。我们建议使用 rawSystem，因为 shell 会给某些字符附加特殊含义，这可能导致安全问题或意外行为。

在这里，我们看到了更改当前工作目录和从系统获取当前工作目录的命令。这些类似于 POSIX shell 中的 cd 和 pwd 命令：

```ghci
ghci> getDirectoryContents "/"
["dev",".vmware","mnt","var","etc","net","..","lib","srv","media","lib64","opt",
".ccache","bin","selinux",".","lost+found","proc",".autorelabel",".autofsck",
"sys","misc","home","tmp","boot",".bash_history","root","sbin","usr"]
```

getDirectoryContents 返回给定目录中每个项目的列表。注意，在 POSIX 系统上，这个列表通常包含特殊值"."和".." 。在处理目录内容时，你通常会想过滤掉这些，像这样：

```ghci
ghci> getDirectoryContents "/" >>= return . filter (`notElem` [".", ".."])
["dev",".vmware","mnt","var","etc","net","lib","srv","media","lib64","opt",
".ccache","bin","selinux","lost+found","proc",".autorelabel",".autofsck",
"sys","misc","home","tmp","boot",".bash_history","root","sbin","usr"]
```

关于过滤 getDirectoryContents 结果的更详细讨论，请参阅第 8 章。

filter (`notElem` [".", ".."]) 这部分令人困惑吗？这也可以写成 filter (\c -> not $ elem c [".", ".."])。在这种情况下，反引号有效地让我们将第二个参数传递给 notElem；有关反引号的更多信息，请参阅第 76 页的"中缀函数"。

你也可以查询系统关于某些目录的位置。这个查询将向底层操作系统获取信息：

```ghci
ghci> getHomeDirectory
"/home/bos"
ghci> getAppUserDataDirectory "myApp"
"/home/bos/.myApp"
ghci> getUserDocumentsDirectory
"/home/bos"
```

## 程序终止

开发者经常编写单独的程序来完成特定任务。这些单独的部分可以组合起来完成更大的任务。shell 脚本或其他程序可能执行它们。调用脚本通常需要一种方式来发现程序是否能够成功完成其任务。Haskell 会在程序因异常而中止时自动指示非成功退出。

但是，你可能需要对比这更细粒度的退出码控制。也许你需要为不同类型的错误返回不同的代码。System.Exit 模块提供了一种退出程序并将特定退出状态码返回给调用者的方法。

你可以调用 exitWith ExitSuccess 返回表示成功终止的代码（在 POSIX 系统上为 0）。或者，你可以调用类似 exitWith (ExitFailure 5) 的东西，这将返回代码 5 给调用程序。

## 日期和时间

从文件时间戳到商业交易，一切都与日期和时间有关。Haskell 提供了操作日期和时间的方法，以及从系统获取日期和时间信息的功能。

### ClockTime 和 CalendarTime

在 Haskell 中，System.Time 模块主要负责日期和时间处理。它定义了两种类型：ClockTime 和 CalendarTime。

ClockTime 是传统 POSIX epoch 的 Haskell 版本。ClockTime 表示相对于协调世界时（UTC）1970 年 1 月 1 日凌晨零点的的时间。负的 ClockTime 表示该日期之前的秒数，而正数表示之后的秒数。

ClockTime 便于计算。因为它追踪 UTC，不需要调整本地时区、夏令时或其他时间处理的特殊情况。每天正好是 (60 * 60 * 24) 即 86,400 秒，† 这使得时间间隔计算很简单。例如，你可以在长任务开始时检查 ClockTime，在结束时再次检查，简单地从结束时间减去开始时间来确定经过了多少时间。然后，如果你愿意，可以除以 3,600 并将以小时数显示经过的时间。

ClockTime 是回答以下问题的理想选择：

- 已经过去了多少时间？
- 从这个精确时刻起 14 天后的 ClockTime 将是什么？
- 文件最后修改是什么时候？
- 现在精确时间是多少？

这些是 ClockTime 的好用法，因为它们指的是精确的、无歧义的时间点。然而，ClockTime 不太容易用于以下问题：

- 今天是星期一吗？
- 明年 5 月 1 日是星期几？
- † 有些人会注意到 UTC 以不规则间隔定义闰秒。Haskell 遵循的 POSIX 标准规定，每天在其表示中正好长度为 86,400 秒，所以你不需要在执行常规计算时担心闰秒。处理闰秒的确切方式取决于系统而且复杂，但通常可以解释为有一个"长秒"。这个细节通常只在执行精确的亚秒计算时才值得关注。

- 在考虑夏令时（DST）潜在存在的情况下，我本地时区的当前时间是多少？

CalendarTime 以人类的方式存储时间：年、月、日、时、分、秒、时区和 DST 信息。这很容易转换为一个方便显示的字符串，或者回答关于本地时间的问题。

你可以随意在 ClockTime 和 CalendarTime 之间转换。Haskell 包含将 ClockTime 转换为本地时区的 CalendarTime 或表示 UTC 的 CalendarTime 的函数。

### 使用 ClockTime

ClockTime 在 System.Time 中定义如下：

```haskell
data ClockTime = TOD Integer Integer
```

第一个 Integer 表示自 epoch 以来的秒数。第二个 Integer 表示额外的皮秒数。因为 Haskell 中的 ClockTime 使用无界的 Integer 类型，它可以有效地表示仅受计算资源限制的日期范围。

让我们看一些使用 ClockTime 的方法。首先，有 getClockTime 函数，它根据系统时钟返回当前时间：

```ghci
ghci> :module System.Time
ghci> getClockTime
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Sat Aug 23 22:30:03 PDT 2008
```

如果你等一秒再运行 getClockTime，它会返回更新后的时间。注意，这个命令的输出是一个很好看的字符串，包含星期几信息。这是因为 ClockTime 的 Show 实例。让我们在更低级别看看 ClockTime：

```ghci
ghci> TOD 1000 0
Wed Dec 31 16:16:40 PST 1969
ghci> getClockTime >>= (\(TOD sec _) -> return sec)
1219555803
```

这里我们首先构造一个 ClockTime，表示 UTC 1970 年 1 月 1 日午夜后 1,000 秒的时间点。该时间点被称为 epoch。根据你的时区，这个时间点在你本地时区可能对应于 1969 年 12 月 31 日晚上。

在第二个例子中，我们从 getClockTime 返回的值中提取秒数。现在我们可以操作它，像这样：

```ghci
ghci> getClockTime >>= (\(TOD sec _) -> return (TOD (sec + 86400) 0))
Sun Aug 24 22:30:03 PDT 2008
```

这将显示从现在起正好 24 小时后你本地时区的时间，因为 24 小时有 86,400 秒。

### 使用 CalendarTime

顾名思义，CalendarTime 表示像我们在日历上那样的时间。它有年、月、日等信息字段。CalendarTime 及其相关类型定义如下：

```haskell
data CalendarTime = CalendarTime
   {ctYear :: Int,         -- 年（后格里高利历）
    ctMonth :: Month,
    ctDay :: Int,          -- 月中的日（1 到 31）
    ctHour :: Int,         -- 一天中的小时（0 到 23）
    ctMin :: Int,          -- 分钟（0 到 59）
    ctSec :: Int,          -- 秒（0 到 61，允许闰秒）
    ctPicosec :: Integer,  -- 皮秒
    ctWDay :: Day,         -- 星期几
    ctYDay :: Int,         -- 年中的日（0 到 364 或 365）
    ctTZName :: String,    -- 时区名称
    ctTZ :: Int,           -- 与 UTC 的偏移秒数
    ctIsDST :: Bool        -- 夏令时是否有效
   }
data Month = January | February | March | April | May | June
             | July | August | September | October | November | December
data Day = Sunday | Monday | Tuesday | Wednesday
          | Thursday | Friday | Saturday
```

这些结构有一些应该强调的地方：

- ctWDay、ctYDay 和 ctTZName 由创建 CalendarTime 的库函数生成，但不用于计算。如果你手工创建 CalendarTime，不必在这些字段中放入准确的值，除非你后来的计算依赖于它们。
- 所有这三种类型都是 Eq、Ord、Read 和 Show 类型类的成员。此外，Month 和 Day 被声明为 Enum 和 Bounded 类型类的成员。有关这些类型类的更多信息，请参阅第 139 页的"重要的内置类型类"。

你可以用几种方式生成 CalendarTime 值。你可以首先将 ClockTime 转换为这样的 CalendarTime：

```ghci
ghci> :module System.Time
ghci> now <- getClockTime
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Sat Aug 23 22:29:59 PDT 2008
ghci> nowCal <- toCalendarTime now
CalendarTime {ctYear = 2008, ctMonth = August, ctDay = 23, ctHour = 22,
ctMin = 29,ctSec = 59, ctPicosec = 877577000000, ctWDay = Saturday,
ctYDay = 235, ctTZName ="PDT", ctTZ = -25200, ctIsDST = True}
ghci> let nowUTC = toUTCTime now
ghci> nowCal
CalendarTime {ctYear = 2008, ctMonth = August, ctDay = 23, ctHour = 22,
ctMin = 29, ctSec = 59, ctPicosec = 877577000000, ctWDay = Saturday,
ctYDay = 235, ctTZName = "PDT", ctTZ = -25200, ctIsDST = True}
ghci> nowUTC
CalendarTime {ctYear = 2008, ctMonth = August, ctDay = 24, ctHour = 5,
ctMin = 29, ctSec = 59, ctPicosec = 877577000000, ctWDay = Sunday,
ctYDay = 236, ctTZName = "UTC", ctTZ = 0, ctIsDST = False}
```

我们使用 getClockTime 从系统时钟获取当前的 ClockTime。接下来，toCalendarTime 将 ClockTime 转换为表示本地时区时间的 CalendarTime。toUTCTime 执行类似的转换，但其结果在 UTC 时区而不是本地时区。

注意 toCalendarTime 是一个 IO 函数，但 toUTCTime 不是。原因是 toCalendarTime 根据本地配置的时区返回不同结果，但 toUTCTime 每当传递相同的源 ClockTime 时都会返回完全相同的结果。

修改 CalendarTime 值很容易：

```ghci
ghci> nowCal {ctYear = 1960}
CalendarTime {ctYear = 1960, ctMonth = August, ctDay = 23,
ctHour = 22, ctMin = 29, ctSec = 59, ctPicosec = 877577000000,
ctWDay = Saturday, ctYDay = 235, ctTZName = "PDT",
ctTZ = -25200, ctIsDST = True}
ghci> (\(TOD sec _) -> sec) (toClockTime nowCal)
1219555799
ghci> (\(TOD sec _) -> sec) (toClockTime (nowCal {ctYear = 1960}))
-295209001
```

在这个例子中，我们首先获取之前的 CalendarTime 值，简单地将其年份切换到 1960。然后，我们使用 toClockTime 将未修改的值转换为 ClockTime，然后是修改后的值，这样你可以看到差异。注意，修改后的值在转换为 ClockTime 时显示为负的秒数。这是可以预期的，因为 ClockTime 是相对于 UTC 1970 年 1 月 1 日午夜的偏移，而这个值在 1960 年。

你也可以手动创建 CalendarTime 值：

```ghci
ghci> let newCT = CalendarTime 2010 January 15 12 30 0 0 Sunday 0 "UTC" 0 False
ghci> newCT
CalendarTime {ctYear = 2010, ctMonth = January, ctDay = 15, ctHour = 12,
ctMin = 30, ctSec = 0, ctPicosec = 0, ctWDay = Sunday, ctYDay = 0,
ctTZName = "UTC", ctTZ = 0, ctIsDST = False}
ghci> (\(TOD sec _) -> sec) (toClockTime newCT)
1263558600
```

请注意，即使 2010 年 1 月 15 日不是星期日——也不是年中的第 0 天——系统仍然可以正常处理这个值。事实上，如果我们将值转换为 ClockTime，然后再转回 CalendarTime，你会发现那些字段被正确填充了：

```ghci
ghci> toUTCTime . toClockTime $ newCT
CalendarTime {ctYear = 2010, ctMonth = January, ctDay = 15, ctHour = 12,
ctMin = 30, ctSec = 0, ctPicosec = 0, ctWDay = Friday, ctYDay = 14,
ctTZName = "UTC", ctTZ = 0, ctIsDST = False}
```

### ClockTime 的 TimeDiff

因为以对人类友好的方式管理 ClockTime 值之间的差异可能很困难，System.Time 模块包含了 TimeDiff 类型。TimeDiff 可以在方便的地方用来处理这些差异。它定义如下：

```haskell
data TimeDiff = TimeDiff
   {tdYear :: Int,
    tdMonth :: Int,
    tdDay :: Int,
    tdHour :: Int,
    tdMin :: Int,
    tdSec :: Int,
    tdPicosec :: Integer}
```

像 diffClockTimes 和 addToClockTime 这样的函数接受一个 ClockTime 和一个 TimeDiff，并通过转换为 UTC 的 CalendarTime、应用差异、再转换回 ClockTime 来内部处理计算。

让我们看看它是如何工作的：

```ghci
ghci> :module System.Time
ghci> let feb5 = toClockTime $ CalendarTime 2008 February 5 0 0 0 0 Sunday 0
"UTC" 0 False
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
ghci> feb5
Mon Feb  4 16:00:00 PST 2008
ghci> addToClockTime (TimeDiff 0 1 0 0 0 0 0) feb5
Tue Mar  4 16:00:00 PST 2008
ghci> toUTCTime $ addToClockTime (TimeDiff 0 1 0 0 0 0 0) feb5
CalendarTime {ctYear = 2008, ctMonth = March, ctDay = 5, ctHour = 0,
ctMin = 0, ctSec = 0, ctPicosec = 0, ctWDay = Wednesday, ctYDay = 64,
ctTZName = "UTC", ctTZ = 0, ctIsDST = False}
ghci> let jan30 = toClockTime $ CalendarTime 2009 January 30 0 0 0 0
Sunday 0 "UTC" 0 False
ghci> jan30
Thu Jan 29 16:00:00 PST 2009
ghci> addToClockTime (TimeDiff 0 1 0 0 0 0 0) jan30
Sun Mar  1 16:00:00 PST 2009
ghci> toUTCTime $ addToClockTime (TimeDiff 0 1 0 0 0 0 0) jan30
CalendarTime {ctYear = 2009, ctMonth = March, ctDay = 2, ctHour = 0, ctMin = 0,
ctSec = 0, ctPicosec = 0, ctWDay = Monday, ctYDay = 60, ctTZName = "UTC", ctTZ =
0, ctIsDST = False}
ghci> diffClockTimes jan30 feb5
TimeDiff {tdYear = 0, tdMonth = 0, tdDay = 0, tdHour = 0, tdMin = 0, tdSec = 31104000,
tdPicosec = 0}
ghci> normalizeTimeDiff $ diffClockTimes jan30 feb5
TimeDiff {tdYear = 0, tdMonth = 12, tdDay = 0, tdHour = 0, tdMin = 0, tdSec = 0,
tdPicosec = 0}
```

我们首先生成一个表示 UTC 2008 年 2 月 5 日午夜的 ClockTime。注意，除非你的时区与 UTC 相同，否则当这个时间显示在屏幕上时，它可能显示为 2 月 4 日晚上，因为它被格式化为你本地时区。

接下来，我们通过调用 addToClockTime 向它加一个月。2008 年是闰年，但系统正确处理了这个问题，我们得到了 3 月份相同日期和时间的结果。使用 toUTCTime，我们可以在原始 UTC 时区看到这对它的影响。

对于第二个实验，我们设置了一个表示 UTC 2009 年 1 月 30 日午夜的时间。2009 年不是闰年，所以当我们试图给它加一个月时可能会想知道会发生什么。我们可以看到，由于 2009 年不存在 2 月 29 日或 30 日，我们最终得到了 3 月 2 日。

最后，我们可以看到 diffClockTimes 如何将两个 ClockTime 值转换为 TimeDiff，尽管只有秒和皮秒被填充。normalizeTimeDiff 函数获取这样的 TimeDiff 并将其重新格式化为人类期望看到的形式。

### 文件修改时间

许多程序需要找出特定文件最后修改的时间。诸如 ls 或图形文件管理器之类的程序通常显示文件的修改时间。System.Directory 模块包含一个跨平台的 getModificationTime 函数。它接受一个文件名并返回一个表示文件最后修改时间的 ClockTime。例如：

```ghci
ghci> :module System.Directory
ghci> getModificationTime "/etc/passwd"
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package filepath-1.1.0.0 ... linking ... done.
Loading package directory-1.0.0.1 ... linking ... done.
Mon Jul 14 04:06:29 PDT 2008
```

POSIX 平台不仅维护修改时间（称为 mtime），还维护最后读取或写入访问的时间（atime）和最后状态更改的时间（ctime）。由于此信息是 POSIX 特定的，跨平台的 System.Directory 模块不提供对它的访问。相反，你需要使用 System.Posix.Files 中的函数。以下是这样做的示例函数：

```haskell
-- file: ch20/posixtime.hs
-- posixtime.hs
import System.Posix.Files
import System.Time
import System.Posix.Types
-- | 给定一个路径，返回 (atime, mtime, ctime)
getTimes :: FilePath -> IO (ClockTime, ClockTime, ClockTime)
getTimes fp =
    do stat <- getFileStatus fp
       return (toct (accessTime stat),
               toct (modificationTime stat),
               toct (statusChangeTime stat))
-- | 将 EpochTime 转换为 ClockTime
toct :: EpochTime -> ClockTime
toct et =
    TOD (truncate (toRational et)) 0
```

注意对 getFileStatus 的调用。那个调用直接映射到 C 函数 stat()。它的返回值存储了大量信息，包括文件类型、权限、所有者、组以及我们感兴趣的三个时间值。System.Posix.Files 提供了各种函数，如 accessTime，从 getFileStatus 返回的不透明 FileStatus 类型中提取我们感兴趣的信息。

诸如 accessTime 之类的函数返回以 POSIX 特定类型 EpochTime 表示的数据，它使用 toct 函数转换为 ClockTime。

System.Posix.Files 还提供了 setFileTimes 函数来设置文件的 atime 和 mtime。‡

## 扩展示例：管道

我们已经看到了如何调用外部程序。有时我们需要比那更多的控制。也许我们需要从那些程序获取输出、提供输入，甚至将多个外部程序链接在一起。管道可以帮助满足所有这些需求。管道在 shell 脚本中经常使用。当你在 shell 中设置管道时，你会运行多个程序。第一个程序的输出被发送到第二个程序的输入。它的输出被作为输入发送到第三个，依此类推。最后一个程序的输出通常发送到终端，或者可以发送到文件。以下是说明管道的 POSIX shell 示例会话：

```bash
$ ls /etc | grep 'm.*ap' | tr a-z A-Z
IDMAPD.CONF
MAILCAP
MAILCAP.ORDER
MEDIAPRM
TERMCAP
```

这个命令运行三个程序，在它们之间管道传输数据。它从 ls /etc 开始，输出 /etc 中所有文件或目录的列表。ls 的输出作为输入发送到 grep。我们给 grep 一个正则表达式，它将导致只输出以'm'开头然后在行中某处包含"ap"的行。最后，结果被发送到 tr。我们给 tr 选项将所有内容转换为大写。tr 的输出没有设置在特定位置，所以显示在屏幕上。

在这种情况下，shell 处理在程序之间设置所有管道。通过在 Haskell 中使用一些 POSIX 工具，我们可以完成相同的事情。

‡ 在 POSIX 系统上通常不可能设置 ctime。

## 使用管道进行重定向

POSIX 定义了一个创建管道的函数。这个函数返回两个文件描述符（FD），它们在概念上类似于 Haskell 的 Handle。一个 FD 是管道的读端，另一个是写端。任何写入写端的东西都可以通过读端读取。数据被"塞进管道"。在 Haskell 中，你调用 createPipe 来访问这个接口。

拥有管道是能够在外部程序之间传输数据的第一步。我们还必须能够将程序的输出重定向到管道，并将另一个程序的输入重定向自管道。Haskell 函数 dupTo 实现了这一点。它接受一个 FD 并在另一个 FD 编号处制作一个副本。标准输入、标准输出和标准错误的 POSIX FD 分别具有预定义的 FD 编号 0、1 和 2。通过将管道的端点重新编号为这些数字之一，我们可以有效地使程序的重定向其输入或输出。

然而，还有另一块拼图。我们不能在调用 rawSystem 之前使用 dupTo，因为这会扰乱我们主 Haskell 进程的 标准输入或输出。此外，rawSystem 会阻塞直到被调用的程序执行完毕，这让我们无法启动多个并行运行的进程。为了实现这一点，我们必须使用 forkProcess。这是一个非常特殊的函数。它实际上复制了当前运行的程序，我们最终会得到两个同时运行的程序副本。Haskell 的 forkProcess 函数接受一个要在新进程（称为子进程）中执行的函数。我们让那个函数调用 dupTo。在完成那之后，它调用 executeFile 来实际调用命令。这也是一个特殊的函数：如果一切顺利，它永远不会返回。那是因为 executeFile 用不同的程序替换了运行中的进程。最终，原始 Haskell 进程将调用 getProcessStatus 来等待子进程终止并了解它们的退出码。

每当你在 POSIX 系统上运行命令时，无论是刚刚在命令行中输入 ls 还是使用 Haskell 中的 rawSystem，在底层，forkProcess、executeFile 和 getProcessStatus（或者它们的 C 等价物）总是被使用。为了设置管道，我们复制系统用来启动程序的进程，并在过程中添加一些涉及管道和重定向的步骤。

还有一些其他的清理事情我们必须小心。当你调用 forkProcess 时，你的程序的几乎所有东西都会被克隆。§ 这包括打开文件描述符（句柄）的集合。程序通过检查文件结束指示器来检测何时完成从管道接收输入。当管道写端的进程关闭管道时，管道读端的进程将收到文件结束指示。然而，如果写文件描述符存在于多个进程中，文件结束指示器不会发送，直到所有进程都关闭了那个特定的 FD。因此，我们必须跟踪哪些 FD 是打开的，以便我们可以在子进程中关闭它们。我们还必须尽快在父进程中关闭管道的子端。

以下是我们用 Haskell 实现管道系统的初步实现：

```haskell
-- file: ch20/RunProcessSimple.hs
{-# OPTIONS_GHC -fglasgow-exts #-}
-- RunProcessSimple.hs
module RunProcessSimple where
import System.Process
import Control.Concurrent
import Control.Concurrent.MVar
import System.IO
import System.Exit
import Text.Regex
import System.Posix.Process
import System.Posix.IO
import System.Posix.Types
{- | 运行外部命令的类型。元组的第一部分是程序名称。
列表表示要传递给命令的命令行参数。 -}
type SysCommand = (String, [String])
{- | 运行任何命令的结果 -}
data CommandResult = CommandResult {
    cmdOutput :: IO String,              -- ^ 产生输出的 IO 操作
    getExitStatus :: IO ProcessStatus    -- ^ 产生退出结果的 IO 操作
    }
{- | 处理在客户端始终关闭的 FD 全局列表的类型 -}
type CloseFDs = MVar [Fd]
{- | 代表任何可运行命令的类 -}
class CommandLike a where
    {- | 给定命令和一个表示输入的 String，
         调用命令。返回一个表示命令输出的 String。 -}
    invoke :: a -> CloseFDs -> String -> IO CommandResult
-- 支持运行系统命令
instance CommandLike SysCommand where
    invoke (cmd, args) closefds input =
        do -- 创建两个管道：一个处理 stdin，另一个
           -- 处理 stdout。我们在这个程序中不重定向 stderr。
           (stdinread, stdinwrite) <- createPipe
           (stdoutread, stdoutwrite) <- createPipe
           -- 我们将父 FD 添加到此列表，因为我们始终需要在
           -- 客户端关闭它们。
           addCloseFDs closefds [stdinwrite, stdoutread]
           -- 现在，获取关闭的 FD 列表并 fork 子进程。
           childPID <- withMVar closefds (\fds ->
                          forkProcess (child fds stdinread stdoutwrite))
           -- 现在，在父端，关闭客户端侧的 FD。
           closeFd stdinread
           closeFd stdoutwrite
           -- 将输入写入命令。
           stdinhdl <- fdToHandle stdinwrite
           forkIO $ do hPutStr stdinhdl input
                       hClose stdinhdl
           -- 准备从命令接收输出
           stdouthdl <- fdToHandle stdoutread
           -- 设置当准备好等待子进程退出时调用的函数。
           let waitfunc =
                do status <- getProcessStatus True False childPID
                   case status of
                       Nothing -> fail $ "Error: Nothing from getProcessStatus"
                       Just ps -> do removeCloseFDs closefds
                                         [stdinwrite, stdoutread]
                                    return ps
           return $ CommandResult {cmdOutput = hGetContents stdouthdl,
                                   getExitStatus = waitfunc}
        -- 定义子进程中发生的情况
        where child closefds stdinread stdoutwrite =
                do -- 将我们的管道复制到常规 stdin/stdout FD
                   dupTo stdinread stdInput
                   dupTo stdoutwrite stdOutput
                   -- 现在关闭原始管道 FD
                   closeFd stdinread
                   closeFd stdoutwrite
                   -- 关闭我们从父进程继承的所有打开的 FD
                   mapM_ (\fd -> catch (closeFd fd) (\_ -> return ())) closefds
                   -- 启动程序
                   executeFile cmd True args Nothing
-- 将 FD 添加到子进程 fork 后必须关闭的列表
addCloseFDs :: CloseFDs -> [Fd] -> IO ()
addCloseFDs closefds newfds =
    modifyMVar_ closefds (\oldfds -> return $ oldfds ++ newfds)
-- 从列表中移除 FD
removeCloseFDs :: CloseFDs -> [Fd] -> IO ()
removeCloseFDs closefds removethem =
    modifyMVar_ closefds (\fdlist -> return $ procfdlist fdlist removethem)
    where
    procfdlist fdlist [] = fdlist
    procfdlist fdlist (x:xs) = procfdlist (removefd fdlist x) xs
    -- 我们只想移除任何给定 fd 的第一个出现
    removefd [] _ = []
    removefd (x:xs) fd
        | fd == x = xs
        | otherwise = x : removefd xs fd
{- | 代表管道的类型。'PipeCommand' 由源和目标两部分组成，
它们都必须 是 'CommandLike' 的实例。 -}
data (CommandLike src, CommandLike dest) =>
     PipeCommand src dest = PipeCommand src dest
{- | 用于创建 'PipeCommand' 的便捷函数。 -}
(-|-) :: (CommandLike a, CommandLike b) => a -> b -> PipeCommand a b
(-|-) = PipeCommand
{- | 使 'PipeCommand' 可作为命令运行 -}
instance (CommandLike a, CommandLike b) =>
         CommandLike (PipeCommand a b) where
    invoke (PipeCommand src dest) closefds input =
        do res1 <- invoke src closefds input
           output1 <- cmdOutput res1
           res2 <- invoke dest closefds output1
           return $ CommandResult (cmdOutput res2) (getEC res1 res2)
{- | 给定两个 'CommandResult' 项，评估两个的退出码，
然后返回一个"组合"退出码。如果两者都成功退出，则是 ExitSuccess。
否则，它反映遇到的第一个错误。 -}
getEC :: CommandResult -> CommandResult -> IO ProcessStatus
getEC src dest =
    do sec <- getExitStatus src
       dec <- getExitStatus dest
       case sec of
            Exited ExitSuccess -> return dec
            x -> return x
{- | 执行一个 'CommandLike'。 -}
runIO :: CommandLike a => a -> IO ()
runIO cmd =
    do -- 初始化我们的 closefds 列表
       closefds <- newMVar []
       -- 调用命令
       res <- invoke cmd closefds []
       -- 处理其输出
       output <- cmdOutput res
       putStr output
       -- 等待终止并获取退出状态
       ec <- getExitStatus res
       case ec of
            Exited ExitSuccess -> return ()
            x -> fail $ "Exited: " ++ show x
```

让我们看看这个程序。第一行有一个特殊的 OPTIONS_GHC 子句。这与传递给 ghc 或 ghci 的 -fglasgow-exts 相同。我们正在使用一个 GHC 扩展，允许我们使用 (String, [String]) 类型作为类型类的实例。‖ 把它放在源文件中意味着我们不必每次使用这个模块时都记住指定它。

在 import 行之后，我们定义了一些类型。首先，我们将 type SysCommand = (String, [String]) 定义为别名。这是一个要由系统执行的命令的类型。我们在使用上述示例执行中的每个命令时都使用了这种类型的数据。CommandResult 类型表示执行给定命令的结果，而 CloseFDs 类型表示在 fork 新子进程时我们必须关闭的 FD 列表。

接下来，我们定义一个名为 CommandLike 的类，它将用于运行"东西"，其中"东西"可能是独立程序、在两个或多个程序之间设置的管道，或者将来甚至是纯 Haskell 函数。要成为这个类的成员，只需要为给定类型存在一个函数 invoke。这将让我们使用 runIO 来启动独立命令或管道。这对于定义管道也很有用，因为我们可能在给定命令的一侧或两侧有一整堆命令。

我们的管道基础设施将使用字符串作为将数据从一个进程发送到另一个进程的方式。我们可以在读取数据时利用 Haskell 对惰性读取的支持（通过 hGetContents），并使用 forkIO 让写入在后台发生。这会工作得很好，尽管不如直接连接两个进程的端点那样快。# 但它使实现相当简单。然而，我们需要注意不要做任何需要缓冲整个 String 的事情，让 Haskell 的惰性来完成其余的工作。

接下来，我们为 SysCommand 定义 CommandLike 的一个实例。我们创建两个管道：一个用于新进程的标准输入，另一个用于其标准输出。这创建了四个端点，因此也有四个文件描述符。我们将父文件描述符添加到必须在所有子进程中关闭的列表中。这些将是子进程标准输入的写端和子进程标准输出的读端。接下来，我们 fork 子进程。在父进程中，我们可以关闭对应于子进程的 文件描述符。我们不能在 fork 之前执行此操作，因为它们对子进程不可用。我们为 stdinwrite 文件描述符获取一个句柄，并通过 forkIO 启动一个线程来向它写入输入数据。然后我们定义 waitfunc，这是调用者在准备好等待被调用进程终止时要调用的操作。

同时，子进程使用 dupTo，关闭它不需要的文件描述符，并执行命令。

接下来，我们定义一些实用函数来管理文件描述符列表。之后，我们定义帮助设置管道的工具。首先，我们定义一个新的 PipeCommand 类型，它有源和目标。源和目标都必须是 CommandLike 的成员。我们还定义了 -|- 便捷操作符。然后，我们让 PipeCommand 成为 CommandLike 的一个实例。它的 invoke 实现用给定的输入启动第一个命令，获取其输出，并将该输出传递给第二个命令的调用。然后它返回第二个命令的输出，并使 getExitStatus 函数等待并检查两个命令的退出状态。

我们通过定义 runIO 来结束。这个函数建立必须在客户端关闭的 FD 列表，启动命令，显示其输出，并检查其退出状态。

## 更好的管道

我们之前的例子解决了让我们设置类似 shell 的管道的基本需求。尽管如此，还有一些其他功能会很不错：

- 支持更多类似 shell 的语法
- 让人们将数据管道到外部程序或常规 Haskell 函数的能力，自由混合搭配两者
- 以 Haskell 程序可以方便使用的方式返回最终输出和退出码的能力

幸运的是，我们已经将大部分必要的部分准备好了。我们只需要添加几个更多的 CommandLike 实例来支持这些功能，以及几个类似于 runIO 的更多函数。以下是一个实现所有这些功能的修订示例：

```haskell
       case sec of
            Exited ExitSuccess -> return dec
            x -> return x
{- | 从 'run' 获取数据的不同方式。
 * IO () 运行，错误时抛出异常，并将 stdout 发送到 stdout。
 * IO String 运行，错误时抛出异常，将 stdout 读入缓冲区，并作为字符串返回。
 * IO [String] 与 IO String 相同，但将结果作为行返回。
 * IO ProcessStatus 运行并返回带有退出信息的 ProcessStatus。
   stdout 被发送到 stdout。不会抛出异常。
 * IO (String, ProcessStatus) 类似于 IO ProcessStatus，
   但还包括管道中最后一个有错误的命令的描述
   （如果没有错误，则是最后一个命令）。
 * IO Int 直接返回程序的退出码。如果信号导致命令被回收，返回 128 + SIGNUM。
 * IO Bool 如果程序正常退出（退出码 0，未被信号停止）返回 True，否则返回 False。
-}
class RunResult a where
    {- | 运行一个命令（或命令管道），结果以多种不同方式呈现。 -}
    run :: (CommandLike b) => b -> a
-- | 供 'RunResult' 实例使用的实用函数
setUpCommand :: CommandLike a => a -> IO CommandResult
setUpCommand cmd =
    do -- 初始化我们的 closefds 列表
       closefds <- newMVar []
       -- 调用命令
       invoke cmd closefds []
instance RunResult (IO ()) where
    run cmd = run cmd >>= checkResult
instance RunResult (IO ProcessStatus) where
    run cmd =
        do res <- setUpCommand cmd
           -- 处理其输出
           output <- cmdOutput res
           putStr output
           getExitStatus res

instance RunResult (IO Int) where
    run cmd = do rc <- run cmd
                 case rc of
                   Exited (ExitSuccess) -> return 0
                   Exited (ExitFailure x) -> return x
                   Terminated x -> return (128 + (fromIntegral x))
                   Stopped x -> return (128 + (fromIntegral x))
instance RunResult (IO Bool) where
    run cmd = do rc <- run cmd
                 return ((rc::Int) == 0)
instance RunResult (IO [String]) where
    run cmd = do r <- run cmd
                 return (lines r)
instance RunResult (IO String) where
    run cmd =
        do res <- setUpCommand cmd
           output <- cmdOutput res
           -- 强制输出被缓冲
           evaluate (length output)
           ec <- getExitStatus res
           checkResult ec
           return output
checkResult :: ProcessStatus -> IO ()
checkResult ps =
    case ps of
         Exited (ExitSuccess) -> return ()
         x -> fail (show x)
{- | 一个便捷函数。仅引用返回 @IO ()@ 的 'run' 版本。
这可以防止当你不在乎 'run' 的结果时，
不得不一直强制转换到它。 -}
runIO :: CommandLike a => a -> IO ()
runIO = run
------------------------------------------------------------
-- Utility Functions
------------------------------------------------------------
cd :: FilePath -> IO ()
cd = setCurrentDirectory

{- | 获取一个字符串并将其作为标准输出发送。
此函数的输入永远不会被读取。 -}
echo :: String -> String -> String
echo inp _ = inp

-- | 在行中搜索正则表达式。返回匹配的那些。
grep :: String -> [String] -> [String]
grep pat = filter (ismatch regex)
    where regex = mkRegex pat
          ismatch r inp = case matchRegex r inp of
                            Nothing -> False
                            Just _ -> True
{- | 创建给定目录。mode 的典型值是 0o755。
System.Posix.Directory.createDirectory 的别名。 -}
mkdir :: FilePath -> FileMode -> IO ()
mkdir = createDirectory
{- | 从文件中删除重复行（类似于 Unix uniq）。
接受一个表示文件或输出的 String，并通过 lines
然后 nub 以行为基础去重。 -}
uniq :: String -> String
uniq = unlines . nub . lines
-- | 计算行数。wc -l
wcL, wcW :: [String] -> [String]
wcL inp = [show (genericLength inp :: Integer)]
-- | 计算文件中的词数（类似于 wc -w）
wcW inp = [show ((genericLength $ words $ unlines inp) :: Integer)]
sortLines :: [String] -> [String]
sortLines = sort
-- | 计算输入中的行数
countLines :: String -> IO String
countLines = return . (++) "\n" . show . length . lines
```

## 关于管道的最后说明

我们在这里开发了一个复杂的系统。我们之前警告过你 POSIX 可能很复杂。另一件我们需要强调的事情是：你必须始终确保在尝试评估子进程的退出码之前，对这些函数返回的 String 进行求值。子进程通常不会退出，直到它可以写入所有数据，如果你以错误的顺序执行此操作，你的程序将挂起。

在本章中，我们从头开发了一个简化版本的 HSH。如果你希望在自已的程序中使用这些类似 shell 的功能，我们推荐 HSH 而不是这里开发的示例，因为 HSH 中存在优化。HSH 还提供了更大的实用函数集和更多功能，但库背后的源代码要复杂和庞大得多。这里展示的一些实用函数实际上是直接从 HSH 复制的。HSH 可在 http://software.complete.org/hsh 获得。
