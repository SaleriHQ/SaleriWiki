# 第02章-类型与函数

## 为什么关心类型？

Haskell 中每个表达式和函数都有一个类型。例如，值 True 的类型是 Bool，而值 "foo" 的类型是 String。值的类型表示它与相同类型的其他值共享某些属性。例如，我们可以添加数字和连接列表；这些是这些类型的属性。我们说表达式具有类型 X，或属于类型 X。

在我们深入讨论 Haskell 类型系统之前，让我们谈谈为什么我们应该关心类型——它们究竟有什么用？在最低层级，计算机关心的是字节，几乎没有额外的结构。类型系统给我们的是抽象。类型为普通字节添加了含义：它让我们可以说"这些字节是文本"、"那些字节是机票预订"等等。通常，类型系统远不止于此，它会阻止我们意外混合类型。例如，类型系统通常不会让我们将酒店预订当作汽车租赁收据。

引入抽象的好处是它让我们能够忘记或忽略低级细节。如果我知道程序中的一个值是字符串，我就不需要知道字符串是如何实现的细节。我可以简单地假设我的字符串会像我以前使用过的所有其他字符串一样表现。

使类型系统有趣的是它们并不都一样。事实上，不同的类型系统通常甚至不关心相同种类的问题。编程语言的类型系统深刻地影响着我们在该语言中思考和编写代码的方式。

Haskell 的类型系统允许我们以非常抽象的层次思考，它允许我们编写简洁、强大的程序。

## Haskell 的类型系统

Haskell 类型有三个有趣的方面：它们是强类型的、静态的，并且可以自动推断。让我们更详细地讨论这些概念的每一个。在可能的情况下，我们将展示 Haskell 类型系统中的概念与其他语言中相关概念之间的相似之处。我们还将触及这些属性各自的优缺点。

### 强类型

当我们说 Haskell 具有强类型系统时，我们的意思是类型系统保证程序不能包含某些类型的错误。这些错误来自于尝试编写没有意义的表达式，例如将整数用作函数。例如，如果函数期望处理整数而我们传递给它一个字符串，Haskell 编译器将拒绝它。

我们称遵守语言类型规则的表达式为类型良好的。违反类型规则的表达式是类型错误的，它将导致类型错误。

Haskell 强类型观的另一个方面是它不会自动将值从一种类型强制转换为另一种类型。（强制转换也称为转换或转换。）

例如，C 编译器会在函数期望 float 类型参数时自动且静默地将 int 类型的值强制转换为 float，但在类似情况下 Haskell 编译器会发出编译错误。我们必须通过应用强制转换函数来显式转换类型。

强类型偶尔确实会使编写某些类型的代码变得更困难。例如，在 C 语言中编写低级代码的一种经典方式是获取一个字节数组并将其转换以将字节视为真正的复杂数据结构。这是非常高效的，因为它不需要我们来回复制字节。Haskell 的类型系统不允许这种强制转换。为了获得相同的结构化数据视图，我们需要进行一些复制，这会在性能上付出一些代价。

强类型的巨大好处是它在我们代码中的真正错误造成问题之前就能捕获它们。例如，在强类型语言中，我们不能意外地在期望整数的地方使用字符串。

### 较弱和较强的类型

值得注意的是，许多语言社区对强类型有自己的定义。但是，我们将简要且广泛地谈谈类型系统中强度的概念。

在计算机科学学术领域，强和弱的含义具有狭窄的技术含义。强度指的是类型系统的宽容程度，而较弱的类型系统比更强的类型系统将更多表达式视为有效。

例如，在 Perl 中，表达式 "foo" + 2 求值为数字 2，但表达式 "13foo" + 2 求值为数字 15。Haskell 拒绝这两个表达式为无效，因为 (+) 运算符要求其两个操作数都是数值。因为 Perl 的类型系统比 Haskell 更宽容，在这个狭窄的技术解释下，我们说它更弱。

类型系统周围的烟火源于普通英语，人们对"弱"和"强"这个词附加了价值观念——我们通常认为 strength 比 weakness 更好。更多程序员说普通英语而不是学术术语，而且相当频繁的是，学术界的人确实在向他们不喜欢的类型系统扔砖头。结果通常是流行的互联网消遣——火焰战。

### 静态类型

拥有静态类型系统意味着编译器在代码执行之前的编译时就知道每个值和表达式的类型。Haskell 编译器或解释器会在我们尝试使用类型不匹配的表达式时检测到这一点，并在我们运行代码之前用错误消息拒绝我们的代码：

```ghci
ghci> True && "false"
<interactive>:1:8:
    Couldn't match expected type `Bool' against inferred type `[Char]'
    In the second argument of `(&&)', namely `"false"'
    In the expression: True && "false"
    In the definition of `it': it = True && "false"
```

我们之前见过这种错误消息。编译器推断表达式 "false" 的类型是 [Char]。(&&) 运算符要求其每个操作数都是 Bool 类型，而它的左操作数确实具有此类型。由于 "false" 的实际类型与要求的类型不匹配，编译器拒绝此表达式为类型错误。

静态类型偶尔会使编写某些有用的代码变得困难。在 Python 等语言中，鸭子类型很常见，其中一个对象的行为足够类似于另一个，可以用作它的替代品。* 幸运的是，Haskell 的类型类系统（我们将在第 6 章介绍）以安全方便的形式提供了动态类型的大部分好处。Haskell 对真正动态类型的编程有一些支持，尽管它不像全心全意拥抱该概念的语言那样容易。

Haskell 强类型和静态类型的结合使得类型错误不可能在运行时发生。虽然这意味着我们需要做更多的前期思考，但它也消除许多可能非常难以找到的简单错误。Haskell 社区中有一句格言：一旦代码编译通过，它比用其他语言更可能正确工作。（也许更现实的说法是 Haskell 代码通常有更少的琐碎错误。）

用动态类型语言编写的程序需要大量测试套件来保证不会发生简单的类型错误。测试套件不能提供完整覆盖：一些常见任务（如重构程序以使其更具模块化）可能引入测试套件可能无法暴露的新类型错误。

在 Haskell 中，编译器为我们证明类型错误的缺失：一个编译通过的 Haskell 程序在运行时不会遭受类型错误。重构通常是移动代码的问题，然后重新编译和整理几次，直到编译器给我们"全部清除"。

理解静态类型值的一个有用类比是将其视为将 pieces 放入拼图。在 Haskell 中，如果一块的形状错误，它根本不会 fit。在动态类型语言中，所有 pieces 都是 1×1 正方形并且总是 fit，所以你必须不断检查结果图片并检查（通过测试）它是否正确。

### 类型推断

最后，Haskell 编译器可以自动推断程序中几乎所有表达式的类型。这个过程称为类型推断。Haskell 允许我们显式声明任何值的类型，但类型推断的存在意味着这几乎总是可选的，而不是我们必须做的。

### 我们对类型系统的期望

我们对 Haskell 类型系统主要能力和优点的探索将跨越多个章节。早期，您可能会发现 Haskell 的类型处理起来很麻烦。

例如，不要只是编写一些代码并运行它来查看它是否按您期望的方式工作（就像您在 Python 或 Ruby 中可能做的那样），您首先需要确保您的程序通过类型检查器的审查。为什么要坚持学习曲线？

虽然强静态类型使 Haskell 安全，但类型推断使其简洁。结果是强大的：我们最终得到一种比流行的静态类型语言更安全且通常比动态类型语言更具表现力的语言。这是一个强有力的声明，我们将在本书中用证据支持它。

修复类型错误最初可能感觉比使用动态语言做更多工作。这可能有助于将其视为将大量调试工作提前。编译器向您展示代码中的许多逻辑缺陷，而不是让您在运行时遇到问题。

此外，因为 Haskell 可以推断您的表达式和函数的类型，您可以获得静态类型的好处，而无需更弱的静态类型语言带来的"手指打字"负担。在其他语言中，类型系统为编译器服务。在 Haskell 中，它为您服务。权衡是您必须学会在其提供的框架内工作。

我们将在本书中始终介绍 Haskell 类型的新用法，以帮助我们编写和测试实际代码。结果，关于类型系统为何值得的完整图景将逐渐显现。虽然每一步都应该自我证明，但整体将大于其各部分的总和。

## 一些常见的基本类型

在第 12 页的"类型的第一步"中，我们介绍了一些类型。这里有更多最常见的基本类型：

- **Char 值**：表示一个 Unicode 字符。
- **Bool 值**：表示布尔逻辑中的值。Bool 类型的可能值是 True 和 False。
- **Int 类型**：用于有符号、固定宽度的整数值。表示为 Int 的确切值范围取决于系统的最长"本机"整数：在 32 位机器上，Int 通常是 32 位宽，而在 64 位机器上，它通常是 64 位宽。Haskell 标准只保证 Int 至少为 28 位宽。（存在恰好 8、16 等位宽的数值类型，有有符号和无符号两种；我们稍后会介绍这些。）
- **Integer 值**：大小无界的有符号整数。Integer 不如 Int 常用，因为它们在性能和空间消耗方面更昂贵。另一方面，Integer 计算不会静默溢出，因此它们给出更可靠正确的结果。
- **Double 类型的值**：用于浮点数。Double 值通常是 64 位宽，使用系统的本机浮点表示。（也存在更窄的类型 Float，但其使用不受鼓励；Haskell 编译器作者更多关注使 Double 高效，所以 Float 慢得多。）

我们之前在第 12 页的"类型的第一步"中简要看到了 Haskell 的类型表示法。当我们显式编写类型时，我们使用 notation expression :: MyType 来表示 expression 具有类型 MyType。如果我们省略 :: 和随后的类型，Haskell 编译器将推断表达式的类型：

```ghci
ghci> :type 'a'
'a' :: Char
ghci> 'a' :: Char
'a'
ghci> [1,2,3] :: Int
<interactive>:1:0:
    Couldn't match expected type `Int' against inferred type `[a]'
    In the expression: [1, 2, 3] :: Int
    In the definition of `it': it = [1, 2, 3] :: Int
```

:: 和类型组合称为类型签名。

## 函数应用

现在我们已经充分了解了数据类型，让我们把注意力转向使用函数处理我们看到的一些类型。

要在 Haskell 中应用函数，我们写函数名，后跟它的参数：

```ghci
ghci> odd 3
True
ghci> odd 6
False
```

我们不使用括号或逗号来分组或分隔函数的参数；只需写函数名，后跟每个参数即可。

例如，让我们应用接受两个参数的 compare 函数：

```ghci
ghci> compare 2 3
LT
ghci> compare 3 3
EQ
ghci> compare 3 2
GT
```

如果您习惯于其他语言中的函数调用语法，这种表示法可能需要一些时间来适应，但它简单且统一。

函数应用的优先级高于使用运算符，所以以下两个表达式的含义相同：

```ghci
ghci> (compare 2 3) == LT
True
ghci> compare 2 3 == LT
True
```

前面代码中的括号没有害处，但增加了一些视觉噪音。然而，有时候我们必须使用括号来表示我们如何解析复杂表达式：

```ghci
ghci> compare (sqrt 3) (sqrt 6)
LT
```

这将 compare 分别应用于 sqrt 3 和 sqrt 6 的结果。如果我们省略括号，看起来就像我们试图将四个参数传递给 compare，而不是它接受的两个。

## 有用的复合数据类型：列表和元组

复合数据类型是从其他类型构建的。Haskell 中最常见的复合数据类型是列表和元组。

我们之前在第 11 页的"字符串和字符"中已经看到过列表类型的提及，我们发现 Haskell 将文本字符串表示为 Char 值列表，而"Char 列表"的类型写作 [Char]。

head 函数返回列表的第一个元素：

```ghci
ghci> head [1,2,3,4]
1
ghci> head ['a','b','c']
'a'
```

它的对应物 tail 返回列表除头部外的所有部分：

```ghci
ghci> tail [1,2,3,4]
[2,3,4]
ghci> tail [2,3,4]
[3,4]
ghci> tail [True,False]
[False]
ghci> tail "list"
"ist"
ghci> tail []
*** Exception: Prelude.tail: empty list
```

正如您所见，我们可以将 head 和 tail 应用于不同类型的列表。将 head 应用于 [Char] 值返回 Char 值，而将其应用于 [Bool] 值返回 Bool 值。head 函数不关心它处理的列表是什么类型。

因为列表中的值可以是任何类型，所以我们称列表类型为多态的。‡ 当我们想写一个多态类型时，我们使用类型变量，它必须以小写字母开头。类型变量是一个占位符，我们最终将用真实类型替换它。

‡ 我们将在第 36 页的"Haskell 中的多态性"中更多讨论多态性。

我们可以通过将类型变量括在方括号中来写"a 的列表"：[a]。这相当于说："我不在乎我有什么类型；我可以从中创建一个列表。"

### 区分类型名和类型变量

我们现在可以看到为什么类型名必须以大写字母开头：它使它与必须以小写字母开头的类型变量区别开来。

当我们谈论具有特定类型值的列表时，我们用我们的类型变量替换该类型。因此，例如，类型 [Int] 是类型 Int 的值列表，因为我们用 Int 替换了 a。同样，类型 [MyPersonalType] 是类型 MyPersonalType 的值列表。我们可以递归地执行此替换：[[Int]] 是类型 [Int] 的值列表，即 Int 列表的列表。

```ghci
ghci> :type [[True],[False,False]]
[[True],[False,False]] :: [[Bool]]
```

此表达式的类型是 Bool 列表的列表。

### 列表是特殊的

列表是 Haskell 集合的面包和黄油。在命令式语言中，我们可能通过循环迭代多次执行任务。这是我们经常在 Haskell 中通过遍历列表做的事情，无论是通过递归还是使用为我们递归的函数。列表是进入我们可以使用数据来构建程序及其控制流这一想法的最简单的垫脚石。我们将花费更多时间讨论列表在第 4 章。

元组是固定大小的值集合，其中每个值可以具有不同的类型。这将它们与列表区分开来，列表可以具有任意长度，但其元素必须都是相同类型。

为了帮助理解这种差异，假设我们想要跟踪关于一本书的两条信息：它的出版年份——一个数字——和它的标题——一个字符串。我们不能将这两条信息保存在列表中，因为它们类型不同。相反，我们使用元组：

```ghci
ghci> (1964, "Labyrinths")
(1964,"Labyrinths")
```

我们通过将元素括在括号中并用逗号分隔来写元组。我们使用相同的表示法来编写其类型：

```ghci
ghci> :type (True, "hello")
(True, "hello") :: (Bool, [Char])
ghci> (4, ['a', 'm'], (16, True))
(4,"am",(16,True))
```

有一种特殊类型 ()，充当零元素元组。此类型只有一个值，也写作 ()。类型和值通常都发音为"unit"。如果您熟悉 C，() 有点像 void。

Haskell 没有一元组的概念。元组通常用元素数量作为前缀来引用。2 元组有两个元素，通常称为对。3 元组（有时称为三元组）有三个元素；5 元组有五个；依此类推。在实践中，使用包含超过几个元素的元组会使代码变得笨拙，所以很少使用多于几个元素的元组。

元组的类型表示其元素的数量、位置和类型。这意味着包含不同数量或类型的元素的元组有不同的类型，类型出现顺序不同的元组也是如此：

```ghci
ghci> :type (False, 'a')
(False, 'a') :: (Bool, Char)
ghci> :type ('a', False)
('a', False) :: (Char, Bool)
```

在这个例子中，表达式 (False, 'a') 的类型是 (Bool, Char)，这与 ('a', False) 的类型不同。尽管元素数量和类型相同，但这两个类型是不同的，因为元素类型的位置不同：

```ghci
ghci> :type (False, 'a', 'b')
(False, 'a', 'b') :: (Bool, Char, Char)
```

此类型 (Bool, Char, Char) 与 (Bool, Char) 不同，因为它包含三个元素，而不是两个。

我们经常使用元组从函数返回多个值。我们也可以在任何需要固定大小值集合的时候使用它们，如果情况不需要自定义容器类型。

## 列表和元组上的函数

我们对列表和元组的讨论提到了如何构造它们，但之后如何处理它们却很少涉及。到目前为止，我们只介绍了两个列表函数，head 和 tail。

两个相关的列表函数 take 和 drop 接受两个参数。给定一个数字 n 和一个列表，take 返回列表的前 n 个元素，而 drop 返回列表除前 n 个元素外的所有元素。（因为这些函数接受两个参数，注意我们使用空白分隔每个函数及其参数。）

```ghci
ghci> take 2 [1,2,3,4,5]
[1,2]
ghci> drop 3 [1,2,3,4,5]
[4,5]
```

对于元组，fst 和 snd 函数分别返回对的第一和第二个元素：

```ghci
ghci> fst (1,'a')
1
ghci> snd (1,'a')
'a'
```

如果您的背景是其他任何语言，这些中的每一个看起来都像是应用于两个参数的函数应用。在 Haskell 的函数应用约定下，每一个都是一个函数应用于单个对。

### Haskell 元组不是不可变列表

如果您来自 Python 世界，您可能习惯于列表和元组几乎可以互换。虽然 Python 元组的元素是不可变的，但它可以使用与列表相同的方法进行索引和迭代。在 Haskell 中情况并非如此，因此不要尝试将这个想法带入陌生的语言领域。

作为说明，请查看 fst 和 snd 的类型签名：它们只为对定义，不能用于其他大小的元组。Haskell 的类型系统使得编写通用的"从任何宽度的元组获取第二个元素"函数变得棘手。

### 将表达式传递给函数

在 Haskell 中，函数应用是左结合的。这最好通过示例说明：表达式 `a b c d` 等价于 `(((a b) c) d)`。如果我们想将一个表达式用作另一个的参数，我们必须使用显式括号来告诉解析器我们真正想要什么。这里有一个例子：

```ghci
ghci> head (drop 4 "azerty")
't'
```

我们可以将其解读为"将表达式 drop 4 "azerty" 作为参数传递给 head。"如果我们省略括号，违规表达式类似于将三个参数传递给 head。编译将因类型错误而失败，因为 head 需要一个参数，一个列表。

## 函数类型和纯度

让我们看看函数的类型：

```ghci
ghci> :type lines
lines :: String -> [String]
```

我们可以将 -> 读作"到"，大致翻译为"返回"。整个签名 thus reads as "lines 的类型是 String 到 list-of-String"。让我们尝试应用该函数：

```ghci
ghci> lines "the quick\nbrown fox\njumps"
["the quick","brown fox","jumps"]
```

lines 函数在行边界上拆分字符串。请注意，它的类型签名暗示了函数实际可能做什么的提示：它接受一个 String，返回许多。这是一种功能语言中类型极具价值的属性。

副作用在函数行为和系统全局状态之间引入依赖关系。例如，让我们暂时离开 Haskell 思考一下，考虑一下命令式编程语言。考虑一个读取并返回全局变量值的函数。如果其他代码可以修改该全局变量，那么我们函数特定应用的结果取决于全局变量的当前值。该函数有副作用，即使它本身从不修改变量。

副作用本质上是函数的不可见输入或输出。在 Haskell 中，默认情况下函数没有副作用：函数的结果仅取决于我们明确提供的输入。我们称这些函数为纯函数；有副作用的函数是非纯的。

如果函数有副作用，我们可以通过阅读其类型签名来判断——函数结果的类型将以 IO 开头：

```ghci
ghci> :type readFile
readFile :: FilePath -> IO String
```

Haskell 的类型系统阻止我们意外混合纯代码和非纯代码。

## Haskell 源文件，以及编写简单函数

既然我们知道如何应用函数，现在是时候将注意力转向编写它们了。虽然我们可以在 ghci 中编写函数，但这不是一个好的环境。它只接受 Haskell 的一个高度受限的子集——最重要的是，用于定义函数的语法与我们用在 Haskell 源文件中不同。§ 相反，我们将最终分解并创建一个源文件。

§ ghci 操作的环境称为 IO monad。在第 7 章中，我们将深入介绍 IO monad，ghci 对我们施加的看似任意的限制将更有意义。

Haskell 源文件通常使用 .hs 后缀标识。一个简单的函数定义是打开一个名为 add.hs 的文件，并向其中添加以下内容：

```haskell
-- file: ch03/add.hs
add a b = a + b
```

在 = 的左侧是函数的名称，后跟函数的参数。右侧是函数体。保存源文件后，我们可以将其加载到 ghci 中，并立即使用我们的新 add 函数（加载文件后 ghci 显示的提示符会改变）：

```ghci
ghci> :load add.hs
[1 of 1] Compiling Main             ( add.hs, interpreted )
Ok, modules loaded: Main.
ghci> add 1 2
3
```

### 如果 ghci 找不到您的源文件怎么办？

当您运行 ghci 时，它可能找不到您的源文件。它会在运行目录中搜索源文件。如果这不是您的源文件实际所在的目录，您可以使用 ghci 的 :cd 命令更改其工作目录：

```ghci
ghci> :cd /tmp
```

或者，您可以将 Haskell 源文件的路径作为 :load 的参数。这可以是绝对路径，也可以是相对于 ghci 当前目录的路径。

当我们将 add 应用于值 1 和 2 时，定义左侧的变量 a 和 b 被给予（或"绑定到"）值 1 和 2，所以结果是表达式 1 + 2。

Haskell 没有 return 关键字，因为函数是单个表达式，而不是语句序列。表达式的值是函数的结果。（Haskell 确实有一个名为 return 的函数，但我们不会很快讨论它；它具有与命令式语言不同的含义。）

当您在 Haskell 代码中看到 = 符号时，它代表"含义"——左边的名称被定义为右边的表达式。

### 变量究竟是什么？

在 Haskell 中，变量提供了一种为表达式命名的方式。一旦变量绑定到（即与）特定表达式，它的值就不会改变：我们总是可以使用变量的名称而不是写出表达式，两种方式都会得到相同的结果。

如果您的背景是命令式编程语言，您可能将变量视为识别可以随时保存不同值的内存位置（或某种等效物）的方式。在命令式语言中，我们可以随时更改变量的值，因此反复检查内存位置可能会每次给出不同的结果。

这两种变量概念之间的关键区别是：在 Haskell 中，一旦我们将变量绑定到表达式，我们知道我们总是可以将其替换为该表达式，因为它不会改变。在命令式语言中，这种可替换性的概念不成立。

例如，如果我们运行以下小型 Python 脚本，它将打印数字 11：

```python
x = 10
x = 11
# value of x is now 11
print x
```

相比之下，在 Haskell 中尝试等效操作会导致错误：

```haskell
-- file: ch02/Assign.hs
x = 10
x = 11
```

我们不能给 x 赋值两次：

```ghci
ghci> :load Assign
[1 of 1] Compiling Main             ( Assign.hs, interpreted )
Assign.hs:4:0:
    Multiple declarations of `Main.x'
    Declared at: Assign.hs:3:0
                 Assign.hs:4:0
Failed, modules loaded: none.
```

## 条件求值

像许多其他语言一样，Haskell 有一个 if 表达式。让我们看看它的运行情况；然后我们将解释发生了什么。作为示例，我们将编写我们自己版本的标准 drop 函数。在我们开始之前，让我们稍微探究一下 drop 的行为，以便我们可以复制它的行为：

```ghci
ghci> drop 2 "foobar"
"obar"
ghci> drop 4 "foobar"
"ar"
ghci> drop 4 [1,2]
[]
ghci> drop 0 [1,2]
[1,2]
ghci> drop 7 []
[]
ghci> drop (-2) "foo"
"foo"
```

从这个代码来看，如果要删除的数字小于或等于零，drop 返回原始列表。否则，它会删除元素直到用完或达到给定数字。这是一个具有相同行为的 myDrop 函数，它使用 Haskell 的 if 表达式来决定做什么。下面的 null 函数检查列表是否为空：

```haskell
-- file: ch02/myDrop.hs
myDrop n xs = if n <= 0 || null xs
              then xs
              else myDrop (n - 1) (tail xs)
```

在 Haskell 中，缩进很重要：它继续现有定义，而不是开始新定义。不要省略缩进！

您可能想知道 Haskell 函数中的变量名 xs 从何而来。这是列表的常见命名模式。您可以将 s 读作后缀，所以该名称本质上是"x 的复数"。

让我们将 Haskell 函数保存在名为 myDrop.hs 的文件中，然后将其加载到 ghci 中：

```ghci
ghci> :load myDrop.hs
[1 of 1] Compiling Main             ( myDrop.hs, interpreted )
Ok, modules loaded: Main.
ghci> myDrop 2 "foobar"
"obar"
ghci> myDrop 4 "foobar"
"ar"
ghci> myDrop 4 [1,2]
[]
ghci> myDrop 0 [1,2]
[1,2]
ghci> myDrop 7 []
[]
ghci> myDrop (-2) "foo"
"foo"
```

现在我们看到了 myDrop 的运行，让我们回到源代码并查看我们引入的所有新内容。

首先，我们引入了 --，单行注释的开始。此注释延伸到行尾。

接下来是 if 关键字本身。它引入了一个有三个部分的表达式：

- 一个 Bool 类型的表达式，紧跟在 if 之后。我们将其称为谓词。
- 然后关键字，后跟另一个表达式。如果谓词求值为 True，此表达式将用作 if 表达式的值。
- else 关键字，后跟另一个表达式。如果谓词求值为 False，此表达式将用作 if 表达式的值。

我们将 then 和 else 关键字之后的表达式称为"分支"。分支必须具有相同的类型；if 表达式也将具有此类型。像 `if True then 1 else "foo"` 这样的表达式，其分支类型不同，所以它是类型错误的，编译器或解释器将拒绝它。

回想一下 Haskell 是一种面向表达式的语言。在命令式语言中，省略 if 的 else 分支可能是有意义的，因为我们处理的是语句，而不是表达式。然而，当我们处理表达式时，缺少 else 的 if 如果谓词求值为 False 将没有结果或类型，所以它会是荒谬的。

我们的谓词包含更多新内容。null 函数表示列表是否为空，而 (||) 运算符对其 Bool 类型参数执行逻辑"或"：

```ghci
ghci> :type null
null :: [a] -> Bool
ghci> :type (||)
(||) :: Bool -> Bool -> Bool
```

### 运算符不是特殊的

注意我们可以通过将 (||) 括在括号中来找到它的类型。(||) 运算符不是语言内置的；它是一个普通函数。

(||) 运算符"短路"：如果其左操作数求值为 True，它不会求值其右操作数。在大多数语言中，短路求值需要特殊支持，但在 Haskell 中不需要。我们很快就会看到原因。

接下来，我们的函数递归地应用自己。这是我们递归的第一个例子，我们稍后会详细讨论。

最后，我们的 if 表达式跨越多行。我们将 then 和 else 分支对齐在 if 下方以保持整洁。只要我们使用一些缩进，确切的数量并不重要。如果我们愿意，我们可以将整个表达式写在一行上：

```haskell
-- file: ch02/myDrop.hs
myDropX n xs = if n <= 0 || null xs then xs else myDropX (n - 1) (tail xs)
```

这个版本的长度使其更难阅读。我们通常会将 if 表达式分成几行，以使谓词和每个分支更容易遵循。

作为比较，这是 Haskell myDrop 的 Python 等效项。两者结构相似——每个都在从列表头部移除元素的同时递减计数器：

```python
def myDrop(n, elts):
    while n > 0 and elts:
        n = n - 1
        elts = elts[1:]
    return elts
```

## 通过示例理解求值

在我们对 myDrop 的描述中，我们目前专注于表面特征。我们需要更深入地开发一个关于函数应用如何工作的有用心智模型。为此，我们将首先通过一些简单的示例工作，直到我们可以逐步完成表达式 `myDrop 2 "abcd"` 的求值。

我们已经很多谈论过用表达式替换变量，我们将在这里利用这个能力。我们的过程将涉及一遍又一遍地重写表达式，用表达式替换变量，直到达到最终结果。这是拿出铅笔和纸的好时机，这样您可以自己尝试我们的描述。

### 惰性求值

我们将首先查看一个简单的非递归函数的定义：

```haskell
-- file: ch02/RoundToEven.hs
isOdd n = mod n 2 == 1
```

这里，mod 是标准的模函数。理解 Haskell 中求值如何工作的第一个大步骤是找出表达式 `isOdd (1 + 2)` 的结果。

在我们解释求值如何在 Haskell 中进行之前，让我们回顾一下更熟悉的语言使用的求值策略。首先，求值子表达式 1 + 2 得到 3。然后应用 odd 函数，n 绑定到 3。最后，求值 `mod 3 2` 得到 1，而 `1 == 1` 得到 True。

在使用严格求值的语言中，函数的参数在函数应用之前被求值。Haskell 选择另一条路径：非严格求值。

在 Haskell 中，子表达式 1 + 2 不会被缩减为值 3。相反，我们创建一个"承诺"，当表达式 `isOdd (1 + 2)` 的值需要时，我们将能够计算它。我们用来跟踪未求值表达式的记录称为 thunk。这一切就这样发生了：我们创建一个 thunk 并将实际求值推迟到真正需要的时候。如果这个表达式的结果随后从未被使用，我们根本不会计算它的值。

非严格求值通常被称为惰性求值。‖

‖ 术语"非严格"和"惰性"有略微不同的技术含义，但我们不会在这里详细介绍它们的区别。

### 一个更深入的示例

现在让我们看看表达式 `myDrop 2 "abcd"` 的求值，我们使用 print 来确保它会被求值：

```ghci
ghci> print (myDrop 2 "abcd")
"cd"
```

我们的第一步是尝试应用 print，这需要对其参数求值。为此，我们将函数 myDrop 应用于值 2 和 "abcd"。我们将变量 n 绑定到值 2，xs 绑定到 "abcd"。如果我们将这些值代入 myDrop 的谓词，会得到以下表达式：

```ghci
ghci> :type  2 <= 0 || null "abcd"
2 <= 0 || null "abcd" :: Bool
```

然后我们求值谓词以找出它的值。这要求我们求值 (||) 表达式。为了确定其值，(||) 运算符需要首先检查其左操作数的值：

```ghci
ghci> 2 <= 0
False
```

将这个值代入 (||) 表达式会产生以下表达式：

```ghci
ghci> :type  False || null "abcd"
False || null "abcd" :: Bool
```

如果左操作数求值为 True，(||) 不需要求值其右操作数，因为它不会影响结果。既然它求值为 False，(||) 必须求值右操作数：

```ghci
ghci> null "abcd"
False
```

现在我们将这个值代入回 (||) 表达式。由于两个操作数都求值为 False，(||) 表达式也是如此，因此谓词求值为 False：

```ghci
ghci> False || False
False
```

这导致 if 表达式的 else 分支被求值。这个分支包含 myDrop 的递归应用。

### 免费短路

许多语言需要特殊对待逻辑或运算符，以便在其左操作数求值为 True 时短路。在 Haskell 中，(||) 是一个普通函数：非严格求值将此能力构建到语言中。

在 Haskell 中，我们可以轻松定义一个新函数来短路：

```haskell
-- file: ch02/shortCircuit.hs
newOr a b = if a then a else b
```

如果我们写一个表达式如 `newOr True (length [1..] > 0)`，它不会求值其第二个参数。（这很好：该表达式尝试计算无限列表的长度。如果它被求值，它会挂起 ghci，循环无限直到我们杀死它。）

如果我们在比如说 Python 中编写一个类似的函数，严格求值会咬我们：两个参数在传递给 newOr 之前都会被求值，我们将无法避免第二个参数上的无限循环。

### 递归

当我们递归应用 myDrop 时，n 被绑定到 thunk 2 - 1，xs 被绑定到 tail "abcd"。

我们现在再次从头开始求值 myDrop。我们将 n 和 xs 的新值代入谓词：

```ghci
ghci> :type (2 - 1) <= 0 || null (tail "abcd")
(2 - 1) <= 0 || null (tail "abcd") :: Bool
```

这里是左操作数求值的压缩版本：

```ghci
ghci> :type (2 - 1) <= 0
(2 - 1) <= 0 :: Bool
ghci> 2 - 1
1
ghci> 1 <= 0
False
```

正如我们现在所期望的，我们直到需要值时才求值表达式 2 - 1。我们还惰性地求值右操作数，将 `tail "abcd"` 的求值推迟到我们需要它的时候：

```ghci
ghci> :type null (tail "abcd")
null (tail "abcd") :: Bool
ghci> tail "abcd"
"bcd"
ghci> null "bcd"
False
```

谓词再次求值为 False，导致 else 分支再次被求值。

### 结束递归

在 myDrop 的下一次递归应用中，我们将 n 绑定到 1 - 1，xs 绑定到 `tail "bcd"`：

```ghci
ghci> :type (1 - 1) <= 0 || null (tail "bcd")
(1 - 1) <= 0 || null (tail "bcd") :: Bool
```

又一次，(||) 需要首先求值其左操作数：

```ghci
ghci> :type (1 - 1) <= 0
(1 - 1) <= 0 :: Bool
ghci> 1 - 1
0
ghci> 0 <= 0
True
```

最终，这个表达式求值为 True！

```ghci
ghci> True || null (tail "bcd")
True
```

因为右操作数不能影响 (||) 的结果，所以它没有被求值，谓词的结果是 True。这导致我们求值 then 分支：

```ghci
ghci> :type tail "bcd"
tail "bcd" :: [Char]
```

### 从递归返回

请记住，我们现在在 myDrop 的第二次递归应用内部。此应用求值为 `tail "bcd"`。我们从函数应用返回，将此表达式替换为 `myDrop (1 - 1) (tail "bcd")` 成为此应用的结果：

```ghci
ghci> myDrop (1 - 1) (tail "bcd") == tail "bcd"
True
```

然后我们从第一次递归应用返回，将第二次递归应用的结果替换为 `myDrop (2 - 1) (tail "abcd")` 成为此应用的结果：

```ghci
ghci> myDrop (2 - 1) (tail "abcd") == tail "bcd"
True
```

最后，我们从原始应用返回，将第一次递归应用的结果替换：

```ghci
ghci> myDrop 2 "abcd" == tail "bcd"
True
```

注意，当我们从每个连续的递归应用返回时，它们都不需要求值表达式 `tail "bcd"`：求值原始表达式的最终结果是一个 thunk。thunk 只在 ghci 需要打印它时才被求值。

```ghci
ghci> myDrop 2 "abcd"
"cd"
ghci> tail "bcd"
"cd"
```

### 我们学到了什么？

我们建立了几个重要点：

- 使用替换和重写来理解 Haskell 表达式求值是有意义的。
- 惰性求值使我们推迟求值直到需要值，并仅求值表达式中足以确定其值的部分。
- 应用函数的结果可能是一个 thunk（一个延迟的表达式）。

## Haskell 中的多态性

当我们介绍列表时，我们提到列表类型是多态的。我们将更详细地讨论 Haskell 的多态性。

如果我们想获取列表的最后一个元素，我们使用 last 函数。它返回的值必须与列表元素的类型相同，但 last 以相同的方式操作，无论这些元素的实际类型是什么：

```ghci
ghci> last [1,2,3,4,5]
5
ghci> last "baz"
'z'
```

为了捕捉这个想法，它的类型签名包含一个类型变量：

```ghci
ghci> :type last
last :: [a] -> a
```

这里，a 是类型变量。我们可以将签名解读为"接受一个列表，其所有元素具有某种类型 a，并返回相同类型 a 的值"。

### 识别类型变量

类型变量总是以小写字母开头。您总是可以通过上下文将类型变量与普通变量区分开来，因为类型和函数的语言是分开的：类型变量存在于类型签名中，普通变量存在于正常表达式中。

将 Haskell 中类型变量的名称保持很短是常见的做法。一个字母极为常见；更长的名称很少出现。类型签名通常很简短；通过保持名称简短，我们获得的比使它们具有描述性更多。

当函数的签名中有类型变量时，表示其某些参数可以是任何类型，我们称该函数为多态的。

当我们想将 last 应用到比如说 Char 列表时，编译器用 Char 替换签名中的每个 a。这给了我们输入为 [Char] 时 last 的类型为 [Char] -> Char。

这种多态性称为参数多态性。通过类比，命名选择很容易理解：就像函数可以有参数，我们以后可以绑定到真实值一样，Haskell 类型可以有参数，我们以后可以绑定到其他类型。

### 一些命名法

如果类型包含类型参数，我们称它为参数化类型或多态类型。如果函数或值的类型包含类型参数，我们称它为多态的。

当我们看到参数化类型时，我们已经注意到代码不关心实际类型是什么。但是，我们可以做一个更强的陈述：它没有办法找出真实类型是什么，也没有办法操作该类型的值。它不能创建值；它也不能检查一个。它只能将其视为完全抽象的"黑匣子"。我们稍后会介绍这很重要的原因。

参数多态性是 Haskell 支持的最可见的多态性种类。Haskell 的参数多态性直接影响了 Java 和 C# 语言的泛型设施的设计。Haskell 中的参数化类型类似于 Java 泛型中的类型变量。C++ 模板也与参数多态性有相似之处。

为了更清楚地说清楚 Haskell 的多态性与其他语言有何不同，这里有一些在其他语言中常见但在 Haskell 中不存在的多态形式。

在主流面向对象语言中，子类型多态性比参数多态性更普遍。C++ 和 Java 的子类机制赋予了它们子类型多态性。基类定义了一组其子类可以修改和扩展的行为。由于 Haskell 不是面向对象的语言，它不提供子类型多态性。

 также常见的 是强制转换多态性，它允许一种类型的值隐式转换为另一种类型的值。许多语言提供某种形式的强制转换多态性；一个例子是整数和浮点数之间的自动转换。Haskell 故意避免甚至这种简单的自动强制转换。

这不是 Haskell 中多态性的全部故事。我们将在第 6 章回到这个主题。

## 推理多态函数

在第 27 页的"函数类型和纯度"中，我们讨论了根据函数的类型签名找出函数的行为。我们可以将同样类型的推理应用于多态函数。让我们再次查看 fst：

```ghci
ghci> :type fst
fst :: (a, b) -> a
```

首先，请注意其参数包含两个类型变量 a 和 b，表示元组的元素可以是不同类型。

fst 的结果类型是 a。我们已经提到参数多态性使真实类型不可访问。fst 没有足够的信息来构造类型 a 的值，也不能将 a 转换为 b。所以唯一可能的有效行为（忽略无限循环或崩溃）是返回对的第一个元素。

### 进一步阅读

任何非病态函数 `(a,b) -> a` 必须完全像 fst 那样做，这有一种深刻的数学意义。而且，这种推理路线延伸到更复杂的多态函数。Philip Wadler 的论文"Theorems for free" (http://citeseerx.ist.psu.edu/viewdoc/summary?doi=10.1.1.38.9875) 深入涵盖了这个过程。

## 具有多个参数的函数的类型

到目前为止，我们还没有过多查看接受多个参数的函数的签名。我们已经使用了一些这样的函数；让我们看看其中一个的签名，take：

```ghci
ghci> :type take
take :: Int -> [a] -> [a]
```

很明显，Int 和一些列表有些事情在发生，但为什么签名中有两个 -> 符号？Haskell 从右到左分组这个箭头链；也就是说，-> 是右结合的。如果我们引入括号，我们可以更清楚地了解这个类型签名是如何解释的：

```haskell
-- file: ch02/Take.hs
take :: Int -> ([a] -> [a])
```

从这里看来，我们应该将类型签名解读为一个接受一个参数（Int）的函数，并返回另一个函数。那个函数也接受一个参数（一个列表）并返回与其结果相同类型的列表。

这是正确的，但不容易看出它的后果是什么。一旦我们花了一些时间编写函数，我们将在第 100 页的"部分函数应用和柯里化"中回到这个主题。现在，我们可以将最后一个 -> 之后的类型视为函数的返回类型，而前面的类型是函数的参数。

我们现在可以为之前定义的 myDrop 函数编写一个类型签名：

```haskell
-- file: ch02/myDrop.hs
myDrop :: Int -> [a] -> [a]
```

## 为什么这么强调纯度？

很少有编程语言像 Haskell 那样坚持默认情况下应该是纯度的。这种选择具有深刻而有价值的后果。

因为应用纯函数的结果只能依赖于它的参数，我们通常可以通过简单阅读它的名字并理解其类型签名来强烈提示纯函数的作用。例如，让我们看看 not：

```ghci
ghci> :type not
not :: Bool -> Bool
```

即使我们不知道这个函数的名字，它的签名本身就限制了它可能有的有效行为：

- 忽略其参数并始终返回 True 或 False。
- 返回其参数不变。
- 求反其参数。

我们也知道这个函数不能做某些事情：访问文件、访问网络、以及告诉时间。

纯度使理解代码的工作变得更容易。纯函数的行为不依赖于全局变量的值，或数据库的内容，或网络连接的状态。纯代码本质上是模块化的：每个函数都是独立的，并且具有定义良好的接口。

纯度默认为止的一个非显而易见的后果是，使用非纯代码变得更容易。Haskell 鼓励一种编程风格，其中我们将必须有副作用的代码与不需要副作用的代码分开。在这种风格中，非纯代码往往是简单的，"繁重的工作"在纯代码中完成。

软件的大部分风险在于与外部世界交谈，无论是处理坏掉的或缺失的数据，还是处理恶意攻击。因为 Haskell 的类型系统准确告诉我们代码的哪些部分有副作用，我们可以适当警惕。因为我们偏好的编码风格保持非纯代码隔离和简单，我们的"攻击面"很小。
