## 第二十八章：软件事务内存

在传统的线程并发编程模型中，当我们在线程之间共享数据时，我们使用锁来保持一致，并使用条件变量来通知线程变化。Haskell 的 MVar 机制在这些工具上有所改进，但仍然存在所有相同的问题：

- 由于忘记锁导致的竞态条件
- 由于锁顺序不一致导致的死锁
- 由于未捕获异常导致的损坏
- 由于省略通知导致的丢失唤醒

这些问题经常影响即使是最小的并发程序，但它们带来的困难在更大的代码库或重负载下会变得更糟。

例如，一个有几个大锁的程序在编写和调试方面还算可控，但这些锁的争用会在重负载下把我们击垮。如果我们用更细粒度的锁来反应，编写能正常工作的软件就变得更加困难。额外的记录工作即使在轻负载时也会损害性能。

## 基础知识

软件事务内存（STM）给我们提供了几个简单但强大的工具，我们可以用它们来解决大多数这些问题。我们使用 atomically 组合器将一组动作作为事务执行。一旦我们进入该块，其他线程就不能看到我们所做的任何修改，直到我们退出，我们的线程也不能看到其他线程所做的更改。这两个属性意味着我们的执行是隔离的。

事务退出时，正好会发生以下情况之一：

- 如果没有其他线程同时修改与我们相同的数据，我们所有的修改将同时对其他线程可见。
- 否则，我们的修改将被丢弃而不执行，我们的动作块将自动重新启动。

这种 atomically 块的"全有或全无"性质被称为原子性，因此有了组合器的名称。如果你使用过支持事务的数据库，你会发现使用 STM 感觉非常熟悉。

## 一些简单的例子

在多人角色扮演游戏中，玩家角色会有一些状态，如生命值、财产和金钱。为了探索 STM 的世界，让我们从几个基于游戏角色状态工作的简单函数和类型开始。我们将随着学习更多 API 来改进我们的代码。

STM API 由 stm 包提供，其模块位于 Control.Concurrent.STM 层次结构中：

```haskell
-- file: ch28/GameInventory.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
import Control.Concurrent.STM
import Control.Monad
data Item = Scroll
          | Wand
          | Banjo
            deriving (Eq, Ord, Show)
newtype Gold = Gold Int
    deriving (Eq, Ord, Show, Num)
newtype HitPoint = HitPoint Int
    deriving (Eq, Ord, Show, Num)
type Inventory = TVar [Item]
type Health = TVar HitPoint
type Balance = TVar Gold
data Player = Player {
      balance :: Balance,
      health :: Health,
      inventory :: Inventory
    }
```

TVar 参数化类型是一个可变变量，我们可以在 atomically 块内读取或写入它。为简单起见，我们将玩家的库存表示为物品列表。

还要注意，我们使用 newtype 声明，这样我们就不会意外地将财富与健康混淆。

要执行从一个 Balance 到另一个 Balance 的基本资金转移，我们只需要调整每个 TVar 中的值：

```haskell
-- file: ch28/GameInventory.hs
basicTransfer qty fromBal toBal = do
  fromQty <- readTVar fromBal
  toQty   <- readTVar toBal
  writeTVar fromBal (fromQty - qty)
  writeTVar toBal   (toQty + qty)
```

让我们编写一个小函数来尝试：

```haskell
-- file: ch28/GameInventory.hs
transferTest = do
  alice <- newTVar (12 :: Gold)
  bob   <- newTVar 4
  basicTransfer 3 alice bob
  liftM2 (,) (readTVar alice) (readTVar bob)
```

如果我们在 ghci 中运行它，它的行为符合我们的预期：

```haskell
ghci> :load GameInventory
[1 of 1] Compiling Main             ( GameInventory.hs, interpreted )
Ok, modules loaded: Main.
ghci> atomically transferTest
Loading package array-0.1.0.0 ... linking ... done.
Loading package stm-2.1.1.1 ... linking ... done.
(Gold 9,Gold 7)
```

原子性和隔离属性保证，如果另一个线程看到 bob 余额的变化，他们也能看到 alice 余额的修改。

即使在并发程序中，我们仍然努力保持尽可能多的代码是纯函数的。这使我们的代码更容易推理和测试。它还为底层 STM 引擎减少工作量，因为涉及的数据不是事务性的。

这是一个纯函数，从我们用来表示玩家库存的列表中移除一个物品：

```haskell
-- file: ch28/GameInventory.hs
removeInv :: Eq a => a -> [a] -> Maybe [a]
removeInv x xs =
    case takeWhile (/= x) xs of
      (_:ys) -> Just ys
      []     -> Nothing
```

结果使用 Maybe，这样我们就可以判断该物品是否真的存在于玩家的库存中。

这是一个事务性函数，用于将物品给予另一个玩家，由于需要确定捐赠者是否真的拥有该物品，它稍微复杂一些：

```haskell
-- file: ch28/GameInventory.hs
maybeGiveItem item fromInv toInv = do
  fromList <- readTVar fromInv
  case removeInv item fromList of
    Nothing      -> return False
    Just newList -> do
      writeTVar fromInv newList
      destItems <- readTVar toInv
      writeTVar toInv (item : destItems)
      return True
```

## STM 和安全性

如果我们要提供原子性、隔离的事务，关键是我们不能故意或意外地从 atomically 块中逃逸。Haskell 的类型系统通过 STM 单子代表我们强制执行此操作：

```haskell
ghci> :type atomically
atomically :: STM a -> IO a
```

atomically 块接受一个 STM 单子中的动作，执行它，并将其结果在 IO 单子中提供给我们。这是所有事务代码执行的单子。例如，我们之前看到的用于操作 TVar 值的函数都在 STM 单子中操作：

```haskell
ghci> :type newTVar
newTVar :: a -> STM (TVar a)
ghci> :type readTVar
readTVar :: TVar a -> STM a
ghci> :type writeTVar
writeTVar :: TVar a -> a -> STM ()
```

这也适用于我们之前定义的事务性函数：

```haskell
-- file: ch28/GameInventory.hs
basicTransfer :: Gold -> Balance -> Balance -> STM ()
maybeGiveItem :: Item -> Inventory -> Inventory -> STM Bool
```

STM 单子不允许我们执行 I/O 或操作非事务性可变状态（如 MVar 值）。这让我们避免可能违反事务保证的操作。

## 重试事务

我们 maybeGiveItem 函数的 API 有点尴尬。它只在角色真正拥有它时才给予物品，这是合理的，但通过返回 Bool，它使调用者的代码复杂化。这是一个必须查看 maybeGiveItem 结果以决定下一步做什么的物品销售函数：

```haskell
-- file: ch28/GameInventory.hs
maybeSellItem :: Item -> Gold -> Player -> Player -> STM Bool
maybeSellItem item price buyer seller = do
  given <- maybeGiveItem item (inventory seller) (inventory buyer)
  if given
    then do
      basicTransfer price (balance buyer) (balance seller)
      return True
    else return False
```

我们不仅要检查物品是否被给予，还必须将成功指示传播回我们的调用者。因此复杂性向外层层传递。

有一种更优雅的方式来处理无法成功的事务。STM API 提供了一个 retry 动作，它将立即终止无法继续的 atomically 块。顾名思义，当发生这种情况时，块的执行将从新开始，先前的任何修改都不会执行。以下是重写 maybeGiveItem 以使用 retry：

```haskell
-- file: ch28/GameInventory.hs
giveItem :: Item -> Inventory -> Inventory -> STM ()
giveItem item fromInv toInv = do
  fromList <- readTVar fromInv
  case removeInv item fromList of
    Nothing -> retry
    Just newList -> do
      writeTVar fromInv newList
      readTVar toInv >>= writeTVar toInv . (item :)
```

我们之前的 basicTransfer 有另一种缺陷：它没有检查发送者的余额以查看她是否有足够的资金转移。我们可以使用 retry 来纠正这个问题，同时保持函数类型相同：

```haskell
-- file: ch28/GameInventory.hs
transfer :: Gold -> Balance -> Balance -> STM ()
transfer qty fromBal toBal = do
  fromQty <- readTVar fromBal
  when (qty > fromQty) $
    retry
  writeTVar fromBal (fromQty - qty)
  readTVar toBal >>= writeTVar toBal . (qty +)
```

现在我们使用 retry，我们的物品销售函数变得非常简单：

```haskell
-- file: ch28/GameInventory.hs
sellItem :: Item -> Gold -> Player -> Player -> STM ()
sellItem item price buyer seller = do
  giveItem item (inventory seller) (inventory buyer)
  transfer price (balance buyer) (balance seller)
```

它的行为与我们之前的函数略有不同。如果卖家没有该物品，它不会立即返回 False，而是会阻塞（如有必要），直到卖家拥有该物品且买家有足够的钱支付。

STM 的美在于它让我们编写的代码干净。我们可以获取两个单独正确工作的函数，并用它们创建一个也会正确行为的第三个函数，而且只需最少的工作。

## 当我们重试时会发生什么？

retry 函数不仅让我们的代码更干净——它的底层行为看起来几乎是神奇的。当我们调用它时，它不会立即重启我们的事务。相反，它阻塞我们的线程，直到在我们调用 retry 之前触摸的一个或多个变量被另一个线程更改。

例如，如果我们用资金不足调用 transfer，retry 将自动等待，直到我们的余额改变后才重新开始 atomically 块。新的 giveItem 函数也会发生同样的情况：如果发送者当前库存中没有该物品，线程将阻塞，直到他有了。

## 在替代方案之间选择

如果 atomically 动作调用 retry 或由于另一个线程的并发修改而失败，我们并不总是想要重启它。例如，我们的新 sellItem 函数将无限重试，只要我们缺少该物品或足够的钱，但我们可能宁愿只尝试一次销售。

orElse 组合器让我们在主动作失败时执行一个"备份"动作：

```haskell
ghci> :type orElse
orElse :: STM a -> STM a -> STM a
```

如果 sellItem 失败，orElse 将调用 return False 动作，导致我们的销售函数立即返回。

## 在事务中使用高阶代码

想象一下，我们想更有雄心一点，从列表中购买第一个卖家拥有且我们负担得起的物品，但如果我们目前买不起任何东西，它什么也不做。当然，我们可以直接编写代码来做到这一点：

```haskell
-- file: ch28/GameInventory.hs
crummyList :: [(Item, Gold)] -> Player -> Player
             -> STM (Maybe (Item, Gold))
crummyList list buyer seller = go list
    where go []                         = return Nothing
          go (this@(item,price) : rest) = do
              sellItem item price buyer seller
              return (Just this)
           `orElse`
              go rest
```

这个函数遭受我们熟悉的将我们想要做什么与我们应该如何做混在一起的问题。稍加检查就会发现这段代码中埋藏着两个可重用的模式。

第一个是使事务立即失败而不是重试：

```haskell
-- file: ch28/GameInventory.hs
maybeSTM :: STM a -> STM (Maybe a)
maybeSTM m = (Just `liftM` m) `orElse` return Nothing
```

其次，我们想要对列表的连续元素尝试一个动作，在第一个成功时停止，或者如果每个都失败则执行重试。方便的是，STM 是 MonadPlus 类型类的实例：

```haskell
-- file: ch28/STMPlus.hs
instance MonadPlus STM where
  mzero = retry
  mplus = orElse
```

Control.Monad 模块按如下定义 msum 函数，这正是我们需要的：

```haskell
-- file: ch28/STMPlus.hs
msum :: MonadPlus m => [m a] -> m a
msum =  foldr mplus mzero
```

我们现在有一些关键的工具片段，可以帮助我们编写一个更清晰的函数版本：

```haskell
-- file: ch28/GameInventory.hs
shoppingList :: [(Item, Gold)] -> Player -> Player
             -> STM (Maybe (Item, Gold))
shoppingList list buyer seller = maybeSTM . msum $ map sellOne list
    where sellOne this@(item,price) = do
            sellItem item price buyer seller
            return this
```

由于 STM 是 MonadPlus 类型类的实例，我们可以将 maybeSTM 推广为适用于任何 MonadPlus：

```haskell
-- file: ch28/GameInventory.hs
maybeM :: MonadPlus m => m a -> m (Maybe a)
maybeM m = (Just `liftM` m) `mplus` return Nothing
```

这给了我们一个在更多情况下都有用的函数。

## I/O 和 STM

STM 单子禁止我们执行任意 I/O 动作，因为它们可能破坏该单子提供的原子性和隔离保证。当然，执行 I/O 的需求仍然存在——我们只需要非常小心地对待它。

大多数情况下，我们需要根据在 atomically 块内做出的决定来执行一些 I/O 动作。在这些情况下，正确的事情通常是返回一个数据片段从 atomically，这会告诉 IO 单子中的调用者下一步做什么。我们甚至可以返回要执行的动作，因为动作是第一类值：

```haskell
-- file: ch28/STMIO.hs
someAction :: IO a
stmTransaction :: STM (IO a)
stmTransaction = return someAction
doSomething :: IO a
doSomething = join (atomically stmTransaction)
```

偶尔我们需要从 STM 内部执行一个 I/O 操作。例如，从必须存在的文件中读取不可变数据并不会违反 STM 的隔离或原子性保证。在这些情况下，我们可以使用 unsafeIOToSTM 来执行 IO 动作。此函数由低级 GHC.Conc 模块导出，所以我们必须想办法使用它：

```haskell
ghci> :m +GHC.Conc
ghci> :type unsafeIOToSTM
unsafeIOToSTM :: IO a -> STM a
```

我们执行的 IO 动作不能启动另一个 atomically 事务。如果一个线程尝试嵌套事务，运行时系统将抛出异常。

由于类型系统无法帮助我们确保我们的 IO 代码正在做明智的事情，如果我们尽可能限制 unsafeIOToSTM 的使用，我们将最安全。这是一个可能在 atomically 块中使用 IO 时出现的典型错误：

```haskell
-- file: ch28/STMIO.hs
launchTorpedoes :: IO ()
notActuallyAtomic = do
  doStuff
  unsafeIOToSTM launchTorpedoes
  mightRetry
```

如果 mightRetry 块导致我们的事务重启，我们将多次调用 launchTorpedoes。事实上，我们无法预测它会被调用多少次，因为运行时系统为我们处理重试。解决方案是不在事务内执行这些类型的非幂等 I/O 操作。

## 线程间通信

除了基本的 TVar 类型，stm 包还提供了两种对线程间通信更有用的类型。TMVar 是 MVar 的 STM 等价物：它可以保存 Just a 值或 Nothing。TChan 类型是 Chan 的 STM 对应物，它实现了一个类型化的 FIFO 通道。

## 一个并发 Web 链接检查器

作为使用 STM 的一个实际例子，我们将开发一个检查 HTML 文件中坏链接的程序——即指向错误网页或已死亡服务器的 URL。这是一个通过并发来解决的好问题：如果我们尝试与一个已死亡的服务器通信，在我们的连接尝试超时之前可能需要长达两分钟。如果我们使用多个线程，当我们有一两个线程卡在与慢速或已死亡服务器通信时，我们仍然可以完成有用的工作。

我们不能简单地为每个 URL 创建一个线程，因为如果（正如我们预期的）大多数链接都是活跃的和响应性的，这可能会使我们的 CPU 或网络连接过载。相反，我们使用固定数量的工作线程，它们从队列中获取要下载的 URL：

```haskell
-- file: ch28/Check.hs
{-# LANGUAGE FlexibleContexts, GeneralizedNewtypeDeriving,
             PatternGuards #-}
import Control.Concurrent (forkIO)
import Control.Concurrent.STM
import Control.Exception (catch, finally)
import Control.Monad.Error
import Control.Monad.State
import Data.Char (isControl)
import Data.List (nub)
import Network.URI
import Prelude hiding (catch)
import System.Console.GetOpt
import System.Environment (getArgs)
import System.Exit (ExitCode(..), exitWith)
import System.IO (hFlush, hPutStrLn, stderr, stdout)
import Text.Printf (printf)
import qualified Data.ByteString.Lazy.Char8 as B
import qualified Data.Set as S
-- This requires the HTTP package, which is not bundled with GHC
import Network.HTTP
type URL = B.ByteString
data Task = Check URL | Done
```

我们的 main 函数为程序提供了顶层脚手架：

```haskell
-- file: ch28/Check.hs
main :: IO ()
main = do
    (files,k) <- parseArgs
    let n = length files
    -- count of broken links
    badCount <- newTVarIO (0 :: Int)
    -- for reporting broken links
    badLinks <- newTChanIO
    -- for sending jobs to workers
    jobs <- newTChanIO
    -- the number of workers currently running
    workers <- newTVarIO k
    -- one thread reports bad links to stdout
    forkIO $ writeBadLinks badLinks
    -- start worker threads
    forkTimes k workers (worker badLinks jobs badCount)
    -- read links from files, and enqueue them as jobs
    stats <- execJob (mapM_ checkURLs files)
                     (JobState S.empty 0 jobs)
    -- enqueue "please finish" messages
    atomically $ replicateM_ k (writeTChan jobs Done)
    waitFor workers
    broken <- atomically $ readTVar badCount
    printf fmt broken
               (linksFound stats)
               (S.size (linksSeen stats))
               n
  where
    fmt   = "Found %d broken links. " ++
            "Checked %d links (%d unique) in %d files.\n"
```

当我们在 IO 单子中时，我们可以使用 newTVarIO 函数创建新的 TVar 值。也有用于创建 TMVar 和 TChan 值的对应函数。

注意我们使用 printf 函数在最后打印报告。与 C 中的对应函数不同，Haskell 的 printf 函数可以在运行时检查其参数类型和数量：

```haskell
ghci> :m +Text.Printf
ghci> printf "%d and %d\n" (3::Int)
3 and *** Exception: Printf.printf: argument list ended prematurely
ghci> printf "%s and %d\n" "foo" (3::Int)
foo and 3
```

尝试在 ghci 提示符下评估 `printf "%d" True`，看看会发生什么。

几个短函数支持 main：

```haskell
-- file: ch28/Check.hs
modifyTVar_ :: TVar a -> (a -> a) -> STM ()
modifyTVar_ tv f = readTVar tv >>= writeTVar tv . f
forkTimes :: Int -> TVar Int -> IO () -> IO ()
forkTimes k alive act =
  replicateM_ k . forkIO $
    act
    `finally`
    (atomically $ modifyTVar_ alive (subtract 1))
```

forkTimes 函数启动若干相同的工作线程，每次线程退出时减少"存活"计数。我们使用 finally 组合器来确保计数总是减少，无论线程如何终止。

接下来，writeBadLinks 函数将每个坏链接或死链接打印到 stdout：

```haskell
-- file: ch28/Check.hs
writeBadLinks :: TChan String -> IO ()
writeBadLinks c =
  forever $
    atomically (readTChan c) >>= putStrLn >> hFlush stdout
```

我们在前面的代码中使用 forever 组合器，它重复一个动作无尽次：

```haskell
ghci> :m +Control.Monad
ghci> :type forever
forever :: (Monad m) => m a -> m ()
```

我们的 waitFor 函数使用 check，如果其参数评估为 False 则调用 retry：

```haskell
-- file: ch28/Check.hs
waitFor :: TVar Int -> IO ()
waitFor alive = atomically $ do
  count <- readTVar alive
  check (count == 0)
```

## 检查链接

这是一个检查链接状态的简单函数。此代码类似于我们在第 22 章开发的播客下载器，有几个小差异：

```haskell
-- file: ch28/Check.hs
getStatus :: URI -> IO (Either String Int)
getStatus = chase (5 :: Int)
  where
    chase 0 _ = bail "too many redirects"
    chase n u = do
      resp <- getHead u
      case resp of
        Left err -> bail (show err)
        Right r ->
          case rspCode r of
            (3,_,_) ->
               case findHeader HdrLocation r of
                 Nothing -> bail (show r)
                 Just u' ->
                   case parseURI u' of
                     Nothing -> bail "bad URL"
                     Just url -> chase (n-1) url
            (a,b,c) -> return . Right $ a * 100 + b * 10 + c
    bail = return . Left
getHead :: URI -> IO (Result Response)
getHead uri = simpleHTTP Request { rqURI = uri,
                                   rqMethod = HEAD,
                                   rqHeaders = [],
                                   rqBody = "" }
```

我们只跟随 HTTP 重定向响应几次，以避免无尽的重定向循环。为了确定 URL 是否有效，我们使用 HTTP 标准的 HEAD 动词，它比完整 GET 占用更少的带宽。

这段代码有我们熟悉的"从屏幕右侧走出去"的风格。以下是一个通过 ErrorT 单子转换器和几个通用函数提供更高清晰度的重写：

```haskell
-- file: ch28/Check.hs
getStatusE = runErrorT . chase (5 :: Int)
  where
    chase :: Int -> URI -> ErrorT String IO Int
    chase 0 _ = throwError "too many redirects"
    chase n u = do
      r <- embedEither show =<< liftIO (getHead u)
      case rspCode r of
        (3,_,_) -> do
            u'  <- embedMaybe (show r)  $ findHeader HdrLocation r
            url <- embedMaybe "bad URL" $ parseURI u'
            chase (n-1) url
        (a,b,c) -> return $ a*100 + b*10 + c
-- This function is defined in Control.Arrow.
left :: (a -> c) -> Either a b -> Either c b
left f (Left x)  = Left (f x)
left _ (Right x) = Right x
-- Some handy embedding functions.
embedEither :: (MonadError e m) => (s -> e) -> Either s a -> m a
embedEither f = either (throwError . f) return
embedMaybe :: (MonadError e m) => e -> Maybe a -> m a
embedMaybe err = maybe (throwError err) return
```

## 工作线程

每个工作线程从共享队列中读取一个任务。它要么检查给定的 URL，要么退出：

```haskell
-- file: ch28/Check.hs
worker :: TChan String -> TChan Task -> TVar Int -> IO ()
worker badLinks jobQueue badCount = loop
  where
    -- Consume jobs until we are told to exit.
    loop = do
        job <- atomically $ readTChan jobQueue
        case job of
            Done  -> return ()
            Check x -> checkOne (B.unpack x) >> loop
    -- Check a single link.
    checkOne url = case parseURI url of
        Just uri -> do
            code <- getStatus uri `catch` (return . Left . show)
            case code of
                Right 200 -> return ()
                Right n   -> report (show n)
                Left err  -> report err
        _ -> report "invalid URL"
        where report s = atomically $ do
                           modifyTVar_ badCount (+1)
                           writeTChan badLinks (url ++ " " ++ s)
```

## 查找链接

我们围绕在 IO 单子上堆叠的状态单子转换器来构建我们的链接查找。我们的状态跟踪我们已经见过的链接（这样我们不会多次检查重复链接）、我们遇到的链接总数，以及我们应该添加要检查的链接的队列：

```haskell
-- file: ch28/Check.hs
data JobState = JobState { linksSeen :: S.Set URL,
                           linksFound :: Int,
                           linkQueue :: TChan Task }
newtype Job a = Job { runJob :: StateT JobState IO a }
    deriving (Monad, MonadState JobState, MonadIO)
execJob :: Job a -> JobState -> IO JobState
execJob = execStateT . runJob
```

严格来说，对于一个小型独立程序，我们不需要这个 newtype 包装器，但这里我们将其作为良好实践的一个例子（它只花费几行代码）。

main 函数将 checkURLs 映射到每个输入文件，所以 checkURLs 只需要读取单个文件：

```haskell
-- file: ch28/Check.hs
checkURLs :: FilePath -> Job ()
checkURLs f = do
    src <- liftIO $ B.readFile f
    let urls = extractLinks src
    filterM seenURI urls >>= sendJobs
    updateStats (length urls)
updateStats :: Int -> Job ()
updateStats a = modify $ \s ->
    s { linksFound = linksFound s + a }
-- | Add a link to the set we have seen.
insertURI :: URL -> Job ()
insertURI c = modify $ \s ->
    s { linksSeen = S.insert c (linksSeen s) }
-- | If we have seen a link, return False.  Otherwise, record that we
-- have seen it, and return True.
seenURI :: URL -> Job Bool
seenURI url = do
    seen <- (not . S.member url) `liftM` gets linksSeen
    insertURI url
    return seen
sendJobs :: [URL] -> Job ()
sendJobs js = do
    c <- gets linkQueue
    liftIO . atomically $ mapM_ (writeTChan c . Check) js
```

我们的 extractLinks 函数不尝试正确解析 HTML 或文本文件。相反，它寻找看起来像 URL 的字符串，并将其视为"足够好"：

```haskell
-- file: ch28/Check.hs
extractLinks :: B.ByteString -> [URL]
extractLinks = concatMap uris . B.lines
  where uris s      = filter looksOkay (B.splitWith isDelim s)
        isDelim c   = isControl c || c `elem` " <>\"{}|\\^[]`"
        looksOkay s = http `B.isPrefixOf` s
        http        = B.pack "http:"
```

## 命令行解析

为了解析我们的命令行参数，我们使用 System.Console.GetOpt 模块。它提供了用于解析参数的有用代码，但使用起来稍微复杂：

```haskell
-- file: ch28/Check.hs
data Flag = Help | N Int
            deriving Eq
parseArgs :: IO ([String], Int)
parseArgs = do
    argv <- getArgs
    case parse argv of
        ([], files, [])                     -> return (nub files, 16)
        (opts, files, [])
            | Help `elem` opts              -> help
            | [N n] <- filter (/=Help) opts -> return (nub files, n)
        (_,_,errs)                          -> die errs
  where
    parse argv = getOpt Permute options argv
    header     = "Usage: urlcheck [-h] [-n n] [file ...]"
    info       = usageInfo header options
    dump       = hPutStrLn stderr
    die errs   = dump (concat errs ++ info) >> exitWith (ExitFailure 1)
    help       = dump info                  >> exitWith ExitSuccess
```

getOpt 函数接受三个参数：

- 参数顺序，指定选项是否可以与其他参数混合（Permute，我们之前使用的）或必须出现在它们之前。
- 选项定义列表。每个包括选项的短名称列表、长名称列表、选项描述（例如，是否接受参数）以及给用户的解释。
- 参数和选项列表，如 getArgs 返回的那样。

函数返回一个三元组，包括解析的选项、剩余参数和发生的任何错误消息。

我们使用 Flag 代数数据类型来表示我们的程序可以接受的选项：

```haskell
-- file: ch28/Check.hs
options :: [OptDescr Flag]
options = [ Option ['h'] ["help"] (NoArg Help)
                   "Show this help message",
            Option ['n'] []       (ReqArg (\s -> N (read s)) "N")
                   "Number of concurrent connections (default 16)" ]
```

我们的选项列表描述了我们接受的每个选项。每个描述必须能够创建一个 Flag 值。看一下我们在前面的代码中对 NoArg 和 ReqArg 的使用。这些是 GetOpt 模块的 ArgDescr 类型的构造函数：

```haskell
-- file: ch28/GetOpt.hs
data ArgDescr a = NoArg a
                | ReqArg (String -> a) String
                | OptArg (Maybe String -> a) String
```

这些构造函数具有以下含义：

NoArg
接受一个参数来表示这个选项。在我们的例子中，如果用户使用 -h 或 --help 调用我们的程序，我们将使用值 Help。

ReqArg
接受一个将必需参数映射到值的函数。它的第二个参数在打印帮助时使用。在这里，我们将一个字符串转换为整数，并将其传递给我们的 Flag 类型的 N 构造函数。

OptArg
类似于 ReqArg 构造函数，但允许使用可以不带参数使用的选项。

## 模式守卫

我们在 parseArgs 的定义中偷偷放入了最后一个语言扩展。模式守卫让我们编写更简洁的守卫表达式。它们通过 PatternGuards 语言扩展启用。

模式守卫有三个组成部分：一个模式、一个 <- 符号和一个表达式。表达式被求值并与模式匹配。如果匹配，模式中存在的任何变量都被绑定。我们可以通过用逗号分隔在单个守卫中混合模式守卫和普通 Bool 守卫表达式：

```haskell
-- file: ch28/PatternGuard.hs
{-# LANGUAGE PatternGuards #-}
testme x xs | Just y <- lookup x xs, y > 3 = y
            | otherwise                    = 0
```

在这个例子中，如果关联的键 x 存在，我们从 alist xs 返回一个值，条件是值大于 3。这个定义等价于以下：

```haskell
-- file: ch28/PatternGuard.hs
testme_noguards x xs = case lookup x xs of
                         Just y | y > 3 -> y
                         _              -> 0
```

模式守卫让我们将守卫和 case 表达式的集合"折叠"成单个守卫，允许我们编写更简洁和描述性的守卫。

## STM 的实践方面

到目前为止，我们一直对 STM 给我们的具体好处保持沉默。最明显的是它组合得多么好——要向事务添加代码，我们只需使用我们通常的 monad 构建块，(>>=) 和 (>>)。

可组合性的概念对于构建模块化软件至关重要。如果我们获取两个单独正确工作的代码片段，两个的组合也应该正确。虽然普通线程编程使可组合性变得不可能，STM 将其恢复为我们可以依赖的关键假设。

STM 单子防止我们意外执行非事务性 I/O 动作。我们不需要担心锁顺序，因为我们的代码不包含锁。我们可以忘记丢失的唤醒，因为我们没有条件变量。如果抛出异常，我们可以使用 catchSTM 捕获它，或者从我们的事务中弹出，留下我们的状态不变。最后，retry 和 orElse 函数给了我们一些构建代码的优美方式。

使用 STM 的代码不会死锁，但线程之间有可能在某种程度上相互饿死。长运行的事务可能导致另一个事务频繁重试，以至于它只会取得相对较少的进展。为了解决这样的问题，尽可能缩短你的事务，同时保持数据一致。

## 习惯于放弃控制

无论是并发还是内存管理，有时我们必须保留控制：一些软件必须对延迟或内存占用做出可靠保证，所以我们将被迫花费额外的时间和精力来管理和调试显式代码。对于许多有趣的、实际的软件用途，垃圾收集和 STM 做得非常好。

STM 不是一个完整的灵丹妙药。将它与使用垃圾收集进行内存管理进行比较是有用的。当我们放弃显式内存管理而支持垃圾收集时，我们放弃控制以换取更安全的代码。同样，使用 STM，我们放弃低级细节，以换取我们更有希望理解的代码。

## 使用不变量

STM 无法消除某些类别的错误。例如，如果我们在一个 atomically 块中从账户取钱，返回 IO 单子，然后在另一个 atomically 块中将其存入另一个账户，我们的代码将存在不一致。在某个时间窗口内，钱既不在第一个账户也不在第二个账户中。

```haskell
-- file: ch28/GameInventory.hs
bogusTransfer qty fromBal toBal = do
  fromQty <- atomically $ readTVar fromBal
  -- window of inconsistency
  toQty   <- atomically $ readTVar toBal
  atomically $ writeTVar fromBal (fromQty - qty)
  -- window of inconsistency
  atomically $ writeTVar toBal   (toQty + qty)
bogusSale :: Item -> Gold -> Player -> Player -> IO ()
bogusSale item price buyer seller = do
  atomically $ giveItem item (inventory seller) (inventory buyer)
  bogusTransfer price (balance buyer) (balance seller)
```

在并发程序中，这些类型的问题众所周知是很难发现和复制的。例如，我们在这里描述的不一致性通常只会在很短的时间内发生。这样的问题通常拒绝在开发过程中出现，只在重负载下的现场发生。

alwaysSucceeds 函数让我们定义一个不变量，即我们数据的一个必须始终为真的属性：

```haskell
ghci> :type alwaysSucceeds
alwaysSucceeds :: STM a -> STM ()
```

当我们创建一个不变量时，它将立即被检查。要失败，不变量必须引发一个异常。更重要的是，不变量随后将在每个事务结束时自动检查。如果它在任何时候失败，事务将被中止，不变量引发的异常将被传播。这意味着当我们的一个不变量被违反时，我们会立即得到反馈。

例如，这里有一些函数来从头开始填充本章游戏世界的玩家：

```haskell
-- file: ch28/GameInventory.hs
newPlayer :: Gold -> HitPoint -> [Item] -> STM Player
newPlayer balance health inventory =
    Player `liftM` newTVar balance
              `ap` newTVar health
              `ap` newTVar inventory
populateWorld :: STM [Player]
populateWorld = sequence [ newPlayer 20 20 [Wand, Banjo],
                          newPlayer 10 12 [Scroll] ]
```

这个函数返回一个不变量，我们可以用它来确保世界的资金余额始终一致——任何时候的余额应该与世界创建时的余额相同：

```haskell
-- file: ch28/GameInventory.hs
consistentBalance :: [Player] -> STM (STM ())
consistentBalance players = do
    initialTotal <- totalBalance
    return $ do
      curTotal <- totalBalance
      when (curTotal /= initialTotal) $
        error "inconsistent global balance"
  where totalBalance   = foldM addBalance 0 players
        addBalance a b = (a+) `liftM` readTVar (balance b)
```

让我们编写一个小函数来测试这个：

```haskell
-- file: ch28/GameInventory.hs
tryBogusSale = do
  players@(alice:bob:_) <- atomically populateWorld
  atomically $ alwaysSucceeds =<< consistentBalance players
  bogusSale Wand 5 alice bob
```

如果我们在 ghci 中运行它，它应该检测到我们不正确地在 bogusTransfer 函数中使用 atomically 引起的不一致：

```haskell
ghci> tryBogusSale
*** Exception: inconsistent global balance
```
