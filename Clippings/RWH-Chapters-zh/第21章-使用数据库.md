## 第二十一章：使用数据库

从网络论坛到播客捕捉器甚至备份程序，一切都经常使用数据库进行持久存储。基于 SQL 的数据库通常非常方便：它们快速，可以从小型扩展到大型，可以跨网络操作，通常有助于处理锁定和事务，甚至可以为应用程序提供故障转移和冗余改进。数据库有許多不同的形式：大型商业数据库（如 Oracle）、开源引擎（如 PostgreSQL 或 MySQL），甚至还有嵌入式引擎（如 Sqlite）。

因为数据库非常重要，所以 Haskell 对它们的支持也很重要。在本章中，我们将向你介绍 Haskell 中用于处理数据库的框架之一。我们还将使用这个框架开始构建一个播客下载器，我们将在第 22 章中进一步开发它。

## HDBC 概述

数据库栈的底部是数据库引擎，负责实际将数据存储在磁盘上。著名的数据库引擎包括 PostgreSQL、MySQL 和 Oracle。

大多数现代数据库引擎支持结构化查询语言（SQL）作为将数据存入和取出关系数据库的标准方式。本书不会提供关于 SQL 或关系数据库管理的教程。*

一旦你有了支持 SQL 的数据库引擎，你就需要一种与其通信的方式。每个数据库都有自己的协议。由于 SQL 在各个数据库之间相当恒定，因此可以创建一个通用接口，使用每个单独协议的驱动程序。

* 如果你没有使用 SQL 的经验，Alan Beaulieu 的《Learning SQL》和 Kevin Kline 等人的《SQL in a Nutshell》（均为 O’Reilly 出版）可能很有用。

Haskell 有几种不同的数据库框架可用，有些在 others 之上提供高层抽象。对于本章，我们将集中讨论 Haskell 数据库连接系统（HDBC）。HDBC 是一个数据库抽象库。也就是说，你可以编写使用 HDBC 的代码，并且可以访问存储在几乎任何 SQL 数据库中的数据，几乎不需要修改。† 即使你永远不需要切换底层数据库引擎，HDBC 的驱动程序系统也能通过单一接口为你提供大量选择。

另一个用于 Haskell 的数据库抽象库是 HSQL，它与 HDBC 有类似的目的。还有一个名为 HaskellDB 的更高层框架，它位于 HDBC 或 HSQL 之上，旨在帮助程序员隔离使用 SQL 的细节。然而，由于其设计限制了某些——尽管相当常见——数据库访问模式，它的吸引力没有那么广。最后，Takusen 是一个使用"左折叠"方法从数据库读取数据的框架。

## 安装 HDBC 和驱动程序

要使用 HDBC 连接到给定数据库，你至少需要两个包：通用接口和特定数据库的驱动程序。你可以从 Hackage（http://hackage.haskell.org/）获取通用 HDBC 包和其他所有驱动程序。‡ 对于本章，我们将使用 HDBC 版本 1.1.3。

你还需要一个数据库后端和后端驱动程序。对于本章，我们将使用 Sqlite 版本 3。Sqlite 是一个嵌入式数据库，所以它不需要单独的服务器，而且易于设置。许多操作系统已经附带了 Sqlite 版本 3。如果你的没有，你可以从 http://www.sqlite.org/ 下载它。HDBC 主页上有已知 HDBC 后端驱动程序的链接。Sqlite 版本 3 的特定驱动程序可以从 Hackage 获得。

如果你想将 HDBC 与其他数据库一起使用，请查看 http://software.complete.org/hdbc/wiki/KnownDrivers 上的 HDBC 已知驱动程序页面。在那里你会找到一个 ODBC 绑定的链接，它让你可以连接到 virtually 任何平台（Windows、POSIX 等）上的 virtually 任何数据库。你还会找到 PostgreSQL 绑定的链接。MySQL 通过 ODBC 绑定支持，MySQL 用户的具体信息可以在 HDBC-ODBC API 文档（http://software.complete.org/static/hdbc-odbc/doc/HDBC-odbc/）中找到。

† 这假定你限制自己使用标准 SQL。

‡ 有关安装 Haskell 软件的更多信息，请参阅第 646 页的"安装 Haskell 软件"。

## 连接到数据库

要连接到数据库，你将使用数据库后端驱动程序的连接函数。每个数据库都有自己独特的连接方法。初始连接通常是你直接从后端驱动程序模块调用的唯一内容。

数据库连接函数将返回一个数据库句柄。这个句柄的精确类型可能因驱动程序而异，但它始终是 IConnection 类型类的实例。你将用于操作数据库的所有函数都将适用于任何是 IConnection 实例的类型。当你完成与数据库的对话时，调用 disconnect 函数断开与它的连接。以下是连接到 Sqlite 数据库的示例：

```ghci
ghci> :module Database.HDBC Database.HDBC.Sqlite3
ghci> conn <- connectSqlite3 "test1.db"

Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
Loading package HDBC-1.1.4 ... linking ... done.
Loading package HDBC-sqlite3-1.1.4.0 ... linking ... done.
ghci> :type conn
conn :: Connection
ghci> disconnect conn
```

## 事务

大多数现代 SQL 数据库都有事务的概念。事务旨在确保修改的所有组件都被应用，或者都不被应用。此外，事务有助于防止其他访问相同数据库的进程看到正在进行的修改中的部分数据。

许多数据库要求你在更改出现在磁盘上之前显式提交所有更改，或者以自动提交模式运行。自动提交模式在每个语句之后运行一个隐式提交。这可能使不习惯事务数据库的程序员更容易适应，但对于实际想要使用多语句事务的人来说，这只是一种阻碍。

HDBC 有意不支持自动提交模式。当你修改数据库中的数据时，你必须显式地将其提交到磁盘。在 HDBC 中有两种方法可以做到这一点：你可以调用 commit 当你准备好将数据写入磁盘时，或者你可以使用 withTransaction 函数包装你的修改代码。withTransaction 将在你的函数成功完成时导致数据被提交。

有时在你将数据写入数据库时会出现问题。也许你从数据库收到错误或发现数据有问题。在这些情况下，你可以"回滚"你的更改。这将导致你自上次提交或回滚以来的所有更改被遗忘。在 HDBC 中，你可以调用 rollback 函数来执行此操作。如果你使用 withTransaction，任何未捕获的异常都会导致发出回滚。

请注意，回滚操作只回滚自上次提交、回滚或 withTransaction 以来的更改。数据库不会像版本控制系统那样维护详细历史。你将在本章后面看到提交的示例。

一个流行的数据库 MySQL，在其默认表类型下不支持事务。在其默认配置中，MySQL 将静默忽略对 commit 或 rollback 的调用，并立即将所有更改提交到磁盘。HDBC ODBC 驱动程序有说明如何配置 MySQL 以向 HDBC 表明它不支持事务，这将导致 commit 和 rollback 产生错误。或者，你可以随 MySQL 使用支持事务的 InnoDB 表。建议将 InnoDB 表与 HDBC 一起使用。

## 简单查询

SQL 中一些最简单的查询涉及不返回任何数据的语句。这些查询可用于创建表、插入数据、删除数据和设置数据库参数。

向数据库发送查询的最基本函数是 run。这个函数接受一个 IConnection、一个表示查询本身的 String 和一个参数列表。让我们用它来在数据库中设置一些东西：

```ghci
ghci> :module Database.HDBC Database.HDBC.Sqlite3
ghci> conn <- connectSqlite3 "test1.db"
Loading package array-0.1.0.0 ... linking ... done.
Loading package containers-0.1.0.2 ... linking ... done.
Loading package bytestring-0.9.0.1.1 ... linking ... done.
Loading package old-locale-1.0.0.0 ... linking ... done.
Loading package old-time-1.0.0.0 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
Loading package HDBC-1.1.4 ... linking ... done.
Loading package HDBC-sqlite3-1.1.4.0 ... linking ... done.
ghci> run conn "CREATE TABLE test (id INTEGER NOT NULL, desc VARCHAR(80))" []
0
ghci> run conn "INSERT INTO test (id) VALUES (0)" []
1
ghci> commit conn
ghci> disconnect conn
```

在这个示例中，连接到数据库后，我们首先创建了一个名为 test 的表。然后我们在表中插入了一行数据。最后，我们提交了更改并断开了与数据库的连接。请注意，如果我们没有调用 commit，根本不会有任何最终更改被写入数据库。

run 函数返回每个查询修改的行数。对于第一个创建表的查询，没有行被修改。第二个查询插入了一行，所以 run 返回了 1。

## SqlValue

在继续之前，我们需要讨论 HDBC 中引入的一个数据类型：SqlValue。由于 Haskell 和 SQL 都是强类型系统，HDBC 尽可能保留类型信息。同时，Haskell 和 SQL 类型并不完全相同。此外，不同的数据库在表示日期或字符串中的特殊字符等方面有不同的方式。

SqlValue 是一个数据类型，有许多构造函数，如 SqlString、SqlBool、SqlNull、SqlInteger 等。这让你可以在到数据库的参数列表中表示各种类型的数据，并在返回的结果中查看各种类型的数据，仍然可以将所有内容存储在列表中。有便捷函数 toSql 和 fromSql，你通常会使用它们。如果你关心数据的精确表示，如果你需要，你仍然可以手工构造 SqlValue 数据。

## 查询参数

HDBC 和大多数数据库一样，支持查询中可替换参数的概念。使用可替换参数有三个主要好处：它们防止 SQL 注入攻击或在输入包含引号字符时出现问题，它们在重复执行类似查询时提高性能，并且它们允许轻松且可移植地将数据插入查询。

假设你想向我们的新表 test 中添加数千行。你可以发出类似 INSERT INTO test VALUES (0, 'zero') 和 INSERT INTO test VALUES (1, 'one') 的查询。这会强制数据库服务器分别解析每个 SQL 语句。如果你可以用占位符替换这两个值，服务器可以解析 SQL 查询一次，然后仅用不同数据执行多次。

第二个问题涉及转义字符。如果你想插入字符串"I don't like 1"怎么办？SQL 使用单引号字符来表示字段的结束。大多数 SQL 数据库会要求你将其写为'I don''t like 1'。但其他特殊字符（如反斜杠）的规则在不同的数据库之间有所不同。HDBC 可以为你处理所有这些，而不是自己编写代码来做到这一点。让我们看一个示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> run conn "INSERT INTO test VALUES (?, ?)" [toSql 0, toSql "zero"]
1
ghci> commit conn
ghci> disconnect conn
```

这个示例中 INSERT 查询中的问号是占位符。然后我们传递将放在那里的参数。run 接受一个 SqlValue 列表，所以我们使用 toSql 将每个项目转换为 SqlValue。HDBC 自动处理将 String "zero" 转换为正在使用的数据库的正确表示。

这种方法在插入大量数据时实际上不会获得任何性能提升。为此，我们需要对创建 SQL 查询的过程有更多控制。我们将在下一节中讨论这个问题。

## 使用可替换参数

可替换参数仅在查询中服务器期望值的部分工作，例如 SELECT 语句中的 WHERE 子句或 INSERT 语句的值。你不能说 run "SELECT * from ?" [toSql "tablename"] 并期望它工作。表名不是值，大多数数据库不会接受此语法。在实践中这不是一个大问题，因为很少需要以这种方式替换不是值的东西。

## 预处理语句

HDBC 定义了一个 prepare 函数，它将准备一个 SQL 查询，但尚不将参数绑定到查询。prepare 返回一个表示编译后查询的 Statement。

一旦你有一个 Statement，你可以用它做很多事情。你可以调用 execute 一次或多次。在调用返回数据的查询的 execute 后，你可以使用其中一个 fetch 函数来检索该数据。诸如 run 和 quickQuery' 之的函数在内部使用语句执行；它们只是让你快速执行常见任务的快捷方式。当你需要对待发生的事情有更多控制时，你可以使用 Statement 而不是 run 之类的函数。

让我们看看使用语句插入具有单个查询的多个值。以下是一个示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> stmt <- prepare conn "INSERT INTO test VALUES (?, ?)"
ghci> execute stmt [toSql 1, toSql "one"]
1
ghci> execute stmt [toSql 2, toSql "two"]
1
ghci> execute stmt [toSql 3, toSql "three"]
1
ghci> execute stmt [toSql 4, SqlNull]
1
ghci> commit conn
ghci> disconnect conn
```

在这里，我们创建一个预处理语句并将其称为 stmt。然后我们执行该语句四次，每次传递不同的参数。这些参数按顺序用于替换原始查询字符串中的问号。最后，我们提交更改并断开与数据库的连接。

HDBC 还提供了一个函数 executeMany，在这种情况下可能很有用。executeMany 简单地接受一个行数据列表来调用语句。以下是一个示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> stmt <- prepare conn "INSERT INTO test VALUES (?, ?)"
ghci> executeMany stmt [[toSql 5, toSql "five's nice"], [toSql 6, SqlNull]]
ghci> commit conn
ghci> disconnect conn
```

## 更高效的执行

在服务器上，大多数数据库将能够进行优化，它们可以应用于 executeMany，这样它们只需编译此查询字符串一次，而不是两次。§ 这可以在一次插入大量数据时带来显著的性能提升。一些数据库也可以将此优化应用于 execute，但不是全部。

## 读取结果

到目前为止，我们已经讨论了插入或更改数据的查询。现在让我们回顾一下从数据库获取数据回来的方式。quickQuery' 函数的类型看起来非常类似于 run，但它返回的是结果列表而不是更改的行数。quickQuery' 通常与 SELECT 语句一起使用。让我们看一个示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> quickQuery' conn "SELECT * from test where id < 2" []
[[SqlString "0",SqlNull],[SqlString "0",SqlString "zero"],
[SqlString "1",SqlString "one"],[SqlString "0",SqlNull],
[SqlString "0",SqlString "zero"],[SqlString "1",SqlString "one"]]
ghci> disconnect conn
```

quickQuery' 与我们刚才讨论的可替换参数一起工作。在这种情况下，我们没有使用任何，所以要替换的值集是 quickQuery' 调用末尾的空列表。quickQuery' 返回一个行列表，其中每行本身表示为 [SqlValue]。行中的值按数据库返回的顺序列出。你可以根据需要使用 fromSql 将它们转换为常规 Haskell 类型。

输出有点难读。让我们扩展这个示例来格式化结果很好。以下是执行此操作的代码：

```haskell
-- file: ch21/query.hs
import Database.HDBC.Sqlite3 (connectSqlite3)
import Database.HDBC
{- | 定义一个函数，接受一个表示要查找的最大 id 值的整数。
将从测试数据库获取所有匹配的行，
并以友好格式将它们打印到屏幕。 -}
query :: Int -> IO ()
query maxId =
    do -- 连接到数据库
       conn <- connectSqlite3 "test1.db"
       -- 运行查询并将结果存储在 r 中
       r <- quickQuery' conn
            "SELECT id, desc from test where id <= ? ORDER BY id, desc"
            [toSql maxId]
       -- 将每一行转换为一个 String
       let stringRows = map convRow r

       -- 打印出这些行
       mapM_ putStrLn stringRows
       -- 并断开与数据库的连接
       disconnect conn
    where convRow :: [SqlValue] -> String
          convRow [sqlId, sqlDesc] =
              show intid ++ ": " ++ desc
              where intid = (fromSql sqlId)::Integer
                    desc = case fromSql sqlDesc of
                             Just x -> x
                             Nothing -> "NULL"
          convRow x = fail $ "Unexpected result: " ++ show x
```

这个程序做的工作与我们在 ghci 中的示例大致相同，但有一个新的添加：convRow 函数。这个函数从数据库获取一行数据并将其转换为 String。这个字符串可以很容易地打印出来。

注意我们如何直接从 fromSql 获取 intid，但将 fromSql sqlDesc 作为 Maybe String 类型处理。如果你记得，我们声明了这个表的第一列永远不能包含 NULL 值，但第二列可以。因此，我们可以安全地忽略第一列中 NULL 的可能性，但不能忽略第二列。可以使用 fromSql 将第二列直接转换为 String，它甚至可以工作——直到遇到该位置有 NULL 的行。这将导致运行时异常。因此，我们将 SQL NULL 值转换为字符串"NULL"。打印时，这将与 SQL 字符串'NULL'难以区分，但这对这个示例来说是可以接受的。让我们尝试在 ghci 中调用这个函数：

```ghci
ghci> :load query.hs
[1 of 1] Compiling Main             ( query.hs, interpreted )
Ok, modules loaded: Main.
ghci> query 2
0: NULL
0: NULL
0: zero
0: zero
1: one
1: one
2: two
2: two
```

## 使用语句读取

正如我们在第 498 页的"预处理语句"中讨论的，你可以使用语句进行读取。有多种方式可以从语句读取数据，在某些情况下可能很有用。像 run 一样，quickQuery' 是一个方便函数，实际上它使用语句来完成其任务。

要创建用于读取的语句，我们像对待将用于写入数据的语句一样使用 prepare。你也使用 execute 在数据库服务器上执行它。然后，我们可以使用各种函数从 Statement 读取数据。fetchAllRows' 函数返回 [[SqlValue]]，就像 quickQuery' 一样。还有一个名为 sFetchAllRows' 的函数，它在返回之前将每列数据转换为 Maybe String。最后，还有 fetchAllRowsAL'，它为每列返回 (String, SqlValue) 对。String 是数据库返回的列名；请参阅第 502 页的"数据库元数据"以了解获取列名的其他方法。

你也可以通过调用 fetchRow 一次读取一行数据，它返回 IO (Maybe [SqlValue])。如果所有结果都已被读取，它将是 Nothing，否则是一行。

## 惰性读取

回到第 178 页的"惰性 I/O"，我们讨论了文件的惰性 I/O。从数据库惰性读取数据也是可能的。当处理返回大量数据的查询时，这可能特别有用。通过惰性读取数据，你仍然可以使用 fetchAllRows 等便捷函数，而不必在数据进入时手动读取每一行。如果我们在使用数据时小心，我们可以避免将所有结果缓冲在内存中。

但是，从数据库惰性读取比从文件读取更复杂。当我们完成从文件惰性读取数据时，文件被关闭——这通常很好。当我们完成从数据库惰性读取数据时，数据库连接仍然打开——例如，你可能正在向其提交其他查询。一些数据库甚至可以支持多个同时查询，所以 HDBC 不能在我们完成时简单地关闭连接。

当使用惰性读取时，在我们尝试关闭连接或执行新查询之前，我们完成读取整个数据集是至关重要的。我们鼓励你在可能的地方使用严格函数或逐行处理，以最大程度地减少与惰性读取的复杂交互。

如果你不熟悉 HDBC 或惰性读取的概念，但有大量数据要读取，重复调用 fetchRow 可能更容易理解。惰性读取是一个强大而有用的工具，但必须正确使用。

要从数据库惰性读取，我们使用与之前相同的函数，但没有重音符号。例如，使用 fetchAllRows 而不是 fetchAllRows'。惰性函数的类型与其严格的对应函数相同。以下是惰性读取的示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> stmt <- prepare conn "SELECT * from test where id < 2"
ghci> execute stmt []
0
ghci> results <- fetchAllRowsAL stmt
[[("id",SqlString "0"),("desc",SqlNull)],[("id",SqlString "0"),
("desc",SqlString "zero")],[("id",SqlString "1"),("desc",SqlString "one")]
,[("id",SqlString "0"),("desc",SqlNull)],[("id",SqlString "0"),
("desc",SqlString "zero")],[("id",SqlString "1"),("desc",SqlString "one")]]
ghci> mapM_ print results
[("id",SqlString "0"),("desc",SqlNull)]
[("id",SqlString "0"),("desc",SqlString "zero")]
[("id",SqlString "1"),("desc",SqlString "one")]
[("id",SqlString "0"),("desc",SqlNull)]
[("id",SqlString "0"),("desc",SqlString "zero")]
[("id",SqlString "1"),("desc",SqlString "one")]
ghci> disconnect conn
```

请注意，你也可以在这里使用 fetchAllRowsAL'。但是，如果你有大量数据集要读取，它会消耗大量内存。通过惰性读取数据，我们可以使用恒定量的内存打印出非常大的结果集。使用惰性版本，结果会被分块求值；使用严格版本，所有结果都会预先读取，存储在 RAM 中，然后打印。

## 数据库元数据

有时程序了解数据库本身的信息可能很有用。例如，程序可能想查看存在哪些表，以便它可以自动创建缺失的表或升级数据库模式。在某些情况下，程序可能需要根据所使用的数据库后端改变其行为。

首先，有一个 getTables 函数，它将获取数据库中已定义表的列表。你也可以使用 describeTable 函数，它将提供有关给定表中已定义列的信息。

你可以通过调用 dbServerVer 和 proxiedClientName 等函数来了解所使用的数据库服务器。例如，dbTransactionSupport 函数可用于确定给定数据库是否支持事务。让我们看看其中一些项目的示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> getTables conn
["test"]
ghci> proxiedClientName conn
"sqlite3"
ghci> dbServerVer conn
"3.5.6"
ghci> dbTransactionSupport conn
True
ghci> disconnect conn
```

你还可以通过从特定查询的语句获取信息来了解查询结果的信息。describeResult 函数返回 [(String, SqlColDesc)]，即对的列表。第一个给出列名，第二个提供有关列的信息：类型、大小以及是否可能为 NULL。完整规范在 HDBC API 参考中给出。

一些数据库可能无法提供所有这些元数据。在这种情况下，将引发异常。例如，Sqlite3 在撰写本文时不支持 describeResult 或 describeTable。

## 错误处理

HDBC 会在发生错误时引发异常。异常具有类型 SqlError。它们传达来自底层 SQL 引擎的信息，如数据库的状态、错误消息和数据库的数字错误代码（如果有）。

ghci 不知道如何在屏幕上显示发生的 SqlError。虽然异常会导致程序终止，但它不会显示有用的消息。以下是一个示例：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> quickQuery' conn "SELECT * from test2" []
*** Exception: (unknown)
ghci> disconnect conn
```

这里我们尝试从不存在的表中 SELECT 数据。我们得到的错误消息没有帮助。有一个实用函数 handleSqlError，它可以捕获 SqlError 并将其重新引发为 IOError。以这种形式，它可以在屏幕上打印，但以编程方式提取特定信息将更加困难。让我们看看它的用法：

```ghci
ghci> conn <- connectSqlite3 "test1.db"
ghci> handleSqlError $ quickQuery' conn "SELECT * from test2" []
*** Exception: user error (SQL error: SqlError {seState = "", seNativeError = 1,
seErrorMsg = "prepare 20: SELECT * from test2: no such table: test2"})
ghci> disconnect conn
```

在这里我们获得了更多信息，包括一条说明没有名为 test2 的表的消息。这更有帮助。许多 HDBC 程序员将 main = handleSqlError $ do 开头作为标准实践，这将确保每个未捕获的 SqlError 都以有帮助的方式打印。

还有 catchSql 和 handleSql——类似于标准的 catch 和 handle 函数。catchSql 和 handleSql 将仅拦截 HDBC 错误。有关错误处理的更多信息，请参阅第 19 章。
