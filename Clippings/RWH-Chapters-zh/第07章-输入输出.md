# 第07章-输入输出

大多数（如果不是全部）程序都致力于从外部收集数据、处理它并将结果返回给外部世界。也就是说，输入和输出是关键。

Haskell 的 I/O 系统功能强大且富有表现力。它易于使用且理解起来很重要。Haskell 严格分离纯代码和可能导致世界发生事情的代码。也就是说，它在纯代码中提供了与副作用的完全隔离。除了帮助程序员推理其代码的正确性，它还允许编译器自动引入优化和并行性。

我们将从 Haskell 中简单、标准化的 I/O 开始本章。然后我们将讨论一些更强大的选项，并提供更多关于 I/O 如何适应纯、惰性、函数式 Haskell 世界的细节。

## Haskell 中的经典 I/O

让我们通过查看一个看起来与其他语言（如 C 或 Perl）中的 I/O 惊人相似的程序来开始 Haskell 中的 I/O：

```haskell
-- file: ch07/basicio.hs
main = do
       putStrLn "Greetings!  What is your name?"
       inpStr <- getLine
       putStrLn $ "Welcome to Haskell, " ++ inpStr ++ "!"
```

你可以将此程序编译为独立可执行文件，使用 runghc 运行它，或从 ghci 内部调用 main。使用 runghc 的示例会话：

```bash
$ runghc basicio.hs
Greetings!  What is your name?
John
Welcome to Haskell, John!
```

这是一个相当简单、明显的结果。你可以看到 putStrLn 写出一个字符串，后跟一个行尾字符。getLine 从标准输入读取一行。`<-` 语法可能对你来说是新的。简单地说，那将 I/O 动作的结果绑定到一个名称。* 我们使用简单的列表连接运算符 ++ 将输入字符串与我们自己的文本连接起来。

让我们看看 putStrLn 和 getLine 的类型。你可以在库参考中找到该信息，或者直接问 ghci：

```ghci
ghci> :type putStrLn
putStrLn :: String -> IO ()
ghci> :type getLine
getLine :: IO String
```

注意这两个函数的返回类型都有 IO。这是你知道它们可能有副作用，或者即使使用相同参数调用也可能返回不同值，或两者兼有的关键。putStrLn 的类型看起来像一个函数。它接受一个 String 类型的参数并返回类型 IO () 的值。但究竟什么是 IO ()？任何类型为 IO something 的东西都是 I/O 动作。你可以存储它，什么都不会发生。我可以说 `writefoo = putStrLn "foo"`，然后什么都不会发生。但如果我们后来在另一个 I/O 动作中使用 writefoo，当其父动作被执行时 writefoo 动作将被执行——I/O 动作可以粘合在一起形成更大的 I/O 动作。() 是一个空元组（发音为"unit"），表示 putStrLn 没有返回值。这类似于 Java 或 C 中的 void。

动作可以被创建、赋值和传递到任何地方。但是，它们可能只能从另一个 I/O 动作内执行（执行）。

让我们用 ghci 查看这个：

```ghci
ghci> let writefoo = putStrLn "foo"
ghci> writefoo
foo
```

在这个例子中，输出 foo 不是 putStrLn 的返回值。而是 putStrLn 实际写入终端的副作用的 foo。

还注意一件事：ghci 实际上执行了 writefoo。这意味着，当给定一个 I/O 动作时，ghci 会当场为你执行它。

### 什么是 I/O 动作？

动作：

- 具有类型 IO t。
- 是 Haskell 中的一级值，与 Haskell 的类型系统无缝配合。
- 在执行时产生效果，但在求值时不产生效果。也就是说，它们只有在被 I/O 上下文中的其他东西调用时才会产生效果。
- 任何表达式都可以产生一个动作作为其值，但该动作在由另一个 I/O 动作执行（或它是 main）之前不会执行 I/O。
- 执行类型 IO t 的动作可以执行 I/O，最终将交付类型 t 的结果。

getLine 的类型可能对你来说看起来很奇怪。它看起来像一个值，而不是一个函数。事实上，这是一种看待它的方式：getLine 存储了一个 I/O 动作。当该动作被执行时，你得到一个字符串。`<-` 运算符用于从执行 I/O 动作中"拉出"结果并将其存储在变量中。

main 本身是一个类型为 IO () 的 I/O 动作。你只能从其他 I/O 动作内执行 I/O 动作。所有 Haskell 程序中的 I/O 都是从 main 的顶层驱动的，这是每个 Haskell 程序执行开始的地方。因此，这就是提供 Haskell 中副作用隔离的机制：你在你的 I/O 动作中执行 I/O，并从那里调用纯（非 I/O）函数。大多数 Haskell 代码是纯的；I/O 动作执行 I/O 并调用该纯代码。

do 是定义动作序列的方便方式。正如你稍后将看到的，还有其他方法。当你以这种方式使用 do 时，缩进很重要；确保你正确对齐你的动作。

只有当你有多个需要执行的动作时，才需要使用 do。do 块的值是最后执行的动作的值。

### 从 I/O 动作中调用纯代码

让我们看一个在 I/O 动作内调用纯代码的示例：

```haskell
-- file: ch07/callingpure.hs
name2reply :: String -> String
name2reply name =
    "Pleased to meet you, " ++ name ++ ".\n" ++
    "Your name contains " ++ charcount ++ " characters."
    where charcount = show (length name)

main :: IO ()
main = do
       putStrLn "Greetings once again.  What is your name?"
       inpStr <- getLine
       let outStr = name2reply inpStr
       putStrLn outStr
```

请注意此示例中的 name2reply 函数。它是一个常规 Haskell 函数，遵守我们告诉你的所有规则：给定相同输入总是返回相同结果，没有副作用，并且惰性操作。它使用其他 Haskell 函数：(++)、show 和 length。

在 main 中，我们将 `name2reply inpStr` 的结果绑定到 outStr。当你在 do 块中工作时，使用 `<-` 从 I/O 动作获取结果，使用 let 从纯代码获取结果。当在 do 块中使用时，不应该在 let 语句后加 in。

你可以看到我们如何从键盘读取人的名字。然后，该数据被传递给纯函数，其结果被打印。实际上，main 的最后两行可以被替换为 `putStrLn (name2reply inpStr)`。所以，虽然 main 确实有副作用——例如，它导致内容出现在终端上——name2reply 没有也不能。因为 name2reply 是一个纯函数，不是动作。

让我们用 ghci 检查这个：

```ghci
ghci> :load callingpure.hs
[1 of 1] Compiling Main             ( callingpure.hs, interpreted )
Ok, modules loaded: Main.
ghci> name2reply "John"
"Pleased to meet you, John.\nYour name contains 4 characters."
ghci> putStrLn (name2reply "John")
Pleased to meet you, John.
Your name contains 4 characters.
```

字符串中的 \n 是行尾（换行）字符，它导致终端在其输出中开始新行。只是在 ghci 中调用 `name2reply "John"` 会向你显示字面的 \n，因为它使用 show 显示返回值。但使用 putStrLn 将其发送到终端，终端解释 \n 开始新行。

## 纯代码与 I/O 的对比

表 7-1 是一个对比表，帮助你理解纯代码和 I/O 之间的差异。当我们谈论纯代码时，我们谈论的是总是给定相同参数返回相同结果且没有副作用的 Haskell 函数。在 Haskell 中，只有 I/O 动作的执行避免这些规则。

表 7-1. 纯与不纯

| 纯 | 不纯 |
|---|---|
| 总是给定相同参数产生相同结果 | 可能为相同参数产生不同结果 |
| 从不有副作用 | 可能有副作用 |
| 从不改变状态 | 可能改变程序、系统或世界的全局状态 |

## 为什么纯度重要

在本节中，我们讨论了 Haskell 与大多数语言不同，在纯代码和 I/O 动作之间划清界限。在 C 或 Java 等语言中，没有任何东西能保证编译器保证函数总是返回相同参数的结果，或者保证函数永远不会产生副作用。知道给定函数是否有副作用的唯一方法是阅读其文档并希望它是准确的。

程序中的许多 bug 是由意外的副作用引起的。更多是由误解函数可能在何种情况下为相同输入返回不同结果引起的。随着多线程和其他形式的并行性越来越普遍，管理全局副作用变得更加困难。

Haskell 将副作用隔离到 I/O 动作中的方法提供了明确的边界。你总是可以知道系统的哪些部分可能改变状态，哪些不会。你总是可以确信程序的纯部分不会有意外结果。这有助于你思考程序。它也帮助编译器思考它。例如，ghc 的最新版本可以为你的代码的纯部分提供一定程度的自动并行性——这是计算的一个圣杯。

## I/O 函数的更多详情

### 读写字符串

putStrLn 写入字符串并添加换行符：

```ghci
ghci> putStrLn "hello"
hello
ghci> putStrLn "world"
world
```

putStr 写入字符串但不添加换行符：

```ghci
ghci> putStr "hello"
helloghci>
```

getLine 读取一行：

```ghci
ghci> inpStr <- getLine
hello
ghci> inpStr
"hello"
```

### 读取和显示值

show 将值转换为字符串：

```ghci
ghci> show 123
"123"
ghci> show True
"True"
```

read 将字符串解析为值：

```ghci
ghci> read "123" + 1
124
ghci> read "True" || False
True
```

### 文件 I/O

readFile 和 writeFile 用于文件：

```haskell
main = do
    content <- readFile "input.txt"
    writeFile "output.txt" (content ++ "processed\n")
```

### 序列化数据

我们可以使用 Show 和 Read 类型类来序列化和反序列化数据：

```haskell
main = do
    let data = [1, 2, 3 :: Int]
    writeFile "data.txt" (show data)
    content <- readFile "data.txt"
    let parsed = read content :: [Int]
    print (sum parsed)
```

## 惰性 I/O

Haskell 的惰性求值意味着 I/O 可以以有趣的方式工作：

```haskell
main = do
    content <- readFile "largefile.txt"
    let lines = filter (\l -> length l > 80) (lines content)
    mapM_ putStrLn lines
```

这不会将整个文件读入内存，而是惰性地处理它。

在本章中，我们涵盖了：
