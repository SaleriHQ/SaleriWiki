# 第03章-定义类型、简化函数

## 定义新的数据类型

虽然列表和元组很有用，但我们经常想要构建自己的新数据类型。这允许我们在程序中的值添加结构。我们可以给相关值的集合一个名称和一个独特的类型，而不是使用匿名元组。定义我们自己的类型也提高了代码的类型安全性：Haskell 不会让我们意外混合两个结构相似但名称不同的类型的值。

为了激励我们思考，我们将考虑一个小在线书店可能需要管理的一些数据种类。我们不会尝试完整或现实的数据定义，但至少我们将它们与现实世界联系起来。

我们使用 data 关键字定义新的数据类型：

```haskell
-- file: ch03/BookStore.hs
data BookInfo = Book Int String [String]
                deriving (Show)
```

data 关键字后面的 BookInfo是我们新类型的名称。我们称 BookInfo 为类型构造函数。一旦我们定义了一个类型，我们将使用它的类型构造函数来引用它。正如我们已经提到的，类型名，因此类型构造函数，必须以大写字母开头。

后面的 Book 是值构造函数的名称（有时称为数据构造函数）。我们使用它来创建 BookInfo 类型的值。值构造函数的名称也必须以大写字母开头。

在 Book 之后，随后的 Int、String 和 [String] 是类型的组件。组件在 Haskell 中起着与另一种语言中结构或类的字段相同的目的：它是一个我们可以保存值的"槽"。（我们经常将组件称为字段。）

在这个例子中，Int 代表书的标识符（例如，在库存数据库中），String 代表它的标题，而 [String] 代表其作者的名称。

为了与我们已经看到的概念联系起来，BookInfo 类型包含与类型 (Int, String, [String]) 的 3 元组相同的组件，但它有一个独特的类型。我们不能意外地（或故意地）在期望另一个的上下文中使用一个。例如，书店也可能销售杂志：

```haskell
-- file: ch03/BookStore.hs
data MagazineInfo = Magazine Int String [String]
                    deriving (Show)
```

尽管这个 MagazineInfo 类型与我们的 BookInfo 类型具有相同的结构，但 Haskell 将类型视为不同的，因为它们的类型和值构造函数具有不同的名称。

### 派生什么？

我们稍后会在第 139 页的"Show"中解释 deriving (Show) 的完整含义。现在，知道我们需要在类型声明中添加这个，以便 ghci 自动知道如何打印此类型的值就足够了。

我们可以通过将 Book 视为一个函数并使用 Int、String 和 [String] 类型的参数应用它来创建 BookInfo 类型的新值：

```haskell
-- file: ch03/BookStore.hs
myInfo = Book 9780135072455 "Algebra of Programming"
         ["Richard Bird", "Oege de Moor"]
```

一旦我们定义了一个类型，我们就可以在 ghci 中尝试它。我们首先使用 :load 命令加载我们的源文件：

```ghci
ghci> :load BookStore
[1 of 1] Compiling Main             ( BookStore.hs, interpreted )
Ok, modules loaded: Main.
```

还记得我们在源文件中定义的 myInfo 变量吗？这里是：

```ghci
ghci> myInfo
Book 9780135072455 "Algebra of Programming" ["Richard Bird","Oege de Moor"]
ghci> :type myInfo
myInfo :: BookInfo
```

我们也可以在 ghci 中交互地构造新值：

```ghci
ghci> Book 0 "The Book of Imaginary Beings" ["Jorge Luis Borges"]
Book 0 "The Book of Imaginary Beings" ["Jorge Luis Borges"]
```

ghci 命令 :type 让我们看到表达式的类型：

```ghci
ghci> :type Book 1 "Cosmicomics" ["Italo Calvino"]
Book 1 "Cosmicomics" ["Italo Calvino"] :: BookInfo
```

### 记住

请记住，如果我们想在 ghci 内部定义一个新变量，语法与 Haskell 源文件的语法略有不同——我们需要在前面放一个 let：

```ghci
ghci> let cities = Book 173 "Use of Weapons" ["Iain M. Banks"]
```

要了解有关类型的更多信息，我们可以使用 ghci 的一些浏览功能。:info 命令让 ghci 告诉我们它知道关于一个名称的所有信息：

```ghci
ghci> :info BookInfo
data BookInfo = Book Int String [String]
      -- Defined at BookStore.hs:4:5-12
instance Show BookInfo -- Defined at BookStore.hs:4:5-12
```

我们还可以找出为什么我们使用 Book 来构造 BookStore 类型的新值：

```ghci
ghci> :type Book
Book :: Int -> String -> [String] -> BookInfo
```

我们可以将值构造函数视为另一个函数——一个恰好创建并返回我们所需类型的新值的函数。

## 命名类型和值

当我们介绍 BookStore 类型时，我们故意选择给类型构造函数 BookStore 一个与值构造函数 Book 不同的名称，完全是为了清楚哪个是哪个。

然而，在 Haskell 中，类型和值的名称是相互独立的。我们在类型声明或类型签名中仅使用类型构造函数（即类型的名称）。我们仅在实际代码中使用值构造函数。因为这些用途是不同的，如果我们将类型构造函数和值构造函数命名为相同名称，就不会有歧义。如果我们正在编写类型签名，我们必须指的是类型构造函数。如果我们正在编写表达式，我们必须使用值构造函数：

```haskell
-- file: ch03/BookStore.hs
-- We will introduce the CustomerID type shortly.
data BookReview = BookReview BookInfo CustomerID String
```

这个定义说名为 BookReview 的类型有一个也名为 BookReview 的值构造函数。

值构造函数与其类型构造函数同名不仅是合法的，而且是正常的。您会在常规 Haskell 代码中一直看到这种情况。

## 类型同义词

我们可以随时引入现有类型的同义词，以使类型更具描述性名称。例如，我们的 BookReview 类型中的 String 没有告诉我们字符串是做什么的，但我们可以澄清这一点：

```haskell
-- file: ch03/BookStore.hs
type CustomerID = Int
type ReviewBody = String
data BetterReview = BetterReview BookInfo CustomerID ReviewBody
```

type 关键字引入类型同义词。新名称在 = 的左边，现有的名称在右边。这两个名称标识相同的类型，所以类型同义词纯粹是为了使代码更易读。

我们也可以使用类型同义词为冗长类型创建更短的名称：

```haskell
-- file: ch03/BookStore.hs
type BookRecord = (BookInfo, BookReview)
```

这说明我们可以使用 BookRecord 作为元组 (BookInfo, BookReview) 的同义词。类型同义词仅创建引用现有类型的新名称。* 我们仍然使用相同的值构造函数来创建该类型的值。

* 如果您熟悉 C 或 C++，它类似于 typedef。

## 代数数据类型

熟悉的 Bool 是称为代数数据类型的类型类别中最简单的常见示例。代数数据类型可以有多个值构造函数：

```haskell
-- file: ch03/Bool.hs
data Bool = False | True
```

Bool 类型有两个值构造函数，True 和 False。每个值构造函数在定义中用 | 字符分隔，我们可以将其读作"或"——我们可以构造一个值为 True 或 False 的 Bool。当一个类型有多个值构造函数时，它们通常被称为替代或情况。我们可以使用任何一种替代来创建该类型的值。

### 关于命名的一个注记

虽然短语"代数数据类型"很长，但我们谨慎地避免使用缩写"ADT"，它已经被广泛理解为代表"抽象数据类型"。由于 Haskell 支持代数数据类型和抽象数据类型，我们将明确并完全避免使用缩写。

代数数据类型的每个值构造函数可以接受零个或多个参数。作为一个例子，这里是我们可能表示账单信息的一种方式：

```haskell
-- file: ch03/BookStore.hs
type CardHolder = String
type CardNumber = String
type Address = [String]

data BillingInfo = CreditCard CardNumber CardHolder Address
                 | CashOnDelivery
                 | Invoice CustomerID
                   deriving (Show)
```

这里，我们说我们支持三种方式向客户计费。如果他们想用信用卡支付，他们必须将卡号、持卡人姓名和持卡人的账单地址作为参数提供给 CreditCard 值构造函数。或者，他们可以向送货的人付款。因为我们不需要存储关于此的任何额外信息，我们为 CashOnDelivery 构造函数指定没有参数。最后，我们可以将发票发送给指定的客户，在这种情况下，我们需要她的 CustomerID 作为 Invoice 构造函数的参数。

当我们使用值构造函数创建 BillingInfo 类型的值时，我们必须提供它所需的参数：

```ghci
ghci> :type CreditCard
CreditCard :: CardNumber -> CardHolder -> Address -> BillingInfo
ghci> CreditCard "2901650221064486" "Thomas Gradgrind" ["Dickens", "England"]
CreditCard "2901650221064486" "Thomas Gradgrind" ["Dickens","England"]
ghci> :type it
it :: BillingInfo
ghci> Invoice
<interactive>:1:0:
    No instance for (Show (CustomerID -> BillingInfo))
      arising from a use of `print' at <interactive>:1:0-6
    Possible fix:
      add an instance declaration for (Show (CustomerID -> BillingInfo))
    In the expression: print it
    In a stmt of a 'do' expression: print it
ghci> :type it
it :: BillingInfo
```

No instance 错误消息出现是因为我们没有向 Invoice 构造函数提供参数。结果，我们试图打印 Invoice 构造函数本身。那个构造函数需要一个参数并返回一个值，所以它是一个函数。我们不能在 Haskell 中打印函数，这就是解释器抱怨的原因。

## 元组、代数数据类型，以及何时使用每个

元组和用户定义的代数数据类型之间存在一些重叠。如果我们愿意，我们可以将之前的 BookInfo 类型表示为 (Int, String, [String]) 元组：

```ghci
ghci> Book 2 "The Wealth of Networks" ["Yochai Benkler"]
Book 2 "The Wealth of Networks" ["Yochai Benkler"]
ghci> (2, "The Wealth of Networks", ["Yochai Benkler"])
(2,"The Wealth of Networks",["Yochai Benkler"])
```

代数数据类型允许我们区分实际上相同的信息片段。具有相同类型元素的两个元组是结构相同的，所以它们具有相同的类型：

```haskell
-- file: ch03/Distinction.hs
a = ("Porpoise", "Grey")
b = ("Table", "Oak")
```

由于名称不同，两个代数数据类型即使结构相同也具有不同的类型：

```haskell
-- file: ch03/Distinction.hs
data Cetacean = Cetacean String String
data Furniture = Furniture String String
c = Cetacean "Porpoise" "Grey"
d = Furniture "Table" "Oak"
```

这让我们能够利用类型系统来帮助编写具有更少 bug 的程序。使用我们刚才定义的元组，我们可能想象将鲸鱼的描述传递给期望椅子的函数，而类型系统无法帮助我们。使用代数数据类型，就没有这种混淆的可能性。

这是一个更微妙的例子。考虑以下二维向量的表示：

```haskell
-- file: ch03/AlgebraicVector.hs
-- x 和 y 坐标或长度。
data Cartesian2D = Cartesian2D Double Double
                   deriving (Eq, Show)
-- 角度和距离（幅度）。
data Polar2D = Polar2D Double Double
               deriving (Eq, Show)
```

笛卡尔形式和极坐标形式使用相同类型表示其两个元素。然而，元素的含义不同。因为 Cartesian2D 和 Polar2D 是不同的类型，类型系统不会让我们意外地在期望 Polar2D 的地方使用 Cartesian2D 值，反之亦然。

```ghci
ghci> Cartesian2D (sqrt 2) (sqrt 2) == Polar2D (pi / 4) 2
<interactive>:1:33:
    Couldn't match expected type `Cartesian2D'
           against inferred type `Polar2D'
    In the second argument of `(==)', namely `Polar2D (pi / 4) 2'
    In the expression:
          Cartesian2D (sqrt 2) (sqrt 2) == Polar2D (pi / 4) 2
    In the definition of `it':
        it = Cartesian2D (sqrt 2) (sqrt 2) == Polar2D (pi / 4) 2
```

( == ) 运算符要求其参数具有相同的类型。

### 比较相等

请注意，在向量类型的 deriving 子句中，我们添加了另一个词 Eq。这导致 Haskell 实现生成代码，让我们可以比较值的相等性。

如果我们使用元组来表示这些值，我们可以通过不恰当地混合两种表示而快速陷入困境：

```ghci
ghci> (1, 2) == (1, 2)
True
```

类型系统在这里无法拯救我们：就它而言，我们比较的是两个 (Double, Double) 对，这是完全有效的做法。事实上，我们无法通过检查来判断这些值中的哪一个应该是极坐标或笛卡尔，但 (1,2) 在每种表示中都有不同的含义。

对于决定何时更好地使用元组或独特的数据类型，没有硬性规定，但这里有一个经验法则。如果您在代码中广泛使用复合值（几乎所有非平凡程序都这样做），添加数据声明将在类型安全性和可读性方面使您受益。对于更小、更本地化的用途，元组通常很好。

### 其他语言中代数数据类型的类比

代数数据类型提供了一种描述数据类型的单一强大方式。其他语言通常需要几种不同的功能来实现相同程度的表达力。以下是 C 和 C++ 中的一些类比，这可能使我们能够更好地理解代数数据类型可以做什么，以及它们如何与可能更熟悉的概念相关联。

#### 结构

只有一个构造函数的代数数据类型类似于元组：它将相关值分组到一个复合值中。它对应于 C 或 C++ 中的 struct，其组件对应于 struct 的字段。这是我们之前定义的 BookInfo 类型的 C 等效项：

```c
struct book_info {
    int id;
    char *name;
    char **authors;
};
```

两者之间的主要区别是 Haskell 类型中的字段是匿名的和位置的：

```haskell
-- file: ch03/BookStore.hs
data BookInfo = Book Int String [String]
                deriving (Show)
```

通过位置的，我们意思是节号在 Haskell 类型的第一字段中，标题在第二中。我们通过位置而不是名称来引用它们。

在本章后面的"模式匹配"第 50 页中，我们将看到如何访问 BookStore 值的字段。在本章 also 的"记录语法"第 55 页中，我们将介绍一种看起来更像 C 一样的定义数据类型替代语法。

#### 枚举

代数数据类型也适用于我们将在 C 或 C++ 中使用 enum 来表示一系列符号值的地方。这样的代数数据类型有时被称为枚举类型。这里有一个 C 中的例子：

```c
enum roygbiv {
    red,
    orange,
    yellow,
    green,
    blue,
    indigo,
    violet,
};
```

这里是 Haskell 等效项：

```haskell
-- file: ch03/Roygbiv.hs
data Roygbiv = Red
             | Orange
             | Yellow
             | Green
             | Blue
             | Indigo
             | Violet
               deriving (Eq, Show)
```

我们可以在 ghci 中尝试这些：

```ghci
ghci> :type Yellow
Yellow :: Roygbiv
ghci> :type Red
Red :: Roygbiv
ghci> Red == Yellow
False
ghci> Green == Green
True
```

在 C 中，enum 的元素是整数。我们可以在期望 enum 的地方使用整数，反之亦然——C 编译器将在两种类型之间自动转换值。这可能是讨厌的 bug 的来源。在 Haskell 中，这种问题不会发生。例如，我们不能在期望 Int 的地方使用 Roygbiv 值：

```ghci
ghci> take 3 "foobar"
"foo"
ghci> take Red "foobar"
<interactive>:1:5:
    Couldn't match expected type `Int' against inferred type `Roygbiv'
    In the first argument of `take', namely `Red'
    In the expression: take Red "foobar"
    In the definition of `it': it = take Red "foobar"
```

#### 有标签的联合

如果代数数据类型有多个替代，我们可以将其视为类似于 C 或 C++ 中的 union。两者之间的一个主要区别是 union 不告诉我们实际存在哪个替代；我们必须显式和手动跟踪我们正在使用的替代，通常在封闭结构的另一个字段中。这意味着 union 可能是讨厌的 bug 的来源，我们认为自己应该使用的替代的想法可能是错误的：

```c
enum shape_type {
    shape_circle,
    shape_poly,
};
struct circle {
    struct vector centre;
    float radius;
};
struct poly {
    size_t num_vertices;
    struct vector *vertices;
};
struct shape
{
    enum shape_type type;
    union {
    struct circle circle;
    struct poly poly;
    } shape;
};
```

在这个例子中，union 可以包含 struct circle 或 struct poly 的有效数据。我们必须手动使用 enum shape_type 来指示当前存储在 union 中的值的种类。

这个代码的 Haskell 版本比 C 等效项明显更短更安全：

```haskell
-- file: ch03/ShapeUnion.hs
type Vector = (Double, Double)
data Shape = Circle Vector Double
           | Poly [Vector]
```

如果我们使用 Circle 构造函数创建 Shape 值，我们创建了一个 Circle 的事实在其中存储。当我们后来使用 Circle 时，我们不能意外地将其视为 Square。我们将在下一节"模式匹配"第 50 页中看到原因。

### 一些注记

阅读了前述部分后，现在应该清楚，我们使用 data 关键字定义的所有数据类型都是代数数据类型。有些可能只有一个替代，而其他可能有几个，但它们都使用相同的机制。

## 模式匹配

现在我们已经看到了如何用代数数据类型构造值，让我们讨论如何处理这些值。如果我们有一个某类型的值，我们有两件事想要做：

- 如果该类型有多个值构造函数，我们需要能够判断哪个值构造函数被用于创建该值。
- 如果值构造函数有数据组件，我们需要能够提取这些值。

Haskell 有一个简单但非常有用的模式匹配机制，让我们能够做这两件事。

模式让我们 look inside a value 并绑定变量到它包含的数据。这里有一个对 Bool 值进行模式匹配的示例：我们将重现 not 函数：

```haskell
-- file: ch03/add.hs
myNot True  = False
myNot False = True
```

看起来我们这里有两个名为 myNot 的函数，但 Haskell 让我们将函数定义为一系列方程：这两个子句为不同输入模式定义相同函数的行为。在每一行上，模式是函数名称后面的项，直到 = sign。

要理解模式匹配如何工作，让我们逐步完成一个示例——比如说 myNot False。

当我们应用 myNot 时，Haskell 运行时检查我们提供的值与第一个模式中的值构造函数。这不匹配，所以它尝试针对第二个模式。那个匹配成功，所以它使用该方程的右侧作为函数应用的结果。

这里有一个稍微扩展的示例。这个函数将列表的元素相加：

```haskell
-- file: ch03/add.hs
sumList (x:xs) = x + sumList xs
sumList []     = 0
```

让我们逐步完成 sumList [1,2] 的求值。列表表示法 [1,2] 是表达式 (1:(2:[])) 的简写。我们首先尝试将 sumList 定义第一个方程中的模式匹配。在 (x:xs) 模式中，: 是熟悉的列表构造函数 (:)。我们现在正在使用它来匹配一个值，而不是构造一个值。值 (1:(2:[])) 是使用 (:) 构造的，所以值中的构造函数与模式中的构造函数匹配。我们说模式匹配或匹配成功。

变量 x 和 xs 现在"绑定到"构造函数的参数，所以 x 被给予值 1，xs 被给予值 2:[]。

我们现在求值的表达式是 1 + sumList (2:[])。我们必须递归地将 sumList 应用到值 2:[]。又一次，这使用 (:) 构造，所以匹配成功。在我们的 sumList 递归应用中，x 现在绑定到 2，xs 绑定到 []。

我们现在求值 1 + (2 + sumList [])。在这个 sumList 的递归应用中，我们匹配的值是 []。值的构造函数与第一个模式中的构造函数不匹配，所以我们跳过这个方程。相反，我们"跳转"到下一个模式，它匹配。该方程的右侧因此被选为此次应用的结果。

因此 sumList [1,2] 的结果是 1 + (2 + (0))，即 3。

### 排序很重要

正如我们已经提到的，Haskell 实现按照我们在方程中指定的顺序检查模式匹配。匹配从头到尾进行，并在第一次成功后停止。低于成功匹配的方程无效。

作为最后一个注记，有一个标准函数 sum 为我们执行此列表求和。我们的 sumList 仅用于说明。

## 构造和析构

让我们退后一步，看看构造值和对其执行模式匹配之间的关系。

我们应用值构造函数来构建值。表达式 `Book 9 "Close Calls" ["John Long"] 应用 Book 构造函数到值 9、"Close Calls" 和 ["John Long"]，以产生 BookInfo 类型的新值。

当我们对 Book 构造函数进行模式匹配时，我们反转构造过程。首先，我们检查该值是否使用该构造函数创建。如果是，我们检查它以获取我们最初在创建值时提供给构造函数的各个值。

### 析构不会销毁任何东西

如果您精通面向对象编程术语，请不要将析构与销毁混淆！匹配一个模式对我们正在检查的值没有影响：它只是让我们"查看"它内部。

## 进一步的冒险

元组上模式匹配的语法类似于构造元组的语法。这是一个返回 3 元组最后一个元素的函数：

```haskell
-- file: ch03/Tuple.hs
third (a, b, c) = c
```

模式 look inside a value 的深度没有限制。这个定义同时查看元组内部和该元组内列表的内部：

```haskell
-- file: ch03/Tuple.hs
complicated (True, a, x:xs, 5) = (a, xs)
```

我们可以交互地尝试这个：

```ghci
ghci> :load Tuple.hs
[1 of 1] Compiling Main             ( Tuple.hs, interpreted )
Ok, modules loaded: Main.
ghci> complicated (True, 1, [1,2,3], 5)
(1,[2,3])
```

每当一个文字值存在于一个模式中时（在前面的模式中为 True 和 5），该值必须精确匹配才能使模式匹配成功。如果一系列方程中的每个模式都匹配失败，我们会收到运行时错误：

```ghci
ghci> complicated (False, 1, [1,2,3], 5)
*** Exception: Tuple.hs:10:0-39: Non-exhaustive patterns in function complicated
```

有关此错误消息的解释，请跳转到"穷尽模式和一个例外"第 54 页。

我们可以使用其值构造函数对代数数据类型进行模式匹配。回想一下我们之前定义的 BookInfo 类型；我们可以如下从 BookInfo 提取值：

```haskell
-- file: ch03/BookStore.hs
bookID      (Book id title authors) = id
bookTitle   (Book id title authors) = title
bookAuthors (Book id title authors) = authors
```

让我们看看它的运行：

```ghci
ghci> bookID (Book 3 "Probability Theory" ["E.T.H. Jaynes"])
3
ghci> bookTitle (Book 3 "Probability Theory" ["E.T.H. Jaynes"])
"Probability Theory"
ghci> bookAuthors (Book 3 "Probability Theory" ["E.T.H. Jaynes"])
["E.T.H. Jaynes"]
```

编译器可以根据我们在模式中使用的构造函数推断访问器函数的类型：

```ghci
ghci> :type bookID
bookID :: BookInfo -> Int
ghci> :type bookTitle
bookTitle :: BookInfo -> String
ghci> :type bookAuthors
bookAuthors :: BookInfo -> [String]
```

如果我们在模式中使用文字值，我们正在匹配的值的相应部分必须包含相同的值。例如，模式 (3:xs) 首先通过匹配 (:) 构造函数检查值是一个非空列表。它还确保列表的头具有确切的值 3。如果这两个条件都成立，列表的尾部将绑定到变量 xs。

## 模式中的变量命名

当你阅读对列表进行匹配函数时，你经常会发现模式内变量的名称类似于 (x:xs) 或 (d:ds)。这是一种流行的命名约定。这个想法是名称 xs 在其名称末尾有一个 s，就好像它是 x 的"复数"，因为 x 包含列表的头，而 xs 包含剩余的元素。

## 通配符模式

我们可以表示我们不关心模式某部分中存在什么。这个表示法是下划线字符 (_)，我们称之为通配符。我们如下使用它：

```haskell
-- file: ch03/BookStore.hs
nicerID      (Book id _     _      ) = id
nicerTitle   (Book _  title _      ) = title
nicerAuthors (Book _  _     authors) = authors
```

这里，我们有我们之前介绍的访问器函数的更整洁版本。现在，毫无疑问我们在每个函数中使用哪个元素。

在模式中，通配符类似于变量，但它不会绑定新变量。如前面的示例所示，我们可以在单个模式中使用多个通配符。

通配符的另一个优点是，如果我们在模式中引入变量名称，然后不在函数体中使用它，Haskell 编译器可以警告我们。定义一个变量但忘记使用它通常可以指示存在 bug，所以这是一个有用的功能。如果我们使用我们不打算使用的变量而不是通配符，编译器不会抱怨。

## 穷尽模式和通配符

在编写一系列模式时，覆盖类型的所有构造函数很重要。例如，如果我们正在检查列表，我们应该有一个匹配非空构造函数 (:) 的方程，一个匹配空列表构造函数 []。

让我们看看如果我们未能覆盖所有情况会发生什么。这里，我们故意省略对 [] 构造函数的检查：

```haskell
-- file: ch03/BadPattern.hs
badExample (x:xs) = x + badExample xs
```

如果我们将其应用于无法匹配的值，我们将在运行时收到错误——我们的软件有 bug！

```ghci
ghci> badExample []
*** Exception: BadPattern.hs:4:0-36: Non-exhaustive patterns in function badExample
```

在这个例子中，函数定义中的哪个方程都不匹配值 []。

### 关于不完整模式的警告

GHC 提供了一个有用的编译选项 -fwarn-incomplete-patterns，它会在编译期间如果一系列模式不匹配类型的所有值构造函数则打印警告。

如果我们需要提供默认行为以处理我们不关心特定构造函数的情况，可以使用通配符模式：

```haskell
-- file: ch03/BadPattern.hs
goodExample (x:xs) = x + goodExample xs
goodExample _      = 0
```

前面代码中显示的通配符将匹配 [] 构造函数，因此应用此函数不会导致崩溃：

```ghci
ghci> goodExample []
0
ghci> goodExample [1,2]
3
```

## 记录语法

为数据类型的每个组件编写访问器函数可能是重复和繁琐的：

```haskell
-- file: ch03/BookStore.hs
nicerID      (Book id _     _      ) = id
nicerTitle   (Book _  title _      ) = title
nicerAuthors (Book _  _     authors) = authors
```

我们称这种代码为样板——必要的，但笨重和麻烦。Haskell 程序员不喜欢样板。幸运的是，语言解决了这个特定的样板问题：我们可以同时定义数据类型及其每个组件的访问器。（这里逗号的位置是偏好问题。如果您愿意，可以将它们放在行尾而不是行首。）

```haskell
-- file: ch03/BookStore.hs
data Customer = Customer {
      customerID      :: CustomerID
    , customerName    :: String
    , customerAddress :: Address
    } deriving (Show)
```

这与以下更熟悉的形式含义几乎完全相同：

```haskell
-- file: ch03/AltCustomer.hs
data Customer = Customer Int String [String]
                deriving (Show)

customerID :: Customer -> Int
customerID (Customer id _ _) = id

customerName :: Customer -> String
customerName (Customer _ name _) = name

customerAddress :: Customer -> [String]
customerAddress (Customer _ _ address) = address
```

对于我们在类型定义中命名的每个字段，Haskell 创建该名称的访问器函数：

```ghci
ghci> :type customerID
customerID :: Customer -> CustomerID
```

我们仍然可以使用通常的应用语法来创建此类型的值：

```haskell
-- file: ch03/BookStore.hs
customer1 = Customer 271828 "J.R. Hacker"
            ["255 Syntax Ct",
             "Milpitas, CA 95134",
             "USA"]
```

记录语法添加了更冗长的表示法来创建值。这有时可以使代码更易读：

```haskell
-- file: ch03/BookStore.hs
customer2 = Customer {
              customerID = 271828
            , customerAddress = ["1048576 Disk Drive",
                                 "Milpitas, CA 95134",
                                 "USA"]
            , customerName = "Jane Q. Citizen"
            }
```

如果我们使用这种形式，我们可以改变列出字段的顺序。这里，我们将名称和地址字段从它们在类型声明中的位置移开。

当我们使用记录语法定义类型时，它也改变了类型值的打印方式：

```ghci
ghci> customer1
Customer {customerID = 271828, customerName = "J.R. Hacker", customerAddress =
["255 Syntax Ct","Milpitas, CA 95134","USA"]}
```

作为比较，让我们看一个 BookInfo 值；我们定义这个类型时没有使用记录语法：

```ghci
ghci> cities
Book 173 "Use of Weapons" ["Iain M. Banks"]
```

当我们使用记录语法时，我们"免费"获得的访问器函数确实是常规 Haskell 函数：

```ghci
ghci> :type customerName
customerName :: Customer -> String
ghci> customerName customer1
"J.R. Hacker"
```

标准 System.Time 模块很好地使用了记录语法。这里是该模块中定义的一个类型：

```haskell
data CalendarTime = CalendarTime {
  ctYear                      :: Int,
  ctMonth                     :: Month,
  ctDay, ctHour, ctMin, ctSec :: Int,
  ctPicosec                   :: Integer,
  ctWDay                      :: Day,
  ctYDay                      :: Int,
  ctTZName                    :: String,
  ctTZ                        :: Int,
  ctIsDST                     :: Bool
}
```

在没有记录语法的情况下，从这样的类型中提取特定字段将是痛苦的。表示法使处理大型结构变得更容易。

## 参数化类型

我们反复提到列表类型是多态的：列表的元素可以是任何类型。我们也可以向我们自己的类型添加多态性。为此，我们在类型声明中引入类型变量。Prelude 定义了一个名为 Maybe 的类型，我们可以用它来表示一个值可以是存在或缺失的，例如，数据库行中可以为空的字段：

```haskell
-- file: ch03/Nullable.hs
data Maybe a = Just a
             | Nothing
```

这里，变量 a 不是常规变量——它是一个类型变量。它指示 Maybe 类型接受另一个类型作为其参数。这允许我们在任何类型的值上使用 Maybe：

```haskell
-- file: ch03/Nullable.hs
someBool = Just True
someString = Just "something"
```

像往常一样，我们可以在 ghci 中尝试这个类型：

```ghci
ghci> Just 1.5
Just 1.5
ghci> Nothing
Nothing
ghci> :type Just "invisible bike"
Just "invisible bike" :: Maybe [Char]
```

Maybe 是一个多态的或泛型类型。我们给 Maybe 类型构造函数一个参数来创建特定类型，例如 Maybe Int 或 Maybe [Bool]。正如我们可能期望的，这些类型是不同的。

我们可以将参数化类型的用法嵌套在彼此内部，但当我们这样做时，我们可能需要使用括号来告诉 Haskell 编译器如何解析我们的表达式：

```haskell
-- file: ch03/Nullable.hs
wrapped = Just (Just "wrapped")
```

为了再次将类比扩展到更熟悉的语言，参数化类型类似于 C++ 中的模板和 Java 中的泛型。请注意，这是一个浅薄的类比。模板和泛型是在各自语言最初定义之后很久才添加的，它们有一种笨拙的感觉。Haskell 的参数化类型更简单且更容易使用，因为语言从一开始就为它们设计了。

## 递归类型

熟悉的列表类型是递归的：它是根据自身定义的。为了理解这一点，让我们创建我们自己的类似列表的类型。我们将使用 Cons 代替 (:) 构造函数，使用 Nil 代替 []：

```haskell
-- file: ch03/ListADT.hs
data List a = Cons a (List a)
            | Nil
              deriving (Show)
```

因为 List a 出现在 = 的左边和右边，类型的定义引用自身。如果我们想使用 Cons 构造函数创建新值，我们必须提供一个类型 a 的值和另一个类型 List a 的值。让我们看看这在实践中把我们引向何方。

我们可以创建的最简单的 List a 类型值是 Nil。将类型定义保存在文件中，然后将其加载到 ghci 中：

```ghci
ghci> Nil
Nil
```

因为 Nil 具有 List 类型，我们可以将其用作 Cons 的参数：

```ghci
ghci> Cons 0 Nil
Cons 0 Nil
```

因为 Cons 0 Nil 具有类型 List a，我们可以使用它作为 Cons 的参数：

```ghci
ghci> Cons 1 it
Cons 1 (Cons 0 Nil)
ghci> Cons 2 it
Cons 2 (Cons 1 (Cons 0 Nil))
ghci> Cons 3 it
Cons 3 (Cons 2 (Cons 1 (Cons 0 Nil)))
```

我们可以无限期地继续这样做，创建越来越长的 Cons 链，每个链末尾只有一个 Nil。

作为递归类型的第三个例子，这里是二叉树类型的定义：

```haskell
-- file: ch03/Tree.hs
data Tree a = Node a (Tree a) (Tree a)
            | Empty
              deriving (Show)
```

二叉树要么是具有两个子节点的节点——它们本身是二叉树——要么是一个空值。

### List 是一个可接受的列表吗？

我们可以轻松地向自己证明我们的 List a 类型与内置列表类型 [a] 具有相同的形状。为此，我们编写一个函数，接受任何 [a] 类型的值并生成 List a 类型的值：

```haskell
-- file: ch03/ListADT.hs
fromList (x:xs) = Cons x (fromList xs)
fromList []     = Nil
```

通过检查，这显然用 Cons 替换每个 (:)，用 Nil 替换每个 []。这涵盖了内置列表类型的两个构造函数。这两种类型是同构的——它们具有相同的形状：

```ghci
ghci> fromList "durian"
Cons 'd' (Cons 'u' (Cons 'r' (Cons 'i' (Cons 'a' (Cons 'n' Nil)))))
ghci> fromList [Just True, Nothing, Just False]
Cons (Just True) (Cons Nothing (Cons (Just False) Nil))
```

这次，让我们通过与我们更熟悉的语言的定义进行比较来寻找洞察。这是 Java 中一个类似的类定义：

```java
class Tree<A>
{
    A value;
    Tree<A> left;
    Tree<A> right;
    public Tree(A v, Tree<A> l, Tree<A> r)
    {
    value = v;
    left = l;
    right = r;
    }
}
```

一个显著的区别是 Java 允许我们在任何地方使用特殊值 null 来表示"nothing"，所以我们可以使用 null 来表示节点缺少左或右子节点。这里是一个小函数，构造具有两个叶子的树（按照惯例，叶子没有子节点）：

```java
class Example
{
    static Tree<String> simpleTree()
    {
    return new Tree<String>(
            "parent",
        new Tree<String>("left leaf", null, null),
        new Tree<String>("right leaf", null, null));
    }
}
```

在 Haskell 中，我们没有 null 的等价物。我们可以使用 Maybe 类型来提供类似的效果，但这会使模式匹配变得臃肿。相反，我们决定使用没有参数的 Empty 构造函数。Java 示例将 null 提供给 Tree 构造函数，我们在 Haskell 中提供 Empty：

```haskell
-- file: ch03/Tree.hs
simpleTree = Node "parent" (Node "left child" Empty Empty)
                           (Node "right child" Empty Empty)
```

2. 定义一个只有一种构造函数的树类型，就像我们的 Java 示例一样。不是使用 Empty 构造函数，而是使用 Maybe 类型来引用节点的子节点。

## 报告错误

Haskell 提供了一个标准函数 `error :: String -> a`，当我们的代码中出现严重问题时，我们可以调用它。我们给它一个字符串参数，这是要显示的错误消息。它的类型签名看起来很奇特：给定一个字符串，它如何能产生任何类型 a 的值？

它有一个结果类型 a，这样我们就可以在任何地方调用它，它将始终具有正确的类型。然而，它不像普通函数那样返回值。相反，它立即中止求值并打印我们给它的错误消息。

mySecond 函数返回其输入列表的第二个元素，但如果输入列表不够长则失败：

```haskell
-- file: ch03/MySecond.hs
mySecond :: [a] -> a
mySecond xs = if null (tail xs)
              then error "list too short"
              else head (tail xs)
```

像往常一样，我们可以在 ghci 中看到它是如何工作的：

```ghci
ghci> mySecond "xi"
'i'
ghci> mySecond [2]
*** Exception: list too short
ghci> head (mySecond [[9]])
*** Exception: list too short
```

注意第三个情况，我们尝试将调用 mySecond 的结果用作另一个函数的参数。求值仍然终止并让我们回到 ghci 提示符。这是使用 error 的主要弱点：它不让调用者区分可恢复的错误和真正应该终止我们程序的严重问题。

正如我们已经看到的，模式匹配失败会导致类似的不可恢复错误：

```ghci
ghci> mySecond []
*** Exception: Prelude.tail: empty list
```

## 一种更可控的方法

我们可以使用 Maybe 类型来表示错误的可能性。

如果我们想表示操作失败，我们可以使用 Nothing 构造函数。否则，我们用 Just 构造函数包装我们的值。

让我们看看如果返回 Maybe 值而不是调用 error，我们的 mySecond 函数会发生什么变化：

```haskell
-- file: ch03/MySecond.hs
safeSecond :: [a] -> Maybe a
safeSecond [] = Nothing
safeSecond xs = if null (tail xs)
                then Nothing
                else Just (head (tail xs))
```

如果传递给我们的列表太短，我们向调用者返回 Nothing。这让他们决定做什么，而调用 error 会强制崩溃：

```ghci
ghci> safeSecond []
Nothing
ghci> safeSecond [1]
Nothing
ghci> safeSecond [1,2]
Just 2
ghci> safeSecond [1,2,3]
Just 2
```

回到早期主题，我们可以使用模式匹配进一步提高这个函数的可读性：

```haskell
-- file: ch03/MySecond.hs
tidySecond :: [a] -> Maybe a
tidySecond (_:x:_) = Just x
tidySecond _       = Nothing
```

第一个模式仅在列表至少有两个元素长时匹配（它包含两个列表构造函数），并且它将变量 x 绑定到列表的第二个元素。如果第一个失败，则匹配第二个模式。

## 引入局部变量

在函数体内，我们可以在需要时使用 let 表达式引入新的局部变量。这是一个简单的函数，决定是否应该借钱给客户。我们的货币储备至少为 100，返回我们在扣除贷款金额后的新余额：

```haskell
-- file: ch03/Lending.hs
lend amount balance = let reserve    = 100
                          newBalance = balance - amount
                      in if balance < reserve
                         then Nothing
                         else Just newBalance
```

这里要注意的关键字是 let，它开始一个变量声明块，和 in，它结束它。每一行引入一个新变量。名称在 = 左边，绑定到它的表达式在右边。

### 特别说明

让我们重申我们的措辞：let 块中的名称绑定到一个表达式，而不是一个值。因为 Haskell 是一种惰性语言，与名称关联的表达式实际上不会被求值，直到需要它。在前面的例子中，如果我们不符合储备，我们就无法计算 newBalance 的值。

当我们在 let 块中定义一个变量时，我们称它为 let 绑定变量。这 simply means what it says：我们在 let 块中绑定了变量。

此外，我们在这里使用的空白很重要。我们稍后将在本章的"边线规则和表达式中的空白"第 64 页更详细地讨论布局规则。

我们可以在声明块中和 in 关键字后面的表达式中使用 let 块中变量的名称。

一般来说，我们将代码中我们可以使用名称的地方称为名称的作用域。如果我们可以使用一个名称，它就在作用域内；否则，它就在作用域外。如果名称在整个源文件中可见，我们称其在顶层。

## 遮蔽

我们可以在表达式中相互"嵌套"多个 let 块：

```haskell
-- file: ch03/NestedLets.hs
foo = let a = 1
      in let b = 2
         in a + b
```

在嵌套 let 表达式中重复变量名称是完全合法的，但确实不明智：

```haskell
-- file: ch03/NestedLets.hs
bar = let x = 1
      in ((let x = "foo" in x), x)
```

这里，内部的 x 隐藏或遮蔽了外部的 x。它具有相同的名称，但具有不同的类型和值：

```ghci
ghci> bar
("foo",1)
```

我们也可以遮蔽函数的参数，导致更奇怪的结果。这个函数的类型是什么？

```haskell
-- file: ch03/NestedLets.hs
quux a = let a = "foo"
         in a ++ "eek!"
```

因为函数的参数 a 由于被 let 绑定的 a 遮蔽而从不在函数体中使用，所以参数可以是任何类型：

```ghci
ghci> :type quux
quux :: t -> [Char]
```

### 编译器警告是您的朋友

遮蔽显然会导致混淆和讨厌的 bug，所以 GHC 有一个有用的 -fwarn-name-shadowing 选项。启用后，GHC 将在我们遮蔽名称时打印警告消息。

## where 子句

我们可以使用另一种机制来引入局部变量：where 子句。where 子句中的定义适用于其前面的代码。这是一个类似的函数借，用 where 而不是 let：

```haskell
-- file: ch03/Lending.hs
lend2 amount balance = if amount < reserve * 0.5
                       then Just newBalance
                       else Nothing
    where reserve    = 100
          newBalance = balance - amount
```

虽然 where 子句最初看起来很奇怪，但它提供了出色的可读性帮助。它让我们将读者的注意力引导到表达式的重要细节上，支持定义在后面出现。一段时间后，您可能会发现在使用缺乏它们的语言时想念 where 子句。

与 let 表达式一样，where 子句中的空白很重要。我们稍后将在"边线规则和表达式中的空白"第 64 页讨论布局规则。

## 局部函数、全局变量

您会注意到 Haskell 定义变量的语法看起来非常像定义函数的语法。这种对称性在 let 和 where 块中保持；我们定义局部函数就像定义局部变量一样容易：

```haskell
-- file: ch03/LocalFunction.hs
pluralise :: String -> [Int] -> [String]
pluralise word counts = map plural counts
    where plural 0 = "no " ++ word ++ "s"
          plural 1 = "one " ++ word
          plural n = show n ++ " " ++ word ++ "s"
```

我们定义了一个局部函数 plural，它由几个方程组成。局部函数可以自由使用封闭它们作用域中的变量；这里，我们使用外部函数 pluralise 定义中的 word。在 pluralise 的定义中，map 函数（我们将在下一章重新讨论）将局部函数 plural 应用到 counts 列表的每个元素。

我们也可以在源文件的顶层定义变量以及函数：

```haskell
-- file: ch03/GlobalVariable.hs
itemName = "Weighted Companion Cube"
```

## 边线规则和表达式中的空白

在我们对 lend 和 lend2 的定义中，我们的文本左边缘相当随意地徘徊。这不是偶然的；在 Haskell 中，空白有意义。

Haskell 使用缩进作为解析代码部分的提示。这种使用布局来传达结构有时称为边线规则。在源文件的开始，第一个顶层声明或定义可以从任何列开始，Haskell 编译器或解释器记住该缩进级别。每个后续顶层声明必须具有相同的缩进。

这里是顶层缩进规则的图示；我们的第一个文件 GoodIndent.hs，表现良好：

```haskell
-- file: ch03/GoodIndent.hs
-- 这是最左边的列。
  -- 顶层声明可以从任何列开始...
  firstGoodIndentation = 1
  -- ...只要所有后续声明也如此！
  secondGoodIndentation = 2
```

我们的第二个 BadIndent.hs，不遵守规则：

```haskell
-- file: ch03/BadIndent.hs
-- 这是最左边的列。
    -- 我们的第一个声明在第 4 列。
    firstBadIndentation = 1
  -- 我们的第二个在第一个的左边，这是非法的！
  secondBadIndentation = 2
```

当我们尝试将两个文件加载到 ghci 时会发生什么：

```ghci
ghci> :load GoodIndent.hs
[1 of 1] Compiling Main             ( GoodIndent.hs, interpreted )
Ok, modules loaded: Main.
ghci> :load BadIndent.hs
[1 of 1] Compiling Main             ( BadIndent.hs, interpreted )
BadIndent.hs:8:2: parse error on input `secondBadIndentation'
Failed, modules loaded: none.
```

空的后续行被视为当前项的延续，进一步向右缩进的后续行也是如此。

let 表达式和 where 子句的规则类似。在 let 或 where 关键字之后，Haskell 编译器或解释器记住它看到的下一个标记的缩进。如果接下来的行是空的，或者它的缩进更靠右，它被视为前一行继续。如果缩进与前一项的开始相同，它被视为在同一块中开始新项：

```haskell
-- file: ch03/Indentation.hs
foo = let firstDefinition = blah blah
          -- 仅注释的行被视为空
                              continuation blah
          -- 我们减少缩进，所以这是一个新定义
          secondDefinition = yada yada
                             continuation yada
      in whatever
```

这里是 let 和 where 的嵌套用法：

```haskell
-- file: ch03/letwhere.hs
bar = let b = 2
          c = True
      in let a = b
         in (a, c)
```

名称 a 仅在内层 let 表达式中可见——它不在外层 let 中可见。如果我们尝试在那里使用名称 a，会收到编译错误。缩进给了我们和编译器一个视觉提示，表明当前什么在作用域内：

```haskell
-- file: ch03/letwhere.hs
foo = x
    where x = y
              where y = 2
```

类似地，第一个 where 子句的作用域是 foo 的定义，但第二个的作用域只是第一个 where 子句。

我们用于 let 和 where 子句的缩进使我们的意图易于理解。

## 关于 Tab 与 Space 的注记

如果您使用 Haskell 感知文本编辑器（例如 Emacs），当您编辑 Haskell 源文件时，它可能已经配置为使用空格字符进行所有空白。如果您的编辑器不是 Haskell 感知的，您应该将其配置为仅使用空格字符。

原因是可移植性。在使用固定宽度字体的编辑器中，tab 停止按照惯例在类 Unix 系统（每八个字符）和 Windows（每四个字符）上的不同间隔放置。这意味着无论您个人对 tab 应该放在哪里有什么信念，您都不能依赖其他人的编辑器遵守您的偏好。任何使用 tab 的缩进在某些人的配置下都会看起来损坏。事实上，这可能导致编译问题，因为 Haskell 语言标准要求实现使用 Unix tab 宽度约定。

使用空格字符完全避免了这个问题的发生。

## 边线规则不是强制性的

我们可以使用显式结构而不是布局来指示我们的意思。为此，我们用左大括号开始一个方程块，用分号分隔每个项，并用右大括号结束块。以下两个 let 用法具有相同的含义：

```haskell
-- file: ch03/Braces.hs
bar = let a = 1
          b = 2
          c = 3
      in a + b + c

foo = let { a = 1;  b = 2;
        c = 3 }
      in a + b + c
```

当我们使用显式结构时，正常的布局规则不适用，这就是为什么我们可以在第二个 let 表达式中使用不寻常的缩进。

我们可以在通常使用布局的任何地方使用显式结构。它适用于 where 子句，甚至适用于顶层声明。只是记住，尽管存在该功能，但在 Haskell 程序中实际上几乎从未使用过显式结构。

## case 表达式

函数定义不是我们可以使用模式匹配的唯一地方。case 构造让我们可以在表达式中进行模式匹配。这是它的样子。这个函数（为我们定义在 Data.Maybe 中）解包一个 Maybe 值，如果值为 Nothing 则使用默认值：

```haskell
-- file: ch03/Guard.hs
fromMaybe defval wrapped =
    case wrapped of
      Nothing     -> defval
      Just value  -> value
```

case 关键字后跟一个任意表达式；模式匹配针对此表达式的结果执行。of 关键字表示表达式的结束和模式与表达式块开始。

块中的每个项由一个模式组成，后跟一个箭头 (->)，后跟一个表达式以在该模式匹配时求值。这些表达式必须都具有相同的类型。case 表达式的结果是第一个匹配模式的关联表达式的结果。匹配从上到下尝试。

要表达"如果其他模式都不匹配，则求值此表达式"，我们只需使用通配符模式 _ 作为模式列表中的最后一个。如果模式匹配失败，我们将获得与之前看到的相同的运行时错误。

## 常见的初学者模式错误

有一些方式是新 Haskell 程序员可能误解或滥用模式的。以下是一些模式匹配出错的尝试。根据您期望这些示例之一做什么，可能会有一些惊喜。

### 错误地匹配变量

看看以下代码：

```haskell
-- file: ch03/BogusPattern.hs
data Fruit = Apple | Orange
apple = "apple"
orange = "orange"

whichFruit :: String -> Fruit
whichFruit f = case f of
                 apple  -> Apple
                 orange -> Orange
```

草率的一瞥表明此代码正在检查值 f 以查看它是否匹配值 apple 或 orange。

如果我们以等式风格重写代码，就更容易发现错误：

```haskell
-- file: ch03/BogusPattern.hs
equational apple = Apple
equational orange = Orange
```

现在你能看出问题了吗？这里，更明显的是 apple 不引用名为 apple 的顶层值——它是一个局部模式变量。

### 不可反驳的模式

我们将始终成功的模式称为不可反驳的。普通变量名和通配符 _ (下划线) 是不可反驳模式的例子。

这是该函数的更正版本：

```haskell
-- file: ch03/BogusPattern.hs
betterFruit f = case f of
                  "apple"  -> Apple
                  "orange" -> Orange
```

我们通过匹配文字值 "apple" 和 "orange" 来修复问题。

### 错误地尝试比较相等性

如果我们想比较存储在 Tree 类型两个节点中的值，然后如果它们相等则返回其中一个怎么办？这里有一个尝试：

```haskell
-- file: ch03/BadTree.hs
bad_nodesAreSame (Node a _ _) (Node a _ _) = Just a
bad_nodesAreSame _            _            = Nothing
```

名称只能在一组模式绑定中出现一次。我们不能将变量放在多个位置来表示"此值与那个应该相同"的意思。

相反，我们将使用卫冕解决这个问题，这是另一个无价的 Haskell 功能。

## 使用卫冕的条件求值

模式匹配限制我们对值的形状执行固定测试。虽然这很有用，但我们经常希望在求值函数体之前进行更具表达力的检查。Haskell 提供了一个称为卫冕的功能，让我们做到这一点。我们将用我们编写的比较树两个节点的函数的修改来介绍这个想法：

```haskell
-- file: ch03/BadTree.hs
nodesAreSame (Node a _ _) (Node b _ _)
    | a == b     = Just a
nodesAreSame _ _ = Nothing
```

在这个例子中，我们使用模式匹配确保我们正在查看正确形状的值，并使用卫冕来比较它们的片段。

模式后面可以跟零个或多个卫冕，每个都是 Bool 类型的表达式。卫冕由 | 符号引入。后面是卫冕表达式，然后是 = 符号（如果我们在 case 表达式中则是 ->），然后是如果卫冕表达式求值为 True 要使用的主体。如果模式匹配，则关联该模式的每个卫冕按照它们编写的顺序求值。如果卫冕成功，与之关联的主体将用作函数的结果。如果没有任何卫冕成功，模式匹配移动到下一个模式。

当卫冕表达式求值时，模式关联中提到的所有变量都被绑定并且可以使用。

这是使用卫冕重写的我们的 lend 函数：

```haskell
-- file: ch03/Lending.hs
lend3 amount balance
     | amount <= 0            = Nothing
     | amount > reserve * 0.5 = Nothing
     | otherwise              = Just newBalance
    where reserve    = 100
          newBalance = balance - amount
```

看起来很特殊的卫冕表达式 otherwise 实际上是一个绑定到值 True 的变量，有助于可读性。

我们可以在可以使用模式的任何地方使用卫冕。将函数写为一组使用模式匹配和卫冕的方程可以使其更清晰。还记得我们在第 29 页的"条件求值"中定义的 myDrop 函数吗？

```haskell
-- file: ch02/myDrop.hs
myDrop n xs = if n <= 0 || null xs
              then xs
              else myDrop (n - 1) (tail xs)
```

这是使用模式和卫冕的重新表述：

```haskell
-- file: ch02/myDrop.hs
niceDrop n xs | n <= 0 = xs
niceDrop _ []          = []
niceDrop n (_:xs)      = niceDrop (n - 1) xs
```

这种风格上的改变让我们预先列举我们期望函数行为不同的情况。如果我们把决定埋在函数内部作为 if 表达式，代码就变得难以阅读。
