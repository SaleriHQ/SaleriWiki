# 第10章-案例研究-解析二进制数据格式

在本章中，我们将讨论一个常见任务：解析二进制文件。我们将把它用于两个目的。第一个确实是谈论一点解析，但我们的主要目标是谈论程序组织、重构和"样板代码消除"。我们将演示如何整理重复的代码，并为第 14 章关于 monads 的讨论做铺垫。

我们将使用的文件格式来自 netpbm 套件，这是一个古老而庄严的用于处理位图图像的程序和文件格式集合。这些文件格式具有双重优势：使用广泛，解析起来相当容易，虽然不是完全简单。对我们来说最方便的是，netpbm 文件是不压缩的。

## 灰度文件

netpbm 灰度文件格式的名称是 PGM（便携式灰度图）。它实际上不是一种格式，而是两种；纯文本（或 P2）格式编码为 ASCII，而更常见的原始（P5）格式主要是二进制。

任何一种格式的文件都以头开始，头又以描述格式的"魔数"字符串开始。对于纯文本文件，字符串是 P2，对于原始文件，它是 P5。魔数字符串后面是空白，然后是三个数字：图像的宽度、高度和最大灰度值。这些数字表示为 ASCII 十进制数字，由空白分隔。

在最大灰度值之后是图像数据。在原始文件中，这是一串二进制值。在纯文本文件中，值表示为由单个空格字符分隔的 ASCII 十进制数字。

原始文件可以包含一系列图像，一个接一个，每个都有自己的头。纯文本文件只包含一个图像。

## 解析原始 PGM 文件

对于我们的第一次解析函数尝试，我们只担心原始 PGM 文件。我们将把 PGM 解析器写成一个纯函数。它不负责获取要解析的数据，只负责实际解析。这是 Haskell 程序中的常见方法。通过将数据的读取与我们随后对其执行的操作分离，我们获得了从哪里获取数据的灵活性。

我们将使用 ByteString 类型来存储我们的灰度图数据，因为它很紧凑。由于 PGM 文件的头是 ASCII 文本但其主体是二进制的，我们同时导入面向文本和二进制的 ByteString 模块：

```haskell
-- file: ch10/PNM.hs
import qualified Data.ByteString.Lazy.Char8 as L8
import qualified Data.ByteString.Lazy as L
import Data.Char (isSpace)
```

就我们的目的而言，使用惰性还是严格的 ByteString 并不重要，所以我们有点随意地选择了惰性类型。

我们将使用一个直接的数据类型来表示 PGM 图像：

```haskell
-- file: ch10/PNM.hs
data Greymap = Greymap {
      greyWidth :: Int
    , greyHeight :: Int
    , greyMax :: Int
    , greyData :: L.ByteString
    } deriving (Eq)
```

通常，Haskell Show 实例应该产生一个字符串表示，我们可以通过调用 read 将其读回。然而，对于位图图形文件，这可能会产生巨大的文本字符串，例如，如果我们显示一张照片。出于这个原因，我们不让编译器自动为我们派生一个 Show 实例；我们将编写自己的并故意简化它：

```haskell
-- file: ch10/PNM.hs
instance Show Greymap where
    show (Greymap w h m _) = "Greymap " ++ show w ++ "x" ++ show h ++
                             " " ++ show m
```

因为我们的 Show 实例故意避免打印位图数据，所以写一个 Read 实例没有意义，因为我们无法从 show 的结果重建一个有效的 Greymap。

这是我们解析函数的一个明显类型：

```haskell
-- file: ch10/PNM.hs
parseP5 :: L.ByteString -> Maybe (Greymap, L.ByteString)
```

这将接受一个 ByteString，如果解析成功，它将返回解析后的 Greymap，以及解析后剩余的字符串。那个剩余字符串将可用于将来的解析。

我们的解析函数必须一次消费一点输入。首先，我们需要确保我们真的在看一个原始 PGM 文件；然后我们需要从头部的剩余部分解析数字；然后我们消费位图数据。

这里有一个表达这个的明显方式，我们将用它作为后续改进的基础：

```haskell
-- file: ch10/PNM.hs
matchHeader :: L.ByteString -> L.ByteString -> Maybe L.ByteString
-- "nat" 这里是的 "natural number" 简称
getNat :: L.ByteString -> Maybe (Int, L.ByteString)
getBytes :: Int -> L.ByteString
         -> Maybe (L.ByteString, L.ByteString)
parseP5 s =
  case matchHeader (L8.pack "P5") s of
    Nothing -> Nothing
    Just s1 ->
      case getNat s1 of
        Nothing -> Nothing
        Just (width, s2) ->
          case getNat (L8.dropWhile isSpace s2) of
            Nothing -> Nothing
            Just (height, s3) ->
              case getNat (L8.dropWhile isSpace s3) of
                Nothing -> Nothing
                Just (maxGrey, s4)
                  | maxGrey > 255 -> Nothing
                  | otherwise ->
                      case getBytes 1 s4 of
                        Nothing -> Nothing
                        Just (_, s5) ->
                          case getBytes (width * height) s5 of
                            Nothing -> Nothing
                            Just (bitmap, s6) ->
                              Just (Greymap width height maxGrey bitmap, s6)
```

这是一段非常字面的代码，在一个长的 case 表达式阶梯中执行所有解析。每个函数返回从其输入字符串消费完所需内容后的剩余 ByteString。我们将每个剩余字符串传递到下一步。我们依次解构每个结果，如果解析步骤失败则返回 Nothing，或者在我们进行时构建最终结果的一部分。以下是我们解析过程中应用的函数体（它们的类型被注释掉了，因为我们之前已经展示过）：

```haskell
-- file: ch10/PNM.hs
-- L.ByteString -> L.ByteString -> Maybe L.ByteString
matchHeader prefix str
    | prefix `L8.isPrefixOf` str
        = Just (L8.dropWhile isSpace (L.drop (L.length prefix) str))
    | otherwise
        = Nothing
-- L.ByteString -> Maybe (Int, L.ByteString)
getNat s = case L8.readInt s of
             Nothing -> Nothing
             Just (num,rest)
                 | num <= 0    -> Nothing
                 | otherwise -> Just (fromIntegral num, rest)
-- Int -> L.ByteString -> Maybe (L.ByteString, L.ByteString)
getBytes n str = let count           = fromIntegral n
                     both@(prefix,_) = L.splitAt count str
                 in if L.length prefix < count
                    then Nothing
                    else Just both
```

## 摆脱样板代码

虽然我们的 parseP5 函数可以工作，但我们编写它的风格不太令人满意。我们的代码稳步走向屏幕右侧，很明显，一个稍微复杂的函数很快就会用尽视觉空间。我们重复一种构造然后解构 Maybe 值的模式，只有在特定值匹配 Just 时才继续。所有相似的 case 表达式都充当样板代码，这些繁忙的工作掩盖了我们真正想要做的事情。简而言之，这个函数迫切需要一些抽象和重构。

如果我们退后一步，可以看到两种模式。首先是我们应用的许多函数有相似的类型。每个接受一个 ByteString 作为其最后一个参数并返回 Maybe something else。其次，我们的 parseP5 函数的"阶梯"中的每一步都解构一个 Maybe 值，要么失败，要么将解包的结果传递给一个函数。

我们可以很容易地写一个函数来捕获这第二种模式：

```haskell
-- file: ch10/PNM.hs
(>>?) :: Maybe a -> (a -> Maybe b) -> Maybe b
Nothing >>? _ = Nothing
Just v  >>? f = f v
```

(>>?) 函数非常简单：它接受一个值作为左参数，一个函数作为右参数。如果值不是 Nothing，它将函数应用于 Just 构造函数中包装的内容。我们将函数定义为运算符，这样我们就可以用它来链接函数。最后，我们没有为 (>>?) 提供 fixity 声明，所以它默认为 infixl 9（左结合、最强运算符优先级）。换句话说，a >>? b >>? c 将从左到右求值，如 (a >>? b) >>? c)。

有了这个链接函数，我们可以第二次尝试我们的解析函数：

```haskell
-- file: ch10/PNM.hs
parseP5_take2 :: L.ByteString -> Maybe (Greymap, L.ByteString)
parseP5_take2 s =
    matchHeader (L8.pack "P5") s      >>?
    \s -> skipSpace ((), s)           >>?
    (getNat . snd)                    >>?
    skipSpace                         >>?
    \(width, s) ->   getNat s         >>?
    skipSpace                         >>?
    \(height, s) ->  getNat s         >>?
    \(maxGrey, s) -> getBytes 1 s     >>?
    (getBytes (width * height) . snd) >>?
    \(bitmap, s) -> Just (Greymap width height maxGrey bitmap, s)
skipSpace :: (a, L.ByteString) -> Maybe (a, L.ByteString)
skipSpace (a, s) = Just (a, L8.dropWhile isSpace s)
```

理解这个函数的关键是思考链接。每个 (>>?) 左边是一个 Maybe 值；右边是一个返回 Maybe 值的函数。因此，每个左右表达式都是 Maybe 类型，适合传递给下一个 (>>?) 表达式。

我们为提高可读性所做的另一个更改是添加了一个 skipSpace 函数。通过这些更改，与我们原来的解析函数相比，代码行数减少了一半。通过删除样板 case 表达式，我们使代码更容易理解。

虽然我们在第 99 页的"匿名（lambda）函数"中警告过不要过度使用匿名函数，但我们在这里的函数链中使用了几个。因为这些函数很小，给它们起名字不会提高可读性。

## 隐式状态

我们还没有走出困境。我们的代码显式地传递对，使用一个元素用于解析结果的中间部分，另一个用于当前剩余的 ByteString。如果我们想扩展代码，例如，跟踪我们到目前为止消费了多少字节以便报告解析失败的位置，我们已经需要在八个不同的地方修改参数，只是为了传递一个三元组。

这种方法即使对于少量代码也会使其难以更改。问题在于我们使用模式匹配从每个对中拉出值：我们将我们总是处理对的知识直接嵌入到代码中。模式匹配是多么令人愉快和有帮助，但如果我们不小心使用它，它可能导致我们走向一些不希望的方向。

让我们做些什么来解决我们新代码的 inflexible。首先，我们将更改解析器使用的状态类型：

```haskell
-- file: ch10/Parse.hs
data ParseState = ParseState {
      string :: L.ByteString
    , offset :: Int64           -- imported from Data.Int
    } deriving (Show)
```

在我们切换到代数数据类型的过程中，我们添加了跟踪当前剩余字符串和我们开始解析以来在原始字符串中的偏移量的能力。更重要的变化是我们使用了记录语法：我们现在可以避免在传递的状态片段上进行模式匹配，而是使用访问函数 string 和 offset。

我们给解析状态起了一个名字。当我们给东西命名时，它可能更容易推理。例如，我们现在可以将解析看作一种函数：它消费一个解析状态并产生一个新的解析状态和一些其他信息。我们可以直接将其表示为一个 Haskell 类型：

```haskell
-- file: ch10/Parse.hs
simpleParse :: ParseState -> (a, ParseState)
simpleParse = undefined
```

为了向用户提供更多帮助，如果解析失败，我们希望报告错误消息。这只需要对解析器的类型进行一个小调整：

```haskell
-- file: ch10/Parse.hs
betterParse :: ParseState -> Either String (a, ParseState)
betterParse = undefined
```

为了使我们的代码面向未来，最好不要向用户暴露解析器的实现。当我们之前显式使用对来表示状态时，一旦我们考虑扩展解析器的能力，我们几乎立即陷入了麻烦。为了避免重复困难，我们将使用 newtype 声明来隐藏解析器类型的细节：

```haskell
-- file: ch10/Parse.hs
newtype Parse a = Parse {
      runParse :: ParseState -> Either String (a, ParseState)
    }
```

记住 newtype 定义只是一个编译时围绕一个函数的包装器，所以它没有运行时开销。当我们想使用函数时，我们将应用 runParser 访问器。

如果我们不从模块中导出 Parse 值构造函数，我们可以确保没有其他人能够意外地创建解析器，也不能通过模式匹配检查其内部结构。

## 恒等解析器

让我们尝试定义一个简单的解析器，即恒等解析器。它所做的只是将它被传递的任何东西转换为解析的结果。这样，它有点类似于 id 函数：

```haskell
-- file: ch10/Parse.hs
identity :: a -> Parse a
identity a = Parse (\s -> Right (a, s))
```

这个函数保持解析状态不变，并使用其参数作为解析的结果。我们在 Parse 类型中包装函数体以满足类型检查器。

我们如何使用这个包装函数来解析东西？

首先，我们必须剥离 Parse 包装器以便访问内部的函数。我们使用 runParse 函数来做到这一点。我们还需要构造一个 ParseState，然后在它上面运行我们的解析函数。最后，我们希望将解析的结果与最终的 ParseState 分开：

```haskell
-- file: ch10/Parse.hs
parse :: Parse a -> L.ByteString -> Either String a
parse parser initState
    = case runParse parser (ParseState initState 0) of
        Left err          -> Left err
        Right (result, _) -> Right result
```

因为恒等解析器和 parse 函数都不检查解析状态，我们甚至不需要创建输入字符串来尝试我们的代码：

```ghci
> :load Parse
[1 of 2] Compiling PNM              ( PNM.hs, interpreted )
[2 of 2] Compiling Parse            ( Parse.hs, interpreted )
Ok, modules loaded: Parse, PNM.
> :type parse (identity 1) undefined
parse (identity 1) undefined :: (Num t) => Either String t
> parse (identity 1) undefined
Loading package array-0.1.0.0 ... linking ... done.
Loading package bytestring-0.9.0.1.1 ... linking ... done.
Right 1
> parse (identity "foo") undefined
Right "foo"
```

一个甚至不检查其输入的解析器可能看起来不有趣，但我们很快就会看到它实际上是有用的。同时，我们获得了我们的类型正确且我们理解代码基本工作原理的信心。

## 记录语法、更新和模式匹配

记录语法不仅仅对访问函数有用——我们可以用它来复制和部分更改现有值。在使用中，符号看起来像这样：

```haskell
-- file: ch10/Parse.hs
modifyOffset :: ParseState -> Int64 -> ParseState
modifyOffset initState newOffset =
    initState { offset = newOffset }
```

这会创建一个与 initState 相同的新 ParseState 值，但其 offset 字段设置为我们为 newOffset 指定的任何值：

```ghci
> let before = ParseState (L8.pack "foo") 0
> let after = modifyOffset before 3
> before
ParseState {string = Chunk "foo" Empty, offset = 0}
> after
ParseState {string = Chunk "foo" Empty, offset = 3}
```

我们可以在花括号内设置尽可能多的字段，用逗号分隔。

## 一个更有趣的解析器

现在让我们专注于编写一个做有意义事情的解析器。我们不会太野心勃勃——我们只是想解析一个字节：

```haskell
-- file: ch10/Parse.hs
-- import the Word8 type from Data.Word
parseByte :: Parse Word8
parseByte =
    getState ==> \initState ->
    case L.uncons (string initState) of
      Nothing ->
          bail "no more input"
      Just (byte,remainder) ->
          putState newState ==> \_ ->
          identity byte
        where newState = initState { string = remainder,
                                     offset = newOffset }
              newOffset = offset initState + 1
```

我们的定义中有很多新函数。

L8.uncons 函数从 ByteString 中取出第一个元素：

```ghci
> L8.uncons (L8.pack "foo")
Just ('f',Chunk "oo" Empty)
> L8.uncons L8.empty
Nothing
```

我们的 getState 函数检索当前解析状态，而 putState 替换它。bail 函数终止解析并报告错误。(==>) 函数链接解析器。我们将很快介绍这些函数。

### 悬挂 lambda

parseByte 的定义有一种我们之前没有讨论过的视觉风格。它包含匿名函数，其中参数和 -> 位于行尾，函数体在下一行继续。

这种布局匿名函数的方式没有官方名称，所以我们称它为"悬挂 lambda"。它的主要用途是在函数体中为更多文本腾出空间。它还使得函数与其后面的函数之间的关系在视觉上更清晰。通常，例如，第一个函数的结果被作为参数传递给第二个函数。

## 获取和修改解析状态

我们的 parseByte 函数不接受解析状态作为参数。相反，它必须调用 getState 获取状态的副本，调用 putState 用新的状态替换当前状态：

```haskell
-- file: ch10/Parse.hs
getState :: Parse ParseState
getState = Parse (\s -> Right (s, s))
putState :: ParseState -> Parse ()
putState s = Parse (\_ -> Right ((), s))
```

阅读这些函数时，回想一下元组的左元素是 Parse 的结果，而右元素是当前 ParseState。这使得更容易理解这些函数在做什么。

getState 函数提取当前解析状态以便调用者可以访问字符串。putState 函数用新的解析状态替换当前解析状态。这将成为 (==>) 链中下一个函数将看到的状态。

这些函数让我们将显式状态处理移动到只有那些需要它的函数体中。许多函数不需要知道当前状态是什么，所以它们永远不会调用 getState 或 putState。这让我们能够写出比我们早期手动传递对的解析器更紧凑的代码。我们将在后面的一些代码中看到效果。

我们已经将解析状态的细节打包到 ParseState 类型中，并使用访问器而不是模式匹配来处理它。现在解析状态被隐式传递，我们获得了进一步的好处。如果我们想向解析状态添加更多信息，我们只需要修改 ParseState 的定义和任何需要新信息的函数体。与我们早期的解析代码相比，我们的所有状态都通过模式匹配暴露，这要 modular 得多：我们影响的唯一代码是需要新信息的代码。

## 报告解析错误

我们仔细定义了我们的 Parse 类型来容纳失败的可能性。(==>) 组合器检查解析失败并在遇到失败时停止解析。但我们还没有介绍 bail 函数，我们用它来报告解析错误：

```haskell
-- file: ch10/Parse.hs
bail :: String -> Parse a
bail err = Parse $ \s -> Left $
           "byte offset " ++ show (offset s) ++ ": " ++ err
```

在我们调用 bail 之后，(==>) 将成功地模式匹配它包装错误消息的 Left 构造函数，并且它不会调用链中的下一个解析器。这将导致错误消息通过先前调用者的链渗透回来。

## 将解析器链接在一起

(==>) 函数服务于与我们的 earlier (>>?) 函数类似的目的——它是让我们将函数链接在一起的"粘合剂"：

```haskell
-- file: ch10/Parse.hs
(==>) :: Parse a -> (a -> Parse b) -> Parse b
firstParser ==> secondParser  =  Parse chainedParser
  where chainedParser initState   =
          case runParse firstParser initState of
            Left errMessage ->
                Left errMessage
            Right (firstResult, newState) ->
                runParse (secondParser firstResult) newState
```

(==>) 的函数体有趣且稍微复杂。回想一下 Parse 类型实际上表示一个包装器内的函数。由于 (==>) 让我们链接两个 Parse 值来产生第三个，它必须返回一个包装器中的函数。

这个函数并没有真正"做"很多事情，它只是创建一个闭包来记住 firstParser 和 secondParser 的值。

闭包简单地说就是函数与其环境的配对，它能看到的绑定变量。闭包在 Haskell 中很常见。例如，section (+5) 是一个闭包。实现必须记录值 5 作为 (+) 运算符的第二个参数，这样生成的函数可以将其添加到你传递给它的任何值。

这个闭包不会被解包并应用，直到我们应用 parse。那时，它将被应用一个 ParseState。它将首先应用 firstParser 并检查其结果。如果解析失败，闭包也会失败。否则，它将把解析的结果和新的 ParseState 传递给 secondParser。

这真的是相当复杂和微妙的东西。我们有效地通过一个隐藏参数将 ParseState 向下传递到 Parse 值链中。（我们将在几章中重新审视这种代码，所以如果这个描述看起来很密集，不要烦恼。）

## 引入函子

我们现在完全熟悉 map 函数，它将函数应用于列表的每个元素，返回一个可能不同类型的列表：

```ghci
> map (+1) [1,2,3]
[2,3,4]
> map show [1,2,3]
["1","2","3"]
> :type map show
map show :: (Show a) => [a] -> [String]
```

这种类似 map 的活动在其他情况下也很有用。例如，考虑一个二叉树：

```haskell
-- file: ch10/TreeMap.hs
data Tree a = Node (Tree a) (Tree a)
            | Leaf a
              deriving (Show)
```

如果我们想获取一个字符串树并将其转换为一个包含这些字符串长度的树，我们可以写一个函数来做到这一点：

```haskell
-- file: ch10/TreeMap.hs
treeLengths (Leaf s) = Leaf (length s)
treeLengths (Node l r) = Node (treeLengths l) (treeLengths r)
```

既然我们的眼睛已经适应于寻找可以转化为通用函数的模式，我们在这里看到了一个可能的案例：

```haskell
-- file: ch10/TreeMap.hs
treeMap :: (a -> b) -> Tree a -> Tree b
treeMap f (Leaf a)   = Leaf (f a)
treeMap f (Node l r) = Node (treeMap f l) (treeMap f r)
```

正如我们可能希望的那样，treeLengths 和 treeMap length 给出相同的结果：

```ghci
> let tree = Node (Leaf "foo") (Node (Leaf "x") (Leaf "quux"))
> treeLengths tree
Node (Leaf 3) (Node (Leaf 1) (Leaf 4))
> treeMap length tree
Node (Leaf 3) (Node (Leaf 1) (Leaf 4))
> treeMap (odd . length) tree
Node (Leaf True) (Node (Leaf True) (Leaf False))
```

Haskell 提供了一个著名的类型类来进一步推广 treeMap。这个类型类名为 Functor，它定义了一个函数 fmap：

```haskell
-- file: ch10/TreeMap.hs
class Functor f where
    fmap :: (a -> b) -> f a -> f b
```

我们可以将 fmap 看作一种提升函数，正如我们在第 223 页的"用提升避免样板代码"中介绍的。它接受一个普通值的函数 a -> b，并将其提升为在容器 f a -> f b 上操作的函数，其中 f 是容器类型。

例如，如果我们将 Tree 替换类型变量 f，那么 fmap 的类型与 treeMap 的类型相同，事实上我们可以使用 treeMap 作为 Tree 上 fmap 的实现：

```haskell
-- file: ch10/TreeMap.hs
instance Functor Tree where
    fmap = treeMap
```

我们也可以使用 map 作为列表的 fmap 实现：

```haskell
-- file: ch10/TreeMap.hs
instance Functor [] where
    fmap = map
```

我们现在可以对不同的容器类型使用 fmap：

```ghci
> fmap length ["foo","quux"]
[3,4]
> fmap length (Node (Leaf "Livingstone") (Leaf "I presume"))
Node (Leaf 11) (Leaf 9)
```

Prelude 为几个常见类型定义了 Functor 的实例，特别是列表和 Maybe：

```haskell
-- file: ch10/TreeMap.hs
instance Functor Maybe where
    fmap _ Nothing  = Nothing
    fmap f (Just x) = Just (f x)
```

Maybe 的实例特别清楚地说明了 fmap 实现需要做什么。实现必须对类型的每个构造函数都有合理的行为。例如，如果一个值包装在 Just 中，fmap 实现对解包的值调用函数，然后用 Just 重新包装它。

Functor 的定义对我们使用 fmap 做一些明显的限制。例如，我们只能从恰好有一个类型参数的类 make Functor 的实例。

例如，我们不能为 Either a b 或 (a, b) 编写 fmap 实现，因为这些有两个类型参数。我们也不能为 Bool 或 Int 编写，因为它们没有类型参数。

此外，我们不能对我们的类型定义放置任何约束。这是什么意思？为了说明，让我们首先看一个正常的数据定义及其 Functor 实例：

```haskell
-- file: ch10/ValidFunctor.hs
data Foo a = Foo a

instance Functor Foo where
    fmap f (Foo a) = Foo (f a)
```

当我们定义一个新类型时，我们可以按照以下方式在 data 关键字后面添加一个类型约束：

```haskell
-- file: ch10/ValidFunctor.hs
data Eq a => Bar a = Bar a
instance Functor Bar where
    fmap f (Bar a) = Bar (f a)
```

这表示只有当 a 是 Eq 类型类的成员时，我们才能将类型 a 放入 Foo。然而，约束使编写 Bar 的 Functor 实例变得不可能：

```ghci
> :load ValidFunctor
[1 of 1] Compiling Main             ( ValidFunctor.hs, interpreted )
ValidFunctor.hs:12:12:
    Could not deduce (Eq a) from the context (Functor Bar)
      arising from a use of `Bar' at ValidFunctor.hs:12:12-16
    Possible fix:
      add (Eq a) to the context of the type signature for `fmap'
    In the pattern: Bar a
    In the definition of `fmap': fmap f (Bar a) = Bar (f a)
```

### 类型定义上的约束是不好的

向类型定义添加约束本质上从来都不是一个好主意。它的效果是强制你向将对那种类型的值进行操作的每个函数添加类型约束。让我们说我们需要一种栈数据结构，我们想查询其元素是否遵守某种排序。这里是数据类型的一个 naive 定义：

```haskell
-- file: ch10/TypeConstraint.hs
data (Ord a) => OrdStack a = Bottom
                           | Item a (OrdStack a)
                             deriving (Show)
```

如果我们想写一个函数来检查堆栈是否递增（即，每个元素都大于其下面的元素），我们显然需要 Ord 约束来执行成对比较：

```haskell
-- file: ch10/TypeConstraint.hs
isIncreasing :: (Ord a) => OrdStack a -> Bool
isIncreasing (Item a rest@(Item b _))
    | a < b     = isIncreasing rest
    | otherwise = False
isIncreasing _  = True
```

然而，因为我们写了类型定义上的类型约束，那个约束最终会在不需要的地方感染。我们需要在 push 上添加 Ord 约束，但它不关心堆栈上元素的顺序：

```haskell
-- file: ch10/TypeConstraint.hs
push :: (Ord a) => a -> OrdStack a -> OrdStack a
push a s = Item a s
```

尝试删除那个 Ord 约束，push 的定义将无法通过类型检查。

这就是为什么我们之前尝试为 Bar 编写 Functor 实例失败的原因：它需要 Eq 约束以某种方式追溯添加到 fmap 的签名中。

既然我们初步确定在类型定义上放置类型约束是 Haskell 的一个缺点，什么是更明智的替代方案？答案很简单：从类型定义中省略类型约束，而是将它们放在真正需要的函数上。

在这个例子中，我们可以从 OrdStack 和 push 中删除 Ord 约束。它需要留在 isIncreasing 上，否则它不能调用 (<)。我们现在把约束放在真正重要的地方。这还有一个额外的好处，就是使类型签名更好地记录每个函数的真正需求。

大多数 Haskell 容器类型遵循这种模式。Data.Map 模块中的 Map 类型要求其键是可排序的，但类型本身没有这样的约束。约束表示在真正需要的函数上，如 insert，而不是在 size 上，那里不使用排序。

## fmap 的中缀使用

很多时候，你会看到 fmap 作为运算符调用：

```ghci
> (1+) `fmap` [1,2,3] ++ [4,5,6]
[2,3,4,4,5,6]
```

也许奇怪的是，普通的旧 map 几乎从不这样使用。

fmap-as-operator 模式持久性的一个可能原因是这种使用让我们可以省略其第二个参数的括号。较少的括号导致在阅读函数时减少精神杂耍：

```ghci
> fmap (1+) ([1,2,3] ++ [4,5,6])
[2,3,4,5,6,7]
```

如果你真的想将 fmap 作为运算符使用，Control.Applicative 模块包含一个名为 (<$>) 的运算符，它是 fmap 的别名。它名字中的 $ 体现了将函数应用于其参数（使用 ($) 运算符）和将函数提升到函子中之间的相似性。当我们回到我们一直在写的代码时，我们会看到这在解析方面效果很好。

## 灵活的实例

你可能希望我们能为类型 Either Int b 编写一个 Functor 实例，它有一个类型参数：

```haskell
-- file: ch10/EitherInt.hs
instance Functor (Either Int) where
    fmap _ (Left n) = Left n
    fmap f (Right r) = Right (f r)
```

然而，Haskell 98 的类型系统不能保证检查这种实例上的约束会终止。非终止约束检查可能会将编译器送入无限循环，所以这种形式的实例是被禁止的：

```ghci
> :load EitherInt
[1 of 1] Compiling Main             ( EitherInt.hs, interpreted )
EitherInt.hs:2:0:
    Illegal instance declaration for `Functor (Either Int)'
        (All instance types must be of the form (T a1 ... an)
         where a1 ... an are type *variables*,
         and each type variable appears at most once in the instance head.
         Use -XFlexibleInstances if you want to disable this.)
    In the instance declaration for `Functor (Either Int)'

```

GHC 比基础的 Haskell 98 标准有更强大的类型系统。它默认在 Haskell 98 兼容模式下运行，以获得最大的可移植性。我们可以使用特殊的编译器指令指示它允许更灵活的实例：

```haskell
-- file: ch10/EitherIntFlexible.hs

instance Functor (Either Int) where
```


## 更多思考函子

我们已经对函子应该如何工作做出了一些隐含的假设。将这些明确化并将它们视为要遵循的规则是有帮助的，因为这让我们将函子视为统一的、行为良好的对象。我们只需要记住两条规则，它们很简单：

- 我们的第一条规则是函子必须保持恒等。也就是说，对一个值应用 fmap id 应该给我们返回一个相同的值：

```ghci
> fmap id (Node (Leaf "a") (Leaf "b"))
Node (Leaf "a") (Leaf "b")
```

- 我们的第二条规则是函子必须是可组合的。也就是说，组合 fmap 的两次使用应该给出与使用相同函数组合的单个 fmap 相同的结果：

```ghci
> (fmap even . fmap length) (Just "twelve")
Just True
> fmap (even . length) (Just "twelve")
Just True
```

看待这两条规则的另一种方式是函子必须保持形状。集合的结构不应被函子影响；只有它包含的值应该改变：

```ghci
> fmap odd (Just 1)
Just True
> fmap odd Nothing
Nothing
```

如果你正在编写一个 Functor 实例，记住这些规则很有用，事实上要测试它们，因为编译器无法检查我们刚刚列出的规则。另一方面，如果你只是使用函子，规则足够"自然"，不需要记住它们。它们只是形式化了一些"做我想做的"的直观概念。

以下是预期行为的伪代码表示：

```haskell
-- file: ch10/FunctorLaws.hs
fmap id       ==  id
fmap (f . g)  ==  fmap f . fmap g
```

## 为 Parse 编写 Functor 实例

对于我们到目前为止看到的类型，我们应该对 fmap 的行为有明显的预期。对于 Parse，由于其复杂性，这就不太清楚了。一个合理的猜测是，我们正在映射的函数应该应用于解析的当前结果，并保持解析状态不变：

```haskell
-- file: ch10/Parse.hs
instance Functor Parse where
    fmap f parser = parser ==> \result ->
                    identity (f result)
```

这个定义很容易阅读，所以让我们进行一些快速实验，看看我们是否遵循了函子的规则。

首先，我们检查恒等是否被保持。让我们首先在应该失败的解析上尝试——从空字符串解析一个字节（记住 (<$>) 是 fmap）：

```ghci
> parse parseByte L.empty
Left "byte offset 0: no more input"
> parse (id <$> parseByte) L.empty
Left "byte offset 0: no more input"
```

好的。现在对于应该成功的解析：

```ghci
> let input = L8.pack "foo"
> L.head input
102
> parse parseByte input
Right 102
> parse (id <$> parseByte) input
Right 102
```

检查这些结果，我们还可以看到我们的 Functor 实例遵守我们保持形状的第二条规则。失败保持为失败，成功保持为成功。

最后，我们将确保可组合性被保持：

```ghci
> parse ((chr . fromIntegral) <$> parseByte) input
Right 'f'
> parse (chr <$> fromIntegral <$> parseByte) input
Right 'f'
```

基于这个 brief 检查，我们的 Functor 实例似乎是行为良好的。

## 使用函子进行解析

所有这些关于函子的讨论都有一个目的：它们经常让我们写出整洁、表达力强的代码。

回想一下我们之前介绍的 parseByte 函数。在重塑我们的 PGM 解析器以使用新的解析器基础设施时，我们经常希望使用 ASCII 字符而不是 Word8 值。

虽然我们可以写一个与 parseByte 结构相似的 parseChar 函数，但我们现在可以利用 Parse 的函子特性来避免这种代码重复。我们需要的是一个将 Word8 转换为 Char 的函数：

```haskell
-- file: ch10/Parse.hs
w2c :: Word8 -> Char
w2c = chr . fromIntegral
-- import Control.Applicative
parseChar :: Parse Char
parseChar = w2c <$> parseByte
```

我们也可以使用函子写一个紧凑的"peek"函数。如果我们在输入字符串的末尾，它返回 Nothing。否则，它返回下一个字符而不消费它（即，它检查但不干扰当前解析状态）：

```haskell
-- file: ch10/Parse.hs
peekByte :: Parse (Maybe Word8)
peekByte = (fmap fst . L.uncons . string) <$> getState
```

让我们定义 parseChar 的相同提升技巧：

```haskell
-- file: ch10/Parse.hs
peekChar :: Parse (Maybe Char)
peekChar = fmap w2c <$> peekByte
```

注意 peekByte 和 peekChar 每个都进行两次 fmap 调用，其中一次伪装成 (<$>)。这是必要的，因为类型 Parse (Maybe a) 是函子内的函子。因此，我们必须将函数提升两次以"获取"内部函子。

最后，我们将编写另一个通用组合器，它是熟悉的 takeWhile 的 Parse 类似物。它在其谓词返回 True 时消费其输入：

```haskell
-- file: ch10/Parse.hs
parseWhile :: (Word8 -> Bool) -> Parse [Word8]
parseWhile p = (fmap p <$> peekByte) ==> \mp ->
               if mp == Just True
               then parseByte ==> \b ->
                    (b:) <$> parseWhile p
               else identity []
```

再一次，我们在几个地方使用函子（必要时双重使用）来减少代码的冗长。这里是同一个函数的一种更直接的风格重写，不使用函子：

```haskell
-- file: ch10/Parse.hs
parseWhileVerbose p =
    peekByte ==> \mc ->
    case mc of
      Nothing -> identity []
      Just c | p c ->
                 parseByte ==> \b ->
                 parseWhileVerbose p ==> \bs ->
                 identity (b:bs)
             | otherwise ->
                 identity []
```

当你不太熟悉函子时，更冗长的定义可能更容易阅读。然而，在 Haskell 代码中使用函子足够普遍，更紧凑的表示应该很快变得第二天性（无论是阅读还是编写）。

## 重写我们的 PGM 解析器

使用我们新的解析代码，原始 PGM 解析函数现在是什么样子？

```haskell
-- file: ch10/Parse.hs
parseRawPGM =
    parseWhileWith w2c notWhite ==> \header -> skipSpaces ==>&
    assert (header == "P5") "invalid raw header" ==>&
    parseNat ==> \width -> skipSpaces ==>&
    parseNat ==> \height -> skipSpaces ==>&
    parseNat ==> \maxGrey ->
    parseByte ==>&
    parseBytes (width * height) ==> \bitmap ->
    identity (Greymap width height maxGrey bitmap)
  where notWhite = (`notElem` " \r\n\t")
```

这个定义利用了几个更多的辅助函数，我们现在按照熟悉的方式呈现它们：

```haskell
-- file: ch10/Parse.hs
parseWhileWith :: (Word8 -> a) -> (a -> Bool) -> Parse [a]
parseWhileWith f p = fmap f <$> parseWhile (p . f)
parseNat :: Parse Int
parseNat = parseWhileWith w2c isDigit ==> \digits ->
           if null digits
           then bail "no more input"
           else let n = read digits
                in if n < 0
                   then bail "integer overflow"
                   else identity n
(==>&) :: Parse a -> Parse b -> Parse b
p ==>& f = p ==> \_ -> f
skipSpaces :: Parse ()
skipSpaces = parseWhileWith w2c isSpace ==>& identity ()
assert :: Bool -> String -> Parse ()
assert True  _   = identity ()
assert False err = bail err
```

(==>&) 组合器链接像 (==>) 这样的解析器，但右边忽略左边的结果。assert 函数让我们检查一个属性，如果属性为 False 则中止解析并显示有用的错误消息。

注意我们写的函数中很少有引用当前解析状态的。特别值得注意的是，我们旧的 parseP5 函数显式地将二元组向下传递到数据流链中，而 parseRawPGM 中的所有状态管理都对我们隐藏了。

当然，我们不能完全避免检查和修改解析状态。这里有一个恰当的例子，parseRawPGM 所需的最后一个辅助函数：

```haskell
-- file: ch10/Parse.hs
parseBytes :: Int -> Parse L.ByteString
parseBytes n =
    getState ==> \st ->
    let n' = fromIntegral n
        (h, t) = L.splitAt n' (string st)
        st' = st { offset = offset st + L.length h, string = t }
    in putState st' ==>&
       assert (L.length h == n') "end of input" ==>&
       identity h
```
