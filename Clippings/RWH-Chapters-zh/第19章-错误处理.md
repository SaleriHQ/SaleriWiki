## 第十九章：错误处理

错误处理是程序员最重要的——也是最容易被忽视的——主题之一，无论使用何种语言。在 Haskell 中，你会发现两种主要的错误处理方式：纯错误处理和异常。

当我们谈论纯错误处理时，我们指的是不需要从 IO monad 中获取任何东西的算法。我们通常可以简单地利用 Haskell 富有表现力的数据类型系统来实现错误处理。Haskell 也有异常系统。由于惰性求值的复杂性，Haskell 中的异常可以在任何地方抛出，但只能在 IO monad 中捕获。在本章中，我们将同时考虑这两种方式。

## 使用数据类型进行错误处理

让我们从一个非常简单的函数开始讨论错误处理。假设我们希望对一系列数字进行除法运算。我们有一个固定的分子，但希望改变分母。我们可能会想到这样一个函数：

```haskell
-- file: ch19/divby1.hs
divBy :: Integral a => a -> [a] -> [a]
divBy numerator = map (numerator `div`)
```

非常简单，对吧？我们可以在 ghci 中稍微玩一下：

```ghci
ghci> divBy 50 [1,2,5,8,10]
[50,25,10,6,5]
ghci> take 5 (divBy 100 [1..])
[100,50,33,25,20]
```

这符合预期：50 / 1 是 50，50 / 2 是 25，以此类推。* 这甚至可以处理无限列表 [1..]。如果我们在列表中偷偷放入一个 0 会发生什么？

* 我们在这里使用整数除法，所以 50 / 8 显示为 6 而不是 6.25。我们在这个例子中没有使用浮点数运算，因为用 Double 做除以零会产生特殊的值 Infinity 而不是错误。

```ghci
ghci> divBy 50 [1,2,0,8,10]
[50,25,*** Exception: divide by zero
```

这很有趣吧？ghci 开始显示输出，然后在遇到零时停止并抛出异常。这是惰性求值在起作用——它按需计算结果。

正如我们将在本章后面看到的，在没有显式异常处理器的情况下，这个异常会使程序崩溃。这显然是不可取的，所以让我们考虑一些更好的方式来在这个纯函数中表示错误。

## Maybe 的使用

一种立即可识别且简单的方式来表示失败是使用 Maybe。† 代替只返回一个列表并在失败时抛出异常，如果输入列表在任何地方包含零，我们可以返回 Nothing，否则返回 Just 和结果。下面是这种算法的一个实现：

```haskell
-- file: ch19/divby2.hs
divBy :: Integral a => a -> [a] -> Maybe [a]
divBy _ [] = Just []
divBy _ (0:_) = Nothing
divBy numerator (denom:xs) =
    case divBy numerator xs of
      Nothing -> Nothing
      Just results -> Just ((numerator `div` denom) : results)
```

如果你在 ghci 中试用，你会发现它可以正常工作：

```ghci
ghci> divBy 50 [1,2,5,8,10]
Just [50,25,10,6,5]
ghci> divBy 50 [1,2,0,8,10]
Nothing
```

调用 divBy 的函数现在可以使用 case 语句来查看调用是否成功，就像 divBy 在调用自己时所做的那样。

你可能注意到你可以使用前面例子的单子式实现，像这样：

```haskell
-- file: ch19/divby2m.hs
divBy :: Integral a => a -> [a] -> Maybe [a]
divBy numerator denominators =
    mapM (numerator `safeDiv`) denominators
    where safeDiv _ 0 = Nothing
          safeDiv x y = x `div` y
```

为了简单起见，我们将在本章中避免使用单子式实现，但想指出它是存在的。

† 关于 Maybe 的介绍，请参阅"更受控的方法"第 61 页。

## 惰性的损失与保持

Maybe 的使用很方便，但它是需要付出代价的。divBy 不再能处理无限列表作为输入。由于结果是 Maybe [a]，在确定我们不会因为列表中某处的零而返回 Nothing 之前，必须检查整个输入列表。你可以通过尝试我们之前的例子来验证情况确实如此：

```ghci
ghci> divBy 100 [1..]
*** Exception: stack overflow
```

请注意，在这里你不会看到部分输出；你什么都看不到。注意，在 divBy 的每一步（除了输入列表为空或零在列表开头的情况），都必须知道每个后续元素的结果，然后才能知道当前元素的结果。因此这个算法不能处理无限列表，对于大型有限列表来说也不是很节省空间。

话虽如此，Maybe 通常是一个不错的选择。在这个特定情况下，我们要到评估整个输入之后才知道是否会有问题。有时我们提前知道问题——例如，ghci 中的 tail [] 会产生异常。我们可以很容易地写一个没有这个问题的无限能力的 tail：

```haskell
-- file: ch19/safetail.hs
safeTail :: [a] -> Maybe [a]
safeTail [] = Nothing
safeTail (_:xs) = Just xs
```

这只是在给定空输入列表时返回 Nothing，对于其他任何情况则返回 Just 和结果。由于我们只需要在知道是否有错误之前确保列表是非空的，在这里使用 Maybe 不会减少我们的惰性。我们可以在 ghci 中测试它，并与常规 tail 进行比较：

```ghci
ghci> tail [1,2,3,4,5]
[2,3,4,5]
ghci> safeTail [1,2,3,4,5]
Just [2,3,4,5]
ghci> tail []
*** Exception: Prelude.tail: empty list
ghci> safeTail []
Nothing
```

在这里，我们可以看到 safeTail 的表现符合预期。但是无限列表呢？我们不想打印出无限数量的结果，所以我们可以测试 take 5 (tail [1..]) 和 safeTail 的类似构造：

```ghci
ghci> take 5 (tail [1..])
[2,3,4,5,6]
ghci> case safeTail [1..] of {Nothing -> Nothing; Just x -> Just (take 5 x)}
Just [2,3,4,5,6]
ghci> take 5 (tail [])
*** Exception: Prelude.tail: empty list
ghci> case safeTail [] of {Nothing -> Nothing; Just x -> Just (take 5 x)}
Nothing
```

在这里，你可以看到 tail 和 safeTail 都能很好地处理无限列表。注意，我们能够更好地处理空输入列表；我们决定在那种情况下返回 Nothing 而不是抛出异常。我们能够在不牺牲惰性的情况下实现错误处理。

但是我们如何将其应用到我们的 divBy 例子中呢？让我们考虑一下那里的情况。失败是个体坏输入的属性，而不是输入列表本身的属性。与其让失败成为输出列表本身的属性，不如让它成为单个输出元素的属性，怎么样？也就是说，代替类型 a -> [a] -> Maybe [a] 的函数，我们将是 a -> [a] -> [Maybe a]。这将有利于保持惰性，加上调用者将能够准确确定问题在列表中的位置——或者如果需要，甚至可以过滤掉有问题结果。下面是一个实现：

```haskell
-- file: ch19/divby3.hs
divBy :: Integral a => a -> [a] -> [Maybe a]
divBy numerator denominators =
    map worker denominators
    where worker 0 = Nothing
          worker x = Just (numerator `div` x)
```

看看这个函数。我们又回到了使用 map，这对惰性和简单性都是好事。我们可以在 ghci 中试用它，验证它对有限和无限列表都能正常工作：

```ghci
ghci> divBy 50 [1,2,5,8,10]
[Just 50,Just 25,Just 10,Just 6,Just 5]
ghci> divBy 50 [1,2,0,8,10]
[Just 50,Just 25,Nothing,Just 6,Just 5]
ghci> take 5 (divBy 100 [1..])
[Just 100,Just 50,Just 33,Just 25,Just 20]
```

我们希望你能从这段讨论中得出一个要点：输入格式不正确（就像 safeTail 的情况一样）和输入可能包含一些坏数据（就像 divBy 的情况）之间是有区别的。这两种情况通常可以证明对结果使用不同的处理方式。

## Maybe 单子 Monad 的使用

回到第 448 页的"May be 的使用"，我们有一个名为 divby2.hs 的示例程序。这个例子没有保持惰性，但返回了类型 Maybe [a] 的值。完全相同的算法可以用单子风格表达。有关单子的更多信息和重要的背景知识，请参阅第 14 章。下面是我们新的单子风格算法：

```haskell
-- file: ch19/divby4.hs
divBy :: Integral a => a -> [a] -> Maybe [a]
divBy _ [] = return []
divBy _ (0:_) = fail "division by zero in divBy"
divBy numerator (denom:xs) =
    do next <- divBy numerator xs
       return ((numerator `div` denom) : next)
```

Maybe 单子使这个算法的表达看起来更漂亮。对于 Maybe 单子，return 和 Just 一样，而 fail _ = Nothing，所以我们的错误解释字符串实际上从未在任何地方被看到。如果我们愿意，可以用我们之前对 divby2.hs 使用的相同测试来测试这个算法：

```ghci
ghci> divBy 50 [1,2,5,8,10]
Just [50,25,10,6,5]
ghci> divBy 50 [1,2,0,8,10]
Nothing
ghci> divBy 100 [1..]
*** Exception: stack overflow
```

我们编写的代码实际上并不特定于 Maybe 单子。通过简单地更改类型，我们就可以让它适用于任何单子。让我们试试：

```haskell
-- file: ch19/divby5.hs
divBy :: Integral a => a -> [a] -> Maybe [a]
divBy = divByGeneric
divByGeneric :: (Monad m, Integral a) => a -> [a] -> m [a]
divByGeneric _ [] = return []
divByGeneric _ (0:_) = fail "division by zero in divByGeneric"
divByGeneric numerator (denom:xs) =
    do next <- divByGeneric numerator xs
       return ((numerator `div` denom) : next)
```

divByGeneric 函数包含与之前 divBy 相同的代码；我们只是给了它一个更通用的类型。实际上，如果不给定类型，这是 ghci 推断的类型。我们还定义了一个具有更具体类型的便捷函数 divBy。

让我们在 ghci 中试试：

```ghci
ghci> :l divby5.hs
[1 of 1] Compiling Main             ( divby5.hs, interpreted )
Ok, modules loaded: Main.
ghci> divBy 50 [1,2,5,8,10]
Just [50,25,10,6,5]
ghci> (divByGeneric 50 [1,2,5,8,10])::(Integral a => Maybe [a])
Just [50,25,10,6,5]
ghci> divByGeneric 50 [1,2,5,8,10]
[50,25,10,6,5]
ghci> divByGeneric 50 [1,2,0,8,10]
*** Exception: user error (division by zero in divByGeneric)
```

前两个输入都产生了我们之前看到的相同输出。由于 divByGeneric 没有特定的返回类型，我们必须给它一个或让解释器从环境中推断它。如果我们不给定特定返回类型，ghci 会推断为 IO 单子。你可以在第三个和第四个例子中看到这一点。IO 单子将 fail 转换为一个异常，你可以在第四个例子中看到。

Control.Monad.Error 模块在 mtl 包中将 Either String 制作成一个单子。如果你使用 Either，你可以得到保留错误消息的纯结果，像这样：

```ghci
ghci> :m +Control.Monad.Error
ghci> (divByGeneric 50 [1,2,5,8,10])::(Integral a => Either String [a])
Loading package mtl-1.1.0.1 ... linking ... done.
Right [50,25,10,6,5]
ghci> (divByGeneric 50 [1,2,0,8,10])::(Integral a => Either String [a])
Left "division by zero in divByGeneric"
```

这将我们带入下一个讨论话题：使用 Either 返回错误信息。

## Either 的使用

Either 类型与 Maybe 类型类似，但有一个关键区别：它可以为错误和成功（"正确答案"）附加数据。虽然语言没有施加限制，但按照惯例，返回 Either 的函数使用 Left 返回值来表示错误，并使用 Right 表示成功。如果这能帮助你记住，你可以想想得到了正确答案（getting the Right answer）。我们可以从前面关于 Maybe 一节中的 divby2.hs 示例开始，并使其适应 Either：

```haskell
-- file: ch19/divby6.hs
divBy :: Integral a => a -> [a] -> Either String [a]
divBy _ [] = Right []
divBy _ (0:_) = Left "divBy: division by 0"
divBy numerator (denom:xs) =
    case divBy numerator xs of
      Left x -> Left x
      Right results -> Right ((numerator `div` denom) : results)
```

这段代码与 Maybe 代码几乎完全相同；我们将每个 Just 替换为 Right。Left 比 Nothing 好，但现在它可以携带消息。让我们在 ghci 中检查一下：

```ghci
ghci> divBy 50 [1,2,5,8,10]
Right [50,25,10,6,5]
ghci> divBy 50 [1,2,0,8,10]
Left "divBy: division by 0"
```

## 用于错误的自定义数据类型

虽然表示错误原因的 String 对人类可能有用，但通常定义一个自定义错误类型是很有帮助的，我们可以根据确切的问题类型以编程方式决定采取什么行动。例如，假设由于某种原因，除了 0 之外，我们也不想除以 10 或 20。我们可以定义一个自定义错误类型，像这样：

```haskell
-- file: ch19/divby7.hs
data DivByError a = DivBy0
                 | ForbiddenDenominator a
                   deriving (Eq, Read, Show)
divBy :: Integral a => a -> [a] -> Either (DivByError a) [a]
```

‡ 有关 Either 的更多信息，请参阅"通过 API 设计处理错误"第 210 页。

divBy _ [] = Right []
divBy _ (0:_) = Left DivBy0
divBy _ (10:_) = Left (ForbiddenDenominator 10)
divBy _ (20:_) = Left (ForbiddenDenominator 20)
divBy numerator (denom:xs) =
    case divBy numerator xs of
      Left x -> Left x
      Right results -> Right ((numerator `div` denom) : results)

现在，在发生错误的情况下，可以检查 Left 数据以找出确切的原因。或者，它也可以简单地用 show 打印出来，这将生成一个关于问题的合理想法。下面是这个函数的实际运行：

```ghci
ghci> divBy 50 [1,2,5,8]
Right [50,25,10,6]
ghci> divBy 50 [1,2,5,8,10]
Left (ForbiddenDenominator 10)
ghci> divBy 50 [1,2,0,8,10]
Left DivBy0
```

所有这些 Either 例子都遭受了我们早期 Maybe 例子缺乏惰性的问题。我们在本章末尾的练习问题中解决了这个问题。

## Either 的单子使用

回到第 450 页的"Maybe 单子的使用"，我们向你展示了如何在单子中使用 Maybe。Either 也可以在单子中使用，但这可能会稍微复杂一些。原因是 fail 被硬编码为只接受 String 作为失败代码，所以我们必须有一种方法将这样的字符串映射到我们用于 Left 的任何类型。正如你之前看到的，Control.Monad.Error 为 Either String a 提供了内置支持，这涉及对 fail 的参数不需要映射。下面是我们如何设置我们的例子来使用单子风格的 Either：

```haskell
-- file: ch19/divby8.hs
{-# LANGUAGE FlexibleContexts #-}
import Control.Monad.Error
data Show a =>
    DivByError a = DivBy0
                  | ForbiddenDenominator a
                  | OtherDivByError String
                    deriving (Eq, Read, Show)
instance Error (DivByError a) where
    strMsg x = OtherDivByError x
divBy :: Integral a => a -> [a] -> Either (DivByError a) [a]
divBy = divByGeneric
divByGeneric :: (Integral a, MonadError (DivByError a) m) =>
                 a -> [a] -> m [a]
```

divByGeneric _ [] = return []
divByGeneric _ (0:_) = throwError DivBy0
divByGeneric _ (10:_) = throwError (ForbiddenDenominator 10)
divByGeneric _ (20:_) = throwError (ForbiddenDenominator 20)
divByGeneric numerator (denom:xs) =
    do next <- divByGeneric numerator xs
       return ((numerator `div` denom) : next)

在这里，我们需要打开 FlexibleContexts 语言扩展，以便为 divByGeneric 提供类型签名。divBy 函数的工作方式与之前完全相同。对于 divByGeneric，我们通过定义当有人调用 fail 时会发生什么（strMsg 函数）使 DivByError 成为 Error 类的成员。我们还将 Right 转换为 return，将 Left 转换为 throwError，以使其通用。

## 异常

GHC 6.10.1 版本在本书付印时发布了。它引入了一个可扩展的扩展系统。在接下来的部分中，我们记录了较旧的异常系统。两者相似，但并不完全兼容。

异常处理存在于许多编程语言中，包括 Haskell。它可能很有用，因为当问题发生时，异常处理可以提供一种简单的处理方式，即使它发生在通过函数调用链的若干层之下。使用异常，不需要检查每个函数调用的返回值以获取错误，就像 C 程序员必须做的那样，我们必须注意生成反映错误的返回值。在 Haskell 中，由于单子和 Either 及 Maybe 类型，我们通常可以在纯代码中实现相同的效果，而不需要使用异常和异常处理。

一些问题——特别是涉及 I/O 的问题——需要使用异常。在 Haskell 中，异常可以从程序的任何位置抛出。然而，由于求值顺序未指定，它们只能被在 IO monad 中捕获。Haskell 异常处理不涉及像 Python 或 Java 中那样的特殊语法。相反，捕获和处理异常的机制是——意想不到的——函数。

## 异常的第一步

在 Control.Exception 模块中，定义了各种与异常相关的函数和类型。那里定义了一个 Exception 类型；所有异常都是 Exception 类型的。还有用于捕获和处理异常的函数。让我们首先看看 try，它的类型是 IO a -> IO (Either Exception a)。这用异常处理包装了一个 IO 操作。如果抛出异常，它将返回一个带有异常的 Left 值；否则返回一个带有原始结果的 Right 值。让我们在 ghci 中试试。我们将首先触发一个未处理的异常，然后尝试捕获它：

```ghci
ghci> :m Control.Exception
ghci> let x = 5 `div` 0
ghci> let y = 5 `div` 1
ghci> print x
*** Exception: divide by zero
ghci> print y
5
ghci> try (print x)
Left divide by zero
ghci> try (print y)
5
Right ()
```

注意 let 语句没有抛出异常。由于惰性求值，这是预期的；除以零不会被执行，直到被打印 x 的尝试所要求。另外，注意 try (print y) 有两行输出。第一行是由 print 产生的，它在终端上显示了数字 5。第二行是由 ghci 产生的，它显示 print y 返回了 () 且没有抛出异常。

## 惰性与异常处理

现在你知道了 try 的工作原理，让我们尝试另一个实验。假设我们想要捕获 try 的结果以供将来求值，这样我们就可以处理除法结果。也许我们会这样做：

```ghci
ghci> result <- try (return x)
Right *** Exception: divide by zero
```

这里发生了什么？让我们试着把它拼凑起来，用另一个尝试说明：

```ghci
ghci> let z = undefined
ghci> try (print z)
Left Prelude.undefined
ghci> result <- try (return z)
Right *** Exception: Prelude.undefined
```

和之前一样，给 z 赋值 undefined 不是问题。这个谜题的关键——以及除法谜题的关键——在于惰性求值。具体来说，在于 return，它不强制求值其参数；它只是将其包装起来。所以，try (return undefined) 的结果将是 Right undefined。现在，ghci 想在终端上显示这个结果。它打印出了"Right "，但我们不能打印出 undefined（或者除以零的结果）。所以当我们看到异常消息时，它来自 ghci，而不是你的程序。

这是一个关键点。让我们想想为什么我们之前的例子有效，而这个无效。早些时候，我们在 try 中放了 print x。打印某些东西的值，当然需要对其进行求值，所以异常在正确的地方被检测到。但是简单地使用 return 不会强制求值。为了解决这个问题，Control.Exception 模块定义了 evaluate 函数。它的行为就像 return，但会立即强制其参数被求值。让我们试试：

```ghci
ghci> let z = undefined
ghci> result <- try (evaluate z)
Left Prelude.undefined
ghci> result <- try (evaluate x)
Left divide by zero
```

这样，这就是预期的结果。这对 undefined 和我们除以零的例子都有效。

记住：每当你试图捕获纯代码抛出的异常时，在你的异常捕获函数中使用 evaluate 而不是 return。

## 使用 handle

通常，你可能希望在一段代码无异常完成时执行一个操作，在另一种情况下执行不同的操作。对于这种情况，有一个叫做 handle 的函数。它的类型是 (Exception -> IO a) -> IO a -> IO a。也就是说，它接受两个参数。第一个是一个函数，在执行第二个参数时发生异常的情况下调用。以下是我们可以使用它的一种方式：

```ghci
ghci> :m Control.Exception
ghci> let x = 5 `div` 0
ghci> let y = 5 `div` 1
ghci> handle (\_ -> putStrLn "Error calculating result") (print x)
Error calculating result
ghci> handle (\_ -> putStrLn "Error calculating result") (print y)
5
```

这样，如果计算中有错误，我们可以输出一条很好的消息。这比程序因除以零错误而崩溃要好得多。

## 选择性处理异常

之前示例的一个问题是，它为任何异常打印"Error calculating result"。可能存在除零异常以外的异常。例如，可能在显示输出时发生错误，或者纯代码可能抛出其他异常。

对于这种情况，有一个 handleJust 函数。它允许你指定一个测试，看看我们是否对给定的异常感兴趣。让我们看看：

```haskell
-- file: ch19/hj1.hs
import Control.Exception
catchIt :: Exception -> Maybe ()
catchIt (ArithException DivideByZero) = Just ()
catchIt _ = Nothing
handler :: () -> IO ()
handler _ = putStrLn "Caught error: divide by zero"
safePrint :: Integer -> IO ()
safePrint x = handleJust catchIt handler (print x)
```

catchIt 定义了一个函数来决定我们是否对给定的异常感兴趣。如果感兴趣，它返回 Just，否则返回 Nothing。另外，附加到 Just 的值将被传递给我们的处理器。我们现在可以很好地使用 safePrint：

```ghci
ghci> :l hj1.hs
[1 of 1] Compiling Main             ( hj1.hs, interpreted )
Ok, modules loaded: Main.
ghci> let x = 5 `div` 0
ghci> let y = 5 `div` 1
ghci> safePrint x
Caught error: divide by zero
ghci> safePrint y
5
```

Control.Exception 模块还提供了许多函数，我们可以用作 handleJust 中的测试来缩小我们关心的异常类型。例如，有一个类型为 Exception -> Maybe ArithException 的函数 arithExceptions，它会挑选出任何 ArithException，但忽略任何其他异常。我们可以这样使用它：

```haskell
-- file: ch19/hj2.hs
import Control.Exception
handler :: ArithException -> IO ()
handler e = putStrLn $ "Caught arithmetic error: " ++ show e
safePrint :: Integer -> IO ()
safePrint x = handleJust arithExceptions handler (print x)
```

这样，我们可以捕获所有类型的 ArithException，但仍然让其他异常通过未修改和未捕获。我们可以看到它像这样工作：

```ghci
ghci> :l hj2.hs
[1 of 1] Compiling Main             ( hj2.hs, interpreted )
Ok, modules loaded: Main.
ghci> let x = 5 `div` 0
ghci> let y = 5 `div` 1
ghci> safePrint x
Caught arithmetic error: divide by zero
ghci> safePrint y
5
```

特别值得注意的是 ioErrors 测试，它对应于一大类 I/O 相关的异常。

## I/O 异常

在任何程序中，异常的最大来源可能是 I/O。当与外部世界打交道时，各种事情都可能出错：磁盘可能已满，网络可能宕机，或者文件在你期望有数据时可能是空的。在 Haskell 中，I/O 异常就像任何其他异常一样，因为 Exception 数据类型可以表示它。另一方面，由于有这么多类型的 I/O 异常，存在一个特殊的模块 System.IO.Error 来处理它们。

System.IO.Error 定义了两个函数 catch 和 try，它们就像 Control.Exception 中的对应函数一样，用于处理异常。然而，与 Control.Exception 函数不同的是，这些函数只会捕获 I/O 错误，而会让所有其他异常通过未捕获。在 Haskell 中，所有 I/O 错误都有类型 IOError，它被定义为与 IOException 相同。

## 小心你使用的名称

因为 System.IO.Error 和 Control.Exception 都定义了同名的函数，如果你都在程序中导入它们，你会得到关于函数引用不明确的错误消息。你可以限定导入其中一个模块，或者从其中一个模块中隐藏这些符号。

注意 Prelude 导出的是 System.IO.Error 版本的 catch，而不是 Control.Exception 提供的版本。请记住前者只能捕获 I/O 错误，而后者可以捕获所有异常。换句话说，Control.Exception 中的 catch 几乎总是你想要的，但默认情况下你得到的不是那个。

让我们看看在 I/O 系统中使用异常的一种方法。回到第 169 页的"使用文件和句柄工作"，我们展示了一个使用命令式风格逐行读取文件内容的程序。虽然我们随后展示了更紧凑、"Haskell 化"的方法来解决这个问题，但让我们在这里重新访问那个例子。在 mainloop 函数中，我们必须在每次尝试从文件中读取一行之前显式测试是否到达输入文件的末尾。相反，我们可以检查读取一行的尝试是否导致 EOF 错误，像这样：

```haskell
-- file: ch19/toupper-impch20.hs
import System.IO
import System.IO.Error
import Data.Char(toUpper)
main :: IO ()
main = do
       inh <- openFile "input.txt" ReadMode
       outh <- openFile "output.txt" WriteMode
       mainloop inh outh
       hClose inh
       hClose outh
mainloop :: Handle -> Handle -> IO ()
mainloop inh outh =
    do input <- try (hGetLine inh)
       case input of
         Left e ->
             if isEOFError e
                then return ()
                else ioError e
         Right inpStr ->
             do hPutStrLn outh (map toUpper inpStr)
                mainloop inh outh
```

在这里，我们使用 System.IO.Error 版本的 try 来检查 hGetLine 是否抛出了 IOError。如果是这样，我们使用 System.IO.Error 中定义的 isEOFError 来查看抛出的异常是否表示我们到达了文件末尾。如果是，我们退出循环。如果异常是其他什么，我们调用 ioError 重新抛出它。

System.IO.Error 中定义了许多这样的测试和方法从 IOError 中提取信息。我们建议你在需要了解它们时查阅库参考中的那一页。

## 抛出异常

到目前为止，我们详细讨论了处理异常。还有另一个难题：抛出异常。§ 在本章中我们访问的示例中，Haskell 系统为你抛出异常。但是，你可以自己抛出任何异常。我们将向你展示如何做到这一点。

你可能会注意到这些大多数函数似乎返回类型 a 或 IO a 的值。这意味着该函数可以看起来返回任何类型的值。实际上，因为这些函数抛出异常，它们在正常意义上永远不会"返回"。这些返回值让你可以在期望各种不同类型的各种上下文中使用这些函数。

让我们从 Control.Exception 中的函数开始我们抛出异常的方式的 tour。最通用的函数是 throw，它的类型是 Exception -> a。这个函数可以抛出任何异常，并且可以在纯上下文中抛出。有一个配套函数——throwIO，类型为 Exception -> IO a——它在 IO monad 中抛出异常。这两个函数都需要一个要抛出的异常。你可以通过手工制作一个异常，也可以重用先前创建的异常。

还有一个函数 ioError，它在 Control.Exception 和 System.IO.Error 中定义相同，类型为 IOError -> IO a。当你想生成一个任意的 I/O 相关异常时使用它。

## 动态异常

动态异常利用了两个很少使用的 Haskell 模块：Data.Dynamic 和 Data.Typeable。我们不会在这里深入讲解这些模块的细节，但我们会给你提供制作和使用你自己的动态异常类型所需的工具。

§ 在其他一些语言中，抛出异常被称为 raising。

在第 21 章中，你将看到 HDBC 数据库库使用动态异常来表示来自 SQL 数据库的错误回到应用程序。数据库引擎的错误通常有三个组成部分：一个表示错误代码的整数、一个状态和一个人类可读的 error message。我们将在这里构建我们自己的 HDBC SqlError 类型实现。让我们从表示错误本身的数据结构开始：

```haskell
-- file: ch19/dynexc.hs
{-# LANGUAGE DeriveDataTypeable #-}
import Data.Dynamic
import Control.Exception
data SqlError = SqlError {seState :: String,
                          seNativeError :: Int,
                          seErrorMsg :: String}
                deriving (Eq, Show, Read, Typeable)
```

通过派生 Typeable 类型类，我们使这个类型可用于动态类型编程。为了让 GHC 自动生成 Typeable 实例，我们必须启用 DeriveDataTypeable 语言扩展。‖

现在，让我们定义一个 catchSql 和一个 handleSql，可用于捕获是 SqlError 的异常（请注意，常规的 catch 和 handle 函数无法捕获我们的 SqlError，因为它不是 Exception 的一种类型）：

```haskell
-- file: ch19/dynexc.hs
{- | 执行给定的 IO 操作。
如果它引发了一个 'SqlError'，那么执行提供的
处理程序并返回它的返回值。否则，正常继续。 -}
catchSql :: IO a -> (SqlError -> IO a) -> IO a
catchSql = catchDyn
{- | Like 'catchSql', with the order of arguments reversed. -}
handleSql :: (SqlError -> IO a) -> IO a -> IO a
handleSql = flip catchSql
```

这些函数只是围绕 catchDyn 的简单包装器，它的类型是 Typeable exception => IO a -> (exception -> IO a) -> IO a。我们在这里简单地限制类型，这样它只捕获 SQL 异常。

通常，当抛出异常但在任何地方都未被捕获时，程序将崩溃并将异常显示给标准错误。但是对于动态异常，系统将不知道如何显示它，所以我们只会看到一个无用的"unknown exception"消息。我们可以提供一个实用程序，这样应用程序作者可以简单地说 main = handleSqlError $ do ... 并且可以确信任何抛出的异常（在该线程中）都将被显示。以下是编写 handleSqlError 的方法：

‖ 可以手工派生 Typeable 实例，但这很麻烦。

```haskell
-- file: ch19/dynexc.hs
{- | 捕获 'SqlError'，并将它们作为 IO 错误与 fail 一起重新引发。
如果你不关心捕获 SQL 错误，但想要在发生错误时看到一条合理的
错误消息，这很有用。人们经常将其用作 SQL 调用周围的高级包装器。 -}
handleSqlError :: IO a -> IO a
handleSqlError action =
    catchSql action handler
    where handler e = fail ("SQL error: " ++ show e)
```

最后，这是如何将 SqlError 作为异常抛出的示例。这是一个会这样做的函数：

```haskell
-- file: ch19/dynexc.hs
throwSqlError :: String -> Int -> String -> a
throwSqlError state nativeerror errormsg =
    throwDyn (SqlError state nativeerror errormsg)
throwSqlErrorIO :: String -> Int -> String -> IO a
throwSqlErrorIO state nativeerror errormsg =
    evaluate (throwSqlError state nativeerror errormsg)
```

提醒一下，evaluate 就像 return，但会强制求值其参数。

这完成了我们的动态异常支持。这是很多代码，你可能不需要那么多，但我们想给你一个动态异常本身的例子以及经常伴随它的实用程序。事实上，这些例子几乎正是 HDBC 库中存在的。让我们在 ghci 中玩一下：

```ghci
ghci> :l dynexc.hs
[1 of 1] Compiling Main             ( dynexc.hs, interpreted )
Ok, modules loaded: Main.
ghci> throwSqlErrorIO "state" 5 "error message"
*** Exception: (unknown)
ghci> handleSqlError $ throwSqlErrorIO "state" 5 "error message"
*** Exception: user error (SQL error: SqlError {seState = "state", seNativeError = 5,
seErrorMsg = "error message"})
ghci> handleSqlError $ fail "other error"
*** Exception: user error (other error)
```

从这里，你可以看到 ghci 本身不知道如何显示 SQL 错误。但是，你也可以看到我们的 handleSqlError 函数对此提供了帮助，但也以未修改的方式传递了其他错误。让我们最后尝试一个自定义处理器：

```ghci
ghci> handleSql (fail . seErrorMsg) (throwSqlErrorIO "state" 5 "my error")
*** Exception: user error (my error)
```

在这里，我们定义了一个自定义错误处理器，它抛出一个由 SqlError 的 seErrorMsg 字段中的消息组成的新异常。你可以看到它按预期工作了。

## 单子 Monad 中的错误处理

因为我们必须在 IO monad 中捕获异常，如果我们尝试在单子中或在单子转换器堆栈中使用它们，我们会被迫进入 IO monad。这几乎从来不是我们真正想要的。

我们在第 438 页的"通过构建一个来理解单子转换器"中定义了 MaybeT 转换器，但它作为理解工具比作为编程工具更有用。幸运的是，一个专用的——更有用的——单子转换器已经存在：ErrorT，它在 Control.Monad.Error 模块中定义。

ErrorT 转换器让我们向单子添加异常，但它使用自己特殊的异常机制，与 Control.Exception 模块提供的机制分离。它给我们提供了一些有趣的能力：

- 如果我们坚持使用 ErrorT 接口，我们可以在这个单子中抛出和捕获异常。
- 遵循其他单子转换器的命名模式，执行函数命名为 runErrorT。未捕获的 ErrorT 异常在到达 runErrorT 时将停止向上传播。我们不会被踢到 IO monad。
- 我们控制异常的类型。

## 不要将 ErrorT 与常规异常混淆

如果我们在 ErrorT 内部使用 Control.Exception 中的 throw 函数（或者如果我们使用 error 或 undefined），我们仍然会被踢到 IO monad。

与其他的 mtl 单子一样，ErrorT 提供的接口由一个类型类定义：

```haskell
-- file: ch19/MonadError.hs
class (Monad m) => MonadError e m | m -> e where
    throwError :: e             -- 要抛出的错误
               -> m a
    catchError :: m a           -- 要执行的操作
               -> (e -> m a)    -- 错误处理器
               -> m a
```

类型变量 e 代表我们想要使用的错误类型。无论我们的错误类型是什么，我们必须让它成为 Error 类型类的实例：

```haskell
-- file: ch19/MonadError.hs
class Error a where
    -- 创建一个没有消息的异常
    noMsg  :: a
    -- 创建一个带有消息的异常
    strMsg :: String -> a
```

ErrorT 的 fail 实现使用 strMsg 函数。它抛出 strMsg 作为异常，并传递它收到的字符串参数。至于 noMsg，它用于为 MonadPlus 类型类提供 mzero 实现。

为了支持 strMsg 和 noMsg 函数，我们的 ParseError 类型将有一个 Chatty 构造函数。如果有人在我们的单子中调用 fail，这将用作构造函数。

我们需要知道的最后一个管道是执行函数 runErrorT 的类型：

```ghci
ghci> :t runErrorT
runErrorT :: ErrorT e m a -> m (Either e a)
```

## 一个小型解析框架

为了说明 ErrorT 的使用，让我们开发一个类似于 Parsec 的解析库的最基本部分：

```haskell
-- file: ch19/ParseInt.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
import Control.Monad.Error
import Control.Monad.State
import qualified Data.ByteString.Char8 as B
data ParseError = NumericOverflow
                | EndOfInput
                | Chatty String
                  deriving (Eq, Ord, Show)
instance Error ParseError where
    noMsg  = Chatty "oh noes!"
    strMsg = Chatty
```

对于解析器的状态，我们将创建一个非常小的单子转换器堆栈。State 单子携带要解析的 ByteString，而 ErrorT 堆叠在上面以提供错误处理：

```haskell
-- file: ch19/ParseInt.hs
newtype Parser a = P {
      runP :: ErrorT ParseError (State B.ByteString) a
    } deriving (Monad, MonadError ParseError)
```

和往常一样，我们将单子堆栈包装在 newtype 中。这在性能上没有损失，但增加了类型安全性。我们有意避免派生 MonadState B.ByteString 的实例。这意味着 Parser 单子的用户将无法使用 get 或 put 来查询或修改解析器的状态。结果，我们被迫进行一些手动的提升以到达堆栈中的 State 单子。然而，这是非常容易做到的：

```haskell
-- file: ch19/ParseInt.hs
liftP :: State B.ByteString a -> Parser a
liftP m = P (lift m)
satisfy :: (Char -> Bool) -> Parser Char
satisfy p = do
  s <- liftP get
  case B.uncons s of
    Nothing         -> throwError EndOfInput
    Just (c, s')
        | p c       -> liftP (put s') >> return c
        | otherwise -> throwError (Chatty "satisfy failed")
```

catchError 函数对于超越简单错误处理的任务很有用。例如，我们可以很容易地化解一个异常，将其转换为更友好的形式：

```haskell
-- file: ch19/ParseInt.hs
optional :: Parser a -> Parser (Maybe a)
optional p = (Just `liftM` p) `catchError` \_ -> return Nothing
```

我们的执行函数只是将各层插入在一起，并将结果重新排列成更整齐的形式：

```haskell
-- file: ch19/ParseInt.hs
runParser :: Parser a -> B.ByteString
          -> Either ParseError (a, B.ByteString)
runParser p bs = case runState (runErrorT (runP p)) bs of
                   (Left err, _) -> Left err
                   (Right r, bs) -> Right (r, bs)
```

如果我们将这个加载到 ghci 中，我们可以让它经过各种考验：

```ghci
ghci> :m +Data.Char
ghci> let p = satisfy isDigit
Loading package array-0.1.0.0 ... linking ... done.
Loading package bytestring-0.9.0.1.1 ... linking ... done.
Loading package mtl-1.1.0.1 ... linking ... done.
ghci> runParser p (B.pack "x")
Left (Chatty "satisfy failed")
ghci> runParser p (B.pack "9abc")
Right ('9',"abc")
ghci> runParser (optional p) (B.pack "x")
Right (Nothing,"x")
ghci> runParser (optional p) (B.pack "9a")
Right (Just '9',"a")
```
