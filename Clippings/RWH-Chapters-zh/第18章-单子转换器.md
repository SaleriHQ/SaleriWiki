# 第18章-单子转换器

## 动机：避免样板代码

Monads 提供了一种构建带效果的计算的强大方式。每个标准 monad 都专门做一件精确的事情。在实际代码中，我们经常需要能够同时使用多种效果。

例如，回想一下我们在第十章开发的 Parse 类型。当我们介绍 monads 时，我们提到这个类型是伪装下的 State monad。我们的 monad 比标准 State monad 更复杂，因为它使用 Either 类型允许解析失败的可能性。在我们的例子中，如果解析提前失败，我们想停止解析，而不是以某种破碎状态继续。我们的 monad 结合了携带状态的效果和提前退出的效果。

正常的 State monad 不能让我们以这种方式逃离；它只携带状态。它使用 fail 的默认实现：这调用 error，抛出我们无法在纯代码中捕获的异常。因此 State monad 似乎允许失败，但这种能力实际上毫无用处。（我们再次建议您几乎总是避免使用 fail！）

如果我们可以某种方式接受标准 State monad 并为其添加失败处理，而不需要手动构建自定义 monads，那就太理想了。mtl 库中的标准 monads 不允许我们组合它们。相反，该库提供了一组 monad 变换器*来实现相同的结果。

monad 变换器类似于常规 monad，但它不是一个独立的实体。相反，它修改底层 monad 的行为。mtl 库中的大多数 monads 都有变换器等价物。按照惯例，monad 的变换器版本具有相同的名称，末尾加上 T。例如，State 的变换器等价物是 StateT；它向底层 monad 添加可变状态。

* 名称 mtl 代表"monad transformer library"。

WriterT monad 变换器使得在另一个 monad 之上堆叠时写入数据成为可能。

## 一个简单的 Monad 变换器示例

在我们介绍 monad 变换器之前，让我们看一个使用我们已经熟悉的技术编写的函数。下面的函数递归进入目录树并返回它在树的每个层级找到的条目数列表：

```haskell
-- file: ch18/CountEntries.hs
module CountEntries (listDirectory, countEntriesTrad) where
import System.Directory (doesDirectoryExist, getDirectoryContents)
import System.FilePath ((</>))
import Control.Monad (forM, liftM)
listDirectory :: FilePath -> IO [String]
listDirectory = liftM (filter notDots) . getDirectoryContents
    where notDots p = p /= "." && p /= ".."
countEntriesTrad :: FilePath -> IO [(FilePath, Int)]
countEntriesTrad path = do
  contents <- listDirectory path
  rest <- forM contents $ \name -> do
            let newName = path </> name
            isDir <- doesDirectoryExist newName
            if isDir
              then countEntriesTrad newName
              else return []
  return $ (path, length contents) : concat rest
```

现在让我们看看使用 Writer monad 来实现相同目标。既然这个 monad 让我们可以随时记录一个值，我们不需要显式地构建结果。

由于我们的函数必须在 IO monad 中执行以便它可以遍历目录，我们不能直接使用 Writer monad。相反，我们使用 WriterT 将记录能力添加到 IO。如果我们查看涉及的类型，会更容易理解。

正常的 Writer monad 有两个类型参数，所以它更恰当地写为 Writer w a。第一个参数 w 是要记录的值类型，a 是 Monad 类型类通常需要的类型。因此 `Writer [(FilePath, Int)] a` 是一个记录目录名和大小列表的 writer monad。

WriterT 变换器有类似的结构，但它添加了另一个类型参数 m：这是我们正在扩充的底层 monad。WriterT 的完整签名是 `WriterT w m a`。

因为我们需要遍历目录，这需要访问 IO monad，我们将 writer 堆叠在 IO monad 之上。因此我们的 monad 变换器和底层 monad 的组合将具有类型 `WriterT [(FilePath, Int)] IO a`。这个 monad 变换器和 monad 的堆叠本身是一个 monad：

```haskell
-- file: ch18/CountEntriesT.hs
module CountEntriesT (listDirectory, countEntries) where
import CountEntries (listDirectory)
import System.Directory (doesDirectoryExist)
import System.FilePath ((</>))
import Control.Monad (forM_, when)
import Control.Monad.Trans (liftIO)
import Control.Monad.Writer (WriterT, tell)
countEntries :: FilePath -> WriterT [(FilePath, Int)] IO ()
countEntries path = do
  contents <- liftIO . listDirectory $ path
  tell [(path, length contents)]
  forM_ contents $ \name -> do
    let newName = path </> name
    isDir <- liftIO . doesDirectoryExist $ newName
    when isDir $ countEntries newName
```

这段代码与我们之前的版本没有太大区别。我们使用 liftIO 在必要时暴露 IO monad，并使用 tell 记录对目录的访问。

为了运行我们的代码，我们必须使用 WriterT 的执行函数之一：

```ghci
ghci> :type runWriterT
runWriterT :: WriterT w m a -> m (a, w)
ghci> :type execWriterT
execWriterT :: (Monad m) => WriterT w m a -> m w
```

这些函数执行 action，然后移除 WriterT 包装并给出一个包装在底层 monad 中的结果。runWriterT 函数给出 action 的结果以及运行时记录的内容，而 execWriterT 丢弃结果只给我们记录的内容：

```ghci
ghci> :type countEntries ".."
countEntries ".." :: WriterT [(FilePath, Int)] IO ()
ghci> :type execWriterT (countEntries "..")
execWriterT (countEntries "..") :: IO [(FilePath, Int)]
ghci> take 4 `liftM` execWriterT (countEntries "..")
[("..",30),("../ch05",28),("../ch05/dist",3),("../ch05/dist/build",9)]
```

我们在 IO 之上使用 WriterT，因为没有 IOT monad 变换器。每当我们将 IO monad 与一个或多个 monad 变换器一起使用时，IO 将始终在堆栈底部。

## Monads 和 Monad 变换器中的常见模式

mtl 库中的大多数 monads 和 monad 变换器在命名和类型类周围遵循一些常见模式。

为了说明这些规则，我们将专注于一个简单直接的 monad：reader monad。reader monad 的 API 详细说明在 MonadReader 类型类中。大多数 mtl monads 有类似命名的类型类。MonadWriter 定义 writer monad 的 API，等等：

```haskell
-- file: ch18/Reader.hs
class (Monad m) => MonadReader r m | m -> r where
    ask   :: m r
    local :: (r -> r) -> m a -> m a
```

类型变量 r 代表 reader monad 携带的不可变状态。Reader r monad 是 MonadReader 类的实例，ReaderT r m monad 变换器也是如此。同样，这种模式被其他 mtl monads 重复：通常既有具体的 monad 也有变换器，每个都是定义 monad API 的类型类的实例。

回到 reader monad 的细节，我们之前没有接触过 local 函数。它临时使用 `r -> r` 函数修改当前环境，然后在修改后的环境中执行其 action。为了使这个想法更具体，这里有一个简单的例子：

```haskell
-- file: ch18/LocalReader.hs
import Control.Monad.Reader
myName step = do
  name <- ask
  return (step ++ ", I am " ++ name)
localExample :: Reader String (String, String, String)
localExample = do
  a <- myName "First"
  b <- local (++"dy") (myName "Second")
  c <- myName "Third"
  return (a, b, c)
```

如果我们在 ghci 中执行 localExample action，我们可以看到修改环境的效果被限制在一个地方：

```ghci
ghci> runReader localExample "Fred"
Loading package mtl-1.1.0.1 ... linking ... done.
("First, I am Fred","Second, I am Freddy","Third, I am Fred")
```

当底层 monad m 是 MonadIO 的实例时，mtl 库为 ReaderT r m 以及一些其他类型类提供实例。这里有一些：

```haskell
-- file: ch18/Reader.hs
instance (Monad m) => Functor (ReaderT r m) where
    ...
instance (MonadIO m) => MonadIO (ReaderT r m) where
    ...
instance (MonadPlus m) => MonadPlus (ReaderT r m) where
    ...
```

同样，大多数 mtl monad 变换器定义这样的实例，以使我们更容易使用它们。

## 堆叠多个 Monad 变换器

正如我们已经提到的，当我们将 monad 变换器堆叠在普通 monad 上时，结果是另一个 monad。这意味着我们可以再次将 monad 变换器堆叠在我们组合的 monad 上，以获得一个新 monad，事实上这是常见的事情。在什么情况下我们可能想要创建这样的堆叠？

- 如果我们需要与外部世界交流，我们将在堆栈底部有 IO。否则，我们将有一些普通的 monad。
- 如果我们添加一个 ReaderT 层，我们给自己访问只读配置信息的能力。
- 添加一个 StateT 层，我们获得可以修改的全局状态。
- 如果我们需要记录事件的能力，我们可以添加一个 WriterT 层。

这种方法的强大之处在于我们可以根据我们的确切需求定制堆栈，指定我们想要支持哪些类型的效果。

作为堆叠 monad 变换器的一个小例子，这里是我们之前开发的 countEntries 函数的重写。我们将修改它，使递归进入目录树的深度不超过给定值，并记录它达到的最大深度：

```haskell
-- file: ch18/UglyStack.hs
import System.Directory
import System.FilePath
import Control.Monad.Reader
import Control.Monad.State
data AppConfig = AppConfig {
      cfgMaxDepth :: Int
    } deriving (Show)
data AppState = AppState {
      stDeepestReached :: Int
    } deriving (Show)
```

我们使用 ReaderT 存储配置数据，以我们要执行的递归最大深度的形式。我们还使用 StateT 记录在实际遍历期间达到的最大深度：

```haskell
-- file: ch18/UglyStack.hs
type App = ReaderT AppConfig (StateT AppState IO)
```

我们的变换器堆叠在底部有 IO，然后是 StateT，ReaderT 在顶部。在这种情况下，ReaderT 或 WriterT 在顶部都没关系，但 IO 必须在底部。

即使是一个小的 monad 变换器堆叠也会很快产生一个笨重的类型名称。我们可以使用类型别名来减少我们编写的类型签名的长度。

### 类型参数在哪里？

你可能已经注意到我们的类型同义词没有与我们关联的 monadic 类型的通常类型参数 a：

```haskell
-- file: ch18/UglyStack.hs
type App2 a = ReaderT AppConfig (StateT AppState IO) a
```

App 和 App2 在正常类型签名中都工作得很好。当我们尝试从其中一个构造另一个类型时，会出现差异。假设我们想要向堆叠添加另一个 monad 变换器：编译器将允许 `WriterT [String] App a`，但拒绝 `WriterT [String] App2 a`。

原因是 Haskell 不允许我们部分应用类型同义词。同义词 App 不接受类型参数，所以不会造成问题。然而，因为 App2 接受一个类型参数，如果我们想使用 App2 创建另一个类型，我们必须为该参数提供某种类型。

这个限制仅限于类型同义词。当我们创建一个 monad 变换器堆叠时，我们通常用 newtype 包装它（我们稍后会看到）。因此，我们实际上很少遇到这个问题。

我们 monad 堆叠的执行函数很简单：

```haskell
-- file: ch18/UglyStack.hs
runApp :: App a -> Int -> IO (a, AppState)
runApp k maxDepth =
    let config = AppConfig maxDepth
        state = AppState 0
    in runStateT (runReaderT k config) state
```

我们对 runReaderT 的应用移除 ReaderT 变换器包装，而 runStateT 移除 StateT 包装，只留下 IO monad 中的结果。

与早期版本相比，我们添加到遍历函数的唯一复杂之处很小。我们跟踪当前深度，并记录我们达到的最大深度：

```haskell
-- file: ch18/UglyStack.hs
constrainedCount :: Int -> FilePath -> App [(FilePath, Int)]
constrainedCount curDepth path = do
  contents <- liftIO . listDirectory $ path
  cfg <- ask
  rest <- forM contents $ \name -> do
            let newPath = path </> name
            isDir <- liftIO $ doesDirectoryExist newPath
            if isDir && curDepth < cfgMaxDepth cfg
              then do
                let newDepth = curDepth + 1
                st <- get
                when (stDeepestReached st < newDepth) $
                  put st { stDeepestReached = newDepth }
                constrainedCount newDepth newPath
              else return []
  return $ (path, length contents) : concat rest
```

无可否认，我们在这里使用 monad 变换器有点人为。因为我们正在编写一个简单直接的函数，我们并没有真正获得什么。然而，这种方法的优势在于它可以扩展到更大的程序。

我们可以在类似于我们的 App monad 的 monad 堆叠中编写大部分应用程序的命令式风格代码。在真实程序中，我们会携带更复杂的配置数据，但我们仍然使用 ReaderT 保持其只读且隐藏，除非需要时。我们会有更多可变状态要管理，但我们仍然使用 StateT 来封装它。

### 隐藏我们的工作

我们可以使用通常的 newtype 技术在自定义 monad 的实现与其接口之间建立坚实的屏障：

```haskell
-- file: ch18/UglyStack.hs
newtype MyApp a = MyA {
      runA :: ReaderT AppConfig (StateT AppState IO) a
    } deriving (Monad, MonadIO, MonadReader AppConfig,
                MonadState AppState)
runMyApp :: MyApp a -> Int -> IO (a, AppState)
runMyApp k maxDepth =
    let config = AppConfig maxDepth
        state = AppState 0
    in runStateT (runReaderT (runA k) config) state
```

如果我们将 MyApp 类型构造函数和 runMyApp 执行函数从一个模块导出，客户端代码将无法知道我们的 monad 内部是 monad 变换器的堆叠。

大型 deriving 子句需要 GeneralizedNewtypeDeriving 语言 pragma。编译器可以为我们派生所有这些实例，这看起来有些神奇。这是如何工作的？

早些时候，我们提到 mtl 库为每个 monad 变换器提供了一些类型类的实例。例如，IO monad 实现 MonadIO。如果底层 monad 是 MonadIO 的实例，mtl 也使 StateT 成为实例，对于 ReaderT 也是如此。

因此这里没有魔法：堆叠顶层的 monad 变换器是我们用 deriving 子句重新派生的所有类型类的实例。这是 mtl 提供精心协调的类型类和实例集的结果，这些类型类和实例很好地配合在一起。这里没有什么比通常我们可以对 newtype 声明执行的自动派生更多的了。

## 向下进入堆叠

到目前为止，我们使用 monad 变换器的方式很简单，mtl 库的管道允许我们避免关于如何构建 monad 堆叠的细节。事实上，我们已经对 monad 变换器了解足够多，可以简化许多常见的编程任务。

有一些有用的方式可以让我们脱离 mtl 的舒适区。大多数情况下，自定义 monad 位于堆栈底部，或者自定义 monad 变换器位于堆栈内的某个位置。要理解潜在的困难，让我们看一个例子。

假设我们有一个自定义 monad 变换器 CustomT：

```haskell
-- file: ch18/CustomT.hs
newtype CustomT m a = ...
```

在 mtl 提供的框架中，堆栈中的每个 monad 变换器通过提供大量类型类的实例使下层的 API 可用。我们可以遵循这个模式并编写一些样板实例：

```haskell
-- file: ch18/CustomT.hs
instance MonadReader r m => MonadReader r (CustomT m) where
    ...
instance MonadIO m => MonadIO (CustomT m) where
    ...
```

如果底层 monad 是 MonadReader 的实例，我们会为 CustomT 写一个 MonadReader 实例，其中 API 中的每个函数都传递给底层实例中对应的函数。这将允许更高层级的代码只关心堆叠整体是 MonadReader 的实例，而不知道或关心哪个层提供了真正的实现。

我们可以明确地做，而不是依赖所有这些类型类实例在幕后为我们工作。MonadTrans 类型类定义了一个名为 lift 的有用函数：

```ghci
ghci> :m +Control.Monad.Trans
ghci> :info MonadTrans
class MonadTrans t where lift :: (Monad m) => m a -> t m a
      -- Defined in Control.Monad.Trans
```

这个函数从堆叠的下层获取一个 monadic action，并将其提升到当前 monad 变换器中的 action。每个 monad 变换器都是 MonadTrans 的实例。

我们根据其与 fmap 和 liftM 的目的相似性来命名 lift。在每种情况下，我们都将某些东西从类型系统的较低层级提升到我们当前工作的层级。不同的选项如下所述：

fmap
将纯函数提升到 functors 的层级
liftM
将纯函数提升到 monads 的层级
lift
将 monadic action 从变换器堆叠中的一层提升到当前层

让我们重新访问我们之前定义的 App monad 堆叠（在我们用 newtype 包装它之前）：

```haskell
-- file: ch18/UglyStack.hs
type App = ReaderT AppConfig (StateT AppState IO)
```

如果我们想访问 StateT 携带的 AppState，我们通常依赖 mtl 的类型类和实例来处理为我们管道：

```haskell
-- file: ch18/UglyStack.hs
implicitGet :: App AppState
implicitGet = get
```

lift 函数让我们通过将 get 从 StateT 提升到 ReaderT 来实现相同的效果：

```haskell
-- file: ch18/UglyStack.hs
explicitGet :: App AppState
explicitGet = lift get
```

显然，当我们可以让 mtl 为我们做这项工作时，我们最终会得到更简洁的代码，但这并不总是可能的。

### 何时需要显式提升

当我们创建的 monad 变换器堆叠中相同类型类的实例出现在多个层级时，我们必须使用 lift 的情况：

```haskell
-- file: ch18/StackStack.hs
type Foo = StateT Int (State String)
```

如果我们尝试使用 MonadState 类型类的 put action，我们将获得的实例是 StateT Int 的，因为它在堆叠顶部：

```haskell
-- file: ch18/StackStack.hs
outerPut :: Int -> Foo ()
outerPut = put
```

在这种情况下，我们访问底层 State monad 的 put 的唯一方式是通过 lift 的使用：

```haskell
-- file: ch18/StackStack.hs
innerPut :: String -> Foo ()
innerPut = lift . put
```

有时，我们需要访问堆叠中多于一层下的 monad，在这种情况下我们必须组合 lift 调用。每次组合使用 lift 给我们访问更深一层：

```haskell
-- file: ch18/StackStack.hs
type Bar = ReaderT Bool Foo
barPut :: String -> Bar ()
barPut = lift . lift . put
```

当我们需要使用 lift 时，写为我们执行提升的包装函数并使用它们可能是好的风格。替代方案是在我们的代码中散布显式的 lift 使用往往看起来很乱。更糟糕的是，它将我们 monad 堆叠布局的细节硬编码到我们的代码中，这将使后续修改复杂化。

## 通过构建一个来理解 Monad 变换器

为了让我们深入了解 monad 变换器一般如何工作，我们将创建一个并描述其机制。我们的目标简单而有用的：MaybeT。但令人惊讶的是，它在 mtl 库中缺失。

这个 monad 变换器通过用 Maybe 包装其类型参数来修改底层 monad m a 的行为，以获得 m (Maybe a)。与 Maybe monad 一样，如果在 MaybeT monad 变换器中调用 fail，执行会提前终止。

为了将 m (Maybe a) 变成 Monad 实例，我们必须通过 newtype 声明使其成为一个独特的类型：

```haskell
-- file: ch18/MaybeT.hs
newtype MaybeT m a = MaybeT {
      runMaybeT :: m (Maybe a)
    }
```

现在我们需要定义三个标准 monad 函数。最复杂的是 `(>>=)`，它的内部最能说明我们实际在做什么。在深入其操作之前，让我们先看看它的类型：

```haskell
-- file: ch18/MaybeT.hs
bindMT :: (Monad m) => MaybeT m a -> (a -> MaybeT m b) -> MaybeT m b
```

要理解这个类型签名，回想一下我们在"多参数类型类"中的讨论。我们打算使其成为 Monad 实例的东西是部分类型 MaybeT m；它有通常的单一类型参数 a，满足 Monad 类型类的要求。

理解我们的 `(>>=)` 实现主体的技巧是 do 块内的所有内容都在底层 monad m 中执行，无论那是什么：

```haskell
-- file: ch18/MaybeT.hs
x `bindMT` f = MaybeT $ do
                 unwrapped <- runMaybeT x
                 case unwrapped of
                   Nothing -> return Nothing
                   Just y -> runMaybeT (f y)
```

我们的 runMaybeT 函数解包 x 中包含的结果。接下来，回想一下 `<-` 符号被脱糖为 `(>>=)`：monad 变换器的 `(>>=)` 必须使用底层 monad 的 `(>>=)`。最后一点 case 分析决定了我们是短路还是链接我们的计算。最后，看一下 body 的顶部。这里，我们必须用 MaybeT 构造函数包装结果，以再次隐藏底层 monad。

以这种方式显示的 do 记法可能阅读起来很愉快，但它隐藏了我们依赖底层 monad 的 `(>>=)` 实现的事实。这里有一个更符合习惯的 MaybeT 版本 `(>>=)` 使这一点更清楚：

```haskell
-- file: ch18/MaybeT.hs
x `altBindMT` f =
    MaybeT $ runMaybeT x >>= maybe (return Nothing) (runMaybeT . f)
```

现在我们理解了 `(>>=)` 在做什么，我们的 return 和 fail 实现不需要解释，我们的 Monad 实例也不需要：

```haskell
-- file: ch18/MaybeT.hs
returnMT :: (Monad m) => a -> MaybeT m a
returnMT a = MaybeT $ return (Just a)
failMT :: (Monad m) => t -> MaybeT m a
failMT _ = MaybeT $ return Nothing

instance (Monad m) => Monad (MaybeT m) where
  return = returnMT
  (>>=) = bindMT
  fail = failMT
```

### 创建一个 Monad 变换器

要将我们的类型变成 monad 变换器，我们必须提供 MonadTrans 类的一个实例，以便用户可以访问底层 monad：

```haskell
-- file: ch18/MaybeT.hs
instance MonadTrans MaybeT where
    lift m = MaybeT (Just `liftM` m)
```

底层 monad 以类型参数 a 开始：我们"注入" Just 构造函数，以便它将获得我们需要的类型 Maybe a。然后我们用 MaybeT 构造函数隐藏 monad。

### 更多类型类实例

一旦我们为 MonadTrans 定义了一个实例，我们可以用它来为许多其他 mtl 类型类定义实例：

```haskell
-- file: ch18/MaybeT.hs
instance (MonadIO m) => MonadIO (MaybeT m) where
  liftIO m = lift (liftIO m)
instance (MonadState s m) => MonadState s (MaybeT m) where
  get = lift get
  put k = lift (put k)
-- ... and so on for MonadReader, MonadWriter, etc ...
```

因为几个 mtl 类型类使用函数依赖，我们的一些实例声明需要大大放宽 GHC 通常的严格类型检查规则。（如果我们忘记这些指令中的任何一个，编译器会在其错误消息中帮助地告诉我们需要哪些。）

```haskell
-- file: ch18/MaybeT.hs
{-# LANGUAGE FlexibleInstances, MultiParamTypeClasses,
             UndecidableInstances #-}
```

是显式使用 lift 更好还是花时间写这些样板实例更好？这取决于我们对 monad 变换器的预期用途。如果我们只在一些受限情况下使用它，我们可以只提供一个 MonadTrans 实例。在这种情况下，更多的实例可能仍然有意义，例如 MonadIO。另一方面，如果我们的变换器将在整个代码体中的不同情况下出现，花一个无聊的小时写这些实例可能是一笔不错的投资。

### 用 Monad 堆叠替换 Parse 类型

现在我们有了一个可以提前退出的 monad 变换器，我们可以使用它来在解析中途失败时保释。例如，我们可以因此替换我们在"隐式状态"中开发的 Parse 类型，用一个根据我们需求定制的 monad：

```haskell
-- file: ch18/MaybeTParse.hs
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
module MaybeTParse
    (
      Parse
    , evalParse
    ) where
import MaybeT
import Control.Monad.State
import Data.Int (Int64)
import qualified Data.ByteString.Lazy as L
data ParseState = ParseState {
      string :: L.ByteString
    , offset :: Int64
    } deriving (Show)
newtype Parse a = P {
      runP :: MaybeT (State ParseState) a
    } deriving (Monad, MonadState ParseState)
evalParse :: Parse a -> L.ByteString -> Maybe a
evalParse m s = evalState (runMaybeT (runP m)) (ParseState s 0)
```

练习

1. 我们的 Parse monad 并不是其早期对应物的完美替代。因为我们使用 Maybe 而不是 Either 来表示结果，如果解析失败，我们无法报告任何有用的信息。
   创建一个 EitherT sometype monad 变换器，并使用它实现一个更有能力的 Parse monad，如果解析失败可以报告错误消息。
   如果你喜欢为了乐趣探索 Haskell 库，你可能遇到过 Control.Monad.Error 模块中 Either 类型的现有 Monad 实例。我们建议你不要将其作为指南。它的设计过于限制：它将 Either String 变成一个 monad，而你本可以使用类型参数代替 String。
   提示：如果你遵循这个建议，你可能需要在定义中使用 FlexibleInstances 语言扩展。

## 变换器堆叠顺序很重要

从我们早期使用 ReaderT 和 StateT 等 monad 变换器的例子来看，很容易得出结论：我们堆叠 monad 变换器的顺序无关紧要。

当我们将 StateT 堆叠在 State 之上时，顺序确实可以产生影响应该更清楚。类型 `StateT Int (State String)` 和 `StateT String (State Int)` 可能携带相同的信息，但我们不能互换使用它们。顺序决定了我们何时需要使用 lift 来获取其中一个或另一个状态。

这里有一个更显著地证明顺序重要性的例子。假设我们有一个可能失败的计算，我们想记录它失败的情况：

```haskell
-- file: ch18/MTComposition.hs
{-# LANGUAGE FlexibleContexts #-}
import Control.Monad.Writer
import MaybeT
problem :: MonadWriter [String] m => m ()
problem = do
  tell ["this is where i fail"]
  fail "oops"
```

这些 monad 堆叠中的哪一个会给我们我们需要的信息？

```haskell
-- file: ch18/MTComposition.hs
type A = WriterT [String] Maybe
type B = MaybeT (Writer [String])
a :: A ()
a = problem
b :: B ()
b = problem
```

让我们在 ghci 中尝试这些替代方案：

```ghci
ghci> runWriterT a
Loading package mtl-1.1.0.1 ... linking ... done.
Nothing
ghci> runWriter $ runMaybeT b
(Nothing,["this is where i fail"])
```

这种结果差异不应该令人惊讶——只要看看执行函数的签名：

```ghci
ghci> :t runWriterT
runWriterT :: WriterT w m a -> m (a, w)
ghci> :t runWriter . runMaybeT
runWriter . runMaybeT :: MaybeT (Writer w) a -> (Maybe a, w)
```

我们的 WriterT-on-Maybe 堆叠将 Maybe 作为底层 monad，所以 runWriterT 必须给我们一个 Maybe 类型的结果。在我们的测试用例中，我们只能看到实际出错时发生的情况的日志！

堆叠 monad 变换器类似于组合函数。如果我们改变应用函数的顺序然后得到不同的结果，我们不会感到惊讶。对于 monad 变换器也是如此。

## 将 Monads 和 Monad 变换器放入视角

退后几步看看用 monads 和 monad 变换器编程的弱点和优势是很有用的。

### 对纯代码的干扰

使用 monads 工作可能最大的实际烦恼是，当我们想使用纯代码时，monad 的类型构造函数经常碍事。许多有用的纯函数需要 monadic 对应物，仅仅是为了附加一些 monadic 类型构造函数的占位符参数：

```ghci
ghci> :t filter
filter :: (a -> Bool) -> [a] -> [a]
ghci> :i filterM
filterM :: (Monad m) => (a -> m Bool) -> [a] -> m [a]
      -- Defined in Control.Monad
```

然而，覆盖并不完整：标准库并不总是提供纯函数的 monadic 版本。

原因在于历史。Eugenio Moggi 在 1988 年引入了使用 monads 进行编程的想法，大约在 Haskell 1.0 标准正在开发的时候。今天 Prelude 中的许多函数可以追溯到 Haskell 1.0，它于 1990 年发布。1991 年，Philip Wadler 开始为更广泛的功能编程受众撰写关于 monads 潜力的文章，此后它们开始被投入使用。

直到 1996 年 Haskell 1.3 发布，标准才获得对 monads 的支持。到那时，语言设计者已经被向后兼容性约束：他们不能更改 Prelude 中函数的签名，因为它会破坏现有代码。

从那以后，Haskell 社区学到了很多关于创建合适抽象的知识，这样我们就可以编写受纯/ monadic 分歧影响较小的代码。你可以在 Data.Traversable 和 Data.Foldable 模块中找到这些想法的现代提炼。虽然这些模块很有吸引力，但我们在本书中没有涵盖它们。部分原因是空间不足，但也因为如果你还在跟着我们走到这一步，你自己弄清楚它们不会有困难。

在理想的世界中，我们会与过去决裂并切换到使用 Traversable 和 Foldable 类型的 Prelude 吗？大概不会。学习 Haskell 对于新手来说已经是一个足够刺激的冒险了。Foldable 和 Traversable 抽象在我们已经理解 functors 和 monads 时很容易掌握，但它们会让早期学习者摄入过多抽象。对于教学语言来说，map 操作列表而不是 functors 是好的。

### 过度的排序

我们使用 monads 的主要原因之一是它们让我们指定效果的排序。再次看看我们之前编写的一小段代码：

```haskell
-- file: ch18/MTComposition.hs
{-# LANGUAGE FlexibleContexts #-}
import Control.Monad.Writer
import MaybeT
problem :: MonadWriter [String] m => m ()
problem = do
  tell ["this is where i fail"]
  fail "oops"
```

因为我们在 monad 中执行，我们保证 tell 的效果发生在 fail 的效果之前。问题是我们即使不一定想要这种排序保证也会得到它：编译器不能重新排列 monadic 代码，即使这样做会更高效。

### 运行时开销

最后，当我们使用 monads 和 monad 变换器时，我们可能会支付效率税。例如，State monad 在闭包中携带其状态。闭包在 Haskell 实现中可能很便宜，但并不是免费的。

monad 变换器为其底层的一切增加自己的开销。我们的 MaybeT 变换器每次使用 `(>>=)` 时都必须包装和解包 Maybe 值。因此，在 StateT 之上的 MaybeT 再加上 ReaderT 的堆叠，每个 `(>>=)` 都有很多簿记要做。

足够聪明的编译器可能会使这些成本部分或全部消失，但这种程度的复杂性目前还无法广泛获得。

有一些相对简单的技术可以避免其中一些成本，尽管我们缺乏空间来更多提及它们的名字。例如，使用 continuation monad，我们可以避免 `(>>=)` 中的恒定包装和解包，只在使用效果时支付代价。这种方法的复杂性已经包装在库中。这个工作领域在撰写本文时仍处于活跃开发中。如果你想让你的 monad 变换器使用更高效，我们建议在 Hackage 上寻找或在邮件列表或 IRC 上询问方向。

### 笨重的接口

如果我们使用 mtl 库作为一个黑盒子，它的所有组件都配合得很好。然而，一旦我们开始开发我们自己的 monads 和 monad 变换器，并与 mtl 提供的那些一起使用，一些缺陷开始显现。

例如，如果我们创建一个新的 monad 变换器 FooT 并想像 mtl 一样遵循相同的模式，我们将让它实现一个类型类 MonadFoo。如果我们真的想将其干净地集成到 mtl 中，我们将不得不在大约十几个 mtl 类型类中的每一个都为它声明实例。

最重要的是，我们还将不得不对每个 mtl 变换器声明 MonadFoo 的实例。这些实例中的大多数几乎相同，写起来很乏味。如果你想继续将新的 monad 变换器集成到 mtl 框架中，我们必须处理的移动部件数量以新变换器数量的平方增加！

公平地说，这个问题只影响极少数人。大多数 mtl 用户根本不需要开发新的变换器，所以他们不受影响。

mtl 设计的这个弱点源于它是开发的第一个 monad 变换器库。考虑到其设计者正在投入未知领域，他们做出了出色的工作，生产了一个对大多数用户易于理解和使用的强大库。

一个更新的 monads 和变换器库 monadLib 纠正了 mtl 中的许多设计缺陷。如果你某天变成一个 monad 变换器的核心黑客，值得一看。

二次方实例定义实际上是使用 monad 变换器方法的问题。已经提出了许多其他组合 monads 的方法，它们没有这个问题，但它们对最终用户来说似乎都不如 monad 变换器方便。幸运的是，根本没有那么多基础的、通用的 monad 变换器。

## 把一切整合起来

在处理效果和类型方面，monads 绝不是道路的尽头。它们是我们迄今为止达到的最实用的停留点。语言研究人员总是在研究提供类似优势而不做出同样妥协的系统。

虽然我们在使用时必须做出妥协，但 monads 和 monad 变换器仍然提供了一种在命令式语言中无与伦比的灵活性和控制水平。只需几个声明，我们就可以重新连接一些基本如分号的东西，赋予它新的含义。
