# 第04章-函数式编程

## 用 Haskell 思考

我们对 Haskell 的早期学习有两个不同的障碍。第一个是习惯从命令式编程到函数式编程的思维转变：我们必须替换来自其他语言的编程习惯。我们这样做不是因为命令式技术不好，而是因为在函数式语言中其他技术效果更好。

我们的第二个挑战是熟悉标准 Haskell 库。与任何语言一样，库作为一个杠杆，使我们能够倍增我们解决问题的能力。Haskell 库往往比其他许多语言中的库在更高层次的抽象上运作。我们需要更加努力地学习使用库，但作为回报，它们提供了很多能力。

在本章中，我们将介绍一些常见的函数式编程技术。我们将借助命令式语言的例子来突出我们需要的思维转变。在此过程中，我们也将走过 Haskell 标准库的一些基础知识。我们也将间歇地介绍一些更多的语言特性。

## 一个简单的命令行框架

在本章的大部分内容中，我们将关注与外部世界没有交互的代码。为了保持我们对实际代码的关注，我们将从开发我们"纯"代码和外部世界之间的网关开始。我们的框架只是读取一个文件的内容，对该文件应用一个函数，然后将结果写入另一个文件：

```haskell
-- file: ch04/InteractWith.hs
-- Save this in a source file, e.g., Interact.hs
import System.Environment (getArgs)

interactWith function inputFile outputFile = do
  input <- readFile inputFile
  writeFile outputFile (function input)

main = mainWith myFunction
  where mainWith function = do
          args <- getArgs
          case args of
            [input,output] -> interactWith function input output
            _ -> putStrLn "error: exactly two arguments needed"
        -- replace "id" with the name of our function below
        myFunction = id
```

这就是编写简单但完整的文件处理程序所需的全部内容。这是一个完整的程序，我们可以将其编译为名为 InteractWith 的可执行文件，如下所示：

```bash
$ ghc --make InteractWith
[1 of 1] Compiling Main             ( InteractWith.hs, InteractWith.o )
Linking InteractWith ...
```

如果我们从 shell 或命令提示符运行此程序，它将接受两个文件名：要读取的文件的名称，以及要写入的文件的名称：

```bash
$ ./Interact
error: exactly two arguments needed
$ ./Interact hello-in.txt hello-out.txt
$ cat hello-in.txt
hello world
$ cat hello-out.txt
hello world
```

我们源文件中的一些表示法是新的。do 关键字引入一个动作块，这些动作可以导致对现实世界的影响，例如读取或写入文件。`<-` 运算符是 do 块内赋值的等价物。这足以让我们开始。我们将更深入地讨论这些表示法细节，以及 I/O 一般，在第 7 章。

当我们想要测试一个不能与外部世界交谈的函数时，我们只需将前面代码中的名称 id 替换为我们想要测试的函数的名称。

无论我们的函数做什么，它都需要具有类型 `String -> String`；换句话说，它必须接受一个字符串并返回一个字符串。

## 热身：可移植地拆分文本行

Haskell 提供了一个内置函数 lines，允许我们在行边界上拆分文本字符串。它返回一个字符串列表，行终止字符被省略：

```ghci
ghci> :type lines
lines :: String -> [String]
ghci> lines "line 1\nline 2"
["line 1","line 2"]
ghci> lines "foo\n\nbar\n"
["foo","","bar"]
```

虽然 lines 看起来很有用，但它依赖于我们以"文本模式"读取文件才能工作。文本模式是许多编程语言的常见特性；当我们读取和写入 Windows 上的文件时，它提供了一种特殊行为。当我们在文本模式下读取文件时，文件 I/O 库将行尾序列 "\r\n"（回车后跟换行）转换为 "\n"（单独换行），在我们写入文件时进行反向转换。在类 Unix 系统上，文本模式不执行任何转换。由于这种差异，如果我们在另一个平台上读取在一个平台上写入的文件，行尾可能会变得一团糟。（readFile 和 writeFile 都在文本模式下操作。）

```ghci
ghci> lines "a\r\nb"
["a\r","b"]
```

lines 函数仅在换行符上拆分，将回车符留在行尾。如果我们在一台 Linux 或 Unix 机器上读取 Windows 生成的文本文件，我们会在每行末尾收到尾随回车符。

多年来，我们已经舒适地使用了 Python 的"通用换行符"支持；它透明地为我们处理 Unix 和 Windows 换行符约定。我们希望在 Haskell 中提供类似的东西。

由于我们仍在阅读 Haskell 代码的早期职业生涯中，我们将相当详细地讨论我们的 Haskell 实现：

```haskell
-- file: ch04/SplitLines.hs
splitLines :: String -> [String]
```

我们函数的类型签名表示它接受一个字符串，具有某种未知换行符约定的文件内容。它返回一个字符串列表，表示文件中的每一行：

```haskell
-- file: ch04/SplitLines.hs
splitLines [] = []
splitLines cs =
    let (pre, suf) = break isLineTerminator cs
    in  pre : case suf of
                ('\r':'\n':rest) -> splitLines rest
                ('\r':rest)      -> splitLines rest
                ('\n':rest)      -> splitLines rest
                _                -> []

isLineTerminator c = c == '\r' || c == '\n'
```

在我们深入细节之前，首先注意我们如何组织代码。我们首先展示了重要的代码片段，将 isLineTerminator 的定义留到后面。因为我们给辅助函数起了一个可读的名称，我们甚至在阅读它之前就可以猜测它的作用，这有助于平滑阅读代码的"流程"。

Prelude 定义了一个名为 break 的函数，我们可以用它将列表分成两部分。它接受一个函数作为其第一个参数。该函数必须检查列表的一个元素并返回一个 Bool，以指示是否在该点拆分列表。break 函数返回一个对，由谓词返回 True 之前消耗的子列表（前缀）和列表的其余部分（后缀）组成：

```ghci
ghci> break odd [2,4,5,6,8]
([2,4],[5,6,8])
ghci> :module +Data.Char
ghci> break isUpper "isUpper"
("is","Upper")
```

因为我们只需要一次匹配一个回车符或换行符，一个接一个地检查列表的每个元素对我们的需求来说已经足够好了。

splitLines 的第一个方程表示，如果我们匹配一个空字符串，我们就没有进一步的工作要做。

在第二个方程中，我们首先将 break 应用于我们的输入字符串。前缀是行终止符之前的子字符串，而后缀是字符串的剩余部分。后缀将包括行终止符（如果存在）。

`pre :` 表达式告诉我们，我们应该将 pre 值添加到行列表的前面。然后我们使用 case 表达式检查后缀，以便决定下一步做什么。case 表达式的结果将用作 (:) 列表构造函数的第二个参数。

第一个模式匹配以回车符开头的字符串，后跟换行符。变量 rest 绑定到字符串的其余部分。其他模式类似，所以它们应该很容易理解。

用散文描述 Haskell 函数并不一定容易理解。我们可以通过进入 ghci 并观察函数在不同情况下的行为来获得更好的理解。

让我们从划分不包含任何行终止符的字符串开始：

```ghci
ghci> splitLines "foo"
["foo"]
```

这里，我们对 break 的应用从未找到行终止符，所以它返回的后缀是空的：

```ghci
ghci> break isLineTerminator "foo"
("foo","")
```

因此，splitLines 中的 case 表达式必须匹配第四个分支，我们就完成了。更有趣的情况呢？

```ghci
ghci> splitLines "foo\r\nbar"
["foo","bar"]
```

我们对 break 的第一次应用给了我们一个非空后缀：

```ghci
ghci> break isLineTerminator "foo\r\nbar"
("foo","\r\nbar")
```

因为后缀以回车符后跟换行符开头，我们匹配 case 表达式的第一个分支。这给了我们绑定到 "foo" 的 pre 和绑定到 "bar" 的 suf。我们递归应用 splitLines，这次仅对 "bar"：

```ghci
ghci> splitLines "bar"
["bar"]
```

结果是，我们构造了一个列表，其头部是 "foo"，尾部是 ["bar"]：

```ghci
ghci> "foo" : ["bar"]
["foo","bar"]
```

这种用 ghci 实验是理解和调试代码片段行为的有效方式。它几乎具有偶然的更重要的好处。从 ghci 测试复杂代码可能很棘手，所以我们将倾向于编写更小的函数，这可以进一步帮助代码的可读性。

这种创建和重用小型但强大的代码片段的风格是函数式编程的基本部分。

## 行尾转换程序

让我们将 splitLines 函数连接到我们之前编写的小框架中。复制 Interact.hs 源文件；让我们称之为新文件 FixLines.hs。将 splitLines 函数添加到新源文件。因为我们的函数必须产生一个 String，我们必须将行列表重新缝合在一起。Prelude 提供了 unlines 函数，它连接字符串列表，将换行符添加到每个末尾：

```haskell
-- file: ch04/SplitLines.hs
fixLines :: String -> String
fixLines input = unlines (splitLines input)
```

如果我们用 fixLines 替换 id 函数，我们可以编译一个可执行文件，该文件将转换文本文件为我们系统的本机换行符：

```bash
$ ghc --make FixLines
[1 of 1] Compiling Main             ( FixLines.hs, FixLines.o )
Linking FixLines ...
```

如果您在 Windows 系统上，找到并下载在 Unix 系统上创建的文本文件（例如，gpl-3.0.txt [http://www.gnu.org/licenses/gpl-3.0.txt]）。在标准 Notepad 文本编辑器中打开它。这些行应该全部连接在一起，使文件几乎不可读。使用您刚刚创建的 FixLines 命令处理文件，然后在 Notepad 中打开输出文件。行尾现在应该已修复。

在类 Unix 系统上，标准分页符和编辑器隐藏 Windows 换行符，使验证 FixLines 实际消除它们变得更加困难。这里有一些应该提供帮助的命令：

```bash
$ file gpl-3.0.txt
gpl-3.0.txt: ASCII English text
$ unix2dos gpl-3.0.txt
unix2dos: converting file gpl-3.0.txt to DOS format ...
$ file gpl-3.0.txt
gpl-3.0.txt: ASCII English text, with CRLF line terminators
```

## 中缀函数

通常，当我们在 Haskell 中定义或应用函数时，我们写函数名，后跟它的参数。这种表示法称为前缀，因为函数名在参数之前。

如果函数或构造函数接受两个或更多参数，我们可以选择以中缀形式使用它，我们将其放在第一个和第二个参数之间。这允许我们将函数用作中缀运算符。

要使用中缀表示法定义或应用函数或值构造函数，我们将其名称括在反引号字符中（有时称为反引号）。这里是函数和类型的简单中缀定义：

```haskell
-- file: ch04/Plus.hs
a `plus` b = a + b

data a `Pair` b = a `Pair` b
                  deriving (Show)

-- we can use the constructor either prefix or infix
foo = Pair 1 2
bar = True `Pair` "quux"
```

因为中缀表示法纯粹是语法便利，它不会改变函数的行为：

```ghci
ghci> 1 `plus` 2
3
ghci> plus 1 2
3
ghci> True `Pair` "something"
True `Pair` "something"
ghci> Pair True "something"
True `Pair` "something"
```

中缀表示法通常可以提高可读性。例如，Prelude 定义了一个函数 elem，它表示值是否存在于列表中。如果我们使用前缀表示法使用 elem，它相当容易阅读：

```ghci
ghci> elem 'a' "camogie"
True
```

如果我们切换到中缀表示法，代码变得更加容易理解。现在很清楚，我们正在检查左边的值是否存在于右边的列表中：

```ghci
ghci> 3 `elem` [1,2,4,8]
False
```

在 Data.List 模块中的一些有用函数中，我们看到了更明显的改进。isPrefixOf 函数告诉我们一个列表是否匹配另一个列表的开头：

```ghci
ghci> :module +Data.List
ghci> "foo" `isPrefixOf` "foobar"
True
```

isInfixOf 和 isSuffixOf 函数分别匹配列表中任何位置和结尾：

```ghci
ghci> "needle" `isInfixOf` "haystack full of needle thingies"
True
ghci> "end" `isSuffixOf` "the end"
True
```

没有硬性规定指示何时应该使用中缀与前缀表示法，尽管前缀表示法更为常见。最好选择在任何特定情况下使您的代码更易读的那个。

### 警惕熟悉表示法在不熟悉语言中

其他一些编程语言使用反引号，但尽管视觉相似，Haskell 中反引号的用途与 Perl、Python 或 Unix shell 脚本中的含义完全不同。

在 Haskell 中，我们可以用反引号做的唯一合法的事情是用它们包装函数名。例如，我们不能用它们包装其值为函数的复杂表达式。这可能很方便，但不是语言现在的样子。

## 处理列表

作为函数式编程的面包和黄油，列表值得一些严肃的关注。标准 Prelude 定义了数十个处理列表的函数。其中许多将是不可或缺的工具，因此尽早学习它们很重要。

无论好坏，本节将有点像函数清单。为什么一次展示这么多函数？因为它们既易于学习又绝对普遍存在。如果我们没有这个工具在指尖，我们将浪费时间为已经存在于标准库中的简单函数重新发明而浪费时间。

所以请忍受我们浏览列表的过程；您节省的努力将是巨大的。

Data.List 模块是所有标准列表函数的"真正的逻辑家园"。Prelude 只是重新导出 Data.List 导出的函数的很大一部分。Data.List 中有几个有用的函数没有被标准 Prelude 重新导出。当我们在下面的章节中浏览列表函数时，我们将明确提及仅在 Data.List 中的那些：

```ghci
ghci> :module +Data.List
```

因为这些函数都不复杂或需要超过大约三行 Haskell 代码编写，我们将在每个函数的描述中简要说明。事实上，快速和有用的学习练习是在您阅读了每个函数后编写其定义。

### 基本列表操作

length 函数告诉我们列表中有多少元素：

```ghci
ghci> :type length
length :: [a] -> Int
ghci> length []
0
ghci> length [1,2,3]
3
ghci> length "strings are lists, too"
22
```

如果您需要确定列表是否为空，请使用 null 函数：

```ghci
ghci> :type null
null :: [a] -> Bool
ghci> null []
True
ghci> null "plugh"
False
```

要访问列表的第一个元素，请使用 head 函数：

```ghci
ghci> :type head
head :: [a] -> a
ghci> head [1,2,3]
1
```

相反，tail 返回列表除头部外的所有部分：

```ghci
ghci> :type tail
tail :: [a] -> [a]
ghci> tail "foo"
"oo"
```

另一个函数 last 返回列表的最后一个元素：

```ghci
ghci> :type last
last :: [a] -> a
ghci> last "bar"
'r'
```

last 的相反是 init，它返回其输入列表除最后一个元素外的所有元素组成的列表：

```ghci
ghci> :type init
init :: [a] -> [a]
ghci> init "bar"
"ba"
```

前面几个函数在空列表上表现不佳，所以如果您不知道列表是否为空，请小心。它们的不当行为采取什么形式？

```ghci
ghci> head []
*** Exception: Prelude.head: empty list
```

在 ghci 中尝试每个前面的函数。哪些在给定空列表时会崩溃？

### 安全和理智地处理崩溃函数

当我们想使用 head 这样的函数时，我们知道如果我们传入空列表可能会崩溃，最初可能有一种强烈的诱惑，在调用 head 之前检查列表的长度。让我们构建一个虚拟示例来说明我们的观点：

```haskell
-- file: ch04/EfficientList.hs
myDumbExample xs = if length xs > 0
                   then head xs
                   else 'Z'
```

如果我们的背景是 Perl 或 Python 等语言，这似乎是一种非常自然的编写测试的方式。在幕后，Python 列表是数组，Perl 数组是，嗯，数组。所以我们必然知道它们有多长，调用 len(foo) 或 scalar(@foo) 是完全自然的事情。但与其他许多事情一样，将这样的假设盲目移植到 Haskell 并不是一个好主意。

我们已经多次看到列表代数数据类型的定义，我们知道列表不明确存储自己的长度。因此，length 唯一能操作的方式是遍历整个列表。

因此，当我们只关心列表是否为空时，调用 length 不是一个好的策略。如果我们要处理的列表是有限的，它可能会做比我们想要的更多的工作。由于 Haskell 让我们轻松创建无限列表，CareTless 使用 length 甚至可能导致无限循环。

这里更合适的函数是 null，它在常数时间内运行。更妙的是，使用 null 使我们的代码表明我们真正关心列表的什么属性。这里有两种改进表达 myDumbExample 的方法：

```haskell
-- file: ch04/EfficientList.hs
mySmartExample xs = if not (null xs)
                    then head xs
                    else 'Z'

myOtherExample (x:_) = x
myOtherExample [] = 'Z'
```

### 偏函数和全函数

仅针对有效输入子集定义返回值的函数称为偏函数（调用 error 不算返回值为合格！）。我们称在整个输入域上返回有效结果的函数为全函数。

知道您使用的函数是偏函数还是全函数始终是一个好主意。用不能处理的输入调用偏函数可能是 Haskell 程序中简单、可避免 bug 的最大单一来源。

一些 Haskell 程序员甚至将偏函数命名为以 unsafe 等前缀开头，这样他们就不会意外地自伤脚。标准 Prelude 定义了相当多的"不安全"偏函数（如 head）而没有同时提供"安全"全等价物，这可以说是一个缺陷。

### 更多简单列表操作

Haskell 对追加函数的名称是 (++)：

```ghci
ghci> :type (++)
(++) :: [a] -> [a] -> [a]
ghci> "foo" ++ "bar"
"foobar"
ghci> [] ++ [1,2,3]
[1,2,3]
ghci> [True] ++ []
[True]
```

concat 函数接受一个列表的列表，所有列表类型相同，并将它们连接成单个列表：

```ghci
ghci> :type concat
concat :: [[a]] -> [a]
ghci> concat [[1,2,3], [4,5,6]]
[1,2,3,4,5,6]
```

它移除一层嵌套：

```ghci
ghci> concat [[[1,2],[3]], [[4],[5],[6]]]
[[1,2],[3],[4],[5],[6]]
ghci> concat (concat [[[1,2],[3]], [[4],[5],[6]]])
[1,2,3,4,5,6]
```

reverse 函数以相反顺序返回列表的元素：

```ghci
ghci> :type reverse
reverse :: [a] -> [a]
ghci> reverse "foo"
"oof"
```

对于 Bool 列表，and 和 or 函数将它们的两个参数对应物 (&&) 和 (||) 推广到列表：

```ghci
ghci> :type and
and :: [Bool] -> Bool
ghci> and [True,False,True]
False
ghci> and []
True
ghci> :type or
or :: [Bool] -> Bool
ghci> or [False,False,False,True,False]
True
ghci> or []
False
```

它们有更多有用的对应物 all 和 any，它们对任何类型的列表进行操作。每一个都将谓词作为其第一个参数；如果该谓词对列表的每个元素都成功，all 返回 True，而如果谓词对列表中至少一个元素成功，any 返回 True：

```ghci
ghci> :type all
all :: (a -> Bool) -> [a] -> Bool
ghci> all odd [1,3,5]
True
ghci> all odd [3,1,4,1,5,9,2,6,5]
False
ghci> all odd []
True
ghci> :type any
any :: (a -> Bool) -> [a] -> Bool
ghci> any even [3,1,4,1,5,9,2,6,5]
True
ghci> any even []
False
```

### 处理子列表

take 函数，我们之前在"函数应用"第 22 页中讨论过，返回列表前 k 个元素的子列表。它的相反，drop，从列表开头丢弃 k 个元素：

```ghci
ghci> :type take
take :: Int -> [a] -> [a]
ghci> take 3 "foobar"
"foo"
ghci> take 2 [1]
[1]
ghci> :type drop
```

## 如何思考循环

与传统语言不同，Haskell 既没有 for 循环也没有 while 循环。如果我们有很多数据要处理，我们用什么代替？这个问题有几个可能的答案。

### 显式递归

从有循环的语言到没有循环的语言的一个直接方法是浏览几个例子，看看差异。这里有一个 C 函数，接受一个十进制数字字符串并将它们转换为一个整数：

```c
int as_int(char *str)
{
    int acc; /* accumulate the partial result */
    for (acc = 0; isdigit(*str); str++) {
    acc = acc * 10 + (*str - '0');
    }
    return acc;
}
```

鉴于 Haskell 没有任何循环结构，我们应该如何思考表示这样一个相当直接的代码？

我们不需要从写类型签名开始，但它有助于提醒我们正在处理什么：

```haskell
-- file: ch04/IntParse.hs
import Data.Char (digitToInt) -- we'll need ord shortly
asInt :: String -> Int
```

C 代码在遍历字符串时增量计算结果；Haskell 代码可以做同样的事情。然而，在 Haskell 中，我们可以将循环的等价物表示为一个函数。我们将 ours 称为 loop 只是为了保持事情清晰和明确：

```haskell
-- file: ch04/IntParse.hs
loop :: Int -> String -> Int
asInt xs = loop 0 xs
```

loop 的第一个参数是我们将使用的累加器变量。传入零等同于在循环开始时在 C 中初始化 acc 变量。

不要急于跳入糟糕的代码，让我们考虑我们必须处理的数据。我们熟悉的 String 只是 [Char] 的同义词，即字符列表。让我们最容易正确遍历的方法是考虑列表的结构：它要么是空的，要么是单个元素后跟列表的其余部分。

我们可以通过直接对列表类型的构造函数进行模式匹配来表达这种结构性思考。首先考虑简单的情况通常很方便；这里，这意味着我们将考虑空列表情况：

```haskell
-- file: ch04/IntParse.hs
loop acc [] = acc
```

空列表不仅仅意味着"输入字符串是空的"；这也是我们遍历非空列表的全部内容时将遇到的情况。所以我们不想在看到空列表时"出错"。相反，我们应该做一些明智的事情。这里，明智的事情是终止循环并返回我们的累积值。

我们必须考虑的另一种情况是输入列表非空时。我们需要处理列表的当前元素，以及列表的其余部分：

```haskell
-- file: ch04/IntParse.hs
loop acc (x:xs) = let acc' = acc * 10 + digitToInt x
                  in loop acc' xs
```

我们为累加器计算一个新值，并将其命名为 acc'。然后我们再次调用 loop 函数，传入更新后的值 acc' 和输入列表的其余部分。这相当于 C 中的循环开始新一轮。

### 变量名中的单引号

记住，单引号是在 Haskell 变量名中使用的合法字符，发音为"prime"。在 Haskell 程序中有一个常见的习惯用法涉及一个变量——比如说 foo——和另一个变量——比如说 foo'。我们通常可以假设 foo' 以某种方式与 foo 相关。

它通常是 foo 的新值，如刚才在我们的代码中所示。

有时我们会看到这个习惯用法被扩展，比如 foo''。由于跟踪附加到名称末尾的单引号数量很快变得乏味，庆幸的是，使用超过两个连续单引号的情况很少见。事实上，即使一个单引号也可能很容易被错过，这可能导致读者混淆。将单引号的使用视为你应该能够识别的编码约定可能更好，而不是你应该真正遵循的约定。

每次 loop 函数调用自身时，它都有一个累加器的新值，并消耗输入列表的一个元素。最终，它将到达列表的末尾，此时 [] 模式将匹配，递归调用将停止。

这个函数的效果如何？对于正整数，它是完全 cromulent：

```ghci
ghci> asInt "33"
33
```

但是因为我们专注于如何遍历列表，而不是错误处理，我们可怜的函数在我们尝试给它喂 nonsense 时表现不佳：

```ghci
ghci> asInt ""
0
ghci> asInt "potato"
*** Exception: Char.digitToInt: not a digit 'p'
```

我们将把修复函数缺陷的工作推迟到第 97 页的"练习"。

因为 loop 做的最后一件事只是调用自身，它是尾递归函数的一个例子。代码中还有另一个常见的习惯用法。考虑列表的结构，并分别处理空和非空情况，是一种称为结构递归的方法。

我们称非递归情况（当列表为空时）为基本情况（或有时称为终止情况）。我们会看到人们将函数调用自身的情况称为递归情况（令人惊讶！），或者他们可能向数学归纳法点头并称之为归纳情况。

作为一种有用的技术，结构递归不限于列表；我们也可以在其他代数数据类型上使用它。我们稍后会更多讨论它。

### 尾递归有什么大不了的？

在命令式语言中，循环在常量空间中执行。缺乏循环，我们在 Haskell 中使用尾递归函数代替。通常，递归函数每次应用自身时都会分配一些空间，所以它知道返回到哪里。

显然，如果递归函数为每个递归应用分配内存，它将相对于循环处于巨大劣势——这将需要线性空间而不是常量空间。然而，函数式语言实现检测尾递归的使用并转换尾递归调用以在常量空间中运行；这称为尾调用优化 (TCO)。

很少有命令式语言实现执行 TCO；这就是为什么在命令式语言中使用任何种类的雄心勃勃的函数式风格通常会导致内存泄漏和糟糕性能的原因。

### 转换每一片输入

考虑另一个 C 函数 square，它将数组中的每个元素平方：

```c
void square(double *out, const double *in, size_t length)
{
    for (size_t i = 0; i < length; i++) {
    out[i] = in[i] * in[i];
    }
}
```

这包含一个简单而常见的循环，它对其输入数组的每个元素执行完全相同的事情。我们如何在 Haskell 中编写这个循环？

```haskell
-- file: ch04/Map.hs
square :: [Double] -> [Double]
square (x:xs) = x*x : square xs
square []     = []
```

我们的 square 函数由两个模式匹配方程组成。第一个"解构"非空列表的开头，以获取其头部和尾部。它将第一个元素平方，然后将其放在新列表的前面，该列表是通过对空列表的剩余部分调用 square 构造的。第二个方程确保 square 在输入列表结束时停止。

square 的效果是构造一个与其输入列表长度相同的新列表，输入列表中的每个元素都被其在输出列表中的平方替代。

这是另一个这样的 C 循环，确保字符串中的每个字母都转换为大写：

```c
#include <ctype.h>
char *uppercase(const char *in)
{
    char *out = strdup(in);

    if (out != NULL) {
    for (size_t i = 0; out[i] != '\0'; i++) {
        out[i] = toupper(out[i]);
    }
    }
    return out;
}
```

让我们看一个 Haskell 等价物：

```haskell
-- file: ch04/Map.hs
import Data.Char (toUpper)
upperCase :: String -> String
upperCase (x:xs) = toUpper x : upperCase xs
upperCase []     = []
```

这里，我们从标准 Data.Char 模块导入 toUpper 函数，其中包含许多用于处理 Char 数据的有用函数。

我们的 upperCase 函数遵循类似于我们早期 square 函数的模式。它在输入列表为空时以空列表终止；当输入不为空时，它对第一个元素调用 toUpper，然后从对输入列表其余部分调用自身的结果构造新列表单元格。

这些例子遵循在 Haskell 中编写列表递归函数的常见模式。基本情况处理我们的输入列表为空的情况。递归情况处理非空列表；它对列表的头做一些事情，并递归地在尾部调用自身。

### 映射列表

我们刚刚定义的 square 和 upperCase 函数生成与其输入列表长度相同的新列表，并且它们对每个元素只执行一件工作。这是一个如此常见的模式，以至于 Haskell 的 Prelude 定义了一个函数 map，以便使其更容易。map 接受一个函数并将其应用于列表的每个元素，返回从这些应用的结果构造的新列表。

这里是我们使用 map 重写的 square 和 upperCase 函数：

```haskell
-- file: ch04/Map.hs
square2 xs = map squareOne xs
    where squareOne x = x * x
upperCase2 xs = map toUpper xs
```

这是我们第一次仔细看一个接受另一个函数作为参数的函数。我们可以通过简单检查其类型来了解 map 的很多信息：

```ghci
ghci> :type map
map :: (a -> b) -> [a] -> [b]
```

签名告诉我们 map 接受两个参数。第一个是一个函数，它接受一个类型 a 的值并返回另一个类型 b 的值。

因为 map 接受一个函数作为参数，我们称它为高阶函数。（尽管有这个名称，高阶函数并不神秘；它只是接受其他函数作为参数或返回函数的函数的术语。）

因为 map 抽象出了我们的 square 和 upperCase 函数共同的模式，以便我们可以更少样板地重用它，我们可以看看这些函数有什么共同点，并弄清楚如何自己实现它：

```haskell
-- file: ch04/Map.hs
myMap :: (a -> b) -> [a] -> [b]
myMap f (x:xs) = f x : myMap f xs
myMap _ _      = []
```

### 那些通配符在那里做什么？

如果您不熟悉函数式编程，以某种方式匹配模式的原因并不总是明显。例如，在前面代码中 myMap 的定义里，第一个方程绑定我们正在映射的函数到变量 f，但第二个对两个参数都使用通配符。怎么回事？

我们使用通配符代替 f 以表明我们不在方程右侧调用函数 f。列表参数呢？列表类型有两个构造函数。我们已经在定义 myMap 的第一个方程中匹配了非空构造函数。通过排除，第二个方程中的构造函数必然是空列表构造函数，所以不需要执行匹配以查看其值真正是什么。

作为风格问题，对于已知简单类型（如列表和 Maybe）使用通配符是可以的。对于更复杂或不太熟悉的类型，显式命名构造函数可能更安全、更易读。

我们尝试我们的 myMap 函数以确保它行为类似于标准 map：

```ghci
ghci> :module +Data.Char
ghci> map toLower "SHOUTING"
"shouting"
ghci> myMap toUpper "whispering"
"WHISPERING"
ghci> map negate [1,2,3]
[-1,-2,-3]
```

发现重复的习惯用法，然后将其实抽象化以便我们可以重用（并编写更少！代码）是 Haskell 编程的常见方面。虽然抽象不是 Haskell 独有的，但高阶函数使其非常容易。

### 选择输入的片段

对数据序列进行操作的另一个常见操作是梳理满足某些条件的元素。这里有一个函数，遍历数字列表并返回那些是奇数的数字。我们的代码有一个比我们早期函数稍微复杂的递归情况——仅当数字是奇数时才将其放入返回的列表中。使用卫冕很好地表达了这一点：

```haskell
-- file: ch04/Filter.hs
oddList :: [Int] -> [Int]
oddList (x:xs) | odd x     = x : oddList xs

```

让我们看看它的运行：

```ghci

```

再一次，这个习惯用法太常见了，Prelude 定义了一个函数 filter，我们已经介绍过了。它不再需要递归遍历列表的样板代码：

### 对集合计算一个答案

将集合归约为单个值也是很常见的。一个简单的例子是对列表的值求和：

```haskell
-- file: ch04/Sum.hs
mySum xs = helper 0 xs
    where helper acc (x:xs) = helper (acc + x) xs
          helper acc _      = acc
```

我们的 helper 函数是尾递归的，并使用累加器参数 acc 来保存列表的当前部分和。正如我们之前在 asInt 中看到的，这是在纯函数式语言中表示循环的"自然"方式。

对于更复杂一点的东西，让我们看看 Adler-32 校验和。它是一种流行的校验和算法；它将两个 16 位校验和连接成单个 32 位校验和。第一个校验和是所有输入字节的总和，加一。第二个是第一个校验和所有中间值的总和。在每种情况下，总和按 65521 计算。这里有一个直接的、未优化的 Java 实现（如果您不读 Java，可以跳过它）：

```java
public class Adler32
{
    private static final int base = 65521;
    public static int compute(byte[] data, int offset, int length)
    {
    int a = 1, b = 0;
    for (int i = offset; i < offset + length; i++) {
        a = (a + (data[i] & 0xff)) % base;
        b = (a + b) % base;
    }
    return (b << 16) | a;
    }
}
```

虽然 Adler-32 是一个简单的校验和，但由于涉及的位操作，这个代码并不是特别容易阅读。我们能用 Haskell 实现做得更好吗？

```haskell
-- file: ch04/Adler32.hs
import Data.Char (ord)
import Data.Bits (shiftL, (.&.), (.|.))

base = 65521

adler32 xs = helper 1 0 xs
    where helper a b (x:xs) = let a' = (a + (ord x .&. 0xff)) `mod` base
                                  b' = (a' + b) `mod` base
                              in helper a' b' xs
          helper a b _     = (b `shiftL` 16) .|. a
```

这个代码不完全比 Java 代码更容易理解，但让我们看看发生了什么。首先，我们引入了一些新函数。shiftL 函数实现逻辑左移；(.&.) 提供按位"与"；(.|.) 提供按位"或"。

又一次，我们的 helper 函数是尾递归的。我们将 Java 中每次循环迭代更新的两个变量转换为累加器参数。当我们的递归在输入列表末尾终止时，我们计算校验和并返回它。

如果我们退后一步，我们可以重构我们的 Haskell adler32 以更接近我们早期的 mySum 函数。我们可以使用一对作为累加器，而不是两个累加器参数：

```haskell
-- file: ch04/Adler32.hs
adler32_try2 xs = helper (1,0) xs
    where helper (a,b) (x:xs) =
              let a' = a + (ord x .&. 0xff)
                  b' = (a' + b) `mod` base
              in helper (a',b') xs
          helper (a,b) _     = (b `shiftL` 16) .|. a
```

为什么我们要进行这个看似无意义结构变更？因为正如我们已经看到的 map 和 filter，我们可以将 mySum 和 adler32_try2 共享的常见行为提取到一个高阶函数中。我们可以将此行为描述为"对列表的每个元素做某事，在我们进行时更新累加器，当我们完成时返回累加器。"

这种函数称为 fold，因为它"折叠"列表。有两种折叠列表的方法：foldl 用于从左侧（开始）折叠，foldr 用于从右侧（结束）折叠。

### 左侧折叠

这里是 foldl 的定义：

```haskell
-- file: ch04/Fold.hs
foldl :: (a -> b -> a) -> a -> [b] -> a
foldl step zero (x:xs) = foldl step (step zero x) xs
foldl _    zero []     = zero
```

foldl 函数接受一个"step"函数，其累加器的初始值，以及一个列表。"step"接受一个累加器和一个列表元素并返回一个新的累加器值。foldl 所做的是在当前累加器和列表元素上调用"stepper"，然后将新的累加器值传递给它自己以递归消耗列表的其余部分。

我们称 foldl 为左折叠，因为它从左（头部）到右消耗列表。

这是使用 foldl 重写的 mySum：

```haskell
-- file: ch04/Sum.hs
foldlSum xs = foldl step 0 xs
    where step acc x = acc + x
```

那个本地函数 step 只是将两个数字相加，所以我们只需使用加法运算符，并消除不必要的 where 子句：

```haskell
-- file: ch04/Sum.hs
niceSum :: [Integer] -> Integer
niceSum xs = foldl (+) 0 xs
```

注意我们的代码比我们原来的 mySum 简单多少。我们不再使用显式递归，因为 foldl 为我们处理了。我们将问题简化为两件事：累加器的初始值应该是什么（foldl 的第二个参数）以及如何更新累加器（(+) 函数）。作为一个额外的好处，我们的代码现在更短了，这使其更容易理解。

让我们通过手动写出在调用 niceSum [1,2,3] 时 foldl 在这里的求值的每个步骤来进行更深入的查看：

```haskell
-- file: ch04/Fold.hs
foldl (+) 0 (1:2:3:[])
          == foldl (+) (0 + 1)             (2:3:[])
          == foldl (+) ((0 + 1) + 2)       (3:[])
          == foldl (+) (((0 + 1) + 2) + 3) []
          ==           (((0 + 1) + 2) + 3)
```

我们可以使用 foldl 重写 adler32_try2，以让我们专注于重要的细节：

```haskell
-- file: ch04/Adler32.hs
adler32_foldl xs = let (a, b) = foldl step (1, 0) xs
                   in (b `shiftL` 16) .|. a
    where step (a, b) x = let a' = a + (ord x .&. 0xff)
                          in (a' `mod` base, (a' + b) `mod` base)
```

这里，我们的累加器是一对，所以 foldl 的结果也将是一对。当 foldl 返回时，我们将最终累加器拆开，然后将其位操作成"适当的"校验和。

### 为什么要使用 Folds、Maps 和 Filters？

一扫似乎 adler32_foldl 并不比 adler32_try2 短。在这种情况下为什么要使用 fold？这里的优势在于 folds 在 Haskell 中非常常见，并且它们具有规则的、可预测的行为。

这意味着具有一点经验的读者将更容易理解 fold 的使用，而不是使用显式递归的代码。fold 不会产生任何惊喜，但显式递归函数的行为并不是立即明显的。显式递归要求我们密切阅读以准确理解发生了什么。

这一推理路线适用于其他高阶库函数，包括我们已经看到的那些。因为它们是具有明确定义行为的库函数，我们只需要学习它们做什么一次，并且在需要理解使用它们的任何代码时都会占优势。可读性方面的这些改进也延续到编写代码。一旦我们开始用高阶函数思考，我们将更快地产生简洁代码。

### 从右侧折叠

foldl 的对应物是 foldr，它从列表的右侧折叠：

```haskell
-- file: ch04/Fold.hs
foldr :: (a -> b -> b) -> b -> [a] -> b
foldr step zero (x:xs) = step x (foldr step zero xs)
foldr _    zero []     = zero
```

让我们使用与之前在"左侧折叠"第 92 页中 niceSum 相同的手动求值过程来关注 foldr (+) 0 [1,2,3]：

```haskell
-- file: ch04/Fold.hs
foldr (+) 0 (1:2:3:[])
          == 1 +           foldr (+) 0 (2:3:[])
          == 1 + (2 +      foldr (+) 0 (3:[])
          == 1 + (2 + (3 + foldr (+) 0 []))
          == 1 + (2 + (3 + 0))
```

从左右两侧的空列表元素和括号的位置可以看出 foldl 和 foldr 之间的差异应该清楚。对于 foldl，空列表元素在左侧，所有括号向左分组。对于 foldr，零值在右侧，括号向右分组。

关于 foldr 如何工作有一个可爱的直观解释：它用零值替换空列表，并用 step 函数的应用程序替换列表中的每个构造函数：

```haskell
-- file: ch04/Fold.hs
1 : (2 : (3 : []))
1 + (2 + (3 + 0 ))
```

初一看，foldr 似乎不如 foldl 有用：什么是有从右侧折叠的函数？但考虑一下我们之前在第 90 页的"选择输入的片段"中遇到的 Prelude 的 filter 函数。如果我们使用显式递归编写 filter，它会看起来像这样：

```haskell
-- file: ch04/Fold.hs
filter :: (a -> Bool) -> [a] -> [a]
filter p []   = []
filter p (x:xs)
    | p x       = x : filter p xs
    | otherwise = filter p xs
```

然而，令人惊讶的是，我们可以使用 foldr 编写 filter：

```haskell
-- file: ch04/Fold.hs
myFilter p xs = foldr step [] xs
    where step x ys | p x       = x : ys
                    | otherwise = ys
```

这是可能导致我们头痛的那种定义，所以让我们稍微深入地检查它。像 foldl 一样，foldr 接受一个函数和一个基本情况（当输入列表为空时做什么）作为参数。从阅读 filter 的类型，我们知道我们的 myFilter 函数必须返回与其消耗的类型相同的列表，所以基本情况应该是这种类型的列表，而 step 辅助函数必须返回一个列表。

因为我们知道 foldr 一次对输入列表的一个元素调用 step，然后以累加器作为其第二个参数，step 的动作必须非常简单。如果谓词返回 True，它会将该元素推到累积列表上；否则，列表保持不变。

我们可以用 foldr 表达的函数类称为原始递归。大量令人惊讶的列表操作函数是原始递归的。例如，这里是用 foldr 编写的 map：

```haskell
-- file: ch04/Fold.hs
myMap :: (a -> b) -> [a] -> [b]
myMap f xs = foldr step [] xs
    where step x ys = f x : ys
```

事实上，我们甚至可以用 foldr 编写 foldl！

```haskell
-- file: ch04/Fold.hs
myFoldl :: (a -> b -> a) -> a -> [b] -> a
myFoldl f z xs = foldr step id xs z
    where step x g a = g (f a x)
```

### 用 foldr 理解 foldl

如果您想为自己设置一个坚实的挑战，请尝试遵循我们用 foldr 定义 foldl 的定义。注意：这不是微不足道的！你可能想要手边有这些东西：一些头痛药和水，ghci（这样您可以找出 id 函数的作用），还有铅笔和纸。

你会想像我们刚才概述的那样遵循相同的手动求值过程，以查看 foldl 和 foldr 真正在做什么。如果您卡住了，在阅读第 100 页的"部分函数应用和柯里化"后可能会发现任务更容易。

回到我们早期对 foldr 做什么的直观解释，另一个有用的思考方式是它转换其输入列表。它的前两个参数是"对列表的每个头/尾元素做什么"和"用什么替换列表末尾"。

因此，使用 foldr 的"恒等"转换用自身替换空列表，并对每个头/尾对应用列表构造函数：

```haskell
-- file: ch04/Fold.hs
identity :: [a] -> [a]
identity xs = foldr (:) [] xs
```

它将列表转换为其自身的副本：

```ghci
ghci> identity [1,2,3]
[1,2,3]
```

如果 foldr 将列表的末尾替换为某个其他值，这给了我们另一种查看 Haskell 的列表追加函数 (++) 的方式：

```ghci
ghci> [1,2,3] ++ [4,5,6]
[1,2,3,4,5,6]
```

要追加一个列表到另一个列表，我们所要做的就是用第二个列表替换第一个列表的末尾：

```haskell
-- file: ch04/Fold.hs
append :: [a] -> [a] -> [a]
append xs ys = foldr (:) ys xs
```

让我们试试：

```ghci
ghci> append [1,2,3] [4,5,6]
[1,2,3,4,5,6]
```

这里，我们用另一个列表构造函数替换每个列表构造函数，但我们用我们想要追加到第一个列表末尾的列表替换空列表。

正如我们对折叠的扩展处理应该表明的，foldr 函数与我们更基本的列表函数列表中更基本的成员几乎一样重要。它可以增量消耗和生成列表，这使其有助于编写惰性数据处理代码。

### 左折叠、惰性和空间泄漏

为保持我们的初步讨论简单，我们在大部分章节中使用 foldl。这对测试很方便，但我们永远不会在实践中使用 foldl。原因是与 Haskell 的非严格求值有关。如果我们应用 foldl (+) [1,2,3]，它会求值为表达式 (((0 + 1) + 2) + 3)。如果我们重新审视函数如何展开，可以看到这种情况发生：

```haskell
-- file: ch04/Fold.hs
foldl (+) 0 (1:2:3:[])
          == foldl (+) (0 + 1)             (2:3:[])
          == foldl (+) ((0 + 1) + 2)       (3:[])
          == foldl (+) (((0 + 1) + 2) + 3) []
          ==           (((0 + 1) + 2) + 3)
```

最终表达式在其值被需要之前不会被求值。在被求值之前，它必须被存储为一个 thunk。不出所料，存储 thunk 比存储单个数字更昂贵，并且 thunked 表达式越复杂，它需要的空间就越大。对于像算术这样便宜的东西，thunking 一个表达式比立即求值它计算成本更高。因此，我们在空间和时间上都付出了代价。

当 GHC 求值一个 thunked 表达式时，它使用内部堆栈来执行此操作。因为 thunked 表达式可能是无限大的，GHC 为此堆栈的最大大小设置了一个固定限制。由于这个限制，我们可以在 ghci 中尝试一个大的 thunked 表达式，而不必担心它可能会消耗所有内存：

```ghci
ghci> foldl (+) 0 [1..1000]
500500
```

从这个展开，我们可以推测这创建了一个由 1,000 个整数和 999 个 (+) 应用程序组成的 thunk。表示一个数字需要这么多内存和努力！对于更大的表达式，虽然大小仍然适中，但结果更显着：

```ghci
ghci> foldl (+) 0 [1..1000000]
*** Exception: stack overflow
```

在小表达式上，foldl 将正确但缓慢地工作，因为它招致了 thunking 开销。我们将这种不可见的 thunking 称为空间泄漏，因为我们的代码运行正常，但使用的内存比应该的多得多。

在更大的表达式上，具有空间泄漏的代码会失败，如上所示。foldl 的空间泄漏是新 Haskell 程序员的经典障碍。幸运的是，这很容易避免。

Data.List 模块定义了一个名为 foldl' 的函数，类似于 foldl，但不会构建 thunk。两者行为差异立即明显：

```ghci
ghci> foldl  (+) 0 [1..1000000]
*** Exception: stack overflow
ghci> :module +Data.List
ghci> foldl' (+) 0 [1..1000000]
500000500000
```

由于 foldl 的 thunking 行为，在真实程序中避免使用此函数是明智的，即使它不会立即失败，它也会不必要地低效。相反，导入 Data.List 并使用 foldl'。

## 匿名（lambda）函数

在我们迄今为止看到的许多函数定义中，我们编写了简短的辅助函数：

```haskell
-- file: ch04/Partial.hs
isInAny needle haystack = any inSequence haystack
    where inSequence s = needle `isInfixOf` s
```

Haskell 让我们编写完全匿名的函数，我们可以用它来避免给辅助函数命名的需要。匿名函数通常被称为"lambda"函数，是对 lambda 演算的点头。我们用反斜杠字符 (\) 引入匿名函数，发音为 lambda。† 后面是函数的参数（可以包括模式），然后是箭头 (->) 以引入函数体。

Lambda 最容易通过示例说明。这里是使用匿名函数重写的 isInAny：

```haskell
-- file: ch04/Partial.hs
isInAny2 needle haystack = any (\s -> needle `isInfixOf` s) haystack
```

我们在这里将 lambda 括在括号中，以便 Haskell 可以判断函数体在哪里结束。

在各方面，匿名函数与有名字的函数行为完全相同，但 Haskell 对我们如何定义它们放置了一些重要限制。最重要的是，虽然我们可以编写使用包含不同模式和卫冕的多个子句的正常函数，但 lambda 在其定义中只能有一个子句。

对单个子句的限制限制了我们可以在 lambda 定义中使用模式的方式。我们通常会编写具有多个子句的正常函数来覆盖不同的模式匹配可能性：

```haskell
-- file: ch04/Lambda.hs
safeHead (x:_) = Just x
safeHead _ = Nothing
```

但因为我们不能编写多个子句来定义 lambda，我们必须确保我们使用的任何模式都会匹配：

```haskell
-- file: ch04/Lambda.hs
unsafeHead = \(x:_) -> x
```

如果我们使用模式匹配失败的值调用此 unsafeHead 定义，它将爆炸：

```ghci
ghci> :type unsafeHead
unsafeHead :: [t] -> t
ghci> unsafeHead [1]
1
ghci> unsafeHead []
*** Exception: Lambda.hs:7:13-23: Non-exhaustive patterns in lambda
```

该定义通过类型检查，所以它会编译，错误会在运行时发生。这个故事的寓意是：在定义匿名函数时小心使用模式：确保你的模式不会失败！

关于之前显示的 isInAny 和 isInAny2 函数，另一件要注意的事情是，第一个版本使用具有名称的辅助函数，比将匿名函数放入中间的版本略易读。命名的辅助函数不会破坏其使用的函数的"流程"，并且明智选择的名称为我们提供了一点关于函数预期做什么的信息。

相比之下，当我们在函数体中间遇到 lambda 时，我们必须切换齿轮并相当仔细地阅读其定义以理解它做什么。因此，为了可读性和可维护性，我们倾向于在许多情况下避免使用 lambda，即使我们可以使用它们来从函数定义中减少几个字符。通常，我们会使用部分应用的函数，从而产生比 lambda 或显式函数更清晰、更易读的代码。不知道什么是部分应用的函数？继续阅读！

我们无意暗示这些警告表明 lambda 是无用的，只是当我们考虑使用它们时，我们应该意识到潜在的陷阱。在后面的章节中，我们将看到它们通常作为"胶水"是无价的。

## 部分函数应用和柯里化

您可能想知道为什么在函数类型签名中 -> 箭头似乎用于两种目的：

```ghci
ghci> :type dropWhile
dropWhile :: (a -> Bool) -> [a] -> [a]
```

看起来 -> 在分隔 dropWhile 的参数之间，但也将参数与返回类型分开。实际上 -> 只有一种含义：它表示一个函数，它接受左边的参数类型并返回右边的值类型。

这里的含义非常重要。在 Haskell 中，所有函数只接受一个参数。虽然 dropWhile 看起来像接受两个参数的函数，它实际上是一个函数的函数，它返回一个接受一个参数的函数。这里是一个完全有效的 Haskell 表达式：

```ghci
ghci> :module +Data.Char
ghci> :type dropWhile isSpace
dropWhile isSpace :: [Char] -> [Char]
```

好吧，那看起来有用。值 dropWhile isSpace 是一个从字符串中剥离前导空白字符的函数。这有什么用？作为一个例子，我们可以将其作为参数传递给高阶函数：

```ghci
ghci> map (dropWhile isSpace) [" a","f","   e"]
["a","f","e"]
```

每次我们向函数提供一个参数时，我们都可以从其类型签名"切下"一个元素。让我们以 zip3 为例看看我们的意思；这是一个将三个列表压缩成三元组列表的函数：

```ghci
ghci> :type zip3
zip3 :: [a] -> [b] -> [c] -> [(a, b, c)]
ghci> zip3 "foo" "bar" "quux"
[('f','b','q'),('o','a','u'),('o','r','u')]
```

如果我们只应用 zip3 的一个参数，我们得到一个接受两个参数的函数。无论我们为这个复合函数提供什么参数，其第一个参数将始终是我们指定的固定值：

```ghci
ghci> :type zip3 "foo"
zip3 "foo" :: [b] -> [c] -> [(Char, b, c)]
ghci> let zip3foo = zip3 "foo"
ghci> :type zip3foo
zip3foo :: [b] -> [c] -> [(Char, b, c)]
ghci> (zip3 "foo") "aaa" "bbb"
[('f','a','b'),('o','a','b'),('o','a','b')]
ghci> zip3foo "aaa" "bbb"
[('f','a','b'),('o','a','b'),('o','a','b')]
ghci> zip3foo [1,2,3] [True,False,True]
[('f',1,True),('o',2,False),('o',3,True)]
```

当我们向函数提供比函数能接受的更少的参数时，我们称之为函数的部分应用——我们仅将函数应用于其部分参数。

在前面的示例中，我们有一个部分应用的函数 zip3 "foo"，和一个新函数 zip3foo。我们可以看到两者的类型签名和行为是相同的。

这也适用于我们修复两个参数，给我们一个仅一个参数的函数：

```ghci
ghci> let zip3foobar = zip3 "foo" "bar"
ghci> :type zip3foobar
zip3foobar :: [c] -> [(Char, Char, c)]
ghci> zip3foobar "quux"
[('f','b','q'),('o','a','u'),('o','r','u')]
ghci> zip3foobar [1,2]
[('f','b',1),('o','a',2)]
```

部分函数应用让我们避免编写繁琐的一次性函数。对于这个目的，它通常比我们之前在本章介绍的匿名函数更有用。回看我们定义的 isInAny 函数，这里是我们如何使用部分应用的函数而不是命名辅助函数或 lambda：

```haskell
-- file: ch04/Partial.hs
isInAny3 needle haystack = any (isInfixOf needle) haystack
```

这里，表达式 isInfixOf needle 是部分应用的函数。我们采用函数 isInfixOf 并将其第一个参数"修复"为我们参数列表中的 needle 变量。这给了我们一个部分应用的函数，其类型和行为与早期定义中的辅助函数和 lambda 完全相同。

部分函数应用以逻辑学家 Haskell Curry 命名，称为柯里化（这是 Haskell 语言名称的来源）。

作为另一个使用的柯里化示例，让我们回到我们在"左侧折叠"第 92 页中写的列表求和函数：

```haskell
-- file: ch04/Sum.hs
niceSum :: [Integer] -> Integer
niceSum xs = foldl (+) 0 xs
```

我们不需要完全应用 foldl；我们可以从参数列表和 foldl 的参数中省略列表 xs，我们将得到一个更紧凑的函数，其类型相同：

```haskell
-- file: ch04/Sum.hs
nicerSum :: [Integer] -> Integer
nicerSum = foldl (+) 0
```

## 片段

Haskell 提供了一个方便的表示法快捷方式来让我们以中缀样式编写部分应用的函数。如果我们把运算符括在括号中，我们可以在括号内提供其左或右参数来获得部分应用的函数。这种部分应用称为片段：

```ghci
ghci> (1+) 2
3
ghci> map (*3) [24,36]
[72,108]
ghci> map (2^) [3,5,7,9]
[8,32,128,512]
```

如果在片段内提供左参数，那么调用结果函数时提供一个参数会提供运算符的右参数，反之亦然。

回想一下，我们可以将函数名包装在反引号中以将其用作中缀运算符。这允许我们使用带有函数的片段：

```ghci
ghci> :type (`elem` ['a'..'z'])
(`elem` ['a'..'z']) :: Char -> Bool
```

前面的定义修复了 elem 的第二个参数，给了我们一个检查其参数是否为小写字母的函数：

```ghci
ghci> (`elem` ['a'..'z']) 'f'
True
```

将其用作 all 的参数，我们得到一个检查整个字符串是否全为小写的函数：

```ghci
ghci> all (`elem` ['a'..'z']) "Frobozz"
False
```

如果我们使用这种风格，我们可以进一步改进我们早期的 isInAny3 函数：

```haskell
-- file: ch04/Partial.hs
isInAny4 needle haystack = any (needle `isInfixOf`) haystack
```

## As-模式

Haskell 的 Data.List 模块中的 tails 函数概括了我们早期介绍的 tail 函数。它不是返回列表的一个"尾部"，而是返回所有尾部：

```ghci
ghci> :m +Data.List
ghci> tail "foobar"
"oobar"
ghci> tail (tail "foobar")
"obar"
ghci> tails "foobar"
["foobar","oobar","obar","bar","ar","r",""]
```

这些字符串中的每一个都是初始字符串的后缀，所以 tails 生成所有后缀的列表，加上末尾的一个额外空列表。它总是产生那个额外的空列表，即使其输入列表为空：

```ghci
ghci> tails []
[[]]
```

如果我们想要一个行为类似于 tails 但仅返回非空后缀的函数怎么办？一种可能性是我们自己编写版本。我们将使用一个新表示法，@ 符号：

```haskell
-- file: ch04/SuffixTree.hs
suffixes :: [a] -> [[a]]
suffixes xs@(_:xs') = xs : suffixes xs'
suffixes _ = []
```

模式 xs@(_:xs') 称为 as-模式，意思是"将变量 xs 绑定到与 @ 符号右侧匹配的值"。

在我们的例子中，如果 @ 后的模式匹配，xs 将绑定到匹配的整个列表，而 xs' 将绑定到列表除头部外的所有部分（我们使用通配符 (_) 模式来表示我们不关心列表头部值）：

```ghci
ghci> tails "foo"
["foo","oo","o",""]
ghci> suffixes "foo"
["foo","oo","o"]
```

as-模式使我们的代码更易读。要查看它如何提供帮助，让我们比较一个没有 as-模式的定义：

```haskell
-- file: ch04/SuffixTree.hs
noAsPattern :: [a] -> [[a]]
noAsPattern (x:xs) = (x:xs) : noAsPattern xs
noAsPattern _ = []
```

这里，我们在模式匹配中解构的列表只是在函数体中直接重新组合在一起。

as-模式比简单可读性更有实际用途：它们可以帮助我们共享数据而不是复制它。在我们对 noAsPattern 的定义中，当我们匹配 (x:xs) 时，我们在函数体中构造了一个新的副本。这导致我们在运行时分配一个新的列表节点。这可能很便宜，但不是免费的。相比之下，当我们用 as-模式定义 suffixes 时，我们重用了我们匹配的现有值 xs。因为我们重用了现有值，我们避免了一点分配。

## 通过组合实现代码重用

引入一个与现有 tails 函数做几乎相同事情的新函数 suffixes 似乎是一种耻辱。当然我们可以做得更好？

回想一下我们在第 77 页的"处理列表"中介绍的 init 函数——它返回列表除最后一个元素外的所有元素：

```haskell
-- file: ch04/SuffixTree.hs
suffixes2 xs = init (tails xs)
```

这个 suffixes2 函数行为与 suffixes 完全相同，但只有一行代码：

```ghci
ghci> suffixes2 "foo"
["foo","oo","o"]
```

如果我们退后一步，我们看到了模式的曙光。我们正在应用一个函数，然后对其结果应用另一个函数。让我们将该模式变成一个函数定义：

```haskell
-- file: ch04/SuffixTree.hs
compose :: (b -> c) -> (a -> b) -> a -> c
compose f g x = f (g x)
```

现在我们有了一个函数 compose，我们可以用来将两个其他函数"粘合"在一起：

```haskell
-- file: ch04/SuffixTree.hs
suffixes3 xs = compose init tails xs
```

Haskell 的自动柯里化让我们删除 xs 变量，所以我们可以使我们的定义更短：

```haskell
-- file: ch04/SuffixTree.hs
suffixes4 = compose init tails
```

幸运的是，我们不需要编写我们自己的 compose 函数。像这样将函数插入彼此非常常见，Prelude 通过 (.) 运算符提供函数组合：

```haskell
-- file: ch04/SuffixTree.hs
suffixes5 = init . tails
```

(.) 运算符不是一种特殊的语言语法——它只是一个正常的运算符：

```ghci
ghci> :type (.)
(.) :: (b -> c) -> (a -> b) -> a -> c
ghci> :type suffixes
suffixes :: [a] -> [[a]]
ghci> :type suffixes5
suffixes5 :: [a] -> [[a]]
ghci> suffixes5 "foo"
["foo","oo","o"]
```

只要（当然）每个 (.) 右侧函数的返回类型与左侧函数可以接受的参数类型相匹配，我们就可以随时通过用 (.) 缝合在一起的组合函数链创建新函数。

作为一个例子，让我们解决一个简单的谜题。计算以大写字母开头的字符串中的单词数：

```ghci
ghci> :module +Data.Char
ghci> let capCount = length . filter (isUpper . head) . words
ghci> capCount "Hello there, Mom!"
2
```

我们可以通过检查其各个部分来理解这个组合函数的作用。(.) 函数是右结合的，所以我们将从右到左进行：

```ghci
ghci> :type words
words :: String -> [String]
```

words 函数的结果类型是 [String]，所以 (.) 左侧的任何内容必须接受兼容参数：

```ghci
ghci> :type isUpper . head
isUpper . head :: [Char] -> Bool
```

如果单词以大写字母开头，此函数返回 True（可以在 ghci 中尝试），因此 filter (isUpper . head) 返回仅包含以大写字母开头的单词的 Strings 列表：

```ghci
ghci> :type filter (isUpper . head)
filter (isUpper . head) :: [[Char]] -> [[Char]]
```

因为此表达式返回一个列表，剩下的一切是计算列表的长度，我们用另一个组合来完成。

这里是另一个来自真实应用的例子。我们想从与 libpcap（一个流行的网络数据包过滤库）一起发布的 C 头文件中提取宏名称列表。头文件包含大量以下形式的定义：

```c
#define DLT_EN10MB      1       /* Ethernet (10Mb) */
#define DLT_EN3MB       2       /* Experimental Ethernet (3Mb) */
#define DLT_AX25        3       /* Amateur Radio AX.25 */
```

我们的目标是从中提取 DLT_EN10MB 和 DLT_AX25 等名称：

```haskell
-- file: ch04/dlts.hs
import Data.List (isPrefixOf)

dlts :: String -> [String]
dlts = foldr step [] . lines
```

我们将整个文件视为一个字符串，用 lines 将其拆分，然后对结果行列表应用 foldr step []。step 辅助函数对单个行进行操作：

```haskell
-- file: ch04/dlts.hs
  where step l ds
          | "#define DLT_" `isPrefixOf` l = secondWord l : ds
          | otherwise                     = ds
        secondWord = head . tail . words
```

如果我们的守卫表达式匹配宏定义，我们将宏的名称加入到返回列表的头部；否则，我们保持列表不变。

虽然体内单个函数对我们来说现在已经很熟悉了，但将这样的组合链在一起可能需要一点练习。让我们一步步地走过这个过程。

再一次，我们从右到左进行。第一个函数是 words：

```ghci
ghci> :type words
words :: String -> [String]
ghci> words "#define DLT_CHAOS    5"
["#define","DLT_CHAOS","5"]
```

然后我们对 words 的结果应用 tail：

```ghci
ghci> :type tail
tail :: [a] -> [a]
ghci> tail ["#define","DLT_CHAOS","5"]
["DLT_CHAOS","5"]
ghci> :type tail . words
tail . words :: String -> [String]
ghci> (tail . words) "#define DLT_CHAOS    5"
["DLT_CHAOS","5"]
```

最后，对 drop 1 . words 的结果应用 head 将给出我们宏的名称：

```ghci
ghci> :type head . tail . words
head . tail . words :: String -> String
ghci> (head . tail . words) "#define DLT_CHAOS    5"
"DLT_CHAOS"
```

### 明智地使用你的头部

在本章前面在"安全和理智地处理崩溃函数"第 79 页警告不安全的列表函数之后，这里我们调用 head 和 tail，这两个都是那些不安全的列表函数。怎么回事？

在这种情况下，我们可以通过检查来确保自己免受运行时失败。在 step 的定义中，模式守卫包含两个词，所以当我们对通过守卫的任何字符串应用 words 时，我们将至少有两个元素的列表："#define" 和一些以 "DLT_" 开头的宏。

这种推理是我们应该做的，以说服自己调用偏函数时我们的代码不会爆炸。不要忘记我们之前的忠告：调用这样的不安全函数需要小心，并且经常会以微妙的方式使我们的代码更脆弱。如果出于某种原因我们修改模式守卫只包含一个词，我们可能会暴露自己崩溃的可能性，因为函数体假定它将收到两个词。

## 编写可读代码的提示

到目前为止，在本章中，我们遇到了两个诱人的 Haskell 特性：尾递归和匿名函数。虽然这些都很好，但我们不经常想使用它们。

许多列表操作最容易使用 map、take 和 filter 等库函数的组合来表达。毫无疑问，需要一些练习来习惯使用这些。作为回报，我们可以更快地编写和阅读代码，并且更少的 bug。

原因很简单。尾递归函数定义与命令式语言中的循环有同样的问题：它完全是通用的。它可能执行一些过滤、一些映射，或者其他谁知道的什么。我们被迫详细查看函数的整个定义以查看它真正在做什么。相比之下，map 和大多数其他列表操作函数只做一件事。我们可以理所当然地认为这些简单的构建块做了什么，可以专注于代码试图表达的想法，而不是它如何操作其输入的细节。

两个折叠位于尾递归函数（具有完全通用性）和我们的列表操作函数工具箱（每个只做一件事）之间的中间地带。理解 fold 比，比如说，组合 map 和 filter 做同样的事情，需要更多的努力，但它的行为比尾递归函数更规则、更可预测。作为一般规则，如果你可以用一些库函数组合，就不要使用 fold，否则尽量使用 fold 而不是手动编写的尾递归循环。

至于匿名函数，它们往往会中断阅读代码时的"流程"。写出本地函数定义在 let 或 where 子句中，然后使用它来代替放入匿名函数，通常一样容易。命名函数的相对优势是双重的：当我们在使用它的代码中阅读时，我们不需要理解函数的定义，并且明智选择的函数名称充当一小段本地文档。

## 空间泄漏和严格求值

我们之前讨论的 foldl 函数不是空间泄漏唯一可能在 Haskell 代码中发生的地方。我们将用它来说明非严格求值有时可能出现问题，以及如何解决出现的困难。

### 你需要马上知道所有这些吗？

完全合理的是跳过本节，直到你在"野外"遇到空间泄漏。只要您在使用 foldr 生成列表时使用 foldr，否则使用 foldl'，空间泄漏在实践中不太可能在相当长的一段时间内打扰您。

### 使用 seq 避免空间泄漏

我们称未惰性求值的表达式为严格的，所以 foldl' 是一个严格的左折叠。它通过使用名为 seq 的特殊函数绕过 Haskell 通常的非严格求值：

```haskell
-- file: ch04/Fold.hs
foldl' _    zero []     = zero
foldl' step zero (x:xs) =
    let new = step zero x
    in  new `seq` foldl' step new xs
```

这个 seq 函数有一个特殊的类型，暗示它不遵守通常的规则：

```ghci
ghci> :type seq
seq :: a -> t -> t
```

它操作如下：当求值 seq 表达式时，它强制其第一个参数被求值，然后返回其第二个参数。它实际上不对第一个参数做任何事情。seq 存在的原因仅仅是强制该值被求值。

让我们走一个简短的应用程序，看看会发生什么：

```haskell
-- file: ch04/Fold.hs
foldl' (+) 1 (2:[])
```

它展开如下：

```haskell
-- file: ch04/Fold.hs
let new = 1 + 2
in new `seq` foldl' (+) new []
```

使用 seq 强制求值 new 到 3 并返回其第二个参数：

```haskell
-- file: ch04/Fold.hs
foldl' (+) 3 []
```

我们最终得到以下结果：

```haskell
-- file: ch04/Fold.hs
3
```

由于 seq，眼前没有 thunk。

### 学习使用 seq

没有一些指导，有效地使用 seq 有一点神秘。这里有一些有效使用它的有用规则。

为了有任何效果，seq 表达式必须是表达式中第一个被求值的内容：

```haskell
-- file: ch04/Fold.hs
-- incorrect: seq is hidden by the application of someFunc
-- since someFunc will be evaluated first, seq may occur too late
hiddenInside x y = someFunc (x `seq` y)

-- incorrect: a variation of the above mistake
hiddenByLet x y z = let a = x `seq` someFunc y
                    in anotherFunc a z

-- correct: seq will be evaluated first, forcing evaluation of x
onTheOutside x y = x `seq` someFunc y
```

要严格求值几个值，请将 seq 的应用程序链接在一起：

```haskell
-- file: ch04/Fold.hs
chained x y z = x `seq` y `seq` someFunc z
```

一个常见的错误是尝试将 seq 与两个不相关的表达式一起使用：

```haskell
-- file: ch04/Fold.hs
badExpression step zero (x:xs) =
    seq (step zero x)
        (badExpression step (step zero x) xs)
```

这里，明显的意图是严格求值 step zero x。因为表达式在函数体中重复了，严格求值它的第一个实例对第二个没有影响。从 foldl' 的定义中使用 let 只是说明如何正确实现此效果。

当求值表达式时，seq 在到达构造函数时立即停止。对于像数字这样的简单类型，这意味着它将完全对它们求值。代数数据类型是另一回事。考虑值 (1+2):(3+4):[]。如果我们对它应用 seq，它将对 (1+2) thunk 求值。因为它会在第一个 (:) 构造函数处停止，它对第二个 thunk 没有影响。对于元组也是如此：seq ((1+2),(3+4)) True 不会对元组内的 thunk 产生任何影响，因为它立即遇到元组的构造函数。

如有必要，我们可以使用常规函数式编程技术来绕过这些限制：

```haskell
-- file: ch04/Fold.hs
strictPair (a,b) = a `seq` b `seq` (a,b)
strictList (x:xs) = x `seq` x : strictList xs
strictList []     = []
```

重要的是要理解 seq 不是免费的：它必须在运行时执行检查以查看表达式是否已被求值。谨慎使用它。例如，虽然我们的 strictPair 函数对一对的内容求值直到第一个构造函数，但它增加了模式匹配、两个 seq 应用程序和构造新元组的开销。如果我们要在一段基准测试的内循环中测量其性能，我们可能会发现它减慢了程序。

除了其性能成本（如果过度使用）外，seq 不是内存消耗问题的万能奇迹。仅仅因为你可以严格求值某事并不意味着你应该。粗心使用 seq 可能什么也不做，移动现有空间泄漏，或引入新的泄漏。

关于 seq 是否必要以及它如何工作的最好的指导是性能测量和分析，我们将在第 25 章介绍。从经验测量的基础，您将培养对 seq 最有用的时候的可靠感觉。
